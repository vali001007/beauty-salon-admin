# Ami_Agent 新一代美业门店运营智能体产品需求文档

版本：v1.0  
日期：2026-06-24  
适用范围：Ami_Agent 独立产品入口、Ami_Core 管理端业务能力、后端业务 API、全新 Agent Runtime、全新自动化执行引擎、权限与审计系统  
产品定位：洞悉美业的新一代门店运营智能体  
设计约束：Ami_Agent 为全新产品形态，不参考、不复用现有 Ami_Aura 与 web app 的界面结构、交互框架和视觉风格。
系统边界：现有 AI Gateway 与现有自动化任务系统已验证不满足 Ami_Agent 目标，不作为 Ami_Agent 的产品或技术底座复用。

---

## 1. 背景与定位

美业门店的日常经营正在从“人找功能、手工看报表、凭经验决策”转向“人表达目标、系统理解业务、智能体调度能力并完成闭环”。门店管理者、店长、前台、美容师、财务、运营等角色不应该被复杂页面、分散菜单和大量报表拖慢，而应该能直接用自然语言表达需求，由 Ami_Agent 识别问题、查询真实数据、给出可执行建议，并在授权范围内完成业务操作。

Ami_Agent 的定位不是普通聊天机器人，也不是报表问答助手，而是一个真正懂美业门店经营的运营智能体：

- 懂美业业务对象：客户、预约、服务、疗程、卡项、项目、商品、库存、收银、营销、员工、排班、提成、财务、会员、门店经营。
- 懂经营目标：拉新、复购、升单、留存、消耗、转化、毛利、现金流、员工效率、库存周转、客户满意度。
- 懂操作闭环：不仅回答“是什么”，还要能告诉用户“为什么、怎么办、下一步怎么做”，必要时直接生成任务、表单、活动、提醒、自动化规则或执行草稿。
- 懂风险提示：在客户流失、库存断货、活动低转化、员工业绩异常、财务波动、排班空档、客诉风险等场景主动提醒。

一句话目标：**让门店用户用一句话获得洞察、操作和结果。**

## 2. 产品目标

### 2.1 总目标

建设一个覆盖 Ami_Core 全业务能力的美业运营智能体，让用户通过自然语言、卡片、表格、图表、文档和自动化任务完成经营查询、业务操作、风险处理和管理决策。

### 2.2 阶段目标

| 阶段 | 目标 | 交付重点 |
| --- | --- | --- |
| MVP | 建立全新 Agent 底座并完成可信问数 | 新 Agent Runtime、工具注册中心、权限网关、多轮问答、数据卡片、人工确认 |
| V1 | 可执行常用业务任务 | 业务工具编排、创建预约、客户跟进、营销活动草稿、库存预警任务、员工提醒、经营复盘 |
| V2 | 建立全新自动化执行引擎 | 根据用户目标生成可执行自动化规则，支持触发器、审批、运行、失败重试、日志和效果归因 |
| V3 | 主动运营智能体 | 主动发现风险和机会，按角色推送每日经营待办、复盘和建议 |

### 2.3 非目标

| 非目标 | 说明 |
| --- | --- |
| 不做泛聊天助手 | 问答必须围绕美业经营、门店管理和 Ami_Core 业务能力 |
| 不让模型绕过系统权限 | 模型不能直接读库、写库或调用未授权接口 |
| 不替代所有页面 | 高频任务由智能体承接，复杂配置、审计、批量治理仍可保留专门后台 |
| 不复用 Ami_Aura 与 web app UI | Ami_Agent 需要独立产品语言、信息架构和视觉系统 |
| 不复用现有 AI Gateway | 现有 AI Gateway 只可作为失败经验参考，不能作为 Ami_Agent 的模型编排、工具调用、权限治理或上下文管理底座 |
| 不复用现有自动化任务系统 | 现有自动化任务系统只可作为反例参考，Ami_Agent 需新建可执行、可审计、可恢复的自动化执行引擎 |

### 2.4 失败资产边界

Ami_Agent 必须明确避开当前已失败的 AI 与自动化资产，避免在旧系统上继续叠功能。

| 资产 | 结论 | 对 Ami_Agent 的要求 |
| --- | --- | --- |
| 现有 AI Gateway | 不复用 | 新建 Agent Runtime，负责意图理解、上下文、工具编排、结构化输出、审批中断、审计和评估 |
| 现有自动化任务系统 | 不复用 | 新建自动化执行引擎，支持触发器、条件、动作、审批、幂等、运行日志、失败重试和效果归因 |
| 现有 Ami_Aura | 不复用产品形态 | 不能沿用终端式工作台、微应用入口、设备化交互和视觉风格 |
| 现有 web app | 不复用产品形态 | 不能沿用传统后台菜单、表格页面和管理端布局 |

新系统可以复用 Ami_Core 中经过验证的业务 API、权限数据、业务数据和领域模型，但必须通过新的 Agent 工具层重新封装，不能让旧 AI Gateway 或旧自动化任务系统继续承担核心链路。

## 3. 用户角色与核心场景

| 角色 | 典型诉求 | Ami_Agent 应提供的价值 |
| --- | --- | --- |
| 老板 / 多店管理者 | 看整体经营、利润、异常、门店对比 | 经营总览、风险预警、跨店对比、行动建议、管理任务派发 |
| 店长 | 管客户、员工、预约、消耗、活动 | 今日重点、客户跟进、排班调整、活动执行、员工辅导 |
| 前台 | 快速查客户、预约、收银、办卡、核销 | 一句话查客、快速建单、业务入口直达、异常提醒 |
| 美容师 | 看本人客户、服务计划、业绩、提成 | 客户护理建议、今日服务提醒、业绩进度、复购机会 |
| 财务 | 查收入、退款、成本、利润、异常流水 | 财务问数、异常解释、口径说明、复盘表格 |
| 库存 / 采购 | 看库存、临期、缺货、消耗、补货 | 库存预警、补货建议、消耗趋势、供应链动作 |
| 营销运营 | 做活动、客群、触达、转化分析 | 客群识别、活动草稿、自动触达、复盘优化 |
| 系统管理员 | 配角色、权限、风格、工具、审计 | Agent 配置、权限授权、工具治理、日志与风险控制 |

### 3.1 美业角色 Agent 规划

Ami_Agent 不应以一个万能助手覆盖全部场景，而应按美业岗位和经营任务拆成角色型 Agent。用户感知上可以是一个统一入口，但底层需要有清晰的角色能力边界、工具授权和任务目标。

| 角色 Agent | 目标用户 | 核心价值 | 阶段 |
| --- | --- | --- | --- |
| 店长经营 Agent | 店长 | 今日经营重点、客户风险、预约重点、员工业绩、库存提醒、门店复盘 | MVP |
| 营销增长 Agent | 店长 / 运营 | 客群筛选、活动草稿、权益推荐、召回话术、活动复盘 | MVP |
| 前台接待 Agent | 前台 | 查客户、查预约、建跟进、解释权益、收银/核销跳转、异常提醒 | MVP |
| 美容师服务 Agent | 美容师 | 今日客户、护理建议、复购机会、疗程消耗、个人业绩 | V1 |
| 库存采购 Agent | 店长 / 库存 | 缺货、临期、消耗趋势、补货建议、供应链动作 | V1 |
| 财务风控 Agent | 财务 / 老板 | 收入、退款、成本、利润、异常流水、经营利润解释 | V1 |
| 顾客服务 Agent | 前台 / 客服 / 顾客侧 | 项目咨询、预约改期、权益解释、售后收集、人工升级 | V1 |
| 老板多店 Agent | 老板 / 多店管理者 | 多店对比、利润异常、门店排名、管理动作、跨店复盘 | V2 |

### 3.2 MVP 角色收敛

MVP 只建议优先交付三个角色 Agent：

1. 店长经营 Agent：覆盖门店日常经营主入口。
2. 营销增长 Agent：最容易证明业务增收价值。
3. 前台接待 Agent：高频、省时间、容易形成使用习惯。

暂缓进入 MVP 的能力：

- 全自动批量触达。
- 全自动修改价格、权限、财务规则。
- 复杂多 Agent 自主协作。
- 顾客侧完全无人客服。
- 传统 RPA 页面点击式执行。

## 4. 用户需求与产品原则

### 4.1 真正懂美业

Ami_Agent 必须能把用户输入识别成美业业务语义，而不是仅做关键词匹配。

| 用户输入示例 | 系统理解 | 输出要求 |
| --- | --- | --- |
| “最近哪些顾客可能要流失？” | 客户活跃度、消费间隔、预约缺失、卡项剩余、历史频次 | 流失风险客户表、原因、跟进建议、一键创建跟进任务 |
| “这个月项目卖得怎么样？” | 项目销售、消耗、复购、客单价、门店 / 员工维度 | 趋势图、Top 项目、异常说明、改进建议 |
| “帮我看看库存有没有风险” | 库存数量、安全线、消耗速度、临期、缺货 | 风险卡片、补货表、处理优先级 |
| “给 60 天没来的顾客做个召回” | 客群筛选、权益匹配、营销内容、触达规则 | 活动草稿、客群明细、预计成本、需确认后执行 |

### 4.2 基于事实回答

所有涉及门店经营事实的回答必须来自 Ami_Core 的真实业务数据或联网搜索结果，并清楚区分：

- 系统数据：来自 Ami_Core 的客户、订单、预约、库存、财务等数据。
- 推理建议：Ami_Agent 基于数据给出的分析和建议。
- 外部资料：来自联网搜索或用户上传文档的参考信息。
- 需要确认：需要用户授权、审批或补充信息后才能执行的动作。

### 4.3 先追问，再执行

当用户需求不明确时，Ami_Agent 不应假装理解，而应进行 1 到 3 个关键追问。

需要追问的典型情况：

- 时间范围不清，例如“最近”“这段时间”。
- 门店范围不清，例如多门店用户未指定门店。
- 操作目标不清，例如“做个活动”未说明拉新、复购、消耗或清库存。
- 风险动作缺少确认，例如批量发券、批量触达、修改价格、删除规则。
- 数据口径存在多种解释，例如“收入”可能是应收、实收、储值消耗或营业额。

### 4.4 回复后主动延展

每次回复完成后，如果存在高价值关联问题，Ami_Agent 可提供 1 到 3 个后续问题供用户选择。

示例：

| 当前回答 | 推荐追问 |
| --- | --- |
| 本月客户流失风险分析 | “要不要生成高风险客户跟进名单？”“要不要按美容师拆分责任客户？”“要不要创建 7 天召回任务？” |
| 库存风险分析 | “要不要生成补货建议单？”“要不要查看近 30 天消耗趋势？”“要不要设置自动库存预警？” |
| 活动复盘 | “要不要找出未转化客户？”“要不要优化下一轮活动话术？”“要不要生成店长复盘文档？” |

### 4.5 主流 Agent 借鉴原则

结合 OpenAI Agents SDK、Salesforce Agentforce、Microsoft Copilot Studio、Adobe Agent Orchestrator、Shopify Sidekick、HubSpot Breeze、ServiceNow AI Agents、UiPath Agentic Automation、Zendesk AI Agents 等成熟产品方向，Ami_Agent 采纳以下原则：

| 借鉴方向 | Ami_Agent 产品要求 |
| --- | --- |
| 角色型 Agent | 不做一个万能助手，而是按美业岗位和经营场景拆分店长、营销、前台、美容师、库存、财务、顾客服务等 Agent |
| 工具化业务能力 | 所有查询、分析和操作必须通过 Ami_Core 工具层执行，工具具备 schema、权限、审计、风险等级和幂等策略 |
| 事件触发自动化 | 自动化从固定规则升级为“触发器 + 条件 + 动作 + 审批 + 日志 + 效果归因” |
| 人工确认与审批 | 查询可直接执行，写操作、批量触达、财务、权限和删除类动作必须确认或审批 |
| 客户旅程编排 | 以新客、体验、办卡、消耗、复购、沉睡、召回、转介绍为美业运营主线 |
| 任务画布 | 回复结果必须能承接业务操作，输出卡片、表格、图表、文档、审批和任务，而不是只有聊天文本 |
| 权限继承 | Agent 不是超级管理员，必须继承当前用户的角色、门店、字段和动作权限 |
| 观测评估 | 从 MVP 开始记录意图识别、工具调用、用户采纳、业务动作和效果归因 |

### 4.6 不照搬的方向

| 不照搬对象 | 原因 | Ami_Agent 处理方式 |
| --- | --- | --- |
| 通用聊天机器人 | 不懂美业经营，难以稳定执行业务 | 建立美业语义层和角色 Agent |
| 大企业 CRM 重流程 | 中小美业门店使用成本高 | 保留轻量、直接、行动导向的交互 |
| 传统 RPA 页面点击 | 稳定性差，难审计 | 优先通过业务 API 和工具层执行 |
| 纯客服机器人 | 只能处理咨询，不能经营门店 | 顾客服务只是其中一个 Agent，不是 Ami_Agent 全部 |
| 第三方 Zap 式拼接 | 核心数据和执行不可控 | 核心门店数据、权限和执行引擎自有可控 |

## 5. 核心功能需求

## 5.1 全新产品底座

Ami_Agent 的底座必须新建，不能沿用现有 AI Gateway 和现有自动化任务系统。新底座不是单一模型接口，而是一套面向美业经营任务的 Agent 产品运行系统。

### 5.1.1 新底座组成

| 模块 | 产品职责 | 必须解决的问题 |
| --- | --- | --- |
| Agent Runtime | 理解用户目标、管理多轮上下文、规划步骤、调用工具、生成结构化回复 | 不能只是转发模型请求，必须能稳定完成多步骤经营任务 |
| 美业语义层 | 统一业务对象、指标口径、同义词、风险规则、操作动作 | 解决模型不懂美业、关键词误判、口径混乱的问题 |
| 工具注册中心 | 将 Ami_Core 能力封装为可授权、可审计、可执行工具 | 解决模型直接访问页面或接口导致不可控的问题 |
| 权限与审批网关 | 在每次工具调用前校验角色、门店、字段、动作和风险等级 | 解决越权查询、越权执行和批量误操作问题 |
| 回复渲染引擎 | 将结果输出为文字、卡片、表格、图表、文档、链接或图片 | 解决回复只有文本、无法承接业务操作的问题 |
| 记忆与归档服务 | 管理会话上下文、用户偏好、长期记忆和按天聊天归档 | 解决对话断裂、偏好无法沉淀、历史不可追踪的问题 |
| 联网搜索服务 | 按权限执行外部搜索并记录来源 | 解决外部信息不可追溯、内外部事实混淆的问题 |
| 自动化执行引擎 | 运行触发器、条件、动作、审批、重试、日志和效果归因 | 解决旧自动化不可执行、不可恢复、不可复盘的问题 |
| 观测与评估系统 | 记录意图识别、工具调用、回答质量、失败原因和用户反馈 | 解决 Agent 效果无法评估、无法持续优化的问题 |

### 5.1.2 新 Agent Runtime 要求

Agent Runtime 需要承担“经营任务大脑”的职责，而不是简单聊天。

| 能力 | 要求 |
| --- | --- |
| 目标识别 | 将用户自然语言识别为查询、分析、操作、自动化、复盘、配置等目标 |
| 任务规划 | 将复杂需求拆成可执行步骤，并在执行前识别风险 |
| 上下文管理 | 管理用户、门店、时间范围、业务对象、历史追问和未完成任务 |
| 工具选择 | 从工具注册中心选择合适工具，不能自由拼接口或绕过权限 |
| 结构化输出 | 输出可渲染结果，而不是只返回自然语言 |
| 中断与确认 | 高风险动作必须暂停，等待用户确认、修改或审批 |
| 失败恢复 | 工具失败、权限不足、数据缺失时给出可操作的修复路径 |
| 质量评估 | 每次回答都能记录是否命中意图、是否被采纳、是否产生业务动作 |

### 5.1.3 新工具层要求

工具层是 Ami_Agent 调度 Ami_Core 的唯一入口。Ami_Core 的业务 API、权限数据、业务数据和领域模型可以被复用，但必须重新封装为 Agent 工具。

工具层需要支持：

- 工具发现：Agent 能知道有哪些业务能力可用。
- 工具说明：每个工具有清晰业务描述、输入、输出和限制。
- 工具授权：不同角色只能调用被授权的工具。
- 工具证据：工具返回的数据必须包含来源、口径、时间范围和字段解释。
- 工具状态：返回成功、失败、部分成功、需确认、需审批等状态。
- 工具幂等：同一个确认动作不能重复创建订单、活动、触达或任务。
- 工具审计：每次调用都进入审计日志。

### 5.1.4 不可接受的底座形态

以下形态不符合 Ami_Agent 要求：

| 形态 | 问题 |
| --- | --- |
| 只把用户问题转发给大模型 | 无权限、无工具、无证据、无执行闭环 |
| 让模型直接拼 SQL 或直接调用业务接口 | 高风险、不可控、难审计 |
| 让 Agent 模拟点击现有后台页面 | 稳定性差，无法形成可治理工具能力 |
| 在旧 AI Gateway 上增加 prompt | 无法解决上下文、工具编排、权限、确认、评估问题 |
| 在旧自动化系统上增加规则 | 无法解决可执行、可恢复、可追踪、可归因问题 |

---

## 5.2 多轮问答与需求识别

### 5.2.1 需求识别

Ami_Agent 需要识别以下信息：

| 识别对象 | 示例 |
| --- | --- |
| 业务领域 | 客户、预约、订单、卡项、项目、商品、库存、财务、员工、营销 |
| 用户意图 | 查询、分析、解释、创建、修改、提醒、自动化、导出、复盘 |
| 时间范围 | 今天、本周、本月、近 30 天、指定日期 |
| 数据范围 | 单店、多店、员工、客户分群、项目类别 |
| 输出形式 | 文字、卡片、表格、图表、图片、文档、链接 |
| 风险等级 | 低风险查询、中风险草稿、高风险写操作 |
| 是否需要确认 | 查询无需确认，写操作和批量触达必须确认 |

### 5.2.2 多轮上下文

系统需支持连续对话中的上下文继承：

- 继承上一次的门店、时间范围、业务对象和筛选条件。
- 支持用户用“那这些人呢”“换成上个月”“按美容师拆一下”等省略表达继续追问。
- 支持用户随时修改条件，并重新生成结果。
- 对高风险动作，必须在执行前重新展示执行摘要并确认。

### 5.2.3 澄清机制

当用户意图不明确时，输出简短追问，不进入冗长解释。

示例：

```text
你想做的是客户召回活动。还需要确认 2 个信息：
1. 召回范围：60 天未到店，还是 90 天未到店？
2. 优惠方式：护理券、现金券，还是指定项目折扣？
```

## 5.3 多模态回复与业务组件

Ami_Agent 的回复不局限于文字，应根据用户任务自动选择最合适的表达形式。

| 输出形式 | 适用场景 | 产品要求 |
| --- | --- | --- |
| 文字 | 解释、建议、追问、总结 | 简洁、分点、明确下一步 |
| 卡片 | 风险、机会、任务、客户、活动、库存 | 可点击、可确认、可跳转、可执行 |
| 表格 | 客户名单、项目排行、库存清单、财务明细 | 支持筛选、排序、导出、字段解释 |
| 图表 | 趋势、对比、占比、转化漏斗 | 支持时间切换、指标解释、数据来源 |
| 图片 | 海报、活动素材、护理建议图 | 明确生成来源，可编辑，可下载 |
| 文档 | 复盘报告、经营分析、店长日报、活动方案 | 支持生成、预览、下载、分享 |
| 链接 | 跳转到业务详情、表单、审批、外部资料 | 链接必须带权限校验和上下文 |

### 5.3.1 回复结构

标准回复建议包含：

1. 结论：先给用户最重要的判断。
2. 证据：列出关键数据、时间范围和来源。
3. 建议：给出可执行步骤。
4. 动作：提供按钮、卡片或链接。
5. 延展：提供 1 到 3 个高价值追问。

### 5.3.2 业务卡片类型

| 卡片 | 核心字段 | 操作 |
| --- | --- | --- |
| 客户风险卡 | 客户名、风险原因、最近到店、卡项余额、建议动作 | 创建跟进、发券、查看客户 |
| 经营指标卡 | 指标值、环比、同比、异常原因 | 查看趋势、导出报表、生成复盘 |
| 库存预警卡 | 商品、当前库存、安全线、预计可用天数 | 生成补货单、设置预警 |
| 营销活动卡 | 客群、权益、预算、预计触达、预计转化 | 编辑草稿、提交审批、执行 |
| 自动化任务卡 | 触发条件、执行动作、运行频率、权限状态 | 启用、暂停、查看日志 |
| 审批确认卡 | 执行摘要、影响范围、风险提示 | 确认执行、修改、取消 |

## 5.4 Ami_Core 全能力调度

### 5.4.1 能力开放原则

整个 Ami_Core 的权限、业务、功能和数据都应向 Ami_Agent 开放调度，但开放方式不是让模型直接访问页面或数据库，而是将 Ami_Core 能力封装为可授权、可审计、可执行的工具。

开放范围包括：

- 客户与会员：客户查询、分群、画像、标签、跟进、会员卡、权益、护理建议。
- 预约与服务：预约查询、创建、改期、取消、服务记录、消耗、核销。
- 收银与订单：订单查询、收银草稿、退款申请、支付记录、异常订单。
- 项目与商品：项目分析、商品销售、价格、套餐、卡项、权益资产。
- 库存与采购：库存查询、消耗、补货、临期、调拨、供应商。
- 员工与排班：排班、业绩、提成、服务效率、任务分配。
- 财务与利润：收入、退款、成本、毛利、经营利润、现金流、报表。
- 营销与自动化：客群、活动、优惠券、触达、转化、复盘、规则。
- 系统配置：门店、角色、权限、审批、审计、Agent 配置。

### 5.4.2 工具化要求

每个可调度能力都需具备：

| 字段 | 要求 |
| --- | --- |
| 工具名称 | 稳定、可读、唯一 |
| 业务描述 | 明确该工具能做什么，不能做什么 |
| 输入参数 | 使用结构化 schema，避免自由文本直传 |
| 输出结构 | 返回可渲染组件、证据和状态 |
| 权限要求 | 标明角色、数据范围、字段权限 |
| 风险等级 | 查询、草稿、写操作、批量写操作、财务敏感 |
| 确认策略 | 是否需要用户确认、审批或二次验证 |
| 审计字段 | 操作人、门店、时间、输入、输出、执行结果 |
| 幂等策略 | 防止重复创建活动、任务、订单或触达 |

### 5.4.3 执行边界

| 动作类型 | 是否可直接执行 | 要求 |
| --- | --- | --- |
| 查询类 | 可直接执行 | 权限通过后返回结果 |
| 分析类 | 可直接执行 | 必须展示数据来源和口径 |
| 草稿类 | 可直接创建 | 草稿需用户确认后生效 |
| 单条低风险写操作 | 可确认后执行 | 展示执行摘要 |
| 批量触达 / 批量修改 | 必须审批 | 展示影响范围、成本和风险 |
| 财务 / 权限 / 删除类 | 强审批 | 需要高权限、二次确认、完整审计 |

## 5.5 自动化任务

### 5.5.1 自动化能力定位

Ami_Agent 支持按用户自然语言创建可执行自动化任务。自动化不是“提醒文案”，也不是旧自动化系统上的规则拼装，而是由全新自动化执行引擎调度、运行、监控和复盘的业务流程。

示例：

| 用户需求 | 自动化任务 |
| --- | --- |
| “每天早上提醒我今天有哪些高价值客户到店” | 每日 8:30 查询今日预约客户，筛选高价值客户，推送店长卡片 |
| “库存低于 5 件就提醒采购” | 库存低于安全线触发补货提醒，生成补货建议单 |
| “60 天没来的客户自动进入召回名单” | 每天扫描客户到店间隔，加入召回客群，生成跟进任务 |
| “每周给我一份经营复盘” | 每周一生成上周经营报告，包含收入、客流、项目、员工、风险 |

### 5.5.2 自动化创建流程

```text
用户提出目标
  -> Ami_Agent 识别触发条件、执行动作、对象范围、频率、权限
  -> 必要时追问补齐信息
  -> 生成自动化任务草稿
  -> 展示执行摘要、风险、预计影响范围
  -> 用户确认或提交审批
  -> 全新自动化执行引擎创建任务
  -> 运行、记录日志、失败重试、异常通知
```

### 5.5.3 自动化任务结构

| 字段 | 说明 |
| --- | --- |
| 任务名称 | 用户可读，例如“高价值客户到店提醒” |
| 任务目标 | 提升转化、降低流失、减少缺货、复盘经营 |
| 触发条件 | 定时、事件、阈值、数据变化、人工触发 |
| 执行动作 | 查询、生成名单、推送、创建任务、生成草稿、提交审批 |
| 执行对象 | 门店、客户群、员工、商品、项目、订单 |
| 执行频率 | 一次性、每日、每周、每月、实时事件 |
| 权限授权 | 创建人、可查看人、可暂停人、可审批人 |
| 风险等级 | 低、中、高 |
| 审批策略 | 无需审批、创建时审批、每次执行审批 |
| 运行日志 | 成功、失败、跳过、异常原因 |

### 5.5.4 全新自动化执行引擎要求

自动化执行引擎必须独立新建，并满足以下产品要求：

| 能力 | 要求 |
| --- | --- |
| 触发器 | 支持定时、事件、阈值、数据变化和人工触发 |
| 条件判断 | 支持客户、库存、订单、预约、员工、财务、营销等条件组合 |
| 动作编排 | 支持查询、筛选、生成、推送、创建草稿、提交审批、执行低风险动作 |
| 审批中断 | 高风险动作运行到审批点时暂停，等待人工确认后继续 |
| 幂等控制 | 防止同一客户重复触达、同一活动重复创建、同一提醒重复发送 |
| 失败恢复 | 支持失败原因展示、重试、跳过、人工接管 |
| 运行日志 | 每次运行记录输入、命中对象、执行动作、输出和异常 |
| 效果归因 | 统计任务带来的跟进、预约、核销、订单、收入或风险下降 |
| 权限继承 | 自动化任务必须继承创建人和被授权角色的权限边界 |
| 暂停熔断 | 当数据异常、触达成本异常或失败率异常时自动暂停并提醒 |

### 5.5.5 自动化任务管理

必须提供以下能力：

- 查看任务列表、运行状态、最近运行结果。
- 启用、暂停、编辑、复制、删除任务。
- 查看每次运行的输入、输出、执行动作和失败原因。
- 支持失败重试和人工接管。
- 支持自动化任务效果复盘，例如召回人数、转化订单、带来收入。
- 支持查看任务由哪一次对话创建、由谁确认、运行时调用了哪些工具。
- 支持按风险等级筛选自动化任务，快速找到可能影响客户、财务或权限的任务。

### 5.5.6 美业内置触发器库

自动化不要从复杂自定义规则起步，应先提供美业高价值内置触发器。用户可以用自然语言启用、暂停、调整阈值或生成对应任务。

| 触发器 | 默认条件示例 | Agent 动作 | 风险等级 |
| --- | --- | --- | --- |
| 沉睡客户触发器 | 客户 45 / 60 / 90 天未到店 | 加入沉睡客户池，生成跟进任务和召回话术 | 中 |
| 高价值客户到店触发器 | 高价值客户明日有预约 | 推送店长和美容师重点服务提醒 | 低 |
| 疗程消耗触发器 | 卡项 / 疗程剩余低于阈值 | 生成续卡、升单或护理建议 | 中 |
| 库存缺货触发器 | 热销商品预计 7 天内缺货 | 生成补货建议单和采购提醒 | 中 |
| 临期库存触发器 | 商品临期或周转异常 | 生成促销、消耗或调拨建议 | 中 |
| 活动低转化触发器 | 活动转化低于预期 | 提醒调整权益、话术或目标客群 | 中 |
| 员工异常触发器 | 美容师复购率、消耗率或业绩连续下降 | 生成员工辅导建议和客户回访名单 | 中 |
| 预约异常触发器 | 爽约率上升、排班空档、预约冲突 | 生成提醒、调班建议或客户确认任务 | 中 |
| 财务异常触发器 | 退款、折扣、收入、成本出现异常波动 | 生成异常解释和复核任务 | 高 |
| 投诉差评触发器 | 退款 / 投诉 / 差评出现 | 创建高优先级风险处理任务并提醒负责人 | 高 |

内置触发器必须满足：

- 可解释：用户能看到触发原因和命中数据。
- 可调整：阈值、频率、对象范围可配置。
- 可暂停：异常或误触发时可一键暂停。
- 可归因：记录任务是否带来跟进、预约、订单、核销、收入或风险下降。
- 可审批：涉及批量触达、财务、权限或高风险客户动作时必须审批。

## 5.6 权限、授权与审计

### 5.6.1 权限原则

Ami_Agent 在产品能力上连接整个 Ami_Core，但执行时必须遵循“用户身份 + 角色权限 + 授权范围 + 风险等级 + 审计留痕”的组合策略。

核心原则：

- Ami_Agent 可以看见 Ami_Core 全能力目录，但不能替用户越权执行。
- 不同角色看到的回答、字段、按钮和可执行动作不同。
- 高风险动作必须确认或审批。
- 所有工具调用、查询、写操作、自动化运行都要留痕。

### 5.6.2 授权模型

| 授权对象 | 示例 |
| --- | --- |
| 角色授权 | 店长可查本店经营数据，不能查其他门店财务明细 |
| 工具授权 | 某角色可使用“创建营销活动草稿”，不可直接“发送营销短信” |
| 数据授权 | 只能访问本门店、本员工、本客户分组 |
| 字段授权 | 手机号、利润、提成、成本等敏感字段需脱敏或额外权限 |
| 动作授权 | 查询、创建草稿、提交审批、直接执行、删除 |
| 自动化授权 | 谁可创建、启用、暂停、审批和查看日志 |

### 5.6.3 审计要求

审计日志至少包含：

- 操作人、角色、门店、设备、时间。
- 用户原始输入和系统识别出的意图。
- 调用的工具、参数、返回摘要。
- 输出给用户的结果。
- 用户确认记录。
- 实际执行结果。
- 失败原因、重试记录和人工接管记录。

## 5.7 可配置内容

系统管理员和授权用户可配置 Ami_Agent 的角色、风格偏好和授权权限。

### 5.7.1 角色配置

| 配置项 | 说明 |
| --- | --- |
| Agent 名称 | 例如“门店运营顾问”“财务分析助手”“营销增长助手” |
| 业务职责 | 该 Agent 主要负责的业务范围 |
| 可用工具 | 可调用的 Ami_Core 工具集合 |
| 数据范围 | 可访问的门店、角色、字段和对象 |
| 风险策略 | 可直接执行、需确认、需审批的动作范围 |
| 推送策略 | 是否主动提醒、提醒频率、提醒渠道 |

### 5.7.2 风格偏好

| 偏好 | 可选项 |
| --- | --- |
| 回复长度 | 简洁、标准、详细 |
| 表达风格 | 专业、直接、温和、经营顾问式 |
| 数据展示 | 先结论、先表格、先图表 |
| 建议强度 | 保守、中性、积极 |
| 追问策略 | 少追问、标准追问、严格补齐信息 |

### 5.7.3 个人偏好

Ami_Agent 应记住用户个人偏好，例如：

- 常看的门店、时间范围、指标。
- 喜欢的报表格式和导出格式。
- 常用业务动作，例如每日看经营日报、每周看活动复盘。
- 习惯的称呼、语言风格和回复详略。

## 5.8 记忆能力与聊天归档

### 5.8.1 记忆类型

| 记忆类型 | 内容 | 生命周期 |
| --- | --- | --- |
| 会话记忆 | 当前对话中的上下文、筛选条件、未完成任务 | 当前会话 |
| 用户偏好 | 回复风格、常用门店、常看指标、常用导出格式 | 长期 |
| 业务偏好 | 门店经营目标、重点项目、重点客群、活动偏好 | 长期，可编辑 |
| 自动化记忆 | 用户创建过的自动化任务和效果 | 长期 |

### 5.8.2 聊天记录按天存档

聊天记录需按自然日归档，支持：

- 按日期浏览历史对话。
- 按关键词、业务对象、客户、活动、任务搜索。
- 继续某一天的历史话题。
- 将一次对话生成文档或任务。
- 删除、隐藏或导出指定对话。

### 5.8.3 记忆可控

用户必须能查看、编辑和删除 Ami_Agent 记住的偏好。

示例：

| 操作 | 产品表现 |
| --- | --- |
| 查看记忆 | “你记住了我常看本月实收、客户复购和库存预警” |
| 删除记忆 | 用户可删除某条偏好 |
| 临时对话 | 本次对话不进入长期记忆 |
| 改写偏好 | “以后回答财务问题先给表格，再给分析” |

## 5.9 联网搜索

### 5.9.1 使用场景

Ami_Agent 支持联网搜索，但必须明确外部资料与门店内部数据的边界。

适用场景：

- 查询美业行业趋势、热门项目、营销玩法。
- 获取节日营销节点、竞品信息、平台规则。
- 参考公开护理知识、产品知识、行业报告。
- 生成外部资料引用的活动方案或经营建议。

### 5.9.2 搜索要求

| 要求 | 说明 |
| --- | --- |
| 显示来源 | 外部资料必须展示来源链接 |
| 区分事实 | 不得把外部行业观点当成本店真实数据 |
| 时间敏感 | 涉及最新政策、平台规则、价格、市场趋势时需标注检索日期 |
| 可关闭 | 管理员可关闭联网搜索或限制搜索范围 |
| 可审计 | 搜索关键词、来源、摘要需进入审计日志 |

## 6. 典型业务流程

### 6.1 经营问数

```text
用户：这个月经营情况怎么样？
Ami_Agent：
1. 识别门店、月份、经营指标。
2. 查询收入、客流、客单、复购、项目、员工、库存和异常。
3. 生成经营总览卡、趋势图、风险卡和建议。
4. 提供追问：看利润、看员工、生成月报。
```

### 6.2 客户召回

```text
用户：帮我找一下快流失的客户，做个召回。
Ami_Agent：
1. 追问或默认使用流失规则。
2. 查询高风险客户。
3. 输出客户表、流失原因、推荐权益。
4. 生成召回活动草稿。
5. 用户确认后提交审批或执行。
```

### 6.3 自动化创建

```text
用户：以后每天早上提醒我今天要重点服务的客户。
Ami_Agent：
1. 识别为每日自动化任务。
2. 追问提醒时间和重点客户规则。
3. 生成任务草稿。
4. 展示运行范围和推送方式。
5. 用户确认后创建任务。
```

### 6.4 运营风险提示

```text
系统主动发现：
- 某项目本月销量下降 35%。
- 某美容师复购率连续两周低于门店均值。
- 某热销商品预计 4 天后缺货。

Ami_Agent 输出：
- 风险说明。
- 数据证据。
- 影响范围。
- 建议动作。
- 一键创建跟进、培训、补货或活动任务。
```

## 7. UI 与交互需求

### 7.1 视觉定位

Ami_Agent 的 UI 风格为：**极简、科技、艺术**。

设计关键词：

- 极简：信息密度高但不拥挤，少装饰，突出任务与结果。
- 科技：清晰的状态、流动的计算感、可信的数据表达。
- 艺术：细腻的排版、留白、光影层次和高级材质感。

禁止项：

- 不复用 Ami_Aura 的终端式工作台、微应用入口和店内设备感。
- 不复用 web app 的传统后台菜单、表格页面和管理端布局。
- 不做营销官网式大横幅。
- 不用大量拟物卡片堆叠。
- 不把聊天窗口做成唯一中心，任务结果和可执行组件同等重要。

### 7.2 信息架构

建议采用“智能会话 + 任务画布 + 运营记忆 + 自动化中心”的新结构。

| 区域 | 作用 |
| --- | --- |
| 智能会话 | 用户输入、追问、多轮上下文、即时回复 |
| 任务画布 | 展示卡片、表格、图表、文档、审批、可执行动作 |
| 运营记忆 | 用户偏好、常用指标、历史结论、长期目标 |
| 自动化中心 | 自动化任务、运行日志、异常处理、效果复盘 |
| 权限与配置 | Agent 角色、工具授权、风格偏好、审计策略 |

### 7.3 核心交互

- 用户可以直接输入自然语言，也可以从任务模板开始。
- 回答中的业务对象可点击展开，例如客户、订单、库存、员工、活动。
- 每个可执行动作必须有明确状态：草稿、待确认、执行中、成功、失败、需审批。
- 长结果默认摘要展示，表格和文档可展开。
- 图表必须能解释口径和来源。
- 自动化任务必须能从对话中生成，也能在自动化中心管理。

## 8. 数据与知识要求

### 8.1 业务语义层

Ami_Agent 需要建立美业经营语义层，统一指标口径。

| 语义对象 | 示例指标 |
| --- | --- |
| 客户 | 新客、老客、沉睡、流失风险、复购、客单价、生命周期价值 |
| 预约 | 到店率、爽约率、预约来源、排班占用 |
| 项目 | 销量、消耗、复购、毛利、转化、护理周期 |
| 商品 | 销售、库存、周转、临期、缺货 |
| 卡项 | 开卡、充值、消耗、剩余、到期、沉淀 |
| 员工 | 业绩、服务数、复购、客诉、提成、工时 |
| 营销 | 触达、打开、领取、核销、转化、ROI |
| 财务 | 应收、实收、退款、成本、毛利、利润 |

### 8.2 客户生命周期模型

美业 Agent 的核心资产不是通用问答，而是客户生命周期模型。Ami_Agent 需要围绕客户从新客到复购、沉睡、召回、转介绍的完整旅程识别机会和风险。

```text
新客到店
  -> 首次体验
  -> 办卡 / 开疗程
  -> 疗程消耗
  -> 复购升单
  -> 沉睡预警
  -> 召回
  -> 转介绍
```

| 阶段 | 关键判断 | Agent 动作 |
| --- | --- | --- |
| 新客到店 | 来源、需求、首次项目、消费能力 | 生成接待提醒、项目推荐、首次体验话术 |
| 首次体验 | 是否满意、是否有复购机会、是否适合办卡 | 生成转化建议、跟进任务、权益推荐 |
| 办卡 / 开疗程 | 卡项类型、金额、周期、消耗节奏 | 生成服务计划和消耗提醒 |
| 疗程消耗 | 消耗进度、到店频率、剩余次数 | 提醒续卡、升单、护理计划调整 |
| 复购升单 | 复购周期、项目偏好、客单价变化 | 推荐项目组合、会员权益、专属活动 |
| 沉睡预警 | 未到店天数、卡项剩余、历史频次 | 识别流失原因，生成召回名单 |
| 召回 | 触达渠道、权益成本、回复情况 | 生成召回话术、跟进任务和转化复盘 |
| 转介绍 | 高满意客户、高价值客户、社交传播 | 生成转介绍权益和邀请话术 |

客户生命周期模型必须输出：

- 客户当前阶段。
- 阶段判断证据。
- 推荐经营动作。
- 可执行任务。
- 预计影响和风险。
- 后续追踪指标。

### 8.3 知识库

需要建设美业知识库，包含：

- 美业业务术语和同义词。
- 门店经营方法论。
- 客户生命周期和护理周期知识。
- 项目、商品、卡项、权益资产说明。
- 营销活动模板和话术模板。
- 常见经营风险规则。
- 系统操作说明和权限说明。

### 8.4 内容与素材生成知识

Ami_Agent 可生成活动文案、私域话术、顾客召回话术、美容师服务提醒、项目介绍页、活动复盘报告和海报素材说明。但内容生成不能脱离业务目标，必须绑定：

- 目标客群。
- 活动目的。
- 项目 / 商品 / 卡项。
- 权益成本。
- 门店风格。
- 合规边界。
- 预期转化指标。

禁止生成：

- 医疗效果承诺。
- 夸大美容效果。
- 未经确认的价格、优惠或疗效。
- 涉及隐私的客户公开内容。

## 9. 安全、可靠性与合规

| 主题 | 要求 |
| --- | --- |
| 数据安全 | 敏感字段脱敏，按角色授权展示 |

| 降级策略 | AI 不可用时，核心查询和自动化日志仍可查看 |
| 旧系统隔离 | 不依赖现有 AI Gateway 和现有自动化任务系统承接核心链路 |

## 13. 关键风险与建议

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 模型误解业务 | 造成错误建议或误操作 | 建立美业语义层、工具 schema、追问机制 |
| 数据口径不统一 | 回答不可信 | 统一经营指标口径和来源说明 |
| 权限过宽 | 敏感数据泄露或越权操作 | 权限前置、字段脱敏、审计留痕 |
| 自动化误执行 | 批量触达、错误提醒、重复任务 | 审批、幂等、运行日志、暂停机制 |
| 沿用失败旧底座 | 新产品继续继承旧 AI 和旧自动化问题 | 立项阶段明确旧 AI Gateway 与旧自动化任务系统不复用，新建 Runtime 和执行引擎 |
| UI 退化成聊天框 | 难以承接复杂业务结果 | 引入任务画布、业务卡片、表格、图表和自动化中心 |
| 与现有产品混淆 | 用户无法理解新产品价值 | 保持独立定位、独立视觉、独立信息架构 |

## 14. 产品结论

Ami_Agent 应作为 Ami_Core 之上的新一代美业门店运营智能体，而不是现有终端或后台的延伸页面。它的核心价值是把 Ami_Core 的全业务能力转化为可被自然语言调度、可被权限治理、可被审计追踪、可持续自动化执行的智能运营入口。

对用户来说，Ami_Agent 不是“我问你答”，而是：

- 看得懂门店经营。
- 查得到真实数据。
- 说得清风险和机会。
- 给得出可执行步骤。
- 能生成任务和自动化。
- 能记住个人偏好。
- 能持续陪伴门店做运营。

最终交付形态应是一套独立、极简、科技、艺术的智能运营工作台，让美业门店从“使用软件管理业务”升级为“与智能体共同经营门店”。
