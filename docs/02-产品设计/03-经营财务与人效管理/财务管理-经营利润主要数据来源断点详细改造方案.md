# 财务管理-经营利润主要数据来源断点详细改造方案

版本：v1.0
日期：2026-06-23
关联需求：`docs/02-产品设计/美容院经营利润看板需求文档.md`
关联计划：`docs/03-开发计划/经营利润一级模块详细开发计划.md`
适用范围：Ami_Core 管理端财务管理-经营利润、server-v2、Ami Aura Lite 智能终端、订单/卡项/库存/提成链路

---

## 1. 背景

经营利润模块已经具备利润看板、商品毛利、项目毛利、会员卡履约、员工人效和成本配置的基础能力，但从真实业务链路看，仍有若干数据来源断点会影响利润可信度。

本方案重点解决：

1. 次卡销售、次卡核销、项目毛利之间的收入确认断点。
2. 储值充值、余额消费、会员卡履约之间的预收负债断点。
3. 商品订单、项目订单、BOM、库存流水、提成记录之间的成本归集断点。
4. 页面能看到结果，但不能追溯到订单/核销/耗材/提成明细的问题。

核心原则：

- 办卡、充值是现金流，不直接计入经营利润。
- 次卡核销是项目履约，应进入项目毛利和经营收入，但不能伪装成新的收款项目订单。
- 商品销售、项目单次服务、次卡核销、储值消费都要有可追溯事实源。
- 缺成本、缺提成、缺 BOM、缺核销单价时必须显式提示，不按 0 成本悄悄计算。

## 2. 当前主要断点总览

| 编号 | 断点 | 当前表现 | 利润影响 | 优先级 |
| --- | --- | --- | --- | --- |
| D1 | 次卡销售订单与 `CustomerCard` 未强关联 | 只能按客户、卡名、时间推断 | 预收现金流和剩余权益难追溯 | P0 |
| D2 | 次卡核销与具体客户卡、项目档案未强关联 | `CardUsageRecord` 只有卡名、项目名 | 同名卡/同名项目会错配 | P0 |
| D3 | 次卡核销单价没有快照 | 当前按 `Card.price / totalTimes` 估算 | 改价、赠送次数、折扣办卡会导致消课收入不准 | P0 |
| D4 | 次卡核销提成无法稳定进入项目毛利 | 项目毛利主要按 `OrderItem.orderItemId` 找提成 | 次卡项目毛利和员工贡献毛利偏高 | P0 |
| D5 | 次卡核销耗材虽有库存流水，但质量不稳定 | 依赖 `sourceType=card_usage` 和项目名匹配 | 历史/手工核销可能漏耗材 | P0 |
| D6 | 储值余额未纳入会员卡履约/预收负债 | 履约页主要看 `CustomerCard` | 充值现金余额和赠送余额负债看不全 | P1 |
| D7 | 商品历史成本快照不完整 | 老订单可能用当前商品成本回算 | 商品毛利历史口径可能偏差 | P1 |
| D8 | 项目毛利缺明细追溯 | 商品毛利有订单明细，项目毛利缺订单/核销明细 | 异常项目难定位原因 | P1 |
| D9 | 经营成本仍依赖手工录入 | 房租、工资、营销、折旧来自 `OperatingCost` | 净利率依赖录入完整度 | P1 |
| D10 | 营销成本、赠品成本未完全归集 | 优惠分摊到收入，活动成本未完整进入成本 | 活动期利润容易高估 | P2 |

## 3. 推荐目标口径

### 3.1 收入口径

| 业务动作 | 是否产生现金收入 | 是否产生经营收入 | 是否进入项目毛利 | 说明 |
| --- | --- | --- | --- | --- |
| 商品销售 | 是 | 是 | 否，进入商品毛利 | 按商品订单明细净收入确认 |
| 单次项目服务收款 | 是 | 是 | 是 | 按项目订单明细净收入确认 |
| 次卡销售/办卡 | 是 | 否 | 否 | 作为预收现金流，不直接算利润 |
| 次卡核销 | 否 | 是 | 是 | 按核销单价快照确认履约收入 |
| 储值充值 | 是 | 否 | 否 | 作为现金流和预收负债 |
| 储值支付项目/商品 | 否或部分现金 | 是 | 按项目/商品归属进入毛利 | 收入来自订单明细，支付方式只是结算方式 |
| 赠送余额/赠送项目 | 否 | 视履约确认 | 是，需标记赠送成本 | 不能把赠送当现金收入 |

### 3.2 次卡口径

不建议把次卡核销直接生成普通项目订单。推荐新增“项目履约收入来源”的统一概念：

```text
项目毛利收入 =
  项目订单明细净收入
  + 次卡核销履约收入

次卡核销履约收入 =
  CardUsageRecord.recognizedUnitValue * 核销次数
```

次卡核销应该具备独立事实源，不重复计入项目订单收款。

## 4. 数据模型改造方案

### 4.1 CustomerCard 增强

目标：让每张客户次卡能追溯到销售订单和成交口径。

建议新增字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `sourceOrderId` | Int? | 来源办卡订单 `ProductOrder.id` |
| `sourceOrderItemId` | Int? | 来源办卡订单明细 `OrderItem.id` |
| `paidAmount` | Decimal | 实际成交金额 |
| `discountAmount` | Decimal | 办卡优惠金额 |
| `giftTimes` | Int | 赠送次数 |
| `recognizedUnitValue` | Decimal | 默认核销单价快照 |
| `pricingSnapshot` | Json? | 卡项价格、项目配比、折扣、赠送权益快照 |

计算建议：

```text
recognizedUnitValue = paidAmount / 可核销总次数
```

如一张卡包含多个项目，需支持按项目配置单次价值；没有配置时才按总价/总次数估算。

### 4.2 CardUsageRecord 增强

目标：让每次核销可以进入项目毛利、耗材、提成和履约收入。

建议新增字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `customerCardId` | Int? | 被核销的客户次卡 |
| `cardId` | Int? | 卡项档案 |
| `projectId` | Int? | 核销项目档案 |
| `storeId` | Int? | 门店，避免只从客户反查 |
| `recognizedUnitValue` | Decimal | 本次核销单次确认收入 |
| `recognizedAmount` | Decimal | 本次核销确认收入 |
| `sourceOrderId` | Int? | 来源办卡订单 |
| `sourceOrderItemId` | Int? | 来源办卡订单明细 |
| `pricingSnapshot` | Json? | 核销时的卡项单价、项目、赠送、折扣快照 |

约束建议：

- 新增索引：`storeId + verifiedAt`、`customerCardId + verifiedAt`、`projectId + verifiedAt`。
- 新核销必须写入 `customerCardId`、`projectId`、`recognizedAmount`。
- 历史数据允许为空，但经营利润需要标记 `missing_card_usage_link` 或 `missing_card_unit_value`。

### 4.3 CommissionRecord 适配核销来源

目标：提成能覆盖项目订单、商品订单、次卡核销三类事实。

建议新增或明确使用：

| 字段 | 说明 |
| --- | --- |
| `sourceType` | `order_item`、`card_usage`、`manual_adjustment` |
| `sourceId` | 对应 `OrderItem.id` 或 `CardUsageRecord.id` |
| `orderItemId` | 保留兼容项目/商品订单 |
| `cardUsageRecordId` | 可选新增，更清晰支持核销提成 |

经营利润聚合规则：

- 商品毛利：按 `OrderItem.id` 汇总 `type=product`。
- 项目订单毛利：按 `OrderItem.id` 汇总 `type=project`。
- 次卡核销项目毛利：按 `CardUsageRecord.id` 汇总 `sourceType=card_usage` 或 `cardUsageRecordId`。

### 4.4 StockMovement 适配核销耗材

当前已存在 `sourceType=card_usage`，建议补强字段和写入规则：

| 字段/规则 | 说明 |
| --- | --- |
| `sourceType=card_usage` | 次卡核销耗材 |
| `sourceId=CardUsageRecord.id` | 直接指向核销记录 |
| `productId` | BOM 耗材商品 |
| `quantity` | 按负数出库或统一用绝对值，计算时取绝对值 |
| `unitCostSnapshot` | 如模型暂不支持，可先在 `remark` 或后续字段中补成本快照 |

新核销流程必须先创建 `CardUsageRecord`，再按 `projectId` 的 BOM 自动写耗材出库。

## 5. 页面与功能改造

### 5.1 利润看板

新增/调整指标：

| 模块 | 调整 |
| --- | --- |
| 收入结构 | 拆分“单次项目收入”“次卡核销履约收入”“商品收入”“办卡现金流”“充值现金流” |
| 成本结构 | 拆分“项目订单耗材”“次卡核销耗材”“商品成本”“项目提成”“次卡核销提成” |
| 数据质量 | 新增次卡关联缺口、核销单价缺口、核销提成缺口 |

口径说明文案：

```text
办卡和充值仅计入现金流；项目毛利按单次项目订单和次卡核销履约收入共同计算。
```

### 5.2 项目毛利

新增字段：

| 字段 | 说明 |
| --- | --- |
| 项目订单收入 | 直接收款项目订单收入 |
| 次卡核销收入 | 按核销单价确认的履约收入 |
| 总服务收入 | 项目订单收入 + 次卡核销收入 |
| 项目订单次数 | 普通项目订单次数 |
| 次卡核销次数 | `CardUsageRecord.times` |
| 核销耗材成本 | `StockMovement.sourceType=card_usage` |
| 核销提成 | 核销来源提成 |

新增操作：

- “明细”按钮：打开项目毛利明细弹窗。
- 明细分两个页签：
  - 项目订单明细。
  - 次卡核销明细。

次卡核销明细字段：

| 字段 | 说明 |
| --- | --- |
| 核销时间 | `verifiedAt` |
| 客户 | 客户姓名/手机号 |
| 次卡 | 卡名、客户卡编号 |
| 核销项目 | 项目名称 |
| 服务员工 | 美容师 |
| 核销次数 | `times` |
| 确认收入 | `recognizedAmount` |
| 耗材成本 | 核销关联库存流水成本 |
| 提成成本 | 核销关联提成 |
| 贡献毛利 | 确认收入 - 耗材 - 提成 |
| 数据缺口 | 缺项目、缺单价、缺 BOM、缺提成 |

### 5.3 会员卡履约/预收负债

建议页面改名：

```text
会员卡履约 -> 预收履约
```

或保留标题，但增加两个页签：

1. 次卡权益。
2. 储值余额。

次卡权益字段补充：

| 字段 | 说明 |
| --- | --- |
| 来源订单号 | 办卡订单 |
| 成交金额 | 实收办卡金额 |
| 已确认收入 | 已核销次数 * 核销单价 |
| 剩余履约价值 | 剩余次数 * 核销单价 |
| 赠送权益 | 赠送次数或赠送项目 |

储值余额字段：

| 字段 | 说明 |
| --- | --- |
| 客户 | 姓名/手机号 |
| 现金余额 | `CustomerBalanceAccount.cashBalance` |
| 赠送余额 | `giftBalance` |
| 最近充值 | 最近 `CustomerBalanceTransaction.type=recharge` |
| 最近消费 | 最近 `type=consume` |
| 风险 | 大额余额、长期未消费、赠送余额过高 |

### 5.4 员工人效

新增拆分：

| 字段 | 说明 |
| --- | --- |
| 项目订单服务收入 | 普通项目订单 |
| 次卡核销服务收入 | 次卡履约收入 |
| 商品销售收入 | 如员工参与商品销售 |
| 办卡销售额 | 现金流，不计贡献毛利 |
| 次卡核销提成 | 核销来源提成 |
| 贡献毛利 | 服务收入 - 耗材 - 提成 |

注意：办卡销售额可以作为销售业绩展示，但不要混入服务贡献毛利。

### 5.5 商品毛利

继续沿用当前订单明细弹窗，但补强：

- 明细中展示成本来源：订单快照、库存出库、商品档案、缺失。
- 若商品订单与项目订单来自同一收银流水，展示 `checkoutGroupNo`。
- 对历史无成本快照订单显示“历史成本估算”。

## 6. 后端聚合改造

### 6.1 新增统一收入事件模型（服务层聚合）

不一定首期新建数据库表，可以先在 `OperationProfitService` 内部形成统一聚合结构：

```ts
type ProfitRevenueEvent =
  | {
      sourceType: 'order_item';
      sourceId: number;
      projectId?: number;
      productId?: number;
      customerId?: number;
      beauticianId?: number;
      amount: number;
      occurredAt: Date;
    }
  | {
      sourceType: 'card_usage';
      sourceId: number;
      customerCardId: number;
      cardId?: number;
      projectId?: number;
      customerId: number;
      beauticianId?: number;
      amount: number;
      occurredAt: Date;
    };
```

经营利润按事件聚合，页面再按项目、商品、员工维度展示。

### 6.2 项目毛利聚合规则

```text
项目订单收入 = sum(OrderItem.netAmount where itemType=project)
次卡核销收入 = sum(CardUsageRecord.recognizedAmount)
服务收入 = 项目订单收入 + 次卡核销收入

项目订单耗材 = sum(StockMovement where sourceType=project_order)
次卡核销耗材 = sum(StockMovement where sourceType=card_usage)
耗材成本 = 优先实际耗材；缺实际耗材时用 BOM 标准成本估算并标记

提成成本 = 项目订单提成 + 次卡核销提成
贡献毛利 = 服务收入 - 耗材成本 - 提成成本
```

### 6.3 预收负债聚合规则

```text
次卡剩余履约价值 = sum(CustomerCard.remainingTimes * CustomerCard.recognizedUnitValue)
储值现金余额 = sum(CustomerBalanceAccount.cashBalance)
储值赠送余额 = sum(CustomerBalanceAccount.giftBalance)
预收负债合计 = 次卡剩余履约价值 + 储值现金余额 + 储值赠送余额
```

赠送余额是否进入负债，可在页面标记为“赠送权益负债”，与现金余额分开。

## 7. 历史数据回填方案

### 7.1 次卡来源订单回填

匹配优先级：

1. `CustomerCard.createdAt` 附近同客户、同卡名、`OrderItem.itemType=card` 的订单。
2. 同客户、同卡名、金额接近 `Card.price` 的订单。
3. 无法匹配则保留为空，标记 `missing_card_source_order`。

回填字段：

- `CustomerCard.sourceOrderId`
- `CustomerCard.sourceOrderItemId`
- `CustomerCard.paidAmount`
- `CustomerCard.recognizedUnitValue`
- `CustomerCard.pricingSnapshot`

### 7.2 次卡核销记录回填

匹配优先级：

1. 同客户、同卡名、核销时间处于客户卡有效期内。
2. 若同客户同卡名多张卡，优先剩余次数变化能解释的卡。
3. 项目名精确匹配 `Project.name`。
4. 项目名模糊匹配仅作为候选，需要数据质量标记。

回填字段：

- `CardUsageRecord.customerCardId`
- `CardUsageRecord.cardId`
- `CardUsageRecord.projectId`
- `CardUsageRecord.storeId`
- `CardUsageRecord.recognizedUnitValue`
- `CardUsageRecord.recognizedAmount`

### 7.3 不可自动回填的数据

以下情况不建议自动写死，需要人工确认或保留缺口：

- 同一客户同一时期购买多张同名卡。
- 卡项改价、赠送次数缺失。
- 核销项目名无法匹配项目档案。
- 历史核销没有服务员工。
- 历史提成规则缺失。

## 8. 数据质量状态

建议扩展 `MissingCostReason`：

| 缺口码 | 说明 |
| --- | --- |
| `missing_card_source_order` | 客户次卡缺来源订单 |
| `missing_card_usage_link` | 核销记录缺客户卡关联 |
| `missing_card_project_link` | 核销记录缺项目档案关联 |
| `missing_card_unit_value` | 核销单价缺失 |
| `missing_card_usage_commission` | 次卡核销提成缺失 |
| `missing_card_usage_consumption` | 次卡核销耗材缺失 |
| `estimated_card_usage_income` | 消课收入为估算 |
| `estimated_historical_product_cost` | 历史商品成本为估算 |

页面展示规则：

- P0 缺口出现时，毛利状态至少为“成本缺口”或“估算”。
- 不允许把缺提成、缺耗材的项目标记为“高利润”。
- 所有估算收入必须有提示，不和真实订单收入混在一起。

## 9. 推荐实施阶段

### 阶段 1：次卡事实源补强

目标：新产生的数据完整。

任务：

1. `CustomerCard` 新增来源订单、成交金额、核销单价快照。
2. `CardUsageRecord` 新增客户卡、卡项、项目、门店、核销收入快照。
3. 管理端办卡、终端办卡写入 `CustomerCard` 快照。
4. 管理端/终端核销写入 `CardUsageRecord` 快照。
5. 核销后自动写 `StockMovement.sourceType=card_usage`。
6. 核销后生成或关联 `CommissionRecord.sourceType=card_usage`。

验收：

- 新办卡后，客户次卡能看到来源订单。
- 新核销后，项目毛利能看到核销收入、耗材、提成。
- 不新增项目订单收款，不重复计现金收入。

### 阶段 2：经营利润聚合改造

目标：利润看板、项目毛利、员工人效按新事实源计算。

任务：

1. `OperationProfitService` 引入统一收入事件。
2. 项目毛利拆分项目订单收入和次卡核销收入。
3. 项目毛利明细弹窗增加核销明细。
4. 员工人效计入次卡核销服务收入和核销提成。
5. 利润看板成本结构拆分次卡核销耗材/提成。

验收：

- 同一项目的项目订单收入和次卡核销收入能分别解释。
- 次卡核销不会出现在项目订单总收款中。
- 项目毛利总收入等于项目订单收入 + 核销履约收入。

### 阶段 3：预收履约页补齐储值余额

目标：预收负债看全。

任务：

1. 会员卡履约页增加储值余额页签。
2. 展示现金余额、赠送余额、最近充值、最近消费。
3. 利润看板展示预收新增和履约消耗。
4. 大额余额、长期未消费、赠送余额过高进入风险提醒。

验收：

- 充值流水能在预收履约页查到。
- 余额消费不重复计现金收入。
- 储值现金余额和赠送余额分开展示。

### 阶段 4：历史数据回填与人工确认

目标：提高历史利润可信度，但不伪造无法确定的数据。

任务：

1. 只读审计脚本输出可自动匹配、需人工确认、不可匹配三类数据。
2. 对高置信数据执行回填。
3. 对低置信数据保留缺口并在页面提示。
4. 输出门店级数据质量报告。

验收：

- 历史客户卡来源订单匹配率可见。
- 历史核销项目匹配率可见。
- 无法匹配的数据不会被强行按 0 成本/0 提成算高利润。

## 10. 验收清单

| 验收项 | 通过标准 |
| --- | --- |
| 次卡销售不计利润 | 办卡只增加现金流和预收，不进入经营收入 |
| 次卡核销计项目毛利 | 核销收入进入项目毛利，且不新增项目订单收款 |
| 次卡核销有单价 | 新核销记录有 `recognizedAmount` |
| 次卡核销有项目 | 新核销记录有 `projectId` |
| 次卡核销有耗材 | BOM 项目核销后有 `StockMovement.sourceType=card_usage` |
| 次卡核销有提成 | 命中提成规则后有核销来源提成 |
| 项目毛利可追溯 | 能展开项目订单明细和次卡核销明细 |
| 预收负债完整 | 次卡剩余权益、储值现金余额、赠送余额均可见 |
| 数据缺口可见 | 缺来源订单、缺项目、缺单价、缺耗材、缺提成都有提示 |
| 经营收入不重复 | 现金收入和经营收入可分别解释 |

## 11. 产品结论

经营利润模块下一步不应继续只加展示字段，而应先补强“钱从哪里来、服务在哪里履约、成本在哪里发生”的事实源。

最关键的改造是：把次卡核销定义为项目履约收入来源，而不是普通项目订单。这样既能让项目毛利真实反映次卡客户的服务成本，也能避免办卡现金流和核销履约收入重复计入。

推荐优先推进阶段 1 和阶段 2。只要新数据链路完整，历史数据即使部分估算，也可以通过数据质量标记逐步修复。
