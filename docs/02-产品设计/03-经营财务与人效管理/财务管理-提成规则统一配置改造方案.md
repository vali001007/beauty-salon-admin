# 财务管理-提成规则统一配置改造方案

## 1. 最新结论

提成体系拆成两个层级：

1. **创建规则**：只创建“提成算法模板”，不选择具体项目、商品、卡项或员工。
2. **规则配置**：选择某条规则，并绑定适用对象和适用员工，真正决定哪些订单会命中这条规则。

统一入口仍放在：

- 财务管理 - 提成规则

项目管理、商品管理、卡项管理不再直接维护提成，只做只读提示或跳转到财务管理。

## 2. 为什么要拆

之前把“规则算法”和“适用范围”放在一个表单里，页面会变复杂：

- 同一套 8% 项目提成，可能要用于多个项目和多个员工。
- 如果每个对象员工组合都展示成一条规则，财务人员会觉得规则很多、很乱。
- 后续修改比例时，如果规则被拆散到很多条记录，容易漏改。

拆成两层后，业务理解更清晰：

- **规则**回答：怎么算钱。
- **配置**回答：这条算法用在哪些对象、哪些员工身上。

## 3. 业务对象定义

### 3.1 规则

规则是可复用的提成算法模板。

规则只保存：

| 字段 | 说明 |
| --- | --- |
| 规则名称 | 如“项目服务 8% 提成” |
| 规则类型 | 项目 / 商品 / 办卡 / 充值 / 新客 |
| 计算方式 | 按比例 / 固定金额 |
| 提成比例 | 按比例时填写 |
| 固定金额 | 固定金额时填写 |
| 计算基数 | 实收金额 / 服务费 / 毛利 |
| 最低提成阈值 | 可选 |
| 指定服务加成 | 可选 |
| 状态 | 启用 / 停用 |
| 备注 | 规则说明 |

规则不保存：

- 项目 ID。
- 商品 ID。
- 卡项 ID。
- 员工 ID。
- 员工等级。
- 优先级。

### 3.2 规则配置

规则配置是规则的适用范围。

规则配置保存：

| 字段 | 说明 |
| --- | --- |
| 规则 | 选择已创建的规则 |
| 规则类型 | 跟随规则，不单独改 |
| 适用对象 | 项目 / 商品 / 卡项，多选 |
| 适用员工 | 员工多选 |
| 状态 | 启用 / 停用 |
| 生效时间 | 阶段 1 可不做，阶段 2 再加 |
| 备注 | 配置说明 |

规则配置才决定订单是否生成提成。

## 4. 页面结构

「财务管理 - 提成规则」建议拆成两个页签。

### 4.1 规则库

规则库只维护提成算法。

列表字段：

- 规则名称
- 类型
- 计算方式
- 比例 / 固定金额
- 计算基数
- 状态
- 已配置数量
- 最近更新时间
- 操作：编辑、停用、删除、去配置

新增规则表单：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 规则名称 | 是 | 如“商品销售 5% 提成” |
| 规则类型 | 是 | 项目 / 商品 / 办卡 / 充值 / 新客 |
| 计算方式 | 是 | 按比例 / 固定金额 |
| 提成比例 | 条件必填 | 按比例时必填 |
| 固定金额 | 条件必填 | 固定金额时必填 |
| 计算基数 | 是 | 实收金额 / 服务费 / 毛利 |
| 最低提成阈值 | 否 | 低于该金额不生成提成 |
| 指定服务加成 | 否 | 指定服务场景加成 |
| 状态 | 是 | 启用 / 停用 |
| 备注 | 否 | 财务说明 |

### 4.2 规则配置

规则配置维护适用范围。

列表字段：

- 配置名称
- 规则名称
- 规则类型
- 适用对象
- 适用员工
- 状态
- 最近更新时间
- 操作：编辑、停用、删除

新增配置表单：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 选择规则 | 是 | 下拉选择规则库中已启用规则 |
| 配置名称 | 否 | 默认用规则名 + 对象范围生成 |
| 适用对象 | 条件必填 | 项目 / 商品 / 办卡规则必填 |
| 适用员工 | 是 | 多选员工 |
| 状态 | 是 | 启用 / 停用 |
| 备注 | 否 | 配置说明 |

## 5. 各类型配置口径

| 类型 | 创建规则时 | 规则配置时 |
| --- | --- | --- |
| 项目提成 | 只定义项目提成算法 | 选择一个或多个项目 + 一个或多个员工 |
| 商品提成 | 只定义商品提成算法 | 选择一个或多个商品 + 一个或多个员工 |
| 办卡提成 | 只定义开卡提成算法 | 选择一个或多个卡项 + 一个或多个员工 |
| 充值提成 | 只定义充值提成算法 | 选择一个或多个员工 |
| 新客提成 | 只定义新客提成算法 | 选择一个或多个员工 |

阶段 1 暂不支持：

- 全部项目。
- 全部商品。
- 全部卡项。
- 项目分类。
- 商品分类。
- 员工等级。
- 门店通用兜底。
- 优先级。

## 6. 命中规则口径

订单或核销发生后，系统按以下步骤生成提成：

1. 判断业务类型：项目、商品、办卡、充值或新客。
2. 识别业务对象：项目 ID、商品 ID、卡项 ID；充值和新客无业务对象。
3. 识别服务员工或办理员工。
4. 查找启用的规则配置。
5. 规则配置中对象命中、员工命中，则取该配置关联的规则算法。
6. 按规则算法生成提成流水。

提成流水仍然记录：

- 员工。
- 来源订单。
- 来源明细。
- 规则。
- 配置。
- 金额基数。
- 比例或固定金额。
- 提成金额。

经营利润、商品毛利、项目毛利、员工绩效仍以 `CommissionRecord` 作为提成成本来源。

## 7. 冲突控制

规则库允许创建多条同类型算法。

规则配置必须控制命中冲突。

以下组合不能同时存在多条启用配置：

```text
storeId + type + objectId + employeeId
```

充值和新客没有对象，冲突组合为：

```text
storeId + type + employeeId
```

冲突示例：

- “肩颈舒压养护 + 唐伊”不能同时配置两条启用项目提成。
- “抗衰紧致眼霜 + 唐伊”不能同时配置两条启用商品提成。
- “综合养护 20 次卡 + 系统管理员”不能同时配置两条启用办卡提成。
- “充值 + 唐伊”不能同时配置两条启用充值提成。

提示文案：

- 该对象与员工组合已存在启用提成配置，请编辑原配置。
- 该员工已存在启用充值提成配置，请编辑原配置。
- 该员工已存在启用新客提成配置，请编辑原配置。

## 8. 数据模型建议

### 8.1 推荐模型

建议拆成两张主表。

#### CommissionRule

规则算法表。

保留字段：

- id
- storeId
- name
- type
- calcMethod
- rate
- fixedAmount
- calcBase
- minThreshold
- isDesignated
- designatedBonus
- status
- remark

不再承载：

- targetType
- targetId
- userId
- levelId
- priority

#### CommissionRuleAssignment

规则配置表。

建议字段：

| 字段 | 说明 |
| --- | --- |
| id | 配置 ID |
| storeId | 门店 |
| ruleId | 关联规则 |
| type | 冗余规则类型，便于查询 |
| targetType | specific / all |
| targetId | 项目、商品、卡项 ID；充值和新客为空 |
| userId | 员工 ID |
| status | active / disabled / archived |
| remark | 备注 |
| createdAt / updatedAt | 时间 |

### 8.2 阶段 1 低成本实现

如果暂不改 Prisma 表结构，也可以用现有 `CommissionRule` 表兼容实现：

- 页面上的“规则库”创建一条无对象、无员工的规则模板。
- 页面上的“规则配置”在保存时复制规则算法，并按对象员工组合拆成多条 `CommissionRule`。
- 用 `sourceRuleId` 或 `groupKey` 标识这些拆分记录来自同一个规则模板。

但这只是过渡方案，缺点是：

- 同一规则算法修改后，需要同步更新所有已拆分配置。
- 规则库和配置记录混在一张表里，后续维护成本高。

因此推荐尽快进入标准双表模型。

## 9. 旧数据迁移口径

现有已落库的 `CommissionRule` 多数同时包含算法和适用范围。

迁移建议：

1. 把现有规则按算法字段聚合，生成规则库记录。
2. 把原有 `targetId + userId` 拆成规则配置记录。
3. 有历史提成流水引用的旧规则不物理删除，保留追溯。
4. 新订单只走新规则配置。
5. 历史订单补算时使用新配置，不覆盖已人工修改的提成流水。

## 10. 页面交互建议

### 10.1 创建规则

用户路径：

1. 进入财务管理 - 提成规则。
2. 打开“规则库”页签。
3. 点击“新增规则”。
4. 只填写算法。
5. 保存后列表出现该规则。
6. 页面提示“规则已创建，请继续配置适用对象和员工”。

### 10.2 配置规则

用户路径：

1. 点击规则库里的“去配置”，或进入“规则配置”页签。
2. 选择规则。
3. 按规则类型选择对象。
4. 选择员工。
5. 保存配置。
6. 系统校验对象员工组合是否冲突。

### 10.3 修改规则

修改规则算法时，需要明确影响范围：

- 如果规则已有启用配置，保存前提示“该规则已被 X 条配置使用，修改后会影响后续新订单提成计算。”
- 已生成的提成流水不自动重算。
- 历史流水如需调整，走“提成明细修改”或“历史补算”。

### 10.4 修改配置

修改配置只影响后续新订单。

可修改：

- 适用对象。
- 适用员工。
- 状态。
- 备注。

不可在配置页修改：

- 提成比例。
- 固定金额。
- 计算基数。

这些必须回到规则库修改。

## 11. 对其他模块的影响

### 11.1 项目管理

项目管理不再出现提成配置步骤。

可选只读提示：

- 当前项目已配置 X 条提成配置。
- 当前项目未配置提成，请到财务管理 - 提成规则 - 规则配置处理。

### 11.2 商品管理

商品详情可展示只读提示：

- 当前商品已配置 X 条商品提成。
- 当前商品未配置提成。

### 11.3 卡项管理

卡项详情可展示只读提示：

- 当前卡项已配置 X 条办卡提成。
- 当前卡项未配置办卡提成。

### 11.4 提成明细

提成明细继续定位为流水修正页：

- 查看系统生成提成。
- 修改员工。
- 修改比例或金额。
- 填写调整备注。
- 修改后同步订单成本和经营利润。

不恢复审核确认。

## 12. 经营利润缺口解释

拆分后，缺口原因更清晰：

| 缺口 | 含义 |
| --- | --- |
| 规则未创建 | 没有对应类型的规则算法 |
| 规则未配置 | 有规则算法，但没有绑定对象和员工 |
| 对象未配置 | 当前项目、商品或卡项没有配置 |
| 员工未配置 | 当前员工不在配置范围内 |
| 服务员工缺失 | 订单或核销没有服务员工 |
| 历史流水缺失 | 规则配置前产生的历史订单未补算 |

## 13. 分阶段开发计划

### 阶段 1：页面和逻辑拆分

目标：把“规则算法”和“规则适用范围”从产品体验上拆开。

开发内容：

1. 提成规则页面增加“规则库”和“规则配置”两个页签。
2. 规则库新增、编辑只维护算法字段。
3. 规则配置新增、编辑选择规则、对象和员工。
4. 项目、商品、卡项对象选择使用下拉多选。
5. 员工选择使用下拉多选。
6. 保存配置时校验对象员工组合冲突。
7. 订单提成生成改为读取规则配置，再取规则算法。

验收标准：

- 创建规则时不再要求选择对象和员工。
- 没有配置的规则不会被订单命中。
- 同一个对象和员工只能命中一条启用配置。
- 新订单可以按配置生成提成流水。

### 阶段 2：数据模型标准化

目标：从数据结构上拆出规则和配置。

开发内容：

1. 新增 `CommissionRuleAssignment` 表。
2. `CommissionRule` 表只保留算法字段。
3. 新增迁移脚本，把旧规则拆成规则库和配置。
4. 提成流水记录 `ruleId` 和 `assignmentId`。
5. API 分成规则库 API 和规则配置 API。

验收标准：

- 规则算法修改能统一影响后续配置命中。
- 配置列表不再依赖复制出来的规则记录。
- 历史提成流水仍可追溯旧规则。

### 阶段 3：历史治理和补算

目标：清理历史复杂规则，补齐历史提成缺口。

开发内容：

1. 输出历史规则审计清单。
2. 标出重复配置和无效配置。
3. 提供历史订单提成补算预览。
4. 财务确认后批量补算。
5. 已人工修改的提成流水不覆盖。

验收标准：

- 历史缺口可解释。
- 补算结果可预览、可追溯。
- 经营利润和员工绩效同步更新。

## 14. 推荐下一步

建议先做阶段 1。

阶段 1 可以先不大改数据库，但必须把页面和业务口径拆开，让财务人员理解为：

- 规则库：维护算法。
- 规则配置：维护适用范围。

等页面和业务流程稳定后，再做阶段 2 的双表模型标准化。
