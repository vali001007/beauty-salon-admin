# Ami_Aura 数字员工商业模式与产品改造方案

版本：v1.0  
日期：2026-06-07  
适用对象：Ami_Aura 智能终端、Ami_Core 管理后台、server-v2 Terminal / AI Gateway  
核心问题：Ami_Aura 未来以“数字员工”的方式参与门店运营，采用“工时 + 提成”的商业模式是否合适，以及产品需要如何改造。

## 1. 结论

Ami_Aura 可以升级为“数字员工”，但不建议直接采用单一的“工时 + 提成”收费。

更适合的商业模式是：

```text
基础订阅费 + 数字员工席位/值守费 + 绩效提成 + 封顶/保护机制
```

其中：

- 基础订阅费覆盖 Ami_Core、门店数据、账号权限、基础终端能力和系统维护。
- 数字员工席位/值守费覆盖 Ami_Aura 在门店承担固定岗位职责的持续可用能力。
- 绩效提成只针对 Ami_Aura 可明确归因、可审计、可复核的增量贡献收取。
- 必须设置封顶、退款扣回、人工覆盖、门店保护期和异常订单排除规则。

如果只按“工时 + 提成”收费，会遇到三个问题：

1. 门店老板很难接受 AI 空闲时间也被计费。
2. 提成归因容易和前台、顾问、美容师产生争议。
3. 当前产品链路还缺少完整的任务归因、转化证明、绩效看板和结算审计，暂时不能支撑严肃结算。

因此，推荐把“工时”改造成“数字员工值守服务费”，把“提成”改造成“可归因成果分成”。

## 2. 为什么这个方向值得做

美业门店本身就有“岗位 + 业绩”的组织结构：

- 前台有接待、预约、核销、收银职责。
- 顾问有推荐、邀约、转化职责。
- 美容师有服务、记录、复购建议职责。
- 店长有经营分析、排班、跟进、风险处理职责。

Ami_Aura 已经不只是一个硬件屏幕，而是逐步参与这些岗位的真实流程：客户识别、今日预约、次卡核销、收银、办卡、充值、服务记录、护理建议、AI 推荐、营销邀约和经营提醒。

这意味着它未来可以从“工具”升级为“门店里的数字员工”：

- 替员工减少重复操作。
- 替店长发现经营机会。
- 替顾问生成推荐话术。
- 替美容师沉淀服务记录。
- 替门店推动复购、升单和邀约。

从商业角度看，数字员工比单纯硬件或 SaaS 更容易解释价值：门店不是只买一台设备，而是在购买一个可以持续参与经营、产生结果、被管理和被考核的岗位能力。

## 3. 但不能直接照搬真人员工的工时和提成

### 3.1 工时的定义需要改造

真人员工的工时代表人在岗劳动时间。Ami_Aura 的“工时”如果按开机时长、模型运行时长或屏幕在线时长计算，门店很难接受。

建议改为三类口径：

| 口径 | 是否建议收费 | 说明 |
| --- | --- | --- |
| 在线时长 | 不建议单独收费 | 只能说明设备在线，不能证明创造价值 |
| 值守时长 | 可以收费 | 终端处于指定岗位模式，并可响应业务任务 |
| 有效任务时长 | 可以作为高级版本指标 | 处理预约、核销、推荐、邀约、服务记录等任务的实际工作时长 |

推荐对外表达：

```text
数字员工值守费，而不是 AI 工时费。
```

### 3.2 提成的定义需要改造

真人员工提成通常按成交额、办卡金额、项目销售、产品销售或复购转化结算。但 Ami_Aura 不应该对所有经过终端的订单都拿提成。

提成只能来自清晰归因的结果：

- Ami_Aura 生成推荐，员工采纳，客户购买。
- Ami_Aura 生成邀约任务，客户到店并成交。
- Ami_Aura 识别沉睡客户，触发复购。
- Ami_Aura 在服务后推荐下次预约，客户完成预约或购买。
- Ami_Aura 辅助办卡、充值、升卡，并有明确推荐链路。

不应计入提成的场景：

- 客户本来就已经决定购买，只是在终端完成收银。
- 员工手动录入的历史订单。
- 退款、撤销、异常订单。
- 与员工归因冲突且无法确认的订单。
- 门店设置为不参与 AI 分成的项目或活动。

推荐对外表达：

```text
只对 Ami_Aura 可证明参与并带来转化的增量成交收取绩效分成。
```

## 4. 推荐商业模式

### 4.1 标准套餐结构

| 收费项 | 建议口径 | 适用阶段 |
| --- | --- | --- |
| Ami_Core 基础订阅 | 按门店/月 | 必选 |
| Ami_Aura 终端服务费 | 按设备/月或数字员工席位/月 | 必选 |
| AI 能力包 | 按套餐、调用量或高级能力 | 可选 |
| 数字员工值守费 | 按岗位包/月，不建议按分钟裸计费 | 成熟后推出 |
| 绩效提成 | 按可归因成交额比例 | 试点后推出 |
| 供应链分成 | 按耗材采购 GMV 或服务费 | 后续增长项 |

### 4.2 建议定价形态

#### 方案 A：保守版，适合早期签约

```text
固定月费 + 免费绩效看板
```

特点：

- 门店决策简单。
- 避免早期归因争议。
- 适合 MVP 和样板店。

适合阶段：现在到第一批样板店验证。

#### 方案 B：增长版，适合标杆门店

```text
固定月费 + 低比例绩效分成 + 月度封顶
```

特点：

- Ami 与门店增长目标绑定。
- 门店感知为“效果付费”。
- 需要完整归因和结算能力。

适合阶段：3-5 家样板店跑通归因后。

#### 方案 C：数字员工版，适合成熟商业化

```text
数字员工席位费 + 岗位值守包 + 可归因绩效提成
```

示例岗位：

- Aura 前台助理
- Aura 顾问助理
- Aura 美容师助理
- Aura 店长助理
- Aura 会员运营助理

特点：

- 更接近“给门店增加一个数字员工”。
- 可按岗位能力售卖。
- 可以做连锁总部统一采购。

适合阶段：终端任务闭环、结算审计、权限体系成熟后。

## 5. 数字员工岗位设计

### 5.1 Aura 前台助理

职责：

- 今日预约查看。
- 客户到店确认。
- 快速建档。
- 次卡核销。
- 收银开单。
- 小票打印。
- 异常提醒。

可计量指标：

- 处理预约数。
- 到店确认数。
- 核销数。
- 收银订单数。
- 平均接待耗时。
- 失败/撤销次数。

适合收费：

- 席位费。
- 不建议按成交提成作为主收费。

### 5.2 Aura 顾问助理

职责：

- 客户画像摘要。
- 项目推荐。
- 升卡/办卡/充值建议。
- 推荐话术生成。
- 推荐采纳与跳过记录。
- 成交归因。

可计量指标：

- 推荐展示次数。
- 员工采纳次数。
- 转化订单数。
- 可归因成交额。
- 推荐转化率。
- 客单价提升。

适合收费：

- 席位费 + 可归因绩效提成。

### 5.3 Aura 美容师助理

职责：

- 服务前客户摘要。
- 皮肤检测解读。
- 服务记录生成。
- 耗材记录。
- 下次护理建议。
- 下次预约建议。

可计量指标：

- 服务记录完成率。
- 服务记录补录减少量。
- 耗材记录完整率。
- 下次预约转化率。
- 护理建议采纳率。

适合收费：

- 席位费。
- 对复购和下次预约可做低比例绩效分成。

### 5.4 Aura 店长助理

职责：

- 今日经营摘要。
- 异常订单提醒。
- 库存风险提醒。
- 员工任务提醒。
- 沉睡客户提醒。
- 待跟进任务分配。

可计量指标：

- 风险提醒数。
- 已处理风险数。
- 跟进任务完成率。
- 库存缺货预警命中率。
- 门店经营动作完成率。

适合收费：

- 高级管理包。
- 不建议直接按营业额提成。

### 5.5 Aura 会员运营助理

职责：

- 沉睡客户识别。
- 生日/周期/项目复购提醒。
- 邀约话术生成。
- 跟进任务创建。
- 触达结果记录。
- 到店与成交归因。

可计量指标：

- 生成跟进任务数。
- 完成跟进数。
- 到店人数。
- 可归因复购订单。
- 沉睡客户唤醒率。

适合收费：

- AI 运营包 + 可归因绩效提成。

## 6. 提成归因规则

要让门店接受提成，必须做到“看得见、算得清、可申诉”。

### 6.1 归因事件链

每一笔可能计提成的订单，都要能回放事件链：

```text
Ami_Aura 识别机会
-> 展示推荐/生成话术/创建跟进任务
-> 员工采纳或执行
-> 客户到店/确认/购买
-> 订单支付完成
-> 退款保护期结束
-> 进入结算
```

### 6.2 推荐归因字段

建议新增统一归因字段：

| 字段 | 说明 |
| --- | --- |
| attributionId | 归因链路 ID |
| sourceAgent | 来源数字员工，例如 aura_consultant |
| sourceIntent | 来源意图，例如 care_advice.generate、card_order.create |
| recommendationId | 推荐记录 ID |
| touchId | 触达/展示/话术记录 ID |
| acceptedBy | 哪位员工采纳 |
| acceptedAt | 采纳时间 |
| convertedOrderId | 转化订单 |
| convertedAmount | 转化金额 |
| commissionEligibleAmount | 可计提金额 |
| attributionWindow | 归因窗口，例如 7 天、14 天、30 天 |
| attributionStatus | pending、confirmed、rejected、settled |
| disputeStatus | none、disputed、resolved |

### 6.3 归因窗口建议

| 场景 | 建议归因窗口 |
| --- | --- |
| 现场推荐成交 | 当日或 24 小时 |
| 服务后下次预约 | 7 天 |
| 沉睡客户邀约 | 14-30 天 |
| 办卡/充值升单 | 24 小时到 7 天 |
| 耗材补货建议 | 7-14 天 |

### 6.4 提成计算建议

基础公式：

```text
可结算提成 = 可计提成交额 × 提成比例 × 有效归因权重
```

有效归因权重示例：

| Aura 参与程度 | 权重建议 |
| --- | --- |
| 只展示信息，员工未采纳 | 0 |
| 生成建议，员工采纳但未成交 | 0 |
| 生成建议，员工采纳并当场成交 | 100% |
| 生成邀约任务，客户到店成交 | 50%-100%，按门店协议 |
| 多员工、多渠道共同影响 | 按规则拆分或只计固定激励 |

### 6.5 结算保护规则

必须支持：

- 退款扣回。
- 异常订单排除。
- 员工手动改归因需审批。
- 门店可申诉。
- 每月封顶。
- 新门店保护期。
- 项目黑名单和活动黑名单。

## 7. 产品需要如何改造

### 7.1 从“终端功能”改成“数字员工工作台”

当前 Aura 更像按角色提供快捷功能的智能终端。未来需要在产品概念上增加数字员工身份：

- 数字员工名称：例如 Aura 前台助理、Aura 顾问助理。
- 当前岗位：前台、顾问、美容师、店长、会员运营。
- 当前值守状态：值守中、忙碌、离线、需人工接管。
- 今日完成任务：接待、核销、推荐、服务记录、邀约。
- 今日贡献：节省时间、推荐采纳、转化金额、待处理风险。

终端首页不只是“我能做什么”，还要体现“我今天帮门店完成了什么”。

### 7.2 增加任务系统

数字员工必须围绕任务运转，而不是围绕聊天运转。

建议新增任务类型：

- customer_reception：客户接待。
- appointment_followup：预约确认/改期/爽约跟进。
- card_consume：次卡核销。
- cashier_checkout：收银开单。
- service_record：服务记录。
- care_advice：护理建议。
- recommendation_followup：推荐跟进。
- customer_invitation：客户邀约。
- manager_alert：店长风险提醒。

任务状态：

```text
created -> suggested -> accepted -> executing -> completed -> converted / closed
```

每个任务都要记录：

- 由谁创建：人或 Aura。
- 谁执行。
- 是否被采纳。
- 是否产生业务结果。
- 是否可计入绩效。

### 7.3 增加推荐采纳与跳过闭环

当前已有推荐事件接口方向，但要支撑提成，需要把推荐生命周期补全：

- shown：展示给员工。
- accepted：员工采纳。
- skipped：员工跳过。
- edited：员工修改话术或推荐内容。
- sent：已触达客户。
- converted：产生转化。
- rejected：门店确认不计入。

前端上不能只显示“AI 建议”，必须给员工明确操作：

- 采纳建议。
- 换一个建议。
- 跳过并选择原因。
- 转为跟进任务。
- 转为预约。
- 转为收银/办卡。

### 7.4 服务记录必须成为核心闭环

如果 Aura 要像数字员工一样参与服务和提成，服务记录不能停留在“完成任务”。

需要补齐：

- 服务前客户摘要。
- 本次项目。
- 服务美容师。
- 使用产品/耗材。
- 护理结果。
- 客户反馈。
- 服务图片。
- 下次护理建议。
- 是否推荐项目/卡项。
- 是否转前台收银。
- 是否创建下次预约。

这部分是后续判断“服务后复购是否由 Aura 贡献”的基础。

### 7.5 收银与办卡必须支持归因

所有可能产生绩效提成的交易，需要在开单或支付时支持选择/确认归因：

- 本单是否由 Aura 推荐触发。
- 关联哪条推荐。
- 关联哪次客户邀约。
- 关联哪次服务记录。
- 关联哪位员工。
- 是否参与数字员工绩效结算。

前台不能被迫填写复杂表单，建议用默认推荐 + 一键确认：

```text
本单可能来自 Aura 顾问助理刚才推荐的“补水次卡升级方案”，是否关联？
```

### 7.6 管理端增加“数字员工管理”

Ami_Core 需要新增数字员工管理模块，至少包括：

- 数字员工列表。
- 岗位配置。
- 能力开关。
- 提成规则。
- 归因规则。
- 任务看板。
- 绩效看板。
- 结算单。
- 申诉与调整。

店长视角要能看到：

- Aura 今天处理了多少任务。
- 哪些建议被采纳。
- 哪些建议带来成交。
- 哪些订单计入 Aura 绩效。
- 哪些订单被排除。
- 本月预计结算金额。

总部视角要能按门店、设备、岗位、能力包对比。

### 7.7 后端增加结算与审计能力

server-v2 需要补齐以下能力：

| 模块 | 说明 |
| --- | --- |
| DigitalAgent | 数字员工主体，关联门店、设备、岗位 |
| AgentShift | 值守记录，记录开始、结束、岗位、在线状态 |
| AgentTask | 数字员工任务 |
| RecommendationEvent | 推荐生命周期事件 |
| Attribution | 归因链路 |
| CommissionRule | 提成规则 |
| CommissionStatement | 月度结算单 |
| CommissionStatementItem | 结算明细 |
| Dispute | 门店申诉与处理 |
| AuditLog | 关键操作审计 |

### 7.8 权限与风控改造

数字员工不能绕过人类权限。

原则：

- Aura 只能在门店授权范围内工作。
- AI 可以建议，但高风险动作必须由人确认。
- 取消预约、退款、余额调整、大额优惠、办卡、充值、核销都必须有人工确认。
- 结算相关规则变更必须由店长或总部管理员操作。
- 所有可计提事件必须可追踪。

## 8. 对现有 Ami_Aura 的改造优先级

### P0：先让终端业务事实闭合

目标：先保证真实业务链路可靠，不能为了提成先做概念包装。

重点：

- 预约到店后生成或关联服务任务。
- 完整服务记录表单。
- 服务记录关联耗材。
- 核销后关联服务记录。
- 收银支付状态收敛。
- 会员余额消费闭环。
- 推荐事件落库并可查询。
- 终端写操作增加幂等、确认、失败兜底。

### P1：建立推荐和归因闭环

目标：让每条 AI 建议都能被追踪。

重点：

- 推荐 shown / accepted / skipped / converted 事件。
- 推荐与订单、预约、服务记录关联。
- 推荐来源区分：客户画像、皮肤检测、服务记录、沉睡客户、库存/项目策略。
- 管理端能看推荐转化漏斗。

### P2：建立数字员工任务和绩效看板

目标：让门店老板看到 Aura 的工作量和贡献。

重点：

- 数字员工任务中心。
- 终端今日贡献面板。
- 管理端 Aura 绩效看板。
- 按岗位统计任务、采纳、转化、节省时间。
- 支持门店导出明细。

### P3：试点绩效分成

目标：在小范围样板店验证商业可接受度。

重点：

- 提成规则配置。
- 月度预结算单。
- 门店确认/申诉。
- 退款扣回。
- 月度封顶。
- 项目白名单。

### P4：推出数字员工套餐

目标：正式商业化。

重点：

- 岗位套餐。
- 连锁总部统一配置。
- 多门店绩效对比。
- 销售物料和合同条款。

## 9. MVP 建议

第一阶段不要急着卖“工时 + 提成”，建议做成“数字员工价值证明版”。

### MVP 范围

聚焦三个最容易证明价值的场景：

1. 服务后复购推荐。
2. 沉睡客户邀约。
3. 办卡/充值/项目升级推荐。

### MVP 指标

必须能回答门店老板的五个问题：

1. Aura 今天帮我处理了多少件事？
2. 员工采纳了多少条 Aura 建议？
3. 哪些建议真的带来了成交？
4. 这些成交金额是多少？
5. 如果按规则结算，Aura 预计应该拿多少绩效？

### MVP 不建议做

- 不按开机时长收费。
- 不对所有终端订单抽成。
- 不做无法解释的 AI 自动成交归因。
- 不让 AI 自动执行退款、取消、余额调整等高风险动作。
- 不在归因未成熟时对客户承诺严格效果付费。

## 10. 对外销售表达建议

不建议说：

```text
我们按 AI 工时收费，还要抽提成。
```

建议说：

```text
Ami_Aura 不是普通收银屏，而是门店里的数字员工。
它可以承担前台助理、顾问助理、美容师助理和会员运营助理的部分工作。
基础费用覆盖数字员工长期值守和系统能力。
只有当 Aura 的推荐、邀约或服务建议被员工采纳，并且产生可确认成交时，才会进入绩效分成。
所有绩效都有明细、有封顶、有退款扣回，门店可以查看和复核。
```

## 11. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 门店认为收费复杂 | 签约阻力上升 | 早期只卖固定套餐，绩效只做报表不收费 |
| 员工担心被抢提成 | 使用抵触 | Aura 先定位为员工助理，提成可与员工共享或不冲突 |
| 归因争议 | 结算纠纷 | 只对明确事件链归因的订单计提 |
| AI 建议不稳定 | 信任下降 | AI 只建议，不直接写业务事实 |
| 退款和异常单 | 结算不准 | 增加保护期、退款扣回、异常排除 |
| 不同门店提成规则差异大 | 实施成本上升 | 提供模板规则 + 白名单项目 + 封顶 |
| 数据不完整 | 无法证明价值 | 先补齐服务记录、推荐事件、订单归因 |

## 12. 最终建议

Ami_Aura 以数字员工方式参与门店运营是正确方向，也比单纯“卖设备”更有长期商业价值。

但商业化顺序必须是：

```text
先闭合业务事实
-> 再记录数字员工任务
-> 再证明推荐和转化
-> 再提供绩效看板
-> 最后收取可归因提成
```

短期建议采用：

```text
Ami_Core 订阅 + Ami_Aura 终端服务费 + AI 增值包
```

中期在样板店推出：

```text
固定月费 + 可归因绩效分成试点
```

长期成熟后升级为：

```text
数字员工席位费 + 岗位值守包 + 可归因绩效提成 + 供应链分成
```

这套方式既能把 Aura 从“智能终端”升级成“门店数字员工”，也能避免早期因为归因不清、提成争议和工时定义模糊导致商业化受阻。
