# 财务管理-提成规则统一配置改造方案

## 1. 背景与问题

当前提成规则存在两个配置入口：

1. 财务管理 - 提成规则：系统正式的提成规则库，写入 `CommissionRule`，订单、次卡核销会按规则生成 `CommissionRecord`。
2. 门店管理 - 项目管理 - 项目编辑：项目弹窗中也可以引用规则、新增项目例外、自定义项目提成，并直接创建或更新 `CommissionRule`。

这种设计会导致产品和交付层面的问题：

- 配置入口不唯一：运营人员不知道应该在财务管理配置，还是在项目管理配置。
- 责任边界不清：项目管理本应维护项目档案、价格、BOM；提成属于财务成本规则。
- 规则优先级难解释：项目侧创建的例外规则会使用更高优先级，后续项目毛利、员工提成、订单成本都依赖它，但来源不直观。
- 历史数据难排查：提成缺口无法快速判断是未配规则、项目侧例外覆盖、员工未绑定，还是历史订单未生成提成流水。
- 培训成本高：同一类规则在两个页面都能改，容易出现重复配置、错配员工、错配项目。

结论：提成规则应统一收敛到「财务管理 - 提成规则」配置；项目管理不再创建、编辑、删除提成规则。

## 2. 当前提成链路分析

### 2.1 数据模型

当前主数据结构如下：

- `CommissionRule`：提成规则。
  - 关键字段：门店、规则类型、适用范围、适用项目/商品/卡项、适用员工、员工等级、比例、固定金额、计算基数、优先级、状态。
  - 规则类型包括：项目、商品、办卡、充值、新客。
- 适用范围包括：全部、分类、指定对象；配置页统一使用下拉多选，支持一次选择多个分类、多个指定对象或多个员工。

- `CommissionRecord`：提成流水。
  - 关键字段：门店、员工、订单、订单明细、来源类型、来源记录、规则、提成类型、计算基数、比例、提成金额、状态、结算月份。
  - 经营利润中的项目成本、商品成本和员工绩效会读取这里的提成金额作为成本来源之一。

### 2.2 规则生成入口

当前存在两类入口：

| 入口 | 当前能力 | 问题 |
| --- | --- | --- |
| 财务管理 - 提成规则 | 创建、编辑、删除、启停提成规则 | 正式入口，应该保留 |
| 项目管理 - 项目编辑 - 提成配置 | 引用规则、新增项目例外、自定义项目规则 | 与财务入口重复，且会直接写入规则库 |

项目管理当前的行为不是“只展示关联规则”，而是会把项目例外写成 `targetType=specific`、`targetId=项目ID` 的项目提成规则，并提高优先级。这是当前链路混乱的核心。

### 2.3 订单生成提成

商品订单、项目订单创建后，后端会按订单明细生成提成流水：

1. 订单明细识别商品、项目、卡项、充值等类型。
2. 根据服务员工找到对应系统用户和员工等级。
3. 调用提成计算服务。
4. 按门店、类型、员工、等级、适用范围、优先级匹配有效规则。
5. 生成 `CommissionRecord`，状态当前为已确认。
6. 经营利润页面把该提成流水计入订单成本。

### 2.4 次卡核销生成提成

次卡核销时，如果核销记录能识别服务员工和项目，会按项目提成规则生成提成流水：

1. 次卡核销确认履约收入。
2. 根据核销项目匹配项目档案。
3. 根据服务员工找到系统用户和等级。
4. 按项目类型提成规则计算提成。
5. 生成来源为次卡核销的 `CommissionRecord`。
6. 项目毛利把次卡核销收入和对应提成成本纳入统计。

### 2.5 提成记录与成本同步

提成规则本身不直接等于订单成本。订单成本实际读取的是已经生成的 `CommissionRecord`：

- 规则用于“计算将来订单或核销的提成”。
- 流水用于“沉淀本次订单实际成本”。
- 修改历史流水时，应同步影响订单利润、项目毛利、商品毛利、员工绩效。
- 修改规则时，不应自动改历史流水，除非执行明确的历史重算或补算任务。

## 3. 统一后的产品原则

### 3.1 一个配置入口

所有提成规则只允许在「财务管理 - 提成规则」创建、编辑、启停、归档。

项目管理、商品管理、卡项管理只展示“当前命中的提成规则”，不直接维护提成。

### 3.2 项目管理只做只读提示

项目管理保留以下能力：

- 展示当前项目会命中的有效提成规则。
- 展示命中原因：全部项目、项目分类、指定项目、员工、等级、优先级。
- 当没有命中规则时，提示“未配置项目提成规则”。
- 提供跳转按钮：去财务管理配置。

项目管理移除以下能力：

- 新增项目例外。
- 从规则库引用并复制成项目规则。
- 自定义项目提成规则。
- 删除或修改提成规则。

### 3.3 财务管理承载规则全生命周期

「财务管理 - 提成规则」需要承担完整配置能力：

- 按规则类型配置：项目、商品、办卡、充值、新客。
- 按适用范围配置：全部、分类、指定对象。
- 按员工或等级配置：指定员工、员工等级、门店通用默认规则。
- 配置比例或固定金额。
- 配置计算基数：订单金额、服务费、毛利。
- 配置优先级。
- 支持启用、停用、归档。
- 支持规则命中预览。

## 4. 建议的规则匹配口径

### 4.1 基础匹配条件

规则必须同时满足：

- 同一门店。
- 同一提成类型。
- 状态为启用。
- 员工条件命中。
- 等级条件命中。
- 适用范围命中。

### 4.2 命中优先级

建议统一为以下顺序：

1. 优先级数值更高的规则优先生效。
2. 指定员工规则优先于等级规则。
3. 等级规则优先于门店通用规则。
4. 指定对象优先于分类。
5. 分类优先于全部。
6. 固定金额和比例同时存在时，固定金额优先。

当前代码已按优先级、员工、等级、适用范围进行排序选择，但产品页面需要把这个规则解释清楚，并在命中预览里显式展示。

多选配置的规则展开口径：

- 一个规则可在页面上选择多个项目、商品、卡项、分类或员工。
- 保存时建议后端按“规则组 + 明细范围”存储；阶段 1 如暂不改表结构，可由前端或后端拆成多条同名规则落库。
- 拆分后的每条规则仍遵守同一套命中优先级。
- 页面需展示为一组规则，避免运营人员看到大量重复规则。
- 后续编辑同一组规则时，应批量更新该组下的全部明细规则。

### 4.3 是否支持“门店通用规则”

当前后端匹配逻辑支持 `userId` 为空的通用规则，但创建和更新规则时要求必须绑定具体员工。

建议分两阶段处理：

- 阶段 1：先保持“规则必须绑定员工”，避免上线后产生不可控的大范围提成。
- 阶段 2：增加“门店通用规则”开关，允许没有员工时作为兜底规则；页面必须明确标识“未配置员工专属规则时生效”。

## 5. 页面改造方案

### 5.1 财务管理 - 提成规则

建议改成统一规则工作台：

- 顶部指标：
  - 启用规则数。
  - 停用规则数。
  - 项目规则数。
  - 商品规则数。
  - 卡项/充值规则数。
  - 存在冲突的规则数。

- 筛选区：
  - 规则类型。
  - 适用对象。
  - 员工。
  - 状态。
  - 关键词。

- 规则列表：
  - 规则名称。
  - 类型。
  - 适用范围。
  - 适用员工或等级。
  - 计算方式。
  - 优先级。
  - 状态。
  - 最近更新时间。
  - 操作：编辑、停用、归档、复制。

- 新增/编辑弹窗：
  - 类型选择后动态切换对象选择器。
  - 项目规则：下拉多选，支持全部项目、多个项目分类、多个指定项目。
  - 商品规则：下拉多选，支持全部商品、多个商品分类、多个指定商品。
  - 办卡规则：下拉多选，支持全部卡项、多个指定卡项。
  - 充值规则：下拉多选，支持全部充值；后续如有充值套餐，可扩展为多个指定充值套餐。
  - 员工选择：下拉多选，支持多个指定员工；阶段 2 支持多个员工等级和门店通用。
  - 计算方式：比例或固定金额。
  - 计算基数：实收金额、服务费、毛利。
  - 优先级：默认 0，高优先级需展示风险提示。

- 命中预览：
  - 选择一个项目或商品。
  - 选择一个员工。
  - 展示最终会命中的规则。
  - 展示未命中原因。

### 5.2 项目管理 - 项目编辑

项目编辑步骤建议调整为：

1. 基本信息。
2. BOM 配置。
3. 提成规则覆盖预览。

第三步只读展示：

- 当前项目命中的提成规则列表。
- 命中来源：全部项目、项目分类、指定项目。
- 适用员工。
- 比例或固定金额。
- 优先级。
- 状态。

页面按钮：

- 去财务管理配置提成规则。
- 刷新命中结果。

移除：

- 新增项目例外。
- 引用规则。
- 自定义规则。
- 删除规则。
- 直接保存项目提成。

### 5.3 提成明细

提成明细定位为“流水和成本校正”，不是规则配置。

建议能力：

- 查看订单、项目、商品、卡项来源。
- 修改员工。
- 修改比例或金额。
- 修改后同步影响订单成本和经营利润。
- 保留修改记录或备注，便于财务追溯。

不建议保留：

- 审核确认流程作为核心动作。当前用户诉求已明确提成不再需要审核确认，操作改为修改。

## 6. 数据和接口改造方案

### 6.1 保留的数据表

阶段 1 不需要新增表，继续使用：

- `CommissionRule`
- `CommissionRecord`

### 6.2 建议新增接口

建议新增规则预览接口：

`GET /commission/rules/preview`

参数：

- `storeId`
- `type`
- `itemId`
- `categoryId`
- `userId`
- `levelId`
- `sourceAmount`

返回：

- 是否命中。
- 命中的规则。
- 计算基数。
- 预计提成金额。
- 未命中原因。
- 被覆盖的候选规则。

用途：

- 项目管理只读预览。
- 财务规则编辑时实时校验。
- 排查项目毛利和商品毛利中的提成缺口。

### 6.3 需要废弃的项目侧写入

项目管理需停止调用：

- 新建项目后创建项目提成规则。
- 编辑项目时更新项目提成规则。
- 删除项目例外规则。

项目保存只处理：

- 项目基本信息。
- 项目 BOM。

### 6.4 历史项目例外规则处理

已从项目管理创建出来的历史规则，本质已经是 `CommissionRule`，不建议物理删除。

处理建议：

1. 在财务管理 - 提成规则中继续展示。
2. 对规则名称进行识别和规范，例如“项目专属：项目名 - 员工名”。
3. 若后续新增 `source` 字段，可把历史项目侧创建规则标记为 `project_legacy`。
4. 上线前输出一次历史规则审计清单：
   - 指定项目规则数。
   - 重复员工规则数。
   - 高优先级规则数。
   - 已停用但仍有历史流水的规则数。

## 7. 经营利润影响

统一提成配置后，对经营利润的影响如下：

- 项目毛利：
  - 收入来自项目订单和次卡核销。
  - 耗材成本来自项目 BOM 或实际耗材流水。
  - 提成成本来自 `CommissionRecord`。
  - 若未生成提成流水，应展示为“提成记录缺失”，并可通过规则预览判断原因。

- 商品毛利：
  - 收入来自商品订单。
  - 商品成本来自订单快照、出库/库存成本或商品档案成本。
  - 提成成本来自 `CommissionRecord`。
  - 若商品已售但未生成提成流水，应展示为“提成记录缺失”。

- 员工绩效：
  - 服务收入、贡献毛利、提成都应基于同一批提成流水。
  - 修改提成流水后，员工绩效和经营利润应同步变化。

## 8. 分阶段开发计划

### 阶段 1：入口收敛

目标：停止项目管理直接配置提成。

开发内容：

1. 项目编辑弹窗移除“新增项目例外、引用规则、自定义规则、保存项目提成”。
2. 项目编辑弹窗改为“提成规则覆盖预览”。
3. 项目保存接口只保存基本信息和 BOM，不再创建或更新 `CommissionRule`。
4. 财务管理 - 提成规则补充项目、商品、卡项对象选择器，避免手填对象 ID。
5. 对象选择器和员工选择器统一改为下拉多选。
6. 文案统一为“提成规则统一在财务管理配置”。

验收标准：

- 项目管理无法新增、编辑、删除提成规则。
- 财务管理仍可新增、编辑、启停提成规则。
- 财务管理新增规则时，可一次选择多个项目、商品、卡项或员工。
- 新项目订单仍可按财务规则生成提成流水。
- 次卡核销仍可按财务规则生成项目提成流水。

### 阶段 2：命中预览和缺口解释

目标：让运营人员知道为什么命中或没命中规则。

开发内容：

1. 新增提成规则命中预览接口。
2. 财务规则编辑页支持实时预览。
3. 项目管理展示当前项目命中的规则。
4. 项目毛利、商品毛利的“提成记录缺失”支持查看原因。
5. 提成明细支持修改员工、比例、金额，并同步经营利润成本。

验收标准：

- 选择项目和员工后，可以看到最终命中的规则。
- 没有命中时能显示原因，例如未绑定员工、规则停用、规则类型不匹配、范围不匹配。
- 修改提成明细后，订单利润、项目毛利、商品毛利同步变化。

### 阶段 3：历史数据治理

目标：解决历史规则和历史订单缺口。

开发内容：

1. 输出历史项目例外规则审计报表。
2. 标记高优先级、重复、已停用、有流水的规则。
3. 提供历史订单提成补算脚本。
4. 补算前支持预览影响范围。
5. 补算后记录操作人、时间、影响订单数、影响金额。

验收标准：

- 能识别历史规则来源和风险。
- 能补齐符合条件但缺失提成流水的历史订单。
- 补算不会覆盖人工修改过的提成流水。
- 经营利润缺口数量下降，并能解释剩余缺口原因。

## 9. 推荐落地顺序

建议先做阶段 1，因为它能马上降低配置混乱：

1. 先把项目管理改成只读预览，不再写提成规则。
2. 再增强财务管理提成规则的对象选择和规则说明。
3. 然后做命中预览和缺口解释。
4. 最后处理历史数据补算。

不建议先做历史补算。当前规则入口未收敛前，补算出来的数据仍可能被后续项目侧例外规则干扰。

## 10. 待确认产品决策

1. 阶段 1 是否继续要求每条规则必须绑定具体员工。
   - 建议：继续要求，避免大范围误发提成。

2. 是否允许门店通用兜底规则。
   - 建议：阶段 2 再支持，并在页面明确“仅在没有员工专属规则时生效”。

3. 项目管理是否保留第三步“提成规则覆盖预览”。
   - 建议：保留只读预览，方便项目负责人确认配置完整性。

4. 历史项目例外规则是否需要改名或标记来源。
   - 建议：先审计，再批量规范命名，不直接删除。

5. 修改历史提成流水是否需要记录调整日志。
   - 建议：需要。提成金额直接影响财务成本和员工结算，必须可追溯。
