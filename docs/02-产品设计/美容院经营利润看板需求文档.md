# 美容院经营利润看板需求文档

版本：v1.0  
日期：2026-06-18  
适用范围：Ami_Core 管理端、server-v2、Ami Aura Lite 智能终端经营摘要、后续 AI 问数能力  
关联调研：`docs/01-市场调研/美容院成本与盈利明细调研报告-2026-06-18.md`

---

## 1. 背景与问题

美容院经营表面上看的是销售额、办卡金额和到店客流，但真实利润会被人工、房租、耗材、营销、折旧、赠送权益和会员卡履约成本持续消耗。单纯看“今天收了多少钱”容易造成三个误判：

1. **把预收当利润**  
   办卡、充值带来现金流，但未来仍要履约。若不看消课、剩余权益、赠送项目和退款风险，门店会高估利润。

2. **项目赚不赚钱看不清**  
   同一个项目需要扣耗材、占用美容师工时、产生提成，仪器类还要计入折旧。没有项目毛利，门店容易把低价长时项目做成主收入。

3. **老板不知道问题出在哪里**  
   月收入下降可能来自新客少、复购慢、客单低、员工人效低、耗材异常、营销 ROI 差或会员卡消课不足。没有统一看板，就只能凭感觉经营。

因此需要建设“经营利润看板”，让店长和老板能在一个页面看清：门店现在赚不赚钱、钱赚在哪里、亏在哪里、接下来该处理什么。

## 2. 产品目标

### 2.1 总目标

建立一套面向中小美容院的轻量利润管理能力，帮助门店从“看销售额”升级为“看经营质量”。

### 2.2 阶段目标

| 阶段 | 目标 | 交付重点 |
| --- | --- | --- |
| MVP | 看清收入、成本、毛利、净利的基本盘 | 单店利润总览、项目毛利、会员卡履约、员工人效、成本录入 |
| V1.1 | 找到利润异常原因 | 毛利异常、耗材异常、营销 ROI、沉睡客户损失、预收履约风险 |
| V1.2 | 形成经营动作闭环 | 自动生成改进建议、推送到终端/任务、支持 AI 问数 |

### 2.3 非目标

MVP 阶段不做成完整财务系统，不替代会计软件，不处理复杂税务申报、工资个税、总账凭证、供应商对账、资产折旧账套。首期只做经营决策口径，保证老板能快速判断门店经营质量。

## 3. 用户角色与使用场景

| 角色 | 关注问题 | 高频动作 |
| --- | --- | --- |
| 老板/投资人 | 这个月赚了多少钱？哪家店最健康？哪些成本失控？ | 看利润总览、看门店排行、看异常提醒 |
| 店长 | 本店哪里拖累利润？哪些客户/项目要跟进？ | 看项目毛利、员工人效、会员卡履约、营销效果 |
| 财务/运营 | 数据口径是否准确？成本是否录入完整？ | 录入固定成本、校准成本分类、导出报表 |
| 库存/采购 | 哪些项目耗材成本过高？哪些产品消耗异常？ | 查看耗材成本、库存流水、低库存和异常消耗 |
| 美容师 | 我的服务收入和复购表现如何？ | 在终端看个人摘要、跟进客户、完成服务记录 |

典型场景：

1. 老板每晚看今日经营摘要，判断今天是“现金流好”还是“真实利润好”。
2. 店长每周复盘项目，把高收入低毛利项目调整为引流项目，主推高贡献项目。
3. 财务每月录入房租、工资、营销费、折旧等固定成本，生成月度利润表。
4. 库存人员发现某项目耗材成本偏离标准，追查是 BOM 设置、员工用量还是库存记录问题。
5. 美容师在 Ami Aura Lite 查看本人服务收入、复购提醒和待跟进客户。

## 4. 核心指标口径

### 4.1 收入口径

| 指标 | 定义 | 首期数据来源 |
| --- | --- | --- |
| 实收金额 | 已完成订单实际收款金额 | `ProductOrder.totalAmount`、支付记录 |
| 服务收入 | 项目/服务类订单收入 | `OrderItem.itemType = project/service` |
| 产品收入 | 商品零售订单收入 | `OrderItem.itemType = product` |
| 办卡收入 | 次卡/疗程卡销售收入 | 卡项订单、`CustomerCard` 创建记录 |
| 充值收入 | 储值充值现金收入 | `CustomerBalanceTransaction.type = recharge` |
| 消课收入 | 会员卡核销对应的履约收入 | `CardUsageRecord` + 卡项单次价值 |
| 退款金额 | 已完成退款金额 | `RefundRecord` |
| 净收入 | 实收金额 - 退款金额 | 聚合计算 |

关键原则：

- 办卡/充值是现金收入，不直接等于利润。
- 消课收入用于判断真实履约价值。
- 报表必须同时显示“现金收入”和“经营收入/履约收入”，避免误判。

### 4.2 成本口径

| 成本类型 | 定义 | MVP 采集方式 |
| --- | --- | --- |
| 耗材成本 | 服务项目消耗的产品、一次性用品、院装产品 | 项目 BOM + `StockMovement` |
| 商品成本 | 零售商品出库成本 | `Product.costPrice` + 销售数量 |
| 人工成本 | 美容师工资、提成、店长/前台工资 | MVP 支持月度手工录入 + 后续提成记录 |
| 房租物业 | 房租、物业、管理费 | 月度固定成本录入 |
| 营销成本 | 团购、达人、广告、活动权益成本 | 活动成本录入 + 营销归因 |
| 水电杂费 | 水电、布草、维修、办公耗材 | 月度成本录入 |
| 折旧摊销 | 装修、仪器、设备按月摊销 | 固定资产简化录入 |
| 赠送权益成本 | 办卡赠送项目、赠送余额、赠送产品 | 订单 payload、卡项/储值赠送记录 |

### 4.3 利润口径

```text
现金收入 = 订单实收 + 办卡实收 + 充值实收 - 退款

经营收入 = 服务消课收入 + 单次服务收入 + 产品销售收入

毛利 = 经营收入 - 耗材成本 - 商品成本 - 直接服务提成

经营利润 = 毛利 - 房租物业 - 固定工资 - 营销成本 - 水电杂费 - 折旧摊销 - 其他经营费用

净利率 = 经营利润 / 经营收入
```

MVP 展示时必须注明口径：

- 若人工成本未录入完整，显示“利润为预估值”。
- 若项目未维护 BOM，项目耗材成本显示“未配置”，不强行按 0 成本计算。
- 若卡项单次价值缺失，消课收入按卡项售价 / 总次数估算，并标注“估算”。

## 5. 功能范围

### 5.1 页面一：利润总览

目标：老板 30 秒看懂门店是否健康。

核心模块：

1. 顶部筛选
   - 时间：今日、本周、本月、本季度、自定义。
   - 门店：单店、多店汇总。
   - 口径：现金流口径、经营收入口径。

2. 核心指标卡
   - 现金收入。
   - 经营收入。
   - 毛利。
   - 经营利润。
   - 净利率。
   - 客单价。
   - 到店客户数。
   - 消课率。

3. 收入结构图
   - 服务收入。
   - 产品收入。
   - 办卡收入。
   - 充值收入。
   - 消课收入。

4. 成本结构图
   - 耗材/商品成本。
   - 人工成本。
   - 房租物业。
   - 营销成本。
   - 折旧摊销。
   - 其他费用。

5. 利润趋势
   - 日/周/月收入趋势。
   - 毛利趋势。
   - 净利趋势。

6. 异常提醒
   - 净利率低于目标。
   - 房租 + 人工占比过高。
   - 项目毛利异常偏低。
   - 会员卡消课过慢。
   - 耗材成本异常升高。

### 5.2 页面二：项目毛利分析

目标：看清每个项目到底赚不赚钱。

列表字段：

| 字段 | 说明 |
| --- | --- |
| 项目名称 | 来自 `Project.name` |
| 项目分类 | 面部、身体、仪器、抗衰等 |
| 标准售价 | `Project.price` |
| 实际成交均价 | 订单明细实际收入 / 数量 |
| 服务次数 | 单次服务 + 卡项核销次数 |
| 服务收入 | 项目订单收入 + 消课收入 |
| 标准耗材成本 | 项目 BOM 标准成本 |
| 实际耗材成本 | `StockMovement` 中项目相关消耗 |
| 美容师提成 | 提成记录或规则估算 |
| 平均服务时长 | `Project.duration` 或预约/服务任务实际时长 |
| 项目贡献毛利 | 服务收入 - 耗材成本 - 直接提成 |
| 毛利率 | 贡献毛利 / 服务收入 |
| 状态 | 高利润、引流、需优化、成本缺失 |

操作：

- 查看项目毛利详情。
- 查看耗材明细。
- 跳转编辑项目 BOM。
- 标记为引流项目或主推项目。
- 导出项目毛利表。

规则建议：

- 毛利率低于 40%：提示“需优化”。
- 服务次数高但毛利低：提示“可能占用技师和床位资源”。
- 无 BOM 且服务次数高：提示“优先维护耗材标准”。

### 5.3 页面三：会员卡与预收履约

目标：区分“现金收进来”和“未来还要服务”的压力。

核心指标：

| 指标 | 定义 |
| --- | --- |
| 期初预收余额 | 上期剩余未履约权益 |
| 本期新增预收 | 办卡、充值、续卡实收 |
| 本期消课价值 | 核销次数对应的履约收入 |
| 本期退款 | 卡项/储值退款 |
| 期末预收余额 | 未履约权益余额 |
| 消课率 | 本期消课价值 / 可履约权益 |
| 临期卡数量 | 即将到期且仍有余额/次数 |
| 高风险卡 | 余额高、长期未消课、快到期或纠纷风险 |

列表字段：

- 客户姓名。
- 卡项名称。
- 购买时间。
- 到期时间。
- 总次数/剩余次数。
- 估算剩余金额。
- 最近一次核销时间。
- 负责美容师。
- 风险状态。

风险规则：

- 30 天未消课且剩余次数较多：提醒跟进。
- 15 天内到期且仍有剩余：提醒预约。
- 赠送权益占比过高：提醒检查活动规则。
- 单客户预收余额过高：提醒履约和退款风险。

### 5.4 页面四：员工人效与提成

目标：看清美容师对收入、复购和利润的贡献。

指标：

| 指标 | 定义 |
| --- | --- |
| 服务收入 | 该美容师服务订单与核销项目收入 |
| 服务次数 | 订单服务次数 + 卡项核销次数 |
| 客户数 | 服务过的去重客户数 |
| 客单价 | 服务收入 / 服务客户数或服务次数 |
| 续卡/办卡贡献 | 关联美容师的卡项销售 |
| 复购贡献 | 服务客户后 30/60 天复购 |
| 直接提成 | 提成记录或规则估算 |
| 贡献毛利 | 服务收入 - 耗材成本 - 直接提成 |

MVP 不要求完整薪酬系统，但需要支持：

- 月度人工成本手工录入。
- 按美容师录入基础工资/提成总额。
- 后续接入 `CommissionRecord`、`CommissionSettlement` 做自动化。

### 5.5 页面五：成本录入与配置

目标：让非财务人员也能维护利润看板需要的最小成本数据。

成本配置项：

| 配置 | 录入频率 | 说明 |
| --- | --- | --- |
| 房租物业 | 月 | 可设置固定金额，按月自动生成 |
| 基础工资 | 月 | 按员工或总额录入 |
| 营销成本 | 活动/日/月 | 可关联营销活动 |
| 水电杂费 | 月 | 简化录入 |
| 折旧摊销 | 月 | 装修、仪器、设备按月金额 |
| 其他费用 | 月 | 支持备注 |
| 项目成本目标 | 一次配置 | 项目目标毛利率、目标耗材成本 |

交互要求：

- 支持“复制上月成本”。
- 支持按门店录入。
- 支持成本分类必填，备注选填。
- 成本未录入时，看板显示缺口提醒。
- 支持导入 Excel，但首期不是必须。

### 5.6 页面六：经营建议

目标：从数据变成行动。

建议类型：

| 建议类型 | 触发条件 | 示例动作 |
| --- | --- | --- |
| 增收建议 | 复购低、沉睡客户多、项目转化低 | 推送复购活动、生成客户清单 |
| 降本建议 | 耗材成本高、库存异常、人工占比高 | 检查 BOM、调整排班、盘点异常产品 |
| 履约建议 | 临期卡多、消课慢 | 自动生成预约提醒任务 |
| 项目建议 | 高销量低毛利、低销量高毛利 | 调整主推项目、优化套餐结构 |
| 员工建议 | 人效差异大 | 培训、排班、客户分配建议 |

首期建议可以规则化生成，不依赖大模型。后续接入 AI 时，AI 只基于看板事实解释原因和生成话术，不直接创造经营数据。

## 6. 信息架构与导航

建议在管理端新增一级/二级入口：

```text
经营分析
  - 利润看板
  - 项目毛利
  - 会员卡履约
  - 员工人效
  - 成本配置
```

若不新增一级菜单，也可放在“智能营销”或“仪表盘”下，但从产品心智看更建议独立为“经营分析”，后续可承接 AI 问数、经营日报、多店对比。

Ami Aura Lite 智能终端只展示轻量摘要：

- 今日现金收入。
- 今日服务/核销次数。
- 今日低库存与耗材异常。
- 本人服务收入和待跟进客户。
- 店长角色可看今日利润预估和异常提醒。

## 7. 数据来源与现有基础

### 7.1 现有可复用数据

| 业务对象 | 现有模型/来源 | 可支持指标 |
| --- | --- | --- |
| 门店 | `Store` | 门店筛选、多店汇总 |
| 项目 | `Project` | 项目售价、时长、分类、状态 |
| 项目耗材 | `ProjectBomItem` | 标准耗材成本、项目 BOM |
| 商品 | `Product` | 成本价、零售价、库存、安全库存 |
| 库存流水 | `StockMovement` | 实际耗材、商品出库、库存异常 |
| 订单 | `ProductOrder` | 实收、退款前收入、支付方式、订单来源 |
| 订单明细 | `OrderItem` | 产品/项目收入、数量、美容师归属 |
| 卡项 | `Card` | 卡项售价、总次数、适用项目 |
| 客户卡 | `CustomerCard` | 剩余次数、到期、状态 |
| 核销记录 | `CardUsageRecord` | 消课次数、项目、美容师、时间 |
| 美容师 | `Beautician` | 员工维度、人效分析 |
| 储值账户 | `CustomerBalanceAccount` | 储值余额、赠送余额 |
| 储值流水 | `CustomerBalanceTransaction` | 充值、消费、余额变化 |
| 提成 | `CommissionRecord`、`CommissionSettlement` | 后续自动提成和人工成本 |

### 7.2 需要新增的数据能力

MVP 建议新增或补齐以下后端能力：

1. **经营成本表**

```text
OperatingCost
- id
- storeId
- periodMonth
- costDate
- category
- amount
- allocationType
- relatedCampaignId
- relatedEmployeeId
- remark
- createdBy
- createdAt
- updatedAt
```

成本分类建议：

```text
rent, salary, commission, marketing, utilities, depreciation, supplies_adjustment, other
```

2. **利润看板聚合服务**

不建议前端直接拉全量订单再计算。后端应提供聚合 API，按门店、时间、口径返回。

3. **项目毛利聚合服务**

按项目聚合订单、核销、BOM、库存流水、提成，返回可解释的毛利明细。

4. **预收履约聚合服务**

按卡项和客户统计剩余次数、消课速度、临期风险、估算剩余权益。

## 8. API 设计建议

接口统一放在 `server-v2`，前端走 `src/api/real/*` 主线，不新增 mock 主业务链路。

### 8.1 利润总览

```http
GET /api/operation-profit/overview?storeId=1&from=2026-06-01&to=2026-06-30&basis=operating
```

返回示例：

```json
{
  "period": {
    "from": "2026-06-01",
    "to": "2026-06-30"
  },
  "basis": "operating",
  "summary": {
    "cashIncome": 186000,
    "operatingIncome": 152000,
    "grossProfit": 96500,
    "operatingProfit": 21800,
    "grossMargin": 0.635,
    "netMargin": 0.143,
    "customerCount": 238,
    "avgTicket": 638.66,
    "cardConsumptionRate": 0.31
  },
  "incomeBreakdown": [],
  "costBreakdown": [],
  "trend": [],
  "alerts": []
}
```

### 8.2 项目毛利

```http
GET /api/operation-profit/project-margins?storeId=1&from=2026-06-01&to=2026-06-30
```

核心返回字段：

- `projectId`
- `projectName`
- `serviceCount`
- `serviceIncome`
- `standardMaterialCost`
- `actualMaterialCost`
- `commissionCost`
- `contributionProfit`
- `marginRate`
- `status`
- `missingCostReasons`

### 8.3 会员卡履约

```http
GET /api/operation-profit/prepaid-liabilities?storeId=1&riskOnly=false&page=1&pageSize=20
```

核心返回字段：

- `customerId`
- `customerName`
- `cardName`
- `remainingTimes`
- `estimatedRemainingValue`
- `expiryDate`
- `lastUsedAt`
- `riskLevel`
- `riskReasons`

## 9. 页面交互与状态

### 9.1 加载状态

经营看板属于高价值页面，不能长时间空白。页面必须具备：

- 顶部指标 skeleton。
- 图表区域 loading。
- 明细表格 loading。
- 分区级错误提示。
- 数据缺口提醒。

### 9.2 空状态

| 场景 | 展示 |
| --- | --- |
| 无订单 | 提示先产生订单或选择更长时间 |
| 无成本配置 | 提示录入本月固定成本，并提供入口 |
| 无项目 BOM | 提示优先配置高频项目耗材 |
| 无核销记录 | 提示会员卡履约数据不足 |
| 无权限 | 显示无权限说明，不暴露财务数据 |

### 9.3 数据可信提示

每个关键指标应展示可信状态：

- `完整`：订单、成本、BOM、提成都有数据。
- `部分估算`：缺少提成或卡项单次价值，系统按规则估算。
- `缺成本`：缺少人工、房租或 BOM。
- `不可计算`：核心数据缺失。

## 10. 权限与安全

建议新增权限码：

```text
core:operation-profit:view
core:operation-profit:export
core:operation-cost:view
core:operation-cost:manage
core:project-margin:view
core:prepaid-liability:view
core:beautician-performance:view
```

权限规则：

- 老板/超级管理员：可看全部利润、成本、门店对比。
- 店长：默认看本门店经营利润和员工人效。
- 财务：可看成本配置、收入、退款、导出。
- 美容师：只能看本人服务和提成摘要，不看全店利润。
- 前台：默认不看净利和成本，只看业务所需的卡项履约状态。

数据导出需要更高权限，并记录操作日志。

## 11. 规则与预警

MVP 预警规则建议：

| 预警 | 条件 | 级别 | 推荐动作 |
| --- | --- | --- | --- |
| 净利率偏低 | 经营净利率 < 8% | 高 | 查看成本结构和项目毛利 |
| 人工房租占比过高 | 人工 + 房租 > 55% 经营收入 | 高 | 调整排班、提升客单或复购 |
| 项目毛利异常 | 项目毛利率 < 40% 且服务次数高 | 中 | 检查定价、耗材、时长和提成 |
| BOM 缺失 | 高频项目无 BOM | 中 | 维护项目耗材标准 |
| 耗材异常 | 实际耗材成本 > 标准成本 130% | 中 | 检查库存流水或服务用量 |
| 消课过慢 | 30 天未核销且剩余次数多 | 中 | 生成客户跟进任务 |
| 临期权益 | 15 天内到期且仍有剩余 | 中 | 提醒预约 |
| 现金好但履约压力大 | 本期预收增长高、消课率低 | 高 | 控制赠送权益，安排履约 |

## 12. MVP 验收标准

### 12.1 产品验收

1. 老板能在利润总览看到本月现金收入、经营收入、毛利、经营利润、净利率。
2. 看板能明确提示哪些指标是估算、哪些成本缺失。
3. 店长能看到项目毛利列表，并识别高收入低毛利项目。
4. 店长能看到会员卡临期、长期未消课、高剩余次数客户。
5. 财务/运营能录入本月房租、工资、营销、水电、折旧等成本。
6. 美容师维度能看到服务收入、服务次数、客户数、贡献毛利的基础统计。
7. 页面有 loading、empty、error、权限不足状态。

### 12.2 技术验收

1. 聚合计算在后端完成，前端不拉全量明细做核心报表。
2. 所有查询必须带 `storeId` 和时间范围。
3. 金额计算使用 Decimal 或后端统一金额处理，避免浮点误差。
4. 缺失 BOM、缺失成本、缺失提成时不按 0 静默计算，必须返回 `missingCostReasons`。
5. 分页接口返回 `{ items, total, page, pageSize }`。
6. API 错误格式遵守 `{ message, code?, status?, details? }`。
7. 权限码接入现有 PermissionGuard、菜单权限和 `usePermission`。
8. 关键聚合 API 至少补服务层单测和前端 API facade 测试。

## 14. 风险与注意事项

1. **数据口径风险**  
   现金收入、经营收入、消课收入必须分开，否则会误导老板。

2. **成本缺失风险**  
   中小门店不会一开始就录完整成本，所以系统必须显示可信度，而不是给出假精确利润。

3. **BOM 维护成本**  
   项目 BOM 是项目毛利的关键，但不能要求一次性全量维护。应优先引导维护高频项目。

4. **预收卡合规风险**  
   预收、赠送、退款、到期规则可能涉及本地监管和消费者权益，产品文案应避免诱导过度办卡。

5. **员工敏感信息**  
   员工人效和提成属于敏感数据，需要严格权限控制。

6. **不要把 AI 当事实源**  
   AI 只能解释和建议，所有金额、客户、订单、成本、利润都必须来自系统聚合数据。
