## 1. 一句话定位

围绕“客户全生命周期价值、服务体验和营销增长”建设一个小而闭环的业务本体，把客户、会员卡、订单、服务项目、护理周期、营销触达、权益、库存消耗和经营结果统一成 AI 能理解、能解释、能触发动作的业务语义层。

它不是新建一套大而全的数据中台，也不是重做推荐算法，而是在现有客户画像、智能推荐、自动营销、活动、权益资产、终端跟进和库存消耗能力之上，补齐一层稳定的业务对象、关系、事件、规则和动作定义。

目标是让系统能回答并执行三类问题：

- 这个客户现在处于什么生命周期阶段，未来价值和流失风险如何？
- 她下一次最适合做什么服务、用什么权益、由谁在什么渠道触达？
- 本次服务和营销是否真的带来了预约、核销、成交、复购、库存消耗和 LTV 提升？

## 2. 为什么现在需要小本体

当前系统已经有客户画像、预测快照、智能推荐、营销活动、自动触达、权益资产、小程序行为和终端增长候选等基础能力，但这些能力仍容易出现四个问题：

1. 语义不统一：客户价值、客户阶段、护理周期、推荐原因、权益、触达结果在不同模块里各自表达，后续复盘困难。
2. 动作不连续：推荐卡可以给建议，但从推荐到活动、自动规则、顾问任务、终端跟进、服务核销之间仍需要很多人工判断。
3. 归因不完整：一次营销到底带来的是浏览、预约、到店、项目成交、卡项续费、商品购买还是库存消耗，缺少统一链路。
4. AI 上下文偏碎片：Agent 问数或经营建议如果只看表名、字段和文本片段，容易知道“发生了什么”，但不理解“为什么重要、下一步能做什么”。

小本体的价值是把这些业务概念固定下来，让产品、算法、后端、前端和 AI Agent 共享同一套业务语言。

## 3. 产品边界

### 3.1 本方案解决什么

- 定义客户生命周期、服务、营销三条主线的核心对象和关系。
- 定义客户从线索、新客、体验、会员、活跃、成长、预流失、沉睡到召回的状态机。
- 定义服务项目、会员卡、护理周期、权益资产、营销触达、库存消耗如何围绕客户形成闭环。
- 定义 AI Agent 能读取的上下文包、可解释推荐依据和可触发动作。
- 给出 P0/P1/P2 落地路径，避免一次性建设过重。

### 3.2 本方案暂不解决什么

- 不重做全量数据仓库。
- 不替代现有营销活动、自动营销规则、客户画像和推荐算法。
- 不在 P0 阶段要求完整知识图谱引擎或复杂图数据库。
- 不让 AI 直接修改客户资产、库存或订单；P0 阶段 AI 只生成建议、草稿和解释，由人确认后执行。

## 4. 小本体总览

本体由五层组成：

| 层级 | 作用 | 典型内容 |
| --- | --- | --- |
| 对象层 | 定义业务名词 | 客户、会员卡、订单、服务项目、权益、营销活动、触达、库存消耗 |
| 关系层 | 定义对象之间如何连接 | 客户拥有会员卡、客户购买订单、服务消耗库存、活动触达客户 |
| 事件层 | 记录状态变化和经营行为 | 预约、到店、服务完成、领券、核销、复购、沉睡、召回 |
| 规则层 | 判断生命周期、机会和风险 | 护理周期到期、卡项即将到期、高 LTV、触达疲劳、库存风险 |
| 动作层 | 让系统能执行下一步 | 创建活动、创建自动规则、生成顾问任务、下发终端提醒、生成权益草稿 |

建议命名为 `CustomerLifecycleOntology`，中文名“客户全生命周期服务营销小本体”。

## 5. 核心业务对象

### 5.1 客户对象 Customer

客户是本体中心节点，所有服务、营销和价值判断都围绕客户展开。

| 字段/属性 | 说明 | 用途 |
| --- | --- | --- |
| `customerId` | 客户唯一 ID | 全链路关联主键 |
| `storeId` | 所属门店 | 多门店隔离与对比 |
| `profile` | 基础资料、肤质、生日、偏好 | 个性化服务与权益匹配 |
| `lifecycleStage` | 生命周期阶段 | 运营策略入口 |
| `ltvTier` | 生命周期价值分层 | 权益强度、顾问优先级 |
| `churnRiskLevel` | 流失风险 | 唤醒、挽回、顾问跟进 |
| `repurchaseWindow` | 复购窗口 | 护理周期和复购提醒 |
| `promotionSensitivity` | 促销敏感度 | 权益类型选择 |
| `preferredChannels` | 偏好渠道 | 小程序、企微、短信、电话、终端 |
| `touchFatigueScore` | 触达疲劳度 | 降低打扰 |
| `servicePreference` | 项目/服务偏好 | 推荐项目与护理方案 |
| `assetSummary` | 卡项、券、积分、余额概览 | 续费、核销、权益提醒 |

### 5.2 生命周期阶段 LifecycleStage

客户生命周期建议采用可解释、可运营的阶段，不只用高低分。

| 阶段 | 定义 | 核心目标 | 典型动作 |
| --- | --- | --- | --- |
| `lead` 线索 | 已有联系方式或小程序访问，但未到店成交 | 建档与预约 | 新客体验券、顾问欢迎任务 |
| `new_customer` 新客 | 首次到店或首次成交后 0-30 天 | 二次到店 | 新客回访、护理建议、二次项目权益 |
| `trial` 体验客 | 做过低价体验或单次项目，未办卡/未复购 | 转会员/疗程 | 项目升级、卡项试算、顾问跟进 |
| `member` 会员 | 有有效会员卡、储值、次卡或疗程 | 提升使用率 | 卡项核销提醒、周期护理 |
| `active` 活跃客 | 近周期持续到店/消费 | 提升客单与连带 | 高毛利项目、商品搭配 |
| `growth` 成长客 | LTV、频次或品类扩展潜力高 | 扩品类/升阶 | 项目组合、VIP 权益 |
| `at_risk` 预流失 | 超过护理周期或复购窗口，风险上升 | 防流失 | 个性化唤醒、顾问关怀 |
| `dormant` 沉睡客 | 长时间未到店或未消费 | 召回 | 分层召回活动、低成本权益 |
| `lost` 流失客 | 长期无互动且多次触达无响应 | 低频维护 | 品牌内容、重大活动再触达 |

阶段不是互斥标签的全部表达。一个客户可以同时是 `member + at_risk + high_ltv`，但主阶段用于运营入口，辅助标签用于推荐排序和动作过滤。

### 5.3 服务对象 Service

服务对象连接客户价值、护理周期、员工能力和库存消耗。

| 对象 | 说明 | 关键属性 |
| --- | --- | --- |
| `ServiceItem` 服务项目 | 可预约、可核销、可推荐的项目 | 项目类型、功效、护理周期、标准时长、价格、毛利 |
| `ServiceSession` 服务履约 | 一次真实服务过程 | 客户、项目、员工、时间、核销卡项、订单 |
| `CarePlan` 护理方案 | 面向客户的阶段性护理计划 | 目标、周期、项目组合、预计次数 |
| `CareCycle` 护理周期 | 项目或客户的下次到店窗口 | 周期天数、提前提醒天数、逾期天数 |
| `ServiceOutcome` 服务结果 | 服务后的反馈和效果 | 满意度、皮肤改善、顾问备注、复购意向 |
| `ServiceConsumption` 项目耗材 | 服务项目消耗的商品/耗材 | 商品、标准数量、实际数量、库存流水 |

### 5.4 会员资产对象 CustomerAsset

会员资产决定客户权益、续费机会和服务承接。

| 对象 | 说明 | 典型机会 |
| --- | --- | --- |
| `MemberCard` 会员卡 | 储值卡、折扣卡、权益卡 | 余额不足、续费、升卡 |
| `PackageCard` 次卡/疗程卡 | 项目次数或套餐权益 | 剩余次数提醒、到期提醒、转疗程 |
| `Coupon` 优惠券 | 已领取或可发放权益 | 到期提醒、领券未核销 |
| `PromotionAsset` 权益资产 | 可复用的权益模板 | 推荐匹配、活动/规则复用 |
| `Points` 积分 | 客户忠诚资产 | 积分兑换、会员关怀 |

### 5.5 营销对象 Marketing

营销对象把“推荐”变成“可投放、可归因、可复盘”的动作。

| 对象 | 说明 | 关键属性 |
| --- | --- | --- |
| `RecommendationCard` 推荐卡 | 系统生成的经营机会 | 人群、策略、权益、推荐项目、预计收益、证据 |
| `MarketingActivity` 单次活动 | 有主题、周期和落地页的活动 | 活动主题、人群、权益、渠道、预算、状态 |
| `AutomationRule` 自动规则 | 按事件或状态持续触发 | 触发条件、频控、动作、退出条件 |
| `TouchTask` 触达任务 | 对客户的一次触达 | 渠道、话术、执行人、状态、结果 |
| `CampaignAudience` 活动人群快照 | 被推荐或投放的人群 | 客户清单、筛选条件、预测批次 |
| `AttributionEvent` 归因事件 | 投放后的结果回流 | 浏览、点击、预约、核销、订单、复购 |

## 6. 核心关系定义

小本体最重要的不是对象数量，而是关系是否能支撑业务闭环。

| 关系 | 说明 | 例子 |
| --- | --- | --- |
| `Customer owns CustomerAsset` | 客户拥有会员资产 | 客户 A 有一张剩余 2 次的补水疗程卡 |
| `Customer purchased Order` | 客户产生订单 | 客户 A 购买补水项目和家居护理商品 |
| `Order includes ServiceItem/Product` | 订单包含项目或商品 | 订单包含小气泡项目和面膜 |
| `ServiceSession fulfills Order` | 服务履约关联订单 | 订单支付后完成一次到店服务 |
| `ServiceItem has CareCycle` | 项目有护理周期 | 补水护理建议 28 天复购 |
| `ServiceSession consumes Inventory` | 服务消耗库存 | 一次护理消耗 1 片面膜和 20ml 精华 |
| `Customer matches RecommendationCard` | 客户命中推荐机会 | 客户 A 命中“补水周期到期” |
| `RecommendationCard uses PromotionAsset` | 推荐卡匹配权益 | 推荐使用“老客补水复购券” |
| `RecommendationCard creates Activity/Rule` | 推荐承接为动作 | 生成活动草稿或自动规则 |
| `Activity/Rule sends TouchTask` | 活动或规则生成触达 | 给客户 A 发小程序消息 |
| `TouchTask leads to AttributionEvent` | 触达产生归因 | 点击、预约、核销、下单 |
| `AttributionEvent updates CustomerProfile` | 结果回流画像 | 客户响应偏好、复购周期更新 |

## 7. 生命周期状态机

建议把生命周期阶段变更设计成可解释事件，而不是每天简单覆盖一个标签。

```text
lead
  -> new_customer: 首次到店 / 首次成交
new_customer
  -> trial: 仅完成体验项目，未办卡且未复购
  -> member: 办卡 / 购买疗程 / 储值
trial
  -> member: 办卡 / 购买疗程
  -> at_risk: 超过二次到店窗口未回访成功
member
  -> active: 周期内持续核销或消费
  -> growth: LTV、频次、品类扩展潜力提升
  -> at_risk: 超过护理周期或资产即将过期未使用
active
  -> growth: 高响应、高复购、高客单
  -> at_risk: 超过复购窗口未到店
at_risk
  -> active: 预约 / 到店 / 成交
  -> dormant: 长时间无响应
dormant
  -> active: 召回成功
  -> lost: 多轮触达无响应且长期无消费
```

每次阶段变更需要记录：

- 原阶段、目标阶段。
- 触发事件。
- 关键证据。
- 计算时间。
- 是否人工确认。
- 影响的推荐、活动或任务。

## 8. 事件体系

### 8.1 事件分类

| 事件类型 | 业务意义 | 示例 |
| --- | --- | --- |
| `profile_event` | 客户资料变化 | 新增肤质、生日、偏好、禁忌 |
| `service_event` | 服务过程变化 | 预约、到店、服务完成、爽约 |
| `asset_event` | 会员资产变化 | 办卡、核销、续费、余额不足、卡项到期 |
| `marketing_event` | 营销互动变化 | 触达、点击、领券、预约、核销 |
| `commerce_event` | 交易变化 | 下单、退款、复购、连带购买 |
| `inventory_event` | 库存消耗变化 | 服务扣耗材、商品销售、临期、缺货 |
| `feedback_event` | 服务反馈变化 | 满意度、投诉、顾问备注、效果反馈 |

### 8.2 P0 必备事件

P0 不追求事件大全，先保证能串起服务和营销闭环。

| 事件 | 触发来源 | 主要用途 |
| --- | --- | --- |
| `first_purchase` | 订单 | 新客入池 |
| `service_completed` | 服务履约/核销 | 更新护理周期、库存消耗、复购窗口 |
| `care_cycle_due` | 周期规则 | 触发周期护理推荐 |
| `card_expiring` | 会员卡/次卡 | 触发卡项到期提醒 |
| `coupon_claimed` | 小程序/活动页 | 判断权益兴趣 |
| `coupon_used` | 核销/订单 | 归因转化 |
| `touch_sent` | 活动/自动规则 | 记录触达 |
| `appointment_created` | 预约 | 归因预约 |
| `order_paid` | 收银/订单 | 归因成交 |
| `dormant_detected` | 客户画像规则 | 触发沉睡召回 |

## 9. 规则体系

### 9.1 客户价值规则

| 规则 | 说明 | 输出 |
| --- | --- | --- |
| LTV 分层 | 结合累计消费、近周期消费、频次、毛利贡献 | `ltvTier` |
| 成长潜力 | 结合服务频次、响应率、项目扩展空间 | `growthPotential` |
| 价格敏感 | 结合领券、核销、折扣订单比例 | `promotionSensitivity` |
| 服务偏好 | 结合项目历史、顾问备注、肤质、浏览行为 | `servicePreference` |
| 渠道偏好 | 结合小程序、短信、企微、电话响应 | `preferredChannels` |

### 9.2 服务周期规则

| 规则 | 说明 | 输出 |
| --- | --- | --- |
| 护理周期到期 | 上次服务后达到项目周期前后窗口 | `careCycleOpportunity` |
| 疗程使用不足 | 次卡/疗程卡剩余次数多但使用慢 | `packageUsageRisk` |
| 服务升级机会 | 当前项目可向高阶项目升级 | `upgradeOpportunity` |
| 商品带回家机会 | 项目关联居家护理商品，且客户未购买 | `homecareOpportunity` |

### 9.3 营销触达规则

| 规则 | 说明 | 输出 |
| --- | --- | --- |
| 触达频控 | N 天内触达次数超过阈值则降权或排除 | `touchAllowed` |
| 触达疲劳 | 多次未打开、未点击、未预约 | `touchFatigueScore` |
| 权益匹配 | 根据客户价值和敏感度匹配权益强度 | `recommendedOffer` |
| 执行方式判断 | 判断适合活动、自动规则或两者均可 | `preferredExecutionMode` |
| 归因窗口 | 不同触达动作对应不同转化观察期 | `attributionWindow` |

### 9.4 库存与产能规则

| 规则 | 说明 | 输出 |
| --- | --- | --- |
| 服务耗材充足 | 推荐项目对应耗材库存可支撑投放 | `inventoryReady` |
| 商品临期消化 | 临期商品与服务项目或客户偏好匹配 | `inventoryMarketingOpportunity` |
| 排班可承接 | 未来 1-7 天有可服务员工和时段 | `capacityReady` |
| 低峰填充 | 低峰时段产能空闲，可给高响应客户权益 | `offPeakOpportunity` |

## 10. 动作体系

动作是小本体区别于普通客户标签体系的关键。每个动作都要有权限、状态、执行人和回滚边界。

| 动作 | 执行对象 | 说明 | P0 策略 |
| --- | --- | --- | --- |
| `create_recommendation_card` | 系统 | 生成推荐卡 | 自动生成，人工查看 |
| `create_activity_draft` | 运营 | 从推荐卡生成活动草稿 | 人工确认后保存 |
| `create_automation_rule_draft` | 运营 | 从推荐卡生成自动规则草稿 | 人工确认后启用 |
| `create_touch_task` | 顾问/终端 | 生成客户跟进任务 | 可自动生成，人工执行 |
| `assign_promotion_asset` | 运营 | 匹配权益资产 | P0 只推荐，不自动发放 |
| `send_touch` | 活动/规则 | 发送触达 | 走现有触达能力 |
| `schedule_followup` | 顾问 | 安排回访 | 终端/管理端提醒 |
| `explain_customer_context` | AI Agent | 解释客户机会 | 只读解释 |
| `generate_campaign_copy` | AI Agent | 生成活动文案和话术 | 人工审核 |

P0 阶段不建议让 AI 直接执行扣库存、改卡项、退款、强制发券、批量群发等高风险动作。

### 11.1 面向 AI 的上下文包

### 11.2 Agent 可回答的问题

| 问题 | 本体提供的能力 |
| --- | --- |

## 15. 指标体系

### 15.1 客户价值指标

- LTV 分层准确率。
- 高价值客户复购率。
- 客户阶段迁移率：新客到会员、预流失到活跃、沉睡到召回。
- 会员资产使用率：次卡核销率、卡项过期前使用率、优惠券核销率。

### 15.2 服务指标

- 护理周期内复购率。
- 服务后 7/14/30 天预约率。
- 项目升级转化率。
- 服务关联商品连带率。
- 项目耗材标准命中率和实际消耗偏差。

### 15.3 营销指标

- 推荐卡采纳率。
- 推荐卡创建活动/规则比例。
- 触达打开率、点击率、预约率、核销率、成交率。
- 触达疲劳下降率。
- 活动 ROI 和推荐归因收入。

### 15.4 经营指标

- 复购收入提升。
- 沉睡客户召回收入。
- 卡项过期损失下降。
- 库存临期损耗下降。
- 顾问人均跟进效率。

## 16. 分阶段落地

### 16.1 P0：客户生命周期和服务营销闭环 MVP

目标：不用大改架构，先让“客户状态、服务周期、推荐动作、营销归因”跑通。

范围：

- 定义生命周期阶段和客户机会类型。
- 给客户详情/画像增加生命周期解释。
- 推荐卡绑定客户、项目、权益、执行方式和证据。
- 支持从推荐卡创建活动草稿、自动规则草稿、顾问任务。
- 打通护理周期到期、次卡到期、沉睡召回、领券未核销、浏览未预约五类场景。
- 归因至少覆盖触达、点击、预约、核销、订单。

验收：

- 任意推荐卡能说明目标客户、推荐原因、服务/权益、执行方式和风险控制。
- 推荐卡能承接到活动或自动规则，不需要运营重新填写核心字段。
- 客户详情能看到生命周期阶段和最近一次阶段变更证据。
- 投放后能看到从触达到订单的基础转化漏斗。

#### 16.1.1 P0 实际落地口径

本轮 P0 采用“轻量持久化”实现，不引入图数据库，也不重做推荐系统。

后端新增三类轻量对象：

| 对象 | 作用 | 说明 |
| --- | --- | --- |
| `CustomerLifecycleSnapshot` | 客户当前生命周期快照 | 记录阶段、LTV、流失风险、触达疲劳、资产摘要、服务偏好和证据 |
| `CustomerLifecycleEvent` | 生命周期阶段变化流水 | 记录阶段初始化/变更、来源和证据 |
| `CustomerOpportunity` | 客户级机会 | 记录护理周期、卡项到期、沉睡召回、领券未核销、浏览未预约等 P0 机会 |

新增接口：

| 接口 | 用途 |
| --- | --- |
| `POST /marketing/lifecycle/rebuild` | 手动重建当前门店客户生命周期快照和机会 |
| `GET /marketing/lifecycle/opportunities` | 查询客户机会，支持类型、优先级、状态、客户筛选 |
| `GET /marketing/lifecycle/customers/:id` | 查询单客户生命周期上下文包 |

已接入现有链路：

- `runPredictions` 完成后自动重建生命周期快照和机会。
- `GET /marketing/recommendations` 优先合并生命周期机会推荐卡，再保留原有推荐卡。
- `GET /customers/:id/profile` 返回 `lifecycle`，客户详情可展示阶段、机会和证据。
- `GET /customers/profile-analytics/prediction` 返回生命周期阶段、机会标签和证据摘要。
- 智能推荐卡继续复用现有活动草稿、自动规则草稿、终端跟进任务承接能力。

P0 明确不做：

- 不自动发券、不自动群发、不自动改卡项、不自动扣库存、不自动创建订单。
- 不新增独立本体管理页面。
- 不做项目级精确护理周期、耗材标准库和库存/产能校验，这些进入 P1。

### 16.2 P1：服务项目、会员资产和库存消耗联动

目标：让推荐不只看客户，还看服务承接能力和库存经营压力。

范围：

- 服务项目补齐护理周期、适用人群、关联商品、耗材标准。
- 服务完成后自动生成下次护理机会。
- 项目关联商品支持居家护理推荐。
- 库存临期、库存充足、低峰产能进入推荐排序。
- 终端今日跟进直接消费后端推荐结果。

验收：

- 服务完成后能自动生成下一次护理周期机会。
- 推荐项目会校验库存和产能，不推荐无法承接的项目。
- 可查看一次营销带来的项目成交、商品连带和库存消耗。

### 16.3 P2：经营 Agent 和本体治理

目标：让 AI 可以基于小本体生成门店经营计划，并让本体持续治理。

范围：

- Agent 支持“本周应该推什么、给谁、怎么推、预计收益”的经营建议。
- 本体规则支持版本管理、灰度和回滚。
- 增加本体质量看板：字段覆盖率、规则命中率、归因完整率。
- 支持多门店对比和优秀门店策略复制。

验收：

- Agent 生成的经营建议全部能追溯到客户、服务、权益、库存、归因证据。
- 推荐规则版本可回溯，异常推荐可定位原因。
- 门店可按周复盘 AI 建议采纳率和实际收入贡献。

## 17. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 一次性建模过大 | 开发周期长，业务不可用 | P0 只围绕 5-6 个高频场景 |
| 标签口径不统一 | 推荐和复盘不可信 | 生命周期、机会、权益、事件统一枚举 |
| AI 直接执行高风险动作 | 误触达、误发券、误改资产 | AI 只出草稿和建议，人确认后执行 |
| 触达过度 | 客户反感，退订增加 | 触达频控和疲劳度作为硬规则 |
| 数据质量不足 | 阶段判断错误 | 输出证据和置信度，允许人工修正 |
| 推荐无法归因 | 无法证明价值 | P0 必须统一归因事件 |
| 库存/产能未校验 | 活动承接失败 | P1 纳入库存和排班校验 |

## 18. 推荐实施顺序

1. 先统一枚举：生命周期阶段、客户机会类型、推荐执行方式、归因事件类型。
2. 再补快照：客户生命周期快照、客户机会、推荐证据。
3. 改推荐卡：让推荐卡从文本建议升级为结构化动作卡。
4. 接动作草稿：活动草稿、自动规则草稿、顾问任务。
5. 补归因：触达、点击、预约、核销、订单回流到推荐卡和客户。
6. 接服务周期：服务完成后生成下一次护理机会。
7. 接库存和产能：推荐前校验能否履约。
8. 最后开放 Agent：让 AI 基于本体解释、问数和生成经营建议。

## 19. 产品结论

这个小本体的重点不是“画一张知识图谱”，而是把美业经营里最重要的闭环固定下来：

```text
客户状态
  -> 服务机会
  -> 权益匹配
  -> 营销触达
  -> 预约/到店/核销/成交
  -> 服务结果/库存消耗/客户价值变化
  -> 下一次生命周期判断
```

对产品交付来说，最现实的第一步是 P0：围绕护理周期到期、次卡到期、沉睡召回、领券未核销、浏览未预约，把推荐卡升级成可执行动作卡，并补齐基础归因。这样既能复用现有客户画像和智能营销能力，又能为后续 Agent V3、服务经营推荐、库存消耗闭环打下统一语义基础。

## 20. P1/P2 实际落地补充（2026-07-08）

本轮 P1/P2 继续沿用 P0 的轻量持久化路线，不引入图数据库，不让 AI 自动改客户资产、库存或订单。P1 先补服务周期、库存/产能承接和归因链，P2 再补规则版本、质量快照和经营 Agent 周计划。

### 20.1 新增轻量模型

| 模型 | 作用 | 交付口径 |
| --- | --- | --- |
| `CustomerServiceCycleState` | 客户-项目维度服务周期状态 | 记录最近服务、周期天数、下一次建议护理时间和结构化证据 |
| `CustomerOpportunityFulfillmentCheck` | 机会承接校验 | 对机会做库存可承接、产能可承接、耗材需求和风险提示，不扣库存 |
| `LifecycleAttributionEvent` | 生命周期统一归因事件 | 汇总触达、行为、预约、核销、订单、库存消耗和排期填充证据 |
| `CustomerLifecycleRuleVersion` | 规则治理版本 | 支持 `draft -> active -> archived/rolled_back`，同门店同类型只保留一个 active |
| `CustomerLifecycleQualitySnapshot` | 本体质量快照 | 字段覆盖率、规则命中率、归因完整率、库存/产能可承接率 |
| `LifecycleBusinessPlan` | 经营 Agent 计划 | 生成周计划和待审批动作，不直接发券、群发、扣库存或下单 |

### 20.2 P1 服务与承接闭环

- 服务周期来源：划卡记录、项目预约/到店记录、项目订单和项目默认护理周期。
- P1 机会类型：`project_cycle_due`、`homecare_bundle`、`service_upgrade`、`project_idle_capacity`、`inventory_clearance`。
- 库存校验：按 `ProjectBomItem.standardQty * 目标人数` 估算耗材需求，对比 `Product.currentStock/safetyStock`，只输出承接风险。
- 产能校验：按未来 7 天预约、排班和空档机会估算可承接时段，只输出承接风险，不自动改排班。
- 推荐卡展示：生命周期证据、库存可承接、产能可承接、归因事件数和风险提示。

### 20.3 P2 治理与 Agent

- `POST /marketing/lifecycle/rebuild` 支持 `includeServiceCycles`、`includeFulfillmentChecks`、`includeAttribution`，默认全开。
- 新增规则治理接口：`GET/POST /marketing/lifecycle/rules`、`POST /marketing/lifecycle/rules/:id/publish`、`POST /marketing/lifecycle/rules/:id/rollback`。
- 新增质量接口：`GET /marketing/lifecycle/quality`，智能推荐页展示字段覆盖率、规则命中率、归因完整率和可承接率。
- 新增经营计划接口：`POST /marketing/lifecycle/business-plans`、`POST /marketing/lifecycle/business-plans/:id/submit-actions`。
- Ami Agent 页面新增“生成本周经营计划”，先生成草稿和待审批动作，提交审批时创建 `AgentRun/AgentApproval`，再进入现有审批链路。

### 20.4 页面验收口径

- 客户资料页“生命周期与下一步机会”展示阶段、机会、服务周期、下一次护理时间、库存/产能承接状态和归因事件。
- 客户画像页“预测视角”展示生命周期阶段、机会标签、下一次护理时间和承接风险标签。
- 智能推荐页展示 P1/P2 推荐卡，包含承接状态、归因链、本体质量指标，并继续复用活动草稿、自动规则草稿、终端跟进入口。
- Ami Agent 只生成经营计划和待审批动作，不直接执行高风险动作。

### 20.5 明确边界

- 本轮只提交 Prisma schema 和 migration 文件，不默认执行真实数据库 migration。
- 库存校验只判断承接能力，不扣减库存。
- 产能校验只判断未来 7 天可承接，不自动调整排班。
- 规则治理先做版本、发布、回滚和质量指标，不做图谱编辑器。
