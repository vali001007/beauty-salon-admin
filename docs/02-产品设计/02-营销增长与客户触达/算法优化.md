# “准模型化预测”开发计划

## Summary
第一版做“规则模型化预测闭环”，覆盖四类预测：流失风险、复购概率、活动转化概率、LTV 预测。它不是训练型机器学习，而是把现有前端规则算法升级为后端可版本化、可落库、可解释、可复盘的预测引擎。

目标交付：
- 后端统一生成客户预测快照，记录模型版本、分数、原因、特征摘要。
- 智能推荐、客户画像、自动营销预览统一读取后端预测结果。
- 自动营销执行时记录客户级触达明细，后续按预约、到店、下单、核销回流效果。
- 页面可看到“为什么推荐、预计价值、实际效果”，形成准模型闭环。

## Key Changes
### 1. 数据模型
在 `packages/server-v2/prisma/schema.prisma` 增加三类表：

- `PredictionRun`
  - 记录一次预测批次：`id`、`storeId`、`modelVersion`、`status`、`startedAt`、`finishedAt`、`customerCount`、`summaryJson`。
  - 用于回答“这批预测是什么时间、什么版本算出来的”。

- `CustomerPredictionSnapshot`
  - 每个客户每次预测一条快照。
  - 字段包含：`runId`、`customerId`、`storeId`、`churnScore`、`churnLevel`、`repurchase30dScore`、`marketingResponseScore`、`ltv6m`、`ltv12m`、`ltvTier`、`featureJson`、`reasonJson`、`recommendedActionsJson`。
  - `featureJson` 存可解释特征，例如最近到店天数、平均到店间隔、消费趋势、会员等级、肤质、次卡状态、未来预约。

- `MarketingAutomationTouch`
  - 自动营销执行的客户级触达记录。
  - 字段包含：`executionId`、`strategyId`、`customerId`、`predictionSnapshotId`、`predictedConversionScore`、`predictedRevenue`、`channel`、`status`、`touchedAt`、`convertedAt`、`conversionType`、`actualRevenue`、`attributionWindowDays`。
  - 用于把“预测名单”与后续真实回店、下单、核销关联起来。

### 2. 后端预测服务
在后端 v2 营销模块内新增预测服务，不新起独立服务，避免系统割裂。

预测算法第一版固定为 `rules-v1`：

- **流失风险**
  - 当前到店间隔 / 个人平均到店间隔：最高 40 分。
  - 绝对未到店天数：最高 25 分。
  - 消费频率下降：最高 20 分。
  - 会员等级修正：普通会员加风险，高等级会员降风险。
  - 已有未来预约、有效次卡可降低风险。
  - 输出：0-100 分、低/中/高/极高、原因列表。

- **复购概率**
  - 最近到店较近、护理周期将到期、有效次卡、历史频率稳定、促销敏感度高会加分。
  - 高流失风险、长期未到店、消费趋势下降会降分。
  - 输出未来 30 天复购概率分。

- **活动转化概率**
  - 在策略预览/执行时按客户和策略动态计算。
  - 输入：客户预测快照、触发规则命中原因、营销渠道、优惠类型、客户促销敏感度、会员等级、肤质/项目匹配度。
  - 输出：单客户预计转化率、预计收入、命中原因。
  - 自动营销执行时固化到 `MarketingAutomationTouch`，后续用于复盘。

- **LTV 预测**
  - 沿用当前月均消费外推，但迁移到后端并去除随机数。
  - `ltv6m = 月均消费 × 6 × 趋势系数 × 流失折扣`
  - `ltv12m = 月均消费 × 12 × 趋势系数 × 流失折扣 × 0.95`
  - 分层：铂金 >= 80000，黄金 >= 30000，白银 >= 10000，其余青铜。

### 3. API / Types
新增或调整接口：

- `POST /marketing/predictions/run`
  - 手动触发一次预测批次。
  - 返回 `PredictionRun` 和汇总。

- `GET /marketing/predictions/latest`
  - 返回当前门店最新预测批次汇总。
  - 包含四类预测分布：流失风险、复购概率、LTV 分层、营销响应分布。

- `GET /marketing/predictions/customers`
  - 分页返回客户预测列表。
  - 支持筛选：`churnLevel`、`ltvTier`、`minRepurchaseScore`、`minMarketingResponseScore`、`storeId`。

- `GET /marketing/predictions/customers/:id`
  - 返回单客户最新预测快照、原因、建议动作、历史预测变化。

- `GET /marketing/recommendations`
  - 改为基于最新预测快照生成推荐卡。
  - 保持前端现有字段兼容：`title`、`reason`、`targetCustomers`、`matchScore`、`expectedRevenue`、`targetCustomerIds`、`source`。
  - 增加：`predictionRunId`、`modelVersion`、`predictionType`、`dataEvidence`。

- `POST /marketing/automation/strategies/:id/execute`
  - 执行策略时生成 `MarketingAutomationTouch`。
  - `triggeredCount`、`reachedCount` 继续写入原执行表，客户级明细写入新表。

前端类型同步增加：
- `CustomerPredictionSnapshot`
- `PredictionRunSummary`
- `PredictionReason`
- `MarketingAutomationTouchEffect`

### 4. Frontend Changes
- 客户画像页增加“预测视角”：
  - 流失风险、30 天复购概率、LTV 分层、营销响应分。
  - 每个分数支持展开“原因”，例如“超过平均到店间隔 2.4 倍”“近 90 天消费下降”。

- 智能推荐页：
  - 推荐卡展示 `模型版本 rules-v1`、预测批次时间、数据依据。
  - 四类来源明确标识：流失预警、复购机会、活动转化、LTV 维护。
  - 点击目标客户列表时展示客户级预测分和原因。

- 自动营销页：
  - 预览受众时不再只估算人数，改为显示预计触达人数、预计转化人数、预计收入。
  - 执行记录详情展示客户级触达明细和后续转化状态。
  - 效果分析页增加预测 vs 实际：预计转化、实际转化、预计收入、实际收入、偏差。

- Mock 模式：
  - 保留现有前端可演示能力。
  - mock 数据按同一字段结构返回，避免 mock/real UI 分叉。

## Test Plan
- 后端单测：
  - 固定客户样本验证四类预测分数稳定、无随机数。
  - 验证无消费记录、无到店日期、无健康档案、无门店权限时的降级结果。
  - 验证自动营销执行会生成客户级 `MarketingAutomationTouch`。
  - 验证效果回流能按归因窗口更新转化状态和实际收入。

- 前端单测：
  - 客户画像能渲染预测分和原因。
  - 智能推荐能兼容旧字段和新增预测字段。
  - 自动营销预览能展示预计转化和预计收入。

- 集成验证：
  - `npm run test`
  - `npm run lint`
  - `npm run build`
  - `cd packages/server-v2 && npm run test`
  - `cd packages/server-v2 && npm run lint`
  - `cd packages/server-v2 && npm run build`

- 手动验收场景：
  - 手动触发预测后，客户画像出现最新预测结果。
  - 智能推荐能生成四类推荐卡。
  - 从推荐卡创建自动营销策略，预览名单能看到预计转化。
  - 执行策略后有客户级触达记录。
  - 新增订单/预约/核销后，效果复盘能看到实际转化更新。

## Assumptions
- 第一版不接入 Python、AutoML 或外部机器学习服务；只做可解释规则模型。
- `rules-v1` 是第一个模型版本，后续可升级为 `rules-v2` 或真实 ML 模型，但前端和 API 不需要重构。
- 预测默认按门店隔离；超级管理员可看全门店汇总。
- 归因窗口默认 30 天；客户在触达后 30 天内预约、到店、下单、核销任一行为即算转化，订单金额计入实际收入。
- 不批量删除历史数据；预测快照和触达明细只追加，后续如需清理必须单独确认。
