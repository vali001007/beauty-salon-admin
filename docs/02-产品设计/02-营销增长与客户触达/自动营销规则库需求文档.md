# 自动营销规则库需求文档

更新时间：2026-06-09

## 1. 背景

当前管理端「智能营销 / 自动营销」已经有触发规则下拉选择器，用户可以在创建自动营销策略时选择一个触发场景，例如「次卡/套餐即将到期」「生日触发」「小程序浏览未预约」「沉睡客户唤醒」等。现有规则来自后端 `/marketing/automation/trigger-options`，前端会读取规则分类、规则名称、说明、优先级、参数表单和 `defaultParams`，选中后自动带入默认参数，并支持用户在策略编辑表单里修改参数。

这说明系统已经不是单纯的静态下拉，而是初步具备「规则目录 + 默认参数 + 用户自定义参数 + 受众预估 + 策略执行」的雏形。

但当前规则仍然停留在「创建策略时可选的触发选项」层面，还没有被产品化为可管理、可复用、可追踪效果的规则资产。随着智能推荐、小程序行为、客户预测、优惠券、卡项、预约、商品补货等数据逐步接入，规则数量和规则来源会继续增加，有必要把它沉淀为独立的「自动营销规则库」。

## 2. 当前现状

### 2.1 已有能力

当前已具备以下能力：

| 能力 | 当前状态 |
| --- | --- |
| 规则目录 | 后端提供 `/marketing/automation/trigger-options`，返回规则列表 |
| 规则分类 | 已分为时间触发、行为触发、属性触发 |
| 默认参数 | 每条规则有 `defaultParams`，用于自动填入推荐值 |
| 参数表单 | 每条规则有 `paramSchema`，前端可动态渲染参数项 |
| 用户自定义 | 用户修改参数后，前端标记为「已自定义」 |
| 自动营销策略 | 可创建、编辑、启用、暂停、删除、执行 |
| 受众预估 | 创建前可预估命中客户数和样本客户 |
| 智能推荐联动 | 智能推荐可生成自动营销规则草稿 |
| 效果统计 | 已有自动营销执行记录和效果统计接口 |

### 2.2 当前已有触发规则

按后端真实规则接口，当前规则大致包括：

| 分类 | 规则 |
| --- | --- |
| 时间触发 | 优惠券即将到期、护理周期到期、季节换肤护理、季节护理、节假日营销、节日营销、生日触发 |
| 行为触发 | 次卡/套餐即将到期、预约放弃、沉睡客户唤醒、小程序浏览未预约、领券未核销、商品补货提醒、老带新/闺蜜同行、最近消费时间、消费金额、到店频次、到店间隔异常、项目偏好 |
| 属性触发 | 高价值客户权益维护、会员等级、肤质类型、新客转化、年龄区间 |

从产品成熟度看，P0/P1 场景已经覆盖美业门店最常见的自动营销闭环：

- 到期提醒：优惠券、次卡/套餐、护理周期。
- 流失召回：沉睡客户、最近消费时间、到店间隔异常。
- 转化召回：预约放弃、小程序浏览未预约、领券未核销。
- 会员经营：会员等级、高价值客户权益维护、生日触发。
- 个性化推荐：肤质类型、项目偏好、季节护理。

### 2.3 主要问题

当前不缺「规则选项」，缺的是「规则资产管理」：

| 问题 | 产品影响 |
| --- | --- |
| 规则只在创建策略时出现 | 用户无法先浏览有哪些推荐规则，也无法按场景选规则 |
| 系统默认规则不可管理 | 默认参数好不好、是否启用、适合什么门店，目前无法展示和维护 |
| 自定义规则只保存在策略里 | 用户改过一次参数后，不能保存为门店自己的规则模板反复使用 |
| 规则缺少版本 | 默认参数升级后，无法区分老策略使用的是旧版本还是新版本 |
| 规则效果没有沉淀到规则维度 | 只能看策略效果，不能判断哪类规则更有效 |
| 规则来源分散 | 后端规则选项、智能推荐默认参数、mock 数据中都有规则逻辑，长期会产生不一致 |
| 缺少规则启停和适用范围 | 不同门店、不同套餐版本、不同数据完备度下，应看到不同规则推荐 |

## 3. 是否需要单独做规则库

结论：建议单独做「自动营销规则库」，但 MVP 不做复杂规则引擎，先做规则模板资产库。

### 3.1 为什么有必要

1. 规则已经超过简单下拉的规模
当前已有 20+ 条规则，且横跨优惠券、卡项、预约、小程序行为、客户画像、消费画像、肤质档案、预测模型等多个数据域。如果继续只用下拉承载，用户会越来越难理解规则用途。

2. 默认规则本身就是产品价值
对美业门店来说，「60 天未到店唤醒」「生日月权益」「优惠券 D-7/D-3/D-1 提醒」「浏览后 24 小时未预约召回」这些默认规则就是可交付的行业运营方法。它们应该以推荐规则的形式呈现，而不是藏在参数默认值里。

3. 用户需要从“配策略”变成“用规则”
门店老板和运营人员更容易理解「启用沉睡客户唤醒规则」而不是从零配置触发条件、动作、频控和渠道。规则库能降低使用门槛。

4. 自定义规则需要复用
例如某门店把沉睡天数从 60 天改成 45 天，把触达渠道改成小程序 + 顾问跟进。这应该能保存为「本店沉睡客户唤醒规则」，后续新建策略时直接复用。

5. 后续 AI 推荐需要规则资产承接
智能推荐页可以推荐「启用某条默认规则」或「基于当前客群生成一条自定义规则」。没有规则库时，推荐只能生成一次性策略草稿，难以形成长期运营资产。

### 3.2 为什么不建议一开始做复杂规则引擎

MVP 阶段不建议做拖拽式复杂规则编排、嵌套条件组、脚本表达式或通用事件流引擎。原因是：

- 美业门店首要诉求是开箱即用，不是自己设计复杂规则。
- 当前最有价值的是默认规则推荐、自定义参数、受众预估、启用执行。
- 复杂规则引擎会显著增加学习成本和测试成本。
- 现有 `triggerRules + ruleRelation` 已可支持单规则和简单 AND/OR 组合，足够覆盖 MVP。

## 4. 产品定位

自动营销规则库是平台沉淀自动化营销方法的资产中心，用于管理系统推荐规则、门店自定义规则、规则效果和规则启用状态。

一句话定义：

> 把「什么时候触发、触发谁、用什么默认参数、建议怎么触达」沉淀成可推荐、可复制、可自定义、可启用的营销规则模板。

## 5. 产品目标

### 5.1 业务目标

- 让门店不用懂复杂配置，也能快速启用高价值自动营销。
- 把当前已有触发规则沉淀为默认推荐规则，形成产品壁垒。
- 支持门店基于默认规则做轻量自定义，并反复复用。
- 支持后续按规则维度做效果分析和推荐优化。

### 5.2 用户目标

- 我能看到系统推荐我应该启用哪些自动营销规则。
- 我能理解每条规则适合什么场景、会命中哪些客户、预计带来什么效果。
- 我能一键启用默认规则，也能修改参数后保存为自己的规则。
- 我能知道规则启用后有没有效果，是否需要调整。

## 6. 范围定义

### 6.1 MVP 包含

| 模块 | 说明 |
| --- | --- |
| 规则库列表 | 展示系统默认规则和门店自定义规则 |
| 规则详情 | 展示规则说明、默认参数、适用场景、数据依赖、受众预估 |
| 规则复制 | 系统默认规则不可直接修改，允许复制为门店自定义规则 |
| 自定义规则 | 用户可修改参数、渠道、频控、推荐动作，并保存为自定义规则 |
| 一键启用 | 从规则库直接创建并启用自动营销策略 |
| 受众预估 | 启用前展示命中客户数、样本客户、命中原因 |
| 规则状态 | 支持启用、停用、草稿、归档 |
| 规则效果 | 展示规则关联策略数、触达人数、核销/预约/回店/收入 |

### 6.2 MVP 暂不包含

| 暂不包含 | 原因 |
| --- | --- |
| 拖拽式复杂条件编排 | 学习成本高，当前业务收益不如默认规则推荐 |
| 跨规则优先级冲突自动决策 | 先用策略级频控解决 |
| 多层嵌套 AND/OR 条件组 | 当前 `triggerRules + ruleRelation` 先满足简单组合 |
| AI 自动生成全新规则并直接上线 | 初期可生成草稿，但上线前必须人工确认 |
| 按行业大模型自动调参 | 先积累规则效果数据，再做智能优化 |

建议在「智能营销」下新增菜单：

```text
智能营销
├─ 智能推荐
├─ 自动营销
├─ 规则库
├─ 营销页面
├─ 活动管理
└─ 效果分析
```

如果不想新增一级菜单，也可以在「自动营销」页面内增加二级标签：

```text
自动营销
├─ 策略列表
├─ 规则库
└─ 执行记录
```

推荐方案：新增「规则库」菜单。
原因：规则库是长期运营资产，不只是策略创建流程的一步，单独入口更利于用户理解和后续扩展。

## 8. 页面需求

### 8.1 规则库列表页

#### 页面目标

让用户快速找到适合启用的规则，并区分「系统推荐」和「我的规则」。

#### 顶部筛选

- 规则来源：全部、系统推荐、我的规则。
- 规则分类：全部、时间触发、行为触发、属性触发。
- 业务场景：到期提醒、流失召回、转化召回、会员经营、个性化推荐、裂变营销。
- 优先级：P0、P1、P2、P3。
- 状态：全部、推荐启用、已启用、未启用、停用、草稿。
- 搜索：规则名称、说明、触发类型。

#### 列表字段

| 字段 | 说明 |
| --- | --- |
| 规则名称 | 如「沉睡客户唤醒」 |
| 来源 | 系统推荐 / 我的规则 |
| 分类 | 时间触发 / 行为触发 / 属性触发 |
| 业务场景 | 流失召回、到期提醒等 |
| 优先级 | P0/P1/P2/P3 |
| 推荐理由 | 简短说明为什么建议启用 |
| 默认触达渠道 | 小程序、短信、微信、门店话术等 |
| 预计命中客户 | 基于当前门店数据预估 |
| 已关联策略 | 使用该规则的策略数量 |
| 最近效果 | 最近 30 天触达、转化或收入 |
| 状态 | 未启用、已启用、停用、草稿 |
| 操作 | 查看、启用、复制、自定义、停用 |

#### 展示方式

MVP 建议使用表格 + 规则详情抽屉。
营销运营用户需要对比规则、状态和效果，表格比卡片更适合扫描。

### 8.2 规则详情抽屉

点击规则后打开右侧抽屉，展示：

- 基础信息：规则名称、分类、业务场景、优先级、来源、版本。
- 规则说明：适合什么场景，解决什么经营问题。
- 触发条件：用业务语言解释参数。
- 默认参数：例如未到店超过 60 天、排除已预约、渠道小程序 + 短信。
- 数据依赖：客户档案、消费记录、卡项、优惠券、小程序行为、预约记录等。
- 受众预估：命中客户数、样本客户、命中原因。
- 建议动作：发券、短信、小程序推送、顾问跟进、活动页承接等。
- 频控建议：同客户 7 天内最多触达 1 次等。
- 效果概览：触达人数、预约数、核销数、回店数、收入、ROI。
- 使用记录：哪些自动营销策略正在使用该规则。

### 8.3 规则编辑页/弹窗

系统默认规则不可直接编辑，只能复制为「我的规则」后编辑。

可编辑内容：

| 区域 | 字段 |
| --- | --- |
| 基础信息 | 规则名称、描述、业务场景、标签 |
| 触发参数 | 按 `paramSchema` 动态渲染，例如天数、金额、次数、渠道 |
| 触达动作 | 优惠券、折扣、礼品、积分、短信、小程序、微信、门店话术 |
| 执行计划 | 实时、每日、每周、每月、活动周期 |
| 频控 | 同客户触达间隔、同渠道触达间隔、单日触达上限 |
| 排除条件 | 排除已预约、近期已购买、近期已触达、黑名单客户 |
| 预览 | 命中客户数、样本客户、预计转化、预计收入 |

### 8.4 一键启用流程

用户从规则库点击「启用」：

1. 系统打开启用确认弹窗。
2. 展示默认参数和预计命中客户。
3. 用户可直接启用，也可点「修改后启用」。
4. 系统创建一条自动营销策略，策略来源标记为规则库。
5. 启用成功后，规则状态变为「已启用」，并关联策略 ID。

## 9. 规则库数据模型

### 9.1 规则模板

```ts
interface MarketingRuleTemplate {
  id: number;
  code: string;
  name: string;

  version: string;
  baseTemplateId?: number;
  triggerType: MarketingTriggerType;
  paramSchema: MarketingTriggerParamSchema[];
  defaultParams: Record<string, MarketingParamValue>;
  recommendedActions: MarketingAction[];
  scheduleDefault: MarketingSchedule;
  frequencyCap: FrequencyCap;
  dataDependencies: string[];
  recommendationReason: string;
  createdBy?: number;
  storeId?: number;
  createdAt: string;
  updatedAt: string;
}
```

### 9.2 规则启用实例

规则模板不直接执行，执行对象仍然是自动营销策略。启用规则时创建策略，并保留规则来源。

```ts
interface MarketingAutomationStrategy {
  id: number;
  name: string;
  source: 'manual' | 'rule_library' | 'recommendation';
  ruleTemplateId?: number;
  ruleTemplateVersion?: string;
  triggerRules: MarketingTriggerRule[];
  actions: MarketingAction[];
  schedule: MarketingSchedule;
  status: 'draft' | 'enabled' | 'paused' | 'archived';
}
```

## 10. 默认推荐规则

MVP 建议把当前已有规则分成三批默认推荐。

### 10.1 P0：优先做成默认推荐

| 规则 | 推荐原因 | 默认参数建议 |
| --- | --- | --- |
| 沉睡客户唤醒 | 直接解决流失召回，是自动营销核心场景 | 60 天未到店，排除已预约和近期已购，小程序 + 短信 |
| 次卡/套餐即将到期 | 直接影响卡项消耗和续费 | 到期前 30 天，剩余 1 次，小程序 + 短信 |
| 优惠券即将到期 | 促进核销和预约 | 到期前 7/3/1 天提醒，排除已预约 |
| 领券未核销 | 承接营销活动后转化 | 领券后 3 天未核销，小程序 + 短信 |
| 小程序浏览未预约 | 承接 Ami Glow 行为数据 | 浏览后 24 小时未预约，推送项目案例或体验券 |
| 预约放弃 | 对预约漏斗做即时召回 | 放弃后 2 小时，推荐相邻时段 |
| 生日触发 | 低风险高接受度的会员关怀 | 生日前 7 天或生日月，小程序 + 短信 |

### 10.2 P1：第二批推荐

| 规则 | 推荐原因 |
| --- | --- |
| 护理周期到期 | 适合推动复购预约 |
| 最近消费时间 | 轻量召回，适合大多数门店 |
| 到店间隔异常 | 比固定未到店天数更个性化 |
| 会员等级 | 适合高价值客户维护 |
| 肤质类型 | 和 Ami Aura Lite、肌肤档案形成闭环 |
| 新客转化 | 新客建档后首单和二次到店引导 |

### 10.3 P2：后续增强

| 规则 | 推荐原因 |
| --- | --- |
| 季节换肤护理 | 更偏活动季节运营 |
| 节假日营销 | 适合活动页和营销页面联动 |
| 商品补货提醒 | 依赖商品消耗周期准确性 |
| 项目偏好 | 依赖消费记录和项目分类质量 |
| 老带新/闺蜜同行 | 需要裂变活动承接 |
| 年龄区间 | 可作为辅助筛选，不建议单独作为强规则 |

## 11. API 需求

### 11.1 新增规则库 API

| Method | Path | 说明 |

### 11.2 与现有 API 的关系

现有 `/marketing/automation/trigger-options` 可以保留兼容，但建议后续由规则库数据生成：

- 规则库用于完整管理和展示。
- `trigger-options` 用于策略编辑器内的轻量选择。
- 自动营销策略仍使用现有 `/marketing/automation/strategies` 创建和执行。
- 智能推荐生成自动营销草稿时，优先引用规则库模板。

## 13. 交互细节

### 13.1 默认规则保护

- 系统默认规则不可直接编辑。
- 用户点击编辑默认规则时，提示「复制为我的规则后可修改」。
- 复制后的规则保留 `baseTemplateId`，便于追踪来源。

### 13.2 参数来源标识

规则启用或编辑时，需要展示：

- 已使用系统推荐值。
- 已使用门店自定义值。
- 来自智能推荐覆盖值。

这与当前策略编辑页已有的 `parameterSource` 保持一致。

### 13.3 数据不足提示

如果规则依赖的数据不足，需要明确提示：

| 数据不足 | 提示 |
| --- | --- |
| 没有小程序行为 | 暂无浏览/点击行为，建议先接入 Ami Glow 行为数据 |
| 没有卡项数据 | 暂无法预估次卡/套餐到期客户 |
| 没有生日字段 | 部分客户缺少生日，生日触发命中人数可能偏低 |
| 没有肌肤档案 | 肤质规则仅能命中已建档客户 |

### 13.4 频控提示

启用规则前必须展示频控，避免用户担心打扰客户：

- 同一客户同一规则 7 天内最多触达 1 次。
- 生日、到期、预约放弃等事件类规则按事件单独计算。
- 已预约、近期已购买、黑名单客户默认排除。

## 14. 与智能推荐的关系

智能推荐页后续可以直接推荐规则库规则：

- 「建议启用：沉睡客户唤醒规则，预计命中 206 人」
- 「建议复制并调整：次卡/套餐即将到期规则，把提前天数从 30 天改为 15 天」
- 「建议新增：小程序浏览未预约召回规则，基于近 7 天浏览行为」

推荐动作分三类：

| 动作 | 说明 |
| --- | --- |
| 启用默认规则 | 直接使用系统默认参数 |
| 自定义后启用 | 基于默认规则改参数 |
| 生成规则草稿 | AI 根据当前数据生成建议，但需人工确认 |

## 15. 开发优先级

### 阶段 1：规则库 MVP

- 新增规则库列表页。
- 将当前 `trigger-options` 规则沉淀为系统默认规则模板。
- 支持查看规则详情、受众预估、一键启用。
- 支持复制系统规则为我的规则。
- 支持我的规则编辑参数和保存。

### 阶段 2：效果和推荐

- 增加规则维度效果统计。
- 在智能推荐页直接推荐启用规则。
- 支持规则效果排序和规则优化建议。

### 阶段 3：规则治理

- 增加规则版本管理。
- 支持系统默认规则灰度升级。
- 支持门店按行业模板、客单价、经营阶段应用不同默认参数。

## 16. 验收标准

MVP 验收标准：

- 用户能在规则库看到系统默认推荐规则和我的规则。
- 用户能按分类、场景、优先级、状态筛选规则。
- 用户能查看每条规则的默认参数、适用场景、数据依赖和受众预估。
- 用户能一键启用系统默认规则，并生成对应自动营销策略。
- 用户能复制系统规则为自定义规则，并修改参数保存。
- 用户启用自定义规则后，自动营销策略使用自定义参数执行。
- 系统默认规则不可被普通门店用户直接修改。
- 规则库里的 P0 规则至少覆盖沉睡客户唤醒、次卡/套餐即将到期、优惠券即将到期、领券未核销、小程序浏览未预约、预约放弃、生日触发。
- 规则效果至少展示关联策略数、触达人数、转化人数、收入。

## 17. 结论

建议单独建设「自动营销规则库」。

当前系统已经有规则目录和默认参数，具备升级为规则库的基础。规则库不是额外增加复杂度，而是把现有规则从「创建策略时的选项」升级为「可推荐、可复用、可自定义、可评估的营销资产」。MVP 应优先围绕默认推荐规则、自定义参数、一键启用和受众预估展开，暂不做复杂规则编排。
