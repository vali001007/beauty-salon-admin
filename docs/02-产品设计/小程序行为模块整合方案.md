# 小程序行为模块整合方案

## 背景

当前管理端出现了三个与“小程序行为”相关的入口：

1. 客户管理 / 客户数据 / 小程序行为
2. 客户管理 / 客户画像 / 小程序行为
3. 智能营销 / 营销资产 / 行为事件

三者底层都与 Ami Glow 客户服务小程序的浏览、点击、预约、测肤、领券等行为事件有关，但业务视角不同。当前最大问题不是“完全重复”，而是原始事件列表在多个位置重复出现，容易让用户误以为是三套数据，也会增加后续筛选、导出、字段解释和权限维护成本。

## 当前关系

### 1. 客户数据 / 小程序行为

定位：客户数据明细视角。

主要作用：

- 查看客户维度的小程序行为流水。
- 支持按客户、事件类型、目标对象等条件查询。
- 用于客服、运营或门店人员排查“某个客户有没有浏览、点击、预约、测肤”等行为。
- 更接近客户档案的“行为明细账”。

数据层级：

- 原始事件明细。
- 面向客户管理，不强调营销资产效果。

建议保留，但应改名为：小程序行为明细。

### 2. 客户画像 / 小程序行为

定位：客户洞察分析视角。

主要作用：

- 汇总小程序绑定客户、活跃客户、预约意向客户等指标。
- 展示行为漏斗、入口模块、客户分层、活跃度等分析结果。
- 用于理解客户群体在小程序里的行为偏好。
- 服务于客户画像、智能推荐、自动营销规则命中。

数据层级：

- 派生分析结果。
- 不应该重复承载完整原始事件流水。

建议保留，而且应作为“小程序行为分析”的主入口。

### 3. 营销资产 / 行为事件

定位：营销资产归因视角。

主要作用：

- 查看营销页面、优惠活动、小程序推荐位等营销资产带来的曝光、点击、预约、转化事件。
- 服务于效果分析、自动营销触发、规则库数据依赖。
- 重点不是“某个客户做了什么”，而是“某个营销对象带来了什么行为”。

数据层级：

- 原始事件明细 + 营销资产上下文。
- 应聚焦营销对象、渠道、活动、页面、优惠、推荐位。

建议保留，但不应和客户数据页展示完全相同的事件表。

## 是否冗余

结论：存在部分冗余，但不是三个模块都冗余。

### 不冗余的部分

客户画像 / 小程序行为不冗余。它是分析层，应该保留，用来做客户分群、行为洞察和营销建议。

营销资产 / 行为事件不冗余。它是营销归因层，应该保留，用来支撑营销资产效果和自动规则触发。

### 存在冗余的部分

客户数据 / 小程序行为 和 营销资产 / 行为事件 都在展示原始事件列表，字段和筛选能力高度接近。

区别只在业务上下文：

- 客户数据页关注客户是谁。
- 营销资产页关注事件来自哪个营销对象。

如果继续各做一套表格，会带来：

- 字段定义重复。
- 筛选逻辑重复。
- 导出逻辑重复。
- 事件类型文案重复。
- 后续埋点字段升级时要改多处。

## 整合原则

1. 一个事件源：所有小程序行为统一进入 `CustomerAppEvent`。
2. 一个事件字典：事件类型、目标对象、渠道、来源统一定义。
3. 一个明细组件：原始事件表格只做一套组件，不同页面传入不同默认筛选和展示列。
4. 三个业务视角保留：客户明细、客户洞察、营销归因不要混成一个页面。
5. 页面命名要让用户一眼知道用途，不要三个地方都叫“小程序行为”。

## 推荐整合方案

### 信息架构

保留三个入口，但调整定位和命名：

| 当前入口 | 推荐名称 | 定位 | 是否保留 |
| --- | --- | --- | --- |
| 客户数据 / 小程序行为 | 小程序行为明细 | 客户维度事件流水 | 保留 |
| 客户画像 / 小程序行为 | 小程序行为分析 | 客户群体行为洞察 | 保留 |
| 营销资产 / 行为事件 | 营销行为事件 | 营销对象归因事件 | 保留 |

### 页面职责

#### 客户数据 / 小程序行为明细

保留为“查客户行为流水”的地方。

默认筛选：

- 当前门店。
- 全部客户。
- 默认 10 条分页。

核心字段：

- 客户名称 / 手机号
- 事件类型
- 目标对象
- 渠道
- 发生时间
- 会话 ID
- 事件摘要

核心能力：

- 搜索客户。
- 按事件类型筛选。
- 查看事件详情。
- 导出明细。

#### 客户画像 / 小程序行为分析

保留为“看客户行为结论”的地方。

核心模块：

- 小程序绑定率。
- 7 日 / 30 日活跃客户。
- 预约意向客户。
- 行为漏斗。
- 常用入口模块。
- 小程序行为分层：高活跃、有意向、低活跃、待绑定。
- 可点击分层跳转到客户数据明细，并带筛选条件。

不建议在此页面放完整原始事件流水，只保留 Top 客户或分层客户列表。

#### 营销资产 / 营销行为事件

保留为“看营销资产带来的行为”的地方。

默认筛选：

- 当前门店。
- 来源为 Ami Glow / 营销页面 / 优惠活动 / 推荐位。
- 默认 10 条分页。

核心字段：

- 事件类型
- 营销对象类型
- 营销对象名称 / ID
- 客户
- 渠道
- 发生时间
- 是否进入转化链路

核心能力：

- 按营销对象筛选。
- 按事件类型筛选。
- 跳转效果分析。
- 跳转对应营销页面、优惠活动、推荐位配置。

## 组件与接口整合建议

### 前端组件

新增一个共享组件：

```text
CustomerAppEventTable
```

支持三种模式：

```text
mode = customerDetail | customerInsight | marketingAsset
```

不同模式只改变：

- 默认筛选条件。
- 显示列。
- 空态文案。
- 行点击跳转。
- 导出文件名。

### API

保留一个分页事件接口作为统一明细源：

```text
GET /api/customer-app/admin/events
```

建议补充统一查询参数：

```text
storeId
customerId
keyword
eventType
targetType
targetId
source
channel
startDate
endDate
page
pageSize
```

客户画像分析接口继续保留：

```text
GET /api/customers/miniapp-behavior-analysis
```

但它只返回分析结果，不返回完整事件明细。

营销效果分析继续消费统一效果接口：

```text
GET /api/marketing/effects/unified
```

它用行为事件做聚合，而不是自己再维护一份事件列表。

## 权限建议

| 能力 | 权限 |
| --- | --- |
| 查看客户行为明细 | `core:customer:view` |
| 查看客户行为分析 | `core:customer:profile` |
| 查看营销行为事件 | `core:marketing:view` |
| 导出行为事件 | 建议新增 `core:customer:export` 或沿用客户数据导出权限 |
| 查看营销效果 | `core:marketing:analytics` |

## 推荐实施路径

### MVP

1. 抽取共享事件表组件 `CustomerAppEventTable`。
2. 客户数据 / 小程序行为 改用共享组件，模式为 `customerDetail`。
3. 营销资产 / 行为事件 改用共享组件，模式为 `marketingAsset`。
4. 客户画像 / 小程序行为 保留分析卡片和分层，不再展示完整事件流水。
5. 页面命名统一：
   - 小程序行为明细
   - 小程序行为分析
   - 营销行为事件

### 第二阶段

1. 建立事件字典配置。
2. 支持事件详情抽屉。
3. 支持从客户画像分层跳转到客户数据明细。
4. 支持从营销行为事件跳转到效果分析或对应营销资产。
5. 增加事件质量检查：未知事件、缺客户、缺目标对象、缺渠道。

### 第三阶段

1. 用事件字典驱动自动营销规则触发条件。
2. 行为事件进入客户预测特征。
3. 营销资产效果分析支持完整归因链路：曝光、点击、留资、预约、到店、支付。

## 产品结论

三个模块不建议简单删除合并成一个页面。

更合理的方式是：

- 底层数据合并：统一事件源、统一事件表组件、统一事件字典。
- 上层场景分开：客户查明细、画像看分析、营销看归因。

这样既减少重复开发，又不会牺牲用户在不同业务场景下的使用效率。
