# Ami Aura Lite Kiosk Prototype 意图驱动即时美业微应用需求文档

版本：v1.1

关联系统：Ami_Core 美业管理平台、Ami Aura Lite 智能终端、Core Terminal API、Core AI Gateway  

## 1. 文档结论

`packages/Ami-Aura-Lite-Kiosk` 当前已经不是单纯 Figma Make 静态原型，而是一个已经开始接入 Ami_Core 的高保真智能终端雏形。它具备三角色首页、快捷按钮、消息流、业务卡片、核销、收银、办卡、充值、登记、预约操作和 AI 建议等关键能力。

但它还没有形成稳定的“意图驱动即时微应用”架构，主要问题是：

- 指令识别仍集中在 `AppContent.tsx`，没有独立 intent router 和 command registry。
- 多处中文文案已经乱码，影响按钮、占位符、指令识别、支付方式和错误提示。
- 部分流程仍依赖 `loadCoreSnapshot()` 一次性拉取客户、预约、库存、卡项、订单、员工等多组接口，导致加载慢和失败面变大。
- AI 已接入为“建议卡片”，但还没有承担统一的意图分类、槽位抽取和多轮澄清能力。
- 业务流程已经有卡片化形态，但仍混有演示逻辑，例如默认取第一个客户、默认生成操作结果、部分提交缺少幂等和二次确认。
- 权限与角色虽然开始从 Core bootstrap 获取，但前端仍存在本地角色切换和本地 action 判断，需要进一步收敛到 Core 返回能力。

下一阶段建议把 Ami Aura Lite 定位为“意图驱动的即时美业微应用”：用户通过自然语言、语音、快捷按钮、扫码、拍照触发业务意图；系统即时生成对应业务卡片和流程；所有事实数据和写回动作来自 Ami_Core；AI 只做理解、编排、解释、建议和话术，不生成业务事实。

## 2. 当前原型现状分析

### 2.1 工程状态

原型目录：`packages/Ami-Aura-Lite-Kiosk`

当前工程特征：

- 包名仍为 `@figma/my-make-file`，说明工程身份仍带有 Figma Make 导出痕迹。
- 使用 Vite + React，未配置 TypeScript 类型检查脚本，`build` 仅执行 `vite build`。
- 已有 `@` 指向根项目 `src`，`@aura` 指向原型 `src` 的 alias。
- 已配置 `envDir` 指向仓库根目录，可读取根目录环境变量。
- 已配置 `/api` 代理到 `http://127.0.0.1:8080`。
- 已允许 Vite dev server 访问仓库根目录源码。
- 已通过 `optimizeDeps.include` 处理 `axios`、`zustand` 预构建。

产品含义：

- 这个原型已经具备接 Ami_Core 的工程基础。
- 后续不建议继续作为孤立 Figma 原型维护，应正式收敛为 Ami Aura Lite 的智能终端主线或重要分支。
- 必须补上类型检查、乱码修复和接口失败兜底，否则不适合进入稳定演示和联调。

### 2.2 Core 接入情况

核心适配层：`src/app/services/auraCoreService.ts`

已经调用的 Core 能力包括：

- 登录与用户：`login`、`getUserInfo`
- 门店上下文：`storeStore`、`getTerminalBootstrap`
- 角色首页：`getTerminalRoleDashboard`
- 预约：`getReservationsPaginated`、`getTerminalReservations`、`createTerminalReservation`、`updateTerminalReservation`、`checkInTerminalReservation`
- 客户：`getCustomers`、`getCustomersPaginated`、`quickCreateTerminalCustomer`
- 美容师与排班：`getBeauticians`、`getSchedule`
- 库存：`getStockItems`、`getStockItemsPaginated`、`getExpiringProducts`、`getReplenishmentSuggestions`
- 卡项：`getCards`、`getTerminalCustomerCards`、`verifyTerminalCardUsage`
- 收银：`createTerminalCashierOrder`、`completeTerminalPayment`
- 办卡：`createTerminalCardOrder`
- 充值：`createTerminalRechargeOrder`
- 服务任务：`getTerminalServiceTasks`、`completeTerminalServiceTask`
- 皮肤检测：`createTerminalSkinTest`
- 打印：`createTerminalPrintJob`
- AI：`sendAiChatMessage`

当前已有优化：

- bootstrap 缓存 30 秒。
- role dashboard 缓存 15 秒。
- 角色首页优先调用 `getTerminalRoleDashboard()`，失败后降级到轻量 Core 查询。

当前主要问题：

- `loadCoreSnapshot()` 会并发拉取大量接口，适合作为兜底，不适合作为每个微应用的默认数据源。
- 提交流程仍经常先拉全量 snapshot，再执行写操作，导致慢和不稳定。
- 部分流程没有使用更细粒度的专用接口，例如客户搜索、客户卡项、可售项目商品、支付方式字典、服务任务详情等。
- 部分 fallback 会默认取第一个客户、第一张卡、第一位美容师，演示可用但生产不可接受。

### 2.3 AI 接入情况

当前 AI 接入点：

- `getAiSuggestion()` 调用 Core 的 `sendAiChatMessage()`。
- AI 系统提示已要求只能基于 Ami_Core 数据做解释、风险提示和下一步建议。
- 业务卡片优先展示，AI 建议跟在业务卡片之后。

这符合正确方向：AI 不生成客户、预约、库存、订单等事实。

当前缺口：

- AI 尚未作为统一 intent classifier 使用。
- 自然语言识别主要靠 `resolveAction()` 的本地关键词判断。
- 没有标准 Intent JSON 输出。
- 没有槽位抽取，例如客户名、手机号、预约时间、卡项、项目、金额、支付方式。
- 没有低置信度澄清流程。
- 没有对高风险动作的 AI 安全分级。
- 语音按钮只有 UI，没有语音转文本或实时语音能力。

### 2.4 三角色体验现状

当前支持角色：

- 店长：经营驾驶舱、员工排班、客户增长、库存预警。
- 前台：今日预约、核销、登记、收银、办卡、充值、打印。
- 美容师：我的预约、客户档案、服务记录、护理建议、完成服务。

已经具备的体验：

- 顶部状态栏显示门店、员工、角色切换、网络/打印机/扫码状态、锁屏。
- 底部智能命令栏显示角色化快捷按钮和文本输入。
- 中间消息流以业务卡片为主。
- 预约卡已补充客户名、手机号、会员、画像标签、状态和确认/改期/取消/到店操作。
- 核销、收银、办卡、登记、充值已经做成步骤式卡片。

问题：

- 多个组件中文已乱码，影响演示和真实使用。
- 快捷按钮虽然来自 role definition，但命令解释仍有本地硬编码。
- 部分角色权限仍是前端模拟，应该完全由 Core bootstrap 和 action capabilities 控制。
- 美容师场景还需要进一步从“查看卡片”深化为“服务前准备、服务中记录、服务后复购建议”的闭环。

### 2.5 业务流程现状

已具备雏形的流程：

- 预约：查看、确认、改期、取消、到店。
- 核销：选择客户、选择卡项项目、确认核销、成功结果。
- 收银：选择客户、选择项目/商品、输入优惠、选择支付方式、完成收银。
- 办卡：选择客户、选择次卡、优惠/赠送、开卡完成。
- 充值：选择客户、金额/赠送、充值完成。
- 登记：录入客户、模拟面部检测、生成客户信息卡、写入客户和皮肤检测。
- 打印：创建打印任务。
- 服务完成：完成服务任务。

需要强化的流程能力：

- 所有写操作增加幂等键，防止重复提交。
- 高风险动作二次确认：取消预约、大额优惠、核销、收银、充值。
- 操作成功后刷新相关卡片，而不是只追加结果。
- 接口失败时展示可理解原因和下一步，不显示技术错误码给一线人员。
- 业务步骤应可恢复，避免刷新后流程丢失。
- 预约改期不能默认加 30 分钟，应进入可选择时段。

## 3. 目标定位

### 3.1 产品定位

Ami Aura Lite 是 Ami_Core 之上的智能终端入口，不是另一个后台。

目标是让店长、前台、美容师通过“说一句话、点一个按钮、扫一下码、拍一张图”快速完成一线高频工作。

### 3.2 新模式定义

“意图驱动的即时美业微应用”指：

1. 用户表达业务目标，而不是寻找后台菜单。
2. 系统识别意图、抽取关键信息、判断权限和风险。
3. 系统调用 Ami_Core 获取确定性数据。
4. 系统即时生成当前任务需要的业务卡片和步骤流程。
5. 涉及提交时，系统要求用户确认，并由 Core 执行写回。
6. AI 负责解释、建议、话术和流程引导，不负责生成事实。

示例：

- 用户说“帮张三核销补水次卡”。
- 系统识别为 `card.consume`。
- 系统抽取客户名“张三”、卡项“补水次卡”。
- 系统调用 Core 搜索客户和可核销卡项。
- 系统展示核销确认卡。
- 用户确认后 Core 写回核销记录。
- 系统展示核销成功卡，并给出“预约下次护理”的建议。

## 4. 产品原则

1. Ami_Core 是唯一业务事实源。
2. Lite 不维护独立业务数据库。
3. Lite 不复制后台页面，只生成一线场景需要的即时卡片。
4. AI 不直接写业务数据，所有写操作都走 Core API。
5. AI 不编造客户、订单、预约、库存、员工业绩等事实。
6. 所有角色权限、门店范围、可用动作由 Core bootstrap 返回。
7. 写操作必须确认、可审计、可追踪、可失败兜底。
8. 一线人员看到的是业务语言，不是接口名、命令码或技术错误。
9. 首页与快捷入口必须快，慢数据卡片级加载。
10. 先保证确定性业务闭环，再增强 AI 体验。

## 6. 意图体系需求

### 6.1 Intent Schema

系统内部应使用统一 Intent 结构。

```json
{
  "intent": "card.consume",
  "role": "reception",
  "confidence": 0.92,
  "source": "text",
  "slots": {
    "customerName": "张三",
    "cardName": "补水次卡",
    "projectName": "巨补水",
    "times": 1
  },
  "riskLevel": "medium",
  "requiresConfirmation": true,
  "missingSlots": []
}
```

字段说明：

- `intent`：标准业务意图编码。
- `role`：当前执行角色。
- `confidence`：识别置信度。
- `source`：触发来源，支持 quick_action、text、voice、scan、camera。
- `slots`：槽位信息。
- `riskLevel`：风险等级，none、low、medium、high。
- `requiresConfirmation`：是否需要确认。
- `missingSlots`：缺失字段，用于追问。

### 6.2 意图命名

建议意图命名：

店长：

- `manager.dashboard.view`
- `manager.risk.view`
- `manager.staff.view`
- `manager.customer_growth.view`
- `manager.inventory.view`
- `manager.trend.view`

前台：

- `appointment.create`
- `appointment.confirm`
- `appointment.reschedule`
- `appointment.cancel`
- `appointment.check_in`
- `customer.search`

美容师：

- `customer.service_profile.view`
- `service_record.create`

通用：

### 6.3 意图识别策略

v1 采用“规则优先 + AI 补充”的混合策略：

1. 快捷按钮直接映射标准 intent，不走大模型。
2. 高频文本指令先走规则识别，保证稳定和低成本。
3. 模糊表达交给 AI 分类和槽位抽取。
4. AI 输出必须被校验到 command registry 中存在的 intent。
5. 低置信度时追问，不执行业务。
6. 涉及交易和取消的 intent 必须进入确认流程。

### 6.4 多轮澄清

当信息不足时，系统不能猜测，应追问一个明确问题。

示例：

- 用户：“给张三核销。”
- 系统：“张三有 2 张可用次卡，请选择要核销的卡项。”

- 用户：“帮她收银。”
- 系统：“请先选择客户，或从今日预约客户中选择。”

- 用户：“改一下时间。”
- 系统：“要修改哪位客户的预约？可从今日预约列表选择。”

## 7. 即时微应用需求

### 7.1 经营驾驶舱微应用

触发：

- 店长点击“经营”。
- 输入“今日经营”“今天怎么样”“本周业绩趋势”。

数据来源：

- Core role dashboard。
- 订单、预约、客户、库存、员工排班聚合。

展示：

- 营业额。
- 订单数。
- 预约数。
- 到店率。
- 客单价。
- 新增客户。
- 低库存。
- 临期商品。
- 员工负载。
- 重点风险。
- Ami 建议。

要求：

- 首屏 2 秒内展示骨架或关键指标。
- 指标标注数据来源和生成时间。
- AI 总结不超过 3 条。
- 无数据时显示“暂无今日经营数据”。

### 7.2 今日预约微应用

触发：

- 前台点击“预约”。
- 输入“今日预约”“今天谁要来”。

展示字段：

- 客户名称。
- 手机号。
- 预约时间。
- 预约项目。
- 美容师。
- 时长。
- 会员等级。
- 可用卡项摘要。
- 画像标签。
- 最近到店。
- 状态。

操作：

- 确认预约。
- 修改时间。
- 取消预约。
- 确认到店。
- 进入核销。
- 进入收银。

要求：

- 默认按时间排序。
- 待确认、即将到店、已到店分组。
- 改期需选择可用时段，不能默认加 30 分钟。
- 取消需二次确认并记录原因。
- 到店后刷新核销和收银候选客户。

### 7.3 次卡核销微应用

触发：

- 前台点击“核销”。
- 输入“核销次卡”“帮张三核销补水卡”。

流程：

1. 选择客户：支持搜索，默认优先显示当天预约客户。
2. 选择次卡核销内容：展示卡名、剩余次数、有效期、适用项目。
3. 确认核销：展示客户、卡项、项目、次数、美容师、剩余次数。
4. 核销完成：展示成功信息卡和时间戳。

规则：

- 不显示 `operation.verify` 等技术命令。
- 不显示“核销已提交”的中间冗余卡片。
- 次数不足、已过期、项目不适用时禁止提交。
- 成功后刷新客户卡项、经营数据、服务任务。
- 写操作带幂等键。

### 7.4 收银结账微应用

触发：

- 前台点击“收银”。
- 输入“给李四开单”“张三收银”。

流程：

1. 开单：选择客户，选择项目/商品，支持多选和数量调整。
2. 收款：输入优惠金额，选择支付方式。
3. 完成：展示收银成功卡、订单号、应收、优惠、实收、支付方式、时间戳。

规则：

- 优惠金额不能大于应收金额。
- 支付方式来自 Core 字典。
- 大额优惠需二次确认或店长授权。
- 成功后刷新经营数据、客户消费记录、库存消耗。

### 7.5 办卡开单微应用

触发：

- 前台点击“办卡”。
- 输入“给张三办补水次卡”。

流程：

1. 选择客户。
2. 选择次卡。
3. 输入优惠金额和赠送项目。
4. 确认开卡。
5. 开卡完成。

规则：

- 卡项价格、次数、有效期继承 Core。
- 优惠和赠送项目写入订单备注或结构化字段。
- 成功后刷新客户卡项。
- 如客户不存在，允许转入客户登记流程。

### 7.6 充值微应用

触发：

- 前台点击“充值”。
- 输入“给李四充值 1000”。

流程：

1. 选择客户。
2. 输入充值金额、赠送金额、赠送项目。
3. 选择支付方式。
4. 确认充值。
5. 充值完成。

规则：

- 金额字段继承 Core 校验。
- 成功后更新客户余额和交易记录。
- 大额赠送需二次确认。

### 7.7 新客登记与面部检测微应用

触发：

- 前台点击“登记”。
- 输入“新增客户”。

流程：

1. 录入用户信息：姓名、手机号、性别、生日、来源、备注等字段继承 Ami_Core。
2. 面部检测：调用摄像头采集，生成检测记录。
3. 生成用户信息卡：展示基础资料、皮肤状态、初步标签、护理建议。
4. 登记完成：写入客户档案和皮肤检测记录。

规则：

- 摄像头必须由用户授权。
- 手机号重复时进入已有客户合并或查看流程。
- 面部检测结果必须标注为“护理建议”，不能等同医疗诊断。
- 摄像头不可用时允许跳过检测完成基础登记。

### 7.8 美容师服务微应用

触发：

- 美容师点击“我的预约”“客户档案”“服务记录”“护理建议”“完成服务”。
- 输入“我的下一个客户”“张三皮肤情况”“记录本次服务”。

展示：

- 我的今日服务排程。
- 下一个客户卡片。
- 客户偏好、禁忌、过敏、皮肤状态。
- 历史服务项目。
- 本次服务记录入口。
- 护理建议。
- 下次预约建议。

操作：

- 开始服务。
- 记录服务。
- 完成服务。
- 转前台收银。
- 预约下次。

规则：

- 美容师只看本人服务相关客户。
- 不展示门店财务和全员绩效。
- 护理建议来自 Core 数据 + AI 解释，必须标注建议来源。

## 8. AI 能力需求

### 8.1 AI 应承担的能力

- 自然语言意图分类。
- 槽位抽取。
- 低置信度澄清问题生成。
- 经营总结。
- 客户摘要。
- 护理建议。
- 接待话术。
- 服务记录摘要。
- 下一步行动建议。
- 语音转文本。
- 后续实时语音对话。

### 8.2 AI 不允许承担的能力

- 不编造客户、预约、订单、卡项、库存、员工等事实。
- 不绕过 Core 权限。
- 不直接完成支付、核销、取消预约等高影响动作。
- 不输出医疗诊断。
- 不将建议包装为确定事实。
- 不在前端保存模型 Key。

### 8.3 AI 编排方式

推荐链路：

```text
用户输入
  -> Lite command router
  -> 规则识别
  -> AI 意图分类兜底
  -> Core permission/action 校验
  -> Core data/service 调用
  -> Lite 业务卡片渲染
  -> AI 解释/建议卡片
```

### 8.4 模型能力结合方向

结合当前 AI 技术趋势，Ami Aura Lite 应优先采用：

- 工具调用：让 AI 输出标准 intent 和 slots，再由系统调用 Core API。
- 结构化输出：要求 AI 返回严格 JSON，便于校验和审计。
- 实时语音：用于门店一线免手操作，先从语音转文本开始，再升级低延迟语音对话。
- 多模态：用于面部检测、图片记录、护理建议解释。
- Agent 编排：用于多步骤任务引导，但高风险动作仍由用户确认。
- 观测与追踪：记录意图识别、接口调用、AI 建议和用户采纳情况。

## 10. 权限与安全需求

### 10.1 权限来源

Core 是唯一权限来源。Lite 不自行推断完整权限。

Lite 只消费：

- `availableRoles`
- `availableActions`
- `permissions`
- `dataScopes`
- `deviceCapabilities`

### 10.2 角色数据范围

店长：

- 可查看门店经营、客户、预约、订单、员工、库存。
- 可发起或分派跟进。
- 可协助交易，但交易入口不抢占首页。

前台：

- 可处理接待、预约、核销、收银、办卡、充值、登记、打印。
- 不默认展示完整经营报表和员工绩效。

美容师：

- 只看本人预约、服务相关客户、服务记录、护理建议。
- 不可见门店完整财务和全员绩效。

### 10.3 隐私与合规

- 客户手机号可按角色脱敏。
- 语音和摄像头采集必须明确授权。
- 皮肤检测不是医疗诊断。
- AI 上下文只传任务必要的最小数据。
- AI 建议和业务事实必须视觉区分。

## 11. 性能与稳定性需求

### 11.1 性能目标

- 应用首屏骨架 1 秒内出现。
- 角色首页关键卡片 2 秒内可见。
- 查询类微应用 2 秒内给出数据或明确 loading。
- 写操作提交后 3 秒内返回结果或可理解错误。
- AI 建议不阻塞业务卡片展示。

### 11.2 接口策略

- 首页优先调用 role dashboard 聚合接口。
- 微应用使用专用轻量接口，不默认调用 `loadCoreSnapshot()`。
- 大列表必须分页、搜索、懒加载。
- 客户、卡项、项目、支付方式等可短缓存。
- 每张业务卡片单独 loading 和错误兜底。

### 11.3 降级策略

Core 不可用：

- 显示离线提示。
- 禁止交易写回。
- 可保留只读缓存，但必须标注缓存时间。

AI 不可用：

- 保留快捷按钮和确定性业务流程。
- 隐藏 AI 建议或显示“AI 建议暂不可用”。

摄像头不可用：

- 允许跳过面部检测完成基础登记。

打印机不可用：

- 允许稍后补打。

- AI 不编造事实。
- AI 建议不阻塞业务流程。

### 阶段五：语音与多模态

目标：形成门店一线更自然的免手操作入口。

## 17. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 中文乱码继续存在 | 指令识别和演示体验不可用 | 第一阶段优先全量修复编码和文案 |
| `loadCoreSnapshot()` 过重 | 加载慢、失败率高 | 改为专用轻量接口和 role dashboard |
| AI 误判意图 | 误触发流程 | 规则优先、低置信度追问、高风险确认 |
| Core 接口不完整 | 微应用不能闭环 | 先补 Core mock/real 双实现 |
| 权限模型变化 | 按钮和数据错乱 | Lite 只消费 bootstrap capabilities |
| 交易重复提交 | 产生重复核销或订单 | 写接口加幂等键和按钮防重复 |
| 门店网络不稳定 | 现场体验差 | 卡片级重试、只读缓存、交易禁写 |
| 摄像头/打印机不可用 | 登记/打印流程中断 | 设备状态前置展示，支持跳过和补打 |
| AI 成本升高 | 商业化压力 | 规则优先、结果缓存、按场景选模型 |

## 19. 参考资料

以下资料用于判断 AI 技术方向，最终实现仍以 Ami_Core 能力和项目实际接口为准：

- OpenAI Realtime API：低延迟、多模态、语音交互能力方向。  
  https://platform.openai.com/docs/guides/realtime
- OpenAI Function Calling / Structured Outputs：工具调用和结构化输出适合 intent + slots。  
  https://platform.openai.com/docs/guides/function-calling
- OpenAI Agents SDK：Agent 编排、工具、追踪、handoff 等能力方向。  
  https://platform.openai.com/docs/guides/agents-sdk
- Anthropic Tool Use：模型调用外部工具/API 的能力方向。  
  https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/overview
- Model Context Protocol：将模型与外部工具、数据源连接的通用协议方向。  
  https://modelcontextprotocol.io/docs
