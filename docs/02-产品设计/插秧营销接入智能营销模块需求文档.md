# 插秧营销接入智能营销模块需求文档

版本：v1.0  
日期：2026-06-26  
适用范围：Ami_Core 管理端智能营销、server-v2 营销服务、推广资产、Ami Aura Lite 终端跟进、Ami Glow 小程序、营销增长 Agent  
关联文档：`docs/02-产品设计/美业插秧营销需求文档.md`、`docs/03-开发计划/智能营销阶段1-2简化改造详细开发计划.md`

---

## 1. 结论

插秧营销不建议做成新的独立一级模块，也不建议另起一套营销系统。当前智能营销已经形成较完整的业务骨架：

```text
营销工作台 -> 智能推荐 / 活动列表 / 自动触达 / 推广资产 / 数据复盘
```

因此，插秧营销应作为“智能营销工作台里的场景化经营模式”加入，核心做法是：

```text
客户田块 -> 插秧策略 -> 推广资产/自动触达/终端跟进 -> 到店承接 -> 数据复盘/补苗
```

产品上新增的是“客户田块、今日插秧、补苗建议、插秧策略草稿、插秧漏斗复盘”这几层业务表达；底层尽量复用现有 `MarketingRecommendation`、`MarketingActivity`、`MarketingAutomationStrategy`、`MarketingPage`、`Promotion`、`TerminalFollowUpTask` 和统一效果接口。

## 2. 当前系统现状

### 2.1 当前菜单与路由

当前智能营销主菜单已收敛为：

| 菜单 | 路由 | 文件 | 当前职责 |
| --- | --- | --- | --- |
| 营销工作台 | `/customer-marketing/workbench` | `src/app/pages/MarketingWorkbench.tsx` | 汇总推荐机会、活动、自动触达、终端跟进、数据复盘 |
| 智能推荐 | `/customer-marketing/intelligent-recommendation` | `src/app/pages/MarketingRecommendation.tsx` | 查看推荐卡、客户受众、发活动、开自动触达 |
| 活动列表 | `/customer-marketing/activity-management` | `src/app/pages/MarketingStrategy.tsx` | 管理单次营销活动 |
| 自动触达 | `/customer-marketing/automation` | `src/app/pages/CreateMarketing.tsx` | 管理自动营销策略、规则、执行和 AI 文案 |
| 推广资产 | `/customer-marketing/assets` | `src/app/pages/MarketingAssets.tsx` | 聚合推广页、权益资产库、Ami Glow 展示、行为数据 |
| 数据复盘 | `/customer-marketing/effect-analysis` | `src/app/pages/MarketingAnalytics.tsx` | 统一复盘活动、自动触达、页面、权益、小程序、推荐和终端跟进 |

仍保留兼容路由：

- `/customer-marketing`
- `/customer-marketing/pages`
- `/customer-marketing/promotions`
- `/customer-marketing/ami-glow`
- `/customer-marketing/rule-library`
- `/customer-marketing/activity-effect/:id`
- `/customer-marketing/strategy-templates`

### 2.2 当前工作台能力

`MarketingWorkbench` 当前已具备插秧营销的雏形：

| 能力 | 当前实现 | 对插秧营销的意义 |
| --- | --- | --- |
| 推荐机会摘要 | `getMarketingRecommendations()` | 可作为“今日该插哪块田”的机会来源 |
| 推荐客户名单 | `getMarketingRecommendationAudience(id)` | 可作为客户田块名单 |
| 终端跟进下发 | `batchCreateRecommendationFollowUpTasks()` | 可作为“员工养护任务” |
| 自动触达摘要 | `getAutomationStrategiesPaginated()` | 可作为自动插秧运行状态 |
| 活动摘要 | `getMarketingActivities()` | 可作为单次插秧战役 |
| 数据摘要 | `getUnifiedMarketingEffects()` | 可作为插秧结果复盘 |

当前缺口：

- 推荐机会仍按“推荐卡”组织，没有显式客户田块视图。
- 工作台有“下发终端跟进”，但没有“今日插秧任务”心智。
- 推荐、自动触达、活动、推广页和权益之间没有统一的“插秧策略”上下文。
- 数据复盘已有对象汇总，但没有按客户田块拆漏斗和补苗建议。

### 2.3 当前推广资产能力

`MarketingAssets` 当前已经把资产聚合为：

| Tab | 文件/组件 | 对插秧营销的作用 |
| --- | --- | --- |
| 推广页 | `MarketingPageManagement` | 承接活动 H5、预约、留资和分享 |
| 权益资产库 | `PromotionManagement` | 提供优惠、赠品、体验、会员权益等营销种子 |
| 小程序展示 | `AmiGlowManagement` | 承接老客小程序入口和推荐位 |
| 数据明细 | `CustomerAppEventTable` | 查看客户浏览、点击、预约行为 |

当前缺口：

- 资产是按类型管理，不是按插秧目标组合管理。
- 权益、页面、推荐位、话术还没有组成“营销种子包”。
- 页面/权益和客户田块之间缺少适配推荐。

### 2.4 当前自动触达能力

`CreateMarketing` 当前支持：

- 创建、保存草稿、编辑、启用、暂停、删除自动策略。
- 触发规则：生日、护理周期、卡项到期、沉睡、复购、转介绍等。
- 动作：短信、小程序、微信、社群、门店话术、优惠权益。
- AI 文案生成，并过滤内部营销/算法术语。
- 从推荐来源带入 `sourceRecommendationId`、`predictionRunId`、`targetAudience`、`offer`、`sourceSignals` 等上下文。

当前缺口：

- 规则语言偏“触发条件”，店长不容易理解为“插秧节奏”。
- 自动触达不直接展示“客户田块 -> 营销种子 -> 养护动作”的完整策略。
- 缺少策略执行后的补苗动作建议。

### 2.5 当前数据复盘能力

`MarketingAnalytics` 当前通过 `getUnifiedMarketingEffects()` 聚合：

| 类型 | 说明 |
| --- | --- |
| activity | 推广活动 |
| auto | 自动触达 |
| page | 推广页 |
| promotion | 权益资产 |
| recommendation | 智能推荐 |
| glow | Ami Glow |

同时已读取 `getMarketingFollowUpTaskSummary()`，展示终端跟进的待处理、处理中、完成、逾期、预约/成交和收入。

当前缺口：

- 复盘按推广对象类型聚合，不按“客户田块”和“插秧策略”聚合。
- 没有明确“补苗池”：已触达未打开、已打开未预约、已预约未到店、已到店未成交。
- 员工跟进结果和客户田块转化之间的关系还不够显性。

### 2.6 当前后端与数据模型

当前可复用主对象：

| 对象 | 现状字段/能力 | 插秧营销复用方式 |
| --- | --- | --- |
| `MarketingActivity` | 活动、`pageSchema`、推荐来源、预测批次、受众快照、权益、推荐项目 | 承接单次插秧战役 |
| `MarketingAutomationStrategy` | 规则、动作、来源、排期、目标人数 | 承接长期自动插秧 |
| `MarketingAutomationTouch` | 客户触达、渠道、状态、预测分、转化 | 承接触达流水 |
| `MarketingAttribution` | 自动触达到订单归因 | 承接自动插秧转化 |
| `MarketingPage` | H5/小程序页、来源对象、分享、二维码 | 承接推广页种子 |
| `MarketingPageEvent` / `Lead` / `Attribution` | 浏览、点击、留资、预约、订单归因 | 承接页面漏斗 |
| `Promotion` | 权益类型、适用范围、毛利守护、审批、成本 | 承接权益种子 |
| `RecommendationEvent` | 终端推荐事件 | 承接终端动作反馈 |
| `TerminalFollowUpTask` | 员工跟进任务、负责人、结果、预约、订单 | 承接员工养护 |
| `CustomerPredictionSnapshot` | 流失、复购、营销响应、LTV、推荐动作 | 承接客户分田输入 |

结论：MVP 不必先新建完整独立表，可先在现有活动、自动触达、任务和页面对象里通过 `source` / `attribution` / `metadata` / `payload` 携带插秧上下文。进入稳定期后，再新增独立的插秧策略表。

## 3. 产品定位

### 3.1 插秧营销在当前系统中的位置

建议定位为智能营销的场景模式：

```text
智能营销
├── 营销工作台
│   ├── 今日机会
│   ├── 客户田块
│   ├── 今日插秧
│   ├── 待养护
│   └── 补苗建议
├── 智能推荐
├── 活动列表
├── 自动触达
├── 推广资产
└── 数据复盘
```

菜单上有两种可选方案：

| 方案 | 做法 | 建议 |
| --- | --- | --- |
| 方案 A | 不新增菜单，在营销工作台增加“插秧营销”Tab | 推荐，理解成本最低 |
| 方案 B | 在营销工作台下新增二级场景入口 `/customer-marketing/workbench?mode=planting` | 可做，适合逐步增强 |
| 方案 C | 新增左侧菜单“插秧营销” | 不推荐，会重新拉高菜单复杂度 |

推荐采用方案 A：插秧营销是工作台里的经营方法，不是另一套系统。

### 3.2 产品心智

面向店长/运营的表达：

```text
今天该经营哪批客户？
系统已经把客户分成几块田。
你选择种子，系统帮你插下去。
员工负责养护，到店和成交自动回流。
没转化的客户进入补苗池。
```

对应系统对象：

| 产品语言 | 系统对象 |
| --- | --- |
| 客户田块 | 推荐受众、客户预测、客户标签、订单/预约/卡项状态 |
| 种子 | 权益、推广页、小程序推荐位、话术、活动内容 |
| 插秧 | 活动、自动触达、终端跟进任务 |
| 养护 | `TerminalFollowUpTask`、预约提醒、服务后回访 |
| 收割 | 预约、到店、核销、订单、转介绍 |
| 补苗 | 二次触达、换权益、人工电话、再次派单 |
| 复盘 | 统一效果、页面归因、自动归因、终端任务结果 |

## 5. 业务流程

### 5.1 主流程

```text
1. 工作台识别机会
   -> 2. 客户分田
   -> 3. 选择种子包
   -> 4. 生成插秧策略草稿
   -> 5. 选择执行方式
      -> 发一次活动
      -> 开启自动触达
      -> 下发终端跟进
      -> 组合执行
   -> 6. 执行与养护
   -> 7. 到店承接
   -> 8. 数据复盘
   -> 9. 补苗与模板沉淀
```

### 5.2 客户分田流程

客户田块来源优先复用当前推荐与预测能力：

| 田块 | 推荐来源 | 规则来源 | 首期执行方式 |
| --- | --- | --- | --- |
| 沉睡补苗田 | 流失风险、最近未到店 | `dormant`、`last_visit`、`visit_gap` | 终端跟进 + 自动触达 |
| 复购成熟田 | 复购分、护理周期到期 | `care_cycle`、`project_cycle_due`、`consumption` | 自动触达 + 小程序 |
| 新客待激活田 | 新客、留资未到店、首单未复购 | `new_customer`、`booking_abandonment` | 活动页 + 前台任务 |
| 疗程消耗田 | 卡项剩余、套餐剩余 | `package_remaining`、`card_expiry` | 终端跟进 + 预约提醒 |
| 转介绍种子田 | 高 LTV、高响应、近期到店 | `referral_campaign`、`member_level` | 活动页 + 推荐权益 |
| 高价值维护田 | LTV 高、会员等级高、复购稳定 | `vip_privilege_care`、`member_level` | 人工关怀 + 小程序展示 |

### 5.3 策略生成流程

用户在工作台选择一个客户田块后，进入“生成插秧策略”：

| 步骤 | 页面表达 | 当前可复用能力 |
| --- | --- | --- |
| 选择客户田块 | 这批客户是谁、为什么值得经营 | 推荐受众、预测客户、客户标签 |
| 选择种子包 | 用什么权益、页面、话术和推荐项目 | Promotion、MarketingPage、Ami Glow、AI 文案 |
| 选择插秧方式 | 发活动、自动触达、终端跟进或组合执行 | MarketingActivity、MarketingAutomationStrategy、TerminalFollowUpTask |
| 预估效果 | 预计触达、预约、收入、成本、风险 | AudiencePreview、推荐预估、权益成本 |
| 确认执行 | 保存草稿、启动、派发、发布 | 现有创建/启用/发布/下发接口 |

### 5.4 执行方式

| 执行方式 | 使用场景 | 当前落点 | 数据回流 |
| --- | --- | --- | --- |
| 发一次活动 | 199 拓客、节日活动、新客激活、转介绍 | `MarketingActivity` + `MarketingPage` | 页面事件、留资、预约、订单归因 |
| 开启自动触达 | 复购、生日、卡项到期、沉睡唤醒 | `MarketingAutomationStrategy` | execution、touch、attribution |
| 下发终端跟进 | 高价值客户、沉睡补苗、顾问解释型场景 | `TerminalFollowUpTask` | 任务状态、预约、订单、收入 |
| 小程序展示 | 老客经营、会员权益、预约入口 | Ami Glow 配置 | 客户行为事件、预约 |
| 组合执行 | 高价值活动或重点田块 | 以上组合 | 统一效果复盘 |

### 5.5 补苗流程

补苗池从数据复盘和任务结果生成：

| 补苗池 | 判断方式 | 推荐动作 |
| --- | --- | --- |
| 未触达 | 发送失败、联系方式缺失、任务未派发 | 转员工电话、更新联系方式 |
| 未打开 | H5/小程序无打开，短信/微信无反馈 | 换标题、换渠道、换发送时间 |
| 未预约 | 已打开/点击但未预约 | 员工私聊、简化预约入口 |
| 未到店 | 已预约未到店或改约失败 | 前台确认、改约、到店提醒 |
| 未成交 | 到店未下单、核销未转卡 | 顾问复盘、换权益、二次护理方案 |
| 未复购 | 首单/体验后无二次消费 | 服务后回访、护理计划、复购提醒 |

补苗不是新建一套触达系统，而是把客户重新送回：

- 自动触达草稿。
- 终端跟进任务。
- 推广页/权益调整。
- 活动二次触达。

## 6. 功能需求

### 6.1 营销工作台改造

在 `MarketingWorkbench` 增加插秧营销区域。

#### 6.1.1 新增视图

| 视图 | 内容 | 数据来源 |
| --- | --- | --- |
| 客户田块 | 沉睡补苗、复购成熟、新客待激活、疗程消耗、转介绍种子、高价值维护 | 推荐、预测、客户行为 |
| 今日插秧 | 今日推荐执行的策略、客户数、建议动作、负责人 | 推荐机会 + 自动触达 + 终端任务 |
| 待养护 | 待跟进、处理中、逾期、已预约、已成交 | `getMarketingFollowUpTaskSummary()` |
| 补苗建议 | 未响应、未预约、未到店、未成交客户池 | 统一效果 + 任务结果 |

#### 6.1.2 新增按钮

| 按钮 | 跳转/动作 |
| --- | --- |

## 12. 验收标准

### 12.1 产品验收

- 店长进入营销工作台，可以看到客户田块和今日插秧建议。
- 从一个客户田块可以生成插秧策略草稿。
- 插秧策略可以选择发活动、开自动触达、下发终端跟进。
- 下发终端跟进后，任务中能看到客户田块、推荐原因、权益和话术。
- 自动触达和活动能带上插秧来源。
- 数据复盘能看到插秧策略带来的触达、预约、到店、成交和待补苗。

### 12.2 技术验收

- 不破坏当前 `/customer-marketing/*` 路由。
- 不新增孤立营销对象导致活动、页面、权益、自动触达重复。
- 插秧上下文可通过现有字段回溯来源。
- `getUnifiedMarketingEffects` 仍是主复盘入口。
- 工作台任一摘要接口失败，不影响其他区块展示。

### 12.3 数据验收

- 客户田块必须来自真实客户、预测、预约、订单、卡项或营销行为数据。
- 终端任务必须能回写完成、预约、成交或未成交原因。
- 活动、页面、自动触达和任务结果能回到数据复盘。
- AI 建议必须展示依据，不能只输出结论。

## 13. 推荐落地顺序

优先做“沉睡补苗田”和“复购成熟田”两个场景：

1. 两者当前已有触发类型支撑：`dormant`、`last_visit`、`visit_gap`、`care_cycle`、`project_cycle_due`。
2. 两者适合复用自动触达和终端跟进，不依赖外部广告渠道。
3. 两者容易用预约、到店、订单和跟进结果验证 ROI。
4. 两者最符合门店“每天该联系谁”的高频使用场景。

第一版完成后，再扩展新客待激活、疗程消耗、转介绍种子和高价值维护。
