# Agent 评测 310 条失败项改进需求文档

版本：v1.0  
日期：2026-06-29  
适用产品：洞悉美业·新一代门店运营智能体、Ami_Agent、Ami Aura Lite 智能终端问答  
输入依据：

- `docs/04-测试数据/agent-eval-questions.md`
- `docs/04-测试数据/agent-eval-remaining-supported-report.json`
- `docs/04-测试数据/agent-eval-test-plan.md`

---

## 1. 背景

本轮评测基于 650 条美业门店自然语言问题，先剔除当前系统整体不支持的业务或数据，再对剩余未测问题做 Planner / Skill / Tool / 输出契约评测。

本轮要解决的核心问题不是“再加关键词规则”，而是让 Agent 真正具备可交付的业务理解能力：

1. 能识别问题属于哪个业务域。
2. 能选择正确 Persona / Skill / Tool。
3. 能基于业务数据给出可核对结论。
4. 能按场景输出 KPI、表格、行动卡、证据来源。
5. 对权限、风险、缺数据场景有明确承接方式。

---

## 2. 本轮测试结论

### 2.1 总体结果

| 指标 | 数量 | 说明 |
|---|---:|---|
| 原始问题 | 650 | 来自 Agent 评测问题库 |
| 系统不支持剔除 | 22 | 当前系统无业务对象、页面/API 或数据闭环 |
| 已测覆盖 | 128 | P0 Daily 119，Kiosk E2E 9 |
| 剩余实际评测 | 500 | 全部属于系统支持或 Agent 应补齐范围 |
| 通过 | 190 | 当前 Planner / Skill / Tool / 输出契约满足要求 |
| 失败/待补齐 | 310 | 系统有业务或可承接，但 Agent 能力不足 |
| 剩余问题通过率 | 38.0% | 不满足上线质量目标 |

### 2.2 重要判断

310 条失败项不能简单理解为“系统没有业务”。其中：

| 类型 | 数量 | 产品含义 |
|---|---:|---|
| `system_supported_testable` | 290 | 系统已有业务、数据或 API，但 Agent 没答好 |
| `system_supported_agent_gap` | 20 | 系统可承接，但 Agent 缺少营销文案、活动设计、自动化、ROI 等高阶 Skill |

因此后续工作重点应放在 Agent 能力补齐，而不是把问题剔除出评测。

## 3. 失败分布分析

### 3.1 按角色

| 角色 | 剩余评测 | 通过 | 失败 | 通过率 | 结论 |
|---|---:|---:|---:|---:|---|
| 店长经营 | 71 | 43 | 28 | 60.6% | 相对最好，但仍有经营趋势、异常洞察、目标进度等缺口 |
| 营销增长 | 84 | 27 | 57 | 32.1% | 活动策划、话术、自动化、ROI 缺口明显 |
| 前台接待 | 80 | 21 | 59 | 26.3% | 当前最弱，客户查询、预约、收银核销、现场协调都不稳 |
| 美容师服务 | 83 | 31 | 52 | 37.3% | 今日安排、护理建议、服务记录、复购建议缺口明显 |
| 库存采购 | 80 | 31 | 49 | 38.8% | 补货、采购计划、安全库存、供应链问题缺口明显 |
| 财务风控 | 83 | 31 | 52 | 37.3% | 收入对账、退款折扣、财务风险输出不完整 |
| Edge Case | 19 | 6 | 13 | 31.6% | 多轮、跨域、模糊输入承接不足 |

优先级建议：

1. P0：前台接待、营销增长、美容师服务。
2. P1：财务风控、库存采购。
3. P2：店长经营和 Edge Case 优化。

### 3.2 按失败原因

| 失败原因 | 次数 | 问题解释 | 交付影响 |
|---|---:|---|---|
| `missing_output_kind` | 246 | 没有输出应有的 KPI、表格、图表或行动卡 | 前端难以渲染成可操作卡片，用户觉得答非所问 |
| `skill_missing` | 243 | Planner 没稳定命中 Skill / Capability | 系统有业务，但 Agent 没找到处理路径 |
| `missing_evidence` | 243 | 缺数据来源或证据块 | 用户无法信任结果，不能用于经营决策 |
| `wrong_intent` | 233 | 查询、分析、草稿、澄清判断不准 | 该直接回答时反问，该生成草稿时只查询 |
| `route_error` | 91 | 业务域识别不准 | 问收银答客户，问预约答经营 |
| `permission_error` | 10 | 权限边界承接不清 | 不知道是无权限、需转交还是需要补充身份 |
| `tool_missing` | 2 | Planner 命中未注册工具 | 工具注册表与 Skill 定义漂移 |

根因判断：

1. 当前 Agent 评测失败主要不是模型表达问题，而是任务规划与输出契约没有形成硬约束。
2. 系统已有很多业务工具，但 Planner 对业务口语表达的 Skill 命中不足。
3. Evidence 没有作为业务问答的默认要求，导致结果无法被审计。
4. 高价值场景没有专属 Skill 包，例如前台客户查询、收银核销、营销自动化、采购计划、财务退款折扣。

|---|---:|---|---|

| 话术与内容生成 | 14 | 私信、朋友圈、生日祝福、短视频脚本 | 营销内容生成 Skill |
| 自动化与触达规则 | 13 | 自动跟进、自动感谢、自动召回 | 自动化规则草稿和确认卡 |

## 4. 系统不支持边界

以下 22 条被剔除，不计入 Agent 失败率。原因是当前 Ami_Core 系统整体没有对应业务闭环。

| 不支持范围 | 数量 | 当前处理建议 |
|---|---:|---|
| 客户投诉、差评、满意度或评价反馈 | 9 | 回复数据缺口，提示后续需新增评价/投诉模块 |
| 服务事故、皮肤过敏、不良反应记录 | 5 | 回复系统未记录，不编造过敏史 |
| 消防安全、税务、发票、纳税 | 4 | 回复超出现有业务范围 |
| 等待离店或排队流失记录 | 1 | 可给通用建议，但不能声称有记录 |
| 员工离职交接或客户带走 | 1 | 可提示缺少离职交接闭环 |
| 供应商涨价通知 | 1 | 有采购数据，但没有涨价通知对象 |
| 美容设备维保或故障 | 1 | 有终端诊断，但无美容设备维保 |

需求边界：

1. 本轮不要求新增上述系统业务模块。
2. Agent 必须能识别这些问题不具备事实数据源，并给出“当前系统暂无数据”的透明说明。
3. 不允许把系统不支持问题误归为 Agent 技能缺失。

## 5. 改进目标

### 5.1 阶段性质量目标

| 阶段 | 覆盖范围 | 当前 | 目标 |
|---|---|---:|---:|
| M1 | 剩余 500 条 Planner / Skill / Tool / 输出契约 | 38.0% | >= 65% |
| M2 | 前台、营销、美容师三大弱项 | 约 31.9% | >= 70% |
| M3 | 全部剩余支持问题 | 38.0% | >= 80% |
| M4 | P0 Daily + Kiosk 核心自然语言 | 已通过 | 持续 100% |

### 5.2 产品体验目标

1. 用户问清单类问题时，必须输出表格，不再只有文字。
2. 用户问指标类问题时，必须输出 KPI 或明确数值，不再反问“你想查询还是分析”。
3. 用户问建议/方案类问题时，必须输出可执行步骤或行动卡。
4. 用户问业务事实时，必须附带数据来源或证据说明。
5. 用户问权限外问题时，必须说明权限边界和可转交路径。
6. 用户问系统无业务闭环的问题时，必须透明说明缺口，不编造。

## 6. 功能需求

### R1：输出契约强约束

优先级：P0  
覆盖失败：`missing_output_kind` 246 次

需求说明：

Planner 输出必须带 `requiredOutputKinds`，Runtime 和前端渲染必须按契约校验。

规则：

| 用户问题类型 | 必须输出 |
|---|---|
| 多少、金额、营业额、收入、利润、客单价、完成率 | `kpi` + `evidence` |
| 哪些、列出、清单、明细、所有、排名 | `table` + `evidence` |
| 趋势、最近几天、对比、变化 | `chart` 或趋势表 + `evidence` |
| 生成、设置、提醒、发券、自动化、退款、采购 | `action_card` + 审批/确认状态 |
| 模糊输入 | `clarify`，且最多追问一个关键问题 |

验收标准：

1. 310 条失败项中 `missing_output_kind` 下降 70% 以上。
2. 表格表头必须是中文业务字段，不得出现 `0/1/2/beauticianId` 这类内部字段。
3. Action Card 必须包含动作标题、影响对象、预览、确认状态。

### R2：Evidence 默认注入

优先级：P0  
覆盖失败：`missing_evidence` 243 次

需求说明：

所有业务查询、分析、建议类回答必须带数据来源说明。

Evidence 至少包含：

1. 数据源表或业务对象。
2. 时间范围。
3. 门店范围。
4. 关键筛选条件。
5. 数据更新时间或统计口径。

示例：

```json
{
  "kind": "evidence",
  "sources": ["ProductOrder", "PaymentRecord"],
  "storeScope": "Ami 全量演示门店",
  "timeRange": "2026-06-01 至 2026-06-29",
  "filters": ["status=paid", "refundExcluded=false"]
}
```

验收标准：

1. 财务、库存、客户、预约、服务、营销回答 Evidence 覆盖率 >= 90%。
2. 没有数据源的问题必须输出 `data_gap`，不能空泛回答。
3. 管理端和 Kiosk 均能展示“数据来源”。

### R3：Skill 命中能力补齐

优先级：P0  
覆盖失败：`skill_missing` 243 次

需求说明：

围绕失败 Top 分类新增或强化 Skill 包，不靠单条关键词补丁。

需要补齐的 Skill：

| Skill | 覆盖场景 | 主要输出 |
|---|---|---|
| `reception.customer.profile` | 客户查询、会员等级、卡项权益、历史项目、退款欠款 | 客户 360 卡片、表格、证据 |
| `reception.appointment.timeline` | 今日预约、待确认、空档、指定美容师排班 | 时间轴、预约表、行动建议 |
| `reception.cashier.checkout` | 支付方式、会员折扣、核销、收银、退款处理 | 明细表、确认卡 |
| `beautician.daily.service` | 今日客人、空档、结束时间、下个客人资料 | 服务日程卡 |
| `beautician.care.recommendation` | 护理建议、项目升级、复购推荐 | 建议卡、依据 |
| `beautician.service.followup` | 服务记录、特殊需求、后续跟进任务 | 草稿卡、跟进任务 |
| `marketing.copywriting` | 私信、朋友圈、生日祝福、短视频脚本 | 文案草稿、多版本 |
| `marketing.campaign.planning` | 老带新、节日、周年、线上引流 | 活动方案卡 |
| `marketing.automation.rule` | 自动召回、自动感谢、服务后提醒 | 自动化规则草稿 |
| `marketing.roi.analysis` | 优惠券核销、活动投入回报、客户质量 | KPI + 漏斗/表格 |
| `inventory.purchase.planning` | 补货计划、采购量、安全库存 | 采购建议卡 |
| `inventory.safety.stock.audit` | 安全库存线是否合理 | 风险表、调整建议 |
| `finance.income.reconcile` | 现金/微信/支付宝、日周月收入 | KPI + 对账表 |
| `finance.refund.discount.audit` | 大额退款、异常折扣、审批人 | 审计表、风险卡 |

验收标准：

1. 每个 Skill 至少覆盖 5 条评测问题。
2. Planner 返回 capabilityId，不再出现业务域已识别但 Skill 为空。
3. 新增 Skill 必须有对应单测和评测样本。

### R4：意图判断和澄清策略改造

优先级：P0  
覆盖失败：`wrong_intent` 233 次

需求说明：

减少过度澄清。对低风险、信息足够的问题，Agent 应直接回答，并声明默认口径。

规则：

1. 含明确时间范围和指标的问题，直接 query。
2. 含“为什么、原因、怎么处理”的问题，进入 analysis_and_recommendation。
3. 含“写、生成、策划、设置、自动”的问题，进入 draft 或 action_card。
4. 缺客户姓名但上下文已有客户时，继承上下文。
5. 真正缺关键对象时才追问，且只问一个问题。

验收标准：

1. `wrong_intent` 下降 60% 以上。
2. “今天现金、微信、支付宝各收了多少”不得进入 clarify。
3. “帮我生成一份本月采购计划”不得只做普通查询。
4. “帮我写一条推荐疗程续购的私信”必须生成文案草稿。

### R5：业务域路由增强

优先级：P1  
覆盖失败：`route_error` 91 次

需求说明：

Router / Planner 需要基于语义域，而不是少量词面规则，识别用户问题属于客户、预约、收银、服务、库存、财务、营销、自动化等业务域。

重点映射：

| 问题表达 | 应识别业务域 |
|---|---|
| 支付方式、微信、支付宝、现金 | finance / cashier |
| 会员折扣、卡项、权益、次卡 | card / customer |
| 下个客人、今天几个客人、空档 | reservation / service |
| 补货、采购量、安全库存 | inventory / purchase |
| 自动跟进、自动感谢、自动召回 | automation / marketing |
| 私信、朋友圈、话术、脚本 | marketing_copy |

验收标准：

1. `route_error` 下降 70% 以上。
2. 前台和美容师问题不得大量回落到 `unknown`。
3. Router 决策必须写入评测报告，便于审计。

### R6：工具注册表与 Skill 对齐

优先级：P1  
覆盖失败：`tool_missing` 2 次

需求说明：

当前出现 Planner 命中 `inventory.product.metadata.suggest`，但评测工具注册表未覆盖的问题。

要求：

1. Skill 声明的 tool 必须在 Tool Registry 中存在。
2. Eval Runner 的 mock tool registry 必须和真实 registry 保持同步。
3. 新增工具时必须补 `outputKinds`、权限、风险等级、是否需要确认。

验收标准：

1. `tool_missing = 0`。
2. CI 中增加 tool registry 一致性检查。

### R7：权限边界承接

优先级：P1  
覆盖失败：`permission_error` 10 次

需求说明：

当角色权限不足时，Agent 不应只失败或泛化追问，而应给出明确承接：

1. 当前角色无权查看哪些字段。
2. 可展示哪些安全摘要。
3. 是否可转交店长/财务。
4. 是否可生成申请或审批卡。

验收标准：

1. 权限问题不计为 Runtime Error。
2. 前台问财务、员工问全店利润、美容师问他人客户时，均有明确权限说明。
3. 高风险动作仍必须确认，不允许自动执行。

## 7. 分角色改进需求

### 7.1 前台接待 Agent

当前问题：80 条剩余评测中失败 59 条，通过率 26.3%，为最弱角色。

核心场景：

1. 客户查询：会员等级、历史项目、卡项权益、退款欠款。
2. 预约管理：今日预约、待确认、指定美容师安排、空档。
3. 收银核销：支付方式、会员折扣、核销、退款处理。
4. 现场协调：推荐项目、优惠活动、客户安抚。

需求：

1. 建立前台客户 360 输出卡。
2. 建立预约时间轴和待确认清单。
3. 建立收银/核销/退款查询 Skill。
4. 现场协调类问题允许输出话术，但必须区分“业务事实”和“建议话术”。

验收目标：

1. 前台剩余问题通过率从 26.3% 提升到 >= 70%。
2. 客户查询、预约管理、收银与核销三类必须优先达标。

### 7.2 营销增长 Agent

当前问题：84 条剩余评测中失败 57 条，通过率 32.1%。

核心场景：

1. 客群识别和召回。
2. 活动策划。
3. 话术与内容生成。
4. 权益 ROI。
5. 自动化触达规则。

需求：

1. 新增营销内容生成 Skill，支持私信、朋友圈、生日祝福、短视频脚本。
2. 新增活动策划 Skill，输出活动目标、客群、权益、成本、风险、执行步骤。
3. 新增自动化规则草稿，输出触发条件、对象、频次、停止条件、确认卡。
4. 新增营销 ROI 分析，至少支持优惠券核销、触达转化、活动成本和复购。

验收目标：

1. 营销剩余问题通过率从 32.1% 提升到 >= 70%。
2. 文案类问题必须输出可直接编辑的草稿。
3. 批量触达必须进入确认卡，不得直接执行。

### 7.3 美容师服务 Agent

当前问题：83 条剩余评测中失败 52 条，通过率 37.3%。

核心场景：

1. 今日服务安排。
2. 客户护理建议。
3. 服务记录和跟进任务。
4. 个人业绩和复购机会。

需求：

1. 新增美容师今日服务卡：客户、时间、项目、上次服务、注意事项。
2. 护理建议必须结合客户历史项目、最近服务记录、卡项权益。
3. 服务记录支持草稿化，不直接写入正式记录。
4. 复购建议输出推荐项目、理由和下一步话术。

验收目标：

1. 美容师剩余问题通过率从 37.3% 提升到 >= 70%。
2. “我今天有没有空档”“下一个客人上次做什么”必须直接回答。
3. 护理建议必须展示依据，不允许纯泛化。

### 7.4 财务风控 Agent

当前问题：83 条剩余评测中失败 52 条，通过率 37.3%。

核心场景：

1. 收入与对账。
2. 退款与折扣。
3. 成本毛利。
4. 财务风险与合规。

需求：

1. 收入查询必须区分营业收入、现金收入、支付方式收入、退款影响。
2. 退款折扣审计输出异常金额、订单、客户、操作人、原因。
3. 成本毛利输出 KPI + 明细表，不只给文字。
4. 税务/发票类属于系统不支持，必须透明说明。

验收目标：

1. 财务剩余问题通过率从 37.3% 提升到 >= 70%。
2. “今天现金、微信、支付宝各收了多少”必须输出 KPI。
3. “有没有异常大额退款”必须输出风险表。

### 7.5 库存采购 Agent

当前问题：80 条剩余评测中失败 49 条，通过率 38.8%。

核心场景：

1. 库存查询与风险。
2. 临期和损耗。
3. 采购建议。
4. 消耗分析。
5. 供应链协调。

需求：

1. 建立采购计划 Skill，输出建议采购品项、数量、依据、风险。
2. 安全库存问题要支持“低于安全库存”和“安全库存设置是否合理”两种判断。
3. 供应商涨价通知属于系统不支持；供应商采购链接属于系统支持。
4. 临期和低库存必须输出表格。

验收目标：

1. 库存剩余问题通过率从 38.8% 提升到 >= 70%。
2. `inventory.product.metadata.suggest` 工具注册一致。
3. “帮我生成本月采购计划”必须输出 action_card。

### 7.6 店长经营 Agent

当前问题：71 条剩余评测中失败 28 条，通过率 60.6%。

核心场景：

1. 经营趋势。

2. 目标进度需要支持目标值缺失时的透明说明。
3. 异常洞察应聚合客户、预约、库存、收入。
4. 趋势类问题必须支持 chart 或趋势表。

验收目标：

1. 店长剩余问题通过率从 60.6% 提升到 >= 80%。

## 11. 建议优先级

### 必须先做

1. 输出契约强约束。
2. Evidence 默认注入。
3. 前台接待 Skill 包。
4. 营销文案和自动化 Skill 包。
5. 美容师今日服务和服务记录 Skill 包。

原因：这五项覆盖了失败最多、体验最明显、用户最容易感知“答非所问”的问题。

### 第二批做

1. 财务退款折扣审计。
2. 财务收入对账。
3. 库存采购计划。
4. 安全库存审计。
5. 权限承接卡。

### 后续优化

1. Edge Case 多轮。
2. 目标进度和经营异常综合洞察。
3. 系统不支持问题的引导型回答。
4. 质量大盘按失败原因趋势展示。

## 12. 结论

当前 310 条失败项反映的是 Agent 产品化能力不足，而不是单纯测试问题。

最关键的改进方向是：

1. 让 Planner 对业务问题有稳定 Skill 命中。
2. 让输出契约成为硬约束。
3. 让 Evidence 成为业务回答默认能力。
4. 让前台、营销、美容师这三类高频角色先达到可用水位。

本需求建议作为下一阶段 Agent 能力补齐的主线输入，后续再拆成开发任务卡。
