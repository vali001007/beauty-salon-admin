## 一、背景与问题定义

### 1.1 为什么要做

美业门店的经营决策高度依赖"经验丰富的店长/老板"这一稀缺角色：他既要盯客流与业绩，又要管美容师排班与手法，还要盘库存、算利润、追客户、定营销。这种全局经营能力难以复制、难以规模化，且在多店扩张时急剧稀释。

现有的信息化系统（含本项目 Ami Core）已经把**业务数据和操作**数字化了，但它是"被动的工具"——人问它才答、人点它才动。店长仍需在几十个菜单、上百张报表间手动串联信息、判断风险、决定动作。

**本智能体的使命**：把"资深店长/老板的经营心智"沉淀为一个可对话、有记忆、能追问、懂全域业务、会主动提示风险与建议、并能在授权下直接执行操作的多智能体系统，让每一家门店都拥有一位"7×24 在线的数字经营专家"。

### 1.2 与工具型系统的本质区别

| 维度 | 传统信息化工具（含现有管理端） | 本智能体 |
|---|---|---|
| 交互 | 菜单 + 表单 + 报表，结构化点击 | 自然语言对话，口语化输入 |
| 记忆 | 无跨会话记忆，每次从零开始 | 分层记忆，记住门店、客户、历史决策与偏好 |
| 主动性 | 被动响应，人不问不动 | 主动巡检、预警、给建议 |
| 理解力 | 精确匹配字段/筛选项 | 行业语义理解，容错、可追问、消歧 |
| 专业度 | 呈现数据，判断靠人 | 内置各角色专家经验，给出判断与行动方案 |
| 协同 | 各模块孤立 | 多角色 Agent 协同完成跨域任务 |

### 1.3 设计红线

- **数据真实性**：所有经营结论必须可溯源到底层业务数据（订单、核销、库存、提成等），严禁"编数据"。无数据时明确说明，不臆测。
- **权限一致性**：智能体的数据可见范围与操作能力，严格继承并不超越登录用户在管理端的权限与门店数据范围。
- **人在环上（Human-in-the-loop）**：涉及资金、客户资产、批量变更等高风险操作，必须经用户确认后才执行。
- **可治理、可迭代**：每一次意图识别、语义映射、工具调用、模型输出都可观测、可评测、可回滚。

## 二、产品目标与成功标准

### 2.1 终极目标

让本智能体的经营管理水平，无限逼近一位"经营 5 年以上、带过多店的资深店长/老板"。具体体现为它能独立或辅助完成如下闭环：

> **看得懂**（读全域数据）→ **想得清**（诊断问题、识别风险与机会）→ **说得明**（给出可执行建议与话术）→ **做得了**（在授权下执行操作）→ **记得住**（沉淀为记忆，下次更懂这家店）。

### 2.2 分阶段目标

| 阶段 | 目标 | 核心能力门槛 |
|---|---|---|
| M1 会问会答 | 全域问数 + 记忆 + 追问 | 单表/跨表查询准确率 ≥ 95%；多轮追问不丢上下文 |
| M2 会诊会荐 | 风险提示 + 业务建议 | 主动巡检覆盖 6 大角色核心指标；建议可落地、带依据 |
| M3 会做会协 | 授权执行 + 多角色协同 | 高风险操作 100% 走确认；跨角色任务可编排完成 |
| M4 会治会进 | 可视化治理 + 持续迭代 | 全链路可观测；评测集回归；能力灰度上线/回滚 |

### 2.3 关键成功指标（KPI）

- **理解力**：口语化输入意图识别准确率 ≥ 92%；歧义主动澄清率（该问就问）≥ 90%。
- **问数**：Text-to-语义查询在评测集上执行成功率 ≥ 95%，数值正确率 ≥ 98%（口径由语义层保证）。
- **主动价值**：店长采纳智能体主动建议的比例 ≥ 40%；预警的真阳性率 ≥ 80%。
- **安全**：越权访问 0 起；高风险操作误执行 0 起；所有操作可审计追溯。
- **效率**：店长日常经营例行工作（看数、追客、排班检查）耗时下降 ≥ 50%。

## 三、总体架构

### 3.1 分层架构总览

本智能体采用**"感知—认知—记忆—协同—执行—治理"六层架构**，各层职责单一、可独立演进：

```
┌──────────────────────────────────────────────────────────────┐
│  L6 治理与迭代层  可观测 / 评测集 / 灰度发布 / 回滚 / 反馈闭环      │
├──────────────────────────────────────────────────────────────┤
│  L5 执行与操作层  技能库(查询/操作/分析) → 底层API/语义查询引擎     │
│                   高风险操作确认 · 幂等 · 事务 · 回执             │
├──────────────────────────────────────────────────────────────┤
│  L4 多智能体协同层  编排器(Supervisor) + 6大角色专家Agent + A2A  │
├──────────────────────────────────────────────────────────────┤
│  L3 记忆层  工作/会话/情景/语义/程序 五层记忆 + 遗忘与巩固          │
├──────────────────────────────────────────────────────────────┤
│  L2 认知层  行业意图语义层 + Ontology + 知识图谱 + 消歧/追问       │
├──────────────────────────────────────────────────────────────┤
│  L1 感知与接入层  多端入口 / 语音文字 / 身份与权限上下文注入         │
└──────────────────────────────────────────────────────────────┘
        │
        ▼  统一经由「语义查询引擎 + 底层能力网关」访问
┌──────────────────────────────────────────────────────────────┐
│  Ami Core 底座：PostgreSQL(80+业务表) · server-v2 业务API · 权限 │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 各层职责

- **L1 感知与接入层**：接收来自管理端、移动端、智能终端的自然语言输入（文字/语音），完成身份认证、门店上下文与权限上下文的注入。所有下游处理都在"当前用户 + 当前门店 + 权限集"的约束下进行。
- **L2 认知层**：把口语化输入翻译成结构化的业务意图，依托美业行业 Ontology 与知识图谱消除歧义、补全实体、纠正术语；不确定时发起澄清式追问。这是"无盲区语义理解"的核心。
- **L3 记忆层**：五层记忆架构，让智能体跨会话记住门店特征、客户历史、店长偏好和过往决策，支撑模糊追问与个性化。
- **L4 多智能体协同层**：一个中枢编排器（Supervisor）+ 店长/前台/美容师/营销/财务/库存/客服等角色专家 Agent。编排器做意图分诊与任务分解，角色 Agent 各自具备极致专业能力，通过 A2A 式协议协同。
- **L5 执行与操作层**：把角色 Agent 的意图落地为对底层的实际调用——数据查询（走语义查询引擎）、业务操作（走底层能力网关/API）、风险与分析计算。高风险动作走确认与事务保护。
- **L6 治理与迭代层**：全链路追踪、评测集回归、能力灰度与回滚、人工反馈闭环，支撑"可视化 Agent 治理与迭代"。

### 3.3 与 Ami Core 底座的关系

本智能体**不重写业务逻辑**。它是底座之上的"经营大脑"：

- **读**：统一经"语义查询引擎"访问数据，不直接裸写 SQL 散落各处，保证口径一致、权限收口。
- **写**：统一经"底层能力网关"调用 server-v2 既有业务 API（下单、核销、排班、建档等），复用后端已实现的校验、事务与副作用，避免绕过业务规则。
- **隔离**：本智能体自身的数据（记忆、语义元数据、编排轨迹、评测、治理配置）存放于独立命名空间的数据表，不侵入既有业务表结构。

## 四、能力域一：记忆与模糊追问（对应需求 ①）

### 4.1 五层记忆架构

参考 2026 年长时记忆智能体的主流实践，采用**工作 / 会话 / 情景 / 语义 / 程序**五层记忆分层，各层生命周期与检索方式不同：

| 记忆层 | 内容 | 生命周期 | 存储/检索 | 例子 |
|---|---|---|---|---|
| 工作记忆 | 当前轮次的即时上下文、临时变量 | 单轮~数轮 | 上下文窗口 | 用户刚说的"这个客户" |
| 会话记忆 | 当前会话的完整对话与中间结论 | 单次会话 | 会话状态存储 | 本次会话已查过的时间范围 |
| 情景记忆 | 跨会话的历史事件、决策、执行回执 | 长期，可归档 | 时间线 + 向量检索 | "上周店长决定给 A 客户升级卡项" |
| 语义记忆 | 关于门店/客户/员工的稳定事实与画像 | 长期，随事实更新 | 结构化 + 图谱 | "3 号店周末客流是平日 2 倍" |
| 程序记忆 | 有效的操作套路、话术模板、偏好 | 长期，随反馈强化 | 规则/模板库 | "该店长偏好先看毛利再看流水" |

### 4.2 记忆的写入、巩固与遗忘

- **写入**：每轮交互后异步抽取值得长期记忆的事实/事件/偏好，避免污染实时链路。
- **巩固（Consolidation）**：定期把零散情景记忆归纳为稳定的语义记忆（如多次"周末爆满"→"该店周末为高峰"），并更新门店/客户画像。
- **遗忘与时效（Staleness）**：为记忆打时间戳与置信度，过期或被新事实推翻的记忆降权或失效；数值型经营事实（如"月流水"）永远回源实时数据，记忆只存"结论与偏好"不存"易变数值"。
- **记忆的可见性**：记忆同样受权限与门店范围约束；跨店记忆仅对有跨店权限的角色（如区域经理）可见。

### 4.3 模糊追问与指代消解

智能体必须像真人店长一样处理"话没说全"的口语输入：

- **指代消解**："他这个月怎么样" → 结合会话/情景记忆确定"他"指代上文提到的某美容师。
- **省略补全**："跟上个月比呢" → 沿用上一轮的指标与对象，仅切换时间范围。
- **澄清式追问**：当存在多个候选实体或口径歧义且无法自信消解时，**主动反问**而非猜测。
  - 例：用户说"看下张姐的业绩" → 系统发现有 2 位"张"姓美容师 → 追问"您是指美容师张丽（3号店）还是张敏（5号店）？"
- **追问预算**：一次澄清尽量一次问清（合并多个待确认项），避免连环追问打断体验；能从记忆/上下文推断的绝不追问。

### 4.4 需求点

| 编号 | 需求 | 优先级 |
|---|---|---|
| MEM-1 | 跨会话记住门店画像、客户历史、店长偏好 | P0 |
| MEM-2 | 支持指代消解与省略补全的多轮追问 | P0 |
| MEM-3 | 歧义无法自信消解时主动澄清，且合并追问 | P0 |
| MEM-4 | 记忆带时效与置信度，易变数值回源实时数据 | P0 |
| MEM-5 | 情景记忆定期巩固为语义画像 | P1 |
| MEM-6 | 记忆可被用户查看、纠正、删除（记忆可解释） | P1 |

## 五、能力域二：美业全领域意图语义层（对应需求 ②）

### 5.1 为什么需要语义层，而不是"裸 Text-to-SQL"

2026 年的行业基准显示：直接把自然语言丢给 LLM 写 SQL，在真实企业库上准确率与一致性都不足；而"自然语言 → 语义层 → 查询"的路径可将可靠性提升到约 95%，覆盖范围内准确率接近 100%。原因是语义层把"业务口径"（指标定义、维度、join 关系、权限）固化下来，LLM 只需理解"问什么"，不必每次重新推导"怎么算"。

因此本智能体在数据理解上采用三件套：**Ontology（本体）+ 知识图谱（实体关系）+ 语义/指标层（度量口径）**，共同构成"美业全领域意图语义层"。

### 5.2 美业行业 Ontology（本体）

Ontology 定义美业经营世界里"有哪些概念、概念有哪些属性、概念之间是什么关系"，是无盲区理解的骨架。基于对底座 80+ 业务表的盘点，抽象为以下核心本体域：

| 本体域 | 核心实体（概念） | 说明 |
|---|---|---|
| 客户域 | 客户、健康档案、消费记录、客户画像、生命周期阶段、客户机会 | 客户是谁、健康状况、消费轨迹、处在什么阶段、有什么商机 |
| 员工域 | 美容师、职级、技能、排班、请假、可用性、提成 | 谁能做什么项目、什么时候在岗、赚多少 |
| 商品域 | 项目、项目类型、产品、品类、BOM（物料清单）、次卡、会员卡 | 卖什么、由什么构成、耗材如何 |
| 交易域 | 产品订单、项目订单、开卡订单、订单项、支付记录、退款记录、余额账户 | 谁买了什么、付了多少、退了多少 |
| 履约域 | 预约、服务任务、卡核销记录、排班、资源占用 | 服务如何被预约、执行、核销 |
| 库存域 | 库存批次、出入库流水、采购单、调拨单、盘点、临期预警、补货 | 有多少货、进销存、过期风险 |
| 财务域 | 收银班次、日结、经营成本、提成规则/记录/结算、日结算 | 钱怎么进出、成本几何、利润多少 |
| 营销域 | 营销活动、营销页、自动化策略、规则模板、触达、归因、推荐、预测快照 | 怎么拉新促活、效果如何归因 |
| 供应链域 | 供应商、资质、供应SKU、报价、采购、发货、结算 | 货从哪来、多少钱、质量如何 |
| 行业域 | 行业数据源、服务模板、产品模板、知识条目、薪酬基准、BOM模板 | 行业标杆与最佳实践 |

> Ontology 中每个实体都携带：业务名称、同义词/口语别名、关键属性、状态枚举、与其他实体的关系（及关系语义），并映射到底层的物理表/字段。

### 5.3 知识图谱（实体关系网络）

在 Ontology 之上构建门店经营知识图谱，把实体用带语义的边连接起来，支撑跨表、跨域推理：

```
客户 ──持有──▶ 次卡 ──核销于──▶ 服务任务 ──由──▶ 美容师
  │                                            │
  ├──产生──▶ 订单 ──包含──▶ 订单项 ──计提──▶ 提成记录
  │                                            │
  └──处于──▶ 生命周期阶段 ──触发──▶ 客户机会 ──转化──▶ 营销触达
项目 ──消耗(BOM)──▶ 产品 ──来自──▶ 库存批次 ──采购自──▶ 供应商
```

- **作用一：跨表查询规划**。用户问"哪些高价值客户的次卡快到期了"，图谱把"客户→次卡→到期日→消费额"的路径显式化，查询引擎据此规划 join，而非让 LLM 盲猜。
- **作用二：消歧与补全**。同名实体、口语术语通过图谱的邻接关系确定最可能的指代。
- **作用三：经营推理**。风险与机会识别可沿图谱边传播（如"某美容师离职→其绑定客户存在流失风险→触发客服角色跟进"）。

### 5.4 语义/指标层（度量口径统一）

把美业经营的关键指标（Metric）与维度（Dimension）定义为可复用的语义对象，口径唯一、集中治理：

| 语义对象类型 | 示例 | 说明 |
|---|---|---|
| 指标 Metric | 实收流水、储值负债、次卡履约率、客单价、连带率、复购率、毛利率、人效、耗占比、拓客成本 | 每个指标绑定明确的计算公式、数据来源、口径边界 |
| 维度 Dimension | 门店、时间（日/周/月）、美容师、项目、品类、客户分层、渠道 | 指标可沿维度切片 |
| 口径约束 | 是否含退款、是否含赠送次数、确认口径 vs 实收口径 | 消除"同一个词不同算法"的分歧 |

> 关键设计：**智能体永远不自行发明指标算法**。所有经营数值必须引用语义层已定义的指标；语义层未覆盖的口径，走"新增指标定义 + 评审"流程，而非临时编算法。这是数值正确率的根本保证。

### 5.5 口语化输入的理解流程

```
口语输入 → 术语归一(别名映射) → 实体识别与链接(挂到Ontology/图谱)
        → 指标/维度识别(挂到语义层) → 意图分类 + 口径消歧
        → [自信?] ── 否 ──▶ 澄清式追问
                    └─ 是 ──▶ 输出结构化意图(交给协同层/查询引擎)
```

## 六、能力域三：多智能体角色协同（对应需求 ③）

### 6.1 编排模式：Supervisor（中枢编排）+ 角色专家

采用 2026 年生产级多智能体系统的主流 **Supervisor（监督者/中枢）模式**：一个中枢编排器负责意图分诊、任务分解、角色调度与结果汇总；各角色 Agent 是领域专家，只专注自己的专业域。必要时角色间通过 **A2A（Agent-to-Agent）式**能力卡片互相委托子任务。

```
                        用户输入
                           │
                     ┌─────▼─────┐
                     │  中枢编排器 │  意图分诊 · 任务分解 · 调度 · 汇总 · 兜底
                     │ Supervisor │
                     └─────┬─────┘
      ┌──────┬──────┬──────┼──────┬──────┬──────┐
   ┌──▼─┐ ┌─▼──┐ ┌─▼───┐ ┌▼───┐ ┌▼───┐ ┌▼───┐ ┌▼───┐
   │店长│ │前台│ │美容师│ │营销│ │财务│ │库存│ │客服│
   │Agent│ │Agent│ │Agent │ │Agent│ │Agent│ │Agent│ │Agent│
   └────┘ └────┘ └─────┘ └────┘ └────┘ └────┘ └────┘
      每个角色 Agent = 专属系统提示 + 专属技能集 + 专属数据范围 + 专属知识
```

**为什么用 Supervisor 而非纯 Swarm（自由群聊）**：门店经营任务需要明确的责任归属、可控的执行顺序和可审计的决策链。中枢模式让每个动作都有清晰的发起者与审批路径，便于治理与追责；角色间协作也不会陷入无收敛的相互调用。

### 6.2 六大（+）角色 Agent 职责与专业能力

每个角色都应达到"该岗位资深从业者"的专业水准。以下定义各角色的**关注指标、典型任务、可用技能、协作关系**：

#### 6.2.1 店长 Agent（总控/经营）
- **定位**：经营全局的第一责任人，也是多数复杂任务的默认入口与汇总者。
- **关注**：营业额、客流、毛利、储值负债、人效、坪效、各角色异常汇总。
- **典型任务**：每日经营晨报/晚报、经营诊断（为什么这周流水降了）、跨角色目标拆解、经营决策建议。
- **专业能力**：能把多角色的碎片信息组织成"结论—归因—建议—行动"的经营叙事；懂美业淡旺季与门店生命周期节律。
- **协作**：向其余角色发起数据请求与任务委托，汇总为统一经营视图。

#### 6.2.2 前台 Agent（接待/预约/收银）
- **关注**：今日预约、到店核销、空档、收银班次、当班异常。
- **典型任务**：查/建/改预约、到店登记、次卡核销引导、收银结账、班次交接检查。
- **专业能力**：熟悉预约冲突与资源占用规则，能主动填补空档（gap opportunity），提示未确认/未到店订单。

#### 6.2.3 美容师 Agent（服务/手法/个人业绩）
- **关注**：个人排班、今日服务任务、个人提成、服务客户健康档案。
- **典型任务**：查看今日任务与客户注意事项、服务记录小结、个人业绩与提成查询、技能匹配。
- **专业能力**：懂项目手法与耗材（BOM），能基于客户健康档案给服务建议与话术。

#### 6.2.4 营销 Agent（拉新/促活/留存）
- **关注**：客户分层与生命周期、活动效果与归因、自动化策略、拓客成本、复购。
- **典型任务**：识别流失预警客户、生成邀约话术与活动草稿、配置自动化触达、活动效果归因分析。
- **专业能力**：懂 AARRR 与生命周期营销，能把"客户机会"转成具体触达方案，并对效果负责（可归因）。

#### 6.2.5 财务 Agent（收支/利润/提成）
- **关注**：实收流水、成本、经营利润、提成规则与结算、储值负债、对账异常。
- **典型任务**：日结/月结核对、经营利润拆解、提成试算与结算、储值负债与履约分析、对账异常排查。
- **专业能力**：懂美业"预收—履约—确认"的收入确认特性，能区分实收口径与确认口径，识别财务风险。

#### 6.2.6 库存 Agent（进销存/耗材/供应链）
- **关注**：库存水位、临期、耗占比、采购与补货、调拨、供应商。
- **典型任务**：临期与缺货预警、补货建议、BOM 耗材核对、采购单与调拨建议、供应商比价。
- **专业能力**：懂 BOM 与项目耗材联动，能根据服务量预测耗材消耗并提前补货。

#### 6.2.7 客服 Agent（关怀/跟进/口碑）
- **关注**：待跟进任务、客户满意度、售后与投诉、回访。
- **典型任务**：生成回访/关怀计划、跟进任务分派与闭环、流失客户挽回、满意度追踪。
- **专业能力**：懂客户关怀节律（服务后回访、生日关怀、周期提醒），话术有温度且合规。

> 角色集可扩展：如"人力/培训 Agent""大客户/VIP Agent"。角色定义（系统提示、技能集、数据范围、知识包）在治理台可配置，无需改代码即可新增或调整角色。

### 6.3 协同编排流程（示例）

**用户（店长）问**："这周业绩不太好，帮我看看问题出在哪，该怎么办。"

```
1. 中枢编排器：识别为"经营诊断"复合任务，分解为多角色子任务
2. 并行调度：
   ├─ 财务Agent：本周 vs 上周/去年同期 流水、毛利、客单价对比 → 定位下滑指标
   ├─ 前台Agent：本周客流、预约转化、空档率
   ├─ 营销Agent：本周新客/复购、活动触达效果、流失客户数
   └─ 库存Agent：是否有缺货导致无法服务
3. 店长Agent：汇总各角色结论 → 归因（如"复购下滑+2名主力美容师请假")
4. 生成"结论—归因—建议—行动"经营报告；高价值行动项可一键委派给对应角色执行
5. 记忆层：记录本次诊断结论与店长采纳的行动，供后续追踪复盘
```

## 七、能力域四：全域底层技能库（对应需求 ④）

### 7.1 技能分层

对底座（Ami Core 管理端 + server-v2 后端）的**全部业务、功能、数据**做技能化封装，供角色 Agent 调用。技能分四类：

| 技能类型 | 说明 | 落地方式 | 风险 |
|---|---|---|---|
| 查询技能 | 单表/跨表数据查询、报表、画像 | 经语义查询引擎 | 低（只读） |
| 分析技能 | 诊断、对比、趋势、归因、预测 | 语义查询 + 计算/模型 | 低（只读） |
| 风险技能 | 主动巡检、阈值预警、异常识别 | 定时/事件触发 + 规则/模型 | 低（提示类） |
| 操作技能 | 建档、下单、核销、排班、发起营销等写操作 | 经底层能力网关调用 server-v2 API | 中~高（写） |

### 7.2 查询技能：无盲区数据访问

统一经**语义查询引擎**，支持两类查询：

- **单表查询**：如"3 号店今天有多少个预约""这个客户的手机号"。
- **跨表组合查询**：如"本月复购客户里，消费额 Top10 且次卡即将到期的人""哪个美容师的连带率最高"。引擎借助知识图谱规划 join 路径，借助语义层套用指标口径。

覆盖范围对齐底座数据资产（按本体域）：客户与画像、员工与排班、商品与 BOM、订单与支付、预约与核销、库存与采购、财务与提成、营销与归因、供应链、行业模板等**全部业务域**，无数据盲区。

### 7.3 操作技能：映射到既有业务 API

操作技能**不重造轮子**，逐一映射到 server-v2 已实现的业务端点，复用其校验与事务。按角色归类的核心操作技能（示例，非穷举）：

| 角色 | 操作技能 | 底层能力（server-v2） |
|---|---|---|
| 前台 | 建预约/确认/改期/到店/取消 | `reservations/*`、`terminal/reservations/*` |
| 前台 | 次卡核销、收银结账、充值 | `terminal/cards/*`、`terminal/cashier/*`、`orders/member-cards/*` |
| 前台/店长 | 客户建档、编辑、健康档案 | `customers/*`、`customers/:id/health-profile` |
| 美容师 | 开始/完成/转交服务任务、服务记录 | `terminal/tasks/*`、`terminal/service-records` |
| 营销 | 生成活动/自动化草稿、发起触达、跟进任务 | `marketing/*`、`marketing/automation/*` |
| 财务 | 日结/月结生成与确认、提成试算/结算 | `commission/*`、`operation-profit/*`、`finance/metrics/*` |
| 库存 | 入库/调整/采购单/调拨/补货 | `inventory/*`、`supply-platform/*` |
| 店长 | 排班、智能排班一键生成/预览/发布 | `scheduling/*`、`scheduling/smart/*` |

> **写操作三原则**：① 幂等（重试不重复副作用）；② 事务（复合操作要么全成要么全败）；③ 回执（返回可读的执行结果与受影响对象）。

### 7.4 风险技能：主动巡检与预警

智能体不只被动应答，还要像资深店长一样"主动盯盘"。定义可配置的巡检规则库，覆盖各角色高价值风险场景：

| 风险域 | 巡检项（示例） | 触发建议 |
|---|---|---|
| 客户 | 高价值客户 N 天未到店、次卡临期未约、流失预警 | 派客服/营销跟进 |
| 财务 | 日结未平、储值负债异常增长、提成规则冲突、对账异常 | 提示财务核查 |
| 库存 | 缺货、临期、耗占比异常、BOM 与实际消耗背离 | 生成补货/盘点建议 |
| 履约 | 大量空档、预约未确认、美容师超负荷/闲置 | 优化排班、填补空档 |
| 营销 | 活动 ROI 低于阈值、自动化触达失败率高 | 调整或暂停策略 |
| 员工 | 主力美容师业绩骤降、请假导致产能缺口 | 排班调整、辅导提醒 |

- **触发方式**：定时巡检（如每日晨间）+ 事件触发（如日结完成后自动核对）。
- **输出**：风险清单按严重度排序，每条附"依据数据 + 建议动作 + 一键处理入口"。

### 7.5 业务建议：从数据到行动

建议必须**可落地、带依据、可执行**，避免正确的废话：

- **结构**：结论 → 依据（引用具体数据/指标）→ 建议动作 → 预期收益 → 一键执行入口。
- **示例**："建议本周对 12 位次卡临期客户发起邀约（依据：这 12 人平均剩余 3 次、到期前 14 天、历史复购率 65%），预计可挽回约 ¥1.8 万储值消耗。[生成邀约话术] [创建跟进任务]"

### 7.6 预测技能（首期纳入）

首期即提供预测类分析，**复用底座已沉淀的预测数据资产**（`PredictionRun`、`CustomerPredictionSnapshot`、`CustomerLifecycleSnapshot` 等），智能体在其上做解释、编排与行动衔接，而非从零训练模型：

| 预测能力 | 输入/来源 | 智能体如何使用 |
|---|---|---|
| 客户流失预测 | 预测快照的 churnScore/churnLevel | 营销/客服角色据此发起挽回，风险巡检据此预警 |
| 复购/响应预测 | repurchase30dScore、marketingResponseScore | 营销角色筛选高响应人群，优化触达 ROI |
| 客户价值预测 | ltv6m/ltv12m、ltvTier | 店长/营销识别高价值客户，差异化经营 |
| 销量/耗材预测 | 历史订单与 BOM 消耗趋势 | 库存角色提前补货，减少缺货与积压 |

> 预测结果一律标注"预测置信度与依据"，并落到"结论—依据—动作"的建议结构中，避免把预测当确定事实呈现。

## 八、能力域五：权限控制与管理（对应需求 ⑤）

### 8.1 三重权限收口

本智能体的权限**完全继承**底座权限体系（`平台:模块:动作` 三段式权限码 + 多门店数据范围 + 角色），并在三个环节强制校验，任一环节不通过即拒绝：

```
① 数据范围收口：注入 X-Store-Id 与用户可见门店集，查询/操作自动加门店过滤
② 技能权限收口：每个技能声明所需权限码，用户无该权限则技能对其"不可见/不可用"
③ 字段级收口：敏感字段（如客户手机、成本价）按角色做字段级脱敏/过滤
```

### 8.2 智能体角色权限 ≠ 放大权限

关键红线：**智能体角色（店长/前台等 Agent）只是"能力分工"，不是"权限来源"**。一个登录用户能通过智能体做什么，恒等于他在管理端本就能做什么。例如：一位只有前台权限的用户，即使调用了"店长 Agent"，也拿不到他无权查看的利润数据。

### 8.3 高风险操作的人在环上确认

| 风险等级 | 操作类型 | 处置 |
|---|---|---|
| 低 | 只读查询、分析、风险提示 | 直接执行 |
| 中 | 建档、建预约、生成草稿、创建跟进任务 | 执行后可撤销，给出回执 |
| 高 | 涉及资金/客户资产/批量变更：收银、退款、充值、核销、批量营销触达、提成结算、删除 | **先给预览与影响面，用户显式确认后执行**；写入审计 |

### 8.4 安全防护

- **提示注入防护**：来自客户输入、外部内容的文本一律视为"数据"而非"指令"，不因其中的"忽略以上指令"等字样改变行为。
- **最小权限**：技能默认最小授权；跨门店、导出、删除等能力单独授权。
- **越权阻断**：查询引擎在 SQL/参数层强制门店与权限过滤，杜绝"绕过前端"的越权。

### 8.5 需求点

| 编号 | 需求 | 优先级 |
|---|---|---|
| PRM-1 | 继承底座权限码与多门店数据范围，三环节强制校验 | P0 |
| PRM-2 | 智能体角色不放大用户权限 | P0 |
| PRM-3 | 高风险操作先预览影响面再确认执行，并审计 | P0 |
| PRM-4 | 敏感字段按角色脱敏/过滤 | P0 |
| PRM-5 | 提示注入防护，外部内容仅作数据 | P0 |

## 九、能力域六：可视化 Agent 治理与迭代（对应需求 ⑥）

### 9.1 全链路可观测

每一次会话都留下完整、可回放的轨迹（Trace），治理台可视化查看：

```
会话Trace
 ├─ 输入(原始口语) + 注入的身份/门店/权限上下文
 ├─ 认知层：术语归一 → 实体链接 → 意图与口径 → (是否追问)
 ├─ 记忆层：命中/写入了哪些记忆
 ├─ 编排层：分诊到哪些角色、任务分解、每步耗时
 ├─ 技能层：调用了哪些技能/查询/API，入参出参，成功/失败
 ├─ 安全：权限校验结果、是否触发确认、是否命中注入防护
 └─ 输出：最终回答 + 引用的数据来源 + Token/延迟/成本
```

### 9.2 评测集与回归

- **评测集**：按角色与场景维护标注用例（问句 → 期望意图/期望查询结果/期望动作）。覆盖问数、诊断、操作、追问、越权拒绝等。
- **评测方法**：确定性断言（数值/结构精确匹配）+ LLM-as-Judge（对话质量、建议合理性）双轨。
- **回归门禁**：语义层、技能、角色提示、模型变更上线前必须跑评测集，指标不回退方可发布。

### 9.3 灰度发布与回滚

- 语义元数据、角色提示、技能清单、巡检规则均**版本化**。
- 支持灰度（按门店/角色/比例）上线新版本，异常时一键回滚到上一稳定版本。

### 9.4 反馈闭环

- 用户可对每次回答"点赞/点踩 + 纠正"，纠正内容进入评测集与记忆修正队列。
- 治理台看板：意图识别准确率、查询成功率、追问率、建议采纳率、预警真阳性率、各角色调用量与失败率、成本趋势。

## 十、技术选型建议（截至 2026 年行业实践）

> 以下为建议而非约束，最终以架构评审为准。均为独立部署，不与历史实现耦合。

| 层 | 选型建议 | 依据 |
|---|---|---|
| 多智能体编排 | 图/状态驱动的编排框架，Supervisor 模式为主，保留 A2A 扩展位 | 2026 生产级多智能体主流模式，责任清晰可审计 |
| 记忆 | 五层记忆：会话状态 + 向量库（情景/语义）+ 结构化画像表 + 规则/模板库 | 2026 长时记忆智能体标准分层 |
| 语义层 | 独立指标/维度语义层 + 图谱（实体关系）联合驱动 NL→查询 | 语义层使问数可靠性达 ~95%，远高于裸 Text-to-SQL |
| 工具接入 | 以 **MCP** 标准封装底层技能为工具；跨 Agent 用 **A2A** 式能力卡片 | MCP 连"Agent↔工具"、A2A 连"Agent↔Agent"，2026 事实标准 |
| 模型 | 通过 AI Gateway 统一接入，Key 不下发前端/终端 | 延续底座安全约定 |
| 治理观测 | 全链路 Trace + 评测集 + LLM-as-Judge + 灰度回滚 | 2026 Agent 治理标准实践 |

### 关于 MCP / A2A 的取舍

- **MCP（Model Context Protocol）**：把 7.3 节的查询/操作技能包装成标准化"工具"，让角色 Agent 以统一方式发现与调用底层能力，未来也便于对接外部工具。
- **A2A（Agent-to-Agent）**：角色 Agent 之间用能力卡片（Agent Card）声明"我能干什么"，中枢按卡片委派子任务。二者互补，不是二选一。

### 16.4 语义查询引擎内部数据流

这是"无盲区 + 数值正确"的核心，把结构化意图安全地翻译为受控查询：

```
结构化意图 { 指标[], 维度[], 过滤条件[], 时间范围, 目标实体 }
      │
      ▼
① 指标解析：从语义层取每个指标的公式/来源/口径 (禁止临时发明算法)
      │
      ▼
② 路径规划：用知识图谱确定实体间 join 路径 (客户→次卡→核销→美容师…)
      │
      ▼
③ 权限与门店注入：强制追加 storeId ∈ 用户可见门店；字段级脱敏
      │
      ▼
④ 生成参数化查询 (白名单表/字段 + 绑定参数, 防注入; 只读事务)
      │
      ▼
⑤ 执行 → 结果 → 口径校验 → 附"数据来源与口径说明" → 返回
      │
      └─(失败/超范围)→ 明确报错, 不猜测、不编数
```

### 16.5 部署与集成架构

```
┌──────────────────────────────────────────────────────────────┐
│  Ami Core 管理端 (React, 现有)                                  │
│   + 智能体对话入口(现有 /ami-agent 之外的独立入口, 新一级菜单)     │
│   + 治理台 (新一级菜单, 复用管理端权限与UI栈)                      │
└───────────────────────────┬──────────────────────────────────┘
                            │ /api/brain/*  (Vite dev 代理 → :8080)
┌───────────────────────────▼──────────────────────────────────┐
│  server-v2 (NestJS, 现有进程) 内新增 BrainModule (独立模块)      │
│   Gateway · Cognition · Memory · Orchestrator · SkillRuntime  │
│   · SemanticQueryEngine · CapabilityGateway · Governance      │
│   ── 模型调用统一经现有 AI Gateway (Key 不下发) ──               │
├───────────────────────────┬──────────────────────────────────┤
│  PostgreSQL: 既有80+业务表(只读访问)  +  brain_* 独立表(读写)      │
├──────────────────────────────────────────────────────────────┤
│  向量存储: 情景/语义记忆检索 (PostgreSQL+pgvector 或独立向量库)     │
└──────────────────────────────────────────────────────────────┘
```

> 集成要点：① 作为 server-v2 内的独立 NestJS 模块，复用其鉴权、Prisma、CSRF、Swagger 与部署管线；② 前端复用管理端路由/权限/组件栈，新增 `brain` 相关权限码与一级菜单；③ 模型访问一律经现有 AI Gateway，前端与终端不持有模型 Key。

### 16.6 关键组件—底座能力映射（落地锚点）

| Brain 组件 | 依赖的底座能力 | 说明 |
|---|---|---|
| 接入网关 | 现有 JWT 认证、`X-Store-Id`、权限码体系 | 直接复用，不新建账号体系 |
| 语义查询引擎 | PostgreSQL 只读、Prisma | 参数化只读查询，强制门店/权限过滤 |
| 能力网关 | server-v2 各业务 Controller（`customers/*`、`orders/*`、`reservations/*`、`inventory/*`、`marketing/*`、`commission/*`、`scheduling/*` 等） | 写操作复用既有 API 校验与事务 |
| 预测技能 | `PredictionRun`、`CustomerPredictionSnapshot`、`CustomerLifecycleSnapshot` | 复用既有预测资产做解释与行动衔接 |
| 记忆向量检索 | pgvector 或独立向量库 | 情景/语义记忆的语义检索 |
| 模型调用 | 现有 AI Gateway（`/api/ai/*`） | 统一收口，Key 不外泄 |
| 治理台 | 管理端 React + 权限体系 | 一级菜单，`brain_*` 数据可视化 |
