# Ami AI 问数与运营数据查询需求文档

版本：v1.0
日期：2026-06-15
适用范围：Ami_Core 管理端、Ami Aura Lite 智能终端、Ami Glow 客户服务小程序、后端 `server-v2`、AI Gateway
核心目标：让用户用自然语言查询门店运营数据，但所有事实必须来自受控数据能力，AI 只做理解、编排、解释和建议。

---

## 1. 背景与问题

当前 Ami 已具备客户、商品、项目、卡项、会员卡、预约、排班、订单、库存、营销、财务等多类经营数据，也已经在智能终端中接入自然语言和 AI 问答能力。

但现有问答存在三个核心风险：

1. **规则误判**
   例如“近期销量增长的商品”被识别为“客户增长”，返回客户增长与流失风险卡片。原因是“增长”被泛化为客户增长关键词，没有结合“商品、销量”的领域语义。

2. **AI 可能瞎答**
   如果问题直接交给大模型，模型可能根据常识补全不存在的订单、客户、金额、库存或趋势，造成经营判断风险。

3. **一次性读全量数据不可接受**
   当前某些兜底链路会一次性拉取客户、订单、预约、库存、卡项等快照。随着数据量增长，这会导致慢、不稳定、成本高，也容易越权暴露数据。

因此需要建立一套面向美业运营的 **AI 问数机制**：自然语言先被解析成结构化查询计划，再由受控数据服务查询，最后由 AI 基于证据生成回答。

## 2. 产品目标

### 2.1 总目标

让店长、前台、美容师、财务、库存等角色可以通过自然语言稳定查询经营数据，系统返回可信的数据卡片、分析结论和下一步动作建议。

### 2.2 设计原则

| 原则 | 要求 |
| --- | --- |
| 事实来自系统 | 客户、订单、金额、库存、卡项、排班等事实只能来自 Ami_Core 查询结果 |
| AI 不直接查库 | AI 只能生成结构化查询计划或基于查询结果总结，不允许自由生成 SQL 直接执行 |
| 不穷举问法 | 不按用户说法枚举规则，而是按业务领域、指标、对象、时间、动作建立能力目录 |
| 不一次性读全库 | 默认先查聚合、Top-K 和摘要，用户追问再查明细 |
| 口径统一 | 销售额、销量、复购、流失、占用率、库存预警等指标必须由语义层统一定义 |
| 权限前置 | 查询计划执行前必须校验用户角色、门店、数据范围和字段权限 |
| 证据可追溯 | 每次回答必须展示数据来源、统计口径、时间范围和关键证据 |
| 可降级 | AI 不可用时，受控查询仍可返回数据卡片和规则化摘要 |

## 3. 用户角色与权限边界

| 角色 | 可查询范围 | 不可查询范围 |
| --- | --- | --- |
| 系统管理员 | 全部门店、全域数据、系统配置、审计 | 无 |
| 店长 | 本门店经营、客户、订单、卡项、排班、库存、营销、财务摘要 | 其他门店明细、系统级配置 |
| 前台 | 本门店预约、客户基础信息、收银、核销、办卡、订单基础数据 | 毛利、提成规则、全量财务报表 |
| 美容师 | 本人排班、本人服务客户、本人提成、本人服务记录、护理建议 | 其他美容师客户、全店财务、全店客户明细 |
| 财务 | 收入、退款、应收、实收、支付方式、结算、Ami 费用 | 非财务必要的客户隐私字段 |
| 库存/采购 | 商品、库存、批次、临期、采购、调拨、供应商 | 客户隐私、提成、营销触达明细 |
| 营销运营 | 活动、触达、转化、客户分群、优惠券、营销归因 | 财务敏感明细、系统配置 |

权限执行要求：

- 所有查询必须带 `storeId`，多门店角色可显式选择门店或查看汇总。
- 查询计划必须带 `operatorId`、`role`、`permissions`。
- 字段级脱敏：手机号默认中间脱敏；导出、批量客户明细需更高权限。
- 美容师默认只能查本人绑定客户和本人服务相关数据。
- 财务、营销、库存等跨域查询要按最小必要字段返回。

## 4. 目标架构

```text
用户自然语言
  -> 业务相关性判断
  -> 领域识别
  -> 指标/对象/时间/条件/动作抽取
  -> 能力目录匹配
  -> 查询计划生成
  -> 权限与参数校验
  -> 受控数据服务执行
  -> 结果压缩与证据生成
  -> AI 基于证据生成解释
  -> 数据卡片 + 文字总结 + 下一步动作
  -> 审计记录
```

### 4.1 核心模块

| 模块 | 作用 |
| --- | --- |
| BusinessQueryRouter | 识别领域、指标、对象和查询意图，决定走哪个数据能力 |
| Semantic Layer | 统一业务指标、时间口径、状态口径、字段别名 |
| Capability Registry | 注册可被 AI 调用的数据查询能力，不依赖裸 SQL |
| Query Planner | 把自然语言转为结构化查询计划 |
| Policy Engine | 校验权限、门店、字段、时间范围、风险等级 |
| Query Executor | 调用受控 service / API / Prisma 聚合查询 |
| Evidence Builder | 生成统计口径、数据来源、样本证据和不确定性说明 |
| Answer Composer | 基于查询结果生成简短解释、建议和动作卡 |
| Audit Logger | 记录问题、查询计划、执行人、数据范围、返回摘要 |

## 5. 领域识别范围

领域识别必须覆盖门店运营涉及的主要业务域。每个领域不是简单关键词，而是由对象、指标、动作和上下文共同判断。

| 领域 | 典型对象 | 典型问题 | 主要数据源 |
| --- | --- | --- | --- |
| 商品 | 商品、产品、SKU、耗材、零售品 | 近期销量增长的商品、哪些商品卖得好、哪个商品库存不够 | Product、OrderItem、ProductOrder、StockMovement、ProductBatch |
| 项目 | 护理项目、服务项目、疗程、加项 | 最近做得最多的项目、项目毛利、项目带动哪些商品 | Project、OrderItem、ServiceTask、Reservation、ProjectBomItem |
| 客户 | 客户、会员、标签、画像、流失、复购 | 哪些客户要流失、高价值客户、最近没来的会员 | Customer、CustomerProfile、PredictionSnapshot、Reservation、ProductOrder |
| 排班 | 美容师、时段、班次、忙闲、请假 | 今天谁有空、哪个时段最忙、排班占用率 | Beautician、Schedule、Reservation、ServiceTask |
| 预约 | 到店、预约、改期、爽约、确认 | 今天预约情况、未到店客户、明天空位 | Reservation、Customer、Project、Beautician |
| 订单 | 订单、收银、退款、支付方式、客单价 | 今天订单收入、哪些订单未收款、退款多少 | ProductOrder、OrderItem、PaymentRecord、RefundRecord |
| 卡项 | 次卡、疗程卡、开卡、核销、到期 | 即将到期次卡、核销最多的卡、剩余次数 | Card、CustomerCard、CardUsageRecord、ProductOrder |
| 会员卡 | 储值、余额、充值、赠送、沉淀资金 | 会员卡余额多少、充值转化、余额沉睡 | CustomerBalanceTransaction、RechargeOrder、Customer |
| 财务 | 收入、费用、利润、结算、Ami 费用 | 今日实收、Ami 费用、ROI、支付渠道分布 | ProductOrder、PaymentRecord、RefundRecord、CommissionRecord、AmiMonthlyBill |
| 库存 | 库存、安全库存、临期、批次、调拨 | 哪些商品低库存、临期商品、库存周转 | Product、ProductBatch、StockMovement、PurchaseOrder、TransferOrder |
| 营销 | 活动、优惠券、触达、转化、归因 | 哪个活动带来订单、沉睡客户活动效果 | MarketingActivity、RecommendationEvent、MarketingAttribution、Promotion |
| 员工 | 美容师、前台、顾问、提成、服务质量 | 员工业绩、提成、服务完成率 | User、Beautician、CommissionRecord、ServiceTask、ProductOrder |
| 门店 | 门店、区域、营业状态、多门店对比 | 哪家店收入最高、门店排名 | Store、ProductOrder、Reservation、Customer |
| 供应链 | 供应商、采购、入库、结算 | 该补哪些货、供应商结算情况 | Supplier、PurchaseOrder、Product、ProductBatch |
| 自动化 | 规则、任务、提醒、执行结果 | 哪些自动化今天执行了、失败原因 | TerminalAutomationStrategy、TerminalAutomationExecution |

## 6. 语义层要求

### 6.1 指标定义

语义层必须统一定义指标，避免 AI、页面、报表各算各的。

| 指标 | 标准口径 |
| --- | --- |
| 销售额 | 已完成或已支付订单的 `totalAmount`，排除取消和退款订单 |
| 商品销量 | `OrderItem.itemType = product` 的 `quantity` 汇总 |
| 项目服务次数 | `OrderItem.itemType = project` 或已完成 `ServiceTask` 次数 |
| 客单价 | 销售额 / 有效订单数 |
| 到店率 | 已到店预约数 / 有效预约数 |
| 排班占用率 | booked、busy、leave 时段 / 总排班时段 |
| 低库存 | 当前库存 <= 安全库存 |
| 临期库存 | 批次有效期在配置天数内，例如 30 天 |
| 流失风险 | 最近到店天数、消费等级、未来预约、触达记录综合判断 |
| 复购率 | 指定周期内复购客户数 / 有消费客户数 |
| 营销转化 | 活动触达后归因窗口内产生订单或预约 |
| Ami 关联收入 | Ami 推荐、自动化、问答或动作卡归因的订单收入 |

### 6.2 时间口径

默认时间范围规则：

| 问法 | 默认解释 |
| --- | --- |
| 今天 | 当日 00:00 到当前时间 |
| 昨天 | 前一自然日 |
| 本周 | 周一到当前日 |
| 本月 | 当月 1 日到当前日 |
| 近期 | 默认近 30 天 |
| 最近 | 默认近 30 天，可根据领域调整 |
| 环比 | 当前周期对比上一等长周期 |
| 同比 | 当前周期对比去年同周期 |

如果用户问题对时间高度敏感但未给出范围，系统应采用默认值并在回答中说明。

### 6.3 别名与同义词

语义层需维护业务别名：

| 标准对象 | 同义词 |
| --- | --- |
| 商品 | 产品、零售品、SKU、耗材 |
| 项目 | 服务、护理、疗程、加项 |
| 次卡 | 卡项、疗程卡、护理卡、次数卡 |
| 会员卡 | 储值卡、余额、账户余额 |
| 订单 | 收银单、流水、消费记录、付款记录 |
| 排班 | 班次、时段、忙闲、请假 |
| 客户 | 会员、顾客、老客、新客 |

## 8. 查询计划要求

### 8.1 Query Plan 标准结构

```json
{
  "requestId": "bq_20260615_001",
  "domain": "product",
  "capability": "product_sales_trend",
  "intent": "query",
  "metrics": ["quantity", "salesAmount", "growthRate"],
  "dimensions": ["productId", "productName"],
  "filters": {
    "storeId": 1,
    "dateRange": {
      "type": "relative",
      "value": "last_30_days"
    },
    "compareWith": "previous_30_days",
    "status": ["completed", "paid"]
  },
  "sort": {
    "field": "growthRate",
    "direction": "desc"
  },
  "limit": 10,
  "needClarification": false,
  "clarificationQuestion": null
}
```

### 8.2 查询计划生成规则

- AI 只能输出 JSON 查询计划，不输出 SQL。
- Query Planner 必须用 schema 校验输出。
- 未识别领域时进入澄清，不直接回答。
- 领域冲突时按对象优先，例如“商品销量增长”优先商品，不是客户增长。
- 动作类问题和问数类问题要分开：
  - “查近期销量增长商品”是查询。
  - “给这些商品创建活动”是写操作，需要确认。
- 默认时间范围必须写入查询计划并在回答中展示。
- 查询计划必须记录原始用户问题。

## 9. 澄清机制

### 9.1 需要澄清的情况

| 情况 | 示例 | 澄清问题 |
| --- | --- | --- |
| 领域不明确 | 最近增长怎么样 | 您想看客户增长、商品销量增长，还是收入增长？ |
| 指标不明确 | 哪个商品最好 | 您想按销量、销售额，还是毛利排序？ |
| 时间不明确且影响较大 | 活动效果怎么样 | 您想看本月、近 30 天，还是指定活动周期？ |
| 对象不明确 | 她上次做了什么 | 请先选择客户，或输入客户姓名/手机号。 |
| 权限不足 | 查全店财务明细 | 当前角色不能查看财务明细，可查看权限范围内的收入摘要。 |

### 9.2 默认值策略

不是所有问题都要追问。可使用默认值并说明：

```text
我先按近 30 天统计，并与前 30 天对比。
```

默认值适用于低风险查询。财务明细、客户批量明细、跨门店查询等高敏感场景优先澄清和权限校验。

## 10. 回答与卡片要求

### 10.1 回答结构

每次问数回答必须包含：

1. 结论：一句话说明核心发现。
2. 数据卡片：结构化展示 Top-K、趋势、异常或明细。
3. 数据依据：时间范围、数据表/业务对象、过滤状态、统计口径。
4. 经营建议：下一步动作，尽量能点击执行。
5. 不确定性：数据不足、权限限制、口径限制时必须说明。

### 10.2 示例：商品销量趋势

用户：

```text
近期销量增长的商品
```

系统应返回：

```text
我先按近 30 天商品销量统计，并与前 30 天对比。

近 30 天销量增长最快的商品：
1. 补水精华：销量 42 件，较前 30 天 +68%，销售额 ¥8,960，当前库存 18 件
2. 修护面膜：销量 31 件，较前 30 天 +45%，销售额 ¥6,510，当前库存 9 件
3. 防晒乳：销量 24 件，较前 30 天 +38%，销售额 ¥4,320，当前库存 6 件

数据依据：已完成/已支付订单，OrderItem=itemType product，统计周期 2026-05-16 至 2026-06-15，对比 2026-04-16 至 2026-05-15。

建议：防晒乳库存接近安全库存，建议先补货；补水精华可与补水护理项目做组合推荐。
```

### 10.3 数据不足示例

```text
当前近 30 天没有足够商品订单明细，无法判断销量增长。可以先查看商品销售排行，或扩大到近 90 天。
```

禁止输出：

```text
近期补水类商品应该增长不错。
```

## 11. 数据查询策略

### 11.1 分层查询

| 层级 | 作用 | 示例 |
| --- | --- | --- |
| L1 聚合摘要 | 快速回答核心问题 | Top 10 商品销量、今日收入、低库存数量 |
| L2 证据明细 | 支撑解释 | 商品订单样本、客户分布、库存变化 |
| L3 关联分析 | 追问或高级分析 | 商品增长与项目、活动、客户标签关联 |
| L4 动作建议 | 转化为业务动作 | 补货、活动、回访、排班调整 |

默认只查 L1。用户追问“为什么”“哪些客户买的”“怎么做活动”时，再进入 L2/L3。

### 11.2 查询限制

- 默认 `limit <= 10`。
- 明细查询默认最多 50 条。
- 时间范围默认最多 180 天，超过需高级权限或异步任务。
- 禁止无条件全表扫描。
- 大查询必须进入后台任务，完成后通知用户。
- AI 上下文只放聚合结果和必要证据，不放全量数据。

## 12. 安全与治理

### 12.1 权限控制

每个 Capability 需要配置：

- 允许角色。
- 必需权限码。
- 数据范围：本人、本店、多店、全局。
- 字段范围：基础字段、敏感字段、财务字段。
- 是否允许导出。
- 是否允许继续执行动作。

### 12.2 审计日志

每次问数记录：

| 字段 | 说明 |
| --- | --- |
| requestId | 查询请求 ID |
| operatorId | 操作人 |
| role | 当前角色 |
| storeId | 数据门店 |
| originalQuestion | 原始问题 |
| queryPlan | 查询计划 |
| capability | 命中能力 |
| executionStatus | 成功/失败/降级 |
| resultSummary | 返回摘要 |
| sensitiveFields | 是否涉及敏感字段 |
| createdAt | 时间 |

### 12.3 幻觉防控

- 回答必须引用 `Evidence Builder` 生成的数据证据。
- AI 输出中出现未在结果集中出现的客户名、商品名、金额、订单号，应拦截或重写。
- 没有数据时返回“数据不足”，不能推测。
- 对异常趋势要展示样本量，避免小样本误导。

## 13. 前端交互要求

### 13.1 对话流

用户输入后，前端展示：

1. 用户原问题。
2. 系统查询状态，例如“正在分析商品销量趋势”。
3. 数据卡片。
4. AI 总结。
5. 下一步动作按钮。

### 13.2 卡片类型

| 卡片 | 关键内容 | 动作 |
| --- | --- | --- |
| 商品销量趋势卡 | 商品、销量、销售额、增长率、库存 | 查看明细、创建活动、补货 |
| 项目服务趋势卡 | 项目、服务次数、收入、耗材成本 | 查看项目、设置推荐、排班优化 |
| 客户风险卡 | 客户、风险原因、最近到店、建议触达 | 分配跟进、生成话术、创建活动 |
| 排班利用率卡 | 美容师、时段、占用率、空闲 | 调整状态、创建预约 |
| 订单分析卡 | 收入、订单数、客单价、支付方式 | 查看订单、导出、处理未收款 |
| 卡项到期卡 | 客户、卡名、剩余次数、到期时间 | 邀约、续卡、核销 |
| 财务摘要卡 | 实收、退款、费用、结算 | 查看明细、导出、对账 |
| 营销转化卡 | 活动、触达、转化、收入 | 查看人群、复制活动、优化规则 |

### 13.3 多轮追问

系统需要保留最近一次查询上下文：

```text
用户：近期销量增长的商品
系统：返回商品销量趋势
用户：这些商品库存够吗
系统：沿用上一轮商品列表，查询库存状态
用户：给补水精华做个活动
系统：生成活动草稿，等待确认
```

多轮上下文必须有过期时间，建议 10 分钟或用户切换领域后失效。

## 14. 后端接口建议

### 14.1 查询路由接口

```http
POST /api/business-query/resolve
```

用途：解析自然语言，返回领域、能力、查询计划或澄清问题。

### 14.2 查询执行接口

```http
POST /api/business-query/execute
```

用途：执行已校验的查询计划，返回结构化结果和证据。

### 14.3 一体化问数接口

```http
POST /api/business-query/ask
```

用途：终端或管理端直接提交自然语言，后端完成解析、执行、总结，返回卡片数据。

返回结构：

```json
{
  "requestId": "bq_20260615_001",
  "status": "success",
  "domain": "product",
  "capability": "product_sales_trend",
  "queryPlan": {},
  "card": {
    "type": "productSalesTrend",
    "title": "近期销量增长的商品",
    "items": []
  },
  "answer": "近 30 天销量增长最快的是补水精华。",
  "evidence": {
    "dateRange": "2026-05-16 至 2026-06-15",
    "compareRange": "2026-04-16 至 2026-05-15",
    "source": ["ProductOrder", "OrderItem"],
    "filters": ["status in completed, paid", "itemType=product"]
  },
  "actions": []
}
```

### 14.4 能力元数据接口

```http
GET /api/business-query/capabilities
```

用途：前端获取当前用户可用问数能力，用于提示词、快捷入口和权限提示。

## 15. 与现有系统的关系

### 15.1 与意图驱动微应用

AI 问数是意图驱动微应用的查询类分支：

- 微应用偏“办事”：预约、核销、收银、办卡、服务记录。
- AI 问数偏“查数 + 分析”：趋势、排行、风险、归因、建议。
- 两者共用 Intent Router、Capability Registry、Policy Engine 和卡片渲染体系。

### 15.2 与底层 API-Agent 模式

AI 问数可作为 API-Agent 模式的第一批低风险能力。它以只读查询为主，适合先做标准化 Tool：

- `queryProductSalesTrend`
- `queryCustomerChurnRisk`
- `queryInventoryAlerts`
- `queryScheduleUtilization`
- `queryOrderRevenue`
- `queryMarketingConversion`

后续再接写操作 Tool，例如创建活动、创建补货单、分配跟进任务。

## 16. 分阶段落地计划

### 阶段 1：P0 问数框架

目标：解决误判和瞎答风险。

范围：

- 新增 `BusinessQueryRouter`。
- 建立领域识别：商品、项目、客户、排班、预约、订单、卡项、会员卡、财务、库存、营销。
- 建立 Capability Registry。
- 支持 P0 能力：商品销量趋势、客户流失、库存预警、今日预约、订单收入。
- 回答必须展示数据依据。
- 修复“近期销量增长的商品”误判。

验收：

- `近期销量增长的商品` 命中商品销量趋势。
- `最近增长客户` 命中客户增长机会。
- `今天收入怎么样` 命中订单收入分析。
- `哪些商品库存不足` 命中库存预警。
- 无数据时不编造，返回数据不足说明。

### 阶段 2：P1 经营分析扩展

目标：覆盖更多经营领域和多轮追问。

范围：

- 项目服务趋势。
- 次卡到期和核销分析。
- 会员卡余额分析。
- 排班利用率。
- 营销转化分析。
- 支持上下文追问。

验收：

- 用户追问“这些商品库存够吗”可沿用上一轮商品列表。
- 用户追问“哪些客户买的”可在权限范围内返回客户分布。
- 用户可从问数结果进入活动草稿或跟进任务。

当前开发进度（2026-06-15）：

- 已接入受控查询：经营概览、商品销量趋势、项目服务趋势、客户流失风险、客户增长机会、库存预警、今日预约、排班利用率、订单收入分析、卡项到期风险、卡项核销分析、会员卡余额分析、财务现金流摘要、营销转化分析。
- 已接入终端问数入口：自然语言问数进入 `business.query`，结果以数据卡片展示，并展示能力、领域、查询结果和数据依据。
- 已验证典型误判：`近期销量增长的商品` 命中商品销量趋势，不再进入客户增长卡片。
- 已接入上下文追问：上一轮商品结果后的 `这些商品库存够吗`、`哪些客户买的` 会携带上一轮 `productId` 进入受控查询，不读全量数据、不让 AI 猜。
- 已接入审计日志：每次问数会记录 `business_query` 场景、原始问题、领域、能力、过滤条件、结果状态、样本量和耗时到 `AiAuditLog`。
- 已接入结果动作入口：问数卡片展示“待确认动作”，用户点击后才进入对应问数、营销草稿或业务入口，不自动执行高风险动作。
- 已接入 P2 高级分析：商品补货机会、项目耗材毛利、供应链采购建议、多门店对比、自动化执行复盘、经营异常主动提醒。
- 多门店对比按 `UserStore` 授权门店执行；当前用户未授权多个门店时返回数据不足说明，不查询其他门店。

### 阶段 3：P2 高级分析与行动闭环

目标：从查数升级为经营动作。

范围：

- 商品补货机会。
- 项目耗材毛利。
- 供应链采购建议。
- 多门店对比。
- 自动化执行复盘。
- 经营异常主动提醒。

验收：

- 问数结果可生成补货单、营销活动、客户跟进任务。
- 所有动作必须经过确认。
- 审计日志完整可查。

## 18. 非目标

当前阶段不做：

- 让 AI 自由生成 SQL 并直接执行。
- 一次性把全量数据库塞给大模型。
- 开放无权限跨门店数据查询。
- 自动执行高风险写操作。
- 通过纯关键词穷举所有问法。
- 将 AI 回答作为财务、合规或医疗诊断结论。

## 19. 成功指标

| 指标 | 目标 |
| --- | --- |
| 领域识别准确率 | P0 高频问题 >= 95% |
| 数据回答可追溯率 | 100% 带数据依据 |
| 幻觉率 | 关键业务事实 0 容忍 |
| 平均响应时间 | P0 聚合查询 <= 3 秒 |
| 澄清率 | 不明确问题可控澄清，不超过 20% |
| 权限误放行 | 0 |
| 用户继续追问率 | 提升，说明数据卡片可承接 |
| 从问数到动作转化 | 可统计创建活动、补货、跟进任务等 |

## 20. 结论

Ami 的 AI 问数不应定位为“聊天机器人查数据库”，而应定位为 **运营数据能力编排层**。

正确路径是：

```text
业务语义层
  + 能力目录
  + 结构化查询计划
  + 权限与审计
  + 受控聚合查询
  + AI 解释与动作建议
```

这样既能覆盖商品、项目、客户、排班、订单、卡项、财务、库存、营销等全运营领域，也能避免瞎答、全量读取和规则穷举。
