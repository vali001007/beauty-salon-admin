# Agent V6 数字店长经营智能体 · 产品需求文档（经营结果驱动版）

- 版本：v2.0（换视角重写：从"能力清单"改为"经营结果闭环"组织）
- 日期：2026-07-09
- 内部代号：Agent V6
- 产品名建议：Ami 数字店长
- 一句话定位：一套按"堵住经营损失"来组织的多角色智能体系统，让门店少漏单、少流失、少缺货、少乱账、少空档、少投诉、少靠店长个人经验
- 面向读者：产品、研发、数据、实施、运营，以及管理端 Ami Core 负责人

## 0. 为什么换一种写法

大多数经营 Agent 需求文档（包括本项目此前的版本）都按"能力"组织：记忆、语义层、多 Agent、权限、治理。这种写法对研发友好，但对经营者无感——老板不会因为"你有分层记忆"而付费，他只关心"这个月是不是少流失了 10 个高价值客户"。

本文因此换一个主轴：**按经营结果（少漏单/少流失/少缺货/少乱账/少空档/少投诉/少靠经验）组织需求**。每一项都是一条可度量的闭环使命，能力（记忆、语义层、多 Agent、扫描、权限、治理）退居为服务这些使命的"横向支撑"（第 9 章），并在第 10 章用对照表证明六项能力要求全部覆盖。

这样做的三个好处：一是需求可按"能救回多少钱/多少客户"排优先级；二是每条能力都能追问"它服务哪个经营结果"，避免为技术而技术；三是验收标准天然可量化（损失下降多少）。

## 1. 产品定位：把"经营损失"变成"可堵的漏斗"

美业门店的利润是被一个个"漏点"吃掉的：客人本该复购却流失了、耗材本该够却断了、卡本该核销确认收入却成了糊涂账、档期本该排满却空着、投诉本该及时补救却发酵成差评。传统后台把这些数据分散在几十个模块里，靠店长手动巡检——好店长稀缺，普通门店只能认损。

V6 的定位不是"问数助手"，而是**一支盯着漏点、主动堵漏的数字经营团队**。它把门店经营抽象成七个"损失漏斗"，每个漏斗配一条闭环使命，由对应角色 Agent 7×24 盯守：发现漏点 → 解释原因 → 给出堵漏动作 → 受控执行 → 度量堵回了多少。

### 1.1 七大经营使命（本文主轴）

| 编号 | 使命 | 堵住的损失 | 主责角色 Agent | 核心度量 |
| --- | --- | --- | --- | --- |
| M1 | 少流失 | 高价值客户悄悄流失、疗程中断、久未回访 | 客服回访 + 营销增长 | 高价值客户流失率、挽回金额 |
| M2 | 少漏单 | 到店未转化、该续卡未续、该升单未升 | 前台接待 + 营销增长 | 转化率、客单价、续卡率 |
| M3 | 少空档 | 档期空置、热门技师排满而他人闲置 | 前台接待 + 店长总控 | 空档利用率、空档挽回收入 |
| M4 | 少缺货 | 耗材断供停服、临期报损、账实不符 | 库存采购 | 缺货停服次数、临期损耗额 |
| M5 | 少乱账 | 退款异常、卡负债失真、对账差异、提成算错 | 财务风控 | 对账差异额、异常拦截数 |
| M6 | 少投诉 | 差评发酵、售后拖延、满意度下滑 | 客服回访 | 投诉响应时长、差评转化率 |
| M7 | 少靠经验 | 决策依赖店长个人、无法复制、交接即失忆 | 店长总控 + 数据审计 | 建议采纳率、经营动作标准化率 |

每个使命都遵循同一条闭环模板（第 2 章），并绑定真实的数据表与业务端点（附录）。

### 1.2 产品不是什么

- 不是单轮聊天机器人；不是只读问数工具；不是无监督自动执行系统。
- 不是历史 Agent 版本的重命名或壳。
- 不是用模型绕过现有业务权限的快捷入口。
- 不是按后台菜单组织能力——而是按经营结果组织能力。

## 2. 使命闭环模板（所有使命共用的骨架）

每条使命都是同一个六段闭环。这套模板是本文的"原子结构"，第 3 章逐一填充七个使命。

```
① 损失场景     这个漏点长什么样、发生在哪个环节、一年吃掉多少利润
      │
② 监测信号     用什么数据/阈值发现它(表 + 字段 + 触发条件)  ← 主动扫描，不等人问
      │
③ 诊断口径     确认是真漏点而非噪声(跨表核对 + 指标口径 + 排除项)
      │
④ 堵漏建议     给出可执行动作 + 预计挽回 + 成本 + 可选方案 + 推荐理由
      │
⑤ 受控执行     按风险分级(L0–L4)：低风险直接办，高风险 dry-run+审批
      │
⑥ 复盘度量     动作是否见效、堵回多少、沉淀为经营记忆与规则
```

关键设计点：

- **②监测是主动的**：使命由 EventScheduler 定时/事件触发，Agent 主动发现漏点推送到驾驶舱，而不是等用户提问。这是"数字店长"和"问数助手"的本质区别。
- **③诊断必须可信**：每个结论带证据（数据来源、时间范围、口径），并由数据审计 Agent 反核，防止把噪声当漏点惊扰经营者。
- **⑤执行默认保守**：涉及客户资产、财务、库存、群发、绩效的动作绝不自动执行，只生成审批草案。
- **⑥复盘形成飞轮**：堵漏效果回写为经营记忆（哪些动作对哪类客户有效），下次同类漏点直接复用已验证策略。

## 3. 七大使命详述

每个使命按闭环模板展开，并标注绑定的真实数据表（详见附录 A）与业务端点（详见附录 C）。

### M1 · 少流失

- **①损失场景**：高价值客户消费频次下降却无人发现，等到彻底流失才察觉；疗程做到一半中断；储值余额睡在账上不再来。
- **②监测信号**：`Customer.lastVisitDate` 超出该客个人到店周期、`CustomerPredictionSnapshot.churnScore` 升高、`CustomerLifecycleSnapshot.lifecycleStage` 转为流失预警、`CustomerBalanceAccount` 有余额但久未消费、疗程 `CustomerServiceCycleState` 中断。触发：每日扫描 + 预测跑批后。
- **③诊断口径**：区分"自然周期未到"与"真流失"——结合客户历史到店间隔、季节性、是否已改约。高价值定义对齐 `ltvTier`/`totalSpent`，避免把低价值偶发客当流失惊扰。
- **④堵漏建议**：按流失原因分层给挽回策略（唤醒券、专属回访、疗程续做提醒、余额提醒），每条带预计挽回金额（基于该客 LTV）与话术草稿。
- **⑤受控执行**：生成跟进任务（`TerminalFollowUpTask`，L2）派给合适的人；营销群发/发券（L3）走审批；绝不自动群发。
- **⑥复盘度量**：高价值客户流失率、挽回成功率与金额；哪类话术/权益对哪类客户有效，回写经营记忆。

### M2 · 少漏单

- **①损失场景**：客人到店只做了单项没升单、次卡该续没续、储值该充没充、推荐时机错过。
- **②监测信号**：`Reservation`/`ServiceTask` 完成但无关联升单订单、`CustomerCard.remainingTimes` 见底未续、`OrderItem` 结构显示单一项目、`recommendation` 快照有高分推荐未采纳。触发：服务完成时 + 每日扫描。
- **③诊断口径**：转化口径以实际成交订单为准（`ProductOrder`/`OrderItem`），排除已升单、已续卡的；升单建议须匹配客户偏好与禁忌（`CustomerHealthProfile`），不硬推。
- **④堵漏建议**：给前台/美容师"此刻该说什么"的升单/续卡/充值建议 + 匹配理由 + 权益，标注预计增收。
- **⑤受控执行**：生成收银/开卡/充值草案（L3，`terminal/cashier/checkout`、`orders/member-cards/*`）走确认与审批；提醒话术（L1）可直接给员工。
- **⑥复盘度量**：到店转化率、客单价、续卡率、升单采纳率；沉淀高转化场景。

### M3 · 少空档

- **①损失场景**：档期空置是纯利润损失——房间/技师闲置一小时，成本照付收入为零；热门技师排满流失客人，其他技师却闲着。
- **②监测信号**：`AppointmentGapOpportunity`（已带 `estimatedRevenue`/`expectedFillRate`/`availableCapacity`）、技师 `Schedule` 与 `Reservation` 的负载不均、`SchedulingRuleConfig` 峰谷规则下的空置时段。触发：实时排班变动 + 每日预判。
- **③诊断口径**：空档挽回潜力按 `estimatedRevenue` 与可匹配客户池估算；区分"结构性空档"（该时段本就无客）与"可填空档"（有潜在客户）。
- **④堵漏建议**：给"填谁、用什么项目、发给谁"的空档填充方案（对接 M1/M2 的可唤醒客户），或调整排班均衡负载，标注预计挽回收入。
- **⑤受控执行**：生成空档营销草案（`gap-opportunities/:id/{follow-up-tasks,benefit-draft}`，L2/L3）走审批；排班调整（`scheduling`，L3）需确认。
- **⑥复盘度量**：空档利用率、空档挽回收入、排班均衡度。

### M4 · 少缺货

- **①损失场景**：耗材断供导致项目做不了、临期未处理直接报损、账实不符导致成本失真。
- **②监测信号**：`Product.currentStock` 低于 `safetyStock`、`StockBatch.expiryDate` 临期、`StockMovement` 消耗速率突增、项目 `ProjectBomItem` 依赖的耗材将不足以支撑近期预约量。触发：库存变动 + 每日扫描。
- **③诊断口径**：补货需求 = 近期预约/疗程排期反推的耗材消耗 − 现有可用库存（含批次效期），排除已在途采购；账实不符靠 `StockMovement` 流水与 `Product.currentStock` 核对。
- **④堵漏建议**：补货建议（补什么、补多少、何时下单）、临期处理（促销消化/调拨）、账实差异核查，标注停服风险与损耗额。
- **⑤受控执行**：采购草案（`inventory/purchase-orders`，L2 草拟）、入库/调整/调拨（L3，`inventory/{inbound,adjustments,transfers}`）走审批。
- **⑥复盘度量**：缺货停服次数、临期损耗额、账实准确率。

### M5 · 少乱账

- **①损失场景**：退款异常无人核、会员卡负债失真看不清真实经营质量、日结对账差异、提成算错引发员工纠纷。
- **②监测信号**：`RefundRecord` 金额/频次异常、`CustomerBalanceTransaction` 调整异常、`DailySettlement.cashDiff` 对账差异、`CommissionRecord` 与规则不符、预收负债（未核销次卡+未消费余额）与实收背离。触发：收银班次关闭 + 日结生成 + 每日扫描。
- **③诊断口径**：严格对齐现有经营口径——次卡收入按核销确认（`CardUsageRecord.recognizedAmount`）而非售卡时确认、材料成本取 `StockMovement.costAmount`、预收负债=未核销次卡价值+未消费储值余额（附录 B）。这是 M5 的生命线，口径错则全盘错。
- **④堵漏建议**：异常明细 + 证据 + 建议核查动作；负债与现金流健康度解读；提成差异定位到规则与订单。
- **⑤受控执行**：财务类几乎全是 L4（退款、余额调整、结算生成/确认、日结确认），默认禁止自动执行，只生成核查任务与申请单，强制人工审批链。
- **⑥复盘度量**：对账差异额、异常拦截数、财务操作合规率。

### M6 · 少投诉

- **①损失场景**：投诉响应慢发酵成差评、售后拖延、满意度下滑无人预警。
- **②监测信号**：投诉/差评工单、回访 `TerminalFollowUpTask` 逾期、服务后满意度低分、疗程效果反馈差。触发：投诉产生 + 服务后 + 每日扫描。
- **③诊断口径**：区分个案与系统性问题（同一技师/项目/门店的投诉聚集）；优先级按客户价值与发酵风险排序。
- **④堵漏建议**：补救方案（补做/补偿/升级处理）、回访话术、需升级给店长/老板的案例，标注客户流失风险。
- **⑤受控执行**：补救任务与回访（L2）派单；补偿类涉及资产的（L4）走审批；自动触达按客户授权。
- **⑥复盘度量**：投诉响应时长、差评转化/挽回率、满意度趋势、系统性问题识别数。

### M7 · 少靠经验

- **①损失场景**：经营决策依赖店长个人经验，好店长难复制、离职即失忆、跨店质量不齐、新店长上手慢。
- **②监测信号**：这是一条元使命——监测的是"其他六个使命的建议是否被采纳、经营动作是否标准化、决策是否有据可依"。信号来自 `AgentV6Feedback`（采纳/拒绝）、经营动作的记录完整度。
- **③诊断口径**：把资深店长的判断显性化为可复用的经营记忆与 SOP（哪类漏点用哪类动作、口径如何、审批给谁），并度量标准化覆盖率。
- **④堵漏建议**：每日经营驾驶舱给出"今天最该处理的事"优先级清单，把散落判断收敛为一张行动卡；为新店长提供带解释的决策辅助。
- **⑤受控执行**：驾驶舱一键生成 M1–M6 的对应任务/方案/审批草案，人确认后推进。
- **⑥复盘度量**：建议采纳率、经营动作标准化率、跨店经营质量方差、新店长上手周期。

## 4. 顶层体验：每日经营驾驶舱作为"使命总汇"

七个使命不是七个独立入口，而是汇聚到一块**每日经营驾驶舱**——这是 M7（少靠经验）的产品化，也是老板/店长每天的第一屏。

驾驶舱每天自动生成"今天最该处理的事"，按预计损失/收益排序，每条卡片对应一个使命的一次堵漏机会：

```
今日经营行动卡（示例）
┌──────────────────────────────────────────────┐
│ 🔴 3 位高价值客户本周流失风险↑   [M1]  预计损失 ¥8,600 │
│    → 查看名单 / 一键生成回访任务 / 看挽回话术            │
├──────────────────────────────────────────────┤
│ 🟠 周六下午 2 个黄金档空置        [M3]  预计挽回 ¥2,400 │
│    → 看可约客户 / 生成填充方案                         │
├──────────────────────────────────────────────┤
│ 🟡 补水精华 3 天后不够用          [M4]  停服风险        │
│    → 看消耗测算 / 生成采购草案                         │
├──────────────────────────────────────────────┤
│ 🔴 昨日退款 2 笔待核             [M5]  差异 ¥1,200    │
│    → 看明细证据 / 生成核查任务(需审批)                 │
└──────────────────────────────────────────────┘
```

驾驶舱可继续对话追问下钻："为什么这个客户排第一""这条谁处理最合适""先给我 3 条跟进话术""这个动作预计带来多少收入"。这就是"经营对话工作台"——它不是一个空白聊天框，而是围绕使命卡片的追问与执行入口。

## 5. 目标用户 × 使命映射

用户价值直接由"他关心的损失被谁堵住"定义：

| 用户 | 最痛的损失 | 主要吃到的使命红利 |
| --- | --- | --- |
| 老板/投资人 | 利润被各种漏点吃掉、跨店质量不齐 | 全部 M1–M7 的汇总视图 + M7 标准化 |
| 店长 | 每天靠人巡检、决策靠经验 | M7 驾驶舱 + M3 空档 + M5 乱账 |
| 前台 | 漏单、空档、接待顾不过来 | M2 漏单 + M3 空档 |
| 美容师/技师 | 不了解客户偏好、错过复购/护理时机 | M1 流失 + M2 升单 |
| 营销运营 | 活动无的放矢、转化归因说不清 | M1 挽回 + M2 转化 + M3 填充 |
| 财务/出纳 | 乱账、对账差异、负债看不清 | M5 乱账 |
| 库存/采购 | 断货停服、临期报损 | M4 缺货 |
| 客服/私域 | 投诉发酵、沉睡客户唤不醒 | M6 投诉 + M1 流失 |
| 平台管理员 | Agent 越权/失控/说不清价值 | 横向支撑（第 9 章治理与安全） |

## 6. 成功指标：以"损失下降"为验收

不以"覆盖多少功能"为成功，而以"堵回多少损失"为成功。指标直接来自七使命的核心度量：

| 层级 | 指标 | P0 目标 | P1 目标 |
| --- | --- | --- | --- |
| 结果层 | 高价值客户流失率(M1) | 建立基线 + 可归因 | 下降 ≥15% |
| 结果层 | 到店转化率/客单价(M2) | 可度量 | 客单价 ↑ ≥8% |
| 结果层 | 空档利用率(M3) | 可度量 | ↑ ≥10 个百分点 |

## 7. 覆盖范围：使命映射到全业务（无盲区的另一种表达）

"无盲区"在本文不是"覆盖 150 张表"这种技术表述，而是"七个使命的监测信号、诊断口径、执行动作，能覆盖门店全部经营域"。下表把使命反向映射到后端 11 个业务域（附录 A），证明覆盖完整：

| 业务域 | 被哪些使命消费 |
| --- | --- |
| 客户/会员 | M1 流失、M2 升单、M6 投诉 |
| 订单/交易/支付/退款 | M2 漏单、M5 乱账 |
| 次卡/储值/余额 | M1 睡眠余额、M2 续卡、M5 负债 |
| 产品/库存/BOM | M4 缺货 |
| 项目/服务 | M2 升单、M4 耗材测算、M6 效果反馈 |
| 美容师/排班/预约 | M3 空档、M2 到店 |
| 财务/提成/结算/成本 | M5 乱账、M7 利润汇总 |
| 营销/生命周期/预测/归因 | M1 挽回、M2 转化、M3 填充 |
| 落地页/小程序 | M1/M2 触点 |

## 8. 横向支撑能力（服务于使命，非独立卖点）

前面七个使命是"做什么"，本章是支撑它们的六类"怎么做"的能力。每类能力都以"它服务哪个使命"来定义需求边界，凡是不服务任何使命的能力都不做。

### 8.1 记忆：让堵漏效果可积累、让追问有上下文

记忆存在的意义是让使命闭环的第⑥步（复盘）真正形成飞轮，以及支撑对话追问。因此记忆按"对使命的作用"分类，而非为分层而分层：

- **堵漏经验记忆**（服务全部使命的第⑥步）：哪类漏点用哪类动作有效（如"对 ltvTier=高 的流失客，专属回访比发券转化高"），下次同类漏点优先复用已验证策略。
- **客户情景记忆**（服务 M1/M2/M6）：客户偏好、禁忌、到店节奏、历史消费与投诉时间线，支撑个性化挽回与升单。
- **门店经营规则记忆**（服务 M7）：门店目标、服务政策、审批规则、活动节奏，是决策标准化的载体。
- **会话短期记忆**：当前对话的代词、时间范围、刚确认的条件，支撑追问连贯。
- **实体别名记忆**：客户昵称、员工昵称、项目口语名，让口语输入能命中实体。

记忆治理红线：敏感 PII 不入明文；进入推理前过权限过滤（不跨角色泄露）；冲突提示确认不静默覆盖门店规则；用户可查看/编辑/删除/禁用。技术承载建议 PostgreSQL 事实表 + pgvector（本仓库已是 PG）。

### 8.2 语义理解与追问：让口语能触发正确的使命

语义层的职责是把一句口语（"小王最近是不是不太行"）准确落到"哪个使命、哪个口径、哪些数据"，并在信息不足时追问。

- **意图落点**：先判断这句话属于哪个使命/业务域，再映射到语义视图与指标。
- **口径消歧**：命中多口径时必须澄清（"收入"指实收/项目收入/毛利？）——错口径的答案在 M5 是灾难。
- **追问策略（澄清预算）**：只读低风险，用高置信默认并声明假设（缺时间默认本月）；写/高风险，必填槽位缺失必须先问，绝不臆断关键客户/金额/时间/门店；一次问全、给可选项、避免连环追问。
- **查询实现**：走"语义视图 + schema linking + 受控 SQL + 执行制导自我纠错"，禁止对裸库直接生成 SQL（真实脏数据下会大幅失准）。指标口径与 Ami Core 经营口径强一致（附录 B）。

追问示例——用户："帮我看看小王最近是不是不太行。" V6："你说的小王是美容师王琳还是前台王晓？看员工业绩我默认按最近 30 天服务客数、项目收入、复购和客诉来看，可以吗？"

### 8.3 多角色协同：一个漏点常常要几个角色一起堵

单使命内多为单角色处理（走单 Agent，省 token/延迟）；但复合经营问题需要多角色协同。协同由"经营问题"驱动，而非为多 Agent 而多 Agent：

- 典型协同——"小王业绩下滑"：营销 Agent 查客源、财务 Agent 查客单与提成、店长 Agent 汇总归因并给行动，数据审计 Agent 反核口径。
- 典型协同——"这个流失客怎么救"（跨 M1/M3）：客服 Agent 出回访方案、前台 Agent 查可约档期、营销 Agent 配权益。
- 编排取舍：单意图默认单步；复合任务才由店长总控 Agent 拆解并按复杂度动态决定子 Agent 数量；子任务下发带"目标+输出格式+可用工具+边界"。
- 角色的可读数据与可执行动作由权限系统（8.5）严格约束，Agent 有效权限 = 用户权限 ∩ Agent 能力上限。

角色清单：店长总控、前台接待、营销增长、财务风控、库存采购、美容师服务、客服回访、数据审计、安全治理（各角色主责使命见 1.1）。

### 8.4 全业务扫描：使命需要什么数据/动作，扫描就得供给什么

扫描（Capability Scanner）不是为了"扫全 150 张表"这个技术目标，而是为了保证每个使命的监测信号、诊断口径、执行动作都有对应的数据与接口可用。

- **扫描对象**：管理端菜单/路由/页面、后端 API/controller/DTO/Swagger、数据库 schema/字段/关系/敏感级、权限码/角色/数据范围。
- **产物 `AmiCoreCapabilityMap`**：每条含 domain/entity/relation/readCapability/writeCapability/riskLevel/permissionPolicy/toolCandidate/evidenceSource/coverageStatus/owner。
- **与使命挂钩**：能力地图的每条读/写能力标注"服务哪个使命"；若某使命所需数据缺元数据或无口径，标记 `missing_meta`，状态"不可执行"，倒逼补齐——这才是"无盲区"的落地机制。
- **原则**：P0 只读扫描；产物须人工确认才进工具注册中心；隐私/支付/资产/绩效数据默认高敏。

### 8.5 权限与安全：使命越能干，越要管得住

使命能主动发现、能建议、能执行，风险随之升高，所以权限与安全是使命的"刹车系统"。

- **权限模型**：直接继承管理端四类作用域——平台 `platformScopes`、数据 `dataScopes`（all/own_store）、字段 `fieldScopes`（visible/masked/hidden）、审批 `approvalScopes`（approve/approve_limited/request/none），不另造。门店隔离强制。
- **风险分级执行**（贯穿所有使命第⑤步）：L0 展示直接办；L1 会话确认；L2 dry-run+确认；L3 审批流+审计；L4（退款/资产/结算/权限/批量删除）默认禁止自动执行，仅生成申请单强制人工审批。真实端点到等级的映射见附录 C。
- **安全护栏**：输入护栏（prompt injection/越权识别，外部内容视为不可信）、查询护栏（只读 AST + 强制注入门店/时间/LIMIT + 字段脱敏 + 成本上限，这是安全红线）、记忆护栏（防投毒/跨角色泄露）、运行护栏（中断/降级/kill switch/人工接管）。
- **审计**：每次运行记录用户/角色/门店/输入/意图/Agent/工具/数据源/输出/证据/追问/执行差异/审批/结果。

### 8.6 可视化治理与迭代：让"堵漏能力"本身可被经营

治理中心（独立于历史治理台）面向管理员/研发，本质是"经营 V6 这支数字团队"：

- **使命健康看板**：每个使命的漏点发现量、建议采纳率、堵回金额、误报率——这是治理的第一视角（对应第 6 章结果指标）。
- **覆盖地图**：哪些业务域已被使命消费、哪些是盲区（第 7 章），标红督办。
- **运行追踪**：每次 run 的步骤/工具/耗时/成本/失败点/证据（建议 OTel GenAI + 可自托管 Langfuse）。
- **能力配置**：Ontology/指标口径/工具注册/权限策略/记忆，全部版本化。
- **评测与迭代**：上线前跑评测集+权限用例；灰度/回滚/单工具禁用；护栏拦截样本回流为新用例；低质回答可追溯到意图错/数据错/工具错/权限错/口径错/模型错。

## 9. 与历史版本、其它文档的关系

- **不复用**：本仓库已存在 Agent V1–V5（呈"每代重写、后代复用前代 text-to-sql"路径依赖）。V6 建独立运行时、独立表、独立 API `/api/agent-v6/*`、独立治理页，不复用任何历史版本的运行时/编排/提示词/能力注册/治理实现/评测题库/适配器代码。
- **作为通用基线写入（非复用实现）**：只读 SQL 安全护栏（安全红线）与经营指标口径（业务事实，附录 B）。二者是"任何版本都必须满足的标准"和"客观业务事实"，不属于复用某版本代码。
- **本文与《技术架构方案》的关系**：本文定义"堵哪些漏、堵到什么程度"（结果导向）；架构方案定义"用什么模块/接口/表实现"（技术导向）。本文的七使命是架构方案里编排层与能力层的需求来源。

## 附录 B. 权限、角色、四类作用域与指标口径

**权限码**：四段前缀 `core`/`assist`/`terminal`/`aura` + `平台:模块:动作`。**内置角色**：`super_admin`(全权限)、`store_manager`、`beautician`、`cashier`、`inventory_manager`、`supplier_admin`。

**四类作用域（V6 直接继承）**：
- 数据 `dataScopes`：all/own_store，维度 store/customer/order/booking/inventory/report/device。
- 字段 `fieldScopes`：visible/masked/hidden，含 customerPhone/Balance/Cost/Profit/staffCommission。
- 审批 `approvalScopes`：approve/approve_limited/request/none，含 refund/discount/priceChange/deleteCustomer/inventoryAdjustment。
- 平台 `platformScopes`：core/assist/terminal 开关。

**关键指标口径（M5 生命线，V6 必须一致）**：
- 次卡收入：按核销确认 `CardUsageRecord.recognizedAmount`，非售卡时确认。
- 材料成本：取 `StockMovement.costAmount`。
- 预收负债：未核销次卡价值（剩余次数×确认单价）+ 未消费储值余额。
- 毛利：收入 − 材料成本 − 提成 − 经营成本。

| L1 | 客户建档/编辑、标签、内部待办 | `POST customers`、`PUT :id`、`POST customer-events` | M1 M6 |
| L0 | 全部查询与聚合 | 各模块 `GET` | 全部使命的②③ |

## 结语

本文与此前版本的根本区别：主轴从"agent 有哪些能力"翻转为"门店要堵哪些经营损失"。七大使命（少流失/少漏单/少空档/少缺货/少乱账/少投诉/少靠经验）是需求骨架，每个使命是"损失场景→监测→诊断→建议→受控执行→复盘"的可度量闭环；你最初的六项能力要求退居为服务使命的横向支撑（第 8 章），并在第 10 章逐条对照证明无漏项。附录用真实系统扫描把使命锚定到具体表、权限、口径与端点。后续可继续输出《使命度量与损失估算模型》《Ontology 与指标口径设计》《P0（M5+M4+M1）开发计划与验收集》。
