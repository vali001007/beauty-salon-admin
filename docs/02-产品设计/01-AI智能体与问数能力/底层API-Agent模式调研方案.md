# 底层 API + Agent 模式调研方案

版本：v0.1
日期：2026-06-12
适用项目：Ami Core 管理平台、Ami Aura Lite 智能终端、Ami Glow 客户小程序、server-v2
调研主题：是否从“传统 SaaS 管理平台 + 智能问答终端”的较重模式，升级为“底层业务 API + Agent 编排 + 多入口轻交互”模式

---

## 1. 结论先行

建议进入专项调研，并以“小步试点”方式验证。

本项目不建议简单把管理平台和智能终端推倒重做，而是把产品重心从“页面驱动”逐步迁移为“API 能力驱动 + Agent 调度”。管理平台继续保留，但定位从一线操作主入口，降级为配置、审核、异常处理、数据治理和管理视图；一线员工、店长、顾客的高频任务，则通过 Agent 入口完成查询、推荐、预约、收银、核销、服务记录、营销触达等闭环。

这与微信正在推进的 AI 生态方向一致：用户不必主动打开复杂小程序逐页操作，而是通过自然语言表达意图，由 AI 调用被授权的小程序能力完成任务。对 Ami 来说，对应的不是“让 AI 点击后台页面”，而是把美业核心业务封装成可被 Agent 调用的稳定 API / Tool / Skill。

短期判断：

- 可以改，但应分阶段改，不要一次性替换现有 SaaS。
- server-v2 已经具备成为底层业务 API 的基础。
- Ami Aura Lite 已经有 intent router、micro app、AI Gateway 的雏形，可以作为 Agent 化试验场。
- Ami Glow 小程序可成为未来微信 AI 生态或客户侧 Agent 入口的前置资产。
- 最大风险不是模型能力，而是权限、审计、确认、幂等、数据一致性和工具边界设计。

## 2. 外部趋势摘要

### 2.1 微信 AI 生态的核心信号

根据 2026-06-08 后多家媒体对“微信公开课”公告的报道，微信开放平台开始面向小程序开发者提供接入微信 AI 生态的能力。开发者需要在“小程序管理后台 - AI 能力”主动授权接入；当前能力处于内测阶段，用户暂时还无法直接体验。

公开报道中提到的两种模式：

| 模式 | 含义 | 对 Ami 的启发 |
| --- | --- | --- |
| 自动模式 | 授权平台在提审时读取小程序源码，分析页面，让微信 AI 能直接操作页面 | 适合低门槛接入，但业务可控性弱；Ami 不应把核心交易依赖 GUI 自动点击 |
| 开发模式 | 开发者基于业务特性自主开发定制化能力，通过评测和审核后被微信 AI 调用 | 更适合 Ami：把预约、核销、收银、办卡、查询等封装成可审核的业务 Skill |

关键趋势：

- 入口从“用户主动找功能”变为“用户表达意图，AI 调用服务”。
- 小程序从“页面容器”变成“可被 AI 调度的能力集合”。
- 开发者仍拥有是否接入、开放哪些能力、如何封装能力的选择权。
- 支付、下单、预约等高信任场景必须有授权、确认和平台审核机制。

### 2.2 Agent 技术栈的行业共识

行业正在形成几层基础设施：

| 层级 | 代表方向 | 产品含义 |
| --- | --- | --- |
| Agent 编排 | OpenAI Agents SDK 等 | 让模型具备工具调用、交接、结构化输出、审批、可观测能力 |
| 工具 / 上下文协议 | MCP | 把数据库、业务 API、工作流以标准方式暴露给 AI 应用 |
| Agent 间协作 | Google A2A | 让不同系统、不同厂商的 Agent 能发现彼此并协作 |
| 平台生态接入 | 微信 AI 小程序生态 | 超级入口用 AI 调用小程序服务，重塑流量和服务分发 |

对本项目而言，现阶段不必追求一次性兼容所有协议，但架构上要提前按“工具化、可授权、可审计、可编排”设计，避免继续把能力锁死在前端页面里。

## 3. 当前项目模式判断

### 3.1 现有模式

当前产品可以概括为：

```text
管理平台 SaaS 页面
  + server-v2 业务 API
  + Ami Aura Lite 智能终端
  + Ami Glow 客户小程序
  + AI Gateway / 智能问答
```

优势：

- 管理端完整，适合配置、管理、批量数据维护。
- server-v2 已有较多业务模块，是真实业务主线。
- 终端已经开始从固定页面转向意图驱动。
- 小程序可以承接客户侧服务触点。

问题：

- 管理平台承载太多一线操作，门店员工学习成本高。
- 智能终端如果只是“问答 + 卡片”，仍然像另一个轻后台。
- 很多业务流程依赖页面路径，而不是可调度能力。
- AI 建议和真实业务动作之间仍存在断层。
- 未来接微信 AI 或其他 Agent 平台时，如果没有 Skill 层，会被迫走不稳定的页面自动化。

### 3.2 目标模式

目标模式建议定义为：

```text
业务事实层：PostgreSQL / Prisma / server-v2
能力层：Core API + Tool Schema + Skill Registry
编排层：Agent Orchestrator + Intent Router + Policy Engine
体验层：管理平台 / Aura Lite / Glow 小程序 / 微信 AI / 语音入口 / 未来第三方 Agent
治理层：权限、确认、幂等、审计、监控、成本、回放
```

产品定位变化：

| 旧定位 | 新定位 |
| --- | --- |
| 管理平台是主入口 | 管理平台是控制台和治理后台 |
| 终端是轻量业务页面 | 终端是 Agent 的现场工作台 |
| AI 是问答和建议 | AI 是意图理解、工具编排、澄清和解释层 |
| API 服务前端页面 | API 是可授权、可组合、可审计的业务能力 |
| 用户找菜单 | 用户表达目标，系统生成任务流 |

## 4. 调研目标

本次调研要回答 6 个核心问题：

1. 产品是否应该从“后台功能堆叠”转为“Agent 调度业务能力”？
2. 哪些美业场景适合第一批 Agent 化，哪些必须保留传统页面？
3. server-v2 现有 API 是否足以成为 Tool / Skill 层，缺哪些原子能力？
4. Agent 执行写操作时，如何做权限、确认、幂等、审计和回滚？
5. 如果未来接入微信 AI 生态，Ami Glow 小程序和 Core API 应如何提前准备？
6. 该模式是否能降低门店使用成本、提高闭环效率，并形成产品差异化？

## 5. 调研范围

### 5.1 业务场景范围

优先调研 P0 场景：

| 场景 | 典型用户表达 | Agent 动作 |
| --- | --- | --- |
| 今日经营 | “今天店里情况怎么样？” | 查询经营指标、异常、预约、库存，生成行动建议 |
| 客户查询 | “查一下张女士最近做过什么” | 搜索客户、汇总档案、卡项、消费、皮肤记录 |
| 预约创建 / 改期 | “帮王芳约明天下午补水” | 补齐客户、项目、美容师、时段，确认后创建预约 |
| 到店接待 | “李娜到了” | 查预约、签到、提示卡项和注意事项 |
| 次卡核销 | “给张三核销补水次卡” | 预核销、展示余额和项目，确认后写入核销 |
| 收银结账 | “这个顾客买单” | 拉取服务/商品/卡项，生成订单，确认支付 |
| 服务记录 | “刚做完小气泡，记录一下” | 生成服务记录、耗材、下次建议，确认后写回 |
| 营销跟进 | “今天哪些客户该联系？” | 生成客户名单、理由、话术，确认后触达或生成任务 |

暂不作为第一批自动执行场景：

- 大额退款、撤销收银、批量删除、批量改价。
- 员工薪酬结算、供应商对账、财务关账。
- 无用户确认的营销群发。
- 无审核的客户敏感信息导出。

### 5.2 系统范围

纳入调研：

- `packages/server-v2`
- `packages/Ami-Aura-Lite-Kiosk`
- `packages/Ami-Glow-MiniApp`
- `src/api/real/*`
- `docs/terminal-api.md`
- `docs/api-contract.md`
- 现有 AI Gateway 与终端 intent 相关实现

不纳入本轮：

- 大规模 UI 重构。
- 删除旧页面或清理历史文档。
- 更换数据库或后端框架。
- 直接上线微信 AI 内测能力。

## 6. 核心调研假设

### 假设 1：API + Agent 可以降低一线员工操作成本

验证方式：

- 选 5 个高频任务，对比传统页面点击路径与 Agent 任务流。
- 记录完成步骤数、耗时、错误率、是否需要培训。
- 输出任务路径对比表。

成功标准：

- 高频任务步骤减少 50% 以上。
- 新员工无需理解完整菜单树即可完成 P0 任务。
- 错误提示能转化为业务语言，而不是技术错误。

### 假设 2：server-v2 现有 API 可支撑第一批 Agent Tool

验证方式：

- 梳理每个场景所需的查询 API、预检查 API、写入 API。
- 判断是否已有专用接口、是否可复用、是否需要新建原子接口。
- 标注每个 Tool 的输入、输出、权限、风险级别。

成功标准：

- P0 场景 70% 以上能力可由现有 API 或轻量改造覆盖。
- 所有写操作都有明确确认点和审计字段。

### 假设 3：Agent 化后管理平台仍有必要，但角色会变化

验证方式：

- 梳理哪些功能应保留页面，哪些可转为 Agent 入口。
- 访谈或模拟店长、前台、美容师的日常路径。
- 输出“页面保留 / Agent 优先 / 双入口”清单。

成功标准：

- 配置型、审核型、批量型、财务型功能继续由后台承载。
- 高频现场任务优先 Agent 化。
- 管理平台不再被要求覆盖所有一线微操作。

### 假设 4：微信 AI 生态会影响客户侧和商家侧入口

验证方式：

- 对照微信 AI 小程序自动模式 / 开发模式。
- 判断 Ami Glow 是否具备被 AI 调用的页面和能力基础。
- 设计一版“微信 AI 可调用 Skill 候选清单”。

成功标准：

- 能明确哪些能力适合未来开放给微信 AI。
- 敏感能力默认不开放或只开放查询 / 预约草稿。
- 客户侧与员工侧权限边界清晰。

## 7. 建议调研方法

### 7.1 外部研究

研究对象：

- 微信 AI 小程序生态接入方式。
- 主流 Agent 编排框架。
- MCP / A2A 等协议的定位。
- 垂直行业 SaaS Agent 化案例。
- AI 执行业务动作的权限、安全和审计模式。

输出物：

- 外部趋势摘要。
- 微信 AI 对 Ami 的机会与限制。
- 可借鉴能力清单。
- 不建议照搬的能力清单。

### 7.2 内部能力盘点

盘点维度：

| 维度 | 需要回答 |
| --- | --- |
| API 完整度 | 是否有原子查询、预检查、写入、回滚或取消能力 |
| 权限 | 是否能按角色、门店、动作控制 |
| 数据一致性 | 写操作是否幂等，是否能避免重复提交 |
| 审计 | 谁在何时通过什么入口让 Agent 做了什么 |
| 错误处理 | 能否把错误转成一线人员可理解的话 |
| 前端承载 | 是否需要页面、卡片、弹窗、确认流 |
| 运营价值 | 是否能减少培训、提升转化、提高复购 |

输出物：

- P0 场景能力矩阵。
- Tool / Skill 候选清单。
- API 缺口清单。
- 管理平台保留功能清单。

### 7.3 原型验证

建议做一个 2 周 PoC，不追求全量改造。

PoC 场景建议：

1. “查客户 + 生成服务建议”
2. “预约创建 / 改期”
3. “次卡核销”
4. “今日经营异常 + 建议动作”

PoC 最小闭环：

```text
用户自然语言
  -> Intent Router
  -> Agent Orchestrator
  -> Tool Registry
  -> server-v2 API
  -> 业务卡片 / 确认卡
  -> 用户确认
  -> 写回 Core
  -> 审计日志
```

PoC 不做：

- 无确认自动交易。
- 微信 AI 正式接入。
- 完整语音实时对话。
- 复杂多 Agent 协作。

## 8. 目标架构草案

### 8.1 Agent 能力分层

```text
入口层
  管理端命令栏 / Aura Lite / Ami Glow / 微信 AI / 语音

意图层
  规则 Intent Parser
  AI Intent Parser
  槽位抽取
  澄清问题生成

编排层
  Agent Orchestrator
  场景 Agent：接待、预约、收银、客户经营、店长经营、营销
  Policy Engine：权限、风险、确认、门店边界

工具层
  Tool Registry / Skill Registry
  查询类 Tool
  预检查类 Tool
  写入类 Tool
  通知类 Tool

业务层
  server-v2 API
  Prisma / PostgreSQL
  AI Gateway

治理层
  审计日志
  幂等键
  Trace
  成本统计
  人工接管
```

### 8.2 Tool 类型建议

| Tool 类型 | 示例 | 风险等级 | 是否需要确认 |
| --- | --- | --- | --- |
| 只读查询 | 查客户、查预约、查库存、查卡项 | 低 | 否 |
| 草稿生成 | 生成预约草稿、营销话术、服务建议 | 中 | 否，但需标注来源 |
| 预检查 | 核销预览、收银预览、可约时段检查 | 中 | 否 |
| 业务写入 | 创建预约、完成核销、完成收银、写服务记录 | 高 | 是 |
| 高风险写入 | 退款、撤销、批量触达、导出敏感数据 | 极高 | 是，且需更高权限或二次审批 |

### 8.3 管理平台的新定位

管理平台不消失，而是承担：

- 业务配置：项目、商品、卡项、角色、门店、支付方式。
- 策略配置：推荐规则、营销规则、Agent 可调用能力范围。
- 审核中心：高风险动作审批、异常交易复核。
- 数据治理：客户合并、字段修正、导入导出。
- 监控运营：Agent 使用量、转化率、失败率、成本、采纳率。
- 兜底操作：Agent 无法处理时，回到完整页面处理。

## 9. 微信 AI 生态映射到 Ami 的机会

### 9.1 可能机会

| 微信 AI 能力趋势 | Ami 可做的准备 |
| --- | --- |
| 用户一句话调用小程序服务 | Ami Glow 提前梳理客户可用 Skill，如预约、查卡、查护理建议 |
| 开发者自主声明能力边界 | 建立 Ami Skill Registry，定义每个 Skill 的权限和输入输出 |
| AI 可操作页面或调用 Skill | 优先做 Skill / API 模式，页面模式只作为低风险兜底 |
| 小程序从流量入口变成服务能力 | 把客户侧复购、预约、售后、护理提醒设计成可调用服务 |
| 平台会审核开发者能力 | 提前规范数据来源、用户授权、隐私说明、敏感动作确认 |

### 9.2 不建议照搬

- 不建议让 Agent 直接模拟点击管理后台完成交易。
- 不建议把所有后台菜单开放给微信 AI。
- 不建议在没有门店授权、顾客授权的情况下开放客户档案。
- 不建议让 AI 自动群发营销内容。
- 不建议把“模型能不能理解”当成唯一验收标准，业务写入必须由 Core 兜底。

## 12. 关键评估指标

产品指标：

- 高频任务完成步骤数下降比例。
- 新员工完成任务所需培训时间。
- Agent 建议采纳率。
- 预约、复购、核销、营销转化提升。
- 门店日常问题自助解决率。

技术指标：

- Intent 命中率。
- 低置信度澄清率。
- Tool 调用成功率。
- 写操作重复提交率。
- API P95 延迟。
- Agent 单次任务成本。
- 审计日志完整率。

安全指标：

- 高风险动作确认覆盖率。
- 越权拦截率。
- 敏感信息脱敏覆盖率。
- Agent 错误执行可追溯率。

## 13. 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| Agent 幻觉导致错误业务动作 | 高 | AI 不直接写事实；写操作必须走预检查、确认和 Core API |
| 权限边界不清 | 高 | 每个 Tool 绑定 permission、role、store scope |
| 重复提交交易 | 高 | 写 Tool 强制 idempotencyKey |
| 错误难追踪 | 高 | 每次 Agent run 记录 traceId、toolCallId、operatorId、storeId |
| API 粒度太粗 | 中 | 从 P0 场景反推原子 Tool，避免直接复用大页面接口 |
| 模型延迟影响现场效率 | 中 | 规则优先、AI fallback；查询并行；结果流式返回 |
| 微信 AI 接入规则变化 | 中 | 先做内部 Skill Registry，不绑定单一平台 |
| 一线员工不信任 Agent | 中 | 所有建议展示数据依据，允许人工改写和接管 |

## 14. 初步实施路线

### 阶段 1：内部 Agent 化，不碰微信正式接入

目标：

- 先在 Ami Aura Lite 内完成 Agent 工作台闭环。
- server-v2 建立 Tool Registry 雏形。
- 管理平台增加 Agent 能力配置和审计查看。

建议周期：3-5 周。

### 阶段 2：客户侧轻 Agent

目标：

- Ami Glow 小程序支持自然语言式预约、查卡、查护理建议。
- 客户授权后才能读取个人档案。
- 不开放高风险交易动作。

建议周期：3-4 周。

### 阶段 3：微信 AI 生态预接入

目标：

- 按微信开发模式思路准备可被调用的 Skill。
- 明确开放能力、隐私说明、用户授权、审核材料。
- 先开放查询、预约草稿、服务介绍、门店活动等低风险能力。

建议周期：视微信 AI 生态开放节奏决定。

### 阶段 4：多 Agent 与生态合作

目标：

- 根据需要评估 MCP / A2A 或类似协议。
- 与供应链、营销、财务、客服等外部系统做 Agent 间协作。

建议周期：中长期。

## 16. 决策建议

建议立项“API + Agent 模式调研”，并把目标定为：

1. 不推翻现有 SaaS，而是重定产品分工。
2. 不把 AI 当聊天插件，而是当业务编排层。
3. 不让 Agent 操作页面完成交易，而是让 Agent 调用受控 API。
4. 不先追求接入微信 AI，而是先把自身 Skill 能力做规范。
5. 先从 Aura Lite 的店内高频任务做 PoC，再扩展到 Ami Glow 和微信生态。

如果调研结论成立，Ami 的产品形态可以从“美业 SaaS + 智能终端”升级为：

```text
Ami Core：美业业务事实与规则底座
Ami Agent：门店经营与服务任务编排层
Ami Console：配置、治理、审核与数据后台
Ami Aura：店内现场 Agent 工作台
Ami Glow：客户侧 Agent 服务入口
```

这会比单纯继续扩管理后台页面更轻，也更符合微信 AI 生态正在释放的方向。
