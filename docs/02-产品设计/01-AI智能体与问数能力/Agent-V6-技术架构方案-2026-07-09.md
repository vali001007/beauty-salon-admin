# Agent V6 数字店长经营智能体 · 技术架构方案

- 版本：v1.0
- 日期：2026-07-09
- 上游文档：《Agent V6 数字店长经营智能体 · 产品需求文档（Clean-Room 独立版）》
- 文档性质：技术架构方案，面向研发、数据与平台负责人
- 目标技术栈：NestJS 11 + Prisma 7 + PostgreSQL（Supabase）+ React 18/Vite（治理台复用管理端）
- 独立性约束：V6 独立运行时、独立数据表、独立 API 命名空间 `/api/agent-v6/*`、独立治理页；不复用历史 Agent V1–V5 的运行时/编排器/提示词/能力注册/治理实现/评测题库/适配器代码

---

## 0. 架构原则

本方案遵循七条架构原则，贯穿所有模块设计：

1. **只读优先、写操作强确认**：读路径默认自动执行，写路径按风险分级（L0–L4），L3/L4 强制 dry-run + 审批 + 审计。
2. **权限是护栏输入、不可协商**：所有数据访问与工具调用先过策略引擎，模型无法绕过 RBAC/ABAC、门店隔离与字段脱敏。
3. **语义层而非裸库**：查询走"语义视图 + 口径/别名/指标"，不对物理表直接生成 SQL，规避 schema boundary problem。
4. **一切可追溯**：每个 run/step/tool-call/approval 全链路 trace，答案必带证据（来源、时间、口径）。
5. **配置驱动、版本化**：Ontology、指标、工具、提示策略、权限策略均为可版本化配置，与核心运行时解耦。
6. **能力来自扫描、须人工确认**：工具与视图由 Capability Scanner 从 Ami Core 扫描生成候选，经人工审核后才可用。
7. **模型可插拔**：编排与业务能力不绑定单一模型供应商，经统一模型网关调用。

## 1. 总体分层架构

V6 分为六层，自上而下：接入层、编排层、能力层、语义与知识层、集成层、数据与可观测层。治理与安全为横切面，贯穿所有层。

```
┌───────────────────────────────────────────────────────────────────────┐
│ 接入层 Access         经营对话工作台 / 每日驾驶舱 / 协同任务卡 / 治理台      │
│                       (React，复用管理端 src/；WebSocket 流式)             │
├───────────────────────────────────────────────────────────────────────┤
│ 编排层 Orchestration  意图理解 → 记忆召回 → 澄清判定 → 计划 → 角色分派      │
│                       Orchestrator / Planner / Router / ClarificationEngine│
├───────────────────────────────────────────────────────────────────────┤
│ 能力层 Capability     角色 Agent(9) + Skill(查询/风险/建议/操作) + ToolRegistry│
│                       PolicyEngine + ApprovalGateway + EvidenceEngine     │
├───────────────────────────────────────────────────────────────────────┤
│ 语义与知识层 Semantic 语义视图注册表 + 指标注册表 + Ontology(节点/边)        │
│                       别名词表 + SchemaLinking + Text-to-SQL + 自我纠错     │
├───────────────────────────────────────────────────────────────────────┤
│ 集成层 Integration    CapabilityScanner + Ami Core API Adapter            │
│                       只读 DB Read Model + 写操作 Command Gateway          │
├───────────────────────────────────────────────────────────────────────┤
│ 数据与可观测层 Data    V6 独立表(run/memory/tool/eval/ontology...)          │
│                       pgvector 记忆 + OTel Trace + 审计日志                │
└───────────────────────────────────────────────────────────────────────┘
      横切：模型网关(ModelGateway) · 治理中心(Governance) · 安全护栏(Guardrails)
```

### 1.1 各层职责与边界

| 层 | 职责 | 关键约束 |
| --- | --- | --- |
| 接入层 | 呈现对话/驾驶舱/任务卡/治理台，流式输出，收集反馈 | 不含业务逻辑，只调用 `/api/agent-v6/*` |
| 编排层 | 把自然语言变成"可执行计划"，决定单/多 Agent、是否追问 | 单意图默认单步；不臆断关键槽位 |
| 能力层 | 角色 Agent 调用技能与工具，产出结论/建议/操作草案 | 工具调用前必过策略引擎；写操作走审批 |
| 语义与知识层 | 把业务概念映射到数据，保证口径统一与查询安全 | 只读语义视图；禁裸库；强制门店/时间/LIMIT 注入 |
| 集成层 | 扫描并适配 Ami Core 的读写能力，屏蔽后端差异 | 读走 Read Model，写走 Command Gateway（幂等+审计） |
| 数据与可观测层 | 独立持久化 V6 运行态、记忆、配置与轨迹 | 与历史 Agent 表物理隔离；PII 脱敏存储 |

### 1.2 独立性与 Ami Core 的关系

V6 是**独立子系统**，通过明确的集成契约接入 Ami Core，而非嵌入其业务模块：

- 读：优先经"只读 Read Model / 语义视图"访问 PostgreSQL，避免与业务事务耦合。
- 写：一律通过 Ami Core 现有业务 API（Command Gateway 封装），复用其事务、校验与权限，V6 不直接改业务表。
- 权限：V6 以发起用户的 JWT 身份调用，继承其权限与门店范围，Agent 自身权限取"用户权限 ∩ Agent 能力上限"。

## 2. 运行时时序

### 2.1 只读问数路径（最高频）

```
用户输入
  │
  ▼
[1] Orchestrator 建 Run(记录 user/role/storeIds/input)
  │
  ▼
[2] MemoryService 召回(会话短期 + 用户偏好 + 实体别名)，做代词/时间消解
  │
  ▼
[3] Router 意图识别 → 命中意图域 + 目标语义视图/指标
  │
  ▼
[4] ClarificationEngine 判定：关键槽位齐? 口径唯一?
  │       ├─ 缺失/歧义(读，低风险) → 用默认并声明假设，继续
  │       └─ 缺失(写/高风险)       → 返回澄清问题，暂停 Run
  ▼
[5] SemanticLayer: SchemaLinking → 生成受控 SQL(仅白名单视图)
  │
  ▼
[6] Guardrails/SqlGuard: AST 校验 + 注入 store_id/时间/LIMIT + 字段脱敏 + 成本护栏
  │
  ▼
[7] ReadModel 执行 → 结果校验(可执行性/合理性)；失败则带错重生成(限重试N次)
  │
  ▼
[8] EvidenceEngine 组装答案(结论+证据来源+时间+口径+不确定项)
  │
  ▼
[9] 流式返回 + 写 Trace/审计；MemoryService 抽取事实写长期记忆
```

### 2.2 写操作路径（风险分级）

```
用户意图(含写动作)
  │
  ▼
[1-4] 同上(意图识别 + 记忆 + 澄清；写操作必须槽位齐全，绝不臆断)
  │
  ▼
[5] PolicyEngine 鉴权：用户权限 ∩ Agent 能力上限 ∩ 门店范围 ∩ approvalScopes
  │       └─ 无权限 → 拒绝并说明
  ▼
[6] 风险分级：
  │   L0/L1 → 会话确认后经 Command Gateway 执行
  │   L2    → dry-run 预览影响 → 用户确认 → 执行(可撤销)
  │   L3    → dry-run + 生成审批单 → 有权限者审批 → 执行 + 审计
  │   L4    → 默认禁止自动执行，仅生成申请单，强制人工审批链
  ▼
[7] Command Gateway 调用 Ami Core 业务 API(幂等键 + 参数校验)
  │
  ▼
[8] 回写结果差异(执行前后) + 审批记录 + 全链路审计；记忆更新决策记录
```

### 2.3 事件驱动主动经营路径（P1+）

```
EventScheduler 定时/事件触发(如库存低于安全值、爽约率升高)
  → 对应角色 Agent 主动分析 → 生成 RiskSignal + 建议 + 操作草案(L2)
  → 推送到驾驶舱/待办，等待经营者确认，不自动执行 L3/L4
```

### 2.4 多 Agent 复合任务路径（P1+）

店长总控 Agent 拆解目标 → 并行/串行分派营销/财务/库存等专业 Agent（各自独立上下文与工具边界）→ 数据审计 Agent 互审口径与证据 → 店长 Agent 汇总为一张任务卡。子任务下发携带"目标 + 输出格式 + 可用工具 + 边界"，按复杂度动态决定子 Agent 数量以控制 token/延迟。

## 3. 模块设计与接口

所有模块以 NestJS Service 组织，置于 `packages/server-v2/src/agent-v6/`（独立目录，不与 `agent`~`agent-v5` 混放）。以下给出关键接口签名（TypeScript 伪代码，定稿以实现为准）。

### 3.1 AgentV6Orchestrator（主编排）

```ts
interface OrchestrateInput {
  runId?: string;            // 续接多轮时传入
  userId: string;
  roleCodes: string[];       // 用户角色
  storeIds: string[];        // 可访问门店(数据隔离)
  permissions: string[];     // 归一化权限码
  channel: 'web' | 'kiosk' | 'event';
  input: { type: 'text' | 'voice' | 'event'; content: string; payload?: unknown };
}
interface OrchestrateResult {
  runId: string;
  status: 'answered' | 'need_clarification' | 'pending_approval' | 'failed';
  answer?: AnswerEnvelope;         // 结论+证据+不确定项
  clarification?: ClarificationAsk; // 需要追问时
  approval?: ApprovalRequest;       // 触发审批时
  taskCard?: MultiAgentTaskCard;    // 多 Agent 协同时
}
class AgentV6OrchestratorService {
  orchestrate(input: OrchestrateInput): Promise<OrchestrateResult>;
  resume(runId: string, userReply: ClarificationReply | ApprovalDecision): Promise<OrchestrateResult>;
}
```

编排状态机：`created → memory_recalled → intent_resolved → (clarifying) → planned → executing → (approving) → composed → done|failed`。状态持久化在 `AgentV6Run.state`，支持中断续接。

### 3.2 AgentV6MemoryService（分层记忆）

```ts
type MemoryKind = 'session' | 'user_pref' | 'store_policy' | 'entity_alias'
                | 'episodic' | 'decision' | 'feedback' | 'task';
interface MemoryItem {
  id: string; kind: MemoryKind;
  scope: { storeId?: string; userId?: string; customerId?: string; role?: string };
  content: string; embedding?: number[];
  source: string; confidence: number;    // 0~1
  validUntil?: Date; createdAt: Date; updatedAt: Date;
  sensitive: boolean;                     // 敏感信息不入明文
}
class AgentV6MemoryService {
  recall(ctx: RecallContext): Promise<MemoryItem[]>;   // 语义相似 + 时间衰减 + 作用域过滤
  write(items: MemoryDraft[]): Promise<void>;          // 抽取事实，冲突"新覆盖旧"并留审计
  resolveAlias(term: string, scope): Promise<EntityRef | null>; // 口语名→实体
  govern(): MemoryAdminApi;                            // 查看/编辑/删除/禁用/合并
}
```

约束：`recall` 结果进入推理前必过权限过滤（不跨角色泄露）；敏感字段（手机号、身份证）只存脱敏或引用；记忆冲突提示确认，不静默覆盖门店经营规则。技术选型：短期记忆用滚动摘要；长期记忆用 PostgreSQL 事实表 + pgvector 向量检索（本仓库已是 PG，落地成本低）；情景记忆按时间戳建序列，支撑"上次做了什么"。

### 3.3 ClarificationEngine（澄清）

```ts
interface ClarificationDecision {
  action: 'proceed_with_defaults' | 'ask';
  assumptions?: Assumption[];   // proceed 时声明的默认(如"默认本月")
  ask?: ClarificationAsk;       // ask 时的问题(合并多缺口，给可选项)
}
class ClarificationEngine {
  decide(intent: ResolvedIntent, ctx: RunContext): ClarificationDecision;
}
```

策略：读+低风险→高置信默认并声明假设；写+中高风险→必填槽位缺失或口径歧义必须 `ask`；一次问全、给可选项、避免连环追问。

### 3.4 SemanticLayer（语义层 + Text-to-SQL）

```ts
interface SemanticView {
  viewName: string; domain: string;         // 附录 A 的 11 域之一
  requiredPermissions: string[]; adminOnly: boolean;
  storeScopeField: string; defaultTimeField?: string;
  fields: { name: string; policy: 'dimension' | 'metric' | 'masked' | 'deny' }[];
  aliases: string[]; sampleQuestions: string[];
  status: 'enabled' | 'planned'; batch: 'P0' | 'P1' | 'P2';
}
interface MetricDefinition {
  canonicalName: string; aliases: string[];
  businessDefinition: string; formula: string;   // 与 Ami Core 口径一致(附录 C)
  sourceViews: string[]; timeGrain: string;
  dimensions: string[]; permissionLevel: string; drillable: boolean;
}
class AgentV6SemanticLayer {
  linkSchema(intent: ResolvedIntent): Promise<{ views: SemanticView[]; metrics: MetricDefinition[] }>;
  buildSql(plan: QueryPlan, ctx: RunContext): Promise<ControlledSql>; // 只选白名单视图/字段
}
```

Text-to-SQL 采用"语义视图 + schema linking + 受控生成"，禁止对物理表直接生成。结果校验失败（执行/EXPLAIN 报错或结果异常）时，带错误信息重生成，重试上限 N 次（建议 2）。指标口径与经营利润聚合逻辑对齐：次卡收入按核销确认、材料成本取库存流水成本额、预收负债=未核销次卡价值+未消费余额（见需求文档附录 C）。

### 3.5 AgentV6ToolRegistry（工具注册中心）

```ts
interface ToolDefinition {
  toolId: string; name: string; kind: 'query' | 'command';
  domain: string; inputSchema: JsonSchema; outputSchema: JsonSchema;
  requiredPermissions: string[]; riskLevel: 'L0'|'L1'|'L2'|'L3'|'L4';
  idempotent: boolean; amiCoreEndpoint?: string;  // command 类对应的业务 API
  version: string; status: 'draft'|'review'|'enabled'|'disabled';
  evidenceSource: string;   // 来自 CapabilityScanner 的溯源
}
class AgentV6ToolRegistry {
  register(draft: ToolDraft): Promise<ToolDefinition>;   // 扫描候选→草稿
  approve(toolId: string, reviewer: string): Promise<void>; // 人工审核后启用
  resolve(intent, ctx): Promise<ToolDefinition[]>;       // 按意图+权限筛可用工具
  disable(toolId: string, reason: string): Promise<void>; // 单工具 kill switch
}
```

工具从 CapabilityScanner 产出为 `draft`，人工审核后转 `enabled`。query 类走语义视图；command 类绑定 Ami Core 业务 API 并携带风险等级。

### 3.6 AgentV6PolicyEngine（权限与安全护栏）

```ts
interface PolicyContext {
  userId: string; roleCodes: string[]; permissions: string[]; storeIds: string[];
  agentId: string; agentPermissionCeiling: string[];    // Agent 能力上限
  fieldScopes: FieldScopeMap; dataScopes: DataScopeMap; approvalScopes: ApprovalScopeMap;
}
class AgentV6PolicyEngine {
  authorizeRead(view: SemanticView, ctx: PolicyContext): PolicyResult;   // 视图+字段
  authorizeTool(tool: ToolDefinition, args: unknown, ctx: PolicyContext): PolicyResult;
  effectivePermissions(ctx): string[];  // 用户权限 ∩ Agent 上限
  maskFields<T>(row: T, ctx: PolicyContext): T;  // 按 fieldScopes 脱敏
}
```

直接继承管理端四类作用域（platform/data/field/approval）与门店隔离，不另造权限体系。安全护栏含：输入护栏（prompt injection/越权识别，外部内容视为不可信）、工具护栏（schema/参数/权限/限流/幂等）、查询护栏（只读 AST + 强制注入 + 脱敏 + 成本上限）、记忆护栏（防投毒/跨角色泄露/过期规则）、运行护栏（中断/降级/kill switch/人工接管）。

### 3.7 AgentV6ApprovalGateway（dry-run + 审批 + 执行）

```ts
interface ApprovalRequest {
  id: string; runId: string; tool: ToolDefinition; args: unknown;
  riskLevel: string; dryRunResult: ImpactPreview;   // 影响对象+回滚方式
  requiredApprovers: ApproverRule; status: 'pending'|'approved'|'rejected'|'executed';
}
class AgentV6ApprovalGateway {
  dryRun(tool, args, ctx): Promise<ImpactPreview>;    // 预览影响，不落库
  submit(req: ApprovalDraft): Promise<ApprovalRequest>;
  decide(id, decision: ApprovalDecision, approver): Promise<void>;
  execute(id): Promise<CommandResult>;                // 经 Command Gateway 执行 + 审计
}
```

L2 需 dry-run + 用户确认；L3 需审批单 + 有权限者审批；L4 默认禁止自动执行，仅生成申请单强制人工审批链。审批人规则对齐现有 `approvalScopes`（如店长 refund `approve_limited`）。

### 3.8 AgentV6CapabilityScanner（Ami Core 能力扫描）

```ts
interface CapabilityMapEntry {
  domain: string; entity: string; relation: string[];
  readCapability?: ReadCap; writeCapability?: WriteCap;
  riskLevel: string; permissionPolicy: PermissionPolicy;
  toolCandidate?: ToolDraft; evidenceSource: string;
  coverageStatus: 'integrated'|'available'|'missing_meta'|'forbidden';
  owner?: string;
}
class AgentV6CapabilityScanner {
  scanRoutes(): Promise<CapabilityMapEntry[]>;   // 管理端菜单/路由/页面
  scanApis(): Promise<CapabilityMapEntry[]>;     // controller/DTO/Swagger 元数据
  scanSchema(): Promise<CapabilityMapEntry[]>;   // Prisma schema 表/字段/关系/敏感级
  scanPermissions(): Promise<CapabilityMapEntry[]>; // 权限码/角色/数据范围
  buildMap(): Promise<AmiCoreCapabilityMap>;     // 汇总为能力地图
}
```

原则：P0 只读扫描，不自动生成写操作；扫描产物须人工确认才进工具注册中心；隐私/支付/会员资产/员工绩效数据默认高敏；识别不到口径或权限的能力标记 `missing_meta` 或 `forbidden`，状态为"不可执行"。

V6 表统一前缀 `AgentV6`，物理隔离于历史 Agent 表。以下为核心表与关键字段（Prisma model，定稿以 migration 为准）。

| --- | --- | --- |
| `AgentV6Run` | id, userId, roleCodes[], storeIds[], channel, inputRaw, intentJson, state, status, cost, latencyMs, createdAt | 一次会话/任务的根 |
| `AgentV6Step` | id, runId, seq, type(recall/route/plan/tool/compose), inputJson, outputJson, agentId, latencyMs | run 的步骤轨迹 |
| `AgentV6Message` | id, runId, role(user/assistant/agent), content, evidenceJson, createdAt | 对话消息与证据 |
| `AgentV6ToolInvocation` | id, runId, stepId, toolId, argsJson(脱敏), resultStatus, riskLevel, approvalId?, idempotencyKey | 工具调用留痕 |
| `AgentV6MemoryItem` | id, kind, scopeJson, content, embedding(vector), source, confidence, sensitive, validUntil | 分层记忆(pgvector) |
| `AgentV6OntologyNode` / `Edge` | id, type, name, aliasesJson / fromId, toId, relation | 经营 Ontology 图 |
| `AgentV6SemanticView` | viewName, domain, requiredPermissions[], storeScopeField, defaultTimeField, fieldsJson, aliasesJson, status, batch | 语义视图注册 |
| `AgentV6MetricDefinition` | canonicalName, aliasesJson, businessDefinition, formula, sourceViewsJson, permissionLevel, drillable | 指标口径 |

## 5. 与 Ami Core 的集成

### 5.1 读集成（Read Model + 语义视图）

- 优先在 PostgreSQL 上建**只读语义视图**（数据库 view 或查询层视图），V6 只查视图不查裸表，规避业务事务耦合与 schema 变动风险。
- 视图强制携带 `store_id` 列供门店隔离注入；敏感列在视图层预脱敏或标记策略。
- 复杂聚合（经营利润、日度财务）复用 Ami Core 现有口径逻辑作为视图定义蓝本（不复用其 Agent 代码，复用的是"口径事实"）。

### 5.2 写集成（Command Gateway）

- V6 不直接写业务表；所有写操作经 Command Gateway 调用 Ami Core 现有业务 API（如 `POST /api/orders/product/:id/refund`）。
- Command Gateway 负责：注入用户 JWT 身份、幂等键、参数校验、调用、结果差异回写、审计。
- 复用 Ami Core 的事务、业务校验与权限，避免绕过后端规则。风险等级映射见需求文档附录 E。

### 5.3 身份与权限透传

V6 以发起用户的 JWT（`Authorization: Bearer` + `X-Store-Id`）调用 Ami Core，天然继承其 `JwtAuthGuard` + `PermissionsGuard` + 门店范围。Agent 有效权限 = 用户权限 ∩ Agent 能力上限，二者取交集后再执行。

## 6. 模型网关（ModelGateway）

- 所有 LLM 调用统一经 Ami Core 现有 AI Gateway（`/api/ai/*`）或 V6 自有网关，前端与终端不持有模型 Key。
- 抽象 `chat/embeddings/tool-call` 三类接口，供应商可配置（不绑定单一模型），支持按场景路由不同模型（意图识别用快模型、复杂规划用强模型）。
- 记录每次调用的 provider/model/tokens/latency/safetyBlocked，接入成本看板。

## 7. 可观测与治理

### 7.1 可观测

- 采用 OpenTelemetry GenAI 语义约定埋点，run/step/tool-call 为 trace/span。
- 接入可自托管观测平台（候选 Langfuse，契合数据合规），或自建轨迹查询接口。
- 每条经营答案携带 EvidenceEnvelope（来源视图、时间范围、口径、行数、置信度）。

### 7.2 治理中心（独立于历史治理台）

治理台复用管理端 React 技术栈，独立路由与权限码（建议 `core:agent-v6-governance:view`/`:manage`）。模块：能力地图、Agent 编排图、运行追踪、权限策略、记忆管理、Ontology 管理、工具注册、评测中心、反馈闭环、价值看板。迭代机制：全对象版本化、上线前跑评测+权限用例、灰度/回滚/单工具禁用、拦截样本回流为用例。

## 8. 安全设计（对齐 OWASP / NIST）

| 风险 | 缓解 |
| --- | --- |
| Prompt injection | 输入护栏识别；外部内容(工具返回/检索)标记为不可信数据，不作为指令执行 |
| 工具误用/过度代理 | 工具须审核启用；写操作风险分级 + 审批；Agent 权限上限约束 |
| 敏感数据泄露 | 字段级脱敏(fieldScopes)；PII 不入记忆明文；查询强制脱敏列 |
| 越权 | PolicyEngine 前置鉴权；门店隔离强制注入；用户权限 ∩ Agent 上限 |
| 数据篡改 | V6 不直接写业务表；写经 Command Gateway + 幂等 + 审计 |
| 记忆投毒 | 记忆写入带来源与置信度；冲突提示确认；可禁用/删除/审计 |
| 自主失控 | L3/L4 禁止自动执行；kill switch(禁用工具/Agent)；异常降级 + 人工接管 |
| 供应链 | 模型经统一网关；工具/依赖版本化可审计 |

## 9. 技术选型（候选与建议，非强锁定）

| 能力 | 建议方案 | 备选 | 取舍理由 |
| --- | --- | --- | --- |
| 运行时 | NestJS 11 独立模块 `agent-v6/` | — | 与后端主线一致，独立目录保证隔离 |
| 编排 | 自建状态机(共享 run/context) + 轻量图 | LangGraph / OpenAI Agents SDK | 单意图为主，先自建可控；复杂协同再评估引入 |
| 记忆 | PostgreSQL 事实表 + pgvector | Mem0 / Zep(时序图谱) | 已是 PG，落地成本低、数据自主可控 |
| 语义/查询 | 语义视图 + schema linking + 受控 SQL + 自我纠错 | GraphRAG | 生产级首选语义层，避免裸库 |
| Ontology | Prisma 图表(node/edge) + 别名词表 | Neo4j | 规模可控，先用关系库承载 |
| 模型 | 经 AI Gateway 多模型路由 | — | 不绑定单一供应商 |
| 可观测 | OTel GenAI + Langfuse(自托管) | Phoenix / LangSmith | 数据合规优先自托管 |
| 治理台 | 复用管理端 React/Vite | — | 复用组件与权限体系 |

## 10. 分阶段落地

- **P0 可信只读闭环**：`agent-v6/` 骨架 + Run/Step/Trace + CapabilityScanner(只读扫描) + P0 语义视图(覆盖 11 域高频) + 单表/跨表查询 + SqlGuard + 记忆与澄清 + 评测集。验收：高频只读+风险覆盖 80%，问数准确率 ≥95%，越权 0 通过。
- **P1 建议 + 低风险操作**：业务建议技能 + ToolRegistry(command 类) + ApprovalGateway(L1/L2 dry-run) + 多 Agent 复合任务 + 价值看板。
- **P2 协同闭环 + 受控高风险**：L3/L4 审批链 + EventScheduler 主动经营 + 灰度/回滚治理成熟 + 记忆时序增强。

## 11. 关键技术风险

1. **语义视图口径漂移**：视图口径与 Ami Core 业务变更不同步 → 建立视图-口径回归测试，schema 变更触发扫描告警。
2. **Text-to-SQL 准确率**：真实脏数据下准确率不达标 → 语义层 + few-shot 已验证 SQL 库 + 执行制导自我纠错，用自家问题集回归(不外推 benchmark)。
3. **多 Agent 成本/延迟**：复合任务 token 数倍膨胀 → 单意图默认单步，按复杂度动态决定子 Agent 数，设成本护栏。
4. **写操作事故**：误执行 L3/L4 → 默认禁止自动执行、强制 dry-run + 审批 + 幂等 + 审计 + 可回滚。
5. **记忆泄露/污染**：跨角色泄露或投毒 → 作用域隔离 + 权限过滤 + 置信度 + 可审计治理。

## 结语

本方案把需求文档的六项能力落到可实施的技术结构：六层架构 + 只读/写/事件/协同四条运行时路径 + 十余个独立模块接口 + V6 独立数据表 + Ami Core 读写集成契约 + 安全与治理设计 + 分阶段落地。核心取舍是"只读优先、写强确认、权限不可协商、语义层而非裸库、能力来自扫描且须人工确认"。后续可继续输出《Capability Scanner 详细设计》《Ontology 与指标口径设计》《权限与安全护栏方案》《P0 开发计划与 tasks.md》《P0 验收用例与评测集》。
