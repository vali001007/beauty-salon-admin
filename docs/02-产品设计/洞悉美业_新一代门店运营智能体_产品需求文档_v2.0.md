## 1. 背景与定位

美业门店的日常经营正在从"人找功能、手工看报表、凭经验决策"转向"人表达目标、系统理解业务、智能体调度能力并完成闭环"。

Ami_Agent 的定位不是普通聊天机器人，也不是报表问答助手，而是一个真正懂美业门店经营的运营智能体：

- **懂美业业务对象**：客户、预约、服务、疗程、卡项、项目、商品、库存、收银、营销、员工、排班、提成、财务、会员、门店经营。
- **懂经营目标**：拉新、复购、升单、留存、消耗、转化、毛利、现金流、员工效率、库存周转、客户满意度。
- **懂操作闭环**：不仅回答"是什么"，还要能告诉用户"为什么、怎么办、下一步怎么做"，必要时直接生成任务、表单、活动、提醒、自动化规则或执行草稿。
- **懂风险提示**：在客户流失、库存断货、活动低转化、员工业绩异常、财务波动、排班空档、客诉风险等场景主动提醒。

**一句话目标：让门店用户用一句话获得洞察、操作和结果。**

## 2. 产品目标与阶段规划

### 2.1 阶段目标

| 阶段 | 目标 | 交付重点 |
|---|---|---|
| **MVP** | AI 优先意图层 + 可信问数 + 多轮上下文 | 意图层重建、AuraResponseBlock 协议、对话上下文、角色首屏、数据卡片、追问与 follow-up chips |
| **V1** | 独立 Agent 工作台 + 六大角色 Agent | 新前端入口（极简/科技/艺术）、六大角色 Agent、工具注册中心、权限网关、结构化任务画布 |
| **V2** | 业务操作闭环 + 记忆归档 | 业务草稿与执行、主动风险提醒、用户偏好记忆、聊天归档、文档生成、效果归因 |
| **V3** | 自动化执行引擎 + 主动运营 | 全新自动化引擎、10 个内置触发器、事件驱动、审批中断、失败恢复、主动推送经营待办 |

### 2.2 非目标

| 非目标 | 说明 |
|---|---|
| 不做泛聊天助手 | 问答必须围绕美业经营和 Ami_Core 业务能力 |
| 不让模型绕过系统权限 | 模型不能直接读库、写库或调用未授权接口 |
| 不替代所有页面 | 高频任务由智能体承接，复杂配置、审计、批量治理仍保留专门后台 |
| 不复用旧 AI Gateway 核心链路 | 旧 AI Gateway 无工具层、无权限、无证据机制，不作为 Agent Runtime 底座 |
| 不复用旧自动化任务系统 | 旧系统不可执行、不可追踪、不可归因，新建自动化执行引擎 |

## 5. 核心产品原则

### 5.1 真正懂美业语义

Ami_Agent 必须能把用户输入识别成美业业务语义，而不是关键词匹配。

| 用户输入 | 系统理解 | 输出要求 |
|---|---|---|
| "最近哪些顾客可能要流失？" | 活跃度、消费间隔、预约缺失、卡项剩余、历史频次 | 流失风险客户表、原因、跟进建议、一键创建跟进任务 |
| "这个月项目卖得怎么样？" | 项目销售、消耗、复购、客单价、员工维度 | 趋势图、Top 项目、异常说明、改进建议 |
| "帮我看看库存有没有风险" | 库存数量、安全线、消耗速度、临期、缺货 | 风险卡片、补货表、处理优先级 |
| "给 60 天没来的顾客做个召回" | 客群筛选、权益匹配、营销内容、触达规则 | 活动草稿、客群明细、预计成本、需确认后执行 |

### 5.2 基于事实回答

所有涉及门店经营事实的回答必须来自 Ami_Core 真实业务数据，并明确区分：

- **系统数据**：来自 Ami_Core 的客户、订单、预约、库存、财务数据
- **推理建议**：Agent 基于数据给出的分析和建议
- **外部资料**：来自联网搜索或用户上传的参考信息
- **需要确认**：需要用户授权、审批或补充信息后才能执行的动作

每条包含经营数据的回答必须标注：数据来源模块、时间范围、口径说明、数据限制。

### 5.3 先追问，再执行

当用户意图不明确时，一次只问一个关键追问，不发散。

**需要追问的典型情况**：
- 时间范围不清（"最近""这段时间"）
- 门店范围不清（多门店用户未指定门店）
- 操作目标不清（"做个活动"未说明目的）
- 风险动作缺少确认（批量发券、批量触达、修改价格）
- 数据口径存在多种解释（"收入"可能是应收、实收或营业额）

**澄清模式**：

低风险/可逆操作 → 声明假设后直接执行：
> "（按本月 1-24 日统计）本月目前营收 ¥128,400，比上月同期高 12%。如想看完整自然月对比，告诉我一声。"

高风险/不可逆操作 → 一次只问一个关键歧义：
> "你想给'长期未回访客户'发优惠券——请先确认：是所有 60 天未到店的客户，还是有其他筛选条件？"

多种合理解读 → 直接给选项：
> "'业绩'是指：A. 服务销售额  B. 产品销售额  C. 两者合计（默认展示合计）"

**硬性规范**：每次最多问一个问题；上下文已有答案不再追问；不在结尾机械添加"还有什么可以帮您"。

### 5.4 多模态输出，按需渲染

不同内容类型对应不同渲染形式，混用会降低信息传达效率：

| 内容类型 | 最优格式 | 错误示范 |
|---|---|---|
| 单一数字结论（今日营收 ¥4,280） | 大号 KPI 卡 + 同比箭头 | 纯文字句子 |
| 多指标横向对比（各技师业绩） | 排名表格 + 条形图 | 列举式文字 |
| 时间趋势（近 30 天客流） | 折线图 | 数字列表 |
| 操作确认（发营销短信） | 确认卡 + 预览 + 操作按钮 | 纯文字描述 |
| 分析结论 + 建议 | 精炼文字（≤3 句） | 满屏 Bullet Points |
| 客户详情 | 结构化信息卡 | 长段文字 |
| 流程说明 | 编号步骤（≤5 步） | 段落散文 |
| 不确定性高的分析 | 文字 + 数据来源注释 | 无来源的确定性断言 |

**标准回复结构**：结论 → 证据 → 建议 → 动作 → 延展

### 5.5 回复后主动延展

每次回复完成后，如存在高价值关联问题，提供 1-3 个后续问题供用户选择。

规范：
- 最多 3 个，三个方向有差异：**深入**（更细节）+ **扩展**（相关维度）+ **行动**（基于结论能做什么）
- 每条 ≤ 8 个字
- 流式输出结束后才显示，不在输出过程中插入
- 动态生成，不静态预设

示例——用户问"本月哪个技师营收最高"后推荐：
- "李美霞的客单价趋势是？"（深入）
- "各技师对比上月变化？"（扩展）
- "给表现最好的技师设置绩效奖励"（行动）

## 6. Agent 底座架构

### 6.1 架构组成

| 模块 | 职责 | 说明 |
|---|---|---|
| Agent Runtime | 意图理解、多轮上下文、步骤规划、工具调用、结构化输出 | 核心编排引擎，不做简单的模型转发 |
| 美业语义层 | 统一业务对象、指标口径、同义词、风险规则、操作动作 | 解决模型不懂美业、关键词误判、口径混乱 |
| 工具注册中心 | 将 Ami_Core 能力封装为可授权、可审计、可执行工具 | 每个工具有 schema、权限、风险等级、幂等策略 |
| 权限与审批网关 | 每次工具调用前校验角色、门店、字段、动作和风险等级 | 解决越权查询和批量误操作 |
| 回复渲染引擎 | 将结果输出为 AuraResponseBlock[] | 解决回复只有文本、无法承接业务操作 |
| 记忆与归档服务 | 管理会话上下文、用户偏好、长期记忆、按天聊天归档 | 解决对话断裂、偏好无法沉淀 |
| 自动化执行引擎 | 运行触发器、条件、动作、审批、重试、日志和效果归因 | 全新建设，不复用旧自动化系统 |
| 观测与评估系统 | 记录意图识别、工具调用、回答质量、用户反馈和业务动作率 | 从 MVP 开始建立，不能上线后补 |

### 6.2 技术资产继承策略

| 资产 | 策略 | 原因 |
|---|---|---|
| 旧 AI Gateway 核心链路 | ❌ 不复用 | 无工具层、无权限、无证据机制，无法支撑 Agent Runtime |
| 旧自动化任务系统 | ❌ 不复用 | 不可执行、不可追踪、不可归因，新建执行引擎 |
| 前端意图层（ruleIntentParser） | ❌ 重建 | 关键词匹配是根本性缺陷，AI 优先替代规则优先 |
| 后端 28 个工具实现 | ✅ 保留并扩展 | 已验证的业务查询逻辑，扩展 schema 和证据包即可 |
| AgentRun / ToolCall / AgentApproval 骨架 | ✅ 保留并扩展 | 状态机和审批流已验证，增加 personaCode 和 renderedBlocks |
| AgentEvidence 机制 | ✅ 保留 | 数据来源可追溯是可信度的基础，已有良好实现 |
| Prisma 全业务域 schema | ✅ 保留 | 35+ 主表覆盖全业务域，无需重建 |
| FlowCard 业务 UI 组件 | ✅ 保留业务逻辑 | 核销/收银/建档流程已验证，可换视觉语言但不重写业务逻辑 |
| Ami Aura 产品形态 | ⚠️ 升级 | 前端意图层重建后，Ami Aura 作为过渡形态继续可用；新 Agent 工作台独立入口 |

## 7. 结构化输出协议（AuraResponseBlock）

后端通过 structured output / function calling 返回 AuraResponseBlock[]，前端按 kind 渲染，AI 逻辑和 UI 完全解耦。

```typescript
type AuraResponseBlock =
  | { kind: 'text'; content: string }
  | { kind: 'kpi_card'; label: string; value: string; delta?: string; deltaType?: 'up' | 'down' }
  | { kind: 'table'; columns: string[]; rows: string[][]; sortable?: boolean }
  | { kind: 'chart'; chartType: 'line' | 'bar' | 'pie' | 'funnel'; title: string; data: unknown }
  | { kind: 'customer_card'; customerId: string; name: string; vipLevel: string; lastVisit: string; suggestion?: string }
  | { kind: 'confirm_action'; title: string; preview: string; actionId: string; riskLevel: string; impactSummary?: string }
  | { kind: 'alert'; level: 'warning' | 'critical'; message: string; actionId?: string }
  | { kind: 'follow_up_chips'; suggestions: string[] }
  | { kind: 'document_preview'; title: string; content: string; downloadable?: boolean }
  | { kind: 'evidence_panel'; sources: string[]; dateRange: string; metricDefinition: string; limitations?: string[] }
```

渐进式披露：先给结论（text + kpi_card），"查看明细"展开 table 和 chart，不让完整内容占满主对话流。

## 8. 操作确认分级

| 级别 | 操作示例 | 确认策略 |
|---|---|---|
| 只读 | 查营收、查客户、看库存 | 直接返回，无需确认 |
| draft | 生成活动草稿、召回话术 | 标注草稿，提供修改入口，不自动生效 |
| low（单条写） | 确认单个预约、创建单条跟进 | confirm_action block 内联确认 |
| high（批量不可逆） | 批量发短信/优惠券 | 展示完整预览+影响范围+二次确认 |
| critical（财务/权限） | 提交财务记录、修改权限 | 强审批，高权限角色确认，完整审计留痕 |

## 9. 多轮对话上下文管理

三层记忆架构：

- **短期**（当前会话滑动窗口 6-10 轮）：解决"她/它/刚才"的代词引用，随每次 AI 调用传入 prompt
- **中期**（session 级）：AgentRun.contextJson 存储任务状态，支持多步任务中途切换后恢复
- **长期**（门店知识库，跨会话）：向量存储（pgvector），本店提成规则、VIP 等级、服务项目说明，每次对话 RAG 检索注入，用户无需重复描述

界面设计：角落显示"当前关注：张雯 / 本月绩效"；提供"清除上下文"快捷操作；跨会话记忆使用要显式声明。

## 10. 六大角色 Agent 详细能力

### 10.1 店长经营 Agent

目标：门店每日经营总入口，帮助店长快速了解今天要关注什么。

| 工具 | 类型 | 说明 |
|---|---|---|
| manager.daily.briefing | 查询/分析 | 生成今日经营简报 |
| customer.priority.rank | 查询/分析 | 推荐优先跟进客户 |
| schedule.diagnose | 查询/分析 | 诊断预约、排班和空档 |
| beautician.performance.diagnose | 查询/分析 | 美容师业绩、复购、服务异常 |
| revenue.diagnose | 查询/分析 | 收入、订单、客单和趋势诊断 |
| inventory.risk.rank | 查询/分析 | 库存缺货、临期、补货优先级 |
| manager.followup.plan.draft | 草稿 | 生成店长待办和派发草稿 |

输出：今日经营重点卡、风险提醒卡、客户跟进表、美容师表现对比表、预约/排班时间轴

权限：可查本店全量数据，不默认查跨店财务明细

验收：问"今天我应该重点关注什么"能返回客户、预约、员工、库存、营销至少 4 类证据；所有建议可追溯数据来源

### 10.2 营销增长 Agent

目标：发现增长机会、识别客群、匹配权益、生成活动草稿、追踪活动效果。

| 工具 | 类型 | 说明 |
|---|---|---|

## 11. 自动化执行引擎

### 11.1 定位

自动化不是"提醒文案"，也不是旧系统的规则拼装，而是由全新执行引擎调度、运行、监控和复盘的业务流程。全新建设，不复用旧自动化任务系统。

自动化创建流程：

1. 用户提出目标
2. Agent 识别触发条件、执行动作、对象范围、频率、权限
3. 必要时追问补齐信息
4. 生成自动化任务草稿
5. 展示执行摘要、风险、预计影响范围
6. 用户确认或提交审批
7. 全新执行引擎创建任务
8. 运行、记录日志、失败重试、异常通知

### 11.2 十个内置触发器

| 触发器 | 默认条件 | Agent 动作 | 风险等级 |
|---|---|---|---|
| 沉睡客户 | 45/60/90 天未到店 | 生成召回名单和跟进任务草稿 | 中 |
| 高价值客户到店 | 高价值客户明日有预约 | 推送店长和美容师重点服务提醒 | 低 |
| 疗程消耗 | 卡项/疗程剩余低于阈值 | 生成续卡/复购建议 | 中 |
| 库存缺货 | 热销商品预计 7 天内缺货 | 生成补货建议单和采购提醒 | 中 |
| 临期库存 | 商品临期或周转异常 | 生成促销、消耗或调拨建议 | 中 |
| 活动低转化 | 活动转化低于预期 | 提醒调整权益、话术或目标客群 | 中 |
| 员工异常 | 美容师复购率/消耗率/业绩连续下降 | 生成员工辅导建议和客户回访名单 | 中 |
| 预约异常 | 爽约率上升、排班空档、预约冲突 | 生成提醒、调班建议或客户确认任务 | 中 |
| 财务异常 | 退款、折扣、收入、成本出现异常波动 | 生成异常解释和复核任务 | 高 |
| 投诉差评 | 退款/投诉/差评出现 | 创建高优先级风险处理任务 | 高 |

### 11.3 执行引擎能力要求

| 能力 | 要求 |
|---|---|
| 触发器 | 支持定时、事件、阈值、数据变化和人工触发 |
| 条件判断 | 支持客户、库存、订单、预约、员工、财务、营销等条件组合 |
| 动作编排 | 支持查询、筛选、生成、推送、创建草稿、提交审批、执行低风险动作 |
| 审批中断 | 高风险动作运行到审批点时暂停，等待人工确认后继续 |
| 幂等控制 | 防止同一客户重复触达、同一活动重复创建、同一提醒重复发送 |
| 失败恢复 | 支持失败原因展示、重试、跳过、人工接管 |
| 运行日志 | 每次运行记录输入、命中对象、执行动作、输出和异常 |
| 效果归因 | 统计任务带来的跟进、预约、核销、订单、收入或风险下降 |
| 暂停熔断 | 当数据异常、触达成本异常或失败率异常时自动暂停并提醒 |

## 13. 权限与审计

### 13.1 角色权限矩阵

| Agent | 默认可用角色 | 敏感边界 |
|---|---|---|
| 店长经营 Agent | 店长、老板、管理员 | 可看本店经营，不默认跨店明细 |
| 营销增长 Agent | 店长、营销运营、老板 | 批量触达、发券必须审批 |
| 前台接待 Agent | 前台、店长 | 财务、成本、提成不可见 |
| 美容师服务 Agent | 美容师、店长 | 美容师只能看本人数据 |
| 库存采购 Agent | 店长、库存、采购、老板 | 正式采购单必须确认 |
| 财务风控 Agent | 财务、老板、管理员 | 成本、毛利、提成字段严格授权 |

### 13.2 审计要求

每次 Agent 运行必须记录：操作人、角色、门店、设备、时间；用户原始输入和识别出的意图；调用的工具、参数、返回摘要；用户确认记录；实际执行结果；失败原因、重试记录和人工接管记录。

### 14.2 需要新增或扩展

## 15. 阶段实施计划

| 阶段 | 周期 | 目标 | 核心交付 |
|---|---|---|---|
| **阶段 0** 基线冻结 | 2-3 天 | 确认资产继承边界，输出工具目录和权限矩阵 | 六大 Agent 工具目录、权限矩阵、数据模型改动清单 |
| **阶段 1** AI 意图层重建 | 1-2 周 | 快速提升现有 Ami Aura 体验（最小改动见效最快） | intentRouter.ts 重建（AI 优先）、conversationContext.ts 新增、AuraResponseBlock 协议定义、BlockRenderer.tsx、角色登录自动首屏、follow-up chips |
| **阶段 2** Agent 工作台骨架 | 1-2 周 | 独立 Ami_Agent 前端入口 + Runtime 骨架 + 店长 Agent + 前台 Agent | /ami-agent 路由入口、AgentPersona 模型、AgentRenderedBlock、店长/前台 Agent 工具实现 |
| **阶段 3** 营销增长 Agent | 1-2 周 | 打通客群、权益、活动草稿、话术、复盘闭环 | 营销增长 Agent 全部工具、活动草稿卡、高风险确认流 |
| **阶段 4** 美容师 + 库存 Agent | 2 周 | 覆盖员工服务与库存采购两个高价值闭环 | 美容师服务 Agent、库存采购 Agent 全部工具 |
| **阶段 5** 财务风控 Agent | 1-2 周 | 打通收入、成本、毛利、利润、异常流水 | 财务风控 Agent 全部工具、字段权限控制 |
| **阶段 6** 记忆 + 观测评估 | 1 周 | 让 Ami_Agent 可持续使用、可复盘、可优化 | AgentMemory、聊天归档、AgentFeedback、eval 覆盖 |
| **阶段 7** 自动化执行引擎 | 2-3 周 | 全新自动化引擎，不依赖旧自动化系统 | 自动化定义/运行/归因模型、10 个内置触发器、审批中断、失败恢复 |

**交付顺序原则**：
- 阶段 1 在现有 Kiosk 上做，2 周内可见效，不影响其他业务
- 阶段 2 独立新入口，阶段 1 的成果可无缝迁移
- 阶段 3-5 按"高频 → 增收价值 → 敏感度"顺序排列
- 阶段 7 风险最高，必须在工具、审批、审计成熟后再上线

## 17. 风险与控制

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 模型误解业务语义 | 错误建议或误操作 | 美业语义层、工具 schema 约束、置信度追问机制 |
| 数据口径不统一 | 回答不可信 | 统一经营指标口径，每条回答标注来源和口径 |
| 权限过宽 | 敏感数据泄露 | 权限前置校验、字段脱敏、审计留痕 |
| 自动化误执行 | 批量触达、错误提醒、重复任务 | 审批中断、幂等控制、运行日志、暂停熔断 |
| 沿用失败旧底座 | 新产品继承旧问题 | 明确不以旧 AI Gateway 和旧自动化系统承接核心链路 |
| 意图层重建质量不稳定 | AI 识别率不达标 | 保留规则快捷路作为 fallback；建立意图识别 eval 集监控质量 |
| UI 退化成聊天框 | 无法承接复杂业务结果 | 任务画布和 AuraResponseBlock 渲染为每阶段验收项 |
| 六大 Agent 同时铺开过大 | 周期不可控 | 阶段门禁通过再继续；阶段 1 先独立验证意图层 |

## 18. MVP 验收清单

| 能力 | 验收标准 |
|---|---|
| AI 优先意图识别 | 自然语言识别率 > 80%；置信度不足时触发追问，不猜测执行 |
| 对话上下文 | "她/它/上面那个"等代词能正确解析；多轮条件修改能继承上下文 |
| 多模态输出 | 至少支持 text、kpi_card、table、chart、confirm_action、follow_up_chips |
| 角色首屏 | 登录后 < 2 秒无需输入即可看到角色仪表盘 |
| 追问规范 | 一次只追问一个问题；上下文已有答案不再追问 |
| Follow-up chips | 流式输出结束后动态生成 3 个，方向差异化（深入/扩展/行动） |
| 数据可信度 | 每条经营数据回答标注来源、时间范围、口径说明 |
| 工具权限 | 不同角色看到不同字段和可执行动作；越权工具调用被拦截 |
| 草稿确认 | 批量不可逆操作必须展示预览 + 二次确认，禁止自动执行 |
| 旧底座隔离 | 核心链路不经过旧 AI Gateway；意图层使用新 AI 路由 |
