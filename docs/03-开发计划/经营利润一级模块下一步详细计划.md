## 2. 下一步总目标

下一步目标不是继续堆功能，而是把经营利润模块从“代码完成”推进到“真实门店可验收、可评审、可发布”。

验收完成后应达到：

1. 测试数据库已执行 6 条经营利润发布必需 migration。
2. 至少一个测试门店具备当月经营成本数据。
3. 6 个经营利润页面均能读取真实 API。
4. 利润看板的现金收入、经营收入、毛利、经营利润、净利率能用真实业务数据解释。
5. 缺商品成本、缺商品提成、缺项目档案、缺 BOM、缺项目提成、缺耗材流水能明确显示。
6. 店长账号只读可见，超级管理员可维护成本。
7. 代码提交、PR、验收记录完整。

已新增成本样例 seed 脚本，默认 dry-run，不写库：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:cost-seed -- --storeId=6 --periodMonth=2026-06
npm.cmd --prefix packages/server-v2 run operation-profit:cost-seed -- --storeId=6 --periodMonth=2026-06 --apply --yes
```

脚本规则：

- 必须显式传 `--storeId`，避免误写未知门店。
- 默认只输出计划成本，不传 `--apply --yes` 不写库。
- 已存在同门店同月份同分类成本时会跳过，避免重复写入。
- 如果当前数据库还没有 `OperatingCost` 表，会输出 `migration_required`，不继续 seed。
- 写入前必须确认目标是本地、测试或试点门店，不直接写生产库。
- 样例总成本为 55000，覆盖房租、工资、营销、水电、折旧、其他 6 类。

完成标准：

- 成本配置页能查到当月成本。
- 利润看板的成本结构能展示录入项。
- 成本缺失提示减少或消失。

风险控制：

- 不直接在生产库写演示数据。
- 如需 seed，先写可重复脚本，并明确只针对测试门店。
- 真实数据库写入前需要用户明确授权。

### 阶段 C：真实 API 联调

优先级：P0
建议耗时：1 天
目标：确认前端 6 个页面通过真实 API 能稳定拿到数据。

进入页面联调前，先跑经营利润 readiness 只读检查：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30
```

readiness 输出含义：

| 状态 | 说明 | 是否进入页面验收 |
| --- | --- | --- |
| `ready` | migration、成本、订单、成本/BOM、服务人、提成均满足基础验收 | 可以进入完整页面验收 |
| `conditional` | 无 fail，但仍有 BOM/提成等 warn | 可以演示，但必须说明数据质量风险 |
| `blocked` | 存在 migration、成本、订单、服务人等 fail | 不建议给老板验收，只能做技术联调 |

readiness 检查项：

| 检查项 | 目的 |
| --- | --- |
| `store` | 试点门店是否存在 |
| `required_prisma_migrations` | 经营利润所需 Prisma migration 是否已应用 |
| `schema_migration_columns` | 经营利润、提成、订单归属所需表/列是否已迁移 |
| `schema_migration_indexes` | 经营利润月度订单查询关键索引是否已迁移 |
| `operating_cost_migration` | `OperatingCost` 表是否已执行 migration |
| `operating_cost_data` | 当月经营成本是否已录入 |
| `margin_source_orders` | 是否有商品/项目订单明细可用于毛利验收 |
| `product_cost_data` | 商品成本是否可计算 |
| `project_master_data` | 项目订单明细是否能关联到项目档案 |
| `project_bom_data` | 项目 BOM 是否足够解释耗材成本 |
| `assignee_data` | 商品/项目订单明细是否有服务人归属 |
| `commission_data` | 商品/项目订单明细是否已有提成记录 |
| `commission_rule_coverage` | 缺提成时，是否存在可覆盖商品/项目明细的有效提成规则 |

### 阶段 D：经营口径业务验收

优先级：P0
建议耗时：1 天
目标：让产品、运营、店长能确认“这些数是否符合经营理解”。

验收方式：

| 指标 | 核对方式 | 通过标准 |
| --- | --- | --- |
| 现金收入 | 对比订单实收、办卡、充值、退款 | 能解释现金流来源 |
| 经营收入 | 对比单次服务、消课、产品销售 | 不把办卡充值直接算利润 |
| 毛利 | 经营收入减耗材、商品成本、商品/项目订单明细提成 | 公式可解释；办卡、充值等非经营收入提成不混入总览毛利 |
| 经营利润 | 毛利减房租、工资、营销、水电、折旧、其他 | 录入成本后变化正确 |
| 净利率 | 经营利润 / 经营收入 | 百分比口径一致 |
| 项目毛利 | 抽查 3 个高频项目 | BOM/耗材/提成缺口可解释 |
| 会员卡履约 | 抽查 5 张会员卡 | 剩余次数和到期风险正确 |
| 员工人效 | 抽查 3 位美容师 | 服务次数和收入归属合理 |

业务验收输出：

```text
验收门店：
验收月份：
现金收入：
经营收入：
经营利润：
主要成本缺口：
主要 BOM 缺口：
是否可给店长试用：
遗留问题：
```

完成标准：

- 产品接受当前 MVP 口径。
- 业务能解释核心指标。
- 未确认的数据缺口已记录，不阻塞试用。

优先级：P1

### 阶段 F：性能与稳定性检查

优先级：P1
建议耗时：0.5-1 天
目标：避免经营利润页面在真实数据量下明显慢或超时。

检查点：

| 场景 | 风险 | 检查方式 |
| --- | --- | --- |
| 利润看板月度查询 | 拉取订单、核销、成本、提成较多 | 看接口耗时和 SQL 查询数 |
| 项目毛利分页 | 项目多、BOM 多、库存流水多 | 检查分页和耗材归因 |
| 会员卡履约 | 客户卡量大 | 检查是否一次性过量拉取 |
| 员工人效 | 美容师聚合逐人查提成 | 检查是否存在 N+1 查询风险 |

优化预案：

1. 短期：限制默认日期范围为当月，分页大小控制在 100 内。
2. 中期：增加日报/月报聚合表。
3. 长期：经营分析接口接入缓存或预计算任务。

完成标准：

- 常规测试门店当月查询在可接受时间内返回。
- 没有明显重复请求或首屏空白等待。
- 慢查询和后续优化项有记录。

### 阶段 H：商品毛利口径冻结

优先级：P0
建议耗时：0.5 天
目标：先把商品毛利的财务公式定死，避免页面做完后口径反复。

商品毛利推荐公式：

```text
商品销售收入 = 已完成/已付款商品订单明细小计 - 已完成退款中对应商品退款金额
商品成本 = 商品销量 × 商品单位成本
商品提成成本 = 商品订单明细关联的有效提成记录金额
商品毛利 = 商品销售收入 - 商品成本 - 商品提成成本
商品毛利率 = 商品毛利 / 商品销售收入
```

商品成本取值优先级：

| 优先级 | 成本来源 | 使用条件 | 产品影响 |
| --- | --- | --- | --- |
| 1 | 订单明细成本快照 | 如果 `OrderItem.payload.costPrice` 或后续新增字段存在 | 最准确，可还原成交当时成本 |
| 2 | 库存出库流水 | 如果商品订单完成时产生 `StockMovement`，且可按 `sourceType/sourceId` 追到订单或订单明细 | 能反映批次/出库真实成本 |
| 3 | 商品档案成本 | 使用 `Product.costPrice` | 当前最可落地，但历史成本变更会影响历史毛利 |

当前实现：

- 新商品订单已在 `OrderItem.payload` 写入 `costPrice`、`costAmount`、`costSource`、`costCapturedAt`，管理端订单和 Ami Aura Lite 终端收银均覆盖。
- 历史订单如果没有成本快照，商品毛利会继续按库存出库确认或 `Product.costPrice` 兜底，并在成本来源中提示。
- 不把采购价、零售价、活动价混用；商品收入只来自订单实收/明细小计，商品成本只来自成本字段或库存出库。

验收标准：

| 验收项 | 通过标准 |
| --- | --- |
| 商品收入 | 只统计 `OrderItem.itemType=product/goods` 且订单状态有效的数据 |
| 商品成本 | 每个商品能看到成本来源：成本快照、库存流水或商品档案 |
| 商品提成 | 只统计 `CommissionRecord.status != cancelled` 且能关联到商品订单明细的数据 |
| 退款影响 | 已退款订单或退款明细不重复计入收入 |
| 数据质量 | 缺商品成本、缺提成、缺订单明细时明确提示 |

### 阶段 K：项目毛利口径加固

优先级：P0
建议耗时：1 天
目标：把项目毛利从“能算”升级为“每一项收入、耗材、提成都能追溯来源”。

项目毛利推荐公式：

```text
项目收入 = 项目订单收入 + 会员卡消课收入
耗材成本 = 优先取实际耗材出库成本；没有实耗时取项目 BOM 标准成本
项目提成成本 = 项目订单明细关联的有效提成记录金额
项目贡献毛利 = 项目收入 - 耗材成本 - 项目提成成本
项目毛利率 = 项目贡献毛利 / 项目收入
```

项目收入来源拆分：

| 收入来源 | 数据对象 | 统计方式 | 注意事项 |
| --- | --- | --- | --- |
| 单次项目订单 | `ProductOrder` + `OrderItem.itemType=project/service/service_project` | 按 `OrderItem.itemId=Project.id` 聚合 `subtotal` | 只取有效订单 |
| 会员卡消课 | `CardUsageRecord` | 通过 `projectName` 匹配项目，按卡单次价值估算收入 | 当前属于估算，缺卡单次价值要提示 |
| 项目订单页面 | `/orders/projects` | 管理端创建项目订单写入统一 `ProductOrder` | 要确保 `itemId` 写入项目 ID |
| 终端项目收银 | Ami Aura Lite 收银 | 终端订单也写入 `ProductOrder` + `OrderItem` | 要确保 `orderItemId` 用于提成追踪 |

耗材成本来源拆分：

| 成本来源 | 数据对象 | 使用优先级 | 说明 |
| --- | --- | --- | --- |
| 实际耗材出库 | `StockMovement` | 1 | `movementType=service_consume/service_consumption`，按项目订单、消课或服务任务来源追踪；同一订单存在多个项目时，必须能按 BOM 商品和流水备注归因到具体项目 |
| 标准 BOM 成本 | `ProjectBomItem` + `Product.costPrice` | 2 | 实耗缺失时用标准用量估算 |
| 缺项目档案 | `OrderItem.itemType=project` 但 `Project` 不存在 | 风险提示 | 历史项目订单收入仍展示，标记 `missing_project_master` |
| 缺 BOM | `ProjectBomItem` 为空 | 风险提示 | 服务次数 > 0 但无 BOM 时标记 `missing_bom` |
| 缺实耗 | 有 BOM 但无 `StockMovement` | 风险提示 | 标记 `missing_actual_consumption` |

提成成本来源拆分：

| 来源 | 数据对象 | 说明 |
| --- | --- | --- |
| 提成规则 | `CommissionRule(type=project)` | 支持全项目、分类、指定项目、等级、指定员工 |
| 提成记录 | `CommissionRecord` | 以 `orderItemId` 回挂项目订单明细 |
| 订单触发 | `OrdersService`、`TerminalService` | 创建/完成项目订单后调用 `CommissionService.calculateOrderCommissions` |
| 缺提成 | 有项目收入但无有效提成记录 | 标记 `missing_commission`，提示规则或触发链路缺失 |

项目毛利后端加固任务：

| 任务 | 当前状态 | 下一步动作 | 验收 |
| --- | --- | --- | --- |
| 项目订单收入 | 已通过 `OrderItem` 聚合 | 补用例覆盖管理端项目订单 + 终端项目订单 | 项目订单收入准确进入项目毛利 |
| 消课收入 | 已按卡单次价值估算 | 页面明确展示“消课收入为估算” | 缺单次价值时有提示 |
| BOM 标准成本 | 已使用 `ProjectBomItem` | 增加 BOM 明细下钻字段或后续详情接口 | 能解释某项目标准耗材成本 |
| 实际耗材成本 | 已查 `StockMovement` | 校准 `sourceType/sourceId` 与订单/消课/服务任务的匹配 | 实耗优先于标准 BOM |
| 提成成本 | 已按 `CommissionRecord.orderItemId` 汇总 | 补回填/重算计划，避免历史订单缺提成 | 有收入无提成时明确提示 |
| 状态规则 | 已有 `cost_missing/high_profit/normal` | 补低毛利、亏损状态 | 页面排序更有经营指导价值 |

### 阶段 L：订单、库存、提成链路补强

优先级：P0
建议耗时：1-2 天
目标：让商品毛利和项目毛利不是“事后估算”，而是和订单、库存、提成形成闭环。

需要核对并补强的链路：

| 链路 | 目标 | 验收方式 |
| --- | --- | --- |
| 商品订单创建 | 已补管理端和终端 `OrderItem.itemType=product/goods`、`itemId=Product.id`、数量、单价、小计、成本快照 | 新建商品订单后商品毛利页面出现该商品收入，且历史成本不随商品档案成本变动 |
| 商品库存出库 | 已有商品订单完成后销售出库 `StockMovement` | 商品成本可从订单成本快照或出库确认追溯 |
| 商品提成 | 已有商品订单完成后按 `CommissionRule(type=product)` 生成 `CommissionRecord` | 商品毛利扣除提成成本 |
| 项目订单创建 | `OrderItem.itemType=project`、`itemId=Project.id`、`beauticianId` 完整 | 项目毛利页面出现该项目收入 |
| 项目 BOM 消耗 | 项目订单完成后按 `ProjectBomItem` 产生服务耗材出库 | 项目毛利用实际耗材成本 |
| 项目提成 | 项目订单完成后按 `CommissionRule(type=project)` 生成提成记录 | 项目毛利扣除提成成本 |
| 退款/取消 | 取消、退款订单不应继续贡献收入和毛利 | 退款后毛利同步减少或标记需人工核对 |

订单成本快照实现：

短期不改 schema，已在 `OrderItem.payload` 中写入：

```json

  "costPrice": 38,
  "costSource": "product_master",
  "costCapturedAt": "2026-06-19T00:00:00.000Z"
}
```

中期如要做严肃财务，应新增字段或明细表：

```text

OrderItem.costAmount
OrderItem.costSource
OrderItem.costCapturedAt
```

这样能避免商品档案成本变化后，历史订单毛利被重新计算。

- 只处理 `product/goods/project/service/service_project` 明细。
- 已有有效 `CommissionRecord` 的 `orderItemId` 不重复生成。
- 缺 `beauticianId` 时输出 `skippedMissingBeautician`；美容师存在但无 `userId` 时输出 `skippedMissingStaffUser`；两类都不硬造提成归属。
- 写入前必须先看 dry-run 输出，确认计划回填条数和异常样本。

缺服务人归属审计脚本规则：

- 只读，不更新 `OrderItem.beauticianId`。
- 输出缺服务人的商品/项目订单明细，并给出候选服务人线索。
- 候选来源包括同订单其他明细、服务任务、预约记录、会员卡消课、终端跟进任务。
- 输出 high/medium/low/none 置信度；即使 high 也需要业务确认后才能写回。
- 商品订单的候选服务人仅作为销售归属线索，不自动等同于项目服务人。

服务人归属写回脚本规则：

- 默认 dry-run，不传 `--apply --yes` 不写库。
- 输入必须是业务确认后的 JSON 清单，至少包含 `orderItemId` 和 `beauticianId`。
- 只允许处理商品/项目明细，卡项、充值等非毛利明细会跳过。
- 校验订单状态必须是 `paid/completed`，订单日期必须落在指定范围内。
- 校验美容师必须存在、门店一致、已绑定 `userId`，否则跳过。
- 写回服务人后，再跑 `operation-profit:backfill` dry-run 生成提成回填计划。

确认清单示例：

```json
{
  "assignments": [
    {
      "orderItemId": 1054,
      "beauticianId": 12,
      "source": "service_task",
      "reason": "业务确认该项目由 12 号美容师服务",
      "confirmedBy": "运营负责人"
    }
  ]
}
```

预检输出：

| 指标 | 说明 |
| --- | --- |
| 商品订单数 | 期间内有效商品订单 |
| 缺商品 ID 明细数 | 无法关联商品档案的订单明细 |
| 缺商品成本数 | 商品档案成本为 0 或空 |
| 项目订单数 | 期间内有效项目订单 |
| 缺项目 ID 明细数 | 无法关联项目档案的订单明细 |
| 缺 BOM 项目数 | 有服务但无 BOM |
| 缺提成订单数 | 有收入但无 `CommissionRecord` |
| 缺服务人账号数 | 美容师存在但未绑定系统用户，无法匹配指定员工提成规则 |
| 服务人候选置信度 | 缺服务人明细可按 high/medium/low/none 区分后续处理优先级 |

执行规则：

- 先跑只读 audit，不写库。
- 写库 backfill 必须用户明确授权。
- backfill 必须幂等：重复执行不重复生成提成、不重复写成本。
- 每次执行输出影响行数和异常清单。

6. 检查商品毛利和项目毛利是否分别扣除成本和提成。

## 9. 本次补充后的更细执行蓝图

本补充计划围绕两个老板最关心的追问展开：商品销售是否赚钱，项目服务扣掉耗材和提成后是否赚钱。下一步执行时不再只验“页面有数”，而是按订单明细逐条复算。

### 9.1 商品毛利闭环

目标：新增商品毛利，并且明确关联已有商品成本、商品订单、提成规则。

商品毛利完整链路：

```text
商品订单 ProductOrder
  -> 商品明细 OrderItem(itemType=product/goods)
  -> 商品档案 Product(costPrice/retailPrice/category)
  -> 库存销售出库 StockMovement
  -> 商品提成规则 CommissionRule(type=product)
  -> 商品提成记录 CommissionRecord(orderItemId)
  -> 商品毛利 ProductMargin
```

详细任务：

| 步骤 | 任务 | 数据对象 | 交付物 | 验收标准 |
| --- | --- | --- | --- | --- |
| 1 | 冻结商品毛利公式 | `OrderItem`、`Product`、`CommissionRecord` | 文档口径 | 公式固定为：商品净收入 - 商品成本 - 商品提成 |
| 2 | 商品订单收入接入 | `ProductOrder`、`OrderItem` | 商品毛利 API 收入字段 | 只统计有效商品订单，不把取消订单计入收入 |
| 3 | 商品成本接入 | `OrderItem.payload.costAmount`、`StockMovement`、`Product.costPrice` | 成本来源字段 | 能区分订单快照、库存出库、商品档案、缺成本 |
| 4 | 商品提成接入 | `CommissionRule(type=product)`、`CommissionRecord.orderItemId` | 提成成本字段 | 商品订单有提成规则时，毛利扣除提成成本 |
| 5 | 新增商品毛利页面 | `/operation-profit/product-margins` | 商品毛利子页面 | 能按商品查看销售额、成本、提成、毛利、毛利率 |
| 6 | 数据质量提示 | 商品成本、提成记录 | 缺口标签 | 缺成本、缺提成不能展示为完整高毛利 |
| 7 | 抽样复算 | 2-3 条商品订单 | 验收记录 | 页面毛利能用订单明细手工复算一致 |

商品成本优先级：

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | 订单明细成本快照 | 新订单优先使用成交当时写入的 `costPrice/costAmount` |
| 2 | 库存销售出库 | 历史订单如有销售出库，用出库侧证明商品成本发生 |
| 3 | 商品档案成本 | 无快照和出库确认时，用 `Product.costPrice` 兜底 |
| 4 | 缺成本 | 无法取得成本时标记 `cost_missing`，不参与完整毛利判断 |

商品提成规则验收：

| 场景 | 配置 | 预期 |
| --- | --- | --- |
| 全商品提成 | `CommissionRule.type=product`、`targetType=all` | 所有商品订单明细生成提成 |
| 指定分类提成 | `targetType=category` | 仅该商品分类生成提成 |
| 指定商品提成 | `targetType=specific`、`targetId=Product.id` | 仅该商品生成提成 |
| 指定员工提成 | 规则绑定 `userId` | 仅归属该员工的商品明细生成提成 |
| 缺服务人 | `OrderItem.beauticianId` 为空 | 不硬造提成，进入异常清单 |

商品毛利验收样本：

| 样本 | 输入 | 预期 |
| --- | --- | --- |
| 正常毛利 | 售价 120，成本 50，提成 6 | 毛利 64，毛利率 53.33% |
| 低毛利 | 售价 100，成本 90，提成 5 | 毛利 5，标记低毛利 |
| 亏损 | 售价 80，成本 90，提成 4 | 毛利 -14，标记亏损 |
| 缺成本 | 有销售收入，无成本 | 标记 `cost_missing` |
| 缺提成 | 有销售收入，无提成记录 | 标记 `missing_commission` |

### 9.2 项目毛利闭环

目标：项目毛利需要把项目收入关联已有项目订单，耗材成本关联已有项目 BOM，提成成本关联已有提成规则。

项目毛利完整链路：

```text
项目订单 ProductOrder
  -> 项目明细 OrderItem(itemType=project/service/service_project)
  -> 项目档案 Project
  -> 项目 BOM ProjectBomItem
  -> 实际耗材出库 StockMovement
  -> 项目提成规则 CommissionRule(type=project)
  -> 项目提成记录 CommissionRecord(orderItemId)
  -> 项目毛利 ProjectMargin
```

详细任务：

| 步骤 | 任务 | 数据对象 | 交付物 | 验收标准 |
| --- | --- | --- | --- | --- |
| 1 | 冻结项目毛利公式 | `OrderItem`、`ProjectBomItem`、`CommissionRecord` | 文档口径 | 公式固定为：项目收入 - 耗材成本 - 项目提成 |
| 2 | 项目订单收入接入 | `ProductOrder`、`OrderItem.itemType=project/service/service_project` | 项目收入字段 | 项目订单收入能按项目聚合 |
| 3 | 消课收入接入 | `CardUsageRecord` | 消课估算收入字段 | 消课收入标记为估算口径，不和现金收入混淆 |
| 4 | BOM 标准耗材接入 | `ProjectBomItem`、`Product.costPrice` | 标准耗材成本 | 无实耗流水时可按 BOM 估算耗材 |
| 5 | 实耗成本接入 | `StockMovement` | 实际耗材成本 | 有实际出库时优先使用实耗成本 |
| 6 | 项目提成接入 | `CommissionRule(type=project)`、`CommissionRecord.orderItemId` | 项目提成成本 | 项目订单明细能扣除对应提成 |
| 7 | 项目缺口提示 | 项目档案、BOM、实耗、提成 | 缺口标签 | 缺项目档案、缺 BOM、缺实耗、缺提成必须可见 |
| 8 | 抽样复算 | 2-3 条项目订单 | 验收记录 | 页面项目毛利能用订单、BOM、提成记录复算一致 |

项目耗材成本优先级：

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | 实际耗材出库 | 已产生 `StockMovement` 时，优先按真实出库成本 |
| 2 | 项目 BOM 标准成本 | 没有实耗时，用 `ProjectBomItem.quantity × Product.costPrice` 估算 |
| 3 | 缺项目档案 | 有项目收入但项目档案不存在，标记 `missing_project_master`，收入不能丢失 |
| 4 | 缺 BOM | 有项目收入但没有 BOM，标记 `missing_bom` |
| 5 | 缺实耗 | 有 BOM 但没有实际耗材出库，标记 `missing_actual_consumption` |

项目提成规则验收：

| 场景 | 配置 | 预期 |
| --- | --- | --- |
| 全项目提成 | `CommissionRule.type=project`、`targetType=all` | 所有项目订单明细生成提成 |
| 项目分类提成 | `targetType=category` | 仅该项目分类生成提成 |
| 指定项目提成 | `targetType=specific`、`targetId=Project.id` | 仅该项目生成提成 |
| 员工等级提成 | 规则绑定 `levelId` | 按美容师等级匹配提成 |
| 指定员工提成 | 规则绑定 `userId` | 按订单服务人匹配提成 |
| 缺服务人 | `OrderItem.beauticianId` 为空 | 不回填提成，进入异常清单 |

项目毛利验收样本：

| 样本 | 输入 | 预期 |
| --- | --- | --- |
| 完整项目毛利 | 项目收入 500，实耗 80，提成 50 | 毛利 370，毛利率 74% |
| BOM 估算项目 | 项目收入 500，无实耗，BOM 标准成本 70，提成 50 | 毛利 380，并提示耗材为估算 |
| 缺项目档案项目 | 项目有收入，但项目档案不存在 | 保留收入行，标记 `missing_project_master`，不从项目毛利页消失 |
| 缺 BOM 项目 | 项目有收入，无 BOM | 标记 `missing_bom`，不误判高毛利 |
| 缺提成项目 | 项目有收入，无提成记录 | 标记 `missing_commission` |
| 亏损项目 | 项目收入 100，耗材 80，提成 40 | 毛利 -20，标记亏损 |

### 9.4 本次补充后的明确边界

| 边界 | 处理方式 |
| --- | --- |
| 历史订单缺服务人 | 不自动生成提成，先输出异常清单，由业务确认归属 |

| 写库回填 | 必须先 dry-run，再经用户明确授权执行 |
