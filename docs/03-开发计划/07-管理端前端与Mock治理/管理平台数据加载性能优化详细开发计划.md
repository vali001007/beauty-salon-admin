# 管理平台数据加载性能优化详细开发计划

生成日期：2026-06-07

关联文档：

- `docs/管理平台数据加载性能优化建议方案.md`

## 1. 目标

本计划要求把建议方案中的所有优化任务全部开发完成，覆盖管理平台核心慢加载问题：

1. 客户数据页首屏不再被消费记录、肌肤档案、全量客户接口拖慢。
2. 所有客户选择场景改为远程搜索，不再通过全量客户接口后前端截取。
3. 消费记录、肌肤档案、推荐卡、自动营销效果等慢接口改为分页、缓存或异步任务。
4. 商品、项目、库存等模块不再依赖无上限全量接口支撑主流程。
5. 前端拆分 loading、请求缓存和大包体，降低页面进入等待。
6. 后端增加必要索引、轻量 select、响应耗时日志和性能测试。

目标指标：

| 场景 | 目标 |
| --- | ---: |
| 客户数据基础信息首屏 | P95 < 1500ms |
| 客户分页接口 | P95 < 800ms |
| 消费记录分页接口 | P95 < 1000ms |
| 肌肤档案分页接口 | P95 < 1000ms |
| 客户远程搜索 | P95 < 500ms |
| 智能推荐读取缓存 | P95 < 800ms |
| 商品/项目/库存分页接口 | P95 < 1000ms |
| 前端单页面首包解析 | P95 < 1500ms |

## 2. 范围

### 2.1 必须开发

- 后端新增分页、搜索、摘要、缓存接口。
- 后端优化已有慢查询和索引。
- 前端替换全量接口调用。
- 前端按 Tab 懒加载。
- 前端引入轻量请求缓存。
- 前端拆分大 chunk。
- 补充单元测试、接口测试、页面验证脚本。
- 输出性能复测报告。

### 2.2 不作为本轮目标

- 不改变业务数据口径。
- 不删除历史 mock 文件。
- 不重构整个权限体系。
- 不迁移数据库供应商。
- 不做 UI 大改版，只调整 loading 和数据获取方式。

## 3. 后端开发任务

### 3.1 客户轻量搜索接口

新增接口：

```text
GET /api/customers/search?keyword=&storeId=&storeName=&limit=20
```

开发文件：

- `packages/server-v2/src/customers/customers.controller.ts`
- `packages/server-v2/src/customers/customers.service.ts`
- `packages/server-v2/src/customers/dto/query-customers.dto.ts` 或新增 `search-customers.dto.ts`

实现要求：

- 支持按姓名、手机号、微信号搜索。
- 支持当前门店隔离，优先使用 `X-Store-Id`。
- `limit` 默认 20，最大 50。
- 返回轻量字段：

```ts
{
  id: number;
  name: string;
  phone: string;
  storeId: number;
  storeName: string;
  memberLevel: string;
  lastVisitDate?: string;
}
```

禁止：

- 不 include `balanceAccounts`。
- 不 include `customerCards`。
- 不返回大字段。

验收：

- 空关键词时只返回当前门店前 20 条最近客户。
- 有关键词时 P95 < 500ms。
- 无权限或跨门店不能返回其他门店数据。

### 3.2 消费记录分页接口

新增接口：

```text
GET /api/customers/consumption-records/paginated?page=1&pageSize=20&customerId=&keyword=&storeId=&startDate=&endDate=
```

开发文件：

- `packages/server-v2/src/customers/customers.controller.ts`
- `packages/server-v2/src/customers/customers.service.ts`
- `src/api/real/customer.ts`
- `src/types/customer.ts`

实现要求：

- 默认 pageSize 20，最大 100。
- 支持按客户、门店、时间范围筛选。
- 返回 `{ items, total, page, pageSize }`。
- select 字段控制在列表需要范围内：
  - id
  - customerId
  - userName/customerName
  - storeName
  - consumeType
  - consumeContent
  - payMethod
  - amount
  - campaign
  - consumeTime

验收：

- 替代当前 `/customers/consumption-records` 全量接口。
- 不再出现 30 秒超时。
- 首屏不请求该接口，切到消费记录 Tab 才请求。

### 3.3 肌肤档案分页接口

```text
GET /api/customers/health-profiles/paginated?page=1&pageSize=20&keyword=&skinType=&storeId=

开发文件：

- `packages/server-v2/src/customers/customers.service.ts`

验收：

```

实现要求：

### 3.6 全量接口限流与上限

改造接口：

- `GET /api/customers`
- `GET /api/customers/consumption-records`
- `GET /api/customers/health-profiles`
- `GET /api/projects`
- `GET /api/products`

实现要求：

- 前端主流程不再使用这些接口。
- 后端保留兼容，但加默认上限：
  - 默认 `limit=200`
  - 最大 `limit=500`
- 如调用方需要全量导出，必须走导出接口，不走列表接口。

验收：

- 任意全量接口不会再 30 秒超时。
- 返回响应头或 payload 标记 `limited: true`，避免误判为全量。

### 3.8 智能推荐每日缓存与异步刷新

新增或改造接口：

```text
GET /api/marketing/recommendations/latest
POST /api/marketing/recommendations/refresh
GET /api/marketing/recommendations/refresh-status/:jobId
```

开发文件：

- `packages/server-v2/src/marketing/marketing.controller.ts`
- `packages/server-v2/src/marketing/marketing.service.ts`
- `packages/server-v2/prisma/schema.prisma`
- 新增 migration
- `src/api/real/recommendation.ts`
- `src/app/pages/MarketingRecommendation.tsx`

新增数据表建议：

```prisma
model MarketingRecommendationCache {
  id            Int      @id @default(autoincrement())
  storeId       Int?
  businessDate  DateTime
  status        String   @default("ready")
  cardsJson     Json
  summaryJson   Json?
  generatedAt   DateTime @default(now())
  expiresAt      DateTime?

  @@unique([storeId, businessDate])
  @@index([storeId, businessDate, status])
}

model MarketingRecommendationRefreshJob {
  id          String   @id
  storeId     Int?
  status      String   @default("pending")
  progress    Int      @default(0)
  errorMessage String?
  startedAt   DateTime?
  finishedAt  DateTime?
  createdAt   DateTime @default(now())

  @@index([storeId, status])
  @@index([createdAt])
}
```

实现要求：

- 页面进入只读当天缓存。
- 如果没有缓存，快速返回 fallback 或最近一次缓存，并标记 `stale: true`。
- 刷新推荐按钮触发异步任务，不阻塞页面。
- 同一门店同一天只能有一个运行中的刷新任务。
- 刷新完成后更新缓存表。

验收：

- `/marketing/recommendations/latest` P95 < 800ms。
- `/marketing/recommendations` 旧接口兼容，内部转 latest。
- 刷新按钮不再让页面等待 45 秒。

### 3.10 数据库索引与字段冗余

新增 migration：

```text
packages/server-v2/prisma/migrations/<timestamp>_performance_indexes/
```

建议索引：

```prisma
@@index([storeId, deletedAt, createdAt])
@@index([storeId, deletedAt, name])
@@index([storeId, deletedAt, phone])
```

消费记录建议：

- 如当前 `ConsumptionRecord` 没有 `storeId`，新增 `storeId Int?`。
- seed 和订单/核销写入时同步写入 storeId。
- 增加：

```prisma
@@index([storeId, consumeTime])
@@index([customerId, consumeTime])
```

肌肤档案建议：

```prisma
@@index([lastCheck])
```

如需按肤质筛选：

```prisma
@@index([skinType])
```

验收：

- migration 可执行。
- seed 和现有数据补 storeId。
- 查询结果与旧口径一致。

### 3.11 请求耗时日志

开发文件：

- `packages/server-v2/src/common/interceptors/logging.interceptor.ts`
- 或新增 `performance.interceptor.ts`

实现要求：

- 记录接口路径、method、status、duration、requestId、storeId。
- 超过 1000ms 标记 slow request。
- 超过 3000ms 打印查询优化提示。
- 日志不能打印手机号、客户姓名等敏感数据。

验收：

- 本地可看到慢接口日志。
- 生产可接入日志平台。
