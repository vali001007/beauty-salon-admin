# 智能营销模块整合详细开发计划

更新时间：2026-06-10

## 1. 背景与目标

当前「智能营销」下包含 8 个二级模块：

- 智能推荐
- 自动营销
- 规则库
- 营销页面
- Ami Glow
- 优惠活动
- 活动管理
- 效果分析

这些模块本身都有业务价值，但当前以平铺菜单呈现，用户会遇到三个问题：

1. 不知道从哪里开始做营销：推荐、规则、活动、页面都像入口。
2. 不清楚模块之间的先后关系：优惠活动、营销页面、Ami Glow 推荐位都属于营销资产，但被拆成三个入口。
3. 不清楚「活动管理」和「效果分析」边界：一个看运营过程，一个看结果复盘，但菜单上没有体现。

本次开发目标不是删功能，而是重构信息架构和页面流转，让智能营销形成清晰闭环：

```text
机会发现 -> 规则/策略生产 -> 营销资产承接 -> 执行运营 -> 效果复盘
```

## 2. 目标信息架构

建议将「智能营销」收敛为 5 个二级模块：

| 新模块 | 承接旧模块 | 产品定位 |
| --- | --- | --- |
| 智能推荐 | 智能推荐 | 发现营销机会，推荐客户群、规则、活动动作 |
| 自动营销 | 自动营销 + 活动管理中自动策略部分 | 管理已启用自动营销策略、执行状态、启停和执行记录 |
| 规则库 | 规则库 | 管理系统默认规则和门店自定义规则模板 |
| 营销资产 | 营销页面 + 优惠活动 + Ami Glow | 管理承接页、优惠权益、小程序推荐位 |
| 效果分析 | 效果分析 + 活动详情效果入口 | 统一复盘营销活动、自动营销、页面、小程序渠道效果 |

阶段性可保留旧路由，先改菜单和页面聚合，降低迁移风险。

## 3. 模块边界定义

### 3.1 智能推荐

定位：营销机会入口。

保留能力：

- 推荐卡片：流失召回、复购提醒、高价值客户维护、新客转化等。
- 推荐依据：RFM、LTV、流失预警、关联规则。
- 推荐客户名单和排序。
- 一键创建活动草稿。
- 一键创建自动营销草稿。
- 跳转规则库启用推荐规则。

边界：

- 不直接管理长期运行策略。
- 不承载营销页面/优惠活动配置。
- 只做「推荐」和「发起下一步」。

### 3.2 规则库

定位：自动营销规则模板资产。

保留能力：

- 系统推荐规则。
- 门店自定义规则。
- 复制系统规则为我的规则。
- 编辑参数、动作、执行计划、频控。
- 预估命中客户。
- 一键启用规则并生成自动营销策略。

边界：

- 规则库只管理模板。
- 启用后的运行状态归「自动营销」管理。
- 效果数据可在规则详情中展示摘要，但完整分析归「效果分析」。

### 3.3 自动营销

定位：长期自动执行的营销策略管理。

保留/调整能力：

- 自动营销策略列表。
- 创建策略、编辑策略、保存草稿、启用、停用。
- 查看触发规则、触达动作、执行计划、频控。
- 查看最近执行记录。
- 标识策略来源：手动创建、规则库、智能推荐。

需要调整：

- 当前路由 `/customer-marketing/strategy-templates` 页面标题是「自动营销」，但路径名仍偏“模板”。建议保留兼容路由，同时新增语义路由：

```text
/customer-marketing/automation
```

- 菜单指向新路由，旧路由重定向或继续兼容。

边界：

- 不再承担活动管理。
- 不管理营销页面、优惠活动、小程序推荐位。
- 只管自动策略的运行。

### 3.4 营销资产

定位：营销承接物和渠道展示配置中心。

合并对象：

- 营销页面
- 优惠活动
- Ami Glow 推荐位

建议页面结构：

```text
营销资产
├── 页面
├── 优惠
├── 小程序推荐位
└── 行为事件
```

Tab 说明：

| Tab | 来源页面 | 说明 |
| --- | --- | --- |
| 页面 | MarketingPageManagement | 管理 H5/小程序承接页、二维码、页面状态 |
| 优惠 | PromotionManagement | 管理券、折扣、权益、有效期 |
| 小程序推荐位 | AmiGlowManagement 展示配置 | 配置 Ami Glow 首页推荐内容和排序 |
| 行为事件 | AmiGlowManagement 行为事件 | 查看小程序浏览、点击、预约等行为记录 |

边界：

- 这里不直接做智能推荐。
- 这里不管理自动策略运行。
- 它提供可被活动、自动营销、智能推荐复用的素材和权益。

### 3.5 效果分析

定位：营销复盘中心。

保留能力：

- 统一查看活动和自动营销效果。
- 按类型筛选：活动、自动营销、页面、小程序渠道。
- 展示触达人数、转化人数、收入、ROI。
- 支持跳转详情。

需要补齐：

- 页面维度效果：营销页面浏览、点击、表单、预约、转化。
- Ami Glow 渠道效果：曝光、点击、跳转、预约、转化。
- 优惠活动效果：领券、核销、带来收入。

边界：

- 活动管理可以保留「查看效果」入口，但完整复盘在效果分析。

## 4. 冗余处理策略

### 4.1 规则库 vs 自动营销

当前问题：

- 规则库可以启用规则生成自动营销。
- 自动营销也可以手动创建策略。
- 用户可能分不清二者差异。

处理方式：

- 规则库文案统一为「规则模板」。
- 自动营销文案统一为「运行策略」。
- 规则库启用成功后，提示：

```text
规则已启用，并生成自动营销策略。可前往自动营销查看运行状态。
```

- 自动营销列表增加来源字段：

```text
来源：手动创建 / 规则库 / 智能推荐
```

### 4.2 营销页面 vs 优惠活动 vs Ami Glow

当前问题：

- 三个模块都是营销资产，但独立展示，用户需要来回跳。

处理方式：

- 新增「营销资产」聚合页。
- 原三个菜单入口下线或隐藏。
- 原路由保留，避免外部链接失效。
- 聚合页内使用 Tab 承载旧页面能力。

### 4.3 活动管理 vs 效果分析

当前问题：

- 活动管理里有查看效果入口，效果分析也看效果。

处理方式：

- 活动管理只保留活动过程管理：状态、时间、渠道、目标客户、活动内容。
- 效果分析承接复盘：曝光、触达、转化、收入、ROI。
- 活动管理的「查看效果」按钮跳到效果分析或活动效果详情。

## 5. 开发阶段计划

### 阶段 1：菜单与路由收口

目标：先降低用户理解成本，不大改业务逻辑。

开发内容：

- 调整 `src/app/components/Layout.tsx` 的智能营销菜单。
- 菜单改为：

```text
智能推荐
规则库
自动营销
营销资产
效果分析
```

- 新增路由：

```text
/customer-marketing/automation
/customer-marketing/assets
```

- 保留旧路由：

```text
/customer-marketing/strategy-templates
/customer-marketing/pages
/customer-marketing/promotions
/customer-marketing/ami-glow
/customer-marketing/activity-management
```

- 第一阶段旧路由不删除，可由聚合页跳转或保持直接访问。

涉及文件：

- `src/app/components/Layout.tsx`
- `src/app/routes.tsx`
- `src/app/pages/CreateMarketing.tsx`
- `src/app/pages/MarketingStrategy.tsx`

验收标准：

- 智能营销菜单只显示 5 个核心入口。
- 旧路由直接访问不 404。
- 自动营销入口打开的是自动营销策略管理能力。
- 不影响已有权限。

### 阶段 2：新增营销资产聚合页

目标：合并营销页面、优惠活动、Ami Glow 三类资产。

开发内容：

- 新增页面：

```text
src/app/pages/MarketingAssets.tsx
```

- 页面使用 Tab：

```text
页面 / 优惠 / 小程序推荐位 / 行为事件
```

- 第一版复用现有组件逻辑，可采用两种方式：

方案 A：抽组件

- 将 `MarketingPageManagement` 拆出列表和弹窗组件。
- 将 `PromotionManagement` 拆出列表和弹窗组件。
- 将 `AmiGlowManagement` 拆出推荐位配置和行为事件组件。
- `MarketingAssets` 组合这些组件。

方案 B：轻量嵌入

- 先在 `MarketingAssets` 里实现统一外壳。
- Tab 内复用现有 API 和表格逻辑。
- 后续再抽公共组件。

建议选择方案 B，开发成本低，适合当前 MVP 后的迭代。

涉及文件：

- `src/app/pages/MarketingAssets.tsx`
- `src/app/pages/MarketingPageManagement.tsx`
- `src/app/pages/PromotionManagement.tsx`
- `src/app/pages/AmiGlowManagement.tsx`
- `src/api/marketingPage.ts`
- `src/api/promotion.ts`
- `src/api/customerApp.ts`

验收标准：

- 用户从「营销资产」可以管理页面、优惠、小程序推荐位。
- 三个旧模块能力不丢失。
- 默认每个 Tab 列表分页 10 条。
- 页面、优惠、小程序推荐位之间可以互相引用或至少展示关联字段。

### 阶段 3：自动营销与活动管理拆边界

目标：让「自动营销」只承载长期自动策略，让「活动」退到活动运营场景。

开发内容：

- 新增或调整自动营销入口：

```text
/customer-marketing/automation
```

- 页面标题统一为「自动营销」。
- 列表字段建议：

| 字段 | 说明 |
| --- | --- |
| 策略名称 | 自动营销策略名称 |
| 来源 | 手动创建 / 规则库 / 智能推荐 |
| 触发规则 | 沉睡客户、生日、领券未核销等 |
| 执行动作 | 短信、小程序、微信、优惠券 |
| 执行计划 | 实时、每日、每周、每月 |
| 状态 | 草稿、已启用、暂停 |
| 最近执行 | 最近一次执行时间 |
| 效果摘要 | 触达、转化、收入 |
| 操作 | 查看、编辑、启用、停用、查看效果 |

- 将 `activity-management` 菜单隐藏或更名为「营销活动」，放入后续阶段。
- 如果保留活动管理，建议菜单不再放在智能营销主入口第一层，后续并入「营销资产」或「活动运营」。

涉及文件：

- `src/app/pages/CreateMarketing.tsx`
- `src/app/pages/MarketingStrategy.tsx`
- `src/api/marketing.ts`
- `packages/server-v2/src/marketing/*`

验收标准：

- 自动营销列表能清晰看到策略来源。
- 从规则库启用的规则能在自动营销中看到来源规则。
- 从智能推荐生成的自动营销草稿能标记来源为智能推荐。

### 阶段 4：效果分析统一入口增强

目标：让效果分析成为唯一复盘入口。

开发内容：

- `MarketingAnalytics` 增加对象类型：

```text
全部 / 营销活动 / 自动营销 / 营销页面 / 优惠活动 / Ami Glow
```

- 增加统一效果模型：

| 指标 | 含义 |
| --- | --- |
| 曝光人数 | 页面、小程序推荐位等可用 |
| 触达人数 | 短信、推送、微信等 |
| 点击人数 | 页面、小程序行为 |
| 预约人数 | 预约转化 |
| 成交人数 | 订单转化 |
| 核销人数 | 优惠活动核销 |
| 收入 | 关联订单收入 |
| ROI | 收入 / 营销成本 |

- 活动管理、自动营销、营销页面、优惠活动、Ami Glow 均提供「查看效果」入口。

涉及文件：

- `src/app/pages/MarketingAnalytics.tsx`
- `src/app/pages/MarketingActivityEffect.tsx`
- `packages/server-v2/src/marketing/*`
- `packages/server-v2/src/marketing-pages/*`
- `packages/server-v2/src/customer-app/*`
- `packages/server-v2/src/promotions/*`

验收标准：

- 用户可以按对象类型查看效果。
- 所有营销对象都能跳到统一效果页或详情页。
- 数据不足时展示「暂无数据」和原因，不显示空白。

## 6. 推荐实施顺序

建议按低风险到高收益推进：

1. 调整菜单为 5 个入口，保留旧路由。
2. 新增「营销资产」聚合页，用 Tab 合并页面、优惠、Ami Glow。
3. 自动营销页面补来源字段和运行状态表达。
4. 规则库启用后增加跳转自动营销入口。
5. 效果分析统一对象类型和指标口径。
6. 旧入口根据使用情况隐藏或重定向。

## 10. 验收标准

### 信息架构验收

- 智能营销菜单只保留 5 个一级心智入口。
- 用户能理解：
  - 智能推荐是机会发现。
  - 规则库是模板。
  - 自动营销是运行策略。
  - 营销资产是页面、优惠、小程序推荐位。
  - 效果分析是复盘。

### 功能验收

- 原 `营销页面` 能力在「营销资产 / 页面」中可用。
- 原 `优惠活动` 能力在「营销资产 / 优惠」中可用。
- 原 `Ami Glow` 展示配置和行为事件在「营销资产」中可用。
- 旧路由不 404。
- 规则库启用后能在自动营销看到策略。
- 智能推荐仍能创建活动或自动营销草稿。
- 效果分析能统一查看不同营销对象。

### 性能验收

- 营销资产各 Tab 默认分页 10 条。
- 切换 Tab 不阻塞整页。
- 任一接口失败时，只影响当前 Tab，不影响其他 Tab。

## 11. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 一次性合并页面过多 | 开发周期拉长 | 先做聚合页壳和菜单，旧页面能力逐步迁入 |
| 旧路由被外部入口引用 | 访问 404 | 保留旧路由，必要时重定向 |
| 权限拆太细 | 增加配置成本 | MVP 继续复用现有营销权限 |
| 效果指标口径不统一 | 用户不信数据 | 先明确曝光、点击、触达、转化、收入定义 |
| Ami Glow 同时是渠道和页面 | 产品边界混乱 | 在营销资产中拆成「推荐位」和「行为事件」两个 Tab |

## 12. 建议优先级

P0：

- 菜单改为 5 个核心入口。
- 新增营销资产聚合页。
- 自动营销和规则库文案边界调整。
- 旧路由保留。

P1：

- 自动营销来源字段展示。
- 规则库启用后跳转自动营销。
- 效果分析增加对象类型筛选。

P2：

- 统一效果统计接口。
- 营销资产之间的关联关系，例如优惠活动关联营销页面、Ami Glow 推荐位关联页面。
- 旧页面组件抽象复用。
