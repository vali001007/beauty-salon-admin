# 会员卡划扣本金/赠送金额按比例分摊改进方案

## 1. 背景与目标

当前 Ami Core 已经把会员卡储值余额拆成两类资产：

- 现金余额：客户真实充值或开卡支付形成的本金余额。
- 赠送余额：充值赠送、开卡赠送或人工赠送形成的权益余额。

但当前会员卡划扣的后端实现是“先扣赠送余额，再扣现金余额”。这会导致前几次消费几乎全部被记为赠送消耗，后续才进入本金消耗，进而影响会员资产、退款、日结、经营收入和提成口径的稳定性。

本方案将产品口径冻结为：**会员卡划扣时按现金余额与赠送余额的当前比例分摊扣减**。

目标是让每一笔会员卡划扣都能清楚回答：

- 本次划扣总额是多少。
- 其中多少来自客户真实充值本金。
- 其中多少来自赠送权益。
- 划扣后现金余额和赠送余额分别剩多少。
- 日结、退款、会员资产、经营利润和提成如何使用这组数据。

## 2. 产品口径冻结

### 2.1 核心规则

会员卡划扣总额 = 本金划扣 + 赠送划扣。

每次划扣时，根据划扣前账户的现金余额和赠送余额计算分摊比例：

```text
现金占比 = 划扣前现金余额 / (划扣前现金余额 + 划扣前赠送余额)
赠送占比 = 划扣前赠送余额 / (划扣前现金余额 + 划扣前赠送余额)

本次本金划扣 = 本次划扣金额 × 现金占比
本次赠送划扣 = 本次划扣金额 × 赠送占比
```

示例：

```text
充值本金 1000，赠送 200，账户总余额 1200
本次会员卡划扣 120

本金划扣 = 120 × 1000 / 1200 = 100
赠送划扣 = 120 × 200 / 1200 = 20
```

### 2.2 边界规则

- 如果赠送余额为 0，本次划扣全部扣现金余额。
- 如果现金余额为 0，本次划扣全部扣赠送余额。
- 如果划扣金额大于现金余额 + 赠送余额，拒绝划扣，提示余额不足。
- 金额精度按分保留，最终 `本金划扣 + 赠送划扣` 必须等于本次划扣金额。
- 分摊后的余额不能小于 0，最后一分钱由现金侧或余额更充足的一侧吸收四舍五入差额。

### 2.3 退款口径

退款分两类处理：

1. **订单退款**
   - 如果原订单使用会员卡划扣支付，退款应按原划扣流水的本金划扣和赠送划扣反向恢复。
   - 即原订单扣了本金 100、赠送 20，退款时恢复本金 100、赠送 20。
   - 退款流水仍要记录恢复后的现金余额和赠送余额。

2. **会员卡退卡/退余额**
   - 只退剩余现金余额。
   - 剩余赠送余额不退，退卡完成后赠送余额应清零或冻结。
   - 退款金额不应包含赠送余额。

### 2.4 报表口径

- 会员卡划扣流水：展示划扣总额、本金划扣、赠送划扣、划扣后现金余额、划扣后赠送余额。
- 收银对账：会员卡划扣仍属于非现金结算，不计入现金收入；但应进入营业收入或履约收入口径。
- 会员资产：储值余额继续拆现金余额和赠送余额；消费流水按本金/赠送消耗汇总。
- 经营利润：赠送划扣可作为优惠/权益消耗分析字段，不应被误认为现金收入。
- 提成：默认可按本次履约金额计算提成，但提成明细应保留本金/赠送拆分，便于后续切换规则。

## 3. 当前系统现状

### 3.1 已有数据基础

当前 schema 已具备基础字段：

- `CustomerBalanceAccount.cashBalance`
- `CustomerBalanceAccount.giftBalance`
- `CustomerBalanceTransaction.amount`
- `CustomerBalanceTransaction.giftAmount`
- `CustomerBalanceTransaction.cashBalanceBefore`
- `CustomerBalanceTransaction.cashBalanceAfter`
- `CustomerBalanceTransaction.giftBalanceBefore`
- `CustomerBalanceTransaction.giftBalanceAfter`

因此 P0 阶段不需要新增 Prisma 字段，即可把划扣口径从“先扣赠送”改为“按比例分摊”。

### 3.2 当前断点

后端 `deductMemberBalanceForOrder` 当前逻辑为：

```text
giftDeduct = min(giftBalanceBefore, amount)
cashDeduct = amount - giftDeduct
```

这表示系统会先扣赠送余额，直到赠送余额耗尽后才扣现金余额。该逻辑需要调整。

前端 mock 里也存在类似“先扣赠送”的本地逻辑，需要同步改掉，避免离线样例和单测口径不一致。

## 4. 改造范围

### 4.1 后端

重点改造：

- `packages/server-v2/src/orders/orders.service.ts`
  - 抽出 `allocateMemberBalanceDeduction` 工具方法。
  - 修改 `deductMemberBalanceForOrder`，按比例计算 `cashDeduct/giftDeduct`。
  - 保证写入 `CustomerBalanceTransaction.amount/giftAmount` 的值就是分摊后的本金/赠送金额。

需要补充测试：

- 现金 1000、赠送 200，划扣 120，得到本金 100、赠送 20。
- 只有现金余额时，全扣现金。
- 只有赠送余额时，全扣赠送。
- 金额存在小数时，总和不差分。
- 余额不足时拒绝。
- 订单退款恢复原划扣拆分。

### 4.2 管理端

重点改造：

- 会员卡划扣流水列表增加或明确展示：
  - 划扣总额
  - 本金划扣
  - 赠送划扣
  - 划扣后现金余额
  - 划扣后赠送余额
- 会员卡账户详情、会员资产页保持现金余额/赠送余额拆分展示。
- 若页面已有“划扣合计”，需要明确这是 `本金划扣 + 赠送划扣`，不是现金收入。

### 4.3 智能终端

重点改造：

- 终端收银使用会员卡划扣时，提交总划扣金额即可。
- 后端统一负责比例分摊，终端不自行计算本金/赠送。
- 收款完成卡片和小票可展示：
  - 会员卡划扣总额
  - 本金划扣
  - 赠送划扣
  - 剩余现金余额
  - 剩余赠送余额

### 4.4 财务模块

重点改造：

- 收银对账：会员卡划扣继续作为非现金结算项展示。
- 日结明细：会员卡划扣明细增加本金/赠送拆分。
- 会员资产：储值消费统计按本金消耗、赠送消耗分别聚合。
- 经营利润：赠送消耗可以作为权益/优惠消耗分析，不和现金收入混用。
- 员工提成：P0 保持现有提成基数不变，只在流水中保留拆分；P1 再支持“只按本金部分提成”或“按总履约金额提成”的规则配置。

## 5. 推荐实施阶段

### P0：冻结口径并修正划扣算法

目标：最小范围修正真实业务口径。

任务：

1. 后端新增比例分摊工具方法。
2. 修改会员卡划扣逻辑，从先扣赠送改为按比例扣减。
3. 同步更新前端 mock 的划扣逻辑。
4. 补充 orders service 单测。
5. 用真实验收样本验证：
   - 充值 1000 赠送 200。
   - 会员卡划扣 120。
   - 交易流水为本金 100、赠送 20。
   - 账户余额为现金 900、赠送 180。

验收标准：

- 会员卡划扣流水拆分正确。
- 账户余额正确。
- 订单和收银结果不受影响。
- 日结仍能刷新成功。

### P1：退款和报表口径补齐

目标：让订单退款、退卡、日结和会员资产完全使用同一套拆分数据。

任务：

1. 订单退款读取原会员卡划扣流水，按原拆分恢复现金/赠送余额。
2. 会员卡退卡只退现金余额，赠送余额清零或冻结。
3. 收银对账和日结详情补充本金/赠送拆分展示。
4. 会员资产页补充本金消耗、赠送消耗统计。
5. 退款后自动刷新日结，确保当日数据可复算。

验收标准：

- 原订单会员卡划扣本金 100、赠送 20，退款后恢复本金 100、赠送 20。
- 退卡时退款金额不包含赠送余额。
- 日结、会员资产、退款记录三处金额能对齐。

### P2：充值批次台账增强

目标：支持更严格的财务追溯和批次级退款。

当前账户级比例分摊已经能满足 P0/P1 的经营管理需求，但如果后续要处理不同充值批次、不同赠送活动、不同有效期，则需要新增充值批次台账。

建议新增对象：

- `CustomerBalanceLot`
  - `accountId`
  - `sourceTransactionId`
  - `cashOriginalAmount`
  - `giftOriginalAmount`
  - `cashRemainingAmount`
  - `giftRemainingAmount`
  - `expiredAt`
  - `status`

批次消耗规则：

- 默认 FIFO：优先消耗最早充值批次。
- 每个批次内部按该批次现金/赠送剩余比例分摊。
- 如果一个划扣金额跨多个批次，生成多条分摊明细或在交易扩展字段中记录批次明细。

P2 不建议和 P0 同时做，避免一次性改动过重。

## 6. 数据迁移与历史数据处理

P0 阶段不强制重算历史流水，原因：

- 历史划扣已经影响客户当前余额和财务报表，直接回写风险较高。
- 现有流水已经是系统当时口径下的事实记录。

建议策略：

- 新增系统配置或发布说明：自上线时间起，新会员卡划扣采用按比例分摊。
- 历史流水保留原记录。
- 报表可加口径说明：上线日前按旧口径，之后按新口径。
- 如确需修复历史数据，单独做数据迁移方案和审计脚本，不和功能改造混在一起。

## 7. 风险与取舍

### 风险 1：历史和新流水口径不一致

处理方式：

- 不重算历史。
- 用上线日期作为口径切换点。
- 报表导出保留交易时间和拆分字段。

### 风险 2：订单退款未按原流水恢复

处理方式：

- P1 必须读取原 `CustomerBalanceTransaction`。
- 不允许退款时重新按当前余额比例计算。

### 风险 3：前端自行计算导致多端不一致

处理方式：

- 终端、管理端都只提交划扣总额。
- 后端统一计算分摊。
- 前端只展示后端返回结果。

### 风险 4：四舍五入导致余额差异

处理方式：

- 使用分为单位计算。
- 最后一分钱由现金侧或余额充足侧吸收。
- 单测覆盖 0.01、0.03、99.99 等边界金额。

### 样本 A：标准比例划扣

- 客户赠送余额 200。

### 样本 B：只有现金余额

- 会员卡划扣 80。

- 本金划扣 80。
- 赠送划扣 0。
- 现金余额 420。

### 样本 C：只有赠送余额

前置：

- 客户现金余额 0。
- 客户赠送余额 100。

- 赠送划扣 30。

- 赠送余额 70。

### 样本 D：订单退款

前置：

- 原流水本金划扣 100，赠送划扣 20。

- 退款记录和会员卡交易流水都可追溯到原订单。

## 9. 推荐结论

建议按以下方式推进：

1. 先做 P0：把真实会员卡划扣改为按比例分摊，并补充单测和真实样本。
2. 再做 P1：补齐订单退款、退卡、日结和会员资产展示。
3. P2 批次台账暂不进入本次改造，等出现批次有效期、分活动赠送追踪或严格退卡审计需求时再做。

这条路线改动可控，能最快解决当前财务可信度问题，同时不会把系统一次性推入复杂的批次会计模型。
