# 经营利润一级模块详细开发计划

更新时间：2026-06-21
关联需求：`docs/02-产品设计/美容院经营利润看板需求文档.md`
关联调研：`docs/01-市场调研/美容院成本与盈利明细调研报告-2026-06-18.md`
目标范围：Ami_Core 管理端在“财务管理”一级模块下新增“经营利润”业务专区，后端 `server-v2` 提供利润聚合与成本配置能力，前端接入真实 API 与权限体系。
开发原则：先口径、后页面；后端聚合、前端展示；缺失成本显式提示；不把办卡/充值现金流误算成利润。

---

## 1. 交付目标

在管理端“财务管理”下新增“经营利润”业务专区，让老板、店长、财务运营可以看清单店或多店的经营质量。

第一版交付后应达到：

1. 左侧菜单在“财务管理”下出现“经营利润”分组和相关入口。
2. 模块下包含 6 个页面：利润看板、商品毛利、项目毛利、会员卡履约、员工人效、成本配置。
3. 后端提供经营利润聚合 API，前端不拉全量明细自行计算核心利润。
4. 支持月度经营成本录入，包括房租、工资、营销、水电、折旧、其他费用。
5. 利润看板能区分现金收入、经营收入、毛利、经营利润、净利率。
6. 商品毛利能识别高毛利、低毛利、亏损、缺成本和缺提成商品。
7. 项目毛利能识别高收入低毛利项目、缺 BOM 项目、耗材异常项目。
8. 会员卡履约能识别临期、沉睡、高剩余权益客户。
9. 员工人效能按美容师展示服务收入、服务次数、客户数、贡献毛利。
10. 所有页面具备 loading、empty、error、权限不足、数据缺口提示。
11. 权限接入现有 `PermissionGuard`、菜单过滤、角色权限配置。

### 5.2 新增 Prisma 模型

在 `schema.prisma` 新增：

```prisma
model OperatingCost {
  id                Int      @id @default(autoincrement())
  storeId           Int
  periodMonth       String
  costDate          DateTime
  category          String
  amount            Decimal  @default(0)
  allocationType    String   @default("store_month")
  relatedCampaignId Int?
  relatedEmployeeId Int?
  remark            String?
  createdBy         Int?
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  store   Store @relation(fields: [storeId], references: [id], onDelete: Cascade)
  creator User? @relation(fields: [createdBy], references: [id], onDelete: SetNull)

  @@index([storeId, periodMonth])
  @@index([category])
  @@index([costDate])
}
```

同时需要在 `Store` 和 `User` 增加 relation 字段。若 `User` relation 命名冲突，需要使用显式 relation name。

成本分类枚举首期先用字符串，避免 Prisma enum 频繁迁移：

```text

commission
marketing
utilities
depreciation
supplies_adjustment
other
```

### 5.4 后端 API

#### 利润总览

```http
GET /api/operation-profit/overview?storeId=1&from=2026-06-01&to=2026-06-30&basis=operating
```

返回：

- `summary`
- `incomeBreakdown`
- `costBreakdown`
- `trend`
- `alerts`
- `dataQuality`

关键计算：

- 现金收入：订单实收 + 办卡/充值 - 退款。
- 经营收入：服务消课收入 + 单次服务收入 + 产品销售收入。
- 毛利：经营收入 - 耗材成本 - 商品成本 - 直接提成。
- 经营利润：毛利 - 经营成本。
- 净利率：经营利润 / 经营收入。

#### 商品毛利

```http
GET /api/operation-profit/product-margins?storeId=1&from=2026-06-01&to=2026-06-30&page=1&pageSize=20&sortBy=grossProfit
```

聚合逻辑：

1. 从 `ProductOrder` + `OrderItem.itemType=product/goods` 聚合商品订单收入。
2. 从 `RefundRecord` 按订单明细金额比例分摊商品退款。
3. 商品成本优先使用 `OrderItem.payload.costPrice/costAmount` 成本快照，其次使用销售出库 `StockMovement` 确认，最后使用 `Product.costPrice` 兜底。
4. 从 `CommissionRecord.orderItemId` 聚合商品提成成本。
5. 输出商品毛利、毛利率、成本来源、缺成本和缺提成提示。

#### 项目毛利

```http
GET /api/operation-profit/project-margins?storeId=1&from=2026-06-01&to=2026-06-30&page=1&pageSize=20
```

聚合逻辑：

1. 从 `Project` 读取项目基础信息。
2. 从 `OrderItem` 聚合项目订单收入。
3. 从 `CardUsageRecord` 聚合核销次数与项目履约收入。
4. 从 `ProjectBomItem` + `Product.costPrice` 计算标准耗材成本。
5. 从 `StockMovement` 按 `sourceType/sourceId` 聚合实际耗材成本。
6. 从 `CommissionRecord` 聚合直接提成。
7. 返回 `missingCostReasons`，例如 `missing_bom`、`missing_commission`、`missing_actual_consumption`。

#### 会员卡履约

```http
GET /api/operation-profit/prepaid-liabilities?storeId=1&riskOnly=false&page=1&pageSize=20
```

聚合逻辑：

1. `CustomerCard` 读取剩余次数、到期时间、状态。
2. `Card` 读取售价、总次数、适用项目。
3. `CardUsageRecord` 获取最近一次核销。
4. 估算剩余价值：`card.price / card.totalTimes * remainingTimes`。
5. 计算风险：临期、长期未消课、高剩余次数。

#### 员工人效

```http
GET /api/operation-profit/beautician-performance?storeId=1&from=2026-06-01&to=2026-06-30
```

聚合逻辑：

1. `Beautician` 读取员工基础信息。
2. `OrderItem.beauticianId` 聚合服务/销售收入。
3. `CardUsageRecord.beauticianId` 聚合核销服务次数。
4. `CommissionRecord` 聚合提成。
5. 项目耗材成本按服务项目归因，首期可按项目平均成本估算。
6. 返回服务收入、服务次数、客户数、客单价、贡献毛利。

#### 成本配置

```http
GET /api/operation-costs?storeId=1&periodMonth=2026-06&page=1&pageSize=50
POST /api/operation-costs
PATCH /api/operation-costs/:id
DELETE /api/operation-costs/:id
POST /api/operation-costs/copy-from-previous-month
```

校验规则：

- `storeId` 必填。
- `periodMonth` 格式为 `YYYY-MM`。
- `amount >= 0`。
- `category` 必须在允许列表中。
- `costDate` 必须落在 `periodMonth` 对应月份内，或允许作为跨月成本并在返回里提示。

### 5.5 计算口径注意事项

1. 缺 BOM 不按 0 成本静默计算，必须返回缺失原因。
2. 缺提成记录时可按 0 参与当前计算，但必须标记 `missing_commission`。
3. 办卡与充值默认作为现金流，不进入经营收入。
4. 卡项消课收入按卡售价/总次数估算，若卡价或总次数缺失则标记估算失败。
5. 退款必须扣减现金收入；是否扣减经营收入需按退款对应订单类型判断。
6. Decimal 统一在后端转 number/string，前端只展示，不二次做精密财务计算。

### 6.3 利润看板页面

页面模块：

1. 筛选区：时间、门店、口径。
2. 指标卡：现金收入、经营收入、毛利、经营利润、净利率、客单价、到店客户数、消课率。
3. 收入结构图：服务收入、产品收入、办卡收入、充值收入、消课收入。
4. 成本结构图：耗材、商品、人工、房租、营销、水电、折旧、其他。
5. 趋势图：收入、毛利、经营利润。
6. 异常提醒：净利率低、人房占比高、BOM 缺失、消课慢。
7. 数据可信度：完整、部分估算、缺成本、不可计算。

交互要求：

- 页面首次进入默认本月、本门店、经营口径。
- 点击成本缺失提示可跳转成本配置页。
- 点击 BOM 缺失提示可跳转服务消耗与 BOM 页面。
- 图表区域失败不能导致整个页面崩溃。

### 6.4 项目毛利页面

- 筛选：时间、门店、项目分类、状态。

- 行操作：查看详情、编辑 BOM、导出。
- 状态标签：高利润、引流、需优化、成本缺失。

首期详情抽屉：

- 项目基础信息。
- 收入构成。
- 耗材构成。
- 缺失成本原因。
- 推荐动作。

## 10. UI 设计要求

经营利润是高密度运营工具，不做营销型页面。

设计原则：

1. 页面信息密度要高，适合老板和店长快速扫描。
2. 指标卡控制在 6-8 个，不堆过多大卡片。
3. 成本/收入结构图要与表格相互印证。
4. 数据质量提示要清楚，但不能喧宾夺主。
5. 所有金额使用统一格式：`¥12,345.67` 或 `¥12,345`，全站保持一致。
6. 净利率、毛利率等百分比保留 1 位小数。
7. 风险状态使用文字 + 颜色，不只靠颜色区分。
8. 移动窄屏下表格允许横向滚动，不挤压字段。

## 12. 关键风险与规避

| 风险 | 影响 | 规避 |
| --- | --- | --- |
| 当前工作区脏改过多 | 容易覆盖并行任务 | 开发前先拆分提交或新建隔离分支 |
| 利润口径争议 | 页面上线后老板不信数据 | 先冻结公式，页面显示口径说明和数据质量 |
| BOM 缺失 | 项目毛利不可用 | 缺 BOM 显示为成本缺失，不按 0 计算 |
| 预收误算成利润 | 严重误导经营 | 现金收入与经营收入分开展示 |
| 提成数据不完整 | 员工贡献毛利偏差 | 标记估算，MVP 支持月度人工成本手工录入 |
| 库存流水归因不完整 | 实际耗材成本偏差 | 优先使用 `StockMovement`，缺归因则回落标准 BOM 并提示 |
| 权限过宽 | 财务敏感数据泄露 | 店长/财务/美容师分权限，导出单独权限 |
| 后端聚合慢 | 看板加载慢 | 后端按时间和门店索引查询，必要时阶段 2 引入日报聚合表 |
