# Ami Glow H5 版本详细开发计划

日期：2026-07-04
对象：`packages/Ami-Glow-MiniApp` 的 H5 版本
建议新增工程：`packages/Ami-Glow-H5`
关联系统：`packages/server-v2/src/customer-app`、Ami Core 管理端、Ami Glow 小程序、`packages/marketing-h5`

## 1. 开发结论

Ami Glow H5 版本建议按“客户服务 H5”独立建设，不建议直接改造成现有 `packages/marketing-h5`。

原因：

1. `packages/marketing-h5` 当前定位是公开营销落地页，核心链路是活动页面展示、留资、点击和渠道归因。
2. `packages/Ami-Glow-MiniApp` 定位是客户服务入口，核心链路是门店首页、项目预约、我的权益、消费记录、会员卡、AI 测肤和客户身份绑定。
3. 两者都面向客户，但业务闭环不同：营销 H5 负责“拉新转化”，Ami Glow H5 负责“到店服务和复访自助”。
4. 后端 `customer-app` 模块已经提供大部分业务接口，H5 版的主要工作不是重建后端，而是补齐 Web 端登录、路由、适配、埋点和部署。

建议第一期交付一个可独立部署的移动 H5 应用：

```text
packages/Ami-Glow-H5
├─ src
│  ├─ app
│  ├─ pages
│  ├─ components
│  ├─ services
│  ├─ stores
│  ├─ types
│  └─ utils
├─ index.html
├─ package.json
├─ tsconfig.json
└─ vite.config.ts
```

第一期目标不是做“网页小程序壳”，而是让顾客通过微信链接、短信链接、公众号菜单、二维码或门店物料打开 H5 后，完成：

```text
打开 Ami Glow H5 -> 浏览门店与项目 -> 绑定手机号 -> 预约服务 -> 查看我的权益/预约/消费记录 -> AI 测肤 -> 管理端承接预约和行为事件
```

| 对象 | 当前定位 | H5 版本处理 |

### 3.2 P1 延后

| 延后项 | 原因 |
| --- | --- |
| 微信网页 OAuth 正式登录 | 需要公众号/开放平台配置，先用手机号绑定闭环 |

| PWA 离线能力 | 客户服务场景不强依赖离线，先保证加载速度和兼容性 |

## 4. 技术方案

### 4.1 工程选择

建议新增 `packages/Ami-Glow-H5`，采用：

| 技术 | 建议 |
| --- | --- |
| 构建 | Vite |
| 框架 | React 18 + TypeScript |
| 路由 | React Router |
| 状态 | 轻量 Zustand 或 React Context，优先少依赖 |
| 样式 | CSS Modules 或普通 CSS，保持移动端主题统一 |
| 图标 | `lucide-react`，与 `marketing-h5` 保持一致 |
| 请求 | 基于 `fetch` 封装 `customer-app` API client |
| 环境变量 | `VITE_API_BASE_URL`，默认 `/api` |
| 部署 | 独立静态产物，可走 Zeabur/Vercel/Nginx 静态托管 |

不建议第一期使用 Taro/uni-app 直接跨端复用小程序代码。现有小程序已经是原生 WXML/WXSS/TS，强行迁移会把主要成本花在框架适配上，反而拖慢 H5 交付。

### 4.2 复用策略

| 层级 | 复用方式 | 说明 |
| --- | --- | --- |
| 业务接口 | 复用 `/customer-app/*` | 大部分接口直接复用 |
| 类型 | 从小程序 `services/types.ts` 迁移到 H5 `src/types/customer-app.ts`，后续可抽共享包 | 第一阶段避免跨包复杂引用 |
| 页面结构 | 参考小程序页面，但用 React 重写 | WXML/WXSS 不直接复用 |
| 视觉风格 | 复用 Ami Glow 粉紫、卡片、服务导向风格 | H5 要兼顾微信浏览器和普通浏览器 |
| 埋点口径 | 复用 `CustomerAppEvent` | H5 事件 `source=ami_glow_h5` 或 `channel=h5` |
| 后端鉴权 | 复用 CustomerAppAuthGuard | 需要补 Web 登录/匿名会话兼容 |

### 4.3 登录与身份方案

H5 版登录建议分三层交付。

| 阶段 | 登录方式 | 交付判断 |
| --- | --- | --- |
| P0 | 手机号 + 姓名绑定，必要时先用后端 Web guest token | 不依赖微信开放平台，最快打通业务闭环 |
| P0 增强 | 微信内浏览器读取 `code` 后换取 unionid/openid | 需要公众号/开放平台配置 |
| P1 | 短信验证码、企业微信/公众号授权、已有会员一键绑定 | 增强安全和转化 |

后端需要评估是否新增：

```text
POST /customer-app/auth/h5-guest
POST /customer-app/auth/h5-bind-phone
POST /customer-app/auth/wechat-oauth-login
```

如果现有 `bind-phone` 强依赖 `CustomerAppAuthGuard` 和 openid，则 H5 必须先拿到 guest token；如果后端允许手机号绑定时创建 H5 identity，则可以不新增 guest 接口。

### 4.5 API client 设计

H5 新增：

```text
packages/Ami-Glow-H5/src/services
├─ request.ts
├─ auth.ts
├─ home.ts
├─ project.ts
├─ reservation.ts
├─ mine.ts
├─ promotion.ts
├─ skin-test.ts
└─ event.ts
```

请求规范：

1. `baseURL = VITE_API_BASE_URL || '/api'`。
2. token 存储在 `localStorage`，key 建议 `ami_glow_h5_token`。
3. 门店上下文存储在 `ami_glow_h5_store_id`。
4. 请求自动附加 `Authorization: Bearer <token>`。
5. 请求自动附加 `X-Store-Id`，方便后端和管理端排查。
6. 错误统一转成客户可读文案，不把后端堆栈或技术字段展示给顾客。
7. 401 自动引导登录绑定，游客可看的页面不强制跳转。

## 5. 阶段计划

按 1 名前端、0.5 名后端、0.5 名测试估算，一期建议 4-5 周完成可验收版本；如果要同时做正式微信网页 OAuth 和短信验证码，增加 1 周。

| 阶段 | 周期 | 目标 | 主要产出 |
| --- | --- | --- | --- |
| 阶段 0：确认与基线 | 1-2 天 | 冻结 H5 范围、登录策略、部署域名和验收数据 | 接口差异清单、H5 工程决策 |
| 阶段 1：工程骨架与公共能力 | 3-4 天 | H5 应用可启动，路由、主题、请求、埋点打底 | `packages/Ami-Glow-H5` 骨架 |
| 阶段 2：首页与项目浏览 | 4-5 天 | 顾客可浏览门店首页、项目列表、项目详情 | 首页/项目闭环 |
| 阶段 3：登录绑定与预约闭环 | 5-7 天 | 顾客可绑定身份、选择时段并提交预约 | 预约管理端可见 |
| 阶段 4：我的服务与权益 | 4-5 天 | 顾客可查预约、次卡、消费记录、会员卡、领权益 | 会员服务闭环 |
| 阶段 5：AI 测肤与推荐 | 4-6 天 | 顾客可上传图片测肤并跳转推荐项目 | 测肤闭环 |
| 阶段 6：联调、部署与验收 | 3-5 天 | 完成兼容、性能、埋点、文档和发布准备 | 发布候选包 |

## 13. 风险与应对

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| H5 没有微信小程序 `wx.login` | 现有登录方式不能直接复用 | P0 走手机号/guest token，P1 接微信网页 OAuth |
| 现有 `bind-phone` 强依赖 CustomerAppAuthGuard | H5 无 token 时无法绑定 | 新增 `h5-guest` 或调整绑定接口 |
| H5 图片上传体积大 | 测肤慢、请求失败 | 前端压缩，后端限制大小并给客户可读错误 |
| 与营销 H5 混用 | 产品边界混乱，后期维护困难 | 独立 `Ami-Glow-H5`，只复用经验和共享类型 |
| 多端事件口径不统一 | 小程序/H5 效果无法比较 | 统一 `CustomerAppEvent`，用 source/channel 区分 |
| 预约并发 | 顾客看到可约但提交失败 | 后端最终校验，前端失败后提示刷新时段 |
| 历史客户手机号不完整 | 绑定后查不到会员权益 | 管理端准备测试客户和手机号治理清单 |
| 微信内置浏览器兼容 | 上传、复制、分享体验不同 | 真机验收列为发布门禁 |
| 首屏加载慢 | 顾客直接退出 | 图片懒加载、骨架屏、首页接口短缓存 |

### 14.1 产品验收

1. 顾客可通过 H5 链接进入首页。
2. 首页可展示当前门店、Banner、推荐项目和基础联系方式。
3. 顾客可浏览项目列表和项目详情。
4. 顾客可完成手机号绑定。

7. 顾客可上传图片完成 AI 测肤，并看到报告和推荐项目。
8. 顾客可领取权益或进入预约转化。
9. 管理端可查看 H5 行为事件。
10. 微信内置浏览器和普通手机浏览器核心流程均可用。

### 14.2 技术验收

1. `packages/Ami-Glow-H5` 构建通过。
2. H5 不保存任何 AI Key、管理端 JWT 或后端敏感配置。
3. 客户个人数据必须走 token 鉴权。
4. 我的预约、次卡、消费记录、会员卡不可越权访问。
5. 预约提交有幂等 key 和后端冲突校验。
6. 测肤上传有大小限制、隐私提示和失败重试。
7. H5 页面无长时间空白等待，关键页面有 skeleton/empty/error 状态。
8. 事件上报失败不影响主流程。
9. 生产 API 地址、静态资源路径和子路径部署可配置。

### 14.3 数据验收

1. H5 绑定手机号后能关联或创建客户。
2. H5 预约能写入真实预约表，并带渠道来源。
3. H5 行为能写入 `CustomerAppEvent`。
4. H5 测肤能关联客户档案或测肤报告。
5. 首页内容按 `storeId` 隔离，不串门店。
6. 权益领取能关联活动、客户和渠道。
