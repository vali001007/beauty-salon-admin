# 终端与管理端业务接口收敛详细开发计划

更新时间：2026-06-26

## 1. 背景与目标

本计划承接“次卡开卡”已经完成的收敛方式：终端和管理端保留各自鉴权入口，但收敛到同一份业务数据契约和同一套后端写入逻辑。

本轮目标覆盖四类已经存在双入口的高频业务：

1. 次卡核销
2. 会员充值/储值充值
3. 收银：管理端商品订单、管理端项目订单、智能终端收银
4. 登记：终端新客登记、管理端新建客户

核心目标不是把所有 URL 强行合并成一个，而是让两个入口生成同一种业务事实：

- 同一业务字段：金额、项目、员工、客户、支付方式、优惠、赠送、备注。
- 同一写库路径：同类业务只由一个 service 决定如何写入主表、明细表和关联表。
- 同一财务副作用：提成、日结、库存消耗、客户画像、余额账户等不再按入口分叉。
- 同一验收口径：从终端和管理端各做一笔业务，后台订单、提成、日结、库存、客户档案表现一致。

## 2. 总体架构原则

### 2.1 保留不同鉴权入口

终端入口继续使用设备鉴权：

- `/terminal/cards/consume`
- `/terminal/recharge-orders`
- `/terminal/cashier/checkout`
- `/terminal/customers/quick-create`

管理端入口继续使用后台用户鉴权：

- `/cards/verify-usage`、`/cards/usage`
- `/orders/member-cards/open`
- `/orders/member-cards/:id/recharge`
- `/orders/product`
- `/orders/project`
- `/customers`

原因：

- 终端鉴权依赖 `DeviceAuthGuard`，请求上下文带 `storeId/deviceId/userId`。
- 管理端鉴权依赖 `JwtAuthGuard` 和权限码，用户权限、门店选择和审计方式不同。
- 合并 URL 会扩大权限边界，收益不如共享 payload 和 service。

### 2.2 收敛到共享业务契约

前端类型上，管理端 payload 尽量复用或兼容终端 payload：

- `TerminalCardUsageVerifyRequest` -> 管理端次卡核销 payload
- `TerminalRechargeOrderCreateRequest` -> 管理端会员卡充值/open payload
- `TerminalCashierOrderCreateRequest` -> 管理端商品/项目订单 payload
- `TerminalQuickCreateCustomerRequest` -> 管理端客户创建 payload 的子集

后端 DTO 上，新增共享的内部输入类型，不要求一开始暴露成同一个 controller DTO：

- `CreateCardUsageInput`
- `CreateRechargeInput`
- `CreateCheckoutOrderInput`
- `CreateCustomerInput`

### 2.3 收敛到共享 service

推荐后端服务边界：

| 业务 | 推荐权威 service | 说明 |
| --- | --- | --- |
| 次卡核销 | `CardsService.verifyCardUsage` | 已具备项目配置校验、项目剩余次数、确认收入、耗材、提成基础能力 |
| 充值/储值 | `OrdersService.createMemberCardRecharge` 或抽出 `RechargeService` | 统一账户、余额流水、订单项、支付、提成、日结 |
| 收银写单 | `OrdersService.createProductOrder` 继续作为主写单，或抽出 `CheckoutOrderService` | 商品/项目/混合订单统一走 `ProductOrder + OrderItem` |
| 新建客户 | `CustomersService.create` | 管理端已有健康档案同步逻辑，终端应复用 |

终端 service 只保留终端特有的上下文补齐：

- 注入 `storeId`
- 注入 `deviceId`
- 注入当前终端操作人 `operatorId`
- 校验收银班次
- 设置 `source = terminal`
- 刷新终端缓存
- 返回终端 UI 需要的轻量结构

## 3. 当前不一致盘点

### 3.1 次卡核销

当前状态：

- 管理端入口：`CardsController.verifyUsage` -> `CardsService.verifyCardUsage`
- 终端入口：`TerminalCardController.consume` -> `TerminalService.consumeCard`

主要不一致：

1. 终端和管理端各自更新 `CustomerCard.remainingTimes`。
2. 终端和管理端各自创建 `CardUsageRecord`。
3. 管理端有更完整的卡内项目校验：
   - 按 `card.projects` 找项目名。
   - 校验项目是否属于当前次卡。
   - 校验该项目剩余次数。
   - 解析真实 `Project.id`。
4. 终端只按 `projectId` 查项目，弱化了“项目是否属于该次卡”的校验。
5. 两边各自处理耗材和提成，后续容易再次出现提成、库存或项目名不同步。

产品影响：

- 同样是核销，终端和管理端可能对“是否可核销”给出不同判断。
- 终端核销生成的项目、提成和耗材归属可能弱于管理端。
- 财务、员工人效、项目毛利会因为入口不同产生可信度差异。

### 3.2 充值/会员储值

当前状态：

- 终端入口：`POST /terminal/recharge-orders`
- 管理端入口：
  - `POST /orders/member-cards/open`
  - `POST /orders/member-cards/:id/recharge`

主要不一致：

1. 终端 payload 使用 `amount`，管理端使用 `rechargeAmount`。
2. 终端 `TerminalService.createRechargeOrder` 自己写 `ProductOrder`、余额账户、余额流水、`OrderItem`、支付记录。
3. 管理端 `OrdersService.createMemberCardRecharge` 也自己写同类数据。
4. 终端充值明确执行提成计算和日结刷新。
5. 管理端充值当前更偏账户交易写入，提成和日结刷新需要补齐或统一到共享逻辑。

产品影响：

- 终端充值和管理端充值对财务结算、提成明细的表现可能不一致。
- 后续修改充值赠送、赠送项目、支付方式、余额交易口径时容易只改一边。

### 3.3 收银：商品/项目订单

当前状态：

- 终端入口：`POST /terminal/cashier/checkout`
- 管理端商品订单：`POST /orders/product`
- 管理端项目订单：`POST /orders/project`

当前已较接近的点：

- 都写入 `ProductOrder`。
- 都写入 `OrderItem`。
- 都支持 `project/product` 明细。
- 都支持优惠分摊字段：
  - `listAmount`
  - `itemDiscountAmount`
  - `orderAllocatedDiscountAmount`
  - `totalDiscountAmount`
  - `netAmount`
  - `discountSource`
  - `allocationMethod`
  - `discountPayload`
- 都支持服务员工 `beauticianId/beauticianName`。

主要不一致：

1. 终端 `TerminalService.checkout` 自己建单，管理端 `OrdersService.createProductOrder` 自己建单。
2. 终端有收银班次 guard，管理端没有。
3. 终端支持混合购物车后按 `orderKind` 拆单；管理端商品订单和项目订单是两个页面入口。
4. 两边库存、BOM、提成、日结副作用分散在两套 service。

产品影响：

- 字段看起来一致，但代码事实源仍是两套。
- 后续补退款、余额支付、券核销、营销归因或提成规则时存在漏改风险。

### 3.4 登记/新建客户

当前状态：

- 终端入口：`POST /terminal/customers/quick-create`
- 管理端入口：`POST /customers`

主要不一致：

1. 终端 `TerminalService.quickCreateCustomer` 直接 `customer.create`。
2. 管理端 `CustomersService.create` 会同步创建 `CustomerHealthProfile`。
3. 终端登记表单有肤质、皮肤状态、面部检测上下文，但基础 quick-create 没有复用管理端健康档案写入逻辑。

产品影响：

- 终端登记的新客可能缺健康档案，影响客户画像、营销推荐、健康档案页。
- “登记 + 面部检测”在产品上是完整建档，但数据上可能拆散。

## 4. 分阶段开发计划

### 阶段 0：契约冻结与测试基线

目标：

1. 新增或整理共享输入类型。

   - `packages/server-v2/src/orders`：定义 `CreateRechargeInput`、`CreateCheckoutOrderInput`。
   - `packages/server-v2/src/customers`：定义 `CreateCustomerInput`。
2. 梳理前端 payload 类型。
   - `src/types/terminal.ts`

   - `src/types/customer.ts`
   - `src/api/real/terminal.ts`
   - `src/api/real/order.ts`
   - `src/api/real/customer.ts`
3. 补现状测试，先证明当前两边入口能创建业务数据。
   - 终端核销测试。
   - 管理端核销测试。
   - 终端充值测试。
   - 管理端充值测试。
   - 终端客户 quick-create 测试。
   - 管理端客户 create 测试。

交付物：

- 共享输入类型草案。
- 当前行为测试。
- 明确哪些测试在阶段 1-4 会改预期。

### 阶段 1：次卡核销收敛

优先级：P0

原因：

次卡核销直接影响履约收入、项目毛利、服务提成、耗材扣减和客户剩余次数，入口不一致会直接损害财务可信度。

目标：

终端和管理端都通过 `CardsService.verifyCardUsage` 完成最终核销。

后端改造：

1. 扩展 `CardsService.verifyCardUsage` 输入：
   - 支持 `projectId`。
   - 支持 `projectName`。
   - 支持 `deviceId`。
   - 支持 `operatorId`。
   - 支持 `beauticianId`。
   - 支持 `times/consumedTimes`。
2. 调整 `TerminalService.consumeCard`：
   - 保留终端校验和返回格式。
   - 不再自己更新卡次数、不再自己创建 `CardUsageRecord`。
   - 调用 `CardsService.verifyCardUsage`。
   - 注入 `deviceId`、终端 `operatorId`。
   - 调用后刷新终端卡项、库存、客户缓存。
3. 调整 `TerminalCardController.consume`：
   - DTO 保持兼容。
   - 让业务进入共享核销 service。
4. 确认管理端 `CardsController.verifyUsage` 不变或只做轻微 DTO 补齐。

前端改造：

1. 终端核销 payload 保持现有字段，但补齐：
   - `beauticianId`
   - `operatorId`
   - `projectName` 或确保后端可由 `projectId` 解析。
2. 管理端次卡核销若存在与终端不同的服务人员字段，统一到 `beauticianId`。

验收口径：

1. 终端核销一笔：
   - `CustomerCard.remainingTimes` 正确减少。
   - `CardUsageRecord` 有 `storeId/customerId/customerCardId/cardId/projectId/projectName/recognizedAmount/beauticianId/operatorId/deviceId`。
   - 命中项目提成规则时生成 `CommissionRecord.sourceType=card_usage`。
   - BOM 耗材生成 `StockMovement.sourceType=card_usage`。
2. 管理端核销一笔：
   - 字段和终端核销同口径。
3. 同一张次卡的同一项目剩余次数校验一致。
4. 不能核销非本卡配置项目。

建议测试：

```powershell
npm.cmd --prefix packages/server-v2 test -- cards.service.spec.ts terminal.service.spec.ts commission.service.spec.ts --runInBand
npm.cmd --prefix packages/server-v2 run build
npm.cmd --prefix packages/Ami-Aura-Lite-Kiosk run build
```

执行记录（2026-06-26）：

- 已完成代码收敛：`TerminalService.consumeCard` 已改为调用 `CardsService.verifyCardUsage`，终端入口不再直接更新 `CustomerCard` 或创建 `CardUsageRecord`。
- 已补齐统一核销能力：`CardsService.verifyCardUsage` 支持终端传入 `projectId`，并统一校验客户归属、卡有效期、卡内项目、项目剩余次数、履约收入、BOM 耗材、服务人员提成。
- 已完成编译级验证：后端核销定向测试、后端构建、终端构建和管理端构建均通过。
- 待真实数据验收：仍需在联调环境从终端和管理端各核销一笔，核对 `CustomerCard`、`CardUsageRecord`、`CommissionRecord`、`StockMovement` 的真实记录一致性。

### 阶段 2：客户登记收敛

优先级：P0

原因：

终端登记是高频前台动作，如果只写客户主表、不写健康档案，会让客户画像、皮肤档案、营销推荐断链。

目标：

终端 quick-create 和管理端 create 都走 `CustomersService.create` 的统一建档逻辑。

后端改造：

1. 扩展 `CreateCustomerDto` 或新增内部 `CreateCustomerInput`：
   - `storeId`
   - `name`
   - `phone`
   - `gender`
   - `birthday`
   - `memberLevel`
   - `source`
   - `skinCondition`
   - `skinType`
   - `tags`
   - `remark`
2. 调整 `TerminalService.quickCreateCustomer`：
   - 不再直接 `this.prisma.customer.create`。
   - 注入 `storeId`。
   - 默认 `source = terminal`。
   - 调用 `CustomersService.create`。
3. 确保 `CustomersService.create` 对终端简化字段友好：
   - `phone` 可选。
   - `gender` 可选或默认。
   - `skinType/skinCondition` 能落到 `CustomerHealthProfile`。
4. 如果终端登记后会继续创建皮肤检测记录，保留现有 `createTerminalSkinTest`，但基础健康档案必须先存在。

前端改造：

1. `TerminalQuickCreateCustomerRequest` 与 `CustomerCreatePayload` 做子集收敛。
2. 终端登记继续保持轻量表单，不要求暴露管理端所有字段。
3. 管理端客户新建不需要改 UI，只需要确认 payload 兼容。

验收口径：

1. 终端登记新客户后：
   - `Customer` 存在。
   - `CustomerHealthProfile` 存在。
   - `skinType/skinStatus/allergyHistory` 等基础字段可解释。
2. 管理端新建客户后：
   - 行为不退化。
3. 客户画像、健康档案页面能看到终端新客。
4. 终端登记 + 面部检测时，客户基础档案和检测记录能串起来。

### 阶段 3：充值/会员储值收敛

优先级：P1

原因：

充值属于现金流和预收负债入口，必须保证终端和管理端对账户、流水、提成、日结的处理一致。

目标：

统一充值 payload 和后端写入逻辑。终端充值、管理端会员卡开卡、管理端会员卡充值共享同一个充值处理函数。

建议契约：

```ts
type CreateRechargeInput = {
  customerId: number;
  customerName?: string;
  customerPhone?: string;
  storeId: number;
  amount?: number;
  rechargeAmount?: number;
  giftAmount?: number;
  discountAmount?: number;
  giftProjects?: string[];
  paymentMethod?: string;
  transactionNo?: string;
  beauticianId?: number;
  operatorId?: number;
  source?: 'admin' | 'terminal';
  type?: 'open' | 'recharge';
  remark?: string;
};
```

后端改造：

1. 抽出充值共享函数：
   - 优先在 `OrdersService` 内把 `createMemberCardRecharge` 改造成可被终端调用的公开/内部方法。
   - 或新增 `RechargeService`，由 `OrdersService` 和 `TerminalService` 共同注入。
2. 统一字段：
   - `amount` 和 `rechargeAmount` 统一归一化为 `rechargeAmount`。
   - `discountAmount` 仅作为兼容字段，最终归入 `giftAmount` 或明确命名为优惠/赠送金额。
   - `giftProjects` 始终写入 `ProductOrder.items`、`OrderItem.payload`、余额交易 remark 或 payload。
3. 统一写入：
   - `CustomerBalanceAccount`
   - `CustomerBalanceTransaction`
   - `ProductOrder`
   - `OrderItem`
   - `PaymentRecord`
4. 统一副作用：
   - 营销归因。
   - 充值提成。
   - 日结刷新。
   - 客户消费统计。
5. 保留来源：
   - 终端传 `source=terminal`。
   - 管理端传 `source=admin`。

前端改造：

1. 管理端 `MemberCardRechargePayload` 兼容 `TerminalRechargeOrderCreateRequest`。
2. 终端充值不需要改 UI，只需要 payload 保持现状。
3. 管理端会员卡开卡和充值可以继续不同入口，但提交字段应统一。

验收口径：

1. 终端充值一笔和管理端充值一笔：
   - 都有余额账户更新。
   - 都有余额流水。
   - 都有 `ProductOrder.orderKind` 可识别为充值。
   - 都有 `OrderItem.itemType=recharge`。
   - 都有 `PaymentRecord`。
   - 都刷新日结。
   - 命中充值提成规则时都生成提成。
2. 财务日结中充值收入一致。
3. 会员卡账户页能看到两类充值流水。

### 阶段 4：收银写单收敛

优先级：P1

原因：

## 7. 风险与控制

### 7.1 权限风险

风险：

把终端和管理端合成同一 URL 会混淆设备权限和后台用户权限。

控制：

- 保留不同 controller。
- 只共享 service 和输入契约。
- 终端 controller 只能注入设备上下文，不能接受任意 `storeId` 覆盖。

### 7.2 财务重复入账风险

风险：

收敛过程中如果旧逻辑和新逻辑同时执行，可能重复写 `PaymentRecord`、`CommissionRecord` 或 `StockMovement`。

控制：

- 每个阶段先选一个权威 service。
- 旧逻辑改为调用权威 service，而不是在调用后继续写同类数据。
- 对库存和提成保留幂等键：
  - `StockMovement.sourceType + sourceId + movementType`
  - `CommissionRecord.sourceType + sourceId` 或业务唯一条件

### 7.3 历史数据兼容风险

风险：

历史记录字段不完整，统一后新逻辑依赖新字段可能导致旧详情页异常。

控制：

- 序列化层保留 fallback。
- 新写入强约束，旧读取兼容。
- 不强制迁移历史数据，除非财务报表必须修复。

### 7.4 大范围改动风险

风险：

收银写单涉及范围大，一次性抽 service 容易引入回归。

控制：

- 先核销、登记、充值。
- 收银最后做。
- 收银阶段先让终端复用 `OrdersService.createProductOrder` 的核心能力，稳定后再抽 `CheckoutOrderService`。

## 9. 最终完成标准

技术完成：

1. 终端和管理端每类业务至少有一套共享 service。
2. 终端和管理端 payload 类型兼容同一业务契约。
3. 不再存在两套独立写同一业务事实的核心代码。
4. 定向测试覆盖终端和管理端两种入口。

产品完成：

1. 前台在终端办业务，管理端能用同一口径查看。
2. 管理端补录业务，财务、提成、库存、客户画像不会少数据。
3. 业务数据按入口可追踪，但财务事实不按入口分叉。
4. 用户用“最新一条核销/充值/收银/登记记录”检查时，链路能解释清楚。

财务完成：

1. `PaymentRecord` 是支付事实。
2. `ProductOrder + OrderItem` 是交易事实。
3. `CustomerBalanceTransaction` 是储值余额事实。
4. `CardUsageRecord` 是次卡履约事实。
5. `CommissionRecord` 能按订单项、充值或核销来源追溯。
6. 日结能在终端和管理端业务后及时刷新。

## 10. 不做事项

本计划不建议在同一轮做以下事情：

1. 不强行把终端和管理端合成同一个 URL。
2. 不重做权限体系。
3. 不重构所有订单页面 UI。
4. 不批量清洗历史数据。
5. 不改变“办卡/充值是现金流和预收，核销/余额消费才是履约收入”的财务口径。
