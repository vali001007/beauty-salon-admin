# 数据治理与经营闭环详细开发计划

更新时间：2026-06-01  
依据文档：`docs/数据现状分析-主数据清单-数据流程图-数据图谱.md`  
适用范围：Ami_Core 管理端、`packages/server-v2` 后端、Ami Aura Lite 终端、AI/营销链路

## 1. 项目目标

本计划的核心目标是把当前“可演示的数据骨架”升级为“可联调、可对账、可分析、可持续运营的数据底座”。

从产品交付角度，最终要达到：

1. 管理端、终端、后端、接口文档的数据口径一致。
2. 核心主数据和经营事实有明确归属、约束和生命周期。
3. 库存、收银、次卡、服务消耗形成可追溯流水。
4. 营销触达与订单支付形成真实归因，ROI 不再只依赖预测或 mock。
5. AI、推荐、自动营销使用同一套客户、消费、健康档案和交易事实。
6. 演示数据与真实数据边界清晰，避免跨端演示口径不一致。

## 2. 总体策略

本次不建议一次性大重构。优先采用“主线收口 + 增量补表 + 兼容旧接口”的方式推进。

### 2.1 推进原则

| 原则 | 说明 |
| --- | --- |
| 先口径，后改表 | 先确认接口、数据字典、主模型事实源，再做迁移，避免反复返工 |
| 先新增，少破坏 | 新增流水表、明细表、归因表优先，旧 JSON 字段保留一段兼容期 |
| 先核心闭环，后治理增强 | 先打通库存、收银、营销归因，再做标签、Prompt 版本、BI 等增强 |
| Mock 与 Real 同步 | 每新增一个 real 能力，同步补 mock，保证演示和测试不断 |
| 小步迁移 | 每个阶段都能独立验收、回滚和继续演示 |

### 2.2 单一事实源约定

| 类型 | 单一事实源 |
| --- | --- |
| 数据库模型 | `packages/server-v2/prisma/schema.prisma` |
| 后端接口实现 | `packages/server-v2/src/**` |
| 管理端 API 契约 | `src/api/real/*.ts` 与 `docs/api-contract.md` |
| 终端 API 契约 | `src/api/real/terminal.ts`、`packages/server-v2/src/terminal/*`、`docs/terminal-api.md` |
| 演示 Mock 数据 | `src/api/mock/**`，后续尽量统一到共享 mock data factory |
| AI Gateway 与兼容入口 | `packages/server-v2/src/ai/**`，承接 AI Gateway 和 legacy `/v1/messages` 兼容，不再保留独立轻量网关 |

## 3. 里程碑规划

建议按 8 周推进，分 4 个双周迭代。若人力不足，可按阶段顺序拉长。

| 阶段 | 周期 | 目标 | 核心交付 |
| --- | --- | --- | --- |
| 阶段 0 | 第 0.5 周 | 口径冻结和开发准备 | 数据字典草案、接口差异清单、迁移策略确认 |
| 阶段 1 | 第 1-2 周 | P0 收口：接口文档和关键外键 | 终端/API 文档更新、Prisma 关键 relation、基础迁移 |
| 阶段 2 | 第 3-4 周 | 库存与订单流水闭环 | 库存流水、订单明细、支付流水、退款流水 |
| 阶段 3 | 第 5-6 周 | 营销归因与终端事件持久化 | 营销转化归因、推荐事件、促销配置、打印任务 |
| 阶段 4 | 第 7-8 周 | Mock 统一、测试和试运行 | 统一演示数据、回归测试、验收报告、上线检查清单 |

## 4. 阶段 0：口径冻结与准备

周期：0.5 周  
优先级：P0  
目标：避免直接开发表结构导致口径反复。

### 4.1 任务清单

| 任务 | 说明 | 产物 |
| --- | --- | --- |
| 确认主数据范围 | 锁定 Store、User、Role、Customer、Project、Product、Card、TerminalDevice 等第一批主数据 | 主数据范围确认表 |
| 确认经营事实范围 | 锁定 ConsumptionRecord、ProductOrder、CustomerCard、CardUsageRecord、Reservation、ServiceTask、SkinTest 等第一批事实表 | 经营事实范围确认表 |
| 接口差异复核 | 以 `src/api/real/terminal.ts` 和 controller 为准，列出终端文档旧路径 | 接口差异清单 |
| 数据迁移策略 | 明确新增 relation、流水表、明细表是否需要 backfill | 迁移策略说明 |
| 兼容策略 | 明确旧 JSON 字段保留周期，旧接口是否继续返回原结构 | 兼容策略说明 |

### 4.2 验收标准

- 产品、前端、后端对主数据和经营事实对象命名达成一致。
- 明确哪些字段保留兼容，哪些字段进入新表。
- 终端接口路径以当前实现为准，不再继续维护旧路径口径。

### 5.2 模块 B：Prisma 关键外键关系补齐

#### 开发任务

| 任务 | 涉及模型 | 说明 |
| --- | --- | --- |
| 美容师归属门店 | `Beautician.storeId -> Store.id` | 增加 relation，支持按门店联查 |
| 排班关系 | `Schedule.storeId -> Store.id`、`Schedule.beauticianId -> Beautician.id` | 增加 relation，支撑排班一致性 |
| 预约关系 | `Reservation.storeId/customerId/projectId/beauticianId` | 增加 Store、Customer、Project、Beautician relation |
| 订单关系 | `ProductOrder.customerId/storeId` | 增加 Customer、Store relation，客户可为空 |
| 核销关系 | `CardUsageRecord.customerId/beauticianId/deviceId` | 增加 Customer、Beautician、TerminalDevice relation |
| 服务任务关系 | `ServiceTask.customerId/beauticianId` | 增加 Customer、Beautician relation |
| 皮肤检测任务关系 | `SkinTest.taskId -> ServiceTask.id` | 增加服务任务 relation |
| 调拨门店关系 | `TransferOrder.fromStoreId/toStoreId -> Store.id` | 增加双门店 relation |
| AI 设备审计关系 | `AiAuditLog.deviceId -> TerminalDevice.id` | 增加终端设备 relation |

#### 技术要点

- 对历史脏数据要先做校验脚本，找出引用不存在的 ID。
- 允许为空的字段先用 `onDelete: SetNull`，强主关系再用 `Restrict` 或 `Cascade`。
- 软删除模型不应通过硬删除触发级联误删。
- 迁移前后要跑 `prisma generate`、后端 build、后端 test。

#### 验收标准

- Prisma migration 可在空库和 seed 后数据库上执行成功。
- 查询预约、排班、服务任务、订单、核销时可通过 relation include 取到关联对象。
- 原有前端页面不因返回字段变化而崩溃。
- `packages/server-v2 npm run build`、`npm run test` 通过。

## 6. 阶段 2：库存与订单流水闭环

周期：第 3-4 周  
优先级：P1  
目标：让库存和收银从“展示型数据”变成“经营可信数据”。

### 6.1 模块 D：库存流水 StockMovement

#### 建议新增模型

```prisma
model StockMovement {
  id              Int      @id @default(autoincrement())
  storeId         Int
  productId       Int
  batchId         Int?
  movementNo      String   @unique
  movementType    String
  quantity        Decimal
  beforeStock     Decimal?
  afterStock      Decimal?
  unit            String?
  sourceType      String?
  sourceId        Int?
  sourceNo        String?
  remark          String?
  operatorId      Int?
  occurredAt      DateTime @default(now())
  createdAt       DateTime @default(now())
}
```

#### movementType 建议枚举口径

| 类型 | 含义 |
| --- | --- |
| `purchase_in` | 采购入库 |
| `transfer_out` | 调拨出库 |
| `transfer_in` | 调拨入库 |
| `service_consume` | 服务耗材消耗 |
| `adjustment` | 人工盘点调整 |
| `loss` | 报损 |
| `return_in` | 退货入库 |

#### 开发任务

| 任务 | 涉及模块 | 说明 |
| --- | --- | --- |
| 新增 Prisma 模型和 migration | `packages/server-v2/prisma/schema.prisma` | 建库存流水表 |
| 入库写流水 | `inventory.service.ts` | `createInbound` 创建 StockBatch/更新 Product 后写 movement |
| 调拨写流水 | `inventory.service.ts` | 创建调拨时至少记录单据，确认调拨时写出入库流水。若当前没有确认流程，先设计状态 |
| 服务完成扣库存 | `terminal.service.ts` | `completeTask` 读取 `consumptionItems`，写 service_consume 流水并扣 Product/Batch |
| 库存查询优化 | `inventory.controller/service` | 支持按商品、门店、来源单据查询流水 |
| Mock 同步 | `src/api/mock/inventory.ts`、`src/api/mock/terminal.ts` | mock 中模拟库存扣减和流水 |

#### 产品影响

- 服务做完后，库存自动减少。
- 能追溯某个商品库存为什么变化。
- 后续可做耗材成本、项目毛利、异常消耗分析。

#### 验收标准

- 采购入库后生成 `purchase_in` 流水，商品库存增加。
- 服务完成提交耗材后生成 `service_consume` 流水，商品库存减少。
- 库存不足时，终端完成服务给出明确提示或允许超耗并标记异常，产品需二选一。
- 库存流水可按商品和时间倒序查询。
- 单测覆盖入库、服务消耗、库存不足。

- 充值后生成订单明细、支付流水和消费记录。
- 退款后订单状态、退款流水、前端展示一致。
- 原订单列表仍能展示历史 JSON 订单。

## 7. 阶段 3：营销归因与终端事件持久化

周期：第 5-6 周  
优先级：P1/P2  
目标：让“推荐和营销有没有带来收入”可以被真实追踪。

### 7.1 模块 F：营销归因闭环

#### 建议新增模型

```prisma
model MarketingAttribution {
  id             Int      @id @default(autoincrement())
  touchId        Int
  strategyId     Int
  executionId    Int?
  customerId     Int
  orderId        Int
  attributionType String
  attributedRevenue Decimal
  attributionWindowDays Int
  occurredAt     DateTime
  createdAt      DateTime @default(now())
}
```

#### 归因规则建议

| 规则 | 说明 |
| --- | --- |
| 客户匹配 | 订单 customerId 必须等于 touch customerId |
| 时间窗口 | 默认使用 touch.attributionWindowDays，当前建议 30 天 |
| 状态条件 | touch.status 为 reached 或 converted 均可归因 |
| 最近触达优先 | 同一客户多次触达时，优先归因最近一次有效触达 |
| 同策略去重 | 同一订单不要重复归因到同一 strategy |
| 金额口径 | 默认归因订单实付金额，退款后需要冲减 |

#### 开发任务

| 任务 | 涉及模块 | 说明 |
| --- | --- | --- |
| 新增归因表 | Prisma | 保存订单与营销触达关系 |
| 支付成功触发归因 | `terminal.service.ts`、`orders.service.ts` | ProductOrder 完成支付后查找有效 touch |
| 更新 touch 状态 | `MarketingAutomationTouch` | 命中归因后更新 convertedAt、conversionType、actualRevenue |
| 退款冲减 | `orders.service.ts` | 退款后更新 attribution revenue 或生成负向调整 |
| 效果看板改造 | `marketing.service.ts`、前端营销效果页 | 使用归因表/Touch actualRevenue 统计 |
| Mock 同步 | `src/api/mock/marketing.ts` | 将模拟转化逻辑调整为接近真实归因 |

#### 产品影响

- 自动营销不再只展示预测收入，可以展示真实转化收入。
- 能区分“触达人数、预计转化、实际成交、实际收入、ROI 偏差”。
- 后续能支持不同渠道、不同策略的运营复盘。

#### 验收标准

- 策略执行后产生 touch。
- 同一客户在归因窗口内下单，touch 状态变为 converted。
- 营销效果页实际收入来自订单归因。
- 订单退款后，营销实际收入同步冲减或标记。
- 支持按 strategyId 查看归因订单列表。

## 8. 阶段 4：Mock 统一、测试与试运行

### 8.1 模块 I：统一 Mock 数据工厂

#### 当前问题

当前至少有三类 Mock：

1. 管理端 Mock：`src/api/mock/*`
2. 管理端大样本 JSON：`src/api/mock/data/*`
3. 终端主线适配层：`packages/Ami-Aura-Lite-Kiosk/src/app/services/auraCoreService.ts`

跨端演示时可能出现客户、预约、卡项、库存不一致。

#### 开发任务

| 任务 | 建议位置 | 说明 |
| --- | --- | --- |
| 建 mock data factory | `src/api/mock/dataFactory.ts` 或 `packages/shared/mock-data` | 统一生成客户、项目、商品、卡项、预约、任务 |
| 终端 mock 改造 | `src/api/mock/terminal.ts`、`packages/Ami-Aura-Lite-Kiosk/src/app/services/auraCoreService.ts` | 优先复用同一批客户、项目、卡项 |
| 固定随机种子 | mock 工厂 | 保证每次演示数据稳定 |
| 导出 seed 数据 | 可选 | 用同一份 mock 生成 Prisma seed |

#### 验收标准

- 同一个客户在管理端、终端 mock 下名称、手机号、卡项、消费记录一致。
- 终端完成服务或核销后，mock 状态能在当前会话内反映。
- 单测不因随机数据波动失败。

1. 管理端登录，切换门店。
2. 新建客户，维护健康档案。
3. 新建服务项目，配置 BOM。

5. 终端登录，搜索客户。
6. 创建预约，签到后生成或关联服务任务。
7. 服务完成提交耗材，确认库存扣减。
8. 办卡并核销，确认客户卡剩余次数变化。
9. 收银支付，确认订单明细、支付流水、消费记录。
10. 执行自动营销，客户下单后确认归因和 ROI。

## 11. 数据迁移与兼容方案

### 11.1 迁移顺序

1. 先新增 relation 和索引，不删除旧字段。
2. 新增流水/明细表。
3. 新写入逻辑同时写旧 JSON 字段和新明细表。
4. 查询接口优先读新表，读不到时回退旧 JSON 字段。
5. 稳定一个版本后，再考虑清理旧字段或保留为快照。

### 11.2 Backfill 建议

| 数据 | Backfill 策略 |
| --- | --- |
| ProductOrder.items | 解析 JSON，生成 OrderItem |
| ProductOrder.payMethod/status | 生成 PaymentRecord，历史订单标记为 imported |
| CustomerCard | 保持原数据，不反推历史办卡订单 |
| CardUsageRecord | 可生成核销类 OrderItem 或保持独立事实 |
| StockBatch/Product.currentStock | 首期不反推出完整历史流水，只从上线后开始可信流水 |
| MarketingAutomationTouch | 历史 mock 转化不进入真实归因，只保留原记录 |

### 11.3 回滚策略

- migration 尽量只新增表和字段，降低回滚风险。
- 写入新表失败时，核心业务可先保留旧逻辑成功，但必须记录错误日志。
- 前端展示保留旧字段兜底。
- 每个阶段上线前导出关键表备份。

## 12. 权限、安全与审计要求

| 数据域 | 要求 |
| --- | --- |
| 客户联系方式 | 遵守 fieldScopes，手机号、微信、地址支持脱敏 |
| 健康档案和皮肤检测图片 | 只允许授权角色查看，AI 输入前做必要摘要和脱敏 |
| 订单与支付流水 | 限制财务/店长/管理员查看，退款需要审批扩展 |
| 库存流水 | 写入不可随意修改，调整类 movement 需要备注和操作人 |
| 营销触达 | 保留策略、渠道、触达时间、归因窗口，便于复盘 |
| AI 审计 | 记录 scenario、provider、model、token、状态，不保存完整敏感原文 |

## 14. 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 历史数据 ID 不一致 | migration 外键失败 | 先跑脏数据检查，允许部分字段 SetNull |
| JSON 明细解析不稳定 | backfill 失败 | 新逻辑双写，历史数据保留 JSON 兜底 |
| 库存扣减影响服务完成 | 终端操作卡住 | 产品确认库存不足策略：阻断、允许负库存、或异常标记 |
| 营销归因争议 | ROI 口径不统一 | 先采用最近触达和 30 天窗口，后续支持配置 |
| Mock 与 real 行为差异 | 演示和联调体验不一致 | 统一 mock factory，关键场景写单测 |
| 涉及面过大 | 交付延期 | 每阶段单独验收，不把 P2 混入 P0/P1 |

## 15. 建议优先级排序

如果只能先做一部分，建议顺序如下：

1. 终端接口文档收口。
2. Prisma 关键 relation 补齐。
3. 库存流水和服务耗材扣减。
4. 订单明细和支付流水。
5. 营销归因。
6. 推荐事件、促销、打印任务持久化。
7. Mock 数据统一。
8. 数据字典和导入模板治理增强。

这个顺序能优先解决联调和经营可信度问题，再解决运营增长和演示一致性问题。
