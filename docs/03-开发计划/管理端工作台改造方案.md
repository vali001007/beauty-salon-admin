# 管理端仪表盘改工作台改造方案

更新时间：2026-06-14  
目标范围：管理端首页 `/dashboard`、左侧菜单、角色化工作台数据、待办事项、快捷入口  
实施原则：先把“用户进来该看什么、该处理什么、该去哪操作”讲清楚；保留 `/dashboard` 路由兼容登录、错误页和历史链接，不做破坏性迁移。

## 1. 改造目标

把当前“仪表盘”从经营数据概览页升级为“工作台”：

1. 不再只是展示统一经营数据，而是按当前用户角色展示最该关注的数据。
2. 不再只是静态入口，而是展示今日最需要处理的待办事项。
3. 不再让用户从完整菜单里找功能，而是提供高频操作快捷入口。
4. 页面名称统一从“仪表盘 / 数据概览”改为“工作台 / 我的工作台”。

产品体验目标：

```text
登录进入系统 -> 看到自己的工作台 -> 先处理待办 -> 再通过快捷入口完成高频动作
```

## 2. 当前基础

### 2.1 已有前端基础

| 能力 | 当前文件 | 当前状态 |
| --- | --- | --- |
| 首页路由 | `src/app/routes.tsx` | `/` 和 `/dashboard` 已指向 `Dashboard` |
| 首页页面 | `src/app/pages/Dashboard.tsx` | 已有店长、前台、美容师三类静态配置 |
| 左侧菜单 | `src/app/components/Layout.tsx` | 仍显示“仪表盘 / 数据概览” |
| 权限目录 | `src/config/permissions.ts` | 已有 `core:dashboard:view` |
| 前端类型 | `src/types/dashboard.ts` | 已有 `DashboardOverview`、指标、优先事项类型 |
| 前端 API | `src/api/real/dashboard.ts` | 调用 `GET /dashboard/overview` |

### 2.2 已有后端基础

| 能力 | 当前文件 | 当前状态 |
| --- | --- | --- |
| Dashboard Controller | `packages/server-v2/src/dashboard/dashboard.controller.ts` | `GET /dashboard/overview`，要求 `core:dashboard:view` |
| Dashboard Service | `packages/server-v2/src/dashboard/dashboard.service.ts` | 按门店聚合客户、收入、库存、活动、预约、服务任务、终端 |
| 用户上下文能力 | `packages/server-v2/src/common/decorators/current-user.decorator.ts` | 已有 `@CurrentUser()` 可复用 |

### 2.3 当前主要问题

1. 页面名称还叫“仪表盘”，用户心智偏“看数据”，不是“处理工作”。
2. 后端 `GET /dashboard/overview` 只知道门店，不知道当前用户、角色和数据权限。
3. 当前待办是通用经营优先事项，没有按店长、前台、美容师、库存管理员分别排序。
4. 快捷入口主要由前端静态配置决定，未完全按权限过滤，可能出现“看得到但进不去”的体验。
5. 当前角色解析偏 Ami Aura Lite 的三角色模型，未显式覆盖 `super_admin`、`inventory_manager` 等管理端角色。

## 3. 目标角色与关注重点

### 3.1 角色范围

第一版建议覆盖这些管理端角色：

| 角色 | 展示名称 | 工作台定位 |
| --- | --- | --- |
| `super_admin` | 总部 / 超级管理员 | 看全局异常、跨门店趋势、系统配置和经营风险 |
| `store_manager` | 店长 | 看本店经营结果、服务进度、库存风险、营销增长机会 |
| `cashier` | 前台 / 收银 | 处理预约到店、客户登记、次卡核销、收银订单 |
| `beautician` | 美容师 | 处理自己的预约、待服务、服务记录、客户护理建议 |
| `inventory_manager` | 库存管理员 | 处理低库存、临期、采购、调拨和消耗异常 |

多角色用户默认进入优先级最高的工作台，并允许在工作台顶部切换当前视角：

```text
super_admin > store_manager > inventory_manager > cashier > beautician
```

### 3.2 角色内容矩阵

| 角色 | 最需要关注的数据 | 最需要处理的待办 | 高频快捷入口 |
| --- | --- | --- | --- |
| 总部 / 超级管理员 | 今日总营收、门店异常数、活跃门店、系统待配置项 | 低活跃门店、异常设备、权限/门店配置缺失、跨店库存风险 | 门店管理、角色权限、平台收入报表、设备管理 |
| 店长 | 今日收入、今日预约、待服务任务、库存预警、进行中活动 | 待确认预约、服务任务积压、低库存/临期、客户流失预警、活动效果异常 | 项目预约、排班管理、库存管理、营销工作台、日结报表 |
| 前台 / 收银 | 今日预约、待到店、待核销、今日收银、新客户登记 | 待确认预约、已到店未开单、待核销次卡、未完成收银、客户资料缺失 | 项目预约、客户登记、次卡核销、商品订单、次卡开卡 |
| 美容师 | 我的预约、待完成服务、待补服务记录、服务客户数、我的提成 | 即将开始服务、超时未完成服务、服务记录未补、护理建议未生成 | 我的排班、客户画像、服务记录、护理建议、提成明细 |
| 库存管理员 | 低库存数、临期批次数、采购待处理、调拨待确认、消耗异常 | 低于安全库存、30 天内临期、采购单待入库、调拨待确认、BOM 消耗异常 | 库存管理、采购管理、过期管理、门店调拨、服务消耗 |

## 4. 页面信息架构

### 4.1 页面结构

建议工作台分为 4 块：

```text
工作台头部
  - 当前角色、当前门店、数据更新时间、手动刷新、角色切换

关键数据
  - 每个角色 4 个核心指标卡
  - 指标卡可点击进入对应业务页面

今日待办
  - 按优先级展示 Top 5-8
  - 每条待办包含严重程度、业务对象、原因、主操作

快捷入口
  - 每个角色 4-8 个高频入口
  - 按权限过滤，不展示无权限入口

Ami 建议
  - 一句话结论 + 依据 + 推荐动作
  - 第一版可继续使用规则结论，后续再接 AI 生成
```

### 4.2 工作台头部

字段建议：

| 字段 | 说明 |
| --- | --- |
| 当前视角 | 例如“店长工作台”“前台工作台” |
| 当前门店 | 继续复用 `StoreSwitcher` 的当前门店 |
| 数据口径 | 全部门店 / 当前门店 / 本人 |
| 更新时间 | 后端 `generatedAt` |
| 刷新按钮 | 重新拉取工作台数据 |
| 角色切换 | 多角色用户可切换工作台视角 |

### 4.3 关键数据卡

指标卡需要具备：

| 字段 | 说明 |
| --- | --- |
| `key` | 指标唯一标识 |
| `label` | 展示名称 |
| `value` | 当前值 |
| `hint` | 同比/环比/解释 |
| `tone` | 颜色状态 |
| `path` | 点击跳转路径 |
| `permission` | 入口权限 |
| `severity` | normal / warning / critical |

店长示例：

```text
今日收入：¥12,800，较昨日 +8.5%
今日预约：16，待确认 3
待服务任务：5，超时 1
库存预警：8，低库存 5 / 临期 3
```

前台示例：

```text
今日预约：16，待到店 6
待核销：4，需前台处理
今日收银：¥8,600，未完成 2 单
新增登记：3，资料待补 1
```

美容师示例：

```text
我的预约：5，下一单 14:30
待完成服务：2，进行中 1
待补记录：3，影响客户画像
我的提成：¥420，本月累计
```

库存管理员示例：

```text
低库存：5，低于安全线
临期批次：3，30 天内到期
采购待处理：2，待下单/入库
调拨待确认：4，跨店处理中
```

### 4.4 今日待办

待办不建议第一版做成完整任务系统，先做“数据驱动的待办摘要”。来源可以是预约、订单、库存、服务任务、营销、设备等业务表。

待办字段建议：

```ts
interface WorkbenchTodo {
  id: string;
  type: 'reservation' | 'service' | 'inventory' | 'order' | 'marketing' | 'device' | 'customer' | 'finance';
  title: string;
  detail: string;
  tag: string;
  severity: 'critical' | 'warning' | 'normal';
  priority: number;
  count?: number;
  dueAt?: string;
  path: string;
  permission: string;
  primaryAction: string;
}
```

优先级建议：

```text
待办优先级 = 角色权重 + 严重程度 + 时间紧急度 + 数量影响 + 是否阻塞主流程
```

示例排序：

| 角色 | 排序靠前的待办 |
| --- | --- |
| 店长 | 库存预警、服务任务积压、今日预约异常、活动转化异常 |
| 前台 | 即将到店、已到店未开单、待核销、客户资料缺失 |
| 美容师 | 30 分钟内预约、进行中服务、待补服务记录、护理建议 |
| 库存管理员 | 低库存、临期、采购待处理、调拨待确认 |

### 4.5 快捷入口

快捷入口分两层：

1. 角色默认入口：按角色预设。
2. 权限过滤入口：只显示用户有权限访问的入口。

第一版不做用户自定义收藏，先使用角色配置。后续可以加“固定到工作台”和“最近访问”。

快捷入口字段建议：

```ts
interface WorkbenchQuickAction {
  key: string;
  label: string;
  path: string;
  icon: string;
  permission: string;
  group: 'operation' | 'management' | 'analytics' | 'system';
}
```

## 5. 前端改造方案

### 5.1 文案和菜单

修改点：

| 文件 | 改造内容 |
| --- | --- |
| `src/app/components/Layout.tsx` | 一级菜单“仪表盘”改“工作台”，子菜单“数据概览”改“我的工作台” |
| `src/config/permissions.ts` | `core:dashboard:view` 名称从“查看仪表盘”改为“查看工作台”，描述同步更新 |
| `src/i18n/locales/zh-CN/dashboard.json` | 如仍使用，文案同步为工作台 |
| `docs/02-产品设计/Admin Panel/*` | 后续产品文档同步，不作为第一版代码阻塞 |

路由策略：

| 路由 | 处理方式 |
| --- | --- |
| `/dashboard` | 保留，继续作为默认工作台 |
| `/` | 保留，继续跳工作台 |
| `/workbench` | 可选新增别名，不建议第一版强依赖 |

### 5.2 组件拆分

当前 `src/app/pages/Dashboard.tsx` 建议拆分为：

```text
src/app/pages/Dashboard.tsx
src/app/pages/workbench/workbenchConfig.ts
src/app/pages/workbench/WorkbenchHeader.tsx
src/app/pages/workbench/MetricCardGrid.tsx
src/app/pages/workbench/TodoList.tsx
src/app/pages/workbench/QuickActionGrid.tsx
src/app/pages/workbench/AmiInsightPanel.tsx
```

拆分收益：

1. 页面文件不继续变大。
2. 角色配置可以独立维护。
3. 后续新增总部、库存管理员视角时不影响主页面结构。

### 5.3 角色解析

现状 `resolveAuraRole(user)` 偏终端三角色。管理端工作台建议新增独立解析：

```ts
type AdminWorkbenchRole =
  | 'super_admin'
  | 'store_manager'
  | 'cashier'
  | 'beautician'
  | 'inventory_manager'
  | 'default';
```

解析逻辑：

1. 优先使用用户显式选择的工作台视角。
2. 其次按用户 `roles` 匹配管理端角色。
3. 再按权限推断可用视角。
4. 最后 fallback 到 `default`，只展示该用户有权限的快捷入口。

### 5.4 权限过滤

所有指标卡、待办、快捷入口都必须带 `permission` 字段。前端渲染前统一过滤：

```text
有权限 -> 展示并可点击
无权限 -> 不展示
无可用入口 -> 展示空状态，引导联系管理员开通权限
```

不能出现“工作台展示入口，点击后进入无权限页”的常态体验。

### 5.5 加载、失败和空状态

| 状态 | 处理 |
| --- | --- |
| 加载中 | 指标卡 skeleton，待办区域显示加载态 |
| 部分接口失败 | 保留可用数据，失败区域显示轻量提示 |
| 后端整体失败 | 展示“工作台数据暂不可用”，保留静态快捷入口 |
| 无待办 | 展示“当前无待办事项”，不要空白 |
| 无权限入口 | 展示“暂无可用快捷入口” |

## 6. 后端改造方案

### 6.1 接口策略

建议新增工作台语义接口，同时保留原接口：

| 接口 | 用途 |
| --- | --- |
| `GET /dashboard/overview` | 保留兼容，旧概览能力继续可用 |
| `GET /dashboard/workbench` | 新工作台接口，按用户、角色、门店返回数据 |

新接口入参：

| 参数 | 来源 | 说明 |
| --- | --- | --- |
| `x-store-id` | Header | 当前门店口径 |
| `role` | Query，可选 | 用户选择的工作台视角 |
| `@CurrentUser()` | JWT | 当前用户、角色、门店、权限 |

### 6.2 响应结构

```ts
interface WorkbenchOverview {
  actor: {
    userId: number;
    name: string;
    roles: string[];
    currentRole: AdminWorkbenchRole;
    availableRoles: AdminWorkbenchRole[];
  };
  scope: {
    storeId: number | null;
    storeName: string;
    mode: 'all' | 'store' | 'self';
  };
  metrics: WorkbenchMetric[];
  todos: WorkbenchTodo[];
  quickActions: WorkbenchQuickAction[];
  insight: {
    conclusion: string;
    basis: string;
    action: string;
    path: string;
    permission: string;
  };
  terminalStatus?: {
    totalDevices: number;
    onlineDevices: number;
  };
  generatedAt: string;
}
```

### 6.3 数据来源

| 模块 | 可贡献数据 |
| --- | --- |
| 客户 | 总客户、本月新增、待补资料、流失/复购机会 |
| 预约 | 今日预约、待确认、待到店、即将开始、取消/爽约 |
| 服务任务 | 待服务、进行中、超时、待补服务记录 |
| 订单 | 今日收银、未完成订单、退款/异常订单 |
| 会员卡 / 次卡 | 待核销、今日核销、开卡收入 |
| 库存 | 低库存、临期、采购待入库、调拨待确认、消耗异常 |
| 营销 | 进行中活动、自动触达、转化异常、推荐机会 |
| 财务 | 日结、提成、平台收入、月度结算 |
| 终端 | 设备在线、终端任务、设备异常 |

### 6.4 用户与门店数据权限

后端必须校验：

1. `x-store-id` 是否在当前用户可访问门店内。
2. 美容师视角只聚合本人相关预约、服务任务、服务客户。
3. 前台视角只聚合当前门店前台操作相关数据。
4. 库存管理员视角不返回客户隐私、订单利润等无关数据。
5. 超级管理员未选门店时可看全局汇总，但待办需要按门店聚合为异常列表。

### 6.5 性能策略

第一版继续使用 `Promise.all` 并行聚合，避免串行查询。后续按压力情况增加缓存：

| 数据 | 建议缓存 |
| --- | --- |
| 工作台主接口 | 15-30 秒，按 `userId + role + storeId` |
| 跨门店总部汇总 | 30-60 秒 |
| 设备在线状态 | 15 秒 |
| 库存预警 | 30-60 秒 |

不要为了工作台第一版新增复杂报表表。若后续数据量变大，再考虑日报快照表。

## 7. 分阶段交付计划

### 阶段 1：低风险改名和前端工作台化

目标：用户马上感知“仪表盘变工作台”，不大改后端。

任务：

1. 左侧菜单改为“工作台 / 我的工作台”。
2. `Dashboard.tsx` 页面标题统一改为“工作台”语义。
3. 增加总部、库存管理员、默认视角配置。
4. 快捷入口增加权限字段并按权限过滤。
5. 待办和指标卡保持兼容现有 `GET /dashboard/overview`，缺失字段用前端兜底。

验收：

1. 登录后默认进入 `/dashboard`，页面显示“工作台”。
2. 不同角色看到不同标题、关键数据、待办和快捷入口。
3. 无权限入口不展示。
4. 后端接口失败时，仍能看到快捷入口和错误提示，不白屏。

### 阶段 2：后端角色化工作台接口

目标：从“前端静态角色配置”升级为“后端按用户和角色返回数据”。

任务：

1. 新增 `GET /dashboard/workbench`。
2. `DashboardController` 引入 `@CurrentUser()`。
3. `DashboardService` 增加 `getWorkbench({ user, storeId, role })`。
4. 返回 `actor`、`metrics`、`todos`、`quickActions`、`insight`。
5. 补齐店长、前台、美容师、库存管理员、总部角色的聚合逻辑。
6. 保留 `GET /dashboard/overview` 兼容旧调用。

验收：

1. 切换门店后，工作台数据随门店刷新。
2. 切换角色视角后，指标和待办排序变化。
3. 美容师用户只看到本人相关预约和服务任务。
4. 库存管理员不看到客户隐私类入口。
5. 超级管理员可看全局口径和异常门店列表。

### 阶段 3：待办闭环和个性化

目标：让工作台从“提示”进一步变成“处理中心”。

任务：

1. 待办支持“已处理后刷新消失”。
2. 快捷入口增加最近访问和固定入口。
3. Ami 建议接入 AI Gateway，生成更像经营助手的建议。
4. 对接营销工作台、终端跟进任务、日结报表等闭环入口。

验收：

1. 用户处理预约、核销、库存等事项后，工作台待办数量刷新。
2. 用户固定入口后，下次登录保留。
3. Ami 建议能基于当前角色和门店口径生成可执行动作。

## 8. 建议优先做的 MVP

为了一次交付可见价值，同时控制风险，建议 MVP 做到：

1. 保留 `/dashboard`，只改显示名称为“工作台”。
2. 前端先拆出工作台配置和组件。
3. 增加 `super_admin`、`inventory_manager`、`default` 视角。
4. 指标、待办、快捷入口全部按权限过滤。
5. 继续复用 `GET /dashboard/overview`，后端只补必要字段。
6. 下一步再新增 `GET /dashboard/workbench` 做真正用户级聚合。

MVP 不做：

1. 不做完整任务中心。
2. 不做用户自定义拖拽。
3. 不新增复杂报表快照表。
4. 不删除 `/dashboard` 旧路由。

## 10. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 数据口径不统一 | 用户不信任工作台数据 | 文案明确“当前口径”，后端统一订单状态、预约状态、门店范围 |
| 待办过多 | 工作台变成杂乱列表 | 只展示 Top 5-8，其余进入对应业务页面 |
| 权限和入口不一致 | 点击后无权限 | 后端返回 permission，前端二次过滤 |
| 多角色用户不知道当前视角 | 误解数据 | 头部展示“当前视角”，提供视角切换 |
| 工作台接口查询重 | 首页变慢 | 并行查询、短缓存、分模块失败兜底 |
| 角色模型混用终端 Aura | 管理端角色覆盖不完整 | 新增管理端 `AdminWorkbenchRole`，不要直接复用终端三角色作为唯一来源 |

## 11. 推荐结论

建议按“两步走”：

1. 第一版先做前端工作台化和权限过滤，快速把用户心智从“看仪表盘”改为“处理工作”。
2. 第二版新增 `GET /dashboard/workbench`，把数据真正按当前用户、角色、门店、权限聚合。

这样既能快速交付可见效果，又不会一次性重构首页、权限、报表和任务系统，风险可控。
