## 4. 交叉依赖与错序风险

### 4.1 业务定义链路不能拆散执行

迁移 2、4、5、10 是同一业务定义/语义证据链：

```text
registry tables + triggers
  -> projection v2 functions
  -> projection v2 backfill
  -> semantic evidence pool
```

跳过任一前置会造成函数、外键或目标表不存在。迁移期间不得让应用在 registry 建表后、projection backfill 前写入新投影，否则不可变触发器和回填更新存在冲突风险。

### 4.2 营销 schema migration 不等于历史数据迁移

3 个营销 migration 负责建立新模型和字段，不会自动完成以下历史治理：

- 39,817 条门店预测快照仍关联 42 个全局预测批次；
- 当前营销机会关联全局预测批次数为 0；该项不再需要历史回填，但 39,817 条预测快照仍需治理；
- 14 条历史推荐活动和 16 条历史终端任务尚未建立稳定推荐实例/采纳关联；
- 460 条旧 `reached` 触达状态仍需按真实证据归一；
- 7 条旧 `completed` execution 仍需归一到统一执行状态；
- 两条 running execution 的排队人数分别为 1009 和 1008，但已生成 touch 只有 65 和 62，不能直接标记成功；
- effect fact 新表创建后仍为空，真实订单、退款和投递事实需要受控回填。

因此发布顺序必须是：schema migration → schema verify → 状态归一 dry-run → 推荐/预测 backfill dry-run → effect fact dry-run → 明确写入授权 → 分批 apply → 全链路验收。

### 4.3 财务迁移不能作为营销迁移的附带动作

财务 migration 会更新真实日结事实，即使当前只有 54 条记录，也需要独立的业务口径验收。营销负责人对营销迁移的授权不能自动视为财务数据回填授权。

### 4.4 应用版本与数据库版本必须保持前向兼容

当前后端 build 已通过，应用版本具备继续发布审查的基础。真实迁移时仍必须保持数据库与应用版本一致，发布门禁要求：

- 当前目标 commit 的 `server-v2 build` 通过；
- 新营销读写开关默认关闭；
- 当前线上版本在新增 nullable 字段和新表存在时仍可运行；
- schema、应用和回填脚本来自同一发布快照。
