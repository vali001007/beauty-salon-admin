## 1. 产品交付结果

本次改造把原来的“前端即时计算推荐 + 串行创建业务对象 + 创建即视为触达”收敛为一条稳定闭环：

```text
门店业务事实
  -> 门店预测批次
  -> 客户预测/生命周期快照
  -> 三类推荐 Provider
  -> 统一推荐编排和权益匹配
  -> 稳定 recommendationInstanceId
  -> 统一采纳事务
  -> 活动 / 自动策略 / 终端任务
  -> 分片 delivery job 和真实渠道对象
  -> 唯一主订单归因
  -> 退款负向事实
  -> 去重效果汇总和多维拆解
```

智能推荐页的“预测覆盖人数”和“单张推荐目标人数”已分开。页面不再把“某张卡命中 1 位客户”误写成“系统只覆盖 1 位客户”；覆盖率取门店预测批次，卡片人数取持久化受众快照。

## 2. 代码完成情况

### 门店预测和推荐实例

- 同门店、同业务日、同模型版本使用稳定 run key 幂等。
- 同进程内同门店、同业务日的并发刷新共享一次完整编排，失败后释放锁；跨进程并发继续由实例唯一键兜底。
- prediction、lifecycle、product_project 三类推荐统一进入编排器。
- 受众、权益、证据、执行模式和风险保存为不可变快照。
- 推荐列表、详情、受众和刷新接口全部要求 `X-Store-Id`。
- 生命周期、规则库、执行、效果、邀请候选、客户画像和旧兼容接口均完成门店边界复核；请求体或 query 中的 `storeId` 不能覆盖请求头门店。
- 营销页详情、编辑、发布/下线、复制、效果、事件、线索和归因接口全部强制使用 `X-Store-Id`；归因汇总的 query `storeId` 不再覆盖请求头，真实页面 8 在门店 6 返回 200、门店 1 返回 404。

### 统一采纳

- 活动模式在一个事务中创建采纳、活动、营销页和页面版本；发布失败整体回滚。
- 自动策略保留 predictionRunId、audienceSnapshotId 和 recommendationInstanceId；启用时不产生 touch。
- 终端跟进批量创建真实任务，并按推荐实例、客户和来源键去重。
- 旧采纳接口保留兼容转发和弃用日志；旧 `follow-up-tasks` 已改为调用采纳服务，不再绕过 adoption 直接创建孤立任务。
- 终端跟进部分失败、全部失败分别记录 `partial_failed`、`failed`；兼容响应返回创建、去重、失败数量和失败明细，负责人角色、用户和美容师分配继续透传。
- 旧列表、受众、增删改、采纳及三类草稿/任务接口共 10 个 handler 统一返回 `Deprecation`、`Sunset`、`Link`，并输出含路由模板、门店和 successor 的结构化 warning，支持 14 天零调用退役门禁。
- 营销工作台首屏统一调用单一 `GET /marketing/recommendation-workspace`。后端按当前门店灰度状态返回 `legacy` 或 `v2`，因此非灰度门店继续展示和执行旧推荐，灰度门店读取稳定推荐实例；首屏仍固定为 3 个主请求。
- V2 受众采用服务端 50 条真分页，终端跟进支持跨页选择，不再一次并发拉取全部客户。活动、自动策略、终端跟进在页面内按工作台模式分流到兼容接口或实例采纳接口。
- 新增日志导出审计门禁，严格校验连续 14 天覆盖、导出新鲜度、导出尾部连续性、内部覆盖分段无缺口、零旧调用和事件时间有效性，并按路由和门店汇总残余调用。
- 新增 Release C/D 只读观察门禁：可独立检查连续 3 个业务日 worker 稳定性、连续 7 天新旧效果口径逐日对账，或执行联合门禁。缺天、陈旧导出、覆盖缺口、stale execution、过期租约、重复投递对象、delivered 缺事实和任一指标差异均失败关闭。
- 旧采纳接口同样只使用 `X-Store-Id`，客户端传入的门店字段会被剥离。

### 执行和投递

- 受众的客户、预测、固定快照和频控查询全部按策略门店限定。
- 无受众快照的兼容路径仍按 500 条分片；持久化受众快照主路径改为数据库内一次原子 `INSERT ... SELECT`，一次创建 execution、1000 个 touch 和 1000 个 delivery job，避免跨区数据库逐批往返。
- Worker 每 10 秒领取最多 100 条，租约 60 秒，使用 `FOR UPDATE SKIP LOCKED` 与事务级 advisory lock 防止多实例重复领取；“领取 100 条”和“单渠道并发 10”已拆成两层，不再把并发上限误当作批量上限。
- 单店并发上限 20，单渠道并发上限 10；站内通知支持 100 条数据库批量适配与批量状态提交，临时错误按 1、5、20 分钟重试。
- 终端任务和站内通知使用 delivery job 幂等；短信、微信未配置时进入死信。
- 终端侧触发自动策略同样先创建真实 `TerminalFollowUpTask`，任务创建成功或命中幂等任务后才记为 `delivered`；创建失败写 `failed` 和错误信息，不再直接写历史 `reached`。
- 站内通知已增加 Customer App 客户+门店隔离读取；首次打开会以条件更新保证幂等，同时把关联自动触达从 `sent/delivered` 推进为 `opened`，并写入一条 `actual/open` 效果事实。Ami Glow H5 可在“我的 → 站内通知”真实查看，不再只停留在数据库记录。
- 自动营销创建终端任务时保存 recommendation、adoption、strategy、execution、delivery job 和 touch 关联；任务产生订单、预约、booked 或 converted 结果后，按门店+客户+投递任务校验来源并把原触达推进为 `converted`，普通 contacted 完成不伪造转化。
- `reachedCount` 只统计真实 delivered，失败和排队不计入触达。
- 部分投递失败时，仅给真实 delivered 客户登记权益领取事件和发放计数，失败客户不再被误记为已领券。
- 站内通知以 deliveryJobId 为唯一幂等键。真实停进程验收覆盖了“通知已创建、job 状态未提交”的崩溃窗口；重启后复用已存在通知，最终通知数与 delivery job 数均为 1000，没有重复触达。

### 归因和效果事实

- 页面 lead 与自动 touch 统一按最后触点选择一个主归因，其他来源只记辅助事实。
- `queued/failed/reached` 在数据库查询和服务层均被排除，不能产生收入归因。
- orders 和 terminal 在运行时复用同一个 `MarketingAttributionService`。
- Orders 和 Terminal 已删除各自的本地归因 fallback，订单创建不再在统一归因后重复执行旧营销页归因。
- 退款生成唯一负向收入事实，并冲减自动触达和页面兼容归因投影。
- 全部总览从事实表聚合一次；活动、推荐、权益、页面、策略和渠道只是同一事实的拆解维度。
- 收入标记 `actual`，预测值标记 `predicted`，固定单价成本标记 `estimated`。
- 通知打开和终端任务转化的效果事实均携带 recommendationInstanceId、adoptionId、strategyId、executionId、touchId 与 deliveryJobId，可直接按同一推荐实例复盘，不再按标题或任务备注反查。
- 兼容效果读取同样只统计 `sent/delivered/opened/clicked/converted`，不再把 queued、failed 或历史 reached 当作曝光。
- 生命周期归因重建同样排除 queued、failed 和历史 reached，防止无效触达重新污染事实层。
- 自动策略、规则、活动、推荐和权益等兼容汇总统一使用可归因触达状态；无触达证据时不再回退到 execution.reachedCount，也不再用 reachedCount 乘固定客单价伪造收入。
- 权益、规则模板、活动列表和 Ami Glow 展示事件均改为批量查询后内存分组，消除逐对象 N+1；活动列表 GET 只计算响应指标，不再隐式回写数据库。

## 7. 灰度和回滚

建议按以下顺序启用：

4. `MARKETING_EFFECT_FACT_WRITE=true` 双写至少 7 天，对比收入、转化、投递和退款净收入。
5. 对比通过后启用 `MARKETING_EFFECT_FACT_READ=true`。

回滚只关闭对应 feature flag，不删除推荐实例、delivery job 或效果事实，不把 delivered 改回 reached。
