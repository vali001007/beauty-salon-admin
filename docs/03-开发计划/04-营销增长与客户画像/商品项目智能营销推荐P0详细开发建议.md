# 商品与项目智能营销推荐 P0 详细开发建议

版本：v1.0
日期：2026-06-14
来源方案：`docs/02-产品设计/商品项目智能营销推荐方案.md`
适用范围：`packages/server-v2`、`src` 管理端、Ami Glow 小程序承接、Ami Aura Lite 终端跟进

## 1. P0 范围

本阶段只做能最快形成经营价值、且当前数据结构基本支撑的 4 类推荐：

| 优先级 | 推荐类型 | 推荐编码 | 目标 |
| --- | --- | --- | --- |
| P0-1 | 商品临期消化推荐 | `product_expiry_clearance` | 降低临期和过期损耗 |
| P0-2 | 美容师排期不满推荐 | `project_idle_capacity` | 把低峰空档转成预约和到店 |
| P0-3 | 商品补货复购推荐 | `product_replenishment` | 提升顾客商品复购 |
| P0-4 | 项目护理周期推荐 | `project_cycle_due` / 兼容 `care_cycle` | 提升项目复购和预约 |

P0 不建议一开始做复杂机器学习。先用规则引擎 + 可解释评分跑通推荐卡、活动草稿、自动营销草稿、终端跟进和效果记录。

## 2. 当前代码落点

### 2.1 后端已有基础

| 能力 | 当前落点 | 可复用方式 |
| --- | --- | --- |
| 智能推荐列表 | `packages/server-v2/src/marketing/marketing.controller.ts`、`marketing.service.ts` | 扩展 `GET /marketing/recommendations` 返回经营推荐卡 |
| 推荐卡结构化字段 | `MarketingService.buildRecommendationCard()` | 继续复用 `executionModes`、`offer`、`recommendedItems`、`recommendedChannels` |
| 活动草稿 | `POST /marketing/recommendations/:id/activity-draft` | P0 推荐卡可直接生成活动草稿 |
| 自动规则草稿 | `POST /marketing/recommendations/:id/automation-draft` | 商品补货、项目周期优先走自动规则 |
| 客户行为事件 | `CustomerBehaviorEvent`、`POST /marketing/customer-events` | 可用于浏览商品/项目、活动页意图 |
| 推荐事件 | `RecommendationEvent` | 记录采纳、跳过、转化、终端跟进 |
| 临期库存 | `InventoryService.getExpiring()`、`StockBatch` | 商品临期推荐核心数据 |
| 补货建议 | `InventoryService.getReplenishment()` | 商品复购和库存安全风控可复用 |
| 排班需求 | `SmartSchedulingService.demand()` | 排期不满推荐可复用现有排班/预约数据 |
| 项目 BOM | `ProjectBomItem` | 商品和项目关联推荐基础 |

### 2.2 数据结构注意点

- 商品销售建议优先读 `OrderItem`，用 `itemType='product'` 和 `itemId=productId` 精确统计。
- 项目销售/服务建议优先读 `OrderItem`，用 `itemType='project'` 和 `itemId=projectId` 精确统计。
- `ConsumptionRecord` 目前主要是文本型 `consumeContent`，可作为补充证据，不建议作为 P0 精确算法主数据。
- 服务消耗和项目 BOM 可结合 `StockMovement`、`ProjectBomItem` 判断商品消耗和项目关联。
- 推荐卡当前是虚拟卡片，不一定有持久化推荐表。P0 可先用稳定的虚拟 ID 段，例如 2000-2999，并在 `sourceSignals` / `payload` 中保留 `recommendationKey`。

## 3. 总体开发建议

### 3.1 建议新增后端服务

不要继续把所有逻辑都堆进 `MarketingService`。建议在营销模块内新增一个经营推荐服务：

```text
packages/server-v2/src/marketing/product-project-recommendation.service.ts
packages/server-v2/src/marketing/dto/product-project-recommendation.dto.ts
```

职责：

- 读取商品、库存、订单、项目、排班、预约、客户预测数据。
- 生成 P0 四类经营推荐卡。
- 提供推荐受众计算。
- 输出兼容现有前端 `Recommendation` 的结构。

`MarketingService` 保持总入口职责：

```ts
async getRecommendations(storeId?: number, scope?: 'all' | 'customer' | 'product-project') {
  const customerCards = scope !== 'product-project' ? await this.getCustomerRecommendationCards(storeId) : [];
  const productProjectCards = scope !== 'customer' ? await this.productProjectRecommendationService.getCards(storeId) : [];
  return [...customerCards, ...productProjectCards].sort(byPriorityAndScore);
}
```

### 3.2 接口兼容策略

现有接口保持不破坏：

```text
GET /marketing/recommendations
```

新增可选查询参数：

```text
GET /marketing/recommendations?scope=product-project
GET /marketing/recommendations?scope=all
GET /marketing/recommendations?scope=customer
GET /marketing/recommendations?type=product_expiry_clearance
```

前端 `src/api/real/recommendation.ts` 增加可选 params，但默认调用不变。

### 3.3 P0 不建议先做数据库迁移

第一阶段推荐以实时查询 + 轻量缓存为主，原因：

- 当前已有 `Product`、`StockBatch`、`OrderItem`、`Schedule`、`Reservation`、`CustomerPredictionSnapshot` 足够支撑 P0。
- 新增快照表会拉长开发和迁移验证周期。
- 推荐卡数量有限，P0 可先按门店维度即时计算。

后续当推荐耗时超过可接受阈值，再加 `ProductDemandSnapshot`、`ProjectCapacitySnapshot` 等日快照表。

### 4.3 BE-03：商品临期消化推荐

推荐编码：`product_expiry_clearance`

数据查询：

- `StockBatch`：`expiryDate <= today + 60d`、`stock > 0`。
- 关联 `Product`：`status='active'`、`deletedAt=null`、按 `storeId` 过滤。
- `OrderItem`：近 30 天同商品销量，用于预测自然消化。
- `ProjectBomItem`：判断该商品可关联哪些项目。
- `CustomerPredictionSnapshot`：优先选择营销响应高、复购概率高、流失非极高的人群。
- `CustomerBehaviorEvent`：近期浏览同类商品/项目的客户加分。

自然销量预测：

```text
dailySales30d = 近30天商品销售数量 / 30
forecastSellThroughQty = dailySales30d * daysToExpiry
gapQty = max(0, batchStock - forecastSellThroughQty)
expectedLossAmount = gapQty * product.costPrice
```

推荐触发：

- `daysToExpiry <= 60`
- `gapQty > 0`
- `expectedLossAmount` 或 `gapQty` 达到展示阈值
- `daysToExpiry <= 7` 时只生成经营提醒，不建议对客促销，避免合规风险

评分：

```text
score =
  daysToExpiryScore * 0.35
  + sellThroughGapScore * 0.30
  + expectedLossScore * 0.20
  + bundleFitScore * 0.15
```

推荐动作：

| 情况 | preferredMode | executionModes |
| --- | --- | --- |
| 31-60 天，缺口小 | `miniapp_slot` 或 `advisor_task` | `['activity', 'advisor_task']` |
| 16-30 天，缺口中 | `activity` | `['activity', 'advisor_task', 'miniapp_slot']` |
| 8-15 天，缺口大 | `activity` | `['activity', 'advisor_task']` |
| 0-7 天 | `advisor_task` | `['advisor_task']`，并提示合规处理 |

目标客户圈选：

1. 近 180 天购买过同商品、同品牌或同品类客户。
2. 近 180 天做过关联项目的客户，关联项目来自 `ProjectBomItem`。
3. 肤质或健康档案与商品功效匹配的客户，P0 可先使用商品分类/名称关键词。
4. `marketingResponseScore >= 60` 或 `repurchase30dScore >= 55`。
5. 排除近 7 天已购买同商品、已有未来预约同活动客户、近期触达疲劳客户。

返回卡片示例字段：

```ts
{
  title: '修护面膜 30 天内临期，建议绑定补水护理消化 55 盒',
  category: 'inventory-opportunity',
  source: 'inventory',
  triggerType: 'product_expiry_clearance',
  priority: 'P0',
  urgency: 'urgent',
  recommendedItems: [
    { type: 'product', id: product.id, name: product.name, reason: '临期库存需消化', confidence: 90 },
    { type: 'project', id: project.id, name: project.name, reason: '项目 BOM/功效匹配', confidence: 78 }
  ],
  offer: { type: 'bundle', label: '补水护理赠修护面膜', reason: '用项目权益消化临期库存，减少直接降价' },
  riskWarnings: ['活动库存上限不得超过临期可用库存', '7天内临期商品不建议大范围对客促销']
}
```

### 4.4 BE-04：美容师排期不满推荐

推荐编码：`project_idle_capacity`

数据查询：

- `Schedule`：未来 1-7 天，状态为 `available` 或本地定义可预约状态。
- `Reservation`：同一门店、同一时段的预约占用。
- `BeauticianProjectSkill`：空闲美容师可服务的项目。
- `Project`：项目 active、未删除。
- `CustomerPredictionSnapshot`：优先高响应、复购窗口客户。
- `CustomerBehaviorEvent`：近期浏览项目但未预约客户。

容量计算建议：

```text
scheduledMinutes = 可预约排班总分钟数
bookedMinutes = 已预约占用分钟数
idleMinutes = max(0, scheduledMinutes - bookedMinutes)
utilizationRate = bookedMinutes / scheduledMinutes
```

推荐触发：

- 未来 1-7 天存在低峰时段。
- `utilizationRate < 60%`。
- `idleMinutes >= 120`。
- 可服务美容师不少于 1 人。

低峰时段默认：

```text
工作日 14:00-17:00
周末不默认低价，仅做余位提醒
```

评分：

```text
score =
  emptySlotScore * 0.35
  + customerDemandScore * 0.25
  + beauticianSkillFitScore * 0.15
  + lowPeakValueScore * 0.15
  + conversionProbabilityScore * 0.10
```

目标客户圈选：

1. 进入项目护理周期客户。
2. 近 7 天浏览项目但未预约客户。
3. 轻度沉睡客户，例如 30-60 天未到店。
4. 该美容师历史服务过且近期未到店客户。
5. 排除已预约客户、近 7 天同类触达客户、高价值客户强折扣触达。

推荐动作：

- `activity`：低峰项目券，权益绑定具体日期/时段。
- `advisor_task`：下发美容师/顾问老客邀约任务。
- `miniapp_slot`：小程序余位提醒。

关键风控：

- 活动草稿必须带 `usableTimeRange`，避免低峰券被黄金时段使用。
- 生成活动前再次校验余位，避免超卖。
- 若项目要求技能认证，必须匹配 `BeauticianProjectSkill.certified=true` 或技能等级阈值。

### 4.5 BE-05：商品补货复购推荐

推荐编码：`product_replenishment`

现状：自动营销规则库已有 `product_replenishment`，当前优先级偏 P2。P0 建议把它提升为 P0 并补齐真实圈人。

数据查询：

- `OrderItem` + `ProductOrder`：客户最近一次购买商品。
- `Product`：商品分类、库存、价格。
- `CustomerPredictionSnapshot`：营销响应、复购概率。
- `MarketingAutomationTouch`：频控。
- `StockBatch` / `Product.currentStock`：库存是否允许扩大营销。

商品消耗周期默认：

| 关键词/分类 | 默认周期 |
| --- | --- |
| 面膜 | 28 天 |
| 精华 | 45 天 |
| 洁面 | 60 天 |
| 面霜 | 60 天 |
| 防晒 | 35 天，夏季可缩短到 28 天 |
| 身体护理 | 75 天 |
| 默认 | 45 天 |

触发条件：

```text
today - lastPurchaseDate >= replenishmentDays
且 lastPurchaseDate 后未再次购买同商品
且 商品 active
且 当前库存 > 安全库存
且 触达频控允许
```

目标客户圈选：

- 同 SKU 复购：上次买 A，这次推荐 A。
- 同功效升级：同 SKU 库存不足时，推荐同品类/同功效商品。
- 项目搭配：客户近期有护理周期到期，推荐“项目 + 商品补货”。

推荐动作：

- 首选 `automation`，生成长期补货提醒规则。
- 可附带小额复购券，不建议强折扣。
- 渠道优先小程序，其次微信/短信。

自动规则草稿默认参数：

```json
{
  "trigger": "product_replenishment",
  "replenishmentDays": 45,
  "excludePurchasedRecently": true,
  "sameProductOnly": true,
  "channels": ["miniapp", "wechat"],
  "frequencyCap": { "sameCustomerDays": 14 }
}
```

### 4.6 BE-06：项目护理周期推荐

推荐编码：`project_cycle_due`，前端和规则引擎兼容现有 `care_cycle`。

数据查询：

- `OrderItem`：`itemType='project'` 的项目消费记录。
- `CardUsageRecord`：次卡核销的项目记录。
- `ServiceTask`：已完成服务任务。
- `Reservation`：未来预约，做排除。
- `Project`：项目状态、时长、分类。
- `Schedule`：近期是否有可预约容量。
- `CustomerPredictionSnapshot`：复购概率和营销响应。

周期默认：

| 项目类型/关键词 | 默认周期 |
| --- | --- |
| 补水/修护/基础面护 | 28 天 |
| 清洁/控油 | 21 天 |
| 抗衰/紧致 | 35 天 |
| 身体护理/SPA | 45 天 |
| 默认 | 28 天 |

触发条件：

```text
today - lastProjectServiceDate >= cycleDays
且 客户未来无同项目/同类项目预约
且 项目 active
且 未来 7 天存在可预约容量
且 触达频控允许
```

目标客户圈选：

- 有项目历史服务记录。
- 复购分或营销响应分较高。
- 当前无未来预约。
- 若客户有未使用次卡，优先提醒核销，而不是直接售卖新项目。

推荐动作：

- 首选 `automation`：项目周期到期自动提醒。
- 若叠加排期低峰，则同时支持 `activity`：低峰复购券。
- 可带顾问跟进任务，尤其是高价值客户。

自动规则草稿默认参数：

```json
{
  "trigger": "care_cycle",
  "cycleDays": 28,
  "lastServiceType": "project",
  "excludeBooked": true,
  "channels": ["miniapp", "sms"],
  "frequencyCap": { "sameCustomerDays": 7 }
}
```

### 5.5 FE-05：活动草稿初始值

现有 `CreateActivityDialog` 已能从推荐卡生成活动。P0 需要补：

- 临期活动：自动带入商品名、项目名、权益、库存上限、有效期。
- 低峰活动：自动带入可用日期、时段、项目、低峰券说明。
- 项目周期活动：自动带入项目名、目标客户、周期提醒文案。

建议 `createInitialDataFromRecommendation()` 识别：

```ts
recommendationType
inventorySnapshot
capacitySnapshot
offer.usableTimeRange
riskWarnings
```

## 6. 自动营销规则库建议

P0 需要补充或调整规则模板：

| 规则 | 当前状态 | 建议 |
| --- | --- | --- |
| `product_replenishment` | 已存在，偏 P2 | 提升为 P0，补圈人逻辑和默认参数 |
| `care_cycle` | 已存在 | 作为 `project_cycle_due` 的规则承接 |
| `product_expiry_clearance` | 新增 | 推荐以活动和顾问任务为主，不建议长期自动触达 |
| `project_idle_capacity` | 新增 | 推荐低峰活动、余位提醒、顾问任务 |

新增模板默认值：

```json
{
  "product_expiry_clearance": {
    "beforeDays": 60,
    "minGapQty": 5,
    "excludeDaysToExpiryLessThan": 7,
    "channels": ["miniapp", "store"]
  },
  "project_idle_capacity": {
    "windowDays": 7,
    "maxUtilizationRate": 0.6,
    "minIdleMinutes": 120,
    "timeRanges": ["14:00-17:00"],
    "channels": ["miniapp", "store"]
  }
}
```

## 8. 验收用例

### 8.1 商品临期

准备数据：

- 某商品有 `StockBatch`，30 天后过期，库存 80。
- 近 30 天销量只有 25。
- 商品关联一个项目 BOM。

期望：

- 推荐列表出现 `product_expiry_clearance`。
- 卡片展示临期库存、预计自然消化、缺口、预计避免损耗。
- 可生成活动草稿。
- 活动草稿包含商品、项目、权益和库存上限提示。

### 8.2 排期不满

准备数据：

- 未来 7 天某工作日下午有美容师可预约排班。
- 预约占用率低于 60%。
- 有客户进入复购窗口或浏览未预约。

期望：

- 推荐列表出现 `project_idle_capacity`。
- 卡片展示空闲时段、空闲工时、占用率、推荐项目。
- 可生成低峰活动草稿。
- 低峰活动草稿带可用时段。

### 8.3 商品补货复购

准备数据：

- 客户 45 天前购买精华。
- 之后没有复购同商品。
- 商品库存充足。

期望：

- 推荐列表出现 `product_replenishment`。
- 推荐受众包含该客户。
- 可生成自动营销规则草稿。
- 草稿触发参数包含 `replenishmentDays` 和频控。

### 8.4 项目护理周期

准备数据：

- 客户 28 天前做过补水项目。
- 未来无预约。
- 未来 7 天该项目有可预约容量。

期望：

- 推荐列表出现 `project_cycle_due` 或 `care_cycle`。
- 推荐受众包含该客户。
- 可生成自动营销规则草稿。
- 若叠加低峰容量，也可生成活动草稿。

## 10. 风险与取舍

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 订单 `items` 历史 JSON 不规范 | 商品/项目销量统计可能漏数 | P0 优先读结构化 `OrderItem`，JSON 仅兜底 |
| 商品功效字段不足 | 商品和项目匹配解释不够准 | P0 用分类、名称关键词、BOM 关系；P1 再补知识库 |
| 排班状态枚举不统一 | 可用工时计算偏差 | 建议统一 `available/active/normal/可预约/空闲` 可用状态 |
| 推荐卡是虚拟 ID | 采纳和受众回查可能不稳定 | 加 `recommendationKey`，并在事件 payload 里记录 |
| 临期促销合规风险 | 客诉和品牌风险 | 7 天内临期只给经营提醒，不默认对客促销 |
| 高价值客户低价触达 | 伤害品牌和客单 | 高 LTV 客户优先顾问权益，不默认低价券 |

## 11. 建议开发顺序

1. 后端先完成 `ProductProjectRecommendationService` 骨架和 `scope=product-project` 接口。
2. 先做商品临期和项目周期两类，因为数据最稳定，业务价值清晰。
3. 再做商品补货复购，补齐自动规则草稿。
4. 再做排期不满，接入排班容量和低峰活动草稿。
5. 前端同步扩展推荐卡展示和筛选。
6. 最后统一补受众回查、推荐事件、单测和联调数据。

## 12. 最小可交付版本

如果需要压缩周期，MVP 可以收敛为：

- 只在智能推荐页展示 P0 四类推荐卡。
- 商品临期和排期不满支持“发一次活动”。
- 商品补货和项目周期支持“开启自动触达”。
- 暂不新增快照表。
- 暂不做小程序推荐位草稿，只复用活动页承接。
- 终端跟进继续复用现有“下发终端跟进”能力。

这样可以先证明门店价值：减少临期损耗、填充低峰排期、提升商品复购、提升项目复购。后续再扩展小程序推荐位、调拨建议、经营归因看板。
