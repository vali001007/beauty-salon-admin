# 智能推荐模块目标架构详细改造方案

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将当前多源动态推荐、旧新采纳入口并存、串行触达和重复效果汇总，改造为严格按门店隔离、推荐实例可追溯、执行可恢复、效果事实去重的智能推荐闭环。

**Architecture:** 保留现有 RFM、流失、复购、营销响应和 LTV 预测能力，新增门店级预测运行服务、统一推荐编排器、持久化推荐实例、受众/权益快照、分片投递任务和统一营销效果事实层。管理端只读取推荐实例，采纳、执行和复盘全部引用同一个 `recommendationInstanceId`，不再按动态数字 ID 重新生成推荐。

**Tech Stack:** NestJS、Prisma 7、PostgreSQL、React、TypeScript、Vite、Vitest、Jest、Ami Aura Lite Terminal、Marketing H5。

- 不接入真实短信、企业微信或公众号供应商。
- 不新增推荐类型。
- 不重做营销 H5 渲染器。
- 不把 Ami Glow、Marketing H5 和管理端合并为一个客户端。
- 不删除历史预测、历史 touch、旧归因记录。

## 2. 第一性原理与架构约束

### 2.1 一个推荐必须是一个可引用的业务对象

推荐不能继续是接口运行时临时拼出的 JSON。每次展示、查看受众、采纳、执行和复盘必须引用同一个持久化推荐实例。

唯一业务链路：

```text
门店业务事实
  -> 门店预测批次 / 实时经营信号
  -> 统一推荐编排器
  -> RecommendationInstance
  -> AudienceSnapshot + OfferSnapshot
  -> Adoption
  -> Activity / Strategy / TerminalTask
  -> DeliveryFact / PageEvent / OrderAttribution
  -> MarketingEffectFact
```

### 2.2 门店范围必须由服务端请求上下文决定

- 管理端请求只使用 `X-Store-Id`。
- body/query 中的 `storeId` 仅在平台级内部任务中使用，普通业务接口拒绝覆盖门店头。
- scheduler 必须显式遍历门店并传入 storeId。
- repository/service 方法不得把 `storeId` 设计成可选参数。
- 历史 `storeId=null` 预测批次保留为 `legacy_global`，不再参与新推荐和执行。

### 2.3 预测快照与实时信号分层

| 数据类型 | 示例 | 生命周期 |
|---|---|---|
| 预测快照 | 流失、复购、响应、LTV | 一个业务日内稳定 |
| 实时信号 | 库存、排班、预约、领券未核销、浏览放弃 | 分钟或小时级变化 |
| 推荐实例 | 预测 + 实时信号编排结果 | 明确 `generatedAt/expiresAt` |

### 2.4 事实只记一次，维度只负责查看

- 一次投递只生成一条投递事实。
- 一笔订单只生成一条主归因收入事实。
- 一笔退款只生成一条负向冲减事实。
- 活动、推荐、权益、页面、渠道是事实的维度，不是再次累加的数据源。

## 4. 数据模型设计

### 4.1 PredictionRun 增量字段

保持历史记录兼容，不直接删除 `storeId=null` 行：

```prisma
model PredictionRun {
  id            Int       @id @default(autoincrement())
  storeId       Int?
  businessDate  DateTime? @db.Date
  runKey        String?   @unique
  scopeStatus   String    @default("store_scoped")
  modelVersion  String
  status        String    @default("running")
  startedAt     DateTime  @default(now())
  finishedAt    DateTime?
  customerCount Int       @default(0)
  summaryJson   Json?

  @@index([storeId, businessDate, status])
}
```

新批次 runKey：

```text
store:{storeId}:date:{YYYY-MM-DD}:model:{modelVersion}
```

数据库补充 CHECK：

```sql
CHECK (
  ("scopeStatus" = 'legacy_global' AND "storeId" IS NULL)
  OR
  ("scopeStatus" = 'store_scoped' AND "storeId" IS NOT NULL AND "businessDate" IS NOT NULL AND "runKey" IS NOT NULL)
)
```

## 6. 统一推荐编排器

### 6.1 Provider 接口

```ts
export type RecommendationBuildContext = {
  storeId: number;
  businessDate: string;
  predictionRunId: number;
  generatedAt: Date;
};

export type RecommendationCandidate = {
  recommendationKey: string;
  sourceType: 'prediction' | 'lifecycle' | 'product_project';
  sourceVersion: string;
  title: string;
  description?: string;
  priority: 'P0' | 'P1' | 'P2';
  urgency: 'urgent' | 'recommended' | 'opportunity';
  preferredMode: 'activity' | 'automation' | 'terminal_follow_up';
  executionModes: Array<'activity' | 'automation' | 'terminal_follow_up'>;
  customerIds: number[];
  audienceRule: Record<string, unknown>;
  audienceReasons: Array<{ customerId: number; score: number; reason: string }>;
  evidenceSnapshot: Record<string, unknown>;
  strategySnapshot?: Record<string, unknown>;
  offerContext: Record<string, unknown>;
  expiresAt: Date;
};

export interface RecommendationProvider {
  readonly sourceType: RecommendationCandidate['sourceType'];
  build(context: RecommendationBuildContext): Promise<RecommendationCandidate[]>;
}
```

### 6.2 Orchestrator 责任

1. 校验预测批次属于当前门店和业务日。
2. 并行调用三个 provider。
3. 统一去重 customerIds。
4. 统一调用服务端权益匹配。
5. 计算候选 fingerprint。
6. 相同 `storeId + recommendationKey + fingerprint` 复用原实例；fingerprint 变化时创建新实例。
7. 在事务中写入 instance、audience snapshot、members、offer snapshot。
8. 将同门店同 recommendationKey 的上一 active 实例标记为 `superseded`。
9. 返回持久化实例，不返回运行时临时卡。

fingerprint 只包含决定推荐内容的稳定字段：

```ts
sha256(JSON.stringify({
  recommendationKey,
  predictionRunId,
  customerIds: [...customerIds].sort((a, b) => a - b),
  selectedPromotionId,
  strategySnapshot,
  expiresAt,
}))
```

```text

双写至少 7 天，对比：

- revenue；
- conversion；
- delivery；
- refund net revenue。

Release D 使用同一份观察导出执行 7 个连续业务日的逐日对账。`deliveryCount`、`conversionCount` 必须完全一致；`revenue`、`refundAmount` 默认允许 0.01 元舍入误差。任一天缺数据、导出陈旧、覆盖有缺口、全窗口没有真实样本或任一指标超差，门禁均返回非零退出码，禁止切换事实层读取。

```

观察导出是外部日志/指标平台生成的只读 JSON，不由审计工具回写数据库。核心结构如下：

```json
{
  "storeId": 6,
  "rangeStart": "2026-07-07T00:00:00.000Z",
  "rangeEnd": "2026-07-14T00:00:00.000Z",
  "exportedAt": "2026-07-14T00:30:00.000Z",
  "observationEndBusinessDate": "2026-07-13",
  "sources": {
    "worker": "delivery-job-worker-metrics",
    "legacyEffect": "legacy-marketing-attribution-projection",
    "factEffect": "marketing-effect-facts"
  },
  "coverageSegments": [{ "start": "...", "end": "..." }],
  "workerDays": [{
    "businessDate": "2026-07-13",
    "executionCount": 2,
    "deliveryJobCount": 100,
    "queuedCount": 0,
    "runningCount": 0,
    "staleExecutionCount": 0,
    "deliveredCount": 99,
    "deadLetterCount": 1,
    "expiredLeaseCount": 0,
    "duplicateDeliveryArtifactCount": 0,
    "deliveredMissingFactCount": 0
  }],
  "factParityDays": [{
    "businessDate": "2026-07-13",

```

每份导出只允许包含一个 `storeId`。`legacyEffect` 与 `factEffect` 必须声明为两个不同来源，防止把同一份查询结果复制两次后得到虚假的“对账一致”。多门店放量时按门店分别执行门禁，不允许用跨店汇总掩盖单店异常。

## 9. 回滚策略

| 发布批次 | 回滚动作 | 数据处理 |
|---|---|---|
| A | 关闭 instance write/read | 新实例表保留，不影响旧链路 |
| B | 关闭 adoption v2，旧接口恢复兼容转发 | 已创建 adoption 和业务对象保留 |
| C | 关闭 job engine，暂停新 execution | queued jobs 保留，禁止删除；修复后续跑 |
| D | 关闭 effect fact read，恢复旧查询 | facts 继续双写，避免丢失观察期数据 |

禁止回滚方式：

- 不删除新实例和 facts。
- 不把 delivered 改回 reached。
- 不通过重跑同一 execution 复制触达。
- 不回滚已经发布的活动页面内容。

## 12. 最终建议

推荐采用本方案的四阶段渐进替换，不采用一次性重写。

原因：

1. 当前推荐算法和业务资产已经可用，价值在于收敛身份和事实，不在于推翻算法。
2. 当前工作区和真实数据库包含大量历史营销数据，必须双轨验证和可回滚。
3. 自动执行与效果复盘属于高风险链路，应在推荐实例稳定后再切换。
4. 通过 feature flags 双写、对比、切读，可以把业务中断风险控制在单个发布批次内。

实施时第一优先级不是开发新推荐卡，而是完成：

```text
门店预测强隔离
  -> 推荐实例稳定身份
  -> 统一采纳
  -> 可恢复执行
  -> 去重事实复盘
```

这条顺序不可调整。
