## 1. 目标与边界

### 1.1 目标

把智能推荐从“给运营一条泛建议”升级为“可执行、可解释、可归因的精准营销动作卡”：

1. 目标群体从粗分层升级为多标签、多维度组合。
2. 权益资产从静态清单升级为可被推荐引擎匹配的动作素材库。
3. 推荐卡自动选择最合适的权益资产，并给出匹配原因、备选权益和风险提示。
4. 推荐卡可一键生成营销活动草稿或自动营销规则草稿。
5. 活动、自动规则、权益领取、预约、核销、下单、收入可以串成复盘闭环。

### 1.2 产品原则

- 不追求一开始上复杂机器学习，P0 先用规则评分 + 可解释权重。
- 不是给客户“最大优惠”，而是用“最低必要权益”推动下一步动作。
- 高价值客户优先服务权益、优先预约、专属顾问，少用大额低价券。
- 已领未核销、券即将到期、卡项剩余等场景优先推动既有权益使用，不新增无关让利。
- 推荐结果必须能直接承接下一步动作：创建活动、创建自动规则、顾问跟进、终端核销。

### 1.3 不做范围

- P0 不做黑盒模型和自动调参。
- P0 不做跨门店复杂协同过滤。
- P0 不做完整营销预算优化器，只做权益成本和毛利保护的规则约束。
- P0 不替代运营审核，高风险权益仍需人工确认。

## 3. 总体方案

### 3.1 核心链路

```text
客户/经营数据
  -> 客户多维标签画像
  -> 推荐信号识别
  -> 目标群体生成
  -> 权益资产匹配评分
  -> 推荐动作卡
  -> 活动草稿 / 自动规则草稿 / 顾问任务
  -> 小程序领取 / 预约 / 终端核销
  -> 效果归因与权重复盘
```

### 3.2 推荐卡必须回答的问题

1. 推荐给谁：目标客户数量、客户标签、命中原因。
2. 为什么现在推荐：护理周期、库存、排期、行为意图、节令、权益到期等证据。
3. 用哪个权益承接：主推权益、备选权益、权益成本、毛利风险。
4. 走哪种执行方式：单次活动、自动规则、两者均可、顾问跟进。
5. 用什么渠道触达：小程序、短信、顾问微信、门店终端、社群。
6. 后续如何复盘：曝光、领取、预约、核销、成交、收入、成本、ROI。

## 4. 多维标签体系

### 4.1 客户标签分层

P0 建议先生成以下 10 类标签，每类标签都要能解释来源。

| 标签维度 | 标签示例 | 数据来源 | 推荐用途 |
| --- | --- | --- | --- |
| 生命周期 | 新客未首单、首单后未复购、活跃老客、沉睡、流失高风险 | 客户、订单、预约、预测快照 | 决定权益强度和触达频率 |
| 消费价值 | 高 LTV、中 LTV、低 LTV、高客单低频、高频低客单 | 订单、卡项、预测快照 | 决定折扣型还是服务型权益 |
| 行为意图 | 浏览项目、浏览活动、领券未核销、预约放弃、加购未买 | 小程序事件、推荐事件、预约记录 | 决定触发时机 |
| 服务偏好 | 补水、清洁、抗衰、敏感修护、身体护理 | 项目订单、服务记录、浏览记录 | 决定适用项目 |
| 肤质问题 | 干皮、油皮、敏感、痘肌、屏障受损 | Ami Aura Lite、客户健康档案 | 决定项目和产品组合 |
| 卡项状态 | 次卡剩余、套餐临期、储值余额、年卡会员 | 卡项、核销、订单 | 决定续卡、核销、升级 |
| 商品周期 | 精华预计用完、防晒预计用完、临期库存适配 | 商品订单、库存、消耗周期 | 决定产品补货和清库存 |
| 预约容量 | 工作日可约、低峰可约、美容师空档 | 排班、预约、项目时长 | 决定低峰权益 |
| 渠道偏好 | 小程序活跃、短信可达、顾问微信、社群活跃 | 事件、触达记录、客户资料 | 决定触达渠道 |
| 触达疲劳 | 7 天已触达、30 天多次未响应、近期已成交 | 营销执行记录、推荐事件、订单 | 决定降权或排除 |

### 4.2 标签结果结构

建议在推荐计算层先形成统一画像对象，可先不落新表，P0 允许即时计算；后续再持久化。

```ts
interface CustomerMarketingProfile {
  customerId: number;
  lifecycleTags: string[];
  valueTags: string[];
  behaviorTags: string[];
  preferenceTags: string[];
  skinTags: string[];
  cardTags: string[];
  productCycleTags: string[];
  capacityTags: string[];
  channelTags: string[];
  fatigueTags: string[];
  evidence: string[];
  updatedAt: string;
}
```

### 4.3 标签更新策略

| 标签类型 | 更新方式 | P0 建议 |
| --- | --- | --- |
| 生命周期、消费价值 | 每日批量 + 推荐即时补算 | 使用现有预测快照和订单数据 |
| 行为意图 | 事件触发更新 | 浏览、领券、预约放弃实时进入推荐 |
| 服务偏好、肤质 | 订单/检测后更新 | 优先使用最近 180 天数据 |
| 卡项、商品周期 | 每日批量 | 临期、剩余、预计用完需要每日刷新 |
| 预约容量 | 实时查询 | 低峰推荐必须基于最新排班 |
| 触达疲劳 | 每次触达后更新 | 推荐评分时实时扣分 |

## 5. 权益资产标签化

### 5.1 权益资产需要补充的匹配元数据

当前 `Promotion` 已有基础字段，建议通过 `metadata` 和 `grossMarginGuard` 承载 P0 所需的匹配规则，避免先大改 schema。

```ts
interface PromotionMatchingMetadata {
  lifecycleTags?: string[];
  valueTags?: string[];
  behaviorTags?: string[];
  preferenceTags?: string[];
  skinTags?: string[];
  cardTags?: string[];
  productCycleTags?: string[];
  channelTags?: string[];
  includeTags?: string[];
  excludeTags?: string[];
  preferredExecutionModes?: ('activity' | 'automation' | 'both' | 'consultant_task')[];
  offerStrength?: 'light' | 'medium' | 'strong';
  fatigueCooldownDays?: number;
  maxPerCustomer?: number;
  requiresManualApproval?: boolean;
  reasonTemplate?: string;
}
```

### 5.2 权益强度分层

| 强度 | 典型权益 | 适合人群 |
| --- | --- | --- |
| `light` | 提醒、优先预约、顾问服务、小样、护理加项 | 高价值、高意图、已领未核销 |
| `medium` | 小额满减、项目升级、产品搭售 | 老客复购、中价值、高响应 |
| `strong` | 新客体验价、沉睡回店券、大额满减 | 新客首单、高流失、久未到店 |

### 5.3 权益禁用规则

| 禁用条件 | 处理 |
| --- | --- |
| `status != active` | 不参与匹配 |
| `approvalStatus != approved` | 不参与匹配 |
| 已超 `maxIssueCount` | 不参与匹配 |
| 当前时间不在 `startAt/endAt` | 不参与匹配 |
| 命中 `excludeTags` | 不参与匹配 |
| 毛利保护不通过 | 不参与匹配或强降权 |
| 客户近期已享同类权益 | 强降权或排除 |
| 同客户触达疲劳过高 | 强降权或转顾问任务 |

## 6. 权益匹配评分器

### 6.1 评分公式

```text
offerFitScore =
  scenarioScore * 0.22
+ audienceScore * 0.18
+ behaviorIntentScore * 0.14
+ itemFitScore * 0.12
+ timingUrgencyScore * 0.10
+ valueProtectionScore * 0.10
+ channelFitScore * 0.06
+ operationFitScore * 0.04
+ historicalEffectScore * 0.04
- fatiguePenalty
- marginRiskPenalty
- conflictPenalty
```

### 6.2 分项说明

| 分数 | 说明 | 示例 |
| --- | --- | --- |
| `scenarioScore` | 推荐场景与权益 `scenario` 是否匹配 | `care_cycle_due` 匹配护理周期预约券 |
| `audienceScore` | 客户标签与权益适用标签是否匹配 | 新客未首单匹配新客体验价 |
| `behaviorIntentScore` | 客户近期行为意图是否强 | 浏览项目、领券、预约放弃 |
| `itemFitScore` | 项目/商品/肤质是否适配 | 干皮匹配补水修护 |
| `timingUrgencyScore` | 时间窗口是否紧迫 | 券 D-3 到期、卡 7 天到期 |
| `valueProtectionScore` | 权益强度与客户价值是否匹配 | VIP 不默认大额低价券 |
| `channelFitScore` | 推荐渠道是否可达 | 小程序活跃优先小程序券 |
| `operationFitScore` | 门店排期/库存/美容师是否承接得住 | 低峰权益绑定空档 |
| `historicalEffectScore` | 同类权益历史转化是否好 | 过去 30 天核销率高 |
| `fatiguePenalty` | 触达疲劳扣分 | 7 天已触达 2 次 |
| `marginRiskPenalty` | 毛利风险扣分 | 折扣低于项目毛利底线 |
| `conflictPenalty` | 权益冲突扣分 | 已领同类券未核销 |

### 6.3 匹配输出

每次匹配返回主推权益和备选权益。

```ts
interface PromotionMatchResult {
  promotionId: number;
  promotionName: string;
  type: string;
  scenario?: string;
  discountText: string;
  fitScore: number;
  fitLevel: 'excellent' | 'good' | 'backup' | 'not_recommended';
  fitReasons: string[];
  riskWarnings: string[];
  estimatedCost?: number;
  expectedGrossProfit?: number;
  promotion: Promotion;
}
```

### 8.2 推荐卡展示要求

推荐卡前端需要展示：

1. 推荐人群：用业务语言展示，不暴露“高流失风险”等敏感词给客户。
2. 推荐权益：展示主推权益、权益类型、有效期、成本提示。
3. 推荐原因：展示 2-4 条可解释证据。

5. 风险提示：毛利、触达疲劳、库存/排期限制。
6. 备选权益：运营可切换，但切换后保留原始推荐用于归因。

## 9. 活动与自动规则承接

### 9.1 生成营销活动草稿

适用：节日活动、新客拓客、临期商品消化、老带新、低峰活动、季节活动。

生成草稿时自动带入：

- 活动名称：推荐卡 `title` + 权益名称。
- 目标客户：推荐卡受众快照。
- 权益资产：`primaryPromotion.promotionId`。
- 活动周期：按 `validDays` 或场景默认周期。
- 推荐项目/商品：`recommendedItems`。
- 推荐渠道：`recommendedChannels`。
- 小程序活动页 schema：从权益、项目、商品和门店信息生成。
- 来源归因：`sourceRecommendationId`、`recommendationKey`、`predictionRunId`、`audienceSnapshotId`。

### 9.2 生成自动营销规则草稿

适用：护理周期、卡到期、券到期、领券未核销、浏览未预约、预约放弃、产品补货、生日。

生成草稿时自动带入：

- 规则类型：`triggerRule.type`。
- 规则参数：`triggerRule.params`。
- 规则关系：默认 `AND`。
- 执行动作：发权益、提醒、顾问任务、推送小程序卡片。
- 权益资产：`primaryPromotion.promotionId`。
- 触达渠道：小程序、短信、顾问任务。
- 频控：同客户 7 天最多 1 次；生日、卡到期、护理周期按事件周期计算。
- 排除条件：已预约、近期已成交、已领同类券未核销、触达疲劳过高。

### 9.3 顾问任务承接

以下场景不应只发券，需要生成顾问跟进动作：

| 场景 | 原因 | 动作 |
| --- | --- | --- |
| 高 LTV 沉睡 | 不宜公开低价刺激 | 顾问私域回访 + 专属护理权益 |
| 差评/低满意度 | 需要服务修复 | 店长/顾问回访 |
| VIP 维护 | 强服务体验 | 专属顾问护理计划 |
| 高风险权益 | 需人工确认 | 审核后再发 |

状态：已完成 P0 承接。推荐页可按推荐卡选择目标客户并批量下发终端跟进任务；任务会进入店长、前台或顾问/美容师队列，脚本中带入当前承接权益，完成后可记录已预约或已成交。

## 14. 验收用例

### 用例 1：护理周期精准复购

输入：

- 客户 30 天前做过补水护理。
- 肤质标签为干皮。
- 最近 7 天浏览过补水项目。
- 当前未预约。

预期：

- 推荐类型：`care_cycle`。
- 目标群体：护理周期到期 + 干皮 + 补水偏好。
- 权益：补水/护理周期类小额券。
- 执行方式：自动规则。
- 推荐原因包含护理周期、肤质、浏览行为。

### 用例 2：VIP 沉睡客户不发大额券

输入：

- 客户高 LTV / VIP。
- 90 天未到店。
- 近期未响应短信。

预期：

- 推荐类型：`churn_winback`。
- 权益：VIP 专属护理礼遇或顾问服务。
- 不默认推荐大额满减券。
- 执行方式：顾问跟进 + 可选自动规则。

### 用例 3：已领未核销不重复让利

输入：

- 客户已领取满减券。
- 5 天未预约未核销。

预期：

- 推荐类型：`coupon_claimed_unused`。
- 权益：已领权益核销提醒。
- 不新增第二张优惠券。
- 执行方式：自动规则。

### 用例 4：低峰排期权益绑定时段

输入：

- 某美容师工作日下午有连续空档。
- 有一批小程序活跃、高响应客户。

预期：

- 推荐类型：`project_idle_capacity`。
- 权益：低峰预约礼。
- 权益带 `usableTimeRange`。
- 活动草稿提示仅限指定时段。

### 用例 5：临期商品消化

输入：

- 某补水产品 30 天内临期。
- 有干皮/补水项目客户。

预期：

- 推荐类型：`product_expiry_clearance`。
- 权益：临期商品消化券或产品搭售券。
- 推荐项目/商品包含临期商品。
- 活动草稿带库存上限和风险提示。

## 16. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 标签过多导致运营看不懂 | 推荐解释复杂 | 前端只展示 2-4 条核心原因，详细标签放抽屉 |
| 权益匹配过度依赖规则 | 初期不够智能 | 保留运营切换权益，并记录效果回流 |
| 高价值客户被低价券打扰 | 伤害品牌和客单 | 高 LTV 默认服务权益优先，大额券需人工确认 |
| 已领未核销重复发券 | 增加成本 | 对同类权益做冲突检查 |
| 低峰券被常态化滥用 | 形成低价心智 | 绑定指定时段、短有效期、限量 |
| 活动/规则来源归因断裂 | 无法复盘 | 必须固化受众快照和推荐来源 |
| 数据缺失导致误推荐 | 客户体验差 | 缺失维度不强行命中，推荐原因必须有证据 |

6. 小程序领取、预约、终端核销、订单收入能进入效果分析。
