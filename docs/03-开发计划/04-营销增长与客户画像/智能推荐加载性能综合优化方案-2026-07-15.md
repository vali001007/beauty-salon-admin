# 智能推荐加载性能综合优化方案

> 日期：2026-07-15
> 适用页面：`/customer-marketing/intelligent-recommendation`
> 目标：消除重复等待、冷启动门店竞态和重型首屏查询，同时保持推荐数据的新鲜度、可解释性与发布安全。

## 一、结论

前端缓存、轻量首屏接口、服务端缓存、数据库快照、SQL 优化和异步刷新不是互斥方案，而是分别解决不同层级的问题：

| 层级 | 解决的问题 | 最终采用方式 |
| --- | --- | --- |
| 页面状态 | 重复进入时重新等待、门店初始化竞态 | 门店就绪门禁 + stale-while-revalidate |
| HTTP 契约 | 首屏响应字段过重、次要数据阻塞主内容 | 摘要与卡片简表首屏返回，详情按需加载 |
| 服务端读取 | 多实例重复查询、3 秒本地缓存价值有限 | 门店级分布式读取缓存 |
| 数据事实 | 每次请求临时聚合多个事实表 | 持久化工作台读模型/快照 |
| 数据库 | 重复分页、读取大 JSON、重复覆盖统计 | 查询复用、字段裁剪、索引和统计缓存 |
| 主动刷新 | 用户等待同步计算完成 | 后台刷新任务 + 进度查询 |

综合最优方案不是把六种缓存叠加，而是建立一条统一的数据版本链：

```text
业务事实变化 / 定时任务 / 手动刷新
              ↓
推荐刷新任务（异步、门店级去重）
              ↓
PredictionRun + RecommendationInstance + Audience/Offer Snapshot
              ↓
RecommendationWorkspaceSnapshot（门店级读模型，带 revision）
              ↓
Redis/共享缓存（按 storeId + revision）
              ↓
轻量 workspace API
              ↓
前端 SWR 缓存（按 storeId + revision）
```

这条链路中，`revision` 是唯一的新鲜度依据。三层缓存都消费同一个版本号，避免各自使用不同 TTL 导致页面显示旧推荐。

## 二、当前问题与第一性原理判断

### 2.1 正常进入不会重新运行预测

页面首次挂载调用：

```text
GET /marketing/recommendation-workspace?page=1&pageSize=50&refresh=false
```

只有点击“刷新推荐”才会传入 `refresh=true`。同一业务日存在 `completed` 的 `PredictionRun` 时，后端复用预测批次，不重新生成全部客户预测快照。

因此当前慢点不是“每次进入重新计算 1252 位客户”，而是：

1. 页面每次挂载都重新发起 3 个主请求。
2. workspace 每次冷读都执行列表、总数、覆盖率和采纳状态查询。
3. 列表直接携带证据、策略、权益、库存、产能等 JSON 快照。
4. 后端读取缓存只有 3 秒且仅在单进程内有效。
5. 页面一次最多加载 50 条并立即渲染全部复杂卡片。

### 2.2 冷启动存在门店状态竞态

`currentStoreId` 初始值为 `null`，门店列表由 `StoreSwitcher` 异步加载。智能推荐页面的加载 effect 不等待门店就绪，也不监听 `currentStoreId`。

结果是：

1. 页面先请求推荐接口。
2. 请求没有 `X-Store-Id`。
3. 推荐请求失败或进入恢复卡。
4. 门店随后初始化完成。
5. 页面不会自动重新请求。

这不是纯性能问题，而是正确性问题，必须作为 P0 优先修复。

## 三、综合目标架构

### 3.1 页面首屏分层

首屏只依赖一个轻量请求：

```http
GET /marketing/recommendation-workspace/summary?page=1&pageSize=20
```

返回：

```json
{
  "storeId": 6,
  "revision": "6:2026-07-15:42",
  "generatedAt": "2026-07-15T00:01:03.338+08:00",
  "freshness": "fresh",
  "coverage": {
    "totalCustomers": 1252,
    "predictedCustomers": 1252,
    "coverageRate": 100
  },
  "counts": {
    "total": 18,
    "P0": 6,
    "P1": 9,
    "P2": 3
  },
  "items": [
    {
      "id": "instance-id",
      "title": "229 位客户命中沉睡客户召回",
      "priority": "P0",
      "targetCount": 229,
      "matchScore": 73,
      "preferredMode": "activity",
      "offerLabel": "回归专享满300减100",
      "inventoryStatus": "ready",
      "capacityStatus": "ready",
      "latestAdoptions": []
    }
  ]
}
```

首屏不返回：

- 完整 `evidenceSnapshot`。
- 完整 `strategySnapshot`。
- 全量权益候选与匹配分解。
- 客户名单。
- 大块库存和产能明细。
- 完整归因事件列表。

用户展开卡片时再请求：

```http
GET /marketing/recommendation-instances/:id
```

用户打开目标客户时继续使用分页受众接口。

### 3.2 次要指标独立加载

以下内容不阻塞推荐卡首屏：

- 终端跟进任务汇总。
- 生命周期质量指标。
- 字段覆盖率、规则命中率、归因完整率、可承接率。

页面先展示推荐摘要和卡片，次要模块各自显示局部骨架屏。单个次要请求失败时，不得回退整个推荐工作台。

### 3.3 门店级工作台读模型

新增持久化读模型：

```text
MarketingRecommendationWorkspaceSnapshot
```

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `storeId` | 门店隔离键 |
| `businessDate` | 上海业务日期 |
| `revision` | 门店级单调递增版本 |
| `predictionRunId` | 对应预测批次 |
| `summaryJson` | 覆盖率、数量、生成时间、质量摘要 |
| `cardListJson` | 首屏轻量卡片列表 |
| `status` | building / active / superseded / failed |
| `generatedAt` | 生成时间 |
| `activatedAt` | 原子切读时间 |
| `expiresAt` | 业务过期时间 |

约束：

- 每个门店只有一个 `active` 快照。
- 新快照完整构建成功后再原子切换为 `active`。
- 构建失败继续读取上一版，不让页面退化为空。
- 历史快照保留用于回滚和问题追踪。

### 3.4 统一缓存规则

#### 前端缓存

缓存键：

```text
marketing-workspace:{storeId}:{page}:{filters}
```

策略：

- `staleTime`：5 分钟。
- `gcTime`：30 分钟。
- 有缓存时立即展示，再后台校验 revision。
- 门店切换时取消旧门店未完成请求。
- 发布活动、启用自动触达、下发跟进只更新对应卡片执行状态，不清空全部推荐卡。
- 手动刷新完成后按新 revision 替换缓存。

#### 服务端共享缓存

缓存键：

```text
marketing:workspace:{storeId}:{revision}:{page}:{filters}
```

策略：

- 使用 Redis 或等价共享缓存，不使用单进程 Map 作为主缓存。
- TTL 10 分钟，revision 变化时旧 key 自然失效。
- 缓存失败不得阻塞数据库快照读取。
- 不缓存鉴权失败、门店缺失和构建失败结果。

#### 数据库快照

数据库快照是缓存未命中时的稳定事实源。页面读取不得重新运行预测、受众生成或权益匹配。

### 3.5 异步刷新任务

手动刷新改为：

```http
POST /marketing/recommendation-refresh-jobs
GET  /marketing/recommendation-refresh-jobs/:jobId
```

刷新流程：

1. 根据 `storeId + businessDate` 获取门店级互斥锁。
2. 已有 running 任务时直接返回原任务。
3. 复用或生成当天预测批次。
4. 构建推荐候选、受众、权益和承接快照。
5. 构建新的 workspace snapshot。
6. 原子激活新 revision。
7. 发布缓存失效事件。
8. 页面获取新 revision 并切换内容。

页面显示明确阶段：

- 排队中。
- 正在更新客户预测。
- 正在生成推荐机会。
- 正在匹配权益与承接能力。
- 已完成。
- 刷新失败，继续使用上一版。

## 四、方案之间的冲突与化解方式

### 4.1 前端缓存与服务端缓存

冲突风险：两层分别按 TTL 判断，刷新完成后前端仍显示旧数据。

统一方式：服务端返回 `revision`，前端只要发现 revision 改变就替换缓存。TTL 只负责减少请求，不负责判断业务新鲜度。

### 4.2 服务端缓存与数据库快照

冲突风险：同时保存完整聚合结果，形成两个事实源。

统一方式：数据库快照是事实源，Redis 只是按 revision 缓存数据库快照的序列化结果。Redis 不独立生成业务数据。

### 4.3 轻量接口与按需详情

冲突风险：接口拆分后请求数量增加。

统一方式：首屏只发一个 summary 请求；详情只在用户展开时请求，并缓存已展开内容。优化目标是减少首屏总数据量，不追求整个会话请求数绝对最少。

### 4.4 定时刷新与手动刷新

冲突风险：两个刷新任务同时写推荐实例和快照。

统一方式：统一进入同一任务队列，并用 `storeId + businessDate` 去重。任何入口都不能直接调用编排器写库。

### 4.5 实时业务变化与推荐快照

冲突风险：库存、产能、预约、订单和权益状态变化速度快于预测批次。

统一方式：

- 客户预测按日或按业务策略刷新。
- 库存、产能、权益可用性作为短周期承接投影更新。
- 卡片详情打开时校验高风险承接字段。
- 发布活动、启用自动触达前执行最终承接门禁。
- 页面首屏快照用于决策提示，不作为资产扣减或发送动作的最终依据。

## 五、实施顺序

### 阶段一：P0 正确性与即时体验

目标：解决冷启动错误和重复等待。

- 为门店状态增加 `isStoreReady`。
- 智能推荐页面在门店就绪后加载。
- 加载 effect 依赖 `currentStoreId`。
- 门店切换取消旧请求，防止串店数据覆盖。
- 接入前端 SWR 缓存。
- 首屏保留上一版内容，刷新时不清空列表。

验收：

- 不再出现缺少 `X-Store-Id` 的推荐请求。
- 硬刷新后能够自动加载当前门店数据。
- 返回页面时 300ms 内展示缓存卡片。

### 阶段二：P1 轻量首屏与 SQL 收敛

目标：降低冷读响应和浏览器渲染成本。

- 增加轻量 summary DTO。
- 默认首屏 20 条。
- 完整详情改为按需读取。
- 第一次列表查询取得实例 ID 后复用 ID 查询采纳状态，不重复分页筛选。
- 覆盖率结果按门店和 predictionRunId 缓存。
- 列表查询显式 select 字段，禁止加载无用大 JSON。
- 卡片列表使用渐进渲染或虚拟列表。

验收：

- 冷启动首屏 P95 小于 800ms。
- workspace 压缩后响应体小于 200KB。
- 首屏不请求客户受众和完整卡片详情。

### 阶段三：P1 工作台读模型与共享缓存

目标：让读取成本不再随客户量和算法复杂度增长。

- 新增 workspace snapshot schema 和 migration。
- 编排完成后生成读模型。
- 原子激活 revision。
- 增加共享缓存与 revision 失效机制。
- 保留上一版快照作为失败兜底。

验收：

- 缓存命中 API P95 小于 200ms。
- 数据库快照读取 API P95 小于 500ms。
- 多实例部署返回同一 revision。
- 刷新失败时页面继续展示上一版并标注失败状态。

### 阶段四：P2 异步刷新与生产观测

目标：刷新动作不阻塞页面，并形成可运营门禁。

- 新增刷新任务表、Worker、租约和重试机制。
- 页面展示刷新进度。
- 增加刷新耗时、缓存命中率、快照年龄、失败率监控。
- 将性能指标纳入 Release C 观察门禁。

验收：

- 点击刷新后 300ms 内返回任务状态。
- 同门店重复刷新只运行一个任务。
- 页面离开后任务继续执行。
- 新快照完成后页面自动切换 revision。

## 六、关键性能与正确性指标

| 指标 | 目标 |
| --- | --- |
| 有前端缓存的返回页面可见时间 | 小于 300ms |
| 冷启动推荐摘要 P95 | 小于 800ms |
| 服务端共享缓存命中 P95 | 小于 200ms |
| 数据库快照冷读 P95 | 小于 500ms |
| 首屏主请求数 | 1 个推荐摘要请求；次要模块独立加载 |
| 首屏推荐响应体 | 小于 200KB |
| 正常进入触发预测次数 | 0 |
| 缺少 `X-Store-Id` 的营销请求 | 0 |
| 同门店同日并发刷新任务 | 最大 1 个 |
| 跨门店数据串读 | 0 |
| 刷新失败导致页面清空 | 0 |

## 七、最终建议

综合最优实施组合为：

1. 立即完成门店就绪门禁和前端 SWR，这是正确性修复，不等待后端重构。
2. 同一迭代完成轻量首屏 DTO、字段裁剪和重复查询消除。
3. 上线放量前完成 workspace snapshot 与共享缓存，revision 作为唯一数据版本。
4. 将手动刷新改为异步任务，但保留上一版快照作为全程兜底。
5. 发布或自动触达前继续执行实时承接校验，避免快照数据直接驱动高风险动作。

该组合没有功能冲突，能够同时覆盖：重复进入慢、冷启动错误、查询成本高、多实例缓存不一致、刷新等待长和推荐数据过期六类问题。
