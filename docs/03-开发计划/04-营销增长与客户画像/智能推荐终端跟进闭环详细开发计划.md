# 智能推荐终端跟进闭环详细开发计划

更新时间：2026-06-15

执行状态：本期 MVP/P0/P1 闭环已完成，补齐推荐跟进记录、人员维度表现和逾期标记；复杂负载均衡、企微/短信真实触达属于后续增强。

## 1. 背景与目标

智能营销推荐页当前已经提供“发一次活动”“开启自动触达”“下发终端跟进”等动作，但“下发终端跟进”的业务边界还不够清晰：

- 什么情况下必须由门店人员跟进，而不是直接发活动或自动触达。
- 当前推荐算法如何判断某条建议适合终端跟进。
- 任务下发后由谁处理，如何确定给店长、顾问/美容师或前台。
- 终端侧如何看到任务、完成任务，以及如何回写转化闭环。

本计划目标是把“下发终端跟进”从一个通用按钮，升级为可解释、可分派、可执行、可复盘的经营闭环能力：

```text
智能推荐生成机会 -> 判断是否需要人工跟进 -> 生成跟进任务 -> 分派到角色/人员 -> 终端执行 -> 完成记录 -> 预约/订单/会员转化归因 -> 效果分析
```

## 2. 当前实现现状

### 2.1 管理端推荐页

当前页面：`src/app/pages/MarketingRecommendation.tsx`

已具备能力：

- 推荐卡读取 `executionModes`、`preferredMode`、`recommendedChannels` 等字段。
- 推荐卡可以跳转创建自动触达。
- 推荐卡可以生成小程序活动预览。
- 点击“下发终端跟进”后，会加载目标客户列表，默认选择前 10 位客户。
- 提交时对每位客户调用 `createTerminalFollowUpTask()`。
- 若推荐有真实 `recommendationId`，同时记录 `recordTerminalRecommendationEvent({ eventType: 'accepted' })`。

主要问题：

- “下发终端跟进”按钮当前在每张推荐卡上都展示，没有严格根据 `executionModes.includes('advisor_task')`、`recommendedChannels` 或风险信号做显隐和排序。
- 弹窗可以选择客户，但还没有推荐“应该由谁跟进”的角色、人员、截止时间和分派理由。
- 当前任务创建后，管理端缺少统一的任务状态看板，不能清楚看到已下发、待处理、已完成、逾期和转化。

### 2.2 后端任务创建与完成

当前后端：`packages/server-v2/src/terminal/terminal.service.ts`

已具备能力：

- `createFollowUpTask(storeId, deviceId, dto)` 校验客户后创建 `RecommendationEvent`。
- 创建事件类型为 `follow_up_created`。
- `payload` 中保存 `status: 'pending'`、`dueAt`、`channel`、脚本和备注等信息。
- `completeFollowUpTask(storeId, id, dto)` 根据 `follow_up_created` 查找任务，并新增 `follow_up_completed` 事件。

主要问题：

- 当前使用 `RecommendationEvent` 兼作任务存储，没有独立任务表，任务状态需要通过事件和 payload 推导。
- 缺少终端待办查询接口，例如按门店、角色、人员、状态、到期时间查询待跟进任务。
- 缺少改派、取消、逾期、批量创建、重复下发拦截等任务生命周期能力。
- `completeFollowUpTask` 是追加完成事件，但原始 `follow_up_created` 事件 payload 不会被更新为 completed，列表侧需要额外聚合才能准确判断状态。

### 2.3 终端侧承接

当前终端：`packages/Ami-Aura-Lite-Kiosk`

已具备能力：

- 终端已有客户增长候选卡片和客户机会展示。
- `auraCoreService.getCustomerGrowthCandidates()` 会读取客户增长候选、增长看板或客户排序。

主要问题：

- 终端客户增长卡片当前主要展示算法候选客户，不是管理端下发的 `follow_up_created` 待办任务。
- 店长、顾问/美容师、前台还没有各自的“待跟进任务队列”。
- 终端侧缺少任务完成操作和结果回填入口，例如已联系、已预约、未接通、拒绝、已成交。

### 2.4 推荐算法已有信号

当前智能推荐已有可复用信号：

- `executionModes`：推荐可执行方式，例如 `activity`、`automation`、`advisor_task`。
- `preferredMode`：推荐首选执行方式。
- `recommendedChannels`：推荐渠道，例如小程序、短信、顾问跟进。
- `urgency / urgencyLabel`：紧急、推荐、机会等优先级。
- `source`：推荐来源，例如客户、商品、项目、库存、产能。
- `triggerType`：触发类型，例如临期库存、低峰产能、护理周期、流失唤醒。
- `targetCustomers`、`matchScore`、`riskWarnings`、`inventorySnapshot`、`capacitySnapshot` 等经营证据。

商品/项目推荐中已经出现明确人工跟进场景：

- 临期商品剩余窗口过短时，`executionModes` 为 `['advisor_task']`，`preferredMode` 为 `advisor_task`。
- 临期库存和低峰产能推荐中，`recommendedChannels` 包含 `{ channel: 'store', label: '顾问跟进' }`。

因此，MVP 不需要重新发明一套判断规则，应先把已有推荐信号接入按钮显隐、任务分派和闭环状态。

## 3. 产品定义：什么情况下需要下发终端跟进

“下发终端跟进”不是所有推荐的默认动作，只适用于需要人工判断、人工解释、人工确认或门店现场协作的场景。

### 3.1 必须下发终端跟进

满足任一条件时，推荐卡应把“下发终端跟进”作为主动作或强推荐动作：

| 条件 | 业务含义 | 示例 |
| --- | --- | --- |
| `executionModes` 包含 `advisor_task` 且 `preferredMode = advisor_task` | 算法判断不适合直接群发，需要人工处理 | 临期只剩 7 天内的商品、敏感客诉风险客户 |
| `recommendedChannels` 包含 `store` 且优先级为紧急 | 需要门店人员一对一跟进 | 高流失风险、高价值客户维护 |
| 存在合规、客诉、库存解释风险 | 需要顾问解释权益边界 | 临期商品、次卡到期、护理套餐调整 |
| 需要确认客户意向后再创建预约/订单 | 直接自动触达转化率低或风险高 | 高客单护理邀约、沉睡客户唤醒 |
| 批量经营风险需要门店管理者分派 | 单个顾问无法直接处理，需要店长协调 | 大量客户流失、低峰排期空档、库存集中临期 |

### 3.2 建议下发终端跟进

满足任一条件时，推荐卡可展示“下发终端跟进”为次动作：

| 条件 | 业务含义 | 示例 |
| --- | --- | --- |
| `executionModes` 同时包含 `activity` 和 `advisor_task` | 可发活动，也可人工跟进重点客户 | 低峰护理活动、高响应客户复购 |
| `recommendedActions` 包含 `consultant_task` | 推荐动作中已有顾问任务 | 老客复购提醒、重点客户回访 |
| 客户数不大但客单价高 | 一对一触达比群发更合适 | 高 LTV 客户维护 |
| 自动触达失败或未响应 | 需要人工二次跟进 | 小程序浏览未预约、优惠券未核销 |

### 3.3 不建议下发终端跟进

满足以下条件时，推荐卡默认不展示该按钮，或收纳到“更多操作”：

| 条件 | 推荐动作 |
| --- | --- |
| `executionModes` 只有 `automation` | 优先开启自动触达 |
| `executionModes` 只有 `activity` 且没有门店跟进渠道 | 优先生成活动 |
| 客户数量大、客单价低、无风险解释需求 | 小程序/短信批量触达 |
| 系统推荐是样例数据或证据不足 | 先提示补全数据，不下发真实任务 |

## 4. MVP 范围

### 4.1 本期必须完成

- 推荐卡基于规则决定是否展示“下发终端跟进”。
- 推荐卡显示“为什么建议/不建议终端跟进”。
- 下发弹窗增加分派预览：跟进角色、建议人员、截止时间、分派原因。
- 后端补齐待办查询接口，终端能读取管理端下发的待跟进任务。
- 终端新增角色化任务队列：店长、顾问/美容师、前台。
- 终端支持完成任务并回写结果。
- 管理端能看到任务状态：待处理、已完成、逾期、已转化。
- 效果分析能统计下发数、完成数、预约数、成交数、成交金额。

### 4.2 本期不做

- 不做复杂 AI 自动派单，只做规则派单和可解释原因。
- 不做跨门店任务流转。
- 不做复杂 CRM 工单系统。
- 不做微信/企微真实消息下发，只先打通终端任务闭环。
- 不强制新增完整任务协同系统，但要为后续扩展保留字段。

## 5. 方案选择

### 5.1 推荐方案：新增 `TerminalFollowUpTask` 独立任务表

虽然当前可以继续使用 `RecommendationEvent` 作为任务来源，但任务天然具有状态、负责人、截止时间、逾期、改派、取消等生命周期。继续只用事件表会导致查询和状态聚合越来越复杂。

建议新增独立任务表，并保留 `RecommendationEvent` 作为行为流水和效果归因。

关系如下：

```text
MarketingRecommendation
  -> TerminalFollowUpTask（任务当前状态）
  -> RecommendationEvent（创建、接单、完成、转化等行为流水）
```

### 5.2 兼容策略

- 新下发任务写入 `TerminalFollowUpTask`。
- 同时继续写入 `RecommendationEvent`，保持现有效果分析和历史事件兼容。
- 历史 `follow_up_created` 事件可作为只读历史任务展示，不强制迁移。
- 如果上线周期非常紧，可先用 `RecommendationEvent` 查询待办，但开发计划按独立任务表设计，避免二次返工。

## 7. 算法与规则设计

### 7.1 是否展示“下发终端跟进”

新增前端/后端共享规则函数：

```ts
shouldShowTerminalFollowUp(recommendation)
```

判断规则：

1. `executionModes.includes('advisor_task')`：展示。
2. `preferredMode === 'advisor_task'`：展示且设为主动作。
3. `recommendedChannels.some(channel => channel.channel === 'store')`：展示。
4. `recommendedActions.some(action => action.type === 'consultant_task')`：展示。
5. `urgency === 'urgent'` 且 `source` 属于 `customer | inventory | capacity | project`：展示。
6. `riskWarnings.length > 0`：展示。
7. 样例推荐、证据不足、无目标客户：禁用并提示原因。

输出结构：

```ts
{
  visible: boolean;
  primary: boolean;
  disabled: boolean;
  reason: string;
  confidence: number;
}
```

### 7.2 跟进优先级评分

用于任务排序和默认截止时间：

```text
score =
  urgencyScore
  + preferredModeScore
  + sourceRiskScore
  + customerValueScore
  + dueWindowScore
  + responseIntentScore
```

评分维度：

| 维度 | 规则 |
| --- | --- |
| 紧急度 | 紧急 +40，推荐 +25，机会 +10 |
| 执行方式 | 首选顾问跟进 +25，包含顾问跟进 +15 |
| 风险 | 有临期、流失、客诉、库存风险 +20 |
| 客户价值 | 高 LTV、高消费频次、会员权益有效 +15 |
| 时间窗口 | 7 天内到期 +20，14 天内 +10 |
| 行为意向 | 浏览/点击/预约放弃等近期行为 +10 |

排序规则：

- 默认按 `score desc`。
- 同分时按 `dueAt asc`。
- 再按客户价值、最近互动时间排序。

### 7.3 默认截止时间

| 场景 | 截止时间 |
| --- | --- |
| 临期商品 7 天内 | 当日 20:00 前 |
| 高流失风险客户 | 48 小时内 |
| 高 LTV 客户维护 | 72 小时内 |
| 低峰排期填充 | 空闲时段前 24 小时 |
| 预约放弃/小程序高意向浏览 | 2 小时内 |
| 次卡/套餐即将到期 | 到期前 3 天 |

## 8. 分派规则：谁来跟进

### 8.1 角色定义

| 角色 | 适合处理 |
| --- | --- |
| 店长 | 批量经营风险、跨人员协调、库存/产能策略、高价值客户兜底 |
| 顾问/美容师 | 客户关系维护、护理建议、项目解释、次卡/套餐续费 |
| 前台 | 预约确认、到店提醒、未接通重拨、基础电话邀约 |

### 8.2 默认分派规则

| 场景 | 默认跟进人 | 分派理由 |
| --- | --- | --- |
| 客户有专属顾问/常服务美容师 | 顾问/美容师 | 关系最熟，转化率最高 |
| 高流失风险客户 | 顾问/美容师，店长可见 | 需要情感维护和服务解释 |
| 高 LTV 客户 | 顾问/美容师，紧急时店长兜底 | 避免普通群发影响体验 |
| 预约放弃/预约提醒 | 前台 | 更接近预约确认流程 |
| 小程序高意向浏览未预约 | 前台或顾问 | 前台先确认时间，顾问补充项目说明 |
| 临期库存 7 天内 | 店长 | 需要控制权益口径和客诉风险 |
| 临期库存 7 天以上 | 店长分派给顾问 | 店长确定策略，顾问触达客户 |
| 低峰排期填充 | 店长或前台 | 店长协调产能，前台确认到店 |
| 次卡/套餐即将到期 | 顾问/美容师 | 结合历史服务体验做续费/核销提醒 |
| 无明确归属客户 | 店长队列 | 店长手动分派 |

### 8.3 人员匹配优先级

顾问/美容师匹配顺序：

1. 客户档案中的专属顾问。
2. 最近一次服务美容师。
3. 最近一次开卡/成交员工。
4. 客户最近预约服务人员。
5. 当前门店可接待且负载最低的顾问/美容师。
6. 无匹配时进入店长待分派队列。

前台匹配顺序：

1. 当前班次前台。
2. 门店默认前台账号。
3. 无前台账号时进入店长待分派队列。

店长匹配顺序：

1. 当前门店店长。
2. 当前门店超级管理员。
3. 无店长时进入“未分派任务”队列。

### 8.4 改派与升级

需要支持：

- 店长可把任务改派给顾问/美容师或前台。
- 任务逾期后自动升级给店长。
- 高价值客户任务逾期后标记为“需店长介入”。
- 顾问无法处理时可退回店长队列，并填写原因。

## 17. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |

| 任务完成后未产生订单但后续转化 | 归因不完整 | 支持 7 天辅助归因并标明口径 |

## 19. 推荐实施顺序

建议按以下顺序执行，避免先做展示但无法闭环：

1. 先做推荐卡动作规则，让“下发终端跟进”只在合适场景出现。
2. 再做独立任务模型和批量创建，解决任务状态真相源。
3. 然后打通终端待办查询，让任务真正到人。
4. 再做完成结果和转化归因，形成闭环。
5. 最后补工作台看板、逾期升级和人员绩效。

这样可以保证每个阶段都有可演示价值，并且不会把“下发终端跟进”停留在管理端按钮层。
