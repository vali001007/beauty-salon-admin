# 客户画像驱动权益资产库营销闭环详细开发计划

更新时间：2026-06-15

关联模块：

- 客户管理 / 客户画像
- 智能营销 / 智能推荐
- 智能营销 / 活动列表
- 智能营销 / 自动触达
- 智能营销 / 规则库
- 智能营销 / 营销资产 / 优惠权益
- Ami Glow 客户服务小程序
- Ami Aura Lite 终端跟进

## 1. 背景与问题

当前系统已经具备客户画像、智能推荐、自动触达、活动列表和优惠权益等模块，但三类能力之间仍偏松散：

1. `优惠权益` 已有独立数据模型 `Promotion`，但更像单独配置页，尚未成为活动和自动触达的统一权益资产来源。
2. `活动列表` 的优惠内容主要保存在 `MarketingActivity.discount` / `offerJson`，当前创建活动时仍以手填权益文案为主。
3. `自动触达` 的触达动作保存在 `MarketingAutomationStrategy.actions`，当前动作值也是文本，未强关联权益资产。
4. `智能推荐` 已能生成推荐权益 `offer`，但该权益更多是推荐结果中的结构化建议，没有沉淀为可核销、可复盘、可复用的权益资产。
5. 客户画像分析能判断客户机会，但还没有形成稳定的“从权益库选择权益，或生成待审核权益草稿”的闭环。

这会导致运营人员在活动、自动触达、推荐卡之间重复填写优惠内容，后续也难以回答：

- 哪类客户用了哪类权益？
- 哪个权益带来了预约、核销、成交？
- 高价值客户是否被过度低价促销？
- 自动触达和活动列表是否在使用同一套权益？
- 小程序、终端、活动页展示的权益是否一致？

## 2. 产品目标

本次开发目标是将“优惠权益”升级为“权益资产库”，并让客户画像分析和智能推荐围绕权益资产形成闭环。

核心目标：

- 将权益资产库作为营销活动和自动触达的统一权益来源。
- 智能推荐不直接上线权益，而是推荐可选权益或生成待审核权益草稿。
- 活动列表和自动触达都能选择权益资产，减少重复填写优惠文案。
- 客户画像分析结果能影响权益匹配，如流失风险、LTV、促销敏感度、肤质、项目偏好、护理周期等。
- 投放后将浏览、点击、领取、预约、核销、成交回流到权益和推荐效果复盘。

目标链路：

```text
客户画像/行为/消费信号
  -> 智能推荐机会
  -> 匹配权益资产或生成权益草稿
  -> 创建活动 / 创建自动触达 / 下发终端跟进
  -> 小程序、活动页、终端承接
  -> 浏览、点击、预约、核销、成交回流
  -> 权益效果、推荐效果、客户画像更新
```

## 3. 产品边界

### 3.1 权益资产库是什么

权益资产库是门店可投放、可展示、可核销、可复盘的权益货架。

包括：

- 优惠券类：满减、折扣、体验价、现金券。
- 礼遇类：赠品、赠护理、生日礼、VIP 专属服务。
- 卡项类：续卡赠次、次卡到期提醒权益、套餐升级权益。
- 预约类：低峰预约礼、护理周期预约权益。
- 商品/项目类：指定项目体验权益、商品组合权益。

### 3.2 客户画像分析负责什么

客户画像分析不直接创建正式权益，不直接自动投放。

客户画像分析负责：

- 判断客户处于哪个营销机会场景。
- 计算客户是否适合权益触达。
- 推荐权益类型、力度、有效期、渠道。
- 从权益库中匹配合适权益。
- 当无合适权益时，生成待审核权益草稿。

### 3.3 活动列表负责什么

活动列表是一次性营销战役管理。

适合：

- 节日营销。
- 季节换肤。
- 老带新。
- 新项目体验。
- 高 LTV 客户维护主题活动。

活动可以关联一个或多个权益资产，但活动本身还包含主题、目标人群、时间、页面、素材和效果数据。

### 3.4 自动触达负责什么

自动触达是长期规则策略管理。

适合：

- 优惠券即将到期。
- 次卡/套餐即将到期。
- 护理周期到期。
- 浏览未预约。
- 领券未核销。
- 预约放弃。
- 沉睡客户唤醒。

自动触达可以关联权益资产，但它的核心是触发规则、频控、渠道和执行闭环。

## 4. 目标数据关系

### 4.1 总体关系

```text
Promotion 权益资产
  <- MarketingActivityPromotion 活动使用权益
  <- MarketingAutomationAction.promotionId 自动触达使用权益
  <- MarketingRecommendationOffer 推荐权益匹配结果
  <- CustomerAppEvent / MarketingPageEvent / MarketingAutomationTouch 效果回流
```

### 4.2 关系定义

| 对象 | 定位 | 与权益资产关系 |
| --- | --- | --- |
| `Promotion` | 权益资产库 | 主数据，承载权益定义、适用范围和状态 |
| `MarketingActivity` | 一次性活动 | 活动可选择权益，保留活动自己的主题和落地页 |
| `MarketingAutomationStrategy` | 自动触达策略 | 动作可选择权益，按规则触达客户 |
| `MarketingRuleTemplate` | 自动营销规则模板 | 可配置默认推荐权益类型或默认权益模板 |
| `MarketingRecommendation` | 智能推荐机会 | 输出推荐权益、匹配理由和推荐执行方式 |
| `CustomerPredictionSnapshot` | 客户画像预测快照 | 提供 LTV、流失、复购、响应、原因和建议动作 |
| `CustomerAppEvent` | 小程序行为流水 | 回流浏览、点击、预约、领取等事件 |
| `MarketingAutomationTouch` | 自动触达客户级记录 | 回流触达、响应、转化 |

## 5. MVP 范围

### 5.1 包含

- 将页面文案从“优惠权益”升级为“权益资产库”。
- 扩展 `Promotion` 模型，让它能表达权益类型、适用人群、核销约束、成本和推荐场景。
- 活动创建支持选择已有权益资产。
- 自动触达动作支持选择已有权益资产。
- 智能推荐卡支持展示“推荐权益资产”或“建议新建权益草稿”。
- 从推荐卡创建活动或自动触达时，带入 `promotionId` 或待审核权益草稿。
- 权益资产库支持系统内置默认权益模板。
- 效果分析支持按权益维度查看曝光、点击、预约、核销、成交。

### 5.2 不包含

- 不做复杂优惠券核销账务系统。
- 不做完整会员账户/积分账户重构。
- 不做 AI 自动创建正式权益并直接上线。
- 不做复杂权益叠加规则引擎。
- 不做跨渠道营销预算自动分配。
- 不重构现有收银、订单、卡项账务主链路。

## 6. 分阶段开发计划

建议按 4 个阶段推进，总周期约 12 到 18 个开发日。

| 阶段 | 周期 | 目标 | 可验收结果 |
| --- | --- | --- | --- |
| 阶段 1 | 3 到 4 天 | 权益资产模型升级和默认权益模板 | 权益资产库可管理结构化权益 |
| 阶段 2 | 3 到 5 天 | 活动和自动触达选择权益 | 活动/触达不再只能手填优惠文案 |
| 阶段 3 | 3 到 5 天 | 智能推荐匹配权益资产 | 推荐卡能带出权益并生成活动/触达草稿 |
| 阶段 4 | 3 到 4 天 | 效果回流和验收 | 可按权益维度复盘曝光、点击、预约、核销、成交 |

## 7. 阶段 1：权益资产库升级

### 7.1 后端数据模型

建议新增字段：

| 字段 | 类型 | 说明 |

| `thresholdAmount` | Decimal? | 满减门槛 |

## 8. 阶段 2：活动和自动触达选择权益

### 8.1 活动列表改造

目标：创建活动时优先选择权益资产，而不是手填优惠文案。

数据模型建议：

轻量 MVP 可先在 `MarketingActivity` 增加：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `promotionIdsJson` | Json? | 活动关联权益 ID 列表 |
| `primaryPromotionId` | Int? | 主推权益 |

更规范方案可新增关联表：

```prisma
model MarketingActivityPromotion {
  id          Int      @id @default(autoincrement())
  activityId  Int
  promotionId Int
  role        String   @default("primary")
  sortOrder   Int      @default(0)
  createdAt   DateTime @default(now())
}
```

MVP 推荐先用 `promotionIdsJson` 和 `primaryPromotionId`，降低迁移和页面改造成本。

前端调整：

- 在活动创建第一步增加 `选择权益`。
- 默认展示智能推荐带入的权益。
- 支持从权益资产库搜索选择。
- 支持“没有合适权益，创建权益草稿”。
- 活动预览页的“本期权益”从选中权益生成。
- 活动详情展示权益来源和权益状态。

兼容策略：

- 保留 `discount` 字段作为展示快照。
- 当选择权益时，`discount = promotion.discountText`。
- 老活动没有 `promotionId` 时，继续展示原 `discount` 文案。

### 8.2 自动触达改造

目标：自动触达动作支持绑定权益资产。

类型调整：

```ts
interface MarketingAction {
  type: 'coupon' | 'discount' | 'gift' | 'points' | 'sms' | 'push' | 'wechat' | 'miniapp';
  value: string;
  promotionId?: number;
  promotionName?: string;
  channel?: 'sms' | 'miniapp' | 'wechat' | 'group' | 'store' | 'moments';
  contentTemplate?: string;
}
```

页面调整：

- 动作配置增加 `选择权益资产`。
- 选择权益后自动填充动作类型、权益内容和文案变量。
- 支持 `自定义权益文案`，但需要提示“不会进入权益核销和权益效果复盘”。
- 规则库启用策略时，如果模板有默认权益场景，自动匹配权益。
- 从智能推荐创建自动触达草稿时，带入推荐权益。

## 9. 阶段 3：智能推荐匹配权益资产

### 9.1 推荐生成逻辑

在智能推荐生成时增加权益匹配步骤。

推荐卡生成流程：

```text
识别机会场景
  -> 计算目标客户画像
  -> 判断执行方式 activity / automation / both
  -> 调用权益匹配
  -> 匹配已有权益或生成权益草稿建议
  -> 输出推荐卡
```

推荐卡字段建议：

```ts
interface RecommendedOffer {
  type: string;
  label: string;
  validDays?: number;
  reason?: string;
  promotionId?: number;
  promotionName?: string;
  fitScore?: number;
  fitReason?: string;
  draftSuggestion?: {
    name: string;
    type: string;
    discountText: string;
    reason: string;
  };
}
```

### 9.2 权益匹配规则

匹配评分建议：

```text
offerFitScore =
  scenarioMatch * 0.25
  + audienceMatch * 0.20
  + projectMatch * 0.15
  + customerValueFit * 0.15
  + costGuardFit * 0.10
  + validityFit * 0.10
  + availabilityFit * 0.05
```

评分解释：

| 维度 | 说明 |
| --- | --- |
| `scenarioMatch` | 权益场景是否匹配推荐类型 |
| `audienceMatch` | 权益适用人群是否匹配客户画像 |
| `projectMatch` | 权益适用项目是否匹配客户偏好和推荐项目 |
| `customerValueFit` | 高价值客户是否避免低价促销，流失客户是否权益力度足够 |
| `costGuardFit` | 是否满足毛利保护 |
| `validityFit` | 有效期是否适合转化窗口 |
| `availabilityFit` | 权益是否在线、未过期、未超发 |

### 9.3 不同画像的权益策略

| 客户画像 | 推荐权益策略 | 不建议 |
| --- | --- | --- |
| 高流失风险 | 明确回店利益，力度可稍强，但保留门槛 | 泛泛问候、无明确行动 |
| 高 LTV | 专属礼遇、优先预约、顾问服务、升级体验 | 直接大额折扣 |
| 护理周期到期 | 小额预约权益、项目组合、护理建议 | 大促销、无项目关联 |
| 新客未首单 | 首护体验价、顾问咨询、低门槛预约 | 高价套餐硬推 |
| 浏览未预约 | 短有效期体验券、浏览项目关联权益 | 长期泛券 |
| 领券未核销 | 提醒已领权益和适配项目 | 再发一张无关新券 |
| 次卡到期 | 剩余权益提醒、续卡赠次、升级套餐 | 单纯低价打折 |
| 敏感肌/肤质明确 | 对应护理项目权益和专业说明 | 与肤质无关的促销 |

### 9.4 推荐卡交互调整

推荐卡增加权益区域：

- 推荐权益名称。
- 权益内容。
- 权益来源：权益资产库 / 推荐生成草稿。
- 适配理由。
- 权益风险提示，如“高价值客户不建议使用大额折扣”。

按钮动作：

- `创建活动`：带入权益。
- `创建自动触达`：带入权益。
- `查看权益`：打开权益资产详情。
- `生成权益草稿`：当无合适权益时创建待审核权益。

- 高 LTV 客户默认不推荐大额低价券，优先服务礼遇。

- 超过发放上限的权益不能继续发放。
- 权益草稿不能直接投放，必须审核通过。
- 自定义权益文案可以保存，但需要标识为“非资产权益”，不进入核销复盘。

## 16. 风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 权益与真实核销链路尚未完全打通 | 早期 ROI 可能不准确 | MVP 先做预约/成交归因，核销后续增强 |
| 活动和自动触达已有大量文本权益 | 老数据无法完整关联权益 | 保留文本快照，旧数据归类为未关联权益 |
| 权益模型过重 | 开发周期变长 | 第一阶段只做必要字段，财务级核销后置 |
| AI 推荐权益可能不符合门店成本 | 造成过度让利 | 推荐生成草稿，必须人工审核 |
| 高价值客户被低价促销 | 损害品牌体验 | 内置高 LTV 权益保护规则 |
| 权益过期后仍被策略使用 | 客户体验和客诉风险 | 自动触达执行前校验权益状态和有效期 |

## 17. 推荐实施顺序

优先做 MVP 闭环，不先追求复杂优惠券系统。

推荐顺序：

1. 权益资产库改名和基础字段扩展。
2. 内置默认权益模板。
3. 活动创建支持选择权益。
4. 自动触达动作支持选择权益。
5. 智能推荐卡匹配权益资产。
6. 推荐卡创建活动/自动触达时带入权益。
7. 效果分析增加权益维度。
8. 后续再补完整领取、核销、财务和库存成本约束。

## 18. 最小可交付版本定义

最小可交付版本必须满足：

- 运营能在权益资产库维护权益。
- 活动和自动触达能选择同一个权益资产。
- 智能推荐能推荐一个权益或生成一个权益草稿。
- 权益被活动或自动触达使用后，能在效果分析里看到基础表现。

若只能选择一个阶段优先开发，建议先做：

```text
权益资产库结构化 + 活动/自动触达选择权益
```

原因是这一步能立即消除“活动列表、自动触达、优惠权益各自手填”的重复问题，也为后续画像推荐权益打下稳定数据基础。
