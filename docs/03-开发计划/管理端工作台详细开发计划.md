# 管理端工作台详细开发计划

更新时间：2026-06-14  
关联方案：`docs/03-开发计划/管理端工作台改造方案.md`  
目标范围：管理端首页 `/dashboard`、工作台角色识别、关键数据、待办事项、快捷入口、后端工作台聚合接口  
开发原则：保留 `/dashboard` 兼容历史入口；不删除旧接口；前后端分阶段上线；所有入口按权限过滤；后端按用户、角色、门店控制数据范围。

## 1. 交付目标

把管理端首页从“仪表盘”改成“工作台”，并做到：

1. 用户登录后看到自己角色最相关的数据。
2. 用户能看到今天最需要处理的待办事项。
3. 用户可以从工作台直接进入高频操作。
4. 多角色用户能切换当前工作台视角。
5. 无权限的指标、待办、快捷入口不展示。
6. `/dashboard` 路由、登录跳转、错误页返回入口继续可用。

第一版交付建议分成两个代码阶段：

| 阶段 | 目标 | 后端改动 | 前端改动 | 推荐优先级 |
| --- | --- | --- | --- | --- |
| 阶段 1 | 快速完成“仪表盘到工作台”的体验改造 | 不新增接口，只兼容现有数据 | 菜单改名、组件拆分、角色配置、权限过滤 | 必做 |
| 阶段 2 | 完成真正按用户和角色聚合 | 新增 `GET /dashboard/workbench` | 切换到新接口，支持角色切换 | 必做 |
| 阶段 3 | 待办闭环和个性化 | 可选新增待办状态、收藏配置 | 最近访问、固定入口、AI 建议增强 | 后续 |

### 3.1 工作台布局

```text

  4-8 个高频入口，按权限过滤

  终端在线、门店状态等轻量状态，仅有数据时展示
```

### 3.2 角色与默认优先级

```ts

  | 'super_admin'
```


### 3.3 角色工作重点

| 角色 | 关键数据 | 今日待办 | 快捷入口 |
| --- | --- | --- | --- |
| `super_admin` | 今日总营收、异常门店、在线终端、系统配置 | 门店异常、设备离线、跨店库存风险、权限配置缺失 | 门店管理、角色权限、平台收入、设备管理 |
| `store_manager` | 今日收入、今日预约、待服务任务、库存预警 | 待确认预约、服务积压、库存风险、营销机会 | 项目预约、排班、库存、营销工作台、日结 |
| `cashier` | 今日预约、待到店、待核销、今日收银 | 即将到店、已到店未开单、待核销、客户资料缺失 | 预约、客户登记、次卡核销、商品订单、开卡 |
| `beautician` | 我的预约、待完成服务、待补记录、我的提成 | 即将开始服务、进行中服务、未补服务记录、护理建议 | 我的排班、客户画像、服务记录、护理建议、提成 |
| `inventory_manager` | 低库存、临期、采购待处理、调拨待确认 | 低于安全线、30 天内临期、采购待入库、调拨待确认 | 库存、采购、过期、调拨、服务消耗 |
| `default` | 可访问模块摘要 | 可访问模块待办 | 用户有权限的入口 |

### 任务 1.3：新增管理端角色解析工具

目标：不要继续直接依赖 `resolveAuraRole(user)` 作为管理端工作台角色。

新增文件：

```text

```

核心函数：

```ts

export function resolveDefaultWorkbenchRole(user: AuthUser | null): AdminWorkbenchRole {}

```

解析规则：

1. `roles` 包含 `super_admin` -> 可用 `super_admin`、`store_manager`、`inventory_manager`、`cashier`、`beautician`。
2. `roles` 包含 `store_manager` -> 可用 `store_manager`、`cashier`、`beautician`。
3. `roles` 包含 `inventory_manager` -> 可用 `inventory_manager`。
4. `roles` 包含 `cashier` -> 可用 `cashier`。
5. `roles` 包含 `beautician` -> 可用 `beautician`。
6. 无匹配角色 -> `default`。

补充规则：

1. 如果用户权限含 `*`，按 `super_admin` 处理。
2. 如果角色缺失但权限包含 `core:inventory:stock`，可补充 `inventory_manager`。
3. 如果角色缺失但权限包含 `core:order:card-usage`，可补充 `cashier`。
4. 如果角色缺失但权限包含 `core:store:scheduling` 且不含订单权限，可补充 `beautician`。

验收：

1. 不同角色用户进入不同默认视角。
2. 多角色用户顶部可切换视角。
3. 未知角色不会报错，展示 `default`。

```ts
getWorkbench(

  @Headers('x-store-id') storeId?: string,
```

### 任务 2.3：解析用户、角色和门店范围

目标：后端知道“当前是谁、能看哪些角色、能看哪个门店”。

改动文件：

```text
packages/server-v2/src/dashboard/dashboard.service.ts
```

新增私有方法：

```ts
private resolveWorkbenchRoles(user: any): AdminWorkbenchRole[] {}
private resolveCurrentWorkbenchRole(availableRoles: AdminWorkbenchRole[], requestedRole?: string): AdminWorkbenchRole {}
private async resolveWorkbenchScope(user: any, storeId?: number, role?: AdminWorkbenchRole): Promise<WorkbenchScope> {}
```

门店校验：

1. `super_admin` 可以不传 `storeId`，表示全局。
2. 非 `super_admin` 必须限制在用户可访问门店。
3. 如果传入无权限门店，返回 403 或降级到用户默认门店，建议返回 403 更清晰。
4. 美容师视角 `scope.mode` 为 `self`，但仍保留所在门店。

验收：

1. 店长不能看无权限门店数据。

## 11. 数据口径确认清单

开发前需要确认或采用默认口径：

| 事项 | 默认口径 | 是否阻塞 |
| --- | --- | --- |
| 今日收入是否包含次卡开卡 | 包含商品订单 + 次卡/会员卡开卡收入 | 不阻塞 |
| 退款是否扣减今日收入 | 第一版沿用现有 activeOrderStatuses，不额外扣减 | 不阻塞 |
| 今日预约状态 | pending / confirmed / 待确认 / 已确认 / 进行中 | 不阻塞 |
| 待核销来源 | 优先 serviceTask pending/in_progress，后续细化次卡核销单 | 不阻塞 |
| 美容师本人关联 | 通过 `beautician.userId` | 阻塞美容师精准视角 |
| 库存临期天数 | 30 天 | 不阻塞 |
| 总部异常门店 | 库存预警、终端离线、无活跃数据 | 不阻塞 |

## 12. 风险和降级

| 风险 | 表现 | 降级策略 |
| --- | --- | --- |
| 后端聚合慢 | 首页加载慢 | 阶段 1 先静态配置；阶段 2 并行查询；必要时 30 秒缓存 |
| 部分模型字段不统一 | 某些指标算不准 | 指标展示“待接入”，不阻塞页面 |
| 用户角色配置不完整 | 进入 default 工作台 | 根据权限推断入口，避免白屏 |
| 权限过滤遗漏 | 点击后无权限 | 后端过滤 + 前端过滤双保险 |
| 美容师未绑定档案 | 查不到本人数据 | 显示绑定提示，保留可用入口 |
| 文案大面积替换误伤 | 终端/财务 dashboard 被误改 | 搜索后逐处确认，只改管理端首页语义 |

## 13. 最小可交付版本定义

1. 菜单从“仪表盘 / 数据概览”改成“工作台 / 我的工作台”。
2. 页面头部使用“工作台”语义。
3. 管理端角色解析不再直接依赖 Aura 三角色。
4. 至少覆盖 `super_admin`、`store_manager`、`cashier`、`beautician`、`inventory_manager`、`default`。
5. 指标、待办、快捷入口按权限过滤。
6. 接口失败不白屏。
7. `/dashboard` 路由保留。

MVP 可以暂不完成：

1. 新增 `/workbench` 路由。
2. 用户自定义快捷入口。
3. 完整任务中心。
4. AI 实时生成建议。
5. 复杂跨门店趋势图。

## 14. 预计工作量

按单人开发估算：

| 阶段 | 工作量 | 说明 |
| --- | --- | --- |
| 阶段 1 前端 MVP | 1.5-2.5 天 | 拆组件、配置、权限过滤、基础测试 |
| 阶段 2 后端接口 | 2-3 天 | 聚合查询、角色/门店校验、单测 |
| 阶段 2 前端接入 | 1-1.5 天 | 新 API、角色切换、fallback |
| 联调与修正 | 1 天 | 多角色、多门店、异常状态 |
| 合计 | 5.5-8 天 | 不含后续个性化和 AI 增强 |

## 16. 推荐执行结论

建议先执行 PR 1，把首页立即改成工作台体验，并把权限过滤和组件结构打稳。  
随后执行 PR 2 和 PR 3，把数据升级为真正按用户、角色、门店聚合。  
阶段 3 的个性化和 AI 建议不要混入第一轮开发，避免首页改造范围过大。
