# 需求热力图负荷程度分级详细开发计划

日期：2026-06-30
关联方案：`docs/03-开发计划/需求热力图负荷程度分级方案-2026-06-30.md`
适用模块：门店管理 / 排班管理 / 需求热力图 / 空档机会 / 店长跟进

## 1. 目标

本轮把“需求热力图”的主表达从“低客流 / 中客流 / 高客流”收敛为“低负荷 / 中负荷 / 高负荷”，让店长看到每个经营时段时，能直接判断下一步动作：

- 低负荷：人手富余，优先补单；补不满时考虑休假、调休或减少在岗。
- 中负荷：人力匹配，保持当前排班。
- 高负荷：人手不足，建议加人、加班或调班。

同时把空档机会和推荐客户从热力图格子中移出，统一放到右侧「空档推荐客户名单」，按周推荐客户并承接“客户详情”和“下发跟进”操作。

### 2.2 当前问题

- 热力图格子仍以“客流”作为颜色和文案主口径，不能表达“已排人力是否匹配需求”。
- 格子内混放预估客流、建议员工、已排员工、空档机会、推荐客户、补位率、预计收益，信息过载。
- 空档补位动作必须依赖格子点击，店长无法从周维度直接看到“本周最应该跟进谁”。
- 后端 `demand` 返回字段只有 `expectedReservations`、`requiredStaff`、`scheduledStaff`、`level`，缺少统一的 `staffDelta`、`loadRatio`、`loadLevel`、`recommendedAction`。

## 4. 产品口径

### 4.1 热力图格子

格子负责回答：这个时段人力是否匹配需求。

推荐展示：

```text
低负荷 / 中负荷 / 高负荷
预测需求：N
建议在岗：N
已排在岗：N
建议：补单 / 保持 / 加人
```

过去时段展示：

```text
已过期
```

空档机会、推荐客户、补位概率、预计收益不再出现在格子内。

### 4.2 图例与汇总

图例从客流口径调整为负荷口径：

```text
低负荷：人手富余，推荐补单或安排休假
中负荷：人力匹配，保持当前配置
高负荷：人手不足，推荐加人或加班
```

顶部汇总建议：

```text
高负荷时段：N
低负荷时段：N
可补单空档：N
推荐客户：N
预计补位收益：¥N
```

### 4.3 右侧空档推荐客户名单

右侧名单负责回答：本周应该跟进哪些客户补位。

默认按周聚合，不要求先点击格子。排序规则：

1. 补位概率高优先。
2. 预计收益高优先。
3. 推荐分高优先。
4. 同一客户如命中多个时段，默认保留最优一条，并可在详情里展示候选时段。

单条推荐字段：

```text
客户名称
补位概率
推荐原因
推荐项目
建议预约时段
客户详情
下发跟进
```

空状态：

```text
本周暂无可补位客户
```

加载状态：

```text
正在生成空档推荐...
```

失败降级：

```text
空档推荐加载失败，可继续查看负荷热力图
```

## 5. 数据口径

### 5.1 后端字段

在 `SchedulingDemandSlot` 上新增字段：

```ts
type DemandLoadLevel = 'low' | 'medium' | 'high';
type DemandRecommendedAction = 'fill_gap' | 'keep' | 'add_staff';

type SchedulingDemandSlot = {
  date: string;
  startTime: string;
  endTime: string;
  expectedReservations: number;
  requiredStaff: number;
  scheduledStaff: number;
  level: 'low' | 'medium' | 'high';
  staffDelta: number;
  loadRatio: number;
  loadLevel: DemandLoadLevel;
  recommendedAction: DemandRecommendedAction;
};
```

兼容策略：

- `level` 暂时保留，继续表达旧客流等级或旧热力等级。
- 新页面优先使用 `loadLevel` 和 `recommendedAction`。
- 如果后端暂未返回新字段，前端按 `scheduledStaff - requiredStaff` 兜底计算。

### 5.2 负荷计算

基础公式：

```text
staffDelta = scheduledStaff - requiredStaff
loadRatio = requiredStaff / max(scheduledStaff, 1)
```

分级：

```text
scheduledStaff > requiredStaff  => low
scheduledStaff = requiredStaff  => medium
scheduledStaff < requiredStaff  => high
```

动作：

```text
low    => fill_gap
medium => keep
high   => add_staff
```

特殊情况：

- `requiredStaff = 0` 且 `scheduledStaff > 0`：低负荷，建议补单或安排休假。
- `requiredStaff = 0` 且 `scheduledStaff = 0`：中负荷或休息时段，前端可展示“无需安排”。
- 过去时段：不预测、不推荐补位、不参与右侧名单。

### 5.3 三时段聚合

页面外显按三个经营时段展示：

- 上午：09:00~12:00
- 下午：14:00~18:00
- 晚上：18:00~20:00

聚合规则：

- `expectedReservations`：取时段内小时级求和或最大值。本轮建议保守采用求和，表达该经营时段总需求。
- `requiredStaff`：取时段内最大值，避免高峰小时被平均掉。
- `scheduledStaff`：取时段内最小值，反映最薄弱小时。
- `staffDelta`：聚合后重新计算。
- `loadLevel`：按聚合后的 `scheduledStaff` 和 `requiredStaff` 判定。
- `recommendedAction`：按聚合后的 `loadLevel` 判定。

## 9. 风险与处理

### 9.1 已排员工数口径风险

风险：热力图的 `scheduledStaff` 若没有严格读取原始排班可服务状态，会导致负荷等级错误。

处理：

- 后端继续以 `Schedule.status=available` 和有效预约口径统计。
- 兼容旧状态时只把可服务状态计为已排员工。
- 测试覆盖预约、请假、已过期、空闲状态。

### 9.2 空档推荐与负荷等级冲突

风险：高负荷时仍出现补单客户，误导店长继续加预约。

处理：

- 右侧名单可以展示全周推荐，但单条推荐需优先来自低负荷或中负荷时段。
- 高负荷时段的候选默认降权或标记“需先加人”。

### 9.3 数据缺失导致空名单

风险：候选客户依赖预测快照、客户卡、历史预约和触达记录，数据不足时名单为空。

处理：

- 保留降级推荐：使用客户卡项和历史预约生成候选。
- 空状态文案说明“暂无可补位客户”，不要显示为加载失败。

### 9.4 脏工作区风险

风险：当前仓库已有大量未提交改动，排班页、后端 scheduling、Prisma schema 都在改动中。

处理：

- 实施时只修改本计划列出的文件。
- 修改前再次确认 `git status --short --branch`。
- 不回退不相关文件。
- 如发现同一文件中存在用户未提交改动，基于当前内容增量修改，不做大范围重排。

6. 构建、测试、文档同步。
