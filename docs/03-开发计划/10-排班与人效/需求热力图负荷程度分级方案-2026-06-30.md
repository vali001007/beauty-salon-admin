# 需求热力图负荷程度分级方案

日期：2026-06-30

## 背景

当前“需求热力图”同时展示客流预测、人力建议、已排员工、空档机会和推荐客户，但颜色分级仍叫“低客流 / 中客流 / 高客流”。这会造成两个问题：

1. 店长看到“低客流”时，不知道是应该减少排班、安排休假，还是应该通过候补客户补单。
2. 当某时段“预估客流低但已排员工多”时，客流分级无法表达真实经营压力，页面会看起来信息很多但行动不清晰。

因此，V1.5 后续建议把热力图主分级从“客流高低”升级为“负荷程度”，即基于预测需求与已排人力的匹配关系给出动作建议。

## 结论

建议改为“负荷程度”分级，而不是继续用“高中低客流”作为主颜色。

客流预测仍保留为基础数据，但不作为颜色主口径。颜色应表达店长下一步应该怎么做：

- 低负荷：人手相对富余，优先补单；补不满时可安排休假或缩短在岗。
- 中负荷：人力与需求基本匹配，保持当前排班。
- 高负荷：人手不足或接近承压，建议加人、加班或调班。

## 当前算法边界

现有后端需求计算不是成熟的机器学习预测模型，主要来源为：

- 当前未来预约数。
- 历史同星期、同时间段预约均值。
- walk-in 缓冲规则。
- 当前原始排班中的已排员工数。

因此当前不适合在产品上强调“小时级精准客流预测”。更稳妥的表达是：

- 内部仍按小时级数据计算空档机会和补单候选。
- 页面外显按上午、下午、晚上三个经营时段聚合。
- 对店长展示“负荷程度”和“推荐动作”。

## 指标定义

### 1. 预测需求

字段名建议：`predictedDemand`

口径：当前时段内预计预约 / 到店需求数量。来源可以继续沿用当前 `expectedReservations`。

展示：作为辅助信息显示，不用于主颜色。

### 2. 建议在岗

字段名建议：`recommendedStaff`

口径：满足预测需求所需的最低在岗美容师数。当前三时段聚合时建议取时段内小时级 `requiredStaff` 的最大值，避免某个小时承压被平均掉。

### 3. 已排在岗

字段名建议：`scheduledStaff`

口径：当前原始排班中该时段实际可服务的人数。三时段聚合时建议取时段内最低在岗数，反映最薄弱时段。

### 4. 负荷差值

字段名建议：`staffDelta`

公式：

```text
staffDelta = scheduledStaff - recommendedStaff
```

解释：

- `staffDelta > 0`：人手富余。
- `staffDelta = 0`：人力匹配。
- `staffDelta < 0`：人手不足。

### 5. 负荷率

字段名建议：`loadRatio`

公式：

```text
loadRatio = recommendedStaff / max(scheduledStaff, 1)
```

解释：

- 越高代表越承压。
- 若 `scheduledStaff = 0` 且 `recommendedStaff > 0`，直接判定高负荷。

## 分级建议

### 低负荷

判定条件：

```text
scheduledStaff > recommendedStaff
```

页面文案：

```text
低负荷
```

推荐动作：

- 有空档机会和推荐客户时：优先补单。
- 无合适候补客户时：建议安排休假、调休或减少在岗。

展示字段：

```text
预测需求：N
建议在岗：N
已排在岗：N
空档机会：N / 推荐客户：N
建议：补单或安排休假
```

### 中负荷

判定条件：

```text
scheduledStaff = recommendedStaff
```

页面文案：

```text
中负荷
```

推荐动作：

- 保持当前排班。
- 若存在高价值候补客户，可以提示“可补单但不强推”。

展示字段：

```text
预测需求：N
建议在岗：N
已排在岗：N
建议：保持当前配置
```

### 高负荷

判定条件：

```text
scheduledStaff < recommendedStaff
```

页面文案：

```text
高负荷
```

推荐动作：

- 推荐加人。
- 推荐加班。
- 推荐调班。
- 若无法加人，则提醒控制新增预约或延后客户。

展示字段：

```text
预测需求：N
建议在岗：N
已排在岗：N
缺口：N 人
建议：加人或加班
```

## 页面调整建议

### 图例区域

当前图例建议从：

```text
低客流 / 中客流 / 高客流 / 人手不足
```

调整为：

```text
低负荷：人手富余，推荐补单或安排休假
中负荷：人力匹配，保持当前配置
高负荷：人手不足，推荐加人或加班
```

顶部汇总建议从：

```text
高需求时段 / 人手不足时段 / 空档机会 / 推荐客户 / 预计收益
```

调整为：

```text
高负荷时段 / 低负荷时段 / 可补单空档 / 推荐客户 / 预计补位收益
```

### 单元格展示

建议每个时段单元格展示：

```text
低负荷 / 中负荷 / 高负荷
预测需求：N
建议在岗：N
已排在岗：N
建议：补单 / 保持 / 加人
```

低负荷时，单元格不再直接展示空档机会和推荐客户明细，只保留轻量动作提示：

```text
建议：可补单
```

空档机会、推荐客户、补位概率和预计收益统一放到热力图右侧的「空档推荐客户名单」中，按当前周聚合展示。这样格子负责回答“负荷是否合理”，右侧名单负责回答“应该跟进谁补位”。

高负荷时，不优先展示补单收益，避免店长误以为应该继续加单；应优先提示人力缺口。

### 右侧空档推荐客户名单

需求热力图右侧新增「空档推荐客户名单」，作为 V1.5 半自动补单闭环的主操作区。

列表按周推荐，不要求店长先点击某个格子。默认排序建议：

1. 补位概率高的客户优先。
2. 推荐收益高的客户优先。
3. 卡项即将到期、近期应回店、历史偏好时间匹配的客户优先。
4. 已有未来预约、近期已高频触达、无可触达方式的客户不展示或降权。

单条推荐展示字段：

```text
客户名称
补位概率
推荐原因
推荐项目
建议预约时段
客户详情
下发跟进
```

操作说明：

- 「客户详情」：打开弹窗，展示客户基础信息、推荐项目、建议时段、推荐原因、风险提示和推荐话术。
- 「下发跟进」：打开确认弹窗，确认后创建跟进任务；V1.5 不自动发短信、不锁档、不自动创建预约。
- 后续如需按时段处理，可在名单上加“上午 / 下午 / 晚上 / 日期”筛选，而不是把推荐信息重新塞回格子。

## 数据与接口建议

后端 `demand` 接口可在后续新增字段，但先兼容旧字段：

```ts
type DemandLoadSlot = {
  date: string;
  startTime: string;
  endTime: string;
  expectedReservations: number;
  requiredStaff: number;
  scheduledStaff: number;
  staffDelta: number;
  loadRatio: number;
  loadLevel: 'low' | 'medium' | 'high';
  recommendedAction: 'fill_gap' | 'keep' | 'add_staff';
};
```

前端短期可以直接基于现有字段计算：

```text
staffDelta = scheduledStaff - requiredStaff
```

中期应由后端统一输出 `loadLevel` 和 `recommendedAction`，避免管理端、终端、智能排班解释口径不一致。

## 与补单闭环的关系

负荷程度应决定“优先动作”：

- 低负荷：优先触发空档机会、候补客户、店长跟进任务。
- 中负荷：仅提示可选补单，不强提示。
- 高负荷：优先提示人力缺口，不建议继续补单，除非补单能自动匹配到新增人手或调整排班。

这能避免当前“低客流但空档机会很多”“高客流还推荐补单”的认知冲突。

## 分阶段实施建议

### V1.5.1 前端口径收敛

- 图例改为低/中/高负荷。
- 单元格主标题改为负荷程度。
- 保留预测需求、建议在岗、已排在岗作为说明字段。
- 低负荷只提示可补单，高负荷显示人力缺口。
- 空档补位信息统一进入右侧「空档推荐客户名单」。

### V1.5.2 后端字段收敛

- `demand` 接口新增 `staffDelta`、`loadRatio`、`loadLevel`、`recommendedAction`。
- 前端不再自行推导颜色，统一使用后端口径。

### V1.6 智能排班联动

- 高负荷单元格直接提供“生成补人方案”入口。
- 低负荷单元格直接提供“生成补单任务”入口。
- 中负荷单元格仅保留查看详情。

## 验收标准

- 店长能在 3 秒内判断每个时段应该“补单、保持、加人”。
- 图例不再混用客流分级和人力缺口分级。
- 高负荷时段不突出补单收益，避免误导继续加预约。
- 低负荷时段只提示可补单，不在格子内堆叠候补客户信息。
- 右侧「空档推荐客户名单」能按周展示客户名称、补位概率、推荐原因、推荐项目、建议预约时段，并支持客户详情和下发跟进。
- 前端和后端对负荷等级、推荐动作口径一致。
