# 经营利润一级模块发布迁移与验收清单

更新时间：2026-06-20

状态：目标库已执行 6 条必需 migration；后续按本清单继续复验和发布评审

## 1. 发布前结论

当前代码门禁已通过，目标库已应用本分支必需的 6 条 Prisma migration，并已写入 `storeId=6`、`2026-06` 经营成本 6 条、总额 55000。最新 readiness 仍为 `blocked`，原因不再是 migration，而是历史商品/项目明细缺服务人归属，以及 3 条项目订单明细引用缺失项目档案。

因此当前可以继续技术联调和页面预检，但还不能进入老板/店长完整业务验收；完整验收必须等服务人归属、提成记录和项目身份缺口得到业务确认或被明确标记为历史异常。

本分支发布不只需要经营利润 3 条 migration，还包含会员卡/客户卡操作人追踪 2 条 migration，以及经营利润月度订单查询组合索引 1 条 migration。原因是当前代码已经读取和写入 `CustomerBalanceTransaction.operatorId`、`CustomerCard.operatorId`，如果目标库漏迁，会员卡流水、办卡记录或终端办卡链路可能运行时报错；经营利润看板、商品毛利和项目毛利会按门店与月份频繁读取订单，缺少组合索引会放大真实门店数据量下的慢查询风险。

## 4. 推荐执行命令

只读检查当前状态：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30
```

其他环境应用 migration：

```powershell
npm.cmd --prefix packages/server-v2 run db:migrate:prod
```

迁移后重新生成 Prisma Client：

```powershell
npm.cmd --prefix packages/server-v2 run db:generate
```

迁移后或数据修复后复验：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选正式确认JSON> --assigneeManualReviewFile=<无候选服务人正式确认JSON> --beauticianUserFile=<美容师账号绑定正式确认JSON> --staffUserFile=<员工账号创建正式确认JSON> --projectMasterFile=<项目档案正式确认JSON> --requireReady --summaryOnly
npm.cmd --prefix packages/server-v2 run operation-profit:confirmed-dry-run -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选正式确认JSON> --assigneeManualReviewFile=<无候选服务人正式确认JSON> --beauticianUserFile=<美容师账号绑定正式确认JSON> --staffUserFile=<员工账号创建正式确认JSON> --projectMasterFile=<项目档案正式确认JSON> --summaryOnly
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30 --assigneeManualReviewFile=<服务人无候选确认JSON>
npm.cmd run check:operation-profit
npm.cmd --prefix packages/server-v2 run operation-profit:cost-seed -- --storeId=6 --periodMonth=2026-06
npm.cmd --prefix packages/server-v2 run operation-profit:sample-reconcile -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --sampleSize=3
```

`operation-profit:cost-seed` 会先通过 `information_schema` 检查 `OperatingCost` 表是否存在。迁移前预期输出 `status=migration_required` 和 `schemaPrecheck.exists=false`，不得在缺表时继续写入或抛出不可读错误。

`check:operation-profit` 是发布前静态门禁，不连接数据库；它用于确认本分支必需 migration、脚本入口、前后端入口、权限、API 导出、readiness 检查项和交付文档没有缺项。

`check:operation-profit:full` 是提交前完整工程门禁，会串行执行静态门禁、经营利润前端集合测试、经营利润 Prisma 脚本 TypeScript 检查、后端经营利润定向测试、前后端构建和 `git diff --check` 空白检查。前端集合覆盖利润总览、商品/项目毛利、成本配置、履约风险、员工人效、Real API 路径、权限和分页/筛选回归；后端经营利润测试固定使用 Jest `--runInBand`，避免 Windows worker 并发触发 `spawn EPERM`；构建门禁会同时执行 `npm.cmd --prefix packages/server-v2 run build` 和 `npm.cmd run build`。

readiness 默认模式用于保留真实库原始缺口；确认包模式用于业务验收准入，只有在服务人无候选确认 JSON 里明确 `resolution=historical_exception` / `ignore_non_margin` 且已业务确认时，才会把这些项从服务人 fail 和提成 warn 中剔除，并通过 `confirmedExceptionGaps` 留痕。

`operation-profit:confirmation-audit --summaryOnly` 用于快速检查确认包是否覆盖当前真实缺口、是否有重复冲突、是否可以进入 dry-run。`operation-profit:confirmed-dry-run --summaryOnly` 只做确认后 dry-run 编排，拒绝 `--apply` / `--yes`；当前确认包未通过时会停在 `blocked_by_confirmation_gate`。发布执行时必须显式传入 4 个核心正式确认 JSON；当前存在员工账号创建计划时，还必须同步传 `--staffUserFile=<员工账号创建正式确认JSON>`，不建议依赖脚本默认 pending 文件。

## 6. 发布风险与回退

| 风险 | 处理 |
| --- | --- |
| 目标库不是预期库 | 停止执行，先确认 `DATABASE_URL` |
| migration 执行中断 | 不继续 seed 或回填，先查看 `_prisma_migrations` 状态 |
| 经营成本为空 | 只跑 `operation-profit:cost-seed` dry-run，不写库 |
| 成本删除或复制上月误触 | 前端必须二次确认，发布前跑 `check:operation-profit` 校验确认逻辑仍存在 |
| 历史订单缺服务人 | 不自动生成提成，先用人工确认 JSON |
| 历史项目缺项目档案 | 先确认真实项目身份或标记历史异常，不把项目误判为高毛利 |
| pending 候选误 apply | `assignee-backfill`、`beautician-user-backfill`、`project-master-backfill` 均拒绝 `confirmedBy=pending_business_confirmation` |
| TODO 草稿误 apply | 统一预检、服务人归属写回、账号绑定写回、项目档案写回和 readiness 确认包模式均拒绝包含 `TODO` 的确认人 |

回退原则：本清单不提供自动回滚命令。若测试/预发库 migration 后需要回退，优先恢复执行前备份或快照。
