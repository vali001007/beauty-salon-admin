## 3. 统一提交与发布门禁

每个提交前必须执行：

1. `git diff -- <显式路径>` 复核业务边界。
2. `git add <显式路径>`，随后检查 `git diff --cached --name-only`、`git diff --cached --stat`、`git diff --cached --check`。
3. 在隔离验证 worktree 复跑该提交的定向验证；共享 schema 改动先执行 Prisma generate/validate。
4. 提交后确认工作区仅残留下一闭环的已知改动。

每个 PR 合并前必须执行：

- 基于最新 `origin/main` 同步分支，保留 merge commit 历史；不 rebase 已发布分支，不强推。
- GitHub 必需检查全部成功，PR 不是 Draft，且不存在未解决评审意见。
- PR 描述包含：业务影响、迁移/回滚说明、测试证据、未执行的真实库或部署动作。

## 4. 最终验收口径

|层级|完成标准|
|---|---|
|代码|退款、P0、评测、终端和管理端改动均按独立闭环提交；无跨域混入。|
|接口与页面|退款由服务端统一承接；管理端与 Kiosk 不复制业务事实；P0 报告不引用本机地址。|
|数据|Prisma schema/migration 经过 validate 和顺序审计；不执行未授权的真实 migration/backfill。|
|交付|每条业务线有独立 PR、可回滚说明和 CI 证据；`main` 无直接推送。|
|治理|原始评测产物保留在被忽略归档；仅已完整替代且无占用的分支/worktree 被退役。|
