# Vibe Coding 工程治理与自动化落地分析

> 日期：2026-07-15  
> 范围：项目 `AGENTS.md` 规则、本地检查脚本、CI 门禁、定时巡检和人工治理边界  
> 本文仅给出落地方案，不直接修改 `AGENTS.md`、不创建自动化、不执行 Git 提交。

## 1. 核心结论

六条改进建议都可以部分固化，但不能全部交给自动化。最稳定的分工是：

- `AGENTS.md` 负责规定 Agent 必须遵守的行为、门禁和交付格式。
- 本地脚本和 CI 负责检查数量、文件、路径、模板完整性和可重复的工程门禁。
- 定时自动化负责生成巡检报告、风险清单和待办，默认只读，不自动提交和推送。
- 人工保留产品取舍、业务口径、版本退役、真实写库、发布和指标解读等不可逆或高语义密度决策。

当前工作区的 `git status --porcelain` 有 393 个状态条目，其中 189 个已跟踪变更、204 个未跟踪条目，同时横跨 Agent、营销、H5、财务、Prisma 和治理文档。因此优先级应为：

1. WIP 与工作区健康门禁。
2. 任务合同与五层验收标准。
3. 分支、worktree 和原子提交纪律。
4. 版本决策与退役治理。
5. 真实业务指标。
6. 每周核心数据链路复核。

## 2. 总体分工表

| 改进项 | 写入 `AGENTS.md` | 本地脚本 | CI/PR 门禁 | 定时自动化 | 必须人工决策 |
|---|---|---|---|---|---|
| 最多两个一级目标并行 | 是 | 是 | 只做提示 | 是 | 例外批准、目标取舍 |
| 分支、worktree、原子提交 | 是 | 是 | 是 | 是 | 提交、push、PR 授权 |
| 五层验收标准 | 是 | 是 | 是 | 可生成缺口报告 | 业务验收结论 |
| Agent/模块版本决策和退役 | 是 | 是 | 是 | 是 | 新版立项、退役批准 |
| 离线评测升级为业务指标 | 是 | 是 | 工程部分 | 是 | 指标口径和效果解读 |
| 每周复核核心数据链路 | 是 | 是 | 否 | 是 | 抽样复算、风险签字 |

## 3. 逐条落地分析

### 3.1 强制控制 WIP：最多两个一级目标并行

#### 内容

将当前正在修改代码、进行真实验证或准备发布的一级业务目标限制在两个以内。文档调研不占用开发 WIP，但调研进入实现后必须占用一个名额。

#### 原因

- 大量并行任务会把共享文件、schema、路由、API facade 和构建配置叠加在同一工作区。
- Agent 需要花大量时间判断哪些改动属于当前任务，导致实现、验证和发布成本同时上升。
- WIP 越大，“构建通过”与“待提交内容通过”之间的差距越大。

#### 建议写入 `AGENTS.md`

```md
## WIP 与任务并行限制
- 同一工作区最多同时存在 2 个进入代码实现、真实验证或发布阶段的一级业务目标。
- 开始第 3 个一级目标前，必须先完成、冻结或转移现有目标，并在回复中说明处理结果。
- 发现工作区横跨 3 个以上一级业务域时，停止扩大修改面，先输出冲突与收口建议。
```

#### 适合自动化的部分

- 新增 `scripts/check-worktree-health.mjs`：按顶层目录和业务域归类 Git 变更，输出 tracked/untracked 数量、跨包范围、共享高风险文件和 WIP 风险。
- 新增 `npm.cmd run check:worktree-health`，在 Agent 开工前和发布计划前执行。
- 每日定时生成只读《工作区健康报告》，当变更跨越 3 个业务域或状态条目超过阈值时标红。

#### 不应自动化的部分

- 不自动删除、stash、reset、commit 或移动当前改动。
- 不由脚本决定哪个业务目标应被暂停。

#### 效益

- 减少任务冲突、上下文浪费和误提交。
- 让每一轮验证能对应到清晰的业务目标。
- 把“工作区太乱”从主观感受转换为可量化预警。

#### 风险

- 只按文件数限制会误伤合理的大型重构。规则应以“业务目标数”为主，文件数只做预警。
- 自动分类会出现共享文件归属不准。报告必须保留 `shared/high-risk` 类别，不强行分配到某个模块。

#### 建议

立即落地为 P0，但先使用“报告 + 预警”模式，不在 CI 中硬阻断。连续运行两周后再根据实际分布设置阈值。

### 3.2 建立分支、worktree 和原子提交纪律

#### 内容

每个一级目标在独立分支上开发；高风险或与当前脏工作区冲突的任务使用独立 worktree；每个提交只承载一个可解释、可验证、可回滚的业务改动。

#### 原因

- 脏工作区下的整树构建不能代表待提交内容通过。
- 大提交会同时降低审查、定位、回滚和多远端同步的可控性。

#### 建议写入 `AGENTS.md`

```md
## 分支与提交边界
- 一个一级业务目标对应一个 `codex/*` 分支；与当前未提交改动冲突时，优先使用独立 worktree。
- 提交必须可独立解释、验证和回滚；不将不同业务域、生成产物和无关文档混入同一提交。
- 只使用显式 `git add <paths>` 控制范围；提交前必须查看 `git diff --cached --name-only` 和对应定向验证结果。
- commit、push 和 PR 始终等待用户明确授权。
```

#### 适合自动化的部分

- 新增 `scripts/check-change-scope.mjs`：检查 staged diff 横跨的业务域、包和高风险文件。
- 新增 `npm.cmd run check:change-scope`，在提交前和 PR CI 中运行。
- PR 过大、同时修改 schema/路由/API client/多客户端时，自动生成风险摘要。
- 每日检查本地分支是否存在长时间未收口、未设 upstream 或与 `main` 偏离过大的情况。

#### 不应自动化的部分

- 不自动创建 commit、push、PR 或删除 worktree。
- 不用简单文件数硬性拒绝一个必须跨包的端到端改动。

#### 效益

- 任务、验证、提交和回滚一一对应。
- 减少精确 staging 时的手工拼补和编码损坏。
- 提高 PR 可审查性和 GitHub/Gitee 同步可解释性。

#### 风险

- worktree 数量失控会造成依赖、环境文件和端口混乱。每个 worktree 必须有目标、分支、负责任务和收口状态。
- 过度切碎提交会使业务上不可独立运行的半成品进入历史。原子提交应以“可独立验证”为边界，不以单文件为边界。

#### 建议

列为 P0。先建立 scope 检查和 PR 风险报告，再逐步强制 staged-tree 验证。

### 3.3 每个任务开工前写五层验收标准

#### 内容

对 L1 及以上任务，开工前定义代码、接口、页面、数据、交付五层完成标准。L2/L3 任务还必须写明真实数据和外部操作授权边界。

#### 原因

- “代码已改”、“构建通过”、“数据库已迁移”、“业务已打通”是不同完成状态。
- 验收条件在实现后才补充，会导致返工和状态误报。

#### 建议写入 `AGENTS.md`

```md
## 任务合同与五层验收
- L1 及以上任务开工前明确：目标、对象、不做什么、交付物、验收标准和外部操作权限。
- 验收标准按代码、接口、页面、数据、交付五层编写；不适用的层级必须标记为“不适用”并说明原因。
- 最终回复按五层分别报告已完成、已验证和未验证项，不用单一“已完成”覆盖多个层级。
```

#### 适合自动化的部分

- 新增任务文档模板，固定包含六个字段和五层验收矩阵。
- 新增 `scripts/check-task-contract.mjs`，检查新增的 PRD/tasks/实施计划是否含必填章节、是否存在 `TODO/TBD/待定`。
- PR 模板要求填写五层验收结果，CI 检查表头和必填项。
- 定时巡检可对未关闭任务生成“已有代码、缺页面验收”或“已有 migration、缺 live DB 证据”的缺口报告。

#### 不应自动化的部分

- 不由脚本判断真实业务验收是否通过。
- 不允许 Agent 为了通过模板检查而自行将未验证项标记为完成。

#### 效益

- 显著减少“是不是已经打通”的二次考古。
- 让计划、实现、验证、日报和发布使用同一状态语言。
- 防止 build 绿灯被误报为业务交付完成。

#### 风险

- 模板过长会把轻量任务变成文档工作。L0 任务免除，L1 使用简版，L2/L3 使用完整版。
- 勾选项会被机械填充。CI 只能检查完整性，不能替代内容审查。

#### 建议

列为 P0，与 WIP 门禁同时落地。这是成本最低、减少误解最直接的改进。

### 3.4 建立 Agent/模块版本决策记录和退役机制

#### 内容

为 Agent、Ami Brain、语义查询、营销推荐等高变化模块建立统一版本登记，明确每个版本的定位、主入口、事实源、代替对象、保留理由、发布状态和退役条件。

#### 原因

- 新版本持续叠加会让路由、API、评测、治理中心和文档同时保留多套概念。
- 没有明确的替代和退役条件，新版会变成并行增量，而不是收敛。

#### 建议写入 `AGENTS.md`

```md
## Agent 与核心模块版本治理
- 新建 Agent/核心模块版本前，必须新增或更新版本决策记录，写明新版本替代什么、不替代什么、主入口、发布门禁和旧版退役条件。
- 不允许只新增版本而不定义旧版的保留、冻结、迁移或退役状态。
- 路由、菜单、API、评测和文档必须引用同一版本登记中的主线定义。
- 退役涉及删除、数据迁移、真实发布时，必须获得用户明确授权。
```

#### 适合自动化的部分

- 建立 `docs/03-开发计划/01-AI智能体与问数能力/agent-version-registry.yaml` 或 Markdown 表格。
- 新增 `scripts/check-agent-version-registry.mjs`，扫描路由、菜单、模块、npm scripts 和评测目录，报告未登记版本、重复主入口、已标记退役但仍被引用的资产。
- PR 如新增 `agent-v*`、`brain-*` 版本目录或路由，CI 要求同步更新版本登记和 ADR。
- 每周生成“版本分布与退役候选”报告。

#### 不应自动化的部分

- 不自动判定哪个 Agent 版本应成为产品主线。
- 不自动删除旧路由、旧表、旧评测结果和兼容代码。

#### 效益

- 减少版本命名、产品定位和运行主线混乱。
- 新版立项时同时形成旧版收口路径。
- 让评测、发布和治理页面不再各自定义版本状态。

#### 风险

- 登记表不同步会变成新的错误事实源。所以必须用扫描脚本检查代码与登记一致性。
- 过早退役会破坏历史兼容和生产回滚。退役必须包含调用量、依赖、数据迁移和回滚证据。

#### 建议

列为 P1。先建登记和扫描报告，不立即执行退役。

### 3.5 把离线评测升级为真实业务指标评测

#### 内容

在现有题库可用率、延迟、失败分类之上，增加场景任务完成率和业务结果指标，形成三层评测：

1. 离线能力：准确性、拒答、权限、查询安全和延迟。
2. 场景任务：店长是否能完成查询、分析、创建任务、发起营销和跟进处理。
3. 业务结果：任务耗时、人工接管率、建议采纳率、触达转化、错账或异常减少量。

#### 原因

- 题库可用率反映“会不会答”，不反映“是否完成了经营任务”。
- 只优化题库会让 Agent 朝测试集收敛，但无法证明真实用户价值。

#### 建议写入 `AGENTS.md`

```md
## Agent 评测与业务效果
- Agent 交付不得只使用离线题库可用率作为完成结论。
- L2/L3 Agent 任务必须分别报告离线能力、场景任务和业务结果三层指标；未接入真实业务指标时，明确标记为“业务效果未验证”。
- 新增指标必须先定义口径、数据源、时间窗口、门店范围和基线，不允许前端自行推导经营结论。
```

#### 适合自动化的部分

- 延续现有评测脚本，为每次运行增加 `scenarioId`、`taskCompleted`、`humanTakeover`、`acceptedAction`、`businessOutcomeWindow`。
- 建立只读数据聚合任务，每周按门店、角色、场景和 Agent 版本生成报告。
- 对金额、转化和归因指标保留 `actual/predicted/estimated` 来源标识。
- CI 只检查埋点合同、指标定义和评测脚本；周期性业务结果由定时任务计算。

#### 不应自动化的部分

- 不由 Agent 自行改变业务指标口径或基线。
- 不将预测值、估算值或短窗口波动标记为真实业务收益。
- 不在没有用户知情和隐私边界时收集对话全文或敏感客户数据。

#### 效益

- 把 Agent 从“能回答”转换为“能产生可验证的经营价值”。
- 为版本保留、灰度、回滚和退役提供业务证据。
- 避免团队只追求题库分数。

#### 风险

- 错误归因会把营销活动、季节、人工操作和 Agent 效果混在一起。必须保留基线、对照和来源元数据。
- 埋点和报表会增加数据治理成本。应先选 2–3 个高价值场景，不一次覆盖全部 Agent 能力。

#### 建议

列为 P2。第一批只选“经营问数”、“营销建议到执行”和“异常巡检到人工处理”三个场景。

### 3.6 每周亲自复核一个核心数据链路

#### 内容

每周按轮换表选择一条核心链路，进行从业务记录、数据表、service、API 到页面/终端展示的抽样复核。自动化准备证据包，人工完成口径判断和签字。

#### 原因

- AI 擅长大范围检索和关系追踪，但不承担业务口径的最终责任。
- 核心数据链路漂移通常不会在编译或单元测试中直接暴露。

#### 建议写入 `AGENTS.md`

```md
## 核心数据链路定期复核
- 每周选择 1 条核心数据链路进行 route/component -> API -> service -> schema/table -> 真实记录抽样复核。
- Agent 负责生成只读证据包、差异和风险；业务口径、异常处理和是否改数据由用户最终确认。
- 复核报告必须标注查询时间、门店范围、样本记录、数据源和未授权操作。
```

#### 适合自动化的部分

- 建立轮换表：收银支付与退款、日结与财务、库存扣减与回补、营销触达与归因、客户画像与推荐、终端任务与业绩、Agent 语义定义与真实查询。
- 定时任务只读生成证据包：当前 schema、迁移状态、最新样本、数量对账、页面/API 路径、已知差异。
- 报告输出到固定目录，并生成“待人工确认”项，不自动修复。
- 对前后两周数量变化和口径变化做 diff，突出新增漂移。

#### 不应自动化的部分

- 不自动修复真实数据、执行 backfill、调整财务口径或重发营销任务。
- 不用单条样本代替整体业务结论。

#### 效益

- 建立用户对核心业务不变量的持续掌控。
- 提前发现 schema、service 口径、页面展示和真实数据之间的漂移。
- 减少只阅读 Agent 总结而没有看原始证据的风险。

#### 风险

- 每周报告过大会导致审阅疲劳。每次只复核一条链路，正文只保留结论、样本、差异和待决策项。
- 定时查询会增加生产库负载。只使用只读、有索引、有时间窗口和有数量上限的查询。

#### 建议

列为 P1。先从“收银支付→退款→日结”和“营销触达→订单归因”两条高风险链路开始。

## 4. 建议的自动化组合

### 4.1 P0：立即做，只读与低风险

1. `check:worktree-health`
   - 输出 Git 状态条目数、业务域分布、跨包范围、高风险共享文件。
   - 只报告，不修改工作区。
2. `check:task-contract`
   - 检查任务文档和 PR 说明是否包含目标、排除项、交付物、五层验收和权限边界。
3. `check:change-scope`
   - 检查 staged diff 是否同时横跨多个不相关业务域。
4. 每日工作区健康报告
   - 只读输出，不 `git add`、不 commit、不 push。

### 4.2 P1：治理与周期复核

1. Agent/核心模块版本登记与一致性扫描。
2. 每周版本分布、重复入口和退役候选报告。
3. 每周一条核心数据链路的只读证据包。
4. PR 大小、业务域、schema/API/路由联动风险报告。

### 4.3 P2：真实业务效果

1. Agent 场景任务事件合同。
2. 人工接管、操作采纳、任务完成和业务结果聚合。
3. 分门店、角色、场景、版本的周报。
4. `actual/predicted/estimated` 指标来源与基线对照。

## 5. 推荐的 `AGENTS.md` 控制粒度

`AGENTS.md` 应保留短、稳定、可执行的规则，不应塞入长脚本说明、当前统计数字和频繁变动的模块列表。

建议仅追加四个稳定规则组：

1. WIP 与并行限制。
2. 分支、worktree 与提交边界。
3. 任务合同与五层验收。
4. Agent/核心模块版本治理和数据链路复核。

详细命令、阈值、报告目录和轮换表应放在本目录的运行手册或对应脚本帮助中。

## 6. 自动化权限边界

### 默认允许自动执行

- Git 和文件状态只读扫描。
- 代码、模板、版本登记和指标合同一致性检查。
- 定向测试、typecheck、build、`git diff --check`。
- 生成工作区健康、版本分布、验收缺口和数据链路证据报告。

### 必须获得用户授权

- commit、push、PR、合并、tag 和 release。
- 真实数据库 migration、backfill、更新、删除和业务动作重放。
- Agent 主线切换、旧版退役、路由删除和兼容层清理。
- 业务指标口径、财务认定规则、营销归因规则和生产发布结论。

## 7. 预期总体效益

- 开工前：Agent 先知道当前可改范围、验收标准和授权边界。
- 开发中：减少任务扩张、共享文件冲突和新旧版本叠加。
- 验证时：明确区分代码、接口、页面、数据和交付状态。
- 发布时：提交边界更清晰，PR 可审查，回滚范围更可控。
- 运营时：Agent 评测从题库分数转向真实任务和业务结果。
- 治理时：用户保留产品、数据和发布的最终决策权，机械检查交给脚本和定时任务。

## 8. 最终建议

不建议一次性实现全部治理能力。最优实施顺序是：

1. 先在用户确认后将 WIP、任务合同、提交边界和版本治理的短规则并入 `AGENTS.md`。
2. 实现 `check:worktree-health`、`check:task-contract`、`check:change-scope` 三个只读脚本。
3. 运行两周只报告模式，根据误报调整阈值。
4. 再将稳定检查接入 PR CI，不稳定项保持 warning。
5. 建立 Agent/核心模块版本登记和每周数据链路证据包。
6. 最后引入真实业务指标，先覆盖三个高价值 Agent 场景。

这个顺序先解决当前最大的工程损耗，再建立长期的版本和业务效果治理，不会在工作区已经复杂的情况下再新建一套重型平台。
