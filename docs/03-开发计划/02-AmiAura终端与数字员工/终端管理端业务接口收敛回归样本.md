# 终端与管理端业务接口收敛回归样本

更新时间：2026-06-26

## 1. 使用范围

本文用于验证终端和管理端保留不同入口，但写入同一类业务事实。联调时每个场景至少各做一笔真实数据，并按表级事实核对，不以页面提示成功作为唯一验收。

## 2. 通用验收原则

- 终端入口继续使用设备鉴权和 `/terminal/*`。
- 管理端入口继续使用 JWT、权限码和后台路由。
- 同一业务必须进入同一权威 service：
  - 次卡核销：`CardsService.verifyCardUsage`
  - 客户登记：`CustomersService.create`
  - 会员充值：`OrdersService.createRechargeOrder` / `createMemberCardRecharge`
  - 商品/项目收银：`OrdersService.createProductOrder`
- 每笔样本都要记录：入口、客户、门店、操作人、服务人员、订单号或流水号、创建时间。

## 3. 回归样本

### 3.1 终端次卡核销

入口：

```http
POST /terminal/cards/consume
```

样本 payload：

```json
{
  "customerCardId": 66,
  "customerId": 10,
  "projectId": 101,
  "times": 1,
  "operatorId": 21,
  "beauticianId": 2
}
```

必查事实：

| 表 | 验收点 |
| --- | --- |
| `CustomerCard` | `remainingTimes` 减少 1 |
| `CardUsageRecord` | 有 `storeId/customerId/customerCardId/cardId/projectId/projectName/operatorId/beauticianId/deviceId/recognizedAmount` |
| `StockMovement` | 有 `sourceType=card_usage`、`sourceId=CardUsageRecord.id` |
| `CommissionRecord` | 命中规则时有 `sourceType=card_usage`、`cardUsageRecordId` |

### 3.2 管理端次卡核销

入口：

```http
POST /cards/verify-usage
```

样本 payload：

```json
{
  "customerCardId": 66,
  "projectName": "深层补水护理",
  "consumedTimes": 1,
  "operatorId": 21,
  "beauticianId": 2
}
```

必查事实：

| 表 | 验收点 |
| --- | --- |
| `CustomerCard` | 次数扣减规则与终端一致 |
| `CardUsageRecord` | 项目名、项目 ID、履约收入与终端同口径 |
| `StockMovement` | BOM 耗材同样从 `card_usage` 追溯 |
| `CommissionRecord` | 服务人员提成同样生成 |

### 3.5 终端充值

入口：

```http
POST /terminal/recharge-orders
```

样本 payload：

```json
{
  "customerId": 10,
  "customerName": "李女士",
  "amount": 500,
  "giftAmount": 50,
  "giftProjects": ["补水护理"],
  "paymentMethod": "wechat",
  "transactionNo": "WX001",
  "beauticianId": 2,
  "remark": "终端充值"
}
```

必查事实：

| 表 | 验收点 |
| --- | --- |
| `ProductOrder` | `orderKind=member_card_recharge`、`source=terminal`、`payMethod=wechat` |
| `OrderItem` | `itemType=recharge`、`netAmount=amount`、`beauticianId` |
| `PaymentRecord` | `method=wechat`、金额等于充值实收 |
| `CustomerBalanceAccount` | 现金余额和赠送余额增加 |
| `CustomerBalanceTransaction` | `type=recharge`、`paymentMethod=wechat` |
| `CommissionRecord` | 命中充值规则时生成 |
| 日结 | 当日刷新后能体现该充值现金流 |

      "unitPrice": 200,

必查事实：

| 表 | 验收点 |
| --- | --- |
| `ProductOrder` | `source=terminal`、`orderKind=project`、`checkoutGroupNo` 存在 |

| `CommissionRecord` | 通过 `orderItemId` 追溯到订单明细 |
| 日结/项目利润 | 收入、耗材、提成可追溯 |

### 3.9 管理端商品/项目订单

入口：

```http

```

必查事实：

| 表 | 验收点 |
| --- | --- |
| `ProductOrder` | 默认 `source=admin`，订单状态和支付方式正确 |
| `OrderItem` | 商品/项目明细结构与终端一致 |
| `StockMovement` | 商品 `sale_out`、项目 `service_consume` 均可追溯 |
| `CommissionRecord` | 提成记录带 `orderItemId` |
| 日结/利润页 | 不因终端收敛而退化 |

## 5. 联调验收记录模板

只读扫描命令：

```powershell
npm.cmd --prefix packages/server-v2 run business-interface:verify -- --storeName="Ami 全量演示门店" --from=2026-06-26 --to=2026-06-26
```

该命令只读取现有真实流水并输出 `pass/warn/fail`，不会创建客户、订单、核销或充值数据。若输出 `warn`，表示对应入口在时间窗内缺少真实样本，需要先通过页面或接口创建样本后重跑。

样本执行计划命令：

```powershell
npm.cmd --prefix packages/server-v2 run build
npm.cmd --prefix packages/server-v2 run business-interface:samples -- --storeName="Ami 全量演示门店"
```

该命令默认也是 dry-run，只输出候选客户、员工、次卡、项目、商品、终端设备和即将调用的业务 service payload。只有显式追加 `--execute --confirm-write-samples` 才会创建真实业务样本。

脚本输出包含两部分：

- `checks`：检查时间窗内已有真实流水是否满足关键字段和副作用要求。
- `samplePlan`：当存在 `warn` 时，基于当前门店真实客户、员工、次卡、项目、商品和余额账户生成可执行样本建议；该部分仍然只读，不自动写库。
