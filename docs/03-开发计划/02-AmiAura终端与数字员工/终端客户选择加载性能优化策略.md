# 终端客户选择加载性能优化策略

## 背景

智能终端在“今日预约-添加预约”弹窗中，客户下拉框长时间停留在“请选择客户/正在加载客户”，导致前台无法快速创建预约。

该问题不是单个 select 组件问题，而是客户选择的数据加载架构过重：一个简单客户选择动作触发了终端全量业务快照，包含客户、预约、终端预约、目录、美容师、库存、库存预警、卡项、订单等多类数据。任一慢请求都会拖慢整个弹窗。

## 当前链路

前端链路：

1. 用户点击“添加预约”
2. `RoleDashboards.openCreateDialog`
3. 调用 `getAppointmentCreateOptions()`
4. 并行调用 `loadCoreSnapshot()` 与 `getAppointmentEditOptions()`
5. 弹窗所有客户、项目、美容师选择项等待同一个 `optionsLoading`

数据链路：

1. `loadCoreSnapshot()` 调用多组接口
2. 其中客户数据使用 `getCustomers({ storeName })`
3. 同时还拉预约、库存、库存预警、卡项、订单等与“选择客户”无直接关系的数据
4. 客户返回后一次性渲染到原生 `<select>`

## 主要问题

### 1. 为客户选择加载了过多无关数据

添加预约只需要：

- 客户候选
- 项目候选
- 美容师候选
- 可选时间段

当前却加载：

- 全量客户
- 预约分页
- 今日终端预约
- 终端目录
- 美容师
- 库存
- 库存预警
- 次卡
- 商品订单

这会把“客户选择”的响应时间绑定到多个无关接口上。

### 2. 客户列表一次性全量加载

全量客户适合管理端列表页，不适合终端弹窗。终端只需要先显示少量高概率客户，再支持搜索。

推荐首屏客户优先级：

1. 今日预约客户
2. 最近到店客户
3. 高价值/活跃客户
4. 搜索命中的客户

### 3. UI 使用原生 select，不适合大量客户

当客户超过几十位后，原生 select 的体验会变差：

- 搜索能力弱
- 渲染选项多
- 无法边输入边查
- 手机号/会员等级/标签展示受限

### 4. 项目、美容师、客户共用一个 loading

当前 `optionsLoading` 同时控制客户、项目、美容师。只要客户慢，项目和美容师也一起不可用；只要项目/美容师慢，客户也不能选。

### 5. 缓存粒度过粗

`loadCoreSnapshot()` 是大缓存，适合经营看板，不适合弹窗级交互。客户选择需要独立缓存、短 TTL、可后台刷新。

## 目标体验

| 场景 | 目标 |
| --- | --- |
| 打开添加预约弹窗 | 300ms 内显示弹窗和项目/美容师基础选项 |
| 客户默认候选 | 1s 内显示 20-50 位高优先级客户 |
| 搜索客户 | 输入 2 个字符后 500ms 内返回结果 |
| 无客户结果 | 支持快速新建客户或跳转登记 |
| 网络慢 | 不阻塞项目/美容师选择，客户区显示局部加载状态 |

## 方案 A：轻量接口替换全量快照（P0，推荐先做）

新增或复用终端轻量客户上下文接口，不再通过 `loadCoreSnapshot()` 获取预约客户。

### 后端建议

新增接口：

```http
GET /api/terminal/appointments/context?keyword=&limit=50
```

返回：

```ts
interface TerminalAppointmentContext {
  storeName: string;
  generatedAt: string;
  customers: Array<{
    id: number;
    name: string;
    phone: string;
    memberLevel: string;
    tags: string[];
    lastVisitDate?: string;
    isAppointedToday: boolean;
    appointmentTime?: string;
  }>;
  projects: Array<{
    id: number;
    name: string;
    duration: number;
    status: string;
  }>;
  beauticians: Array<{
    id: number;
    name: string;
    level: string;
    status: string;
  }>;
}
```

查询原则：

- 客户默认只取 50 位
- 搜索时只取命中客户，最多 50 位
- 只 select 弹窗需要字段
- 今日预约客户优先合并进候选
- 不查库存、订单、卡项等无关数据

### 前端建议

调整 `getAppointmentCreateOptions()`：

- 不再调用 `loadCoreSnapshot()`
- 改用轻量预约上下文
- 项目、美容师走目录/预约上下文
- 客户候选只显示轻量列表

### 收益

- 改动范围可控
- 不影响经营看板
- 立即解决“一个弹窗拉全量业务数据”的问题
- 可复用现有 `getTerminalContextCustomers` 的查询逻辑

### 风险

- 默认客户只返回候选，不是全量客户
- 需要搜索能力兜底

## 方案 B：异步搜索客户选择器（P1，推荐与 P0 连做）

把客户下拉从原生 select 改为可搜索客户选择器。

交互设计：

1. 打开弹窗时显示“今日预约/最近客户”候选
2. 用户输入姓名或手机号时触发后端搜索
3. 搜索结果展示姓名、手机号、会员等级、标签
4. 无结果时展示“新增客户”

技术实现：

- 输入防抖 300ms
- 请求可取消，避免旧请求覆盖新结果
- 每次最多返回 20-50 条
- 客户列表局部 loading，不阻塞项目/美容师
- 后端搜索支持姓名、手机号、拼音首字母可后续扩展

收益：

- 终端使用体验接近收银/CRM 系统
- 客户再多也不会拖慢弹窗
- 可以复用于收银、办卡、充值、核销、客户跟进

## 方案 C：统一终端客户选择服务（P2）

把所有终端涉及客户选择的功能统一到一个数据源和组件：

- 添加预约
- 收银
- 办卡
- 充值
- 核销
- 客户跟进
- 服务记录

统一能力：

```ts
interface CustomerSelectQuery {
  scene: 'appointment' | 'cashier' | 'card_opening' | 'recharge' | 'verification' | 'follow_up';
  keyword?: string;
  onlyWithActiveCards?: boolean;
  onlyMyCustomers?: boolean;
  limit?: number;
}
```

不同场景只改变过滤条件，不改变组件和基础接口。

收益：

- 避免每个业务流各自实现客户加载
- 数据口径一致
- 后续加缓存、分页、权限都在一处完成

## 方案 D：后台预热与分层缓存（P2）

对高频终端场景做预热：

- 进入预约页后后台预取项目/美容师
- 用户点击“添加预约”前预取默认客户候选
- 客户搜索单独缓存 1-3 分钟
- 使用 stale-while-revalidate：先显示缓存，再后台刷新

缓存建议：

| 数据 | TTL |
| --- | --- |
| 项目/美容师目录 | 10 分钟 |
| 默认客户候选 | 1-3 分钟 |
| 搜索结果 | 1 分钟 |
| 可预约时间段 | 30 秒 |

## 方案 E：数据库索引和查询治理（P1/P2）

后端应补齐客户搜索相关索引：

- `Customer(storeId, deletedAt, lastVisitDate)`
- `Customer(storeId, deletedAt, totalSpent)`
- `Customer(storeId, deletedAt, phone)`
- `Customer(storeId, deletedAt, name)`
- 如使用 PostgreSQL 模糊搜索，可评估 trigram index

查询治理：

- 所有终端客户选择接口必须带 `limit`
- 禁止弹窗级接口无分页拉全量客户
- 只返回选择器需要的字段
- 大对象字段、历史记录、订单聚合放到客户详情或后续请求

## 推荐执行顺序

### P0：1-2 天

1. 新增预约创建轻量上下文接口
2. `getAppointmentCreateOptions()` 改为轻量接口
3. 添加预约弹窗拆分客户、项目、美容师 loading
4. 默认客户候选限制 50 条
5. 保留失败降级：轻量接口失败时再走旧快照

验收标准：

- 打开添加预约弹窗不再等待库存、订单、卡项等接口
- 客户默认候选 1 秒内出现
- 项目、美容师不被客户加载阻塞
- 终端构建通过
- 后端终端单测通过

### P1：2-4 天

1. 替换原生 select 为异步搜索客户选择器
2. 增加客户搜索接口参数：`keyword`、`limit`、`scene`
3. 支持无结果快速新建客户
4. 复用到预约、收银、办卡、充值

验收标准：

- 输入姓名/手机号可快速搜索
- 1000+ 客户情况下弹窗不卡顿
- 搜索请求不会互相覆盖
- 所有客户选择控件口径一致

### P2：5-8 天

1. 建立统一 `CustomerSelectService`
2. 所有终端客户选择场景接入统一服务
3. 增加缓存、预热、请求取消、指标埋点
4. 补数据库索引和慢查询监控

验收标准：

- 所有终端客户选择组件共用一个数据源
- 客户选择平均响应低于 500ms
- P95 响应低于 1.5s
- 慢接口可观测、可定位

## 不建议的做法

1. 继续扩大 `loadCoreSnapshot()`
   这会让一个弹窗依赖更多业务接口，后续还会变慢。

2. 只把客户列表 slice 到更少
   这只能减少渲染，不能解决接口慢和无法搜索的问题。

3. 继续用原生 select 承载全量客户
   美业门店客户量增长后必然不可用。

4. 客户、项目、美容师继续共用一个 loading
   用户感知是“整个弹窗卡死”，不符合前台高频操作场景。

## 最终建议

先执行 P0 + P1：

- P0 解决当前“迟迟加载不出来”的问题
- P1 解决“客户多后无法高效选择”的长期体验问题

P2 作为统一治理，把收银、办卡、充值、核销、客户跟进的客户选择能力合并，避免每个模块重复维护。
