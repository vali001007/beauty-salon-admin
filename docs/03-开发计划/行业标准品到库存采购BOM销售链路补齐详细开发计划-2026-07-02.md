## 1. 背景与目标

当前链路的目标是把行业数据平台沉淀的标准商品/耗品，转化为门店可经营、可入库、可采购、可扣耗、可销售的真实 SKU，并最终接入供应链平台的报价和履约。

目标链路：

```text
行业标准商品/耗品模板
  -> 采用/映射
  -> 门店本地产品 SKU
  -> 项目 BOM
  -> 库存入库/服务扣耗/销售出库
  -> 采购建议/供应链 SKU/报价/履约/结算
```

本计划聚焦把链路从“门店内部库存闭环已通”推进到“标准品、门店 SKU、供应链 SKU、采购履约全闭环可验收”。

## 3. 核心对象边界

### 3.1 行业标准商品/耗品模板

模型：`IndustryProductTemplate`

定位：

- 平台级行业标准。
- 定义标准名称、标准编码、分类、规格、规格单位、包装、参考成本、参考零售价、适用服务分类。
- 不直接代表门店库存。
- 不直接代表某个供应商可供货 SKU。

关键字段：

- `standardProductCode`
- `name`
- `category`
- `productType`
- `recommendedSpec`
- `unit`
- `packageUnit`
- `referenceCostMin/referenceCostMax`
- `referenceRetailPriceMin/referenceRetailPriceMax`
- `futureSupplyMappingStatus`

### 3.2 门店本地产品 SKU

模型：`Product`

定位：

- 门店真实库存、采购、销售、BOM 扣耗对象。
- 同一个行业标准品可在不同门店生成不同本地 SKU。
- 本地 SKU 有自己的库存、安全库存、成本价、零售价、供应商、批次、库存流水。

关键字段：

- `storeId`
- `sku`
- `name`
- `specQuantity`
- `specUnit`
- `packageUnit`
- `costPrice`
- `retailPrice`
- `currentStock`
- `safetyStock`
- `supplier`
- `minPurchaseQty`

### 3.3 采用记录

模型：`IndustryAdoptionRecord`

定位：

- 追踪标准模板如何变成本地业务对象。
- 支撑来源追溯、版本提示、重新同步、脏链路巡检。

关键字段：

- `adoptionType`
- `productTemplateId`
- `templateVersion`
- `localProductId`
- `localProjectId`
- `localBomItemIds`
- `payload`

### 3.4 项目 BOM

模型：`ProjectBomItem`

定位：

- 本地项目消耗本地产品的标准用量。
- 服务完成或项目订单确认时，按 BOM 扣减库存。

关键字段：

- `projectId`
- `productId`
- `standardQty`
- `unit`

### 3.5 供应链映射

模型：`SupplyCatalogMapping`

定位：

- 连接行业标准品、本地产品、供应链 SKU。
- 回答“这个标准品/门店 SKU 现在从哪个供应商买，买哪个 SKU”。

关键字段：

- `standardProductTemplateId`
- `productId`
- `supplySkuId`
- `storeId`
- `mappingStatus`
- `isPreferred`

### 3.6 供应链采购履约

模型：

- `SupplySku`
- `SupplyQuote`
- `ProcurementOrder`
- `ProcurementOrderItem`
- `SupplierShipment`
- `SupplySettlement`

定位：

- 从供应商可供 SKU 和报价生成平台采购单。
- 供应商确认、发货。
- 门店收货后写入库存批次、产品库存、库存流水。
- 生成供应商结算。

## 4. 补齐原则

1. 标准品不直接参与库存扣减，库存扣减必须落到门店本地 `Product`。
2. BOM 不绑定供应商 SKU，BOM 绑定本地 `Product` 和用量单位。
3. 采购建议优先使用 `SupplyCatalogMapping + SupplyQuote`，无映射时回退手工采购。
4. 产品详情必须能反查来源标准品、采用记录和供应链映射状态。
5. 所有自动生成动作必须写审计记录，避免门店不知道数据从哪里来。
6. 真实数据验收优先于页面可见，必须验证数据库对象和业务流水。

## 阶段 6：销售与库存扣耗一致性治理

目标：确保销售、服务、BOM 都使用同一套本地 SKU 和规格单位，不再出现包装单位误作扣耗单位。

### 任务 6.1 单位一致性巡检

检查：

- `ProjectBomItem.unit` 是否与 `Product.specUnit` 合理一致。
- `StockMovement.unit` 是否使用规格单位。

- 对历史异常提供修复建议，不自动批量改真实流水。

### 任务 6.2 商品销售出库口径确认

产品决策点：

- 商品销售通常按包装销售，例如 1 瓶、1 盒。
- 服务耗材扣耗通常按规格单位消耗，例如 3ml、1片。

需要明确：

- 商品订单 `sale_out` 是按包装库存扣减，还是按规格库存扣减。
- 如果库存主数量按规格单位管理，销售 1 瓶 30ml 应扣 30ml。
- 如果库存主数量按包装管理，服务 BOM 3ml 扣耗需要做包装拆分和余量管理。

建议口径：

- 短期：库存数量仍按当前系统数量口径，服务 BOM 用 `specUnit` 展示，不做包装拆分。
- 中期：新增包装换算字段 `packageSpecQuantity`，支持 1 瓶 = 30ml，库存可按最小规格单位管理。

验收：

- 形成明确口径文档。
- 不同页面文案不再混用“单位”。

| 采用记录没有强外键到 Product | 可能出现已删除产品仍被追溯 | 增加健康巡检和详情过滤 |
| 供应链映射为空 | 采购建议不能直接平台化 | 先做映射管理，再接采购建议 |
| 报价随时间变化 | 历史采购金额可能被当前报价污染 | 采购单创建时固化 quoteId 和 unitPrice |
| 包装与规格换算未建模 | 服务扣耗和销售出库可能口径不同 | 中期新增包装换算字段 |
| 批量采用可能生成重复 SKU | 产品目录污染 | 采用前做 storeId + SKU 唯一检查和 dry-run |

## 11. 交付顺序建议

推荐按以下顺序实施：

1. 阶段 0：脏链路治理与基线验收。
2. 阶段 1：产品详情补来源与映射状态。
3. 阶段 2：标准品批量采用与现有产品映射。
4. 阶段 3：供应链映射管理。
5. 阶段 4：采购建议接入供应链映射。
6. 阶段 5：平台采购履约闭环。
7. 阶段 6：销售与库存扣耗一致性治理。
8. 阶段 7：链路总览、单品链路详情和 Agent 接入。

最小可交付 MVP：

```text
采用记录巡检
  + 产品详情来源
  + 标准品批量采用
  + 供应链映射 CRUD
  + 采购建议显示映射与报价
```

完整闭环交付：

```text
MVP
  + 从采购建议生成平台采购单
  + 供应商确认发货
  + 门店收货入库
  + 供应链结算
  + 链路总览和 Agent 问答
```

## 12. 完成定义

本链路视为完成，必须同时满足：

1. 任一已发布标准品能看到是否已采用、采用到哪个本地产品。
2. 任一本地产品能看到来源标准品、BOM 使用情况、库存流水、采购记录、销售/服务消耗。
3. 任一低库存产品能判断是否有供应链映射和可用报价。
4. 有映射和报价的产品能从补货建议直接生成平台采购单。
5. 平台采购单能完成供应商确认、发货、门店收货、库存入库。
6. 收货入库能写批次、产品库存和库存流水。
7. 服务完成能按 BOM 扣库存。
8. 商品销售能生成销售出库流水。
9. 链路总览能显示每个阶段的数量和断点。
10. 真实数据库有可复验样例，不只停留在 mock 或单测。
