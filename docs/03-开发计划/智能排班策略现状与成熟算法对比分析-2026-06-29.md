# 智能排班策略现状与成熟算法对比分析

日期：2026-06-29

## 结论摘要

当前 Ami_Core 的智能排班已经不是纯前端展示功能，而是有完整的后端预览、评估、发布、运行记录链路。它的核心策略是“规则启发式 + 贪心补人 + 冲突评分”，适合中小门店做半自动排班建议，但还不是成熟劳动力排班系统常见的“约束求解/全局最优化”。

产品上可以定位为：当前版本是可解释、可人工确认的智能排班助手；如果要对标成熟排班系统，还需要补齐工时法规、休息间隔、成本最小化、员工偏好、技能替代、资源容量、取消/改约滚动重排等能力。

## 当前功能链路

### 管理端入口

页面：`src/app/pages/Scheduling.tsx`

用户点击“智能排班”后展开配置区，可设置：

- 排班周期：当前查看周、本周、下周。
- 生成方式：
  - 复制上周并优化。
  - 从空白生成。
  - 基于当前班表优化。
- 优先目标：
  - 覆盖预约。
  - 覆盖高峰。
  - 工时公平。
  - 减少人力。
- 高峰最低在岗人数。
- 是否保留已确认预约美容师。
- 是否允许覆盖占用时段。
- 是否允许覆盖请假时段。

页面调用接口：

- `POST /api/scheduling/smart/preview`：生成预览，不保存排班。
- `POST /api/scheduling/smart/evaluate`：评估当前排班方案。
- `POST /api/scheduling/smart/publish`：发布方案，写入排班表。
- `GET /api/scheduling/demand`：查看需求热力图。

### 后端核心服务

核心文件：`packages/server-v2/src/scheduling/smart-scheduling.service.ts`

当前会读取的数据：

- `Beautician`：美容师列表和状态。
- `Reservation`：当前周项目预约。
- `Reservation` 历史 28 天数据：估算需求热度。
- `Schedule`：当前周或上周排班。
- `SchedulingRuleConfig`：营业时间、高峰规则、默认最低人手。
- `BeauticianAvailability`：美容师可用/不可用/偏好时段。
- `BeauticianTimeOff`：请假时段。
- `BeauticianProjectSkill`：项目技能匹配。
- `SmartSchedulingRun`：智能排班预览/发布记录。

## 当前算法策略

### 1. 基础班表生成

根据生成方式选择基础班表：

- `blank`：空白开始。
- `copy_last_week_optimize`：复制上周同一天同时间段。
- `optimize_current`：基于当前周已有排班。

这些基础排班会先归一化并去重。

### 2. 预约优先保留

系统会遍历当前周预约：

- 如果预约已经指定美容师，会生成该美容师的工作时段。
- 已确认、已到店、服务中预约默认保留。
- 发布时会再次把已确认预约对应时段合并进去，避免人工调整漏掉预约覆盖。

这说明当前策略对“已有预约不掉线”比较重视。

### 3. 需求热力计算

系统按固定 1 小时时段计算需求：

- 当前周真实预约数量。
- 过去 28 天同星期、同时间段的历史预约均值。
- 页面传入或门店规则配置的高峰最低人手。

每个时段的建议人数取：

`max(配置最低人手, 当前预约数, 历史期望预约数, 有预约则至少 1 人)`

需求等级：

- 高：建议人数 >= 3 或期望预约 >= 5。
- 中：建议人数 >= 2 或期望预约 >= 2。
- 低：其他。

### 4. 贪心补人

对每个需求时段，如果当前排班人数低于建议人数，系统会补美容师：

- 只选择在职美容师。
- 避开不可用时段、请假时段。
- 避免同一美容师同时间段重复排班。
- 从当前已排工时最少的人中选择。
- 工时相同则按美容师 id 排序。

这是典型的贪心启发式：速度快、逻辑可解释，但不保证全局最优。

### 5. 冲突检测

当前会识别：

- 非在职美容师排班。
- 超出营业时间。
- 重复或重叠排班。
- 与请假时段冲突。
- 预约未被排班覆盖。
- 预约与忙碌/请假冲突。
- 项目技能不匹配。

硬冲突会阻止发布；技能不足等问题作为软提醒。

### 6. 评分体系

当前评分固定为 100 分制：

- 预约覆盖率：30 分权重。
- 高峰覆盖率：20 分权重。
- 冲突情况：最高 20 分，硬冲突扣 8 分，软提醒扣 2 分。
- 工时公平：15 分权重。
- 技能匹配：10 分权重。
- 基础分：5 分。

评分用于解释和发布前判断，但不是求解器的优化目标。

## 与成熟排班算法对比

| 维度 | 当前实现 | 成熟排班系统常见能力 | 差距判断 |
| --- | --- | --- | --- |
| 算法类型 | 规则启发式 + 贪心补人 | 约束规划、整数规划、混合启发式、多目标优化 | 当前快且可解释，但不是全局最优 |
| 预约覆盖 | 已接入预约，优先覆盖已指定美容师 | 支持未指定预约自动分配、改约滚动重排、候补池 | 已有基础，缺自动分配和动态重排 |
| 高峰人力 | 支持配置高峰最低人数和历史需求 | 支持预测模型、转化率、天气节假日、营销活动影响 | 当前预测较粗 |
| 工时公平 | 按已排小时数少者优先 | 支持周/月累计、轮班公平、加班均衡、绩效倾斜 | 当前只看本次生成负载 |
| 劳动合规 | 有字段但未完整约束最大日/周工时、最小休息 | 强约束最大工时、连续工作、休息间隔、法规合规 | 关键差距 |
| 员工偏好 | 支持 availability 的 available/preferred/unavailable | 支持偏好权重、请假审批、换班、满意度 | 已有数据口，策略较浅 |
| 技能匹配 | 有技能软提醒和评分 | 技能作为硬/软约束，可做技能替代和等级优先 | 当前不会主动换人优化技能 |
| 成本优化 | 未接入薪资、提成、人效成本 | 支持人工成本最小化、收益最大化、人效约束 | 缺少经营成本目标 |
| 资源容量 | 未接入房间、设备、床位 | 同时约束人员、房间、设备、耗材 | 对美容院真实排班还不完整 |
| 目标切换 | 前端有目标选项 | 不同目标会改变优化权重和求解策略 | 当前后端没有按 objective 改变生成策略 |
| 可解释性 | 有覆盖率、冲突、解释文本 | 有原因解释、替代方案、敏感度分析 | 当前解释够 MVP，但不够决策级 |
| 发布安全 | 硬冲突阻止发布，记录 SmartSchedulingRun | 支持版本对比、回滚、审批、模拟收益 | 当前有记录，缺回滚和方案对比 |

## 关键问题

### 1. “优先目标”目前更像展示项，不是真正优化目标

前端传了 `objective`，DTO 也允许 `cover_reservations / cover_peak / fairness / reduce_staff`，但后端生成逻辑没有按不同 objective 改变候选人选择、需求人数或评分权重。

交付影响：用户选择“减少人力”或“工时公平”时，生成结果可能与“覆盖预约”差异不明显，容易产生“按钮有了但策略没变”的感知。

### 2. 排班状态口径刚收敛，智能排班服务仍保留 legacy busy

页面状态已收敛为空闲、已预约、请假、已过期，但智能排班服务和 DTO 仍允许 `busy`。这对兼容旧数据有价值，但后续要把“占用/忙碌”从产品态改成内部兼容态，避免再次出现在用户界面或新写入数据里。

交付影响：如果继续保留 busy 作为新接口合法状态，会和刚完成的四态收敛冲突。

### 3. 规则配置字段没有全部生效

`SchedulingRuleConfig` 有 `maxDailyHours / maxWeeklyHours / minRestMinutes / slotMinutes`，但当前算法主要使用营业时间、高峰规则、默认最低人手。

交付影响：系统还不能承诺“合规排班”或“防过劳排班”。

### 4. 技能匹配只评估，不主动优化

当前如果预约美容师技能不足，会产生软提醒；但算法不会主动把预约改给更合适的美容师。

交付影响：适合提醒店长确认，不适合自动完成高质量派工。

### 5. 未指定美容师预约的处理不足

当前预约补排依赖 `reservation.beauticianId`。如果预约未指定美容师，当前策略不会主动做派工分配。

交付影响：线上预约场景里，“客户只选项目不选美容师”的预约无法充分智能化。

## 推荐升级路线

### P0：先把现有策略产品化讲清楚

目标：避免用户误解当前是全局最优算法。

建议：

- 页面增加“生成逻辑说明”入口：复制上周/当前班表，优先覆盖预约和高峰，自动避开请假。
- 预览结果明确显示：新增排班数、覆盖预约数、仍需人工确认的问题。
- `objective` 暂时不要宣传为强算法目标，除非后端真的按目标改变策略。

### P1：补齐状态和约束一致性

目标：让智能排班与最新四态一致。

建议：

- DTO 不再对外暴露 `busy`，只作为旧数据兼容输入。
- 智能排班生成和发布只写 `available / leave`，预约状态来自 `Reservation`。
- 使用 `maxDailyHours / maxWeeklyHours / minRestMinutes / slotMinutes` 形成硬约束。

### P2：让不同优先目标真正生效

目标：用户选择不同目标时，生成结果有可感知差异。

建议：

- 覆盖预约：预约覆盖、技能匹配权重最高。
- 覆盖高峰：高峰时段缺口权重最高。
- 工时公平：候选人选择加入周累计工时差惩罚。
- 减少人力：在覆盖预约和高峰最低人手后，压缩低需求时段排班。

### P3：引入成熟优化模型

目标：从规则助手升级为真正的智能排班引擎。

建议采用两层架构：

1. 规则层：负责硬约束过滤。
   - 营业时间。
   - 请假。
   - 最大日/周工时。
   - 最小休息间隔。
   - 预约覆盖。
   - 房间/设备容量。

2. 优化层：负责软目标排序。
   - 高峰覆盖。
   - 工时公平。
   - 技能匹配。
   - 人工成本。
   - 员工偏好。
   - 店长自定义权重。

中小门店可以继续用启发式，复杂门店再引入 CP-SAT 或整数规划求解器。

## 产品定位建议

当前版本适合定位为：

“基于预约、高峰和请假约束的智能排班预览工具，帮助店长快速生成可发布的班表建议。”

不建议定位为：

“自动最优排班系统”或“成熟劳动力优化算法”。

原因是当前缺少全局优化、工时合规、成本目标、资源容量和动态重排。

## 验收口径建议

短期验收：

- 点击智能排班可以生成预览。
- 已确认预约不会被漏排。
- 请假冲突阻止发布。
- 高峰最低人手能被补齐。
- 分数、冲突、解释能在页面展示。
- 发布后写入 `Schedule`，并记录 `SmartSchedulingRun`。

中期验收：

- 四种优先目标生成结果不同且可解释。
- 最大日工时、周工时、休息间隔生效。
- 未指定美容师预约可以自动分配候选人。
- 技能不足可给出更优替代美容师。
- 低需求时段可自动压缩人力。

长期验收：

- 多方案对比。
- 一键回滚。
- 成本/收入/人效联动。
- 预约变动后滚动重排。
- 房间、设备、耗材容量联动。
