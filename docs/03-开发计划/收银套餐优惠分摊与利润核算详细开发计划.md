# 收银套餐优惠分摊与利润核算详细开发计划

更新时间：2026-06-21

关联方案：`docs/03-开发计划/收银套餐优惠分摊与利润核算详细改造方案.md`

## 1. 开发目标

本开发计划用于把“套餐优惠分摊与利润核算”从产品方案落到代码、数据、页面和验收闭环。

核心目标：

1. 收银支持项目、商品、卡项、充值混合结算，并支持整单优惠、套餐价、明细优惠和赠品。
2. 后端统一把整单优惠分摊到 `OrderItem` 明细，写入可复算的原价、优惠、实收和分摊来源。
3. 商品毛利、项目毛利、提成、退款、卡项履约统一使用明细净额 `netAmount`。
4. 终端收银、管理端订单、卡项开卡、充值等入口复用同一套分摊服务，避免口径漂移。
5. 所有关键场景有单测、接口测试、页面测试和手工验收样本。

## 3. 数据模型开发计划

### 3.1 OrderItem 字段

在 `OrderItem` 新增：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `listAmount` | Decimal | 0 | 原价小计 |
| `itemDiscountAmount` | Decimal | 0 | 明细级优惠 |
| `orderAllocatedDiscountAmount` | Decimal | 0 | 整单优惠分摊 |
| `totalDiscountAmount` | Decimal | 0 | 明细总优惠 |
| `netAmount` | Decimal | 0 | 明细实收 |
| `discountSource` | String | `"none"` | 优惠来源 |
| `allocationMethod` | String | `"none"` | 分摊方式 |
| `discountPayload` | Json? | null | 优惠、活动、授权、尾差快照 |
| `isGift` | Boolean | false | 是否赠品 |
| `eligibleForOrderDiscount` | Boolean | true | 是否参与整单优惠分摊 |

字段语义：

- `subtotal` 短期继续保留，并让新订单 `subtotal = netAmount`。
- 历史订单没有 `netAmount` 时，业务查询回退 `subtotal`。
- `discount` 短期继续保留，并兼容为老明细优惠字段。

### 3.2 ProductOrder 字段

在 `ProductOrder` 新增：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `listAmount` | Decimal | 0 | 订单原价合计 |
| `itemDiscountAmount` | Decimal | 0 | 明细优惠合计 |
| `orderDiscountAmount` | Decimal | 0 | 订单级优惠 |
| `totalDiscountAmount` | Decimal | 0 | 总优惠 |
| `netAmount` | Decimal | 0 | 订单应收净额 |
| `discountSource` | String | `"none"` | 订单优惠来源 |
| `allocationMethod` | String | `"none"` | 默认分摊方式 |
| `promotionId` | Int? | null | 优惠活动 |
| `couponId` | Int? | null | 优惠券 |
| `packageId` | Int? | null | 套餐模板，P2 可用 |
| `discountPayload` | Json? | null | 订单级优惠快照 |

兼容规则：

- 新订单 `totalAmount = netAmount`。
- 老订单查询时 `netAmount ?? totalAmount`。

### 3.3 Migration

新增 migration：

```text
packages/server-v2/prisma/migrations/YYYYMMDDHHMMSS_order_discount_allocation/
```

迁移原则：

1. 字段先加默认值，避免历史数据空值影响查询。
2. 不在 migration 内做复杂历史金额重算。
3. 如需历史回填，单独写只读审计脚本和可回滚回填脚本。

建议历史回填策略：

```text
OrderItem.listAmount = subtotal
OrderItem.netAmount = subtotal
OrderItem.totalDiscountAmount = discount 或 0
ProductOrder.netAmount = totalAmount
ProductOrder.listAmount = sum(OrderItem.listAmount) 或 totalAmount
```

## 4. 后端核心服务开发计划

### 4.1 新增 DiscountAllocationService

建议目录：

```text
packages/server-v2/src/orders/discount-allocation.service.ts
packages/server-v2/src/orders/discount-allocation.service.spec.ts
```

职责：

1. 规范化收银明细。
2. 计算订单原价合计。
3. 计算明细优惠。
4. 计算整单优惠。
5. 按规则分摊整单优惠。
6. 处理尾差。
7. 生成订单头和明细落库快照。
8. 校验大额优惠、赠品、不可折扣明细。

核心输入：

```ts
type DiscountAllocationInput = {
  items: DiscountInputItem[];
  discountMode?: 'none' | 'amount' | 'rate' | 'package_price' | 'manual';
  discountAmount?: number;
  discountRate?: number;
  packagePrice?: number;
  allocationMethod?: 'price_ratio' | 'manual';
  discountSource?: 'none' | 'item' | 'order' | 'package' | 'promotion' | 'coupon' | 'manual';
  promotionId?: number;
  couponId?: number;
  authorizedBy?: number;
  reason?: string;
};
```

核心输出：

```ts
type DiscountAllocationResult = {
  order: {
    listAmount: number;
    itemDiscountAmount: number;
    orderDiscountAmount: number;
    totalDiscountAmount: number;
    netAmount: number;
    discountSource: string;
    allocationMethod: string;
    discountPayload: Record<string, unknown>;
  };
  items: Array<{
    itemType: string;
    itemId?: number;
    name: string;
    quantity: number;
    unitPrice: number;
    listAmount: number;
    itemDiscountAmount: number;
    orderAllocatedDiscountAmount: number;
    totalDiscountAmount: number;
    netAmount: number;
    subtotal: number;
    discount: number;
    discountSource: string;
    allocationMethod: string;
    discountPayload?: Record<string, unknown>;
    isGift: boolean;
    eligibleForOrderDiscount: boolean;
  }>;
};
```

### 4.2 分摊算法实现

P0 支持：

- `none`
- `amount`
- `rate`
- `package_price`
- 明细级 `itemDiscountAmount`
- 赠品 `isGift=true`
- 按原价占比 `price_ratio`
- 尾差归到原价最高明细

P1 支持：

- 手工分摊 `manual`
- 优惠券适用范围
- 不可折扣明细
- 大额优惠授权

P2 支持：

- 套餐模板。
- 营销活动直接带入套餐分摊规则。

### 4.3 校验规则

必须覆盖：

1. 优惠金额不能为负。
2. 优惠不能超过可折扣金额。
3. 明细实收不能为负。
4. 赠品实收必须为 0。
5. 订单净额等于明细净额合计。
6. 明细总优惠合计等于订单总优惠。
7. 分摊尾差不超过 0.01 * 明细数量。

## 7. 退款开发计划

### 7.1 退款口径

退款必须基于明细净额。

规则：

- 整单退款：按订单 `netAmount`。
- 部分退款：按被退明细 `netAmount / quantity`。
- 已分摊优惠不能被退给客户。
- 提成按退款比例冲回或标记调整。

### 7.2 后端改造点

- discountAmount

## 13. 高风险点

1. 当前工作区已有大量经营利润、终端、订单、卡项相关未提交改动，实际开发前必须先确认这些改动是否要继续沿用。
2. Prisma schema 已处于高风险共享链路，新增字段和 migration 前需要确认现有 migration 顺序。
3. 商品/项目毛利正在改造中，切换 `netAmount` 时必须保护历史订单回退。
4. 终端和管理端如果各自实现优惠逻辑，会导致口径漂移，必须统一服务。
5. 赠品、充值赠送、办卡赠项目不能混进真实收入。
6. 退款如果不按明细净额，会导致多退或利润复算不一致。

## 14. 开发前确认事项

开始实现前建议确认：

1. 是否允许新增 Prisma migration。
2. 当前经营利润分支的未提交改动是否全部作为本次基础。
3. 提成默认按实收还是按原价。本计划建议按实收。
4. 赠品成本归入营销成本还是商品毛利页单独展示。本计划建议单独标识，并在经营成本中可汇总。
5. 首轮是否只做临时套餐价，不做套餐模板。本计划建议首轮不做模板。
6. 大额优惠授权阈值。本计划建议先配置默认阈值，后续门店配置化。

## 15. 完成定义

本开发完成不以“页面能提交”为准，而以以下闭环为准：

1. 一笔项目 + 商品 + 卡项套餐订单可以输入套餐价并完成收银。
2. 后端订单头和每个明细都保存原价、优惠、实收和分摊来源。
3. 商品库存和项目耗材按真实数量扣减，不受优惠影响。
4. 商品毛利和项目毛利均使用明细净额。
5. 提成记录能回挂 `orderItemId`，并按净额计算。
6. 部分退款按明细净额计算可退金额。
7. 赠品收入为 0，但库存和成本可追踪。
8. 订单详情、利润页面、提成记录、小票金额可以互相复算。
9. 后端定向测试、前端定向测试、终端构建和主应用构建通过。
