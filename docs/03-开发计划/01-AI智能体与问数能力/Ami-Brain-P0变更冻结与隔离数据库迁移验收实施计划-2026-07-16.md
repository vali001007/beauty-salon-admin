## 一、执行结论

本计划可执行，建议按 **范围冻结 → 数据库切片 → 隔离库双场景验证 → 代码切片 → 五层验收 → 分批提交** 的顺序推进。

当前远端数据库只读检查结果为：`packages/server-v2` 共识别 95 条 migration，`prisma migrate status` 返回 `Database schema is up to date!`。这说明当前任务不是“立即给远端补 migration”，而是验证当前未提交 migration 能否在干净环境和代表性历史数据环境中稳定重放，并形成可提交、可部署的证据。

当前 Ami Brain 相关工作面已达到高风险规模：

- 已跟踪修改：Brain runtime 61 个、semantic-query 10 个、管理端 UI 9 个、管理端 API 3 个、Prisma 3 个、semantic-data 2 个。
- 未跟踪新增：Brain runtime 153 个、semantic-data 52 个、Prisma 21 个、管理端 UI 7 个、semantic-query 4 个、管理端 API 2 个。
- 共享高风险点：`schema.prisma`、连续 migration、权限写入、Brain module、semantic-data module、管理端路由/API、发布与评测门禁。

因此，不允许把当前目录作为一个整体提交。

## 三、“现状冻结快照”的风险与控制方式

这里冻结的是 **交付范围**，不是冻结整个仓库，也不是停止其他成员工作。

| 风险 | 具体影响 | 控制措施 |
| --- | --- | --- |
| 快照很快过期 | 继续修改 schema 或 migration 后，旧清单失真 | 快照有效期 4 小时；schema/migration 任一变化立即失效并重做 |
| 错把用户 WIP 当作提交内容 | 混入其他业务或未完成试验 | 每个文件必须标记“纳入、排除、待确认”；默认排除无法解释的文件 |
| 阻塞其他开发 | 全仓冻结会影响并行任务 | 只冻结上述 Ami Brain P0 路径；其他业务继续，但不得修改共享 Prisma/路由/API client |
| 文件清单不能表达同文件多业务改动 | `schema.prisma`、module、package.json 可能包含多个目标 | 对共享文件补充 hunk 级归属说明；无法安全拆分时提升为单独基础切片 |
| 未跟踪文件丢失 | 新增源码未进入 Git，切换 worktree 或清理时易遗漏 | 快照记录未跟踪文件及哈希；执行阶段禁止 stash/reset/clean，迁移工作区前先人工核对 |
| 快照被误认为质量证明 | “文件已登记”不代表 build/test/业务通过 | 快照只证明输入边界；所有完成状态必须来自后续验收证据 |
| Live DB 状态漂移 | 远端后台任务或其他成员写入会改变数据计数 | 本次 migration 验收使用隔离库；共享库只做带时间戳的只读状态核对 |

### 冻结门禁产物

执行时生成一份 `Ami-Brain-P0范围冻结清单-YYYY-MM-DD-HHmm.md`，至少包含：

- 当前分支和 HEAD SHA。
- `git status --short --branch` 摘要。
- 目标路径下已修改/未跟踪文件清单。
- `schema.prisma` 哈希与 migration 目录哈希。
- 每个文件所属切片及排除理由。
- 生成时间、失效时间和重新生成条件。

冻结阶段不执行任何 Git 暂存或工作区搬运。

## Phase 2：隔离数据库双场景验收（1 人日）

### 场景 A：空库全量重放

目的：证明 95 条 migration 从零开始可按顺序创建完整 schema。

步骤：

1. 创建一次性 PostgreSQL 数据库，仅本任务可访问。
2. 将 `DATABASE_URL` 指向隔离库，不修改仓库 `.env`；使用当前 PowerShell 会话临时环境变量。
3. 执行 `npm.cmd run db:migrate:prod`。
4. 再执行 `prisma migrate status`，必须返回 up to date。
5. 检查关键表、字段、索引和外键：
   - `business_definition*`
   - `business_semantic_evidence`
   - `brain_capability_regeneration_job`
   - `brain_skill_registry` 新字段
6. 验证空数据下服务启动和核心查询不会因缺 seed 崩溃。

### 场景 B：历史基线增量升级

目的：证明 migration 能处理已有角色、能力目录、ontology、metric 和业务定义数据。

步骤：

1. 从脱敏的开发库结构和最小代表性数据构造基线；基线停留在目标 migration 之前。
2. 预置以下冲突样本：
   - 已有 `store_manager` 且 permissions 为非空 JSON。
   - `brain_skill_registry` 有旧能力记录。
   - ontology/metric/dimension 已有关联数据。
   - business definition 存在历史版本、重复 alias 候选和待发布投影。
3. 先运行只读预检并保存行数。
4. 执行目标 migration 序列。
5. 对比升级前后行数、权限集合、外键完整性和唯一约束。
6. 再跑一次 migration status；确认无 pending、failed、rolled_back 状态。

### 数据层验收标准

- `_prisma_migrations` 中目标 migration 全部 `finished_at` 非空，且无失败日志。
- `information_schema` 与 `schema.prisma` 的关键结构一致。
- `store_manager` 原权限不丢失，Brain 只读权限仅出现一次。
- 回填后的 definition/projection/evidence 可关联，无孤儿记录。
- 唯一约束和幂等键不存在冲突。
- 隔离库执行前后均保留带时间戳的证据快照。

### 失败处理

- 任一 migration 失败：停止后续 migration，不使用 `resolve --applied` 掩盖失败。
- 修复 SQL 后销毁隔离库并从基线重新完整重放，不在失败库上继续试补丁。
- 若旧数据违反新唯一约束，先新增只读预检和显式修复方案；数据修复脚本与 schema migration 分开评审。

## 六、Go / No-Go 标准

### Go

- 范围冻结清单未过期，且没有新增 schema/migration 漂移。
- 空库全量重放、历史基线增量升级全部通过。
- 六个切片都有独立验证结果和明确文件范围。
- store_manager 权限正反例通过。
- 管理端 build、server-v2 build、相关测试和能力 gate 通过。
- 临时评测目录、`outputs/` 和无关业务改动未进入提交范围。

### No-Go

- migration 需要 `resolve --applied` 才能继续。
- 唯一约束依赖人工删数据才能通过，且没有正式修复方案。
- schema/migration 在隔离库验收后继续变化。
- 权限测试只能证明“能访问”，不能证明写操作被禁止。
- 业务定义、能力目录和管理端页面读取不同事实源。
- 任一提交切片依赖未提交的临时文件或运行目录。
