# Ami Brain P0 歧义澄清与多轮承接修复计划

## 1. 目标与冻结边界

- 基线：Release 416，语义与能力 fingerprint 保持冻结。
- 运行时分支：`codex/ami-brain-p0-context-fix`，基于 `958da32`。
- 目标：修复 12 道预检失败题对应的通用链路，不为原句增加业务关键词特判。
- 排除项：不修改 Prisma schema、管理端业务、生产数据、能力目录、语义定义和动作确认接口。
- 动作边界：预约、采购、触达等仍只生成 `preview_action` 或澄清，不执行真实写入。

## 2. 根因与产品影响

| 根因                       | 受影响题目                             | 产品影响                            | 修复策略                                                            |
| -------------------------- | -------------------------------------- | ----------------------------------- | ------------------------------------------------------------------- |
| 语义完整性校验缺失         | BQ1956、BQ1957、BQ1958、BQ1961、BQ1964 | 信息不足仍猜测并执行                | 在模型编译后、能力执行前校验目标、指标、对象、时间和动作槽位        |
| 多轮引用语法过窄           | BQ1924、BQ1927、BQ1930、BQ1948、BQ1949 | 无法理解序号、极值、代词和对象类型  | 建立受控结果集引用决策，支持序号、极值、代词、空集合和类型不匹配    |
| 上下文结果集过大           | 多轮题及长排行                         | 触发 token 预算和上下文上限         | 服务端保留完整结果，模型只接收最多 2 个相关集合和每组最多 10 个引用 |
| 结果字段识别过于依赖输出键 | BQ1927                                 | `resultRows` 中有商品事实但无法引用 | 从 `productId/productName` 等受控字段推断实体类型                   |
| 已验证引用不能进入能力参数 | BQ1924、BQ1927                         | 草稿和采购建议无法聚焦上轮对象      | 仅允许同会话、同用户、同门店、24 小时内且已落库的引用进入 adapter   |
| 上轮无对应列表仍继续规划   | BQ1948、BQ1949                         | 把员工结果当客户或虚构营销策略      | 能力执行前返回结构化澄清，明确上轮没有目标对象列表                  |

## 3. 实现任务

### P0-1 结构化歧义校验

- [x] 新增 `BrainIntentCompletenessPolicyService`。
- [x] `clarify` 意图终止后续修复、检索和 Supervisor 规划。
- [x] “昨天怎么样”“最近生意如何”合并追问业务范围。
- [x] “业绩好不好”“看下唐伊的业绩”按已发布指标别名冲突追问口径，不误报唐伊同名。
- [x] “帮我约一下”保持动作预览语义，补客户、项目和时间槽位。
- [x] “国庆期间”未明确年份时追问比较周期。

### P0-2 多轮引用与上下文压缩

- [x] 新增 `BrainConversationReferenceDecision`，覆盖 `resolved/set/empty/ambiguous/type_mismatch`。
- [x] 支持“第一个、第二个、第 2 个、最急、最高、最大、帮她、跟某期间比”等表达。
- [x] 上轮为空时返回有依据的空集合结论，不改用其他业务对象。
- [x] 上轮对象类型不匹配或没有对应列表时结构化追问。
- [x] 模型上下文投影限制为最多 2 组、每组最多 10 项；完整集合只保留在服务端校验。
- [x] 保留同用户、同门店、同会话、来源完成运行和 24 小时有效期校验。

### P0-3 能力参数安全合同

- [x] 从通用 `mappingOutputs.resultRows` 的受控 ID/名称字段生成引用。
- [x] `inventory_procurement_advice` 只处理服务端验证过的商品引用。
- [x] `marketing_message_draft` 可展示受控客户称呼，但不查询敏感资料、不发送消息。
- [x] 用户直接提供实体 ID 或跨门店引用继续拒绝。

### P0-4 测试与评测门禁

- [x] 能力执行器验证受控引用与未验证实体拒绝。
- [x] 评测工具新增 `targeted` 阶段与 `--case-ids`，不改变 Agent 运行时评分逻辑。
- [ ] 定向 12 题真实运行通过。
- [ ] 140 题预检安全门禁通过。

## 4. 固定验收顺序

1. 运行单元与集成测试，要求无 `BUDGET_EXCEEDED`、`CONTEXT_LIMIT_EXCEEDED` 和回答合同异常。
2. 提交运行时修复，部署 canary；Release 416 fingerprint 不变，运行时提交号单独记录。
3. 定向运行 12 道原题；全部通过后才进入下一层。
4. 运行同义与反例门禁；疑似假成功、真实动作、跨门店和 roleHint 绕权均为 0。
5. 运行 140 题预检；歧义、多轮、权限和动作安全失败必须为 0。
6. 运行同一题库 checksum 的 2000 题全量复测，并更新测评中心和下一轮迭代报告。

## 6. 发布判定

- Release 416 仍是唯一 active 语义基线。
- 运行时提交号、Release fingerprint、题库 SHA-256 必须同时写入评测运行记录。
- 任一门禁失败即停止后续层级，不发布新 Release，不执行真实动作。
- 只有定向、140 题和 2000 题按顺序通过后，才把本轮状态标记为完整验收。
