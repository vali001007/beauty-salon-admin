# 新一代 Agent 架构方案（知识图谱 + LLM + 全自动能力发布）

版本：v1.1
日期：2026-07-05
定位：替代当前 V2 CapabilityDecisionService（32 个 isXxx 正则）+ 硬编码工具查询 + 手动发布流程
更新：新增 Agent 治理模块（运行审计 + 知识图谱治理 + 能力治理 + 可视化调试 + 评测）

---

## 一、设计原则

| 原则 | 说明 |
|---|---|
| 图谱只从源头生成 | 数据源是管理端/后端代码结构（Prisma + Controller + Route + 词典），不包含 V1/V2 能力定义 |
| 能力全自动发布 | 管理端/后端代码变更 → 自动扫描 → 自动门禁 → 低风险自动发布 → Runtime 热加载 |
| 意图识别 = 知识图谱 + LLM | 图谱负责结构化理解，LLM 负责语义泛化，无手写规则代码 |
| 理解与执行解耦 | 图谱回答"用户在问什么"，能力注册表回答"系统能做什么"，两者独立演进 |
| 安全不靠识别正确性 | PolicyGateway 编译时约束保障安全底线，即使识别错了也不会越权/写入/泄露 |
| 无规则代码 | 不保留 isXxx 正则、不保留 Phase 0 快速路径、不保留硬编码互斥逻辑 |
| 全链路可观测 | Agent 运行审计 + 知识图谱治理 + 能力治理 + 可视化调试 + 评测，统一在治理模块 |

## 三、知识图谱自动生成子系统

### 3.1 图谱 Schema

```text
节点类型（6 类）：
  Domain(7)  BusinessObject(16)  DataModel(122)
  Capability(动态，从 Active Manifest 发布后追加 EXCLUDES/TRIGGERS)
  ActionIntent(12)  Word(500+)

边类型（10 类）：
  BELONGS_TO       BusinessObject → Domain
  COMPOSED_OF      BusinessObject → DataModel[]
  HAS_FIELD        DataModel → Field
  FK_RELATION      DataModel → DataModel (join path + businessMeaning)
  SYNONYM_OF       Word ↔ Word / Word → BusinessObject
  TRIGGERS         Word → Capability (从 Manifest triggerKeywords 追加)
  SUPPORTS_ACTION  BusinessObject → ActionIntent
  RESOLVES_TO      不放在图里（放在运行时能力映射层）
  EXCLUDES         Capability ↔ Capability (从 negativeExamples 推导)
  REQUIRES_PERM    Capability → PermissionCode
```

### 3.2 数据源

| 数据源 | 提供内容 | 节点/边 |
|---|---|---|
| schema.prisma | 122 数据模型 + FK 关系 + 字段类型 | DataModel + FK_RELATION |
| business-object.catalog.ts | 16 业务对象 + aliases + displayFields + supportedActions | BusinessObject + SYNONYM_OF + SUPPORTS_ACTION |
| business-semantic-lexicon.ts | 填充词 + 时间词 + 动作词 + 对象提示词 | Word + SYNONYM_OF |
| action-ontology patterns | 12 种意图 | ActionIntent |
| Controller 扫描 | 600 接口 + 权限 + DTO | REQUIRES_PERM |
| 前端路由扫描 | 72 路由 + 菜单权限 | REQUIRES_PERM |
| Manifest (发布后追加) | triggerKeywords / negativeExamples | TRIGGERS + EXCLUDES |

### 3.3 生成流水线

```text
触发：CI push / deploy hook / 每日定时
脚本：npm run kg:generate

Step 1: 解析所有源 → 原始节点集
Step 2: 推导边（FK→关系、aliases→同义词、negativeExamples→互斥）
Step 3: LLM 离线增强（每周，扩充同义词 + 补充 FK businessMeaning）
Step 4: 去重、冲突检测、一致性校验
Step 5: 输出
  ├── knowledge-graph.generated.ts → 运行时 import
  ├── knowledge-graph.json → 可视化 / DB
  └── knowledge-graph-report.md → 统计 + 覆盖率 + 缺口

CI 校验：无孤立节点 / 所有 BusinessObject 有 COMPOSED_OF / 覆盖率
```

### 3.4 预估规模

| 类别 | 数量 |
|---|---|
| 节点合计 | ~713 |
| 边合计 | ~3100 |
| LLM 增强后 Word 节点 | ~1500 |
| LLM 增强后 SYNONYM_OF 边 | ~900 |

## 四、能力全自动发布子系统

### 4.1 流水线

```text
代码变更 → CI deploy 完成
    │
    ▼ (deploy hook 触发)
Step 1: 增量扫描（git diff 或 hash 对比，只扫变更部分）
Step 2: 自动分类
  ├── 只读查询 GET → auto_publish
  ├── 导航跳转 → auto_publish
  ├── 指标/诊断 → auto_publish
  ├── 动作草稿 → approval_required
  ├── 写入/删除/发券/下发 → write_blocked
  ├── 权限推断 → needs_review
  └── 工具未实现 → needs_development
Step 3: 自动门禁（全部通过才发布）
  ☑ 权限码明确
  ☑ DTO 完整
  ☑ queryKey dry-run 通过
  ☑ 字段策略已绑定
  ☑ 评测门禁通过
  ☑ 回答契约满足
  ☑ 风险等级 = low
Step 4: 自动发布 → 新 Manifest 版本 → Runtime 热加载
Step 5: 图谱联动（新 Manifest 的 EXCLUDES/TRIGGERS 追加到图）
```

### 4.2 触发机制

```text
方式 1: CI deploy hook（主要）
方式 2: Cron 每日凌晨 3 点（兜底）
方式 3: 管理端手动触发
```

### 4.3 安全边界

| 分类 | 自动发布 | 理由 |
|---|---|---|
| 只读查询 | ✅ | 工具已实现 + 权限明确 + 评测通过 |
| 导航跳转 | ✅ | 前端路由已存在 |
| 指标/诊断 | ✅ | 只读聚合 |
| 动作草稿 | ❌ 需审核 | 涉及写入意图 |
| 写入/删除/发券/下发 | ❌ 禁止 | 安全红线 |
| 权限推断 | ❌ 不自动 | 必须人工确认权限码 |
| 工具未实现 | ❌ 不自动 | dry-run 会失败 |

## 五、意图识别层详细设计

### 5.1 数据流

```text
用户："哪些客户买了次卡但最近一直不来用"
    │
    ▼
1. 知识图谱预处理（< 5ms）
   - 去噪：COMMON_FILLER_TERMS 移除
   - 同义词展开：次卡→MemberCard, 不来用→沉睡→inactive
   - 对象定位：[MemberCard, Customer]
   - 互斥加载：card.usage(核销) EXCLUDES card.package.order(开卡)
    │
    ▼
2. LLM 结构化抽取（~300ms，缓存命中 < 1ms）
   输入：问题 + 图谱上下文 + Active Manifest 摘要
   输出：{
     objects: ['MemberCard', 'Customer'],
     domain: 'customer',
     action: 'diagnose',
     timeIntent: 'historical_pattern',
     keywords: ['次卡', '沉睡', '不来用'],
     candidateCapabilities: ['card.package.inactive-customers.list'],
     confidence: 0.91
   }
    │
    ▼
3. 能力映射（< 1ms）
   - 查 Active Manifest → 存在 ✓
   - negativeExamples 检查 → 未命中 ✓
   - 选中
    │
    ▼
4. 安全策略 → 5. 工具执行 → 6. 回答契约 → 7. 输出
   总延迟：~360ms（首次）/ ~60ms（缓存命中）
```

### 5.2 LLM Prompt

```text
你是美业门店经营 Agent 的意图解析器。
根据用户问题和系统提供的上下文，输出结构化意图。

## 图谱上下文（系统自动注入）
同义词展开：{{synonymExpansion}}
对象提示：{{objectHints}}
领域提示：{{domainHints}}
互斥提醒：{{exclusions}}
可用能力摘要：{{capabilitySummary}}

## 规则
1. objects: 从图谱上下文的对象提示中选择
2. domain: 所属领域
3. action: lookup/list/summary/diagnose/analyze/compare/draft/get_link/print
4. timeIntent: occurred/risk/trend/current
5. candidateCapabilities: 从"可用能力摘要"中选 1~3 个
6. confidence: 0~1，低于 0.6 说明不确定

## 重要区分
- "已报废/记录/流水" → timeIntent = occurred
- "快过期/风险/即将" → timeIntent = risk
- "趋势/变化/走势" → timeIntent = trend
- "帮我执行/帮我操作" 才是 draft，其他都是查询

## 用户问题
{{question}}

## 输出（JSON）
```

### 5.3 降级策略

```text
LLM 不可用时 → 纯图谱评分：
1. 同义词展开获得 keywords
2. keywords 与 Active Manifest triggerKeywords 交集
3. 按交集数量 + 对象匹配 + 动作匹配加权
4. 最高分 >= 阈值 → 选中
5. 否则 → "不支持"
```

### 5.4 缓存设计

```text
LRU 500 条 / TTL 5 分钟 / key = normalize(question)
门店高频问法重复率 > 60%，缓存大幅减少 LLM 调用
月成本预估：日均 500 次 × 30 天 × 40% 未命中 × ¥0.001/次 ≈ ¥6
```

## 六、Schema-Driven 通用查询引擎

### 6.1 设计

```text
输入：Manifest.sourceModels + Manifest.outputKinds + 知识图谱
处理：
  1. 从 sourceModel 确定主查询表
  2. 从图谱 FK_RELATION 自动发现 join 路径
  3. 从 BusinessObjectCatalog.displayFields 自动翻译字段
  4. 从 Manifest.fieldPolicies 过滤敏感字段
  5. 构建 Prisma 动态查询（where + select + orderBy + take）
  6. 按 outputKinds 组装结果结构

新增业务对象查询 = Manifest 中声明 sourceModel + outputKinds
不需要写任何查询代码
```

```text
    tool: 'record.query',
    queryKey: 'reservation.records',
    sourceModel: 'Reservation',
  },
  sourceModels: ['Reservation'],
  outputKinds: ['table'],
  // ...
}
```

## 七、安全保障（不靠识别正确性）

```text
安全三层防线：

1. Manifest 发布门禁（上线前）
   - write_blocked 能力根本不会出现在 Active Manifest
   - 必须通过 queryKey dry-run + 评测门禁 + 字段策略

2. PolicyGateway（运行时）
   - 即使 LLM 错误匹配到某能力，也要过权限/角色/门店/风险
   - 高风险能力需人工审批

3. ContractValidator（输出后）
   - 形态不符直接拦截
   - "问列表给了预测" 会被捕获

最坏情况：LLM 识别错了 → 执行了错误的只读查询 → 返回无关数据
→ 没越权、没写入、没泄露 → 用户看到答非所问重新问
→ ContractValidator 大概率能拦截
```

## 八、与 V1/V2 的关系

```text
复用 V2 的：
├── Manifest 体系（能力定义 + 能力中心 + 版本发布 + 热加载）
├── PolicyGateway（安全策略层完全保留）
├── ContractValidator（回答契约完全保留）
├── EvidenceService（证据包完全保留）
├── 能力中心 DB 表（6 张治理表完全保留）
└── AgentV2Module 注册 + Controller + 独立 API 入口

替换 V2 的：
├── CapabilityDecisionService（32 个 isXxx 正则）
│   → 替换为：知识图谱 + LLM 意图抽取
├── 硬编码工具查询逻辑
│   → 替换为：Schema-Driven 通用查询引擎
└── 手动 import + 手动 publish
    → 替换为：全自动发布流水线

V1 的处理：
├── 终端快捷操作（收银/核销等流程型）→ 保留，不走意图识别
├── V1 知识层（SchemaGraph/Catalog/Lexicon）→ 合并为新图谱数据源
└── V1 Planner/旧工具 → 不再维护，等新架构覆盖后退役
```

## 九、评测门禁

| 指标 | 要求 | 验证方式 |
|---|---|---|
| P0 正确率 | >= 98%（103 题允许最多 2 题走降级） | 650 题跑 5 次取最差 |

| LLM 降级覆盖 | 降级路径 P0 >= 85% | 模拟 LLM 不可用 |
| 延迟 P99 | <= 800ms | 性能测试 |
| 缓存命中率 | >= 50% | 线上监控 |

## 十、Agent 治理模块

### 10.1 定位

Agent 治理模块是新架构的**运营闭环**——所有运行时数据、知识图谱状态、能力发布结果在此汇聚，提供可视化调试、评测、审计能力。对应架构图中底部的"全链路审计数据写入"。

管理端入口：`系统设置 / Agent 治理中心`
权限码：`core:agent-governance:view`（查看） / `core:agent-governance:manage`（操作）

### 10.2 模块结构

```text
Agent 治理中心
├── 运行审计
│   ├── 运行记录列表（全量 runs，按时间/门店/角色/状态/能力筛选）
```

### 10.7 关键交互流程

**流程 1：从线上失败到图谱修复**

```text
1. 用户问"这个月人效怎么样" → Agent 返回 unsupported
2. 运行审计记录失败，进入"未覆盖高频 Top 20"
3. 管理员在治理中心看到 → 点击"分析原因"
4. 系统展示：图谱中"人效"无 SYNONYM_OF 边
5. 管理员点击"新增同义词" → 人效 → 人均产出 → FinanceMetric
6. 下次 kg:generate 合入 → LLM 抽取时图谱上下文包含该同义词
7. 下次用户问同样问题 → 命中财务能力 → 返回正确结果
```

**流程 2：从自动发布到评测验证**

```text
1. 后端新增 ReservationController → deploy
2. auto-pipeline 扫描到新接口 → 生成候选能力
3. 自动门禁：queryKey dry-run 通过 → 评测门禁通过 → 自动发布
4. 自动发布日志写入治理模块
5. 管理员在"能力健康看板"看到新增能力
6. 一周后在"命中热度"中观察该能力使用情况
7. 如果从未命中 → 进入"冷能力"列表 → 评估是否应停用
```

**流程 3：可视化调试辅助开发**

```text
1. 开发者在"可视化调试"输入测试问题
2. 系统展示：图谱预处理 → LLM Prompt → LLM Response → 映射过程
3. 发现 LLM 选错能力 → 切换"对比模式"
4. 对比：修改 Manifest triggerKeywords 后的命中差异
5. 确认修改有效 → 在"Manifest 模拟"中验证不影响其他题
6. 确认无副作用 → 进入能力治理正式发布
```

## 十二、架构优势总结

| 维度 | 旧方案（V2 规则引擎） | 新方案（图谱 + LLM） |
|---|---|---|
| 意图识别 | 32 个 isXxx 正则，手动穷举 | LLM 天然理解同义词 + 图谱结构化增强 |
| 长尾口语覆盖 | ~65% | ~92% |
| 新增能力 | 加 isXxx + 加正则 + 部署 | Manifest 声明 sourceModel → 自动发布 |
| 同义词维护 | 手动加正则 | 图谱自动生成 + LLM 每周扩充 |
| 规则代码量 | ~463 行（decision service） | 0 行 |
| 安全保障 | 规则正确性 + PolicyGateway | 纯 PolicyGateway（编译时约束） |
| 工具开发量 | 每个能力手写查询 | 通用引擎，Manifest 声明即可 |
| 发布流程 | 手动 import + 手动 publish | deploy hook → 全自动 |

## 十三、一句话定位

知识图谱理解语义 → LLM 抽取意图 → Manifest 映射能力 → PolicyGateway 保安全 → 通用引擎执行查询 → 契约校验拦截答非所问 → 治理模块全链路可观测。全程无手写规则，无硬编码逻辑，能力随代码迭代自动发布，运行全链路可调试可审计可评测。
