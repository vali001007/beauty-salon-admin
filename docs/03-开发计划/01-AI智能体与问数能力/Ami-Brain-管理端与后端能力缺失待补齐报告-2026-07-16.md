# Ami Brain 管理端与后端能力缺失待补齐报告

> 日期：2026-07-17
> 依据：Ami Brain r40 前 120 题、r44 假阳性、r81/r82 财务对比与诊断真实请求路径评测，以及对 Prisma Schema、server-v2 Controller/Service、管理端路由与页面的代码扫描。
> 目的：把“Ami Brain 没有接技能”和“管理端/后端本身没有业务事实”分开，避免 Agent 团队在缺少事实源时继续堆提示词、关键词或假查询。

## 一、结论

当前失败问题分为三类：

| 类型 | 含义 | 处理责任 | 产品风险 |
| --- | --- | --- | --- |
| A. 数据和 API 已存在，Brain 未接入 | 后端已有可靠事实，可直接补 Business Definition、Capability 和回答合同 | Ami Brain | 可在下一阶段快速转化为可用率 |
| B. 底层存在零散数据，但缺统一关系或业务口径 | 能查到局部数据，无法形成可审计结论 | 管理端/后端主责，Brain 配合 | 直接做 Agent 会产生错误归因和假阳性 |
| C. 业务模型、API、管理页面均缺失 | 系统没有事实入口，Brain 无数据可读 | 管理端/后端 | 必须先建设业务闭环，不能由模型猜测 |

扫描确认：r40 的 61 条可评测非可用题中，大量问题不是“再加一个关键词”就能解决。投诉、满意度、员工生命周期、考勤、服务事故、消防巡检、客户归属转移等事实在当前 schema 和管理端没有完整落点。Ami Brain 应明确拒答，并把缺口进入本报告跟踪。

## 二、代码扫描边界与现有基础

### 2.1 已存在的可复用事实

- 客户基本资料、来源、消费、预约、卡项、健康档案：`Customer`、`CustomerHealthProfile`、`CustomerCard`、`CardUsageRecord`。
- 订单、折扣、支付、退款、成本与提成：`ProductOrder`、`OrderItem`、`PaymentRecord`、`RefundRecord`、`CommissionRecord`、`DailySettlement`。
- 服务、项目、员工技能与排班：`ServiceTask`、`Project`、`ProjectBomItem`、`BeauticianProjectSkill`、`BeauticianAvailability`、`BeauticianTimeOff`。
- 库存、批次、耗用、供应商和采购：`StockBatch`、`StockMovement`、`SupplySupplier`、供应平台采购/收货/结算接口。
- 客户生命周期和营销归因：`CustomerLifecycleEvent`、`CustomerOpportunity`、`LifecycleAttributionEvent`、`CustomerBehaviorEvent`、`CustomerAppEvent`、营销触达与归因表。

### 2.2 明确不存在的独立业务模块

server-v2 当前没有投诉工单、客户满意度、HR 员工档案/合同/转正、考勤、服务事故、消防巡检等模块目录；管理端页面也没有对应业务页面。员工管理目前主要是美容师基础资料和等级，不能承担完整人事管理。

### 2.3 当前局部事实的关键断点

- `Reservation` 有 `checkedInAt`，`ServiceTask` 有 `startedAt`，但两者没有直接关系，无法可靠计算“到店后等待多久”及等待流失。
- `BeauticianProjectSkill` 只有技能等级、认证和优先级，没有项目服务满意度或客户反馈。
- `ProductOrder` 有折扣来源和折扣金额，但没有折扣审批实例、授权额度、审批人和规则命中记录。
- `CustomerBalanceTransaction` 能记录充值/扣减，但没有提现申请、审批、风险判定和资金去向模型。
- `CustomerLifecycleEvent` 与 `LifecycleAttributionEvent` 能记录生命周期变化，但缺统一“流失原因”和“重新激活证据”规则中心。
- `PurchaseOrder` 仍是弱结构 JSON，供应平台采购能力更完整，但采购退货和退货原因闭环未形成统一事实。

## 四、B 类：已有局部数据，但管理端/后端必须先统一

## 六、建议交付顺序

### Phase B1：先补 P0 事实入口

1. 客诉与满意度中心。
2. 服务事故与过敏处置。
3. 员工档案、合同、试用转正和考勤。
4. 折扣审批审计与客户归属台账。
5. 预约排队、等待超时和离店原因。

完成标准：管理端能录入、处理和关闭；后端有门店隔离 API 和审计字段；Brain 只读查询能获得结构化事实。

### Phase B2：补经营与供应链关系

1. 项目成本版本与成本上涨解释。
2. 缺料阻断、库存预占和服务失败原因。
3. 采购退货与供应商责任归因。
4. 储值提现申请与风险审批。
5. 统一风险事件/任务中心。

### Phase B3：补增长归因

1. 统一转介绍关系。
2. 多端项目兴趣事件合并。
3. 流失原因与再激活证据规则。
4. 会员权益满意度。

## 七、与 Ami Brain 的对接合同

每个新增管理端/后端能力完成时，必须同时提供以下机器可发现信息，避免两端再次失配：

1. Prisma Model/Relation/Enum 和字段级门店范围。
2. Controller/DTO/Service 的只读与写操作声明。
3. 权限码、敏感字段、审批要求和风险等级。
4. Business Definition：业务对象、指标、维度、口径、时间语义和证据来源。
5. Capability Scanner 可读取的执行器绑定和正反例。
6. 变更指纹与 drift diff；破坏性变更阻断候选发布。
7. 管理端只负责统一业务定义和风险审批，不在 Ami Brain 内重复维护口径。

## 八、验收门禁

- 新增业务必须具备：数据模型、API、权限、管理页面、审计、测试六项，不以“表已建”判定完成。
- Brain 不得在底层事实缺失时返回估算、通用名单或模型推断结论。
- 所有列表、排行、原因分析必须返回业务对象维度和证据引用。
- 新增写操作先进入 `preview_action`，用户只做通过、修改或拒绝；生产执行需单独授权。
- 每轮评测报告同时输出：Brain 能力缺口、底层能力缺口、数据质量缺口，三类不得混报。

## 十、2026-07-17 财务诊断新增底层缺口

### 10.1 P0：历史结算与成本归属可信度

R81/R82 使用门店 6 真实数据诊断时，上一可比期出现：

- 毛利率 `-278.4%`
- 物料成本率 `378.4%`
- 经营费用率 `8040.9%`

Ami Brain 已增加可信度门禁，不再据此输出“毛利上升/下降根因”。但这只是防止误导，不等于底层数据已修复。

| 缺口 | 当前状态 | 后端待补 | 管理端待补 | 验收标准 |
| --- | --- | --- | --- | --- |

该缺口必须在客户主数据层统一解决，不能由 Ami Brain、营销页和报表各自维护一套测试姓名黑名单。

### 24.3 当前开发边界

- 当前 Ami Brain 只继续修复“已有事实但路由、工具选择、规划或输出合同未接好”的失败项。
- 需要新增管理端业务流程、后端事实表、migration 或真实采集的数据问题，统一登记后另起开发任务。
- 外部任务完成后，由 Capability Scanner 自动发现契约变化并生成候选能力，经审批、评测和发布接入；不手工同步双份技能定义。

## 二十六、2026-07-18 创建预约业务表级幂等缺口

### 26.1 当前事实

- `Reservation` 当前只有门店、客户、项目、美容师、日期、时间、状态和备注等字段，没有 `externalRequestId`、`idempotencyKey` 或来源请求唯一键。
- `ReservationsService.create` 完成参数构造后直接执行 `prisma.reservation.create`。
- `BrainActionExecution` 的唯一键只能保证 Brain 不主动创建第二条执行记录，不能证明下游预约写入没有成功。
- 当业务预约已经写入、但 Brain 回执更新或网络响应失败时，盲目重试创建预约会生成重复预约。

### 26.2 后续统一业务任务

| 层级 | 待补能力 | 验收标准 |
| --- | --- | --- |
| 后端数据模型 | 为预约创建增加来源系统、外部请求幂等键和门店范围唯一约束 | 同一门店、同一来源、同一请求键重复提交只返回同一预约 |
| 预约 Service | 提供事务化 `createIdempotent` 或等价合同，区分首次创建、重复命中和参数冲突 | 首次请求创建；同参数重放返回原单；不同参数复用同键明确报冲突 |
| API 契约 | 管理端、客户 H5、终端和 Brain 共用统一幂等请求头或 DTO 字段 | 不由各客户端在备注中自行编码幂等信息 |
| 管理端 | 在预约详情显示来源、外部请求标识和重复请求对账信息 | 用户可判断动作是否已落单，不需要查询 Brain 内部表 |
| 测试 | 增加并发重复提交、业务写入后响应失败、跨门店同键和参数冲突用例 | 不产生重复预约，跨门店互不影响，冲突可解释 |

### 26.3 当前 Ami Brain 行为

- 创建预约失败继续返回 `manual_reconcile`，不显示自动重试按钮。
- 用户刷新或重新打开会话后，可从持久化动作状态看到业务回执、失败原因和人工核对要求。
- 在统一预约幂等合同上线前，不把客户、项目、日期、时间拼成推测性唯一键，也不把幂等 token 写入备注。
- 后端任务完成后由 Capability Scanner 发现契约变化，经动作安全测试和发布门禁升级为可安全重放能力。

## 二十七、2026-07-18 次卡核销底层合同收口与剩余幂等缺口

### 27.1 本轮已补齐的统一后台能力

| 问题 | 修复 | 产品影响 |
| --- | --- | --- |
| 同一卡并发核销可能同时读取旧余次 | 核销事务对 `CustomerCard` 执行行锁后再统计项目余次和扣减 | 同一卡并发请求串行处理，不会因丢失更新超扣 |
| 同客户同名卡项目次数可能混算 | 项目核销累计改为按 `customerCardId + projectName` | 每张客户卡独立计算项目余次 |
| 跨门店美容师可能先写入核销记录 | 扣次前验证美容师属于当前门店且为启用状态 | 核销、提成和服务归属保持同门店 |
| 请求体可伪造 operatorId | Controller 强制使用当前登录用户 ID | 核销审计人不可由客户端覆盖 |

这些修复属于统一 `CardsService` 和 `/cards/verify-usage` 合同，管理端、终端和 Ami Brain 共用，不在各客户端重复维护。

### 27.2 仍需后续后端任务补齐

`CardUsageRecord` 当前没有外部幂等请求键和唯一约束。Ami Brain execution 唯一键可以避免同一确认被主动重复执行，但无法覆盖“核销事务已提交、Brain 回执持久化失败”的不确定结果。

| 层级 | 待补能力 | 验收标准 |
| --- | --- | --- |
| 数据模型 | 为核销流水增加来源系统、外部请求键和门店/来源唯一约束 | 同一请求键重复提交只返回同一核销记录 |
| 核销 Service | 支持同参数重放返回原记录，不同参数复用同键明确冲突 | 不重复扣次数、耗材、收入和提成 |
| 管理端 | 核销详情展示来源、外部请求键和重复请求对账状态 | 用户可判断是否已经核销，不依赖 Brain 内部执行表 |
| 测试 | 覆盖事务提交后响应失败、并发同键、跨门店同键和参数冲突 | 所有副作用只发生一次且可追溯 |

### 27.3 当前安全策略

- 次卡核销在 Brain 中固定为 `critical + requiresConfirmation + manual_reconcile`。
- 执行失败不开放自动重试，用户先核对核销记录和客户余次。
- 业务表级幂等上线并完成隔离库故障注入前，不将该动作改为 `safe_replay`。
