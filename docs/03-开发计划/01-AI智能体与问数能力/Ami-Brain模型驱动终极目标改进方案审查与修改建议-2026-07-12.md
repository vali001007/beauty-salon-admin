# Ami Brain 模型驱动终极目标改进方案审查与修改建议

生成日期：2026-07-12

审查对象：`Ami-Brain-模型驱动经营智能体终极目标改进方案与实施计划-2026-07-12.md`

审查结论：**架构方向成立，但当前文档不能直接进入实施。必须先完成一次 v3 单一执行版重写，消除范围冲突、历史包袱、任务顺序错误和不可验收指标。**

## 1. 审查边界

本建议遵守以下边界：

- 只修改 Ami Brain 目标方案，不建议复用任何历史 Agent 代码、架构、运行时、API 或能力目录。
- 借鉴综合历史复盘中已经确认的失败教训：范围失控、评测假阳性、语义多真相源、能力供给靠人工补洞、fallback 掩盖失败、问答与动作不闭环、旧链路长期不退役。
- 历史内容只作为风险证据，不作为 Ami Brain 的设计输入或实现资产。
- Ami Brain 继续保持独立 `brain` 命名空间、`brain_*` 数据模型和 `/api/brain/*` 协议。

## 2. 总体评价

方案的核心方向正确：

- 模型负责结构化理解和受控规划。
- 系统负责权限、参数、预算和执行。
- 单技能快速通道与复合任务通道分离。
- 模型不能写 SQL、扩大权限或自动确认写操作。
- 计划、Observation、完成度和回答均可追踪。
- 模型上线前先做真实 provider spike、离线回放和小流量灰度。

但方案存在四个结构性问题：

1. v2 只在文档顶部宣布范围降级，正文、文件清单、Task 和验收仍保留原重方案。
2. 同一期同时建设模型认知、Ontology Runtime、Capability Scanner、Catalog、Supervisor、DAG、Response Blocks、Memory、Actions、Inspection、Governance 和跨端渲染，范围仍然过大。
3. 关键评测资产和真实业务任务没有成为第一阶段输入，仍然先建平台、后证明产品可用。
4. 多个核心合同允许模型输出过多控制字段，确定性系统没有完全掌控权限、预算、执行状态和完成判定。

因此，本方案应由“大一统终极实施计划”改为“分阶段产品门禁计划”。每个阶段先证明用户价值，再开放下一层复杂度。

## 3. P0 必须修改项

### 3.1 将 v2 覆盖层改成完整 v3 单一执行版

当前文档写明“正式执行以第 0 节为准”，但执行人员仍会从正文和 Task 获取文件、步骤和验收命令。文档内部存在直接冲突：

| v2 已声明 | 正文仍保留的冲突内容 |
| --- | --- |
| 去掉 shadow，改离线回放 | §5.3 仍有 `shadow` 模式；P16、文件清单、Task 20、测试和治理台仍要求 shadow diff |
| Scanner 只生成 draft + 漂移 | §5.4 自动化等级 A、Task 8A 仍生成 executor binding、codegen、分支提案和 `brain:capability:generate` |
| 全自动能力生成移出本期 | §7 仍创建 `brain-capability-codegen.service.ts`，Task 8A 仍执行 codegen |
| 业务语义中台独立立项 | Task 8B 仍包含 migration、Registry、Projection Compiler、Controller、管理端页面和路由 |
| 首批 executor 全部手写 | §12.3 最终判定仍要求 Scanner 自动生成 executor binding |

修改建议：

- 不再保留“原方案正文 + v2 覆盖说明”的双轨结构。
- 直接生成 v3 完整方案，只保留本期真实执行内容。
- 移出本期的内容放入单独“后续研究项”文档，不留在 Implementation Tasks、文件清单、验收和排期中。
- v3 文档中每个目标只出现一次，不使用“以顶部修订为准”覆盖后文。

### 3.2 删除本期所有 shadow 运行内容

方案已经选择“离线回放 + 1% 小流量灰度”，就必须完整删除 shadow 机制，避免研发同时建设两套验证链路。

应删除或改写：

- `BRAIN_MODEL_SHADOW_PERCENT`。
- 运行模式中的 `shadow`。
- `brain-cognition-shadow.service.ts`。
- P16 的 shadow diff 和 shadow 灰度。
- `ami-brain-eval.ts` 的 shadow 对比输出。
- 治理台 rules/model shadow diff。
- E2E 中查看 shadow diff。
- Task 20 的 shadow 发布步骤。

保留以下验证链路：

```text
离线真实问句回放
-> 人工抽检
-> 1% canary
-> 5%
-> 20%
-> 50%
-> 100%
```

每个灰度阶段必须定义最小请求数、最短观察时间、错误预算和自动停止条件。

### 3.3 将 Task 0 移到所有实施任务之前

当前 Task 0 位于 Task 6 之后，执行顺序与“先 spike 再定门禁”冲突；单人排期也从 P11 开始，遗漏 P10。

建议顺序：

1. Task 0A：模型结构化生成 spike。
2. Task 0B：Clean-room 产品任务和评测集。
3. Task 1：运行配置。
4. Task 2：语义合同。
5. Task 3：结构化生成正式实现。
6. 后续任务。

P10 未通过时，不得进入 P11，不得提前实现 Supervisor、Scanner 或写操作。

### 3.4 新增 Task 0B：Clean-room 产品任务与评测基线

方案仍以既有 650 题作为主要意图准确率来源，这会把研发带回“围绕固定题库做高分”的路径。历史复盘已经证明，题库结构完成率不能替代真实任务可用性。

建议新建 Ami Brain 独立评测集，不继承任何历史 Agent 标签或成功口径。

Task 0B 必须交付：

- 12 个核心用户任务。
- 每个任务 10 条基础表达、5 条同义改写、3 条纠正或冲突表达、2 条权限或安全表达。
- 固定门店、冻结时间、稳定业务对象和精确预期结果。
- `expectedIntent`、`expectedCapability`、`expectedOutputShape`、`expectedCompletionState`、`expectedEvidence`。
- 排行、名单、对比、趋势和动作预览的结构化预期。
- 人工标注规范和双人复核流程。

既有 650 题只允许作为额外压力集，不作为核心产品门禁，也不允许按原标签直接训练或调规则。

### 3.5 先锁定 12 个核心任务，停止全域同时扩展

当前方案在 Task 4、Task 9、Task 13、Task 15 和 Task 18 同时铺开多个领域和复杂能力。建议第一阶段只完成以下任务：

1. 今日经营概览。
2. 指标精确查询。
3. 时间对比。
4. 商品或项目排行。
5. 客户精确查询。
6. 预约清单。
7. 员工服务安排。
8. 实收、退款和折扣核对。
9. 库存数量和风险清单。
10. 采购建议。
11. 预约动作预览。
12. 跟进任务动作预览。

核心任务未达到门禁前，不进入跨域 Supervisor、主动巡检、长期记忆和自动能力发现。

### 3.6 从主计划彻底移除 Task 8B 业务语义中台建设

第 0 节已经决定把业务语义中台独立立项，但 Task 8B 仍包含完整后端、数据库、管理端和迁移任务。这会再次把 Brain 项目变成全公司数据治理项目。

本期 Task 8B 应替换为：

> **Business Definition Reference Adapter**

只交付：

- 读取现有稳定指标 key 和版本。
- 校验引用存在、可用且未过期。
- 保存 `businessDefinitionRef`、版本和 fingerprint。
- 引用失效时阻断 Capability 发布。
- 生成上游口径缺口报告。

以下内容全部移出本期：

- 新 Business Definition Registry 数据模型。
- 共享 Projection Compiler。
- 业务口径 Controller 和管理端页面。
- 报表、API、Semantic Query 全量迁移。
- 自动发布别名和指标口径。

### 3.7 从主计划彻底移除自动 codegen

Task 8A 仍要求生成 executor binding、契约测试骨架、独立分支变更提案和 `brain:capability:generate`。这与 v2 的“首批 executor 全部手写”直接冲突。

本期 Scanner 只允许输出：

- source metadata。
- Capability draft。
- input/output schema draft。
- 权限、store scope 和风险证据。
- sourceFingerprint。
- added/changed/removed/stale/blocked diff。

禁止输出：

- 可运行 executor binding。
- 自动代码修改。
- migration 提案。
- 自动发布。
- 自动生成后直接进入运行目录。

首批 12 个 Capability Handler 全部人工实现、人工审查并绑定独立测试。

### 3.8 不允许 P11-P12 完成后直接宣称“经营问数助手”

当前排期声明 P11-P12 完成后可对外定位为“模型驱动的经营问数助手”，但 Grounded Answer、Completion Guard、结构化 Response、跨端渲染和事实引用都在 P14/Task 15。

没有完成回答合同和完成度护栏时，系统只能证明“模型能理解并选择能力”，不能证明用户获得可信答案。

建议调整：

- 将 Task 15 的最小版提前到 R1。
- R1 同时完成 Semantic Intent、单 Capability、Observation、Completion State 和最小 Response Envelope。
- 只有 12 个核心任务达到真实可用门禁后，才能宣称“模型驱动经营问数助手”。

### 3.9 将主动巡检从真实动作阶段拆出

P15 同时包含真实动作和主动巡检，两个系统的风险、验收和运营指标完全不同。

建议拆分：

- P15A：受控动作 preview/confirm/execute/receipt。
- P15B：动作稳定后再接 inspection finding。

P15B 的前置条件：

- P15A 连续试点无越权、跨店、重复写入和假回执。
- 巡检人工标注集已经建立。
- 巡检具备真阳性率、采纳率、处置率和去重率指标。

### 3.10 长期记忆不得与核心回答同时上线

P14 同时交付回答、角色和长期记忆，增加了不可控变量。历史失败经验表明，错误上下文和错误记忆会让回答看起来连贯，但事实更难追踪。

建议拆分：

- R1 只保留会话内槽位、上一轮对象和用户即时纠正。
- R2 再开放用户偏好记忆。
- 门店规则记忆必须由有权限用户确认。
- 实时指标、客户敏感信息和模型推断不得进入长期记忆。
- 长期记忆未通过冲突、撤销、过期和跨店测试前，不进入规划输入。

### 3.11 禁止修改历史命名的 CI 工作流

Task 8A 要求修改 `.github/workflows/agent-v2.yml`，这直接违反 Ami Brain clean-room 隔离边界。

建议：

- 创建 `.github/workflows/ami-brain.yml`，或把 Brain 检查接入明确的通用 server-v2 CI。
- Brain 代码、测试、迁移和评测不引用任何历史 Agent 工作流、命令或产物目录。
- 评测产物改存到独立目录，例如：
  `docs/04-测试数据/Ami-Brain模型驱动评测-2026-07-12/`。

### 3.12 排期必须在 P10 后重新估算

文档一方面承认原 16 周明显乐观，另一方面仍保留“总工期 16 周”和单人 28 周结论。这会把未经校准的数字继续传递给产品和管理层。

建议：

- 当前只给阶段范围，不给最终固定总工期。
- P10 spike 后根据 provider 合法率、意图准确率、单次成本和延迟重新估算 P11。
- R1 完成后根据 Capability 实现成本和评测标注速度重新估算后续阶段。
- 单人串行顺序必须包含 P10 和 Task 0B。

## 4. 核心合同修改建议

### 4.1 `BrainSemanticIntent`

当前合同中的 `filters: Record<string, unknown>` 过于宽松，`reasoningSummary` 也不应要求模型输出内部推理摘要。

建议修改：

```typescript
interface BrainFilterClause {
  fieldKey: string;
  operator: 'eq' | 'neq' | 'in' | 'contains' | 'gt' | 'gte' | 'lt' | 'lte';
  value: string | number | boolean | Array<string | number>;
}

interface BrainSemanticIntent {
  schemaVersion: '1.0';
  objective: string;
  intent: BrainIntentType;
  domains: string[];
  entities: BrainEntityReference[];
  metrics: string[];
  dimensions: string[];
  filters: BrainFilterClause[];
  timeRange?: BrainSemanticTimeRange;
  orderBy: BrainOrderClause[];
  answerShape: BrainAnswerShape;
  missingSlots: string[];
  assumptions: string[];
  confidence: number;
  decisionSummary: string;
}
```

必须增加：

- `missingSlots`。
- `assumptions`。
- 实体来源和原文 mention。
- 字段、指标和维度的 definition/version 引用。

必须删除：

- 任意 `Record<string, unknown>` 过滤参数。
- 要求模型输出隐藏推理过程的字段。
- 任何由模型决定的 userId、storeId、permissions 和 data scope。

### 4.2 `BrainCapabilityCard`

建议补充：

- `handlerKey`。
- `businessDefinitionRefs`。
- `sourceFingerprint`。
- `owner`。
- `status: draft | active | stale | disabled`。
- `deprecation` 和替代能力。
- `retryPolicy`。
- `evidenceSchema`。
- `evalPackVersion`。
- `createdFrom: manual | scanner_draft`。

Capability 描述和 examples 属于不可信检索内容，发布前必须经过审核并锁定版本，不能在运行时被用户输入修改。

### 4.3 `BrainExecutionPlan`

当前合同允许模型输出 `roleKey`、节点 timeout、总 timeout 和 maxReplans。模型不应决定执行身份和系统预算。

建议拆成两个对象：

```text
BrainProposedPlan
-> Plan Validator
-> BrainValidatedPlan
```

`BrainProposedPlan` 只包含：目标、节点、Capability、参数、依赖、成功条件和失败建议。

`BrainValidatedPlan` 由系统补充：

- actor 和 store scope。
- required permissions。
- timeout。
- retry policy。
- maxReplans。
- risk level。
- confirmation boundary。
- plan version 和 hash。

模型不得输出或覆盖 `roleKey`、用户身份、门店范围、权限和系统预算。

### 4.4 `BrainObservation`

建议增加：

- `outputSchemaVersion`。
- `dataClassification`。
- `retryable`。
- `businessObjectRefs`。
- `sourceVersion`。
- `redactionApplied`。
- `partialFailure`。

工具原始输出必须先通过 Schema、权限和敏感字段检查，再进入下一节点或回答生成。

### 4.5 `BrainCompletionAssessment`

不能只依赖模型判断是否完成。每个 `successCriteria` 必须使用稳定 ID，并绑定确定性 evaluator。

建议：

```typescript
interface BrainCriterionResult {
  criterionId: string;
  status: 'satisfied' | 'missing' | 'failed' | 'blocked';
  evaluator: 'schema' | 'data' | 'permission' | 'model';
  evidenceRefs: string[];
}
```

模型只能判断诊断质量和建议完整性，不能覆盖以下确定性失败：

- 缺少必需节点。
- 输出 Schema 错误。
- 无证据事实。
- 权限不足。
- 时间或门店范围错误。
- 写操作未确认。

### 4.6 `BrainResponseEnvelope`

当前状态只有 `completed | partial | failed`，无法区分业务未覆盖、澄清、权限、安全和无数据。

建议改为：

```typescript
type BrainResponseStatus =
  | 'completed'
  | 'partially_completed'
  | 'clarification_required'
  | 'capability_unavailable'
  | 'no_data'
  | 'permission_denied'
  | 'safety_blocked'
  | 'failed';
```

R1 先返回最终结构化 Envelope 和文本 SSE。`block_delta` 流式渲染移到后续阶段，避免第一阶段同时改动三端流式协议。

## 5. 推荐重排后的交付阶段

### R0：可行性与产品基线

交付：

- 真实 provider 结构化生成 spike。
- 12 个核心任务。
- Clean-room Golden Dataset。
- 隐私脱敏后的真实问句回放集。
- 延迟、Token、成本、Schema 合法率和意图准确率基线。

停止条件：

- 模型结构化输出稳定性无法满足业务门禁。
- 模型在真实问句上的目标理解没有超过现有确定性基线。
- 单请求成本或延迟无法进入产品预算。

### R1：单能力模型驱动问数闭环

交付：

- Semantic Intent Compiler。
- Intent Validator。
- 手写 Capability Catalog 和 12 个 Handler。
- Single-step Planner。
- Observation 和确定性 Completion Guard。
- 最小 Response Envelope。
- 离线回放、1% canary 和一键回滚。

出口门禁：

- 12 个核心任务真实可用率不低于 90%。
- 数值 Golden Case 正确率 100%。
- 假阳性 0。
- 权限和跨店错误 0。
- `clarification_required` 与 `capability_unavailable` 分类准确率不低于 95%。

### R2：Capability draft 扫描与漂移治理

交付：

- `@BrainCapability`。
- 只读 source adapters。
- draft candidate。
- sourceFingerprint。
- drift diff 和 CI 门禁。
- 人工 Handler 绑定与发布。

出口门禁：

- 显式标注接口的漂移检出率 100%。
- 未标注或证据冲突对象不进入 active Catalog。
- 自动生成可运行代码和自动发布均为 0。

### R3：复合任务 Supervisor

前置条件：R1 连续灰度稳定，Capability 输入输出 Schema 和 Observation 已稳定。

交付：

- Proposed Plan。
- Plan Validator。
- Validated Plan。
- Bounded Executor。
- 有限 Replan。
- 复合任务 Completion Guard。

首批只允许 3 个复合任务，不做全域 DAG：

- 临期库存处置建议。
- 明日下午空档补齐建议。
- 周经营风险诊断。

### R4：受控动作

交付：

- preview。
- confirm。
- execute。
- receipt。
- 幂等、乐观锁、错误分类和补偿入口。

首批只开放：

- 预约创建或改期。
- 客户跟进任务。
- 采购草稿。
- 营销触达草稿。

### R5：记忆、主动巡检与治理增强

该阶段不进入 R1-R4 的关键路径。

交付顺序：

1. 会话外偏好记忆。
2. 记忆冲突和撤销。
3. 巡检人工标注集。
4. 巡检建议和动作预览。
5. 真阳性、采纳、处置和归因闭环。

## 6. 评测与灰度门禁修改建议

### 6.1 不再以单一“意图准确率”代表产品可用

必须分别报告：

- 目标理解准确率。
- Capability 选择准确率。
- 参数准确率。
- 数值正确率。
- 输出粒度正确率。
- 任务完成率。
- 澄清正确率。
- 正确拒答率。
- 假阳性。
- 权限、安全和跨店错误。

### 6.2 高风险数据采用更严格门禁

- 财务、收银、会员资产和库存写操作的 Golden Case 数值正确率必须为 100%。
- 全局 98% 数值正确率不能覆盖高风险领域错误。
- 无证据事实、假完成、未确认写入、跨店和越权均执行一票否决。

### 6.3 Canary 必须有样本量和停止规则

每个阶段至少定义：

- 最小请求数。
- 最短观察时间。
- 核心任务覆盖数。
- 假阳性上限。
- 错误率上限。
- P95 延迟上限。
- 单请求成本上限。

出现以下任一情况立即切回 `rules`：

- 假阳性大于 0。
- 跨店、越权或未确认写入大于 0。
- 核心任务正确率低于阶段门禁。
- P95 延迟或成本连续超出预算。

### 6.4 真实问句回放必须增加隐私治理

Task 0 和 Task 7 应补充：

- 导出审批。
- 手机号、姓名、备注和敏感字段脱敏。
- 样本用途和保留期限。
- provider 数据保留策略。
- 禁止记录未脱敏 prompt 和 raw response。
- 报告只保留必要错误样本和匿名标识。

## 8. 对排期的修改建议

不再保留固定“16 周/4 人”和“28 周/单人”承诺。建议采用阶段承诺：

| 阶段 | 建议口径 |
| --- | --- |
| R0 | 1–2 周，结束后决定是否继续模型主线 |
| R1 | 根据 R0 结果估算，目标是完成 12 个核心任务闭环 |
| R2 | 根据 12 个 Handler 的真实开发和漂移成本估算 |
| R3 | R1 稳定后立项，不提前承诺 |
| R4 | 逐个动作按风险单独排期 |
| R5 | 不进入首版关键路径 |

管理层只获得当前阶段的承诺和下一阶段的进入条件，不获得未经 spike 校准的终局日期。

## 9. 最终修改结论

该方案不需要推翻，必须做的是彻底收口：

1. 生成无冲突的 v3 单一执行版。
2. 先做模型 spike 和 Clean-room 产品评测基线。
3. 首版只完成 12 个单能力核心任务。
4. 提前完成最小 Response Envelope 和 Completion Guard。
5. Scanner 只做 draft 与漂移，不生成可运行代码。
6. 业务语义中台彻底移出本计划。
7. Supervisor、长期记忆、主动巡检和全自动能力生成全部延后。
8. 所有灰度以假阳性、安全、数值和任务完成度为硬门禁。

修改后，Ami Brain 的开发顺序将从“先建设完整智能体平台”变为“先证明 12 个真实任务可用，再逐层开放规划、动作和主动经营”。这能直接吸收历史失败经验，同时保持 Ami Brain 架构、代码和产品边界完全独立。
