## 1. 当前结论

本轮 Agent 知识治理底座已经从“能发现缺口”推进到“核心缺口清零”：

- SchemaGraph 已同步：Prisma model 120 个，generated graph 120 个。
- P0 knowledge-map gate 通过：149/149。
- 阻断项：0。
- 中高风险工具审批缺口：0。
- ToolRegistry 注册缺口：0。
- BusinessQuery 执行分支缺口：0。
- BusinessObjectCatalog 映射缺口：0。
- 重要字段中文名缺口：0。
- SkillRegistry 暴露缺口：0。
- Eval 覆盖缺口：0。
- 前端页面候选提醒：0。
- legacy fallback 命中：0。

因此当前 Agent 的主要问题不再是“底层链路不通”或“对象字典缺失”，而是进入下一阶段：

1. **持续治理**：管理端继续新增页面、API、Prisma model 时，需要自动发现并输出人工确认清单。
2. **能力转化**：API capability 草案已有 120 条候选，需要按业务价值分批确认，不能自动全部开放给 Agent。
3. **运行质量**：知识图谱门禁已通过，但仍要用真实问答失败样本持续补 Skill、Tool、Eval。
4. **产品边界**：内部对象、证据对象、必查对象已经分类，但部分对象是否面向门店角色展示，仍要按权限和场景确认。

## 2. 改进目标

### 2.1 产品目标

让“洞悉美业·门店运营智能体”从能回答部分问题，升级为可持续覆盖管理端业务增长的智能体体系。

重点不是继续堆规则，而是让每个新增业务都能自动进入以下链路：

管理端新增业务 -> Schema/API/Page 被扫描发现 -> 产品确认业务语义 -> 研发补 Catalog/Skill/Tool/Eval -> Agent 可稳定回答。

### 2.2 工程目标

- P0 门禁继续保持 100%。
- 让 99 个模型全部进入明确状态：必查、证据或内部白名单。
- 保持 SkillRegistry、ToolRegistry、BusinessQuery 执行分支、Eval 覆盖缺口为 0。
- 降低前端页面候选误报，避免治理台长期噪音过高。
- 把“新 API / 新页面 / 新模型”自动转为候选草案和人工确认清单，而不是靠用户发现 Agent 答错后再补丁。

## 4. 改进方案

### 4.1 业务对象字典治理

#### 问题

当前 99 个 Prisma model 未映射到 BusinessObjectCatalog。这里面既有真正需要 Agent 查询的业务对象，也有系统内部审计、日志、配置、运行态对象。

如果全部直接加入 Catalog，会让 Agent 看到太多低价值对象，反而增加误判。

#### 方案

建立 BusinessObjectCatalog 三层处理策略：

| 分类 | 数量 | 说明 | 处理方式 |
|---|---:|---|---|
| Agent 必查对象 | 51 | 用户会自然询问的核心业务对象 | 进入 BusinessObjectCatalog |
| Agent 证据对象 | 41 | 用户不直接问，但回答时需要作为证据来源 | 进入 Catalog，但标记 evidenceOnly |
| 系统内部对象 | 7 | 审计、日志、运行态、配置、权限、认证等不应暴露给门店用户 | 标记 internalOnly 或进入白名单 |

完整 99 个模型明细见：

`docs/03-开发计划/Agent_BusinessObjectCatalog_99模型建议分类表-2026-07-02.md`

#### 已采纳的首批 P1 必查对象

优先处理直接影响问答准确性和业务语言表达的 19 个对象：

- `Card`
- `CashierShift`
- `CommissionRecord`
- `CommissionRule`
- `CommissionSettlement`
- `CustomerBalanceAccount`
- `CustomerBalanceTransaction`
- `DailySettlement`
- `MarketingPageLead`
- `MarketingRecommendationSnapshot`
- `OperatingCost`
- `ProjectBomItem`
- `Schedule`
- `ServiceTask`
- `StockBatch`
- `StockMovement`
- `SupplierSettlement`
- `TerminalFollowUpTask`
- `TransferOrder`

这些对象覆盖财务、客户资产、库存、排班、营销、提成和供应链，是店长、前台、财务、库存角色最容易直接询问的业务对象。

#### 已采纳的首批 P1 证据对象

优先处理会影响解释可信度和证据链完整性的 11 个对象：

- `AgentFeedback`
- `AgentToolCall`
- `AiAuditLog`
- `BeauticianProjectSkill`
- `ConsumptionRecord`
- `CustomerAppEvent`
- `CustomerBehaviorEvent`
- `CustomerPredictionSnapshot`
- `MarketingAttribution`
- `MarketingPageAttribution`
- `MarketingPageEvent`

这些对象不一定作为用户直接查询入口，但应作为 evidence source，支撑“为什么推荐、为什么预警、为什么判断客户需要召回”等解释。

#### 系统内部对象处理建议

以下 7 个对象不应进入普通门店 Agent 查询候选：

- `AgentDefinition`
- `AgentEvalCase`
- `AgentPersona`
- `IndustryDataSource`
- `RefreshToken`
- `Role`
- `UserRole`

处理方式：

- `RefreshToken` 必须 `internalOnly + sensitive`。
- `Role`、`UserRole` 只允许系统管理员治理视角查看。
- `AgentDefinition`、`AgentPersona`、`AgentEvalCase` 只用于 Agent 治理台和研发审计，不进入门店经营问答。

#### 交付物

- 更新 `business-object.catalog.ts`。
- 每个对象补齐：
  - 中文名。
  - 业务解释。
  - sourceModels。
  - queryableFields。
  - displayFields。
  - supportedActions。
  - 是否 `evidenceOnly` 或 `internalOnly`。
- 增加 BusinessObjectCatalog 元数据：
  - `evidenceOnly: true`
  - `internalOnly: true`
  - `sensitive: true`
  - `defaultVisible: false`
- 对明确不进入普通 Agent 查询的模型，写入治理白名单或 internalOnly 策略，并注明原因和负责人。

### 4.2 字段中文名和业务口径补齐

#### 问题

当前 66 个重要字段缺中文名。用户看到表格时，如果字段仍显示 `amount`、`status`、`rate`、`count`，会觉得 Agent 不懂业务。

#### 方案

按字段类型建立统一口径：

| 字段类型 | 示例 | 处理原则 |
|---|---|---|
| 金额 | amount、income、revenue、payment | 必须说明是应收、实收、退款、充值本金还是赠送金 |
| 比率 | rate、margin、discount | 必须说明分母和计算口径 |
| 状态 | status、auditStatus、reviewStatus | 必须给出中文枚举 |
| 计数 | count、total、times | 必须说明统计范围和时间口径 |
| 敏感字段 | phone、token、secret | 默认脱敏或禁止输出 |

#### 交付物

- 补齐核心对象 `displayFields`。
- 对财务、会员余额、卡项、营销转化、库存风险字段补充 `metricDefinition`。
- 更新表格渲染中文字段映射，减少 `beauticianId / status / growthCount` 这类技术字段直出。

### 4.4 Eval 覆盖补齐

#### 问题

当前 8 个 implemented capability 缺 Eval 覆盖：

- `automation_execution_summary`
- `business_anomaly_alert`
- `business_overview`
- `member_balance_analysis`
- `multi_store_comparison`
- `product_customer_distribution`
- `product_sales_trend`
- `project_material_margin`

这些能力没有回归样本保护，后续改 Planner、Router、Skill 时可能被破坏。

#### 方案

每个能力至少补 3 类 Eval：

1. 标准问法：用户明确提出业务问题。
2. 随机自然问法：用户口语化、缺少完整字段。
3. 边界问法：权限不足、数据为空、时间范围不明确。

#### 示例

| 能力 | Eval 示例 |
|---|---|
| business_overview | 今天我应该重点关注什么 |
| business_anomaly_alert | 最近一个月门店有哪些经营风险 |
| product_sales_trend | 最近 30 天哪些商品卖得最好 |
| project_material_margin | 哪些项目耗材成本偏高 |
| member_balance_analysis | 当前会员储值沉淀资金是多少 |

#### 交付物

- 更新 `agent-eval.cases.ts`。
- 更新 Skill 内置 eval cases。
- `agent:eval:knowledge-map:gate:p2` 继续保持 100%。

---

### 4.5 页面候选降噪

#### 问题

当前前端页面候选 59 个全部进入提醒，说明扫描规则偏保守。

不是每个页面都需要 Agent 能力，例如配置页、纯管理页、内部审计页，可能只需要进入白名单或 internalOnly。

#### 方案

增强页面扫描判断：

| 页面类型 | 处理 |
|---|---|
| 列表/详情/统计页 | 生成 Agent 能力候选 |
| 配置页 | 默认不生成 Agent 能力，只做知识字典候选 |
| 审计/日志页 | 标记 internalOnly |
| 高风险动作页 | 生成审批策略检查 |
| 已有对应 Capability 的页面 | 不再提醒 |

#### 交付物

- 优化 `agent-knowledge-scan.ts` 的 route candidate 判断。
- 引入页面扫描白名单或分类配置。
- 治理台展示“新增页面但 Agent 未覆盖”时，只展示真正高价值页面。

### 4.6 自动生成草案能力补齐

#### 问题

当前系统已有高频问题候选池和 `candidateCapabilityId`，也已有负反馈导入 Eval 草稿池能力，但知识治理报告还没有自动输出完整的 capability 草案和 eval case 草案。

这会导致第三优先级中的“自动生成 capability 草案 / eval case 草案”还没有完全闭环。

#### 方案

新增两个只读草案生成器，不直接写正式 Catalog：

1. `agent:knowledge:capability-drafts`
   - 输入：API 候选、页面候选、Skill 缺口、AgentRun 高频问题。
   - 输出：`docs/04-测试数据/agent-capability-drafts.md/json`。
   - 内容包含 capabilityId、persona、业务对象、建议工具、输出类型、风险等级、示例问法。

2. `agent:knowledge:eval-drafts`
   - 输入：Eval 覆盖缺口、负反馈样本、remaining-supported 失败项。
   - 输出：`docs/04-测试数据/agent-eval-drafts.md/json`。
   - 内容包含 caseId、question、expectedPersona、expectedCapability、expectedTool、expectedOutputKinds。

#### 原则

- 草案只生成，不自动写入 `capability-catalog.service.ts` 或 `agent-eval.cases.ts`。
- 需要产品或研发确认后再进入正式代码。
- 草案必须带证据来源，避免模型凭空补能力。

## 6. 建议优先级

### 第一批必须做

1. 补 29 个 SkillRegistry 暴露缺口。
2. 补 8 个 Eval 覆盖缺口。

原因：这两类直接影响 Agent 是否能稳定规划和持续回归。

### 第二批尽快做

1. BusinessObjectCatalog P1 对象确认。
2. 字段中文名和业务口径补齐。

原因：这两类直接影响回答是否像“懂美业”，而不是输出数据库字段。

### 第三批持续做

1. 页面候选降噪。
2. capability/eval 草案生成器。
3. 白名单治理。

原因：这三类提升团队协作效率，但不应替代人工确认业务语义。

## 9. 产品确认边界

本轮已采纳 99 模型建议分类并完成落地，但产品上仍需要继续确认以下展示和权限边界：

1. `User` 是否允许被门店 Agent 查询，还是只作为员工、操作人、审核人证据。
2. `SkinTest` 当前是否已有完整业务闭环；如果只是记录，不应承诺医疗或诊断建议。
3. `AmiMonthlyBill` 和 `AmiPerformanceRecord` 是给平台运营看，还是也给门店店长看。
4. 供应链相关对象是否已经开放给门店端，还是只在供货平台内部使用。
5. Agent 治理类对象是否只在系统管理员角色下可见。
6. `RefreshToken`、`Role`、`UserRole` 必须保持 internalOnly，不进入自然语言业务问答。

## 10. 交付建议

建议下一轮不要继续从单个用户问题打补丁，而是按以下顺序推进：

1. **治理台确认 API capability 草案**：从 120 条候选里按业务价值确认本周要 Agent 化的能力。
2. **把真实失败样本转 Eval**：凡是用户反馈“答非所问/查不到/格式错”，必须沉淀成 Eval case。
3. **按角色细化对象权限**：店长、前台、美容师、库存、财务看到的对象、证据和动作要分层。
4. **持续执行 daily/weekly 治理**：每日看阻断和 fallback，周报输出人工确认清单。

这样能把 Agent 改进从“问题驱动补丁”转成“知识治理驱动迭代”。
