# Ami Brain 歧义意图追问功能优化方案

> 日期：2026-07-21
> 范围：管理端 Ami Brain 对话页与 `server-v2` Brain 对话链路
> 目标：为 Ami Brain 全部业务领域建立统一的歧义识别、最小追问和上下文续答机制。当业务领域已经明确、但用户目标、指标、对象、维度或动作参数不明确时，优先追问并在同一会话继续执行，不再返回统一能力介绍。
> P0 状态：已完成代码、接口、页面和定向测试闭环；未执行真实数据写入、能力发布或数据库迁移。

## 1. 产品结论

追问功能不能只修复会员卡问法，而要成为所有业务领域的基础能力。只要系统已经识别出用户在问财务、预约、客户、会员资产、项目商品、库存采购、员工经营、营销增长、订单收银或业务动作，但无法确定完成任务所必需的目标或槽位，就应提出一个最小追问。

“会员卡负责情况”只是一个代表性样例：系统已经识别到“会员卡”业务对象，但无法确定用户要查询的是会员卡未履约负债、储值余额、开卡充值、剩余次数，还是到期风险。相同原则也适用于“库存怎么样”“客户情况”“员工最近如何”“营销效果怎么样”等全领域模糊表达。

目标交互：

1. 用户输入一个领域明确但目标宽泛的问题，例如“会员卡负责情况”或“库存怎么样”。
2. 如果该领域存在已发布、可聚合且当前角色有权限访问的总览能力，Ami Brain 先直接输出整体概述。
3. 概述下方提供 3 个更细维度的候选问题，引导用户继续下钻。
4. 如果该领域没有可信总览，或不同目标会产生互斥口径、敏感数据或业务动作，才输出结构化追问。
5. 用户点击候选问题或自由输入后，Ami Brain 在原会话中继承领域和时间范围，继续执行真实业务查询。

## 2. 第一性原理与边界

追问只用于“知道用户在问什么业务，但缺少完成查询所必需的信息”。它不能成为所有失败的统一包装。

| 场景                                             | 系统行为                                |
| ------------------------------------------------ | --------------------------------------- |
| 领域、目标和口径明确                             | 直接查询，不追问                        |
| 领域明确、问题宽泛，且存在可信总览能力           | 先输出整体概述，再提供 3 个细分候选问题 |
| 领域明确，但没有可信总览或多个候选口径互斥       | 结构化追问                              |
| 排行、对比、名单等任务明确，但缺少必要维度或对象 | 针对缺失槽位追问                        |
| 问题明确但能力尚未接入                           | 明确说明能力缺口，不伪装成歧义          |
| 越权、跨门店或高风险操作                         | 权限拒绝或确认流程，不通过追问绕过      |
| 完全无法识别业务领域                             | 返回简短范围说明和示例，不无限追问      |

“现在接入 Ami Brain 的是哪个版本”属于目标明确但当前业务问数入口未提供版本信息，不应按歧义追问处理。后续应由版本/治理信息能力直接回答，或明确说明该入口的业务边界。

## 3. 当前可复用能力与真实断点

现有链路已经具备：

- `BrainResponseBlock.kind = "clarification"` 和 `BrainChatResponse.clarification`。
- `completed` 运行状态下的 `clarification` block、`completion: partial`、`missingSlots`、`ambiguities` 和候选选项结构。
- `BrainConversationContextService.pendingClarification`，可在同一会话保存待补信息并在下一轮解析。
- 管理端 `BrainResponseRenderer` 已能显示追问问题和选项。
- 能力检索器已能在置信度不足或候选差距过小时返回 `clarify`。

当前断点：

1. `BrainQuestionIntentService` 的确定性规则分支在未命中完整指标时直接返回 `unknown + unsupportedAnswer`，没有把“已识别领域、目标不明确”转换为澄清请求。
2. `BrainChatService` 在没有 metric 时直接返回统一能力介绍，绕过了现有结构化追问链路。
3. 管理端追问选项当前使用静态标签展示，无法点击后继续发送。
4. 追问是否解决、被放弃或进入重复追问没有统一埋点和发布门禁。

## 4. 后端改造方案

### 4.1 将“领域识别”和“完整意图识别”拆开

在确定性意图识别中先产出业务线索，再判断是否具备可执行口径：

```ts
interface BrainIntentHint {
  domains: string[];
  objective?: string;
  metricCandidates: string[];
  capabilityCandidates: string[];
  missingSlots: string[];
  ambiguities: Array<{
    slot: string;
    reason: string;
    candidates: Array<{ id: string; label: string; value: string }>;
  }>;
}
```

识别结果分三类：

- `selected`：目标和指标唯一，直接执行。
- `clarify`：领域明确，但目标、指标、维度或对象存在多个合理候选。
- `unsupported`：目标明确，但当前没有对应能力。

不得继续用 `unknown` 同时承载“真正未知”“目标明确但未支持”“领域明确但口径不清”三种不同产品状态。

### 4.2 全业务领域统一候选生成

追问候选不得以会员卡专用规则或新的静态领域表为主线。候选必须来源于当前治理发布状态下的业务定义、指标、实体、维度和能力目录，确保新增能力发布后自动进入追问候选范围。

候选生成顺序：

1. 使用已发布 Ontology 和业务定义识别 `domains/entities/metrics/dimensions`。
2. 从当前激活版本的能力目录检索 top-K 能力。
3. 根据当前角色、门店和权限过滤不可访问能力，避免向用户展示点击后必然越权的选项。
4. 将技术能力按用户目标合并，去除同义能力、旧版本和内部实现差异。
5. 候选唯一且必要槽位齐全时直接执行。
6. 问题宽泛且存在可信总览能力时，执行总览并生成 3 个下钻问题。
7. 候选唯一但缺少对象、时间、维度或动作参数时，针对缺失槽位追问。
8. 不存在总览且有 2–4 个高相关候选时，生成选择式追问。
9. 领域明确但没有已发布能力时，明确说明能力缺口，不生成无法执行的假选项。

全领域首批追问矩阵：

| 业务领域     | 典型模糊问题   | 首要追问维度或候选目标                                          |
| ------------ | -------------- | --------------------------------------------------------------- |
| 门店经营     | 店里情况怎么样 | 经营总览、风险预警、今日待办；若经营总览能力可覆盖则直接执行    |
| 财务与利润   | 财务情况怎么样 | 实收与支付构成、毛利、退款异常、预付/卡项负债、经营风险         |
| 预约与前台   | 预约情况怎么样 | 今日预约总览、预约空档、到店状态、指定员工/日期、创建或改期动作 |
| 客户与会员   | 客户情况怎么样 | 新老客、流失风险、高价值低活跃、消费记录、客户跟进              |
| 会员卡与卡项 | 会员卡负责情况 | 未履约负债、储值余额与流水、开卡充值、剩余次数与到期风险        |
| 项目与商品   | 项目情况怎么样 | 销售排行、收入、毛利、服务表现、耗材消耗、指定项目/商品         |
| 库存与采购   | 库存怎么样     | 低库存、临期风险、库存金额、耗材消耗、采购建议或采购单草稿      |
| 员工与排班   | 员工最近怎么样 | 个人/团队业绩、排班忙闲、客户服务、退款率、指定员工             |
| 订单与收银   | 订单情况怎么样 | 项目订单、商品订单、支付流水、退款、核销、指定订单              |
| 营销与增长   | 营销效果怎么样 | 活动效果、客群分析、触达内容、自动化规则、营销执行预览          |
| 服务与跟进   | 服务情况怎么样 | 今日服务、服务记录、客户跟进、投诉反馈、指定客户或美容师        |
| 业务动作     | 帮我处理一下   | 动作类型、目标对象、时间、数量和成功标准；生成预览后仍需确认    |

矩阵只定义产品覆盖范围和追问维度，不作为第二套硬编码能力清单。实际选项必须从当前已发布能力目录动态生成。

### 4.3 总览优先与三问下钻

典型模糊问题应优先视为“总览请求”，而不是天然的歧义请求。只要系统可以用一个已治理的总览能力覆盖该问题，就先回答用户最关心的整体状态，再通过 3 个细分问题进行渐进式下钻。

统一输出结构：

```json
{
  "status": "completed",
  "answer": "当前库存整体可用，低库存 6 项，临期风险 3 项，预计风险金额 2,480 元。建议优先处理临期商品和低于安全库存的高频耗材。",
  "blocks": [
    {
      "kind": "kpi",
      "items": [
        { "label": "低库存", "value": "6 项" },
        { "label": "临期风险", "value": "3 项" },
        { "label": "风险金额", "value": "¥2,480" }
      ]
    },
    {
      "kind": "follow_up_questions",
      "questions": [
        { "id": "inventory_low_stock", "label": "哪些商品低于安全库存？", "value": "列出当前低于安全库存的商品" },
        { "id": "inventory_expiring", "label": "哪些库存即将到期？", "value": "列出临期库存及预计风险金额" },
        { "id": "inventory_procurement", "label": "现在应该优先采购什么？", "value": "根据库存风险给出采购优先级建议" }
      ]
    }
  ]
}
```

新增 `follow_up_questions` 展示块，与 `clarification` 严格区分：

- `follow_up_questions`：当前问题已经完成回答，3 个问题用于继续深入；点击前不阻塞本轮完成状态。
- `clarification`：当前问题缺少完成所必需的信息；用户选择前不能执行最终查询或动作。

总览能力准入条件：

1. 必须是当前发布版本中状态为 active/published 的能力。
2. 必须明确声明支持 overview、summary 或多指标聚合答案形态。
3. 所有输出指标都有业务定义、真实数据来源和当前门店隔离。
4. 当前用户具备总览内全部数据的读取权限；不能先展示越权指标再隐藏明细。
5. 总览不得将排行、对比、原因诊断或预测结果用单期总值替代。
6. 不满足任一条件时退回结构化追问，不拼接多个不一致的单指标结果伪造总览。

三问生成规则：

- 固定输出 3 个问题，不输出内部 capability key 或 metric key。
- 三个问题必须属于不同的下钻维度，避免“低库存有哪些 / 哪些库存低 / 库存不足清单”这类重复表达。
- 优先级默认为：风险或异常、结构或明细、建议或下一步；若领域不适用则按已发布能力动态替换。
- 问题必须是当前角色有权限、当前版本能执行、点击后能返回真实结果的完整自然语言问题。
- 结合本轮异常信号排序：存在严重风险时优先推荐风险明细；无异常时推荐趋势、结构或优化机会。
- 继承本轮门店、角色、时间范围和已确认对象，用户无需重复输入。
- 用户可以忽略候选问题继续自由输入，不强制点击。

全领域示例：

| 典型模糊问题         | 首屏整体概述                                     | 3 个细分候选问题示例                                                                         |
| -------------------- | ------------------------------------------------ | -------------------------------------------------------------------------------------------- |
| 店里情况怎么样       | 实收、订单、客户、预约、到店、员工忙闲和主要风险 | 今天最需要关注什么风险？ / 哪项经营指标变化最大？ / 接下来优先处理什么？                     |
| 财务情况怎么样       | 实收、支付构成、毛利、退款和预付负债摘要         | 各支付方式分别收了多少？ / 毛利和毛利率是多少？ / 退款和异常流水有哪些？                     |
| 预约情况怎么样       | 今日预约、到店、未到店、空档和员工负荷           | 今天还有哪些预约未到店？ / 哪些员工有可用空档？ / 哪些预约需要跟进？                         |
| 客户情况怎么样       | 新老客、活跃、流失风险、高价值低活跃和待跟进摘要 | 哪些客户流失风险最高？ / 哪些高价值客户最近不活跃？ / 今天应该优先跟进谁？                   |
| 会员卡情况怎么样     | 未履约负债、储值余额、开卡充值和到期风险摘要     | 当前会员卡未履约负债是多少？ / 最近储值充值和消耗情况如何？ / 哪些卡项剩余次数少或即将到期？ |
| 项目和商品情况怎么样 | 销售、收入、毛利、服务表现和耗材消耗摘要         | 哪些项目销售最好？ / 哪些项目毛利偏低？ / 哪些项目耗材消耗异常？                             |
| 库存怎么样           | 低库存、临期风险、库存金额和采购风险摘要         | 哪些商品低于安全库存？ / 哪些库存即将到期？ / 现在应该优先采购什么？                         |
| 员工最近怎么样       | 团队业绩、排班负荷、客户服务和异常指标摘要       | 谁的业绩变化最大？ / 谁还有可用服务空档？ / 哪些员工需要关注服务或退款异常？                 |
| 订单情况怎么样       | 项目/商品订单、支付、退款、核销和异常订单摘要    | 今天有哪些项目订单？ / 哪些订单发生退款或异常？ / 哪些卡项等待核销？                         |
| 营销效果怎么样       | 活动效果、触达、客群响应和转化风险摘要           | 哪个活动效果最好？ / 哪类客群响应最高？ / 下一步应该优化哪个触达策略？                       |
| 服务情况怎么样       | 今日服务、完成状态、客户反馈和待跟进摘要         | 今天完成了哪些服务？ / 哪些客户服务后需要跟进？ / 有哪些投诉或异常反馈？                     |

表中的问题是产品示例。运行时必须根据已发布能力、真实异常信号和权限动态生成；不存在对应能力时不得展示该候选问题。

  "status": "completed",
  "answer": "你想了解会员卡的哪方面？",
  "blocks": [
    {
      "kind": "clarification",
      "question": "你想了解会员卡的哪方面？",
      "options": [
        { "id": "card_liability", "label": "未履约负债", "value": "会员卡未履约负债是多少" },
        { "id": "member_balance_flow", "label": "储值余额与流水", "value": "查看会员储值余额与流水" },
        { "id": "member_card_orders", "label": "开卡与充值", "value": "查看会员卡开卡与充值情况" },
        { "id": "card_usage_risk", "label": "剩余次数与到期风险", "value": "查看会员卡剩余次数与到期风险" }
      ]
    }
  ],
  "clarification": {
    "missingSlots": ["objective", "metric"],

  },

“负责”与“负债”不得直接静默纠错后查询金额。它可以作为低置信度近似词触发会员卡领域追问，由用户确认真实目标，避免错查财务口径。其他领域的错别字、简称和口语表达使用同一原则：可以帮助定位领域，但不能在关键业务口径不确定时静默替用户做决定。

### 4.4 接入现有澄清链路

在 `BrainChatService` 的 `!metric` 统一兜底之前增加判断：

1. 如果存在明确领域和多个有效候选，调用统一的 clarification builder。
2. 保持运行记录 `status = "completed"`，通过 clarification block 与 `completion: partial` 表达等待用户补充，不新增运行状态枚举。
3. 将 `pendingClarification` 写入现有 `BrainConversation.contextSnapshot` JSON。
4. 下一轮输入沿用 `resolve_pending_or_new` 模式，只补齐待确认槽位，不重新丢失原始业务对象。
5. 追问解决后清除 `pendingClarification`，再进入能力检索、权限校验和执行。

不新增 Prisma 表，不需要数据库迁移。

### 4.5 防止无限追问

- 同一目标最多连续追问 2 轮。
- 第二轮仍无法补齐时，返回当前可执行的具体示例，不再重复相同问题。
- 用户输入新的完整问题时，主动废弃旧 `pendingClarification`。
- 用户回答“都看、全部”时，不能一次隐式执行多个不同权限能力；返回可执行范围并要求选择首个主题，或进入已治理的多步骤计划。
- 不得在追问阶段提前读取敏感业务数据；选定目标后再进行权限和门店隔离校验。

## 5. 管理端交互改造

### 5.1 追问选项改为可点击按钮

调整 `BrainResponseRenderer`：

- 新增 `onClarificationSelect(option)` 回调。
- 选项由静态 `span` 改为按钮。
- 点击后调用当前 `BrainWorkspace.handleSend`，在同一会话发送 option 的规范化 `value`。
- 发送中禁用所有追问按钮，防止重复请求。
- 已选择选项显示选中状态；历史会话中的旧选项保持只读。

第一期继续复用现有 `/brain/conversations/:id/messages`，不新增接口。用户手工输入“看负债”与点击“未履约负债”走同一后端上下文解析链路。

### 5.2 三个下钻问题组件

扩展前后端 `BrainResponseBlock`：

```ts
type BrainFollowUpQuestionBlock = {
  kind: 'follow_up_questions';
  questions: Array<{
    id: string;
    label: string;
    value: string;
  }>;
};
```

管理端将 3 个问题显示在已完成答案下方：

- 使用完整问句按钮或问题卡片，不使用只有“详情”“更多”的模糊标签。
- 点击后调用当前 `BrainWorkspace.handleSend`，沿用同一会话、门店、角色和上下文。
- 点击问题后不修改上一条回答的完成状态。
- 请求期间禁用三个问题，成功发送后保留历史展示但禁止重复触发。
- 小屏下纵向排列，大屏可使用一行三列或自动换行。
- 键盘可聚焦并支持 Enter/Space 发送。

### 5.3 消息文案规则

- 追问必须使用当前领域的业务语言，例如“你想看低库存、临期风险、库存金额，还是采购建议？”
- 禁止用“请换个问法”“请输入更明确的问题”把理解成本全部转给用户。
- 每次最多给 4 个主要选项，另允许用户自由输入。
- 选项标签使用业务语言，不暴露 metric key、capability key 或内部版本号。

## 6. 运行状态、审计与指标

新增或统一以下运行轨迹事件：

- `intent_clarification_requested`
- `intent_clarification_selected`
- `intent_clarification_resolved`
- `conversation_guidance_decision`
- `follow_up_questions_generated`
- `follow_up_question_selected`
- `follow_up_questions_insufficient`

事件只记录来源 run、option id、缺失槽位、候选能力 key、置信度、候选差距和决策码；不得写入未脱敏客户数据或把点击来源当作权限凭证。

产品监控指标：

- 已识别领域问题的统一兜底率。
- 典型宽泛问题的总览直接回答率。
- 三问展示率、点击率和点击后执行成功率。
- 三个问题的重复度与不可执行候选率。
- 追问触发率。
- 一轮追问解决率。
- 两轮内追问解决率。
- 追问后真实能力执行成功率。
- 重复追问率和放弃率。
- 用户手工输入与点击选项的解决率差异。

## 7. 测试与验收

### 7.1 后端定向测试

必须覆盖：

- 测试从当前已发布能力目录读取领域集合，逐领域验证宽泛问题能够正确分流到“总览 + 三问”或“澄清 + 选择”，不允许只维护会员卡专用用例。
- 对存在总览能力的领域，逐领域验证“宽泛问题 -> 完成整体概述 -> 返回 3 个可执行下钻问题”，不得错误返回 `clarify`。
- 对不存在总览能力或候选口径互斥的领域，验证仍返回 `clarify`，不得拼接伪总览。
- 三个下钻问题必须去重、权限可用、当前版本可执行，并继承原门店和时间范围。
- “会员卡负责情况”返回 `clarify`，不返回统一能力介绍。
- “会员卡情况”返回四类业务选项。
- “会员卡负债是多少”直接执行，不产生追问。
- “看会员卡余额流水”直接命中余额流水能力。
- 首轮“会员卡情况”后回复“看负债”，继承原问题并执行负债查询。
- “库存怎么样”追问低库存、临期风险、库存金额或采购建议；“哪些商品低于安全库存”直接执行。
- “预约情况”在存在多个目标时追问预约总览、空档、到店状态或预约动作；“今天有多少预约”直接执行。
- “客户情况”追问新老客、流失风险、高价值低活跃或消费记录；带明确客户和目标时直接执行。
- “财务情况”追问实收、支付构成、毛利、退款异常或预付负债；明确指标不得追问。
- “员工最近怎么样”追问个人/团队、业绩/排班/服务表现和目标员工；角色权限会过滤不可用选项。
- “项目情况”与“商品情况”不会混为同一指标，必要时先追问业务对象，再追问排行、收入、毛利或消耗口径。
- “订单情况”追问订单类型、支付、退款、核销或指定订单；明确订单号时直接进入详情能力。
- “营销效果怎么样”追问具体活动、时间范围、客群或效果口径；“帮我发一下”进入动作目标和确认链路。
- “服务情况”追问门店总览、指定美容师、指定客户或服务记录，不用预约总数替代回答。
- 追问后切换到其他领域的完整问题时，废弃旧澄清状态并执行新问题。
- 无权限用户选择财务负债后仍被权限门禁拦截。
- 同一歧义最多追问两轮，不产生循环。
- “当前接入哪个版本”不会被错误归类为会员卡或经营指标歧义。

### 7.2 管理端测试

- clarification block 显示问题和 1–4 个按钮。
- follow-up block 在已完成概述下方显示且严格为 3 个完整问题。
- 点击下钻问题会发送其规范化自然语言 `value`，不改变上一轮 completed 状态。
- 点击选项只发送一次消息，沿用当前会话和角色。
- 请求期间按钮禁用。
- 历史消息中的选项不会误触发重复查询。
- 键盘操作、焦点状态和小屏布局可用。
- 手工输入追问答案与点击选项结果一致。

### 7.3 评测集门禁

新增“歧义意图与追问解决”专项集，覆盖当前已发布能力目录中的全部业务领域，不采用固定六领域白名单：

- 明确问题不得被过度追问。
- 存在可信总览能力的宽泛问题必须先回答概述，并提供 3 个细分问题。
- 领域明确但目标不清的问题必须追问。
- 目标明确但能力缺失的问题必须明确说明缺口。
- 正常闲聊或系统版本问题不得误路由到经营指标。
- 追问解决后必须进入真实能力执行，不能再次返回统一兜底。
- 每个已发布业务领域至少包含：1 个模糊问题、1 个明确问题、1 个缺槽位问题、1 个能力缺失或权限受限对照用例。
- 发布检查从能力目录动态计算领域覆盖率；新增领域或能力时，缺少对应追问评测将阻止发布。

发布门禁：

- 明确意图误追问率 = 0。
- 已识别领域的统一兜底率 = 0。
- 总览能力可覆盖的问题误追问率 = 0。
- 三问数量正确率、去重率、权限映射正确率和可执行率 = 100%。
- 追问选项与权限能力映射正确率 = 100%。
- 两轮内追问解决率不低于 95%。
- 已发布业务领域的追问评测覆盖率 = 100%。
- 不产生跨门店读取、权限绕过或假动作确认。

## 10. 排除项

- 本方案不把 Ami Brain 改成无限制闲聊助手。
- 不通过追问绕过尚未建设的业务能力。
- 不在追问阶段执行写操作或预读取敏感数据。
- 不修改数据库表结构。
- 不处理历史 Agent V1–V5 路径。
