# Ami Brain 当前错题集第一性原理诊断与问题优先级

> 诊断时间：2026-07-22 02:09（Asia/Shanghai）
> 诊断对象：管理端 Ami Brain 当前标记为“需改进”的 15 条问题、对应 `brain_run` / `brain_run_step`、当前发布状态、能力快照和最新 650 题评测证据
> 执行边界：只读核对；未发送新问题，未修改代码、配置、数据库和发布状态

> 后续处置更新（2026-07-22 02:58）：诊断完成后，已为用户 1 / 门店 6 定向激活模型驱动 Release #396，包含 10 项核心只读能力；评测 Release #395 的 Run #103 为 31/31 通过。其他账号仍使用 rules 基线。本报告的故障结论描述切换前状态，全量发布、动作能力和 650 题问题仍未解决。

## 一、结论

用户对产品现状的判断成立：**按当前真实生效的运行链路，Ami Brain 对门店日常基础问数基本不可用。**

但根因不是“8 天内完全没有开发业务能力”，而是更严重的交付装配问题：

1. 已开发的真实业务能力曾被装入 41 项能力的 model/shadow 发布，但该发布因语义版本与能力快照不一致连续回滚。
2. 回滚后的生产基线不是可工作的旧版 Ami Brain，而是仅含一个 `P10 发布演练技能` 的 rules 基线空壳。
3. 当前请求因此不会进入能力检索和真实业务执行，而是先经过一组硬编码正则；14/15 条错题在业务能力执行前就被拒绝或兜底。
4. 唯一进入业务执行的题目被错误路由：用户问“最近一周有多少到店消费客户”，系统返回“最近 30 天预约与员工状态”，却仍把运行标记为 `completed`。
5. 最新 650 题报告自身只有 56.0% 真实可用率，而且没有完整复现当前生产发布选择和回滚链路，因此不能证明当前产品可用。

从第一性原理看，一个经营智能体只有在以下链路全部成立时才算可用：

```text
用户问题
  -> 选择正确且完整的生产发布
  -> 正确理解业务意图、对象、指标、时间和答案形态
  -> 命中有权限的真实能力
  -> 查询正确业务事实
  -> 校验答案与问题一致
  -> 返回可解释结果或有效追问
```

当前主故障发生在最前面的“发布选择”和“意图路由”，导致后面的 41 项能力、语义定义和真实数据接口大部分根本没有执行机会。

## 三、第一性原理根因

### 根因 1：生产发布已回滚到非功能性 rules 空壳

2026-07-22 01:51:17，`ami-brain-final-p1-20260721-v3-canary-50-r3`（Release #392）因：

```text
semantic_definition_v3_requires_capability_refresh
```

被回滚。当前门店 6、用户 1、角色 `store_manager` 实际解析结果为：

```json
{
  "mode": "rules",
  "releaseId": 3,
  "releaseKey": "ami-brain-rules-baseline-20260714",
  "itemCount": 1,
  "capabilityCandidateCount": 0
}
```

Release #3 的唯一资源不是经营能力，而是：

```text
p10_release_drill_1783789709 / P10 发布演练技能 v1
```

与此同时，包含 41 项真实能力的 Release #386、#387、#388 已全部归档。包括：

- `finance_payment_breakdown`
- `product_sales_ranking`
- `project_service_ranking`
- `staff_performance_ranking`
- `customer_facts`
- `inventory_operations_overview`
- `store_operations_overview`

因此当前产品不是“模型效果不好”，而是**生产发布根本没有把这些能力交给当前用户**。

### 根因 2：运行模式存在两个互相独立的控制源

数据库发布记录按请求决定 `rules` / `shadow` / `model`；但语义快照是否在服务启动时加载，又由环境变量 `BRAIN_COGNITION_MODE` 决定。

当前实际 `.env` 未配置 `BRAIN_COGNITION_MODE`，代码默认值为 `rules`。在 rules 模式启动时，`BrainOntologyRuntimeService.onModuleInit()` 直接返回，不加载生产语义快照。

这解释了回滚前最早 4 条错题的 shadow 轨迹：

```text
cognition_model -> failed -> production_ready_snapshot_unavailable
```

即数据库可以把流量切到 shadow/model，但进程仍按 rules 启动且没有模型认知所需快照。发布系统与运行时初始化没有形成原子契约。

### 根因 3：回滚策略只保证“能回滚”，没有保证“回滚后产品可用”

现有回滚会沿发布链寻找 rules release，却没有验证该 rules release 是否包含最低可用业务能力，也没有执行基础经营问答 smoke gate。

结果是技术上回滚成功，产品上从 41 项能力退回一个发布演练技能。回滚机制把“系统不报错”错误地当成“产品可服务”。

### 根因 4：当前 rules 认知是关键词开关，不是业务意图理解

当前生产路径的关键规则包括：

- `本月实收` 不命中；`本月实收多少` 或 `实收流水` 才可能命中。
- 只要出现“排行、排名、最多、最好、最高”等词，就先返回 `ranking_question_requires_grouped_shape`。
- 趋势、客户累计消费、活跃客户、商品销量等常见表达没有进入可执行意图。

这套规则位于能力检索之前，因此即使能力快照里已经存在完全对应的能力，仍然不会执行。

### 根因 5：答案完成门禁没有校验问题与结果的一致性

“最近一周有多少到店消费客户”至少包含四个硬约束：

1. 时间：最近一周；
2. 对象：客户；
3. 条件：到店且消费；
4. 形态：去重人数标量。

实际答案是最近 30 天预约状态、待到店名单和员工忙闲。时间、对象、条件、答案形态全部不一致，但 run 仍为 `completed`。

这说明当前 `completed` 只代表代码执行结束，不代表回答完成用户任务。假成功比明确拒答风险更高，必须作为 P0 阻断项。

### 根因 6：650 题门禁验证的不是当前生产交付状态

[最新 650 题报告](../../04-测试数据/Agent评测与知识治理-2026-06-30至07-03/ami-brain-eval-run-2026-07-21-release362-final-650/ami-brain-model-driven-eval-report-2026-07-15.md)显示：

- 650 题中可评测 643 题；
- 真实可用 360 题，真实可用率 56.0%；
- intent 通过率 78.3%，tool 72.1%，plan 65.8%，completion 63.2%，answer 63.1%；
- 315 题没有 grounding；
- 126 次 `intent_mismatch`；
- 平均耗时 12.680 秒，P95 27.292 秒；
- 当前 15 条人工错题在该 650 题中精确匹配 0 条。

更关键的是，评测冻结逻辑会把 `shadow` 发布强制按 `model` 主路径运行；当前生产用户却落在 Release #3 的 `rules` 路径。因此 650 题没有验证“用户最终会命中哪个生产发布”。

两个典型反例：

- 评测/能力示例中的“本月实收多少”可以进入 `finance_payment_breakdown`，当前用户输入“本月实收”直接兜底。
- 650 题中的“帮我看一下库存整体情况”为 `usable_exact`，当前用户输入“当前库存情况”直接兜底。

这不是普通边缘表达，而是基础同义改写没有被门禁保护。

## 四、问题优先级

## P0：立即阻断“基本问题不可回答”和“错误答案被当成完成”

### P0-1 建立唯一、可工作的生产基线

目标：任何流量切换或回滚后，当前角色都必须拿到一份包含生产语义快照和核心只读能力的完整发布。

验收：

- 当前用户的 runtime summary 可见明确 `releaseId`、`mode` 和能力数量；
- 不允许把发布演练技能作为经营智能体回滚基线；
- 发布/回滚事务同时校验语义版本、能力快照、权限和最小 smoke 集；
- 15 条当前错题全部进入正确能力、有效追问或真实能力缺失说明，不得通用兜底。

### P0-2 合并运行模式控制源

目标：数据库发布状态成为请求运行模式和快照准备状态的统一事实源，环境变量只决定平台允许范围，不能制造第二套实际模式。

验收：

- shadow/model 发布激活前必须确认生产语义快照已加载；
- 快照缺失时禁止激活，而不是上线后在每次请求中失败；
- 每个 run 固化记录 `releaseId`、declared mode、effective mode、snapshot fingerprint 和能力数量。

### P0-3 取消“规则先拒绝、能力后检索”的倒置链路

目标：规则只承担安全、明确快速路径和歧义判定；业务问题必须先查询当前发布能力目录，再决定执行、追问或能力缺失。

首批必须覆盖：

- 实收、收入、流水及省略问法；
- 商品销量/销售额；
- 项目、员工、客户排行；
- 经营概览、库存概览；
- 趋势、对比；
- 客户画像、累计消费、客户档案。

### P0-4 增加答案契约校验，消灭假成功

目标：在 run 标记完成前，逐项校验时间、对象、指标、过滤条件、粒度和答案形态。

验收：

- “一周客户人数”不能返回“30 天预约名单”；
- 任一关键维度不一致时，只能重新规划、追问或明确失败；
- 假成功率必须为 0，优先级高于总体回答率。

### P0-5 重建面向生产路径的发布门禁

目标：评测必须从“冻结候选能力能否运行”升级为“真实用户最终命中的生产发布能否运行”。

门禁至少包含：

- 发布选择与角色/门店/用户 cohort；
- 进程启动配置与语义快照 readiness；
- 当前 15 条错题及其同义改写；
- 基础业务 smoke 100% 通过；
- 假成功、跨店、越权、错误时间范围均为 0；
- 650 题真实可用率未达到发布标准时禁止使用“final”作为可发布证据。

## P1：形成可持续修复闭环

### P1-1 错题自动进入回归集

“需改进”反馈应自动生成待审计 eval case，补齐预期意图、能力、时间、粒度和答案形态；问题修复后才能关闭反馈。

### P1-2 补齐可观测性

运行轨迹必须显示发布选择、每层输入/输出、能力候选、最终能力、token、阶段耗时和失败原因。当前缺失的信息使“规则拒答”和“能力缺失”在页面上无法快速区分。

### P1-3 处理性能和连接池问题

650 题报告中出现 31 次数据库 `max clients reached`，当前错题又出现 53.734 秒拒答。正确性修复后必须治理并发连接、总超时、模型超时和异步 shadow 任务，基础拒答也不能消耗几十秒。

## P2：产品表达与治理体验

### P2-1 修正文案和状态语义

当前兜底文案宣称“已接入实收流水”等能力，但同一轮“本月实收”无法回答，属于误导。`completed` 应区分“技术完成”和“用户任务完成”。

### P2-2 治理中心显示真实生效关系

治理页面需要直接展示：当前生效发布、覆盖 cohort、语义快照、能力数量、回滚目标和最近门禁结果，避免只看到大量草稿/历史版本却无法判断用户实际走哪条链路。

## 五、建议的处理顺序

```text
先停用空壳基线作为产品入口
  -> 修复发布/快照双控制源
  -> 建立完整且可回滚的生产基线
  -> 让能力检索先于业务拒答
  -> 加入答案契约与假成功阻断
  -> 用当前错题 + 基础同义改写重跑生产路径门禁
  -> 再处理性能、可观测性和治理 UI
```

不要先逐条补正则。逐条补正则只能让当前 15 题短期变绿，下一批同义问法仍会失败，并继续掩盖发布装配错误。
