# Ami Brain P3安全与业务闭环修复 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** 在 P2 已达成 650 题真实可用率 30.0%、假阳性 0 的基础上，先补齐后端权限、门店范围、动作确认、指标口径和评测门禁，再继续扩大六角色经营能力覆盖。

**Architecture:** 下一阶段不继续堆关键词和模板技能，先建立“请求入口授权 -> 门店/用户范围 -> 技能权限 -> 只读口径 -> 动作确认 -> 评测门禁”的硬闭环。Ami Brain、AskData、AI Gateway 使用同一类后端权限与 store scope 规则；Brain trace/action/governance 不能再只靠前端菜单隐藏。评测从“意图/粒度匹配”升级为“安全边界 + 数据口径 + 事实 grounding + 业务闭环”并行门禁。

**Tech Stack:** NestJS、TypeScript、Prisma、PostgreSQL、Jest、React API facade、Ami Brain `BrainChatService`、`BrainActionConfirmationService`、`PermissionsGuard`、`ami-brain-eval.ts`。

## 1. 下一阶段验收标准

### P0 安全门禁

- 非授权用户访问 `/brain/governance/*`、`/brain/actions/*`、`/brain/runs/:runId/events` 返回 `403`。
- 用户 A 不能读取用户 B 的 run events、trace、conversation messages。
- 用户只能访问自己 `storeIds` 内的 `X-Store-Id`。
- 仅有 `core:brain:use` 的用户不能通过伪造 `roleHint=finance` 获取财务摘要。
- AskData 不能通过手改 `X-Store-Id` 查询其他门店。

### P1 业务闭环门禁

- action preview 必须写入 `brain_action_confirmation`。
- confirm/reject 必须校验 `actionId + runId + storeId + userId + pending status`。
- 当前没有真实执行器的动作，确认后返回 `confirmed_preview_only`，不能让用户误以为已创建预约。
- 预约数不统计取消预约。
- 低库存定义和 SQL 一致，不把 `safetyStock=0/currentStock=0` 计为缺货预警。
- 时点指标不支持期间过滤时明确拒答，不返回伪期间值。

### P2 评测门禁

- 650 题真实可用率不低于 `30.0%`。
- 假阳性数保持 `0`。
- 新增 adversarial eval：
  - 权限绕过：`0` 通过。
  - 跨门店读取：`0` 通过。
  - roleHint 伪造：`0` 通过。

- [x] **Step 3: 修改 PermissionsGuard**

Update `packages/server-v2/src/common/guards/permissions.guard.ts`:

```ts
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { PERMISSIONS_KEY } from '../decorators/index.js';

@Injectable()
export class PermissionsGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredPermissions = this.reflector.getAllAndOverride<string[]>(PERMISSIONS_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);

    if (!requiredPermissions || requiredPermissions.length === 0) return true;

    const { user } = context.switchToHttp().getRequest();
    if (!user) return false;

    const permissions: string[] = user.permissions ?? [];
    const deniedPermissions: string[] = user.deniedPermissions ?? [];

    if (deniedPermissions.includes('*')) return false;
    if (requiredPermissions.some((permission) => deniedPermissions.includes(permission))) return false;
    if (permissions.includes('*')) return true;

    return requiredPermissions.every((permission) => permissions.includes(permission));
  }
}
```

import { BadRequestException, ForbiddenException, Req } from '@nestjs/common';
import type { Request } from 'express';
import type { AskDataRequestContext } from './ask-data.types.js';

interface AuthenticatedAskDataRequest extends Request {
  user?: {
    id?: number;
    storeIds?: number[];
    permissions?: string[];
    deniedPermissions?: string[];
  };
}

private contextFromRequest(req: AuthenticatedAskDataRequest): AskDataRequestContext {
  const rawStoreId = req.headers['x-store-id'];
  const storeId = Number(Array.isArray(rawStoreId) ? rawStoreId[0] : rawStoreId);
  if (!Number.isInteger(storeId) || storeId <= 0) {
    throw new BadRequestException('缺少有效的 X-Store-Id');
  }
  const visibleStoreIds = req.user?.storeIds ?? [];
  if (!visibleStoreIds.includes(storeId)) {
    throw new ForbiddenException(`store_scope_denied:${storeId}`);
  }
  return {
    userId: Number(req.user?.id),
    storeId,
    visibleStoreIds,
    permissions: req.user?.permissions ?? [],
    deniedPermissions: req.user?.deniedPermissions ?? [],
  };
}
