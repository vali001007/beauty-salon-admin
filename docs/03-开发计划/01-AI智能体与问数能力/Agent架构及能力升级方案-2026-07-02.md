# Agent 架构及能力升级方案

日期：2026-07-02
适用范围：Ami Aura Lite、Ami_Agent、Ami_Core 管理端业务 API、server-v2 Agent Runtime、经营问数、终端智能问答、自动化任务和权限治理。

## 1. 背景与问题

当前 Agent 已具备一定基础，包括任务预解析、LLM 任务草稿、能力目录、工具注册、语义查询、回复契约校验和评测体系。但从近期问题看，现有架构仍存在明显交付风险：

1. **业务语义理解与工具路由脱节**
   - 用户问“本周有哪些报废产品”，系统答成“哪些产品有报废风险”。
   - 本质不是大模型完全不理解，而是工具层把“已发生报废记录”和“报废/临期风险”混在一个库存风险能力里。

2. **能力目录存在，但不是唯一决策入口**
   - 代码中已有 `CapabilityCatalogService`、`BusinessTaskCompilerService`、`AgentToolRegistryService` 等能力。
   - 但实际链路仍存在关键词技能路由、预解析指标映射、专用工具兜底等多套入口，导致同一个问题可能被不同层错误接管。

3. **工具粒度偏碎，持续补丁成本高**
   - 如果每个页面、每个问题都手工开发一个 Agent 工具，会导致工具越来越多、语义越来越乱。
   - 管理端持续迭代时，Agent 工具跟不上业务变化。

4. **权限和数据安全不能靠最终文案兜底**
   - 可以让大模型自由理解问题，但不能让它自由读取全量数据后再靠最终回复隐藏。
   - 正确做法是：模型可以自由规划，取数必须通过权限、字段、门店、风险等级网关。

5. **最终回复缺少强校验**
   - 当前已有输出契约校验，但还需要进一步校验“回答的问题是否和用户问题一致”。
   - 例如用户问“已发生记录”，回答“风险预测”应直接判为不合格。

## 2. 总体目标

升级目标不是继续堆工具，而是把 Agent 从“关键词路由 + 专用工具补丁”升级为：

> **大模型自由理解问题，能力目录约束计划，通用工具授权取证，最终契约校验回复。**

目标效果：

- 用户像问人一样提问，Agent 能理解真实业务意图。
- Agent 不直接访问数据库、不绕过权限、不自由调用接口。
- 管理端新增表、接口、页面后，Agent 能自动发现候选能力，减少手工补工具。
- 回答必须有证据包，且证据包已按权限过滤和字段脱敏。
- 能通过评测持续发现“语义错路由、权限越界、格式错误、数据缺口”等问题。

## 3. 最新目标架构

```text
用户问题
  ↓
会话上下文与身份解析
  ↓
LLM 自由理解与结构化任务草稿
  ↓
能力目录候选召回
  ↓
能力计划裁决
  ↓
权限 / 门店 / 字段 / 风险网关
  ↓
通用业务工具执行
  ↓
授权证据包生成
  ↓
LLM 组织业务回复
  ↓
回复契约与安全校验
  ↓
返回用户 / 要求澄清 / 拦截重试
```

## 4. 核心设计原则

### 4.1 大模型负责理解，不负责越权取数

允许大模型在规划阶段自由发挥：

- 理解用户真实意图。
- 区分“已发生记录”“风险预测”“经营诊断”“动作草稿”。
- 生成结构化计划。
- 识别问题中隐含的时间、对象、范围、输出格式。

禁止大模型：

- 自由拼 SQL。
- 自由访问数据库。
- 自由调用未授权 API。
- 看到当前用户权限外的原始数据。

### 4.2 能力目录是唯一决策入口

未来 Agent 调用业务能力必须先命中能力目录。能力目录定义：

- 这个能力能回答什么问题。
- 支持哪些业务对象。
- 来源表/API 是什么。
- 允许哪些意图。
- 支持哪些过滤条件。
- 返回哪些输出形态。
- 需要哪些权限。
- 风险等级是什么。
- 正例和反例是什么。

示例：

```json
{
  "capabilityId": "inventory.scrap.records.list",
  "name": "查询报废记录",
  "domain": "库存管理",
  "businessObject": "库存流水",
  "sourceModels": ["StockMovement", "Product", "Store", "User"],
  "allowedIntents": ["查询已发生记录"],
  "eventTypes": ["scrap_out"],
  "filters": ["storeId", "timeRange", "productId", "operatorId"],
  "outputKinds": ["table", "summary", "evidence_panel"],
  "permissions": ["core:inventory:view"],
  "riskLevel": "low",
  "examples": ["本周有哪些报废产品", "昨天报废了哪些耗材"],
  "negativeExamples": ["哪些产品快报废了", "哪些库存有临期风险"]
}
```

### 4.3 少量通用工具替代大量专用补丁

不建议为每个问题开发一个工具。建议沉淀 5 类通用工具：

| 通用工具 | 用途 | 示例 |
|---|---|---|
| `business.record.query` | 查询业务流水/已发生记录 | 报废记录、核销记录、收银记录、充值记录、订单明细 |
| `business.metric.query` | 查询指标 | 营收、毛利、库存金额、转化率、服务次数 |
| `business.trend.query` | 查询趋势 | 本周 vs 上周、本月趋势、环比变化 |
| `business.detail.query` | 查询对象详情 | 客户详情、订单详情、商品详情、活动详情 |
| `business.action.draft` | 生成待确认动作草稿 | 采购草稿、营销活动草稿、跟进任务草稿 |

每个工具不直接代表一个业务功能，而是按能力目录提供的 `object + event + filters + output` 执行。

例如：

```json
{
  "tool": "business.record.query",
  "object": "StockMovement",
  "eventType": "scrap_out",
  "timeRange": "this_week",
  "output": "table"
}
```

## 5. 自动化能力生成方案

### 5.1 自动扫描来源

能力目录不应完全手工维护，应从以下来源自动生成候选能力：

| 来源 | 生成内容 |
|---|---|
| Prisma Schema | 业务对象、字段、关联、枚举、时间字段 |
| Nest Controller | 可用 API、入参、出参、动作类型 |
| DTO / TypeScript Types | 字段含义、必填项、可筛选项 |
| 权限码 | 能力所需权限、角色边界 |
| 前端路由与菜单 | 能力所属模块、页面入口 |
| 业务字典 / 语义字典 | 中文同义词、业务事件、正反例 |
| 历史评测失败 | 需要补充的能力、反例和边界 |

### 5.2 自动生成产物

自动扫描后生成：

- `agent-capability-drafts.json`：候选能力目录。
- `agent-capability-drafts.md`：产品和研发可读版本。
- `agent-eval-drafts.json`：自动生成的评测题。
- `agent-knowledge-scan-report.json`：能力覆盖率和缺口。
- `agent-knowledge-governance-report.md`：治理报告。

### 5.3 发布策略

| 能力类型 | 是否自动发布 | 原因 |
|---|---:|---|
| 低风险查询 | 可自动发布 | 只读能力，执行前仍经过权限、门店、字段和输出契约校验 |
| 经营指标汇总 | 可自动发布 | 指标口径来自能力目录和语义指标注册，发布后由评测持续校验 |
| 客户明细 | 可自动发布 | 可自动发布，但必须前置字段脱敏、门店范围和客户数据权限控制 |
| 财务明细 | 可自动发布 | 可自动发布，但必须前置财务权限、字段级授权、证据包和审计日志 |
| 写操作草稿 | 可自动发布 | 只生成草稿，不直接改业务数据，进入确认或审批流 |
| 直接写入/删除/发券/下发 | 不自动发布，强审批 | 会真实改变业务结果、客户权益或外部触达，必须人工确认或审批 |

结论：**能力目录候选可以自动生成；除直接写入、删除、发券、下发等真实执行动作外，其他能力在权限、字段、证据包和评测通过后均可自动发布。**

## 6. 关键链路设计

### 6.1 问题理解层

输入：

```text
本周有哪些报废产品
```

输出结构化任务：

```json
{
  "domain": "inventory",
  "intent": "list_records",
  "businessObject": "stock_movement",
  "event": "scrap_out",
  "timeRange": "this_week",
  "outputIntent": "show_table",
  "riskLevel": "low"
}
```

重点：

- 不把“报废”直接映射为库存风险。
- 必须区分：
  - 已报废：事实记录。
  - 快报废/临期：风险预测。
  - 帮我报废：动作草稿。

### 6.2 能力匹配层

候选能力：

1. `inventory.scrap.records.list`：查询报废记录。
2. `inventory.expiring.list`：查询临期库存。
3. `inventory.product.stock.lookup`：查询库存状态。

裁决规则：

- 用户问“有哪些报废产品”且时间为“本周”，优先匹配已发生记录。
- `inventory.expiring.list` 是反例，因为它回答风险，不回答已发生。

### 6.3 权限与证据网关

执行前必须校验：

- 用户是否有库存查看权限。
- 用户是否只能看当前门店。
- 是否允许查看操作人。
- 是否允许查看成本金额。
- 是否需要手机号、客户姓名脱敏。

工具只返回授权后的证据包：

```json
{
  "capabilityId": "inventory.scrap.records.list",
  "sourceModels": ["StockMovement", "Product"],
  "filtersApplied": {
    "storeId": 1,
    "timeRange": "2026-06-29 至 2026-07-05",
    "movementType": "scrap_out"
  },
  "fields": ["报废时间", "产品", "批次", "数量", "单位", "操作人", "备注"],
  "items": []
}
```

### 6.4 回复生成层

大模型只能基于证据包组织回复：

- 有数据：输出表格和简短结论。
- 无数据：说明“本周没有报废记录”，不能改答风险预测。
- 权限不足：说明权限边界和可转交角色。
- 能力缺失：说明当前系统还没有该查询能力，并生成缺口。

### 6.5 最终校验层

最终回复必须通过四类校验：

| 校验 | 作用 |
|---|---|
| 问题一致性校验 | 用户问已发生记录，不能回答风险预测 |
| 输出契约校验 | 用户问“哪些”，必须有表格或列表 |
| 证据校验 | 回答中的数字、对象必须来自证据包 |
| 权限校验 | 不允许输出未授权字段或跨门店数据 |

不通过时：

- 低风险场景允许自动重试一次。
- 仍失败则返回受控失败卡，不生成误导性答案。

## 7. 数据与权限模型升级

### 7.1 能力定义表

建议新增或规范化能力目录存储：

| 字段 | 说明 |
|---|---|
| capabilityId | 能力唯一 ID |
| displayName | 中文名称 |
| domain | 业务域 |
| objectTypes | 业务对象 |
| sourceModels | 来源模型 |
| allowedIntents | 支持意图 |
| events | 支持事件 |
| filters | 支持过滤 |
| outputKinds | 支持输出 |
| requiredPermissions | 所需权限 |
| fieldPolicy | 字段授权与脱敏规则 |
| riskLevel | 风险等级 |
| examples | 正例 |
| negativeExamples | 反例 |
| executor | 通用执行器 |
| status | draft / active / disabled |
| version | 版本 |

### 7.2 字段级权限

建议增加字段策略：

| 字段类型 | 默认策略 |
|---|---|
| 手机号 | 中间脱敏 |
| 客户姓名 | 当前门店可见 |
| 成本、毛利、提成 | 财务/店长可见 |
| 操作人 | 当前门店管理角色可见 |
| 跨门店汇总 | 店长以上或授权角色 |
| 删除、发券、改价 | 强审批 |

### 7.3 证据包标准

每次工具返回必须包含：

- `capabilityId`
- `queryPlan`
- `sourceModels`
- `filtersApplied`
- `permissionScope`
- `fieldPolicyApplied`
- `items`
- `summary`
- `dataFreshness`
- `limitations`

## 8. 与当前代码的升级关系

当前已有基础：

| 当前模块 | 现状 | 升级方向 |
|---|---|---|
| `BusinessTaskCompilerService` | 已支持预解析 + LLM 草稿合并 | 改为 LLM 先生成结构化计划，再由能力目录裁决 |
| `CapabilityCatalogService` | 已有能力目录雏形 | 扩展为唯一能力入口，增加 event、fieldPolicy、negativeExamples |
| `AgentToolRegistryService` | 已注册大量专用工具 | 保留必要工具，同时新增通用执行器，逐步减少专用补丁 |
| `SemanticQueryExecutorService` | 支持部分语义查询 | 升级为通用 record/metric/trend/detail 查询执行层 |
| `AnswerContractValidatorService` | 已校验 outputKinds | 增加问题一致性、证据一致性和权限字段校验 |
| Agent eval | 已有评测体系 | 增加能力目录回归、错路由回归、权限回归 |

## 9. 分阶段开发计划

### 阶段 1：统一决策入口

目标：避免多个路由层抢答。

任务：

1. 梳理现有 Agent 决策链路。
2. 将 `CapabilityCatalogService` 提升为统一裁决入口。
3. 保留预解析，但只作为 slot 提取，不直接决定工具。
4. 降级旧关键词技能路由为候选召回，不再最终拍板。
5. 增加能力裁决 trace，记录为什么选这个能力、为什么排除其他能力。

验收：

- “本周有哪些报废产品”不会命中库存风险能力。
- “哪些产品快过期了”仍命中临期库存能力。
- 每次回答可看到 `capabilityId` 和排除理由。

### 阶段 2：能力目录自动生成

目标：管理端迭代后，Agent 能自动发现候选能力。

任务：

1. 从 Prisma Schema 扫描业务对象、字段和关联。
2. 从 Controller 扫描 API、权限和动作类型。
3. 从前端路由和菜单补充业务入口。
4. 从语义字典生成正例、同义词和反例。
5. 输出候选能力草稿和治理报告。

验收：

- 新增业务对象后，可在能力草稿中看到候选能力。
- 每个能力至少包含：来源模型、权限、输出形态、正例、反例。
- 高风险能力默认不自动启用。

### 阶段 3：通用业务执行器

目标：减少手工工具数量。

任务：

1. 实现 `business.record.query`。
2. 实现 `business.metric.query`。
3. 实现 `business.trend.query`。
4. 实现 `business.detail.query`。
5. 实现 `business.action.draft`。
6. 将低风险查询能力迁移到通用执行器。

验收：

- 报废记录、订单记录、核销记录、充值记录可以共用 record 查询。
- 营收、毛利、库存金额、服务次数可以共用 metric 查询。
- 不再为每个问法新增专用工具。

### 阶段 4：权限与证据网关

目标：模型自由规划，但取数严格受控。

任务：

1. 增加统一 `AgentPolicyGateway`。
2. 校验角色、门店、字段、动作、风险等级。
3. 输出字段脱敏后的证据包。
4. 所有工具结果必须带 `filtersApplied` 和 `sourceModels`。
5. 写操作、财务、客户批量明细进入审批或确认流。

验收：

- 非授权角色不能查看财务明细。
- 门店角色不能跨门店取明细。
- 手机号、成本、提成等敏感字段按策略脱敏或隐藏。

### 阶段 5：回复契约升级

目标：防止答非所问。

任务：

1. 在现有 `AnswerContractValidatorService` 基础上增加问题一致性校验。
2. 增加证据一致性校验。
3. 增加 forbidden intent 检查。
4. 增加自动重试策略。
5. 将失败原因写入评测和治理报告。

验收：

- 用户问“已报废”，不能回答“报废风险”。
- 用户问“有哪些”，没有表格/列表时判失败。
- 回答中的数字必须能追溯到证据包。

### 阶段 6：评测驱动治理

目标：让 Agent 迭代可量化。

任务：

1. 建立能力目录覆盖率评测。
2. 建立错路由评测。
3. 建立权限越界评测。
4. 建立输出契约评测。
5. 将线上失败沉淀为评测用例。

验收指标：

| 指标 | 目标 |
|---|---:|
| 核心问答能力命中率 | ≥ 95% |
| 错路由率 | ≤ 2% |
| 权限越界 | 0 |
| 输出契约通过率 | ≥ 98% |
| 高风险动作无确认执行 | 0 |

## 10. 全量场景验收用例

本章节不再只列少量典型样例，而是将既有 docs/04-测试数据/agent-eval-questions.md 的 650 条测试问题全部纳入本方案，作为能力目录驱动规划、通用工具执行、权限拦截和回复契约校验的基础验收集。

### 10.1 使用规则

1. 每条问题必须至少完成能力目录匹配判断：命中能力、无权限、系统暂不支持、需要澄清四类结果之一。
2. 所有“哪些、列出、明细、清单”类问题必须返回表格或列表输出契约。
3. 所有经营指标类问题必须返回指标口径、时间范围和证据来源。
4. 所有客户、员工、财务、成本、提成相关问题必须经过字段级权限和脱敏策略。
5. 所有动作类问题必须区分草稿、确认、审批和真实执行；直接写入、删除、发券、下发不得自动发布执行。
6. 系统暂不支持的问题不能乱答，必须进入能力缺口和产品待建清单。

## 11. 产品交付影响

升级后，产品层面会有以下变化：

1. **Agent 不再像“关键词问答机器人”**
   - 能先理解业务语义，再选择系统能力。

2. **用户问法不需要被穷举**
   - 通过能力目录和语义字典理解问法，而不是不断加关键词补丁。

3. **管理端迭代不会导致 Agent 大量掉队**
   - 新接口、新表、新页面进入候选能力池，再通过治理发布。

4. **权限边界更清晰**
   - Agent 能知道系统有什么能力，但只能在当前用户权限内执行。

5. **错误可被归因**
   - 是语义理解错、能力缺失、权限不足、工具失败、数据缺口还是回复格式错，都能明确分类。

## 12. 不建议采用的方案

### 12.1 完全自由 Agent

不建议让大模型直接读库、拼接口、自由调用所有能力。

风险：

- 权限不可控。
- 敏感数据进入模型上下文。
- 回答可能受未授权数据影响。
- 难以审计和复现。

### 12.2 继续手工补丁式工具

不建议每遇到一个问题就新增一个专用工具。

风险：

- 工具数量失控。
- 语义边界混乱。
- 维护成本随管理端迭代快速上升。
- 评测通过率靠堆 case，不具备泛化能力。

### 12.3 只做最终回复拦截

不建议先让 Agent 拿全量数据，最后再隐藏。

风险：

- 已经进入上下文的数据无法真正“删除影响”。
- 模型可能基于不该看的数据做判断。
- 最终拦截只能防展示，不能防推理污染。

## 13. 推荐实施优先级

P0：

1. 能力目录成为唯一裁决入口。
2. 区分事实记录、风险预测、动作草稿三类意图。
3. 增加报废记录、核销记录、订单记录等 record 查询能力。
4. 回复契约增加问题一致性校验。

P1：

1. 能力目录自动扫描和候选生成。
2. 通用 `business.record.query`、`business.metric.query` 执行器。
3. 字段级权限和证据包标准。
4. 线上失败自动沉淀为评测用例。

P2：

1. 高风险动作审批流。
2. 自动化任务能力目录化。
3. 多 Agent 角色协同。
4. 管理端能力变更自动治理报告。

## 14. 结论

最新 Agent 架构应明确为：

> **模型负责理解和规划，能力目录负责边界，通用工具负责授权取证，契约校验负责拦截错误，评测体系负责持续进化。**

这条路线能同时解决三个核心矛盾：

1. **让 Agent 像人一样理解业务问题。**
2. **避免模型越权读取或错误回答。**
3. **避免管理端每迭代一次就手工补一堆 Agent 工具。**

下一步建议直接进入 P0 开发：先把“事实记录 / 风险预测 / 动作草稿”的意图区分和能力目录裁决链路改掉，再补通用 record 查询执行器。这样可以优先解决当前最影响信任的问题。
