# Agent 管理模块 V1/V2 能力对比与信息架构建议

> 日期：2026-07-04
> 适用范围：Ami_Core 管理端、Ami Aura Lite 终端、`server-v2` Agent V1/V2、Agent 能力中心、Agent 审计、AI 审计、数字员工账单。

---

## 1. 核心结论

当前不建议把 V1 和 V2 做成两个并列的“用户入口”，否则店长和运营会困惑：到底该问哪个 Agent。

建议新增一级菜单 **Agent 管理**，并采用“双层结构”：

1. **业务使用层**：用户只看到“智能体工作台”，默认使用推荐引擎。
2. **治理管理层**：管理员、产品、研发可以对比 V1/V2、看审计、看能力、跑评测、做灰度。

产品上要把 V1/V2 定义为 **Agent 引擎版本**，不是两个不同产品。

---

## 2. 当前 V1/V2 状态判断

| 维度 | Agent V1 | Agent V2 |
|---|---|---|
| 入口 | `/api/agent/*` | `/api/agent-v2/*` |
| 当前定位 | 旧主线、兼容生产、覆盖已有问答和治理功能 | 新一代能力 Manifest 驱动引擎 |
| 能力来源 | Persona、CapabilityCatalog、SkillRegistry、BusinessQuery、旧 Planner | V2 Capability Manifest、能力中心草案、V2 Tool Registry |
| 编排方式 | `AgentOrchestratorService` 为主，历史逻辑较多 | `AgentV2OrchestratorService` 独立编排 |
| 工具体系 | 旧 Tool Registry + BusinessQuery + 语义查询 | V2 原生工具：record、metric、trend、detail、draft、navigation |
| 输出契约 | 已有 structured blocks，但部分能力仍依赖旧格式和补丁 | 强制 evidence、字段策略、AnswerContract 校验 |
| 权限与风险 | 旧权限策略和部分角色规则 | V2 Policy Gateway + Manifest permissionCodes / fieldPolicies |
| 可治理性 | 能看运行审计、记忆、质量报告，但能力来源较分散 | 能力中心可导入、审核、发布、dry-run、eval gate |
| 回归评测 | V1 knowledge-map / remaining-supported 等评测 | V2 eval gate、capability drafts、manifest 版本治理 |
| 适合场景 | 稳定兜底、保留已有终端体验、旧能力兼容 | 新能力发布、能力对比、严格验收、后续主推 |

### 产品判断

- **V1 是兼容层**：短期保留，避免已有终端和管理端问答突然断。
- **V2 是目标层**：后续新增能力、评测、治理、发布都应优先进入 V2。
- **管理模块必须支持 V1/V2 对比**：否则无法回答“V2 是否真的比 V1 好”“哪些能力可以切换”。

## 3. Agent 管理模块推荐菜单

建议新增一级菜单：

```text
Agent 管理
├─ 智能体工作台
├─ V1/V2 对比中心
├─ Agent 运行审计
├─ Agent 能力中心
├─ Agent 评测中心
├─ 知识图谱治理
├─ 自动化与审批
├─ AI 调用审计
└─ 数字员工账单
```

### 3.1 菜单解释

| 菜单 | 主要作用 | 默认用户 |
|---|---|---|
| 智能体工作台 | 统一问答入口，不让普通用户选择 V1/V2 | 店长、运营、前台 |
| V1/V2 对比中心 | 同一问题同时跑 V1/V2，比较路由、答案、证据、耗时、成功率 | 产品、研发、测试 |
| Agent 运行审计 | 查看每次问答由哪个引擎处理、命中能力、工具调用、失败原因 | 运营、产品、研发 |
| Agent 能力中心 | 管理 V2 capability drafts、manifest、发布、dry-run、权限、字段策略 | 产品、研发 |
| Agent 评测中心 | 650 题、P0/P1/P2、V1/V2 通过率、失败分类、趋势 | 测试、产品、研发 |
| 知识图谱治理 | SchemaGraph、BusinessObjectCatalog、字段中文名、对象映射 | 数据治理、研发 |
| 自动化与审批 | Agent 自动化任务、草稿、审批、执行结果 | 店长、运营 |
| AI 调用审计 | 模型调用、Token、费用、延迟、安全审计 | 管理员、研发 |
| 数字员工账单 | Ami 数字员工费用、分成、门店账单 | 管理员、财务 |

## 4. 推荐首页：Agent 管理总览

进入 **Agent 管理** 后，不要直接落到某个列表页，建议做一个总览页。

### 4.1 顶部 KPI

| 指标 | 说明 |
|---|---|
| 今日问答量 | V1/V2 分开统计 |
| V2 占比 | 用于观察灰度迁移进度 |
| 成功率 | 完成回答且契约通过 |
| 用户有用率 | 来自 👍 / 👎 和采纳 |
| 平均耗时 | 对比 V1/V2 响应速度 |
| 失败数 | runtime_error、missing_capability、contract_failed 等 |
| 费用 | Token / 模型调用费用 |

### 4.2 中部模块

1. **V1/V2 能力覆盖对比**
   - 按领域：经营、客户、预约、库存、财务、营销、员工、终端。
   - 展示：V1 可答、V2 可答、V2 已发布、V2 待审核、V2 缺口。

2. **最近失败问题**
   - 展示用户原问题、引擎版本、失败原因、是否已有能力草案。
   - 一键生成 Eval case 或能力草案。

3. **待治理能力**
   - V2 drafts：待审核、待补齐、可发布、发布失败。
   - 高风险能力必须显示审批策略。

4. **评测趋势**
   - V1 pass rate、V2 pass rate、P0 pass rate。
   - V2 未映射、契约失败、权限复核、字段策略缺口。

## 5. V1/V2 对比中心设计

这是新增 Agent 管理模块里最关键的页面。目标不是给普通用户用，而是给产品验收和研发治理用。

### 5.1 页面能力

| 功能 | 说明 |
|---|---|
| 单问题对比 | 输入一个问题，同时请求 V1 和 V2 |
| 批量题库对比 | 选择 eval 题库，跑 V1/V2 shadow 对比 |
| 同屏答案展示 | 左 V1、右 V2，展示最终答案和 blocks |
| 路由/能力对比 | 展示 V1 persona / capability 与 V2 manifest capability |
| 工具调用对比 | 展示 toolNames、queryKey、耗时、错误 |
| 证据对比 | 对比 evidence sources、dateRange、metricDefinition |
| 契约对比 | V1 outputKinds 与 V2 AnswerContract 校验结果 |
| 用户反馈对比 | 哪个答案被采纳、哪个被点“无用” |

### 5.2 对比维度

| 对比项 | V1 | V2 | 产品意义 |
|---|---|---|---|
| 是否回答 | completed / failed | completed / failed | 基础可用性 |
| 是否答对 | Eval 判定 | Eval 判定 | 真实能力 |
| 命中能力 | Skill/Capability | Manifest capability | 可治理性 |
| 数据来源 | BusinessQuery / Tool | V2 tool + evidence | 可信度 |
| 输出形态 | blocks / text | contract blocks | 体验一致性 |
| 权限处理 | 旧策略 | Policy Gateway | 安全 |
| 耗时 | ms | ms | 体验 |
| 费用 | token / model | token / model | 成本 |
| 是否可迁移 | 人工判断 | 人工判断 | 灰度决策 |

### 5.3 对比结论标签

每条问题跑完后，系统应输出一个结论标签：

| 标签 | 含义 |
|---|---|
| `v2_better` | V2 答案更准、证据更完整，可优先切换 |
| `v1_better` | V1 当前表现更好，V2 需要补能力 |
| `same_quality` | 两者质量接近，可看耗时和成本 |
| `v2_missing_capability` | V2 没有对应能力 |
| `v2_contract_failed` | V2 命中了能力但输出契约失败 |
| `permission_gap` | 权限或字段策略不一致 |
| `both_failed` | 两者都失败，需要补业务能力或数据 |

## 6. Agent 能力中心如何承接 V1/V2

能力中心不建议混管 V1 和 V2。建议定位为：

> **V2 能力发布与治理中心，V1 只作为对照基线。**

### 6.1 能力中心 Tab

```text
Agent 能力中心
├─ V2 能力草案
├─ 已发布能力
├─ Manifest 版本
├─ Tool QueryKey
├─ 字段策略
├─ 权限与风险
└─ V1 对照能力
```

### 6.2 V1 对照能力不要可编辑

V1 对照能力只展示：

- V1 capabilityId / skill / persona。
- V1 能力覆盖的问题样本。
- 对应的 V2 capabilityId。
- 迁移状态：未映射、已映射、已灰度、已替代。

不要在 V1 对照页继续新增复杂配置，否则会继续扩大双轨维护成本。

## 7. Agent 运行审计如何设计

运行审计必须把“引擎版本”作为一等字段。

### 7.1 列表字段

| 字段 | 说明 |
|---|---|
| 时间 | run 创建时间 |
| 用户问题 | 原始输入 |
| 入口 | admin / kiosk / api / eval / shadow |
| 引擎 | agent_v1 / agent_v2 |
| 状态 | completed / failed / blocked |
| Persona | 店长经营、营销增长等 |
| Capability | V1 capability 或 V2 manifest capability |
| Tool | 实际调用工具 |
| 耗时 | 总耗时 |
| 反馈 | 有用/无用/采纳 |

### 7.2 详情页必须分层

```text
运行详情
├─ 用户问题与最终回答
├─ 引擎版本与入口
├─ 路由 / 能力决策
├─ 工具调用链路
├─ 数据证据
├─ 输出契约 / 渲染 Blocks
├─ 权限与字段脱敏
├─ 错误与 fallback
└─ 生成 Eval / 生成能力草案
```

### 7.3 关键原则

V2 不应被 V1 隐式兜底污染。如果真的允许回退，必须是显式策略：

- `fallbackPolicy = none`：默认。
- `fallbackPolicy = explicit_legacy`：用户或灰度策略明确允许。
- `fallbackPolicy = shadow_only`：只做对比，不影响用户答案。

## 8. 智能体工作台如何设计

普通业务用户不应该看到 V1/V2 技术选择。

### 8.1 默认体验

- 页面名称：**洞悉美业智能体** 或 **智能体工作台**。
- 用户只输入问题。
- 系统按灰度策略选择 V1 或 V2。
- 回答卡片可以显示“由 店长经营 Agent 处理”，但不默认显示“V1/V2”。

### 8.2 调试模式

只有管理员或 URL debug 参数显示：

- 引擎选择：自动 / V1 / V2 / V1+V2 对比。
- 显示 capabilityId、toolNames、contract status。
- 一键跳转运行审计详情。

### 8.3 灰度策略

| 策略 | 说明 |
|---|---|
| V1 默认 | 生产兜底，适合未完成迁移时 |
| V2 优先 | 命中已发布 V2 能力则用 V2，否则明确提示未覆盖 |
| Shadow 对比 | 用户拿 V1 答案，后台同时跑 V2，用于对比 |
| V2 强制 | 验收和灰度门店使用 |

## 9. Agent 评测中心设计

评测中心要从“测试报告页面”升级为“产品能力回归中心”。

### 9.1 核心视图

| 视图 | 内容 |
|---|---|
| 总览 | V1/V2 pass rate、P0 pass rate、失败趋势 |
| 题库 | 650 题、P0/P1/P2、系统不支持、Agent 能力缺口 |
| V1/V2 对比 | 同题双跑结果 |
| 失败分类 | route_error、skill_missing、tool_missing、contract_failed、permission_error |
| 能力缺口 | 可一键生成 V2 capability draft |
| 回归历史 | 每日/每周趋势 |

### 9.2 发布门禁

V2 能力发布前必须满足：

- P0 对应题通过。
- 输出契约通过。
- 权限策略通过。
- 字段策略通过。
- 高风险能力有审批策略。
- 至少一条真实 dry-run 或 post-publish smoke test 通过。

## 10. 推荐迁移路径

### 阶段 1：管理入口归并

- 新增一级菜单 **Agent 管理**。
- 把 AI 智能体、Agent 审计、Agent 能力中心、AI 审计、数字员工账单统一归入。
- 保留原入口跳转，避免用户找不到。

### 阶段 2：V1/V2 可观测

- 运行审计列表增加引擎字段。
- 工作台 debug 模式支持选择 V1/V2。
- 增加 V1/V2 对比中心。

### 阶段 3：V2 能力中心成为主治理入口

- V2 drafts -> review -> validate -> dry-run -> eval gate -> publish。
- V1 只读对照，不再扩展新能力。

### 阶段 4：灰度切换

- 按 capability、persona、store 三个维度灰度。
- 对已通过 V2 的能力设置 `v2_preferred`。
- 对风险能力保持 V1 或人工确认。

### 阶段 5：V1 退役评估

- 连续 2-4 周 V2 在某领域 pass rate、用户有用率、耗时、失败率都优于 V1。
- 该领域 V1 标记为 legacy。
- 仅保留回滚开关和历史审计。

---

## 11. 最终推荐方案

Agent 管理模块不要只是“把几个页面放一起”，而要承担三个产品目标：

1. **统一入口**：业务用户只用一个智能体工作台。
2. **双引擎对比**：产品和研发能清楚看到 V1/V2 哪个更好。
3. **能力治理闭环**：失败问题能进入能力草案，能力发布前能评测，发布后能审计。

推荐落地结构：

```text
Agent 管理
├─ 总览
├─ 智能体工作台
├─ V1/V2 对比中心
├─ Agent 运行审计
├─ Agent 能力中心
├─ Agent 评测中心
├─ 知识图谱治理
├─ 自动化与审批
├─ AI 调用审计
└─ 数字员工账单
```

其中：

- **普通用户只常用：智能体工作台、自动化与审批、数字员工账单。**
- **产品/研发/测试常用：V1/V2 对比中心、运行审计、能力中心、评测中心、知识图谱治理。**
- **V2 是未来主线，V1 是兼容和对照基线。**
