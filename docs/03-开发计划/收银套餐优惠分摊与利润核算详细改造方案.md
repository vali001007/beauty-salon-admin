# 收银套餐优惠分摊与利润核算详细改造方案

更新时间：2026-06-21

## 1. 背景与目标

美容门店收银经常出现“项目 + 商品 + 卡项”的组合销售，例如：

- 补水项目 + 居家面膜，原价 800，套餐价 680。
- 办 10 次卡赠护理 1 次或赠商品。
- 项目套餐 8 折，同时叠加会员券、手工抹零。
- 商品第二件半价，或项目带动临期商品出清。

前台希望录入简单：选择项目、商品、卡项后，直接输入套餐价、折扣或优惠金额即可。

老板和财务希望结果准确：项目毛利、商品毛利、卡项履约、员工提成、库存出库、退款都能解释清楚。

因此本方案的核心目标是：

1. 前台收银允许整单录入套餐价、整单优惠或折扣。
2. 后台必须把优惠分摊到订单明细行，形成可追溯的项目、商品、卡项净收入。
3. 项目毛利、商品毛利、卡项履约、提成和退款统一使用分摊后的明细净额。
4. 对赠品、充值赠送、办卡赠项目等美业高频场景建立明确口径，避免把优惠误算成收入或利润。

## 2. 当前系统基础

当前系统已经具备改造基础：

- `ProductOrder` 作为统一交易订单。
- `OrderItem.itemType` 区分 `project`、`product`、`card`、`recharge` 等明细类型。
- `OrderItem.subtotal` 当前承担明细小计功能。
- `OrderItem.discount` 已存在，但未形成完整的“整单优惠分摊 + 明细净额”口径。
- 项目毛利已经按项目订单明细、项目 BOM、实际耗材、项目提成计算。
- 商品毛利已经按商品订单明细、商品成本、库存销售出库、商品提成计算。
- 终端收银 `CheckoutDto` 已有 `discountAmount`，但更偏订单级优惠输入，缺少分摊结果字段。

当前缺口：

1. 缺少订单头和明细行之间的优惠分摊规则。
2. 缺少明细原价、明细优惠、整单分摊优惠、明细实收的稳定字段。
3. 商品/项目毛利目前主要读取 `subtotal`，后续应切到明细净额，或明确兼容规则。
4. 卡项、充值、赠品、退款对优惠分摊的处理需要统一。
5. 页面和小票缺少“原价、优惠、实收、分摊方式”的可解释展示。

## 3. 产品原则

### 3.1 前台录入简单

前台收银不要求每一次都拆到项目、商品、卡项分别手动填优惠。

支持三种输入方式：

| 输入方式 | 使用场景 | 系统处理 |
| --- | --- | --- |
| 明细优惠 | 某个项目体验价、某个商品第二件半价 | 优惠直接落到该明细 |
| 整单优惠 | 手工减 100、整单 9 折、抹零 | 按规则分摊到参与优惠的明细 |
| 套餐价 | 原价合计 2000，套餐价 1600 | 反算优惠 400，再按规则分摊 |

### 3.2 后台核算必须拆明细

无论前台如何输入，保存时都必须形成每一行的：

- 原价小计。
- 明细手工优惠。
- 整单分摊优惠。
- 总优惠。
- 明细实收。
- 分摊方法。
- 优惠来源。

### 3.3 利润使用明细实收

项目、商品、卡项不能继续只看整单金额。所有经营分析都以明细净额为主：

- 项目毛利 = 项目明细实收 - 项目耗材成本 - 项目提成。
- 商品毛利 = 商品明细实收 - 商品成本 - 商品提成。
- 卡项销售 = 卡项明细实收进入预收或履约负债。
- 充值 = 本金与赠送金额分开，本金进入储值负债，赠送金额不作为真实收入。

### 3.4 赠品不是收入

“赠送商品”“赠送护理”“赠送次数”不能被当成折扣后低价销售收入。

推荐口径：

- 赠品商品：收入为 0，商品成本记为营销成本或赠品成本，库存正常出库。
- 赠送项目：核销时收入为 0 或估算履约成本，不计现金收入。
- 赠送金额：进入赠送余额，不计充值真实本金。

## 4. 业务场景与处理规则

### 4.1 项目 + 商品套餐优惠价

例子：补水项目 500 + 面膜商品 300，套餐价 680。

处理：

1. 原价合计 = 800。
2. 套餐优惠 = 800 - 680 = 120。
3. 按原价占比分摊：
   - 项目分摊优惠 = 120 * 500 / 800 = 75。
   - 商品分摊优惠 = 120 * 300 / 800 = 45。
4. 项目实收 = 425。
5. 商品实收 = 255。

利润：

- 项目毛利用 425 扣项目耗材和项目提成。
- 商品毛利用 255 扣商品成本和商品提成。

### 4.2 买项目送商品

例子：做补水项目 500，送面膜 1 盒，面膜零售价 100，成本 40。

推荐处理：

- 项目明细实收 500。
- 商品明细作为赠品行，原价 100，实收 0，优惠来源 `gift`。
- 商品库存正常出库。
- 商品成本 40 记为营销成本或赠品成本，不算商品销售收入。

不建议把项目收入硬分一部分给赠品商品，否则项目毛利和营销成本会被混淆。

### 4.3 办卡赠项目

例子：购买 10 次补水卡 1980，赠肩颈护理 1 次。

推荐处理：

- 卡项明细实收 1980，进入预收或卡项履约负债。
- 赠送项目作为卡项权益或赠送核销权益，不在购买当日确认为项目现金收入。
- 赠送项目实际核销时：
  - 收入为 0 或单独标记为赠送履约。
  - 产生项目耗材成本。
  - 是否给美容师服务提成由门店规则决定，不能默认按项目原价提成。

### 4.4 储值充值赠送

例子：充值 1000 赠 200。

推荐处理：

- 本金余额 = 1000。
- 赠送余额 = 200。
- 收款金额 = 1000。
- 不把 1200 全部当收入。
- 后续消费时，按本金/赠送余额分别扣减，赠送部分不计真实收入或单独计为折扣成本。

### 4.5 整单手工抹零

例子：应收 803，实收 800，抹零 3。

处理：

- 抹零作为整单优惠。
- 按参与优惠明细的原价占比分摊。
- 金额较小时不要求前台选择归属，系统自动分摊即可。

### 4.6 满减券与活动优惠

例子：满 1000 减 200，订单包含项目 600、商品 400。

处理：

- 如果优惠券配置了适用范围，只分摊到适用范围内。
- 如果优惠券适用全部，则按项目 60%、商品 40% 分摊。
- 分摊结果需要保留 `promotionId`、`couponId` 或 `marketingActivityId`，便于营销复盘。

## 5. 分摊算法

### 5.1 默认算法：按原价占比分摊

```text
参与分摊原价合计 = sum(listAmount of eligibleItems)
明细分摊优惠 = orderDiscountAmount * item.listAmount / 参与分摊原价合计
明细实收 = listAmount - itemManualDiscount - orderAllocatedDiscount
```

适用：

- 套餐价。
- 整单折扣。
- 整单满减。
- 抹零。
- 无明确归属的手工优惠。

### 5.2 明细直减

```text
明细实收 = listAmount - itemManualDiscount
```

适用：

- 单个项目体验价。
- 单个商品第二件半价。
- 指定卡项立减。

### 5.3 手工指定分摊

适用少数复杂场景：

- 店长明确要求优惠只算到某个项目。
- 合作活动只补贴商品。
- 平台团购券只覆盖固定项目。

系统应允许高级用户在订单详情或收银确认页调整分摊，但需要记录操作人和调整原因。

### 5.4 分摊舍入规则

金额必须保留两位小数，分摊会出现尾差。推荐规则：

1. 先按比例计算每行优惠，保留两位小数。
2. 计算尾差 = 整单优惠 - 明细优惠合计。
3. 尾差加到原价最高的参与明细；若原价相同，加到最后一行。
4. 保证明细优惠合计永远等于订单优惠。

### 5.5 不参与分摊的明细

以下明细默认不参与普通整单优惠分摊，除非优惠规则明确允许：

- 已标记为赠品的明细。
- 已经实收为 0 的赠送项目或赠送商品。
- 充值赠送金额。
- 不可折扣商品或高监管项目。
- 已使用单独固定优惠价且禁止叠加的套餐明细。

## 6. 数据模型改造

### 6.1 OrderItem 明细字段

建议在 `OrderItem` 增加以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `listAmount` | Decimal | 明细原价小计，通常为 `unitPrice * quantity` |
| `itemDiscountAmount` | Decimal | 明细级手工优惠或单品优惠 |
| `orderAllocatedDiscountAmount` | Decimal | 整单优惠分摊到该明细的金额 |
| `totalDiscountAmount` | Decimal | 明细总优惠 = 明细优惠 + 整单分摊优惠 |
| `netAmount` | Decimal | 明细实收 = 原价 - 总优惠 |
| `discountSource` | String | `none`、`item`、`order`、`package`、`promotion`、`coupon`、`gift`、`manual` |
| `allocationMethod` | String | `none`、`direct`、`price_ratio`、`manual` |
| `discountPayload` | Json | 优惠券、活动、套餐、分摊明细、授权人等扩展信息 |

兼容规则：

- 老数据没有 `netAmount` 时，默认 `netAmount = subtotal`。
- 老数据没有 `listAmount` 时，默认 `listAmount = subtotal + discount` 或 `subtotal`。
- 短期内保留 `subtotal`，但新逻辑应明确 `subtotal` 是否等于实收金额。推荐后续让 `subtotal` 表示明细实收，`listAmount` 表示原价。

### 6.2 ProductOrder 订单头字段

建议在 `ProductOrder` 增加：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `listAmount` | Decimal | 订单原价合计 |
| `orderDiscountAmount` | Decimal | 订单级优惠总额 |
| `itemDiscountAmount` | Decimal | 明细优惠合计 |
| `totalDiscountAmount` | Decimal | 总优惠 |
| `netAmount` | Decimal | 订单实收或应收净额 |
| `promotionId` | Int? | 关联优惠活动 |
| `couponId` | Int? | 关联优惠券 |
| `packageId` | Int? | 关联套餐模板 |
| `discountPayload` | Json | 整单优惠、套餐价、授权记录、分摊结果快照 |

兼容规则：

- 老数据 `totalAmount` 暂时视为 `netAmount`。
- 新订单建议 `totalAmount` 与 `netAmount` 保持一致，减少页面兼容成本。
- 如果未来要区分“应收净额”和“实收支付金额”，可由 `PaymentRecord` 表示实收。

### 6.3 套餐模板 Package

如果只是支持临时套餐价，不一定第一阶段建套餐模板。

如果要支持可复用套餐，建议新增：

```text
PackageTemplate
- id
- storeId
- name
- packageType: project_product / project_card / card_gift / product_bundle
- listAmount
- packagePrice
- discountAmount
- allocationMethod
- status
- validFrom
- validTo
- createdBy

PackageTemplateItem
- id
- packageId
- itemType
- itemId
- nameSnapshot
- quantity
- unitPrice
- listAmount
- discountCapAmount
- isGift
- eligibleForAllocation
```

第一阶段建议不做复杂模板，先支持收银临时组合和明细分摊；第二阶段再沉淀常用套餐模板。

## 7. API 改造

### 7.1 收银提交 DTO

`CheckoutDto` 建议增加：

```ts
type CheckoutDiscountInput = {
  discountMode?: 'none' | 'amount' | 'rate' | 'package_price' | 'manual';
  discountAmount?: number;
  discountRate?: number;
  packagePrice?: number;
  allocationMethod?: 'price_ratio' | 'manual';
  discountSource?: 'order' | 'package' | 'promotion' | 'coupon' | 'manual';
  promotionId?: number;
  couponId?: number;
  reason?: string;
  authorizedBy?: number;
};
```

`CheckoutItemDto` 建议增加：

```ts
type CheckoutItemDiscountInput = {
  listAmount?: number;
  itemDiscountAmount?: number;
  allocatedDiscountAmount?: number;
  netAmount?: number;
  eligibleForOrderDiscount?: boolean;
  isGift?: boolean;
  discountSource?: string;
  allocationMethod?: string;
};
```

### 7.2 后端分摊服务

新增服务建议：

```text
DiscountAllocationService
- normalizeCheckoutItems()
- calculateOrderDiscount()
- allocateDiscountByPriceRatio()
- applyManualAllocation()
- validateDiscountLimits()
- buildOrderDiscountSnapshot()
```

所有入口统一调用该服务：

- 管理端商品订单新增。
- 管理端项目订单新增。
- 终端收银 checkout。
- 办卡。
- 充值。
- 营销活动或优惠券核销。

不要在多个 controller 或 service 内重复写分摊逻辑。

### 7.3 校验规则

必须校验：

1. 优惠金额不能小于 0。
2. 总优惠不能大于参与优惠明细原价合计。
3. 单个明细 `netAmount` 不能小于 0。
4. 赠品明细 `netAmount` 必须为 0。
5. 不可折扣明细不能参与整单优惠，除非店长授权。
6. 大额优惠需要二次确认或授权，阈值可配置。
7. 手工分摊时，明细分摊合计必须等于整单优惠。

## 8. 前端与终端改造

### 8.1 收银台交互

收银台建议结构：

1. 添加项目、商品、卡项、充值。
2. 每行展示：原价、数量、单行优惠、实收。
3. 订单底部展示：
   - 原价合计。
   - 明细优惠。
   - 整单优惠。
   - 套餐价。
   - 应收净额。
4. 支持切换优惠方式：
   - 立减金额。
   - 折扣比例。
   - 输入套餐价。
   - 明细优惠。
5. 点击“分摊明细”可展开查看系统如何分配优惠。

### 8.2 优惠录入建议

默认只给前台简单入口：

- 输入套餐价。
- 输入优惠金额。
- 输入折扣。

高级入口给店长或管理员：

- 手动调整某一行优惠。
- 选择哪些明细参与优惠。
- 选择优惠来源和活动。
- 填写大额优惠原因。

### 8.3 订单详情展示

订单详情必须展示：

| 明细 | 原价 | 明细优惠 | 整单分摊 | 实收 | 成本 | 提成 | 毛利 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |

产品经理验收时重点看两点：

1. 前台能看懂客户到底优惠了多少。
2. 老板能看懂优惠后每个项目/商品是否赚钱。

### 8.4 小票展示

小票建议：

- 面向客户展示简单版：原价、优惠、实收。
- 不展示复杂分摊算法。
- 内部订单详情保留完整分摊。

## 9. 利润、提成与退款口径

### 9.1 项目毛利

新口径：

```text
项目收入 = sum(OrderItem.netAmount where itemType in project/service/service_project)
项目毛利 = 项目收入 - 实际耗材成本或 BOM 标准成本 - 项目提成
```

缺 `netAmount` 的历史订单：

```text
项目收入 = OrderItem.subtotal
```

### 9.2 商品毛利

新口径：

```text
商品收入 = sum(OrderItem.netAmount where itemType in product/goods)
商品毛利 = 商品收入 - 商品成本 - 商品提成
```

赠品商品：

- 商品收入 = 0。
- 商品成本仍计入营销成本或赠品成本。
- 是否进入商品毛利页，需要页面明确标识“赠品成本”。

### 9.3 卡项履约

卡项销售：

```text
卡项销售实收 = OrderItem.netAmount
预收负债增加 = 卡项销售实收
```

卡项核销：

- 按次确认履约收入或消课估算收入。
- 赠送次数核销不确认现金收入。
- 但仍应记录耗材成本和服务提成规则。

### 9.4 提成

提成计算推荐使用明细净额：

```text
提成基数 = OrderItem.netAmount
```

门店可配置：

| 配置 | 说明 |
| --- | --- |
| 按原价提成 | 激励服务，门店承担优惠成本 |
| 按实收提成 | 默认推荐，优惠后收入更真实 |
| 赠品无提成 | 赠品项目/商品不产生销售提成 |
| 赠送服务固定工费 | 赠送项目只给服务工费，不按销售额提成 |

第一阶段建议默认按实收提成，并在提成规则中保留配置能力。

### 9.5 退款

退款必须按明细净额处理。

整单退款：

```text
退款金额 = sum(明细 netAmount)
恢复库存/撤销消耗/撤销提成按明细执行
```

部分退款：

- 用户选择要退的明细和数量。
- 系统按该明细 `netAmount / quantity` 计算可退金额。
- 不能按原价退，否则套餐优惠会被多退。

例子：

项目原价 500，分摊后实收 425。退该项目时最高退 425，不是 500。

## 10. 库存与成本口径

### 10.1 商品

商品销售：

- 按商品明细数量出库。
- 商品成本按订单快照、库存流水或商品档案成本读取。
- 商品收入按 `netAmount`。

商品赠品：

- 仍然出库。
- 收入为 0。
- 成本进入赠品成本或营销成本。

### 10.2 项目耗材

项目销售：

- 按项目 BOM 或实际耗材产生 `StockMovement`。
- 项目收入按 `netAmount`。
- 耗材成本不因优惠而减少。

这点很关键：优惠减少的是收入，不减少真实耗材成本。

## 11. 数据迁移策略

### 11.1 新字段默认值

历史订单迁移建议：

```text
OrderItem.listAmount = OrderItem.subtotal + OrderItem.discount
OrderItem.itemDiscountAmount = OrderItem.discount
OrderItem.orderAllocatedDiscountAmount = 0
OrderItem.totalDiscountAmount = OrderItem.discount
OrderItem.netAmount = OrderItem.subtotal
OrderItem.discountSource = discount > 0 ? 'item' : 'none'
OrderItem.allocationMethod = discount > 0 ? 'direct' : 'none'
```

如果历史 `discount` 语义不稳定，则保守迁移：

```text
OrderItem.listAmount = OrderItem.subtotal
OrderItem.netAmount = OrderItem.subtotal
OrderItem.totalDiscountAmount = 0
```

推荐先做只读审计，再决定迁移策略。

### 11.2 订单头迁移

```text
ProductOrder.netAmount = ProductOrder.totalAmount
ProductOrder.listAmount = sum(OrderItem.listAmount)
ProductOrder.totalDiscountAmount = max(0, listAmount - netAmount)
```

如果订单缺明细，则保持 `listAmount = totalAmount`。

## 12. 分阶段实施计划

### 阶段 0：口径冻结与测试样本

目标：先冻结业务口径，不动核心链路。

交付：

1. 固化本方案中的分摊公式。
2. 准备 8 类验收样本：
   - 项目 + 商品套餐价。
   - 商品第二件半价。
   - 买项目送商品。
   - 办卡赠项目。
   - 充值赠金额。
   - 整单抹零。
   - 优惠券满减。
   - 部分退款。
3. 明确各样本的项目收入、商品收入、卡项负债、库存、提成、退款预期。

验收：

- 产品、财务、研发对字段和公式达成一致。
- 样本可以手工复算。

### 阶段 1：后端分摊服务与字段落库

目标：新增字段和统一分摊服务。

交付：

1. Prisma schema 增加订单头和明细优惠字段。
2. 新增 migration。
3. 新增 `DiscountAllocationService`。
4. 终端收银、管理端订单新增统一调用分摊服务。
5. 新订单写入 `listAmount`、`netAmount`、`discountPayload`。

验收：

- 单测覆盖比例分摊、尾差处理、赠品、不可折扣明细。
- 新订单明细优惠合计等于订单总优惠。
- 老订单仍可查询。

### 阶段 2：利润、提成、退款切换净额口径

目标：经营结果使用明细净额。

交付：

1. 商品毛利 API 使用 `OrderItem.netAmount`，缺失时回退 `subtotal`。
2. 项目毛利 API 使用 `OrderItem.netAmount`，缺失时回退 `subtotal`。
3. 提成计算默认使用 `netAmount`。
4. 退款计算按明细净额和数量处理。
5. 卡项销售写入实收净额，赠送权益单独标识。

验收：

- 同一套餐订单在项目毛利和商品毛利中收入加总等于订单实收。
- 商品赠品不产生商品销售收入，但有库存出库和赠品成本。
- 部分退款不会按原价多退。

### 阶段 3：前端收银与订单详情改造

目标：让前台易录入，让老板看得懂。

交付：

1. 收银台支持套餐价、整单立减、整单折扣、明细优惠。
2. 订单确认页展示优惠分摊预览。
3. 订单详情展示原价、优惠、实收、分摊方式。
4. 小票展示客户友好版优惠信息。
5. 大额优惠二次确认或店长授权入口。

验收：

- 前台能在 30 秒内完成项目 + 商品套餐收银。
- 店长能在订单详情看到每个明细分摊了多少优惠。
- 页面金额合计无尾差。

### 阶段 4：套餐模板与营销联动

目标：把高频组合沉淀为可复用套餐。

交付：

1. 新增套餐模板管理。
2. 营销活动可引用套餐模板。
3. 小程序/H5 可展示套餐价。
4. 收银选择营销活动后自动带入套餐价和分摊规则。
5. 营销复盘统计套餐收入、优惠成本、转化和毛利。

验收：

- 从营销活动到收银核销再到利润复盘口径一致。
- 活动优惠不会只停留在文案，能落到订单明细。

## 13. 验收样本

### 样本 1：项目 + 商品套餐价

| 明细 | 原价 | 分摊优惠 | 实收 | 成本 | 提成 | 毛利 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 补水项目 | 500 | 75 | 425 | 80 | 42.5 | 302.5 |
| 面膜商品 | 300 | 45 | 255 | 120 | 25.5 | 109.5 |
| 合计 | 800 | 120 | 680 | 200 | 68 | 412 |

验收点：

- 分摊优惠合计 = 120。
- 明细实收合计 = 680。
- 项目/商品收入合计 = 订单实收。

### 样本 2：买项目送商品

| 明细 | 原价 | 优惠 | 实收 | 成本 | 收入口径 |
| --- | ---: | ---: | ---: | ---: | --- |
| 补水项目 | 500 | 0 | 500 | 80 | 项目收入 |
| 赠送面膜 | 100 | 100 | 0 | 40 | 赠品成本 |

验收点：

- 商品不计销售收入。
- 商品库存扣减。
- 赠品成本可在营销成本中看到。

### 样本 3：办卡赠项目

| 明细 | 原价 | 实收 | 处理 |
| --- | ---: | ---: | --- |
| 10 次补水卡 | 1980 | 1980 | 进入卡项预收 |
| 赠肩颈护理 | 200 | 0 | 赠送权益，核销时记耗材 |

验收点：

- 当日不确认赠送项目现金收入。
- 后续核销赠送项目不重复确认收入。

### 样本 4：部分退款

项目原价 500，分摊后实收 425。

验收点：

- 退项目最高退款 425。
- 不能退 500。
- 项目提成按退款比例冲回。
- 若已产生耗材，是否冲回库存按门店规则处理。

## 14. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 历史 `subtotal` 语义不一致 | 毛利切换后口径漂移 | 新字段兼容回退，先审计历史订单 |
| 前台不理解分摊 | 操作负担变重 | 默认自动分摊，只在详情页解释 |
| 大额优惠滥用 | 利润失真 | 阈值授权、原因必填、审计日志 |
| 赠品被当收入 | 虚增销售额 | 赠品 `netAmount=0`，成本单独归集 |
| 卡项收入提前确认 | 利润虚高 | 卡项销售进入预收，核销再确认服务履约 |
| 退款按原价退 | 多退钱 | 退款基于 `netAmount` |
| 多入口重复实现 | 终端和管理端口径不一致 | 统一 `DiscountAllocationService` |

## 15. 权限与审计

建议增加以下控制：

- 大额优惠阈值：例如优惠超过 300 元或超过 30%，需要店长授权。
- 手工分摊调整：需要记录操作人、原因、调整前后金额。
- 赠品出库：需要记录赠品原因，可关联营销活动或客户安抚。
- 退款：展示原价、分摊优惠、可退净额，防止前台误退。

## 17. 推荐落地顺序

推荐先做“临时组合套餐 + 自动分摊”，再做“套餐模板”。

原因：

1. 门店真实收银最先需要的是快速处理现场优惠。
2. 自动分摊是利润、提成、退款准确的基础。
3. 套餐模板属于运营效率增强，不应挡住基础核算。

建议优先级：

| 优先级 | 能力 | 说明 |
| --- | --- | --- |
| P0 | 明细净额字段与分摊服务 | 所有后续核算基础 |
| P0 | 终端收银整单优惠自动分摊 | 高频现场场景 |
| P0 | 商品/项目毛利使用净额 | 解决利润失真 |
| P1 | 退款按净额处理 | 防止多退 |
| P1 | 赠品和充值赠送专项口径 | 美业高频 |
| P2 | 套餐模板 | 提升复用效率 |
| P2 | 营销活动联动套餐 | 支撑活动闭环 |

## 18. 最终完成标准

本改造完成后，应满足：

1. 前台可以一笔订单同时收项目、商品、卡项，并录入套餐价或整单折扣。
2. 系统自动把整单优惠分摊到每个明细。
3. 订单详情能解释每个明细的原价、优惠、实收。
4. 项目毛利、商品毛利、卡项预收、提成、退款都使用一致的明细净额。
5. 赠品、赠送项目、充值赠送不会被误算为真实收入。
6. 同一订单在收银、小票、订单详情、利润看板、提成记录中的金额可以闭环复算。
