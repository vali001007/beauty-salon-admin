## PR 与 GitHub CI 计划

1. 推送分支：

```powershell
git push -u origin codex/release-v0.9.0-rc4
```

2. 创建 PR：

```powershell
gh pr create --base main --head codex/release-v0.9.0-rc4 --title "v0.9.0-rc.4：Agent Runtime 与前端对话内核统一预发布"
```

3. PR 必须等待以下 CI 全部通过：

- `frontend`
- `backend`
- `terminal-prototype`
- `ami-semantic-agent`

4. CI 失败处理：

- 先读取失败 job 日志。
- 本地复现失败命令。
- 修复后追加提交到同一分支。
- 不绕过失败 CI 合并。

5. CI 通过后执行 squash merge：

```powershell
gh pr merge <PR_NUMBER> --squash --delete-branch --subject "v0.9.0-rc.4：Agent Runtime 与前端对话内核统一预发布"
```

## Release 计划

合并后执行：

```powershell
git switch main
git fetch --prune origin
git pull --ff-only origin main
```

等待 `main` 分支 CI/CD 完成，确认：

- workflow conclusion 为 `success`。
- `frontend`、`backend`、`terminal-prototype`、`ami-semantic-agent` 成功。
- `deploy-production` job 成功。
- 若实际部署步骤因 secrets 缺失被跳过，必须写入 Release notes。

创建 tag：

```powershell
git tag -a v0.9.0-rc.4 -m "v0.9.0-rc.4"
git push origin v0.9.0-rc.4
```

创建 GitHub Pre-release：

```powershell
gh release create v0.9.0-rc.4 --prerelease --target <MAIN_MERGE_SHA> --title "v0.9.0-rc.4 - Agent Runtime 与前端对话内核统一预发布" --notes-file <release-notes-file>
```

Release notes 必须包含：

- 核心能力。
- 本地验证结果。
- PR CI 结果。
- main CI/CD 与 deploy-production 状态。
- 已知风险。
- 回滚方式。

## 已知风险

- 当前改动跨 `server-v2`、管理端、Kiosk、共享内核，属于 L3 高风险发布。
- `packages/agent-core` 是新增共享包，需要确认 Vite、Vitest、TS alias 在根应用和 Kiosk 中都解析一致。
- Kiosk Agent 对话体验改动较大，需要重点验证首屏、快捷指令、消息渲染、事实上下文和 micro app 执行。
- 管理端订单/商品/预约页面引入复用选择器，需确认原有新增/编辑流程不退化。
- Release 后如果生产 deploy secrets 仍缺失，GitHub job 可能 success 但实际 Vercel 部署仍跳过。

## 回滚方式

- 代码层：revert `v0.9.0-rc.4` 对应 squash merge commit。
- 发布层：保留 tag 并在 Release notes 标记 superseded，或删除 prerelease。
- 数据层：本计划不自动执行真实写库脚本；如涉及数据库变更，必须单独授权和单独回滚。

## 建议执行顺序

1. 先按本计划进行分批暂存和提交。
2. 每批提交前检查 `git diff --cached --name-only`。
3. 完成提交后跑完整本地验证。
4. 推送分支并创建 PR。
5. CI 全绿后 squash merge。
6. main CI/CD 成功后创建 `v0.9.0-rc.4` Pre-release。
