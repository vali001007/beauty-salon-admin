# 一键智能排班成熟算法需求文档

日期：2026-06-29

## 1. 背景

当前排班管理已经具备智能排班入口，并已接入 `Reservation`、`Schedule`、`Beautician`、`BeauticianAvailability`、`BeauticianTimeOff`、`BeauticianProjectSkill`、`SchedulingRuleConfig`、`SmartSchedulingRun` 等对象。

现有算法定位是“规则启发式 + 贪心补人 + 冲突评分”，适合生成可人工确认的排班建议，但与成熟排班算法相比，还缺少全局最优化、工时合规、资源容量、成本收益、动态重排和多方案对比。

本需求目标是把“智能排班”升级为“按成熟排班算法执行的一键智能排班”：店长只需点击一次，系统自动读取预约、员工、技能、请假、资源、门店规则和历史需求，生成可解释、可对比、可发布、可回滚的最优或近似最优排班方案。

## 2. 产品目标

### 2.1 核心目标

- 一键生成整周排班，不要求店长逐项配置。
- 已预约项目必须优先覆盖，不能漏排。
- 无预约时段默认空闲，不再生成无意义的忙碌状态。
- 自动识别高峰时段并补齐人手。
- 自动避开请假、不可用时段、非在职员工、营业时间外时段。
- 自动考虑技能匹配、房间/设备容量、最大工时、休息间隔。
- 在多个可行方案中选择综合评分最高方案。
- 输出清晰解释：为什么这样排、还剩哪些风险、哪些地方需要人工确认。
- 发布前可预览，发布后可回滚。

### 2.2 业务目标

- 降低店长排班时间：从手工 30-60 分钟降到 1-3 分钟确认。
- 降低预约漏排和请假冲突。
- 提升高峰时段人手覆盖率。
- 提升项目与美容师技能匹配率。
- 控制低峰冗余排班，减少无效人力。
- 为终端、预约、收银、绩效提供统一排班真相源。

## 3. 用户角色

| 角色 | 目标 | 关注点 |
| --- | --- | --- |
| 店长/门店管理员 | 快速生成可执行班表 | 是否覆盖预约、是否有人手不足、是否有冲突 |
| 美容师 | 查看自己的班表和预约安排 | 工时是否合理、请假是否生效、是否被排到不擅长项目 |
| 系统管理员 | 配置门店排班规则 | 营业时间、最大工时、资源容量、算法权重 |
| 财务/运营 | 看人效与成本 | 高峰覆盖、低峰冗余、人力成本、产出效率 |

## 4. 使用场景

### 4.1 周排班一键生成

店长进入排班管理，点击“一键智能排班”，系统自动生成下一周或当前周未发布班表。

系统自动读取：

- 当前门店营业时间。
- 当前周项目预约。
- 未指定美容师的预约。
- 美容师在职状态。
- 美容师技能与项目关系。
- 美容师请假与不可用时段。
- 房间、床位、设备等资源容量。
- 历史 4-8 周预约需求。
- 已有排班和已发布排班。
- 门店规则和算法权重。

### 4.2 预约变更后滚动重排

当发生新增预约、取消预约、改约、改美容师、请假审批通过时，系统提示“当前班表有变化，建议重新优化”。店长可以点击“一键重新优化”，系统只调整受影响时段，尽量保持已确认班表稳定。

### 4.3 发布前人工确认

系统生成方案后展示：

- 综合评分。
- 预约覆盖率。
- 高峰覆盖率。
- 技能匹配率。
- 工时公平度。
- 人力成本预估。
- 冲突数量。
- 与当前班表差异。

店长确认后发布；如果存在硬冲突，不允许发布。

### 4.4 回滚

发布后如果发现异常，店长可以回滚到上一版已发布班表。

## 5. 范围定义

### 5.1 本期必须包含

- 一键生成入口。
- 默认算法策略，无需手动配置也能运行。
- 预约覆盖。
- 请假避让。
- 营业时间约束。
- 在职状态约束。
- 技能匹配。
- 高峰需求补人。
- 最大日工时、最大周工时。
- 最小休息间隔。
- 未指定美容师预约自动分配。
- 多目标评分。
- 预览、发布、回滚。
- 算法运行记录。

### 5.2 本期暂不包含

- 跨门店借调排班。
- 员工自主换班。
- 复杂劳动法地区差异配置。
- AI 自然语言解释生成。
- 与外部日历双向同步。

这些可以进入后续版本。

## 6. 一键智能排班用户流程

### 6.1 默认流程

1. 店长进入 `门店管理 / 排班管理`。
2. 点击“一键智能排班”。
3. 系统弹出确认窗口：
   - 排班周期：默认下周；如果当前周未发布，则默认当前查看周。
   - 策略：默认“平衡模式”。
   - 提示：系统会保留已确认预约和请假，不会直接发布。
4. 店长点击“生成方案”。
5. 系统生成 1 个推荐方案，并可选生成 2 个备选方案。
6. 页面展示方案评分、冲突、解释、与当前班表差异。
7. 店长点击“发布排班”。
8. 系统再次校验硬冲突。
9. 发布成功，写入排班表并记录版本。

### 6.2 高级流程

高级设置默认收起，可供店长或管理员调整：

- 策略模式：
  - 平衡模式。
  - 预约优先。
  - 高峰优先。
  - 成本优先。
  - 公平优先。
- 是否允许调整未确认预约美容师。
- 是否允许调整已确认预约美容师。
- 是否允许低峰压缩人手。
- 是否生成备选方案。
- 是否只优化冲突时段。

## 7. 状态口径

排班页对用户只展示四种状态：

| 状态 | 来源 | 是否可手动改 |
| --- | --- | --- |
| 空闲 | 无预约、无请假、未过期 | 可改为请假 |
| 已预约 | 项目预约 `Reservation` | 不可在排班页手动改 |
| 请假 | 请假/不可用时段或人工设置 | 可手动改为空闲 |
| 已过期 | 系统按当前时间自动判断 | 不可手动改 |

内部兼容旧状态 `busy/normal`，但新接口、新算法、新写入数据不再产生面向用户的“忙碌/正常”状态。

## 8. 成熟算法设计

### 8.1 算法总体架构

采用两层架构：

1. 约束层：判断方案是否可行。
2. 优化层：在可行方案中寻找综合最优。

推荐实现路径：

- MVP：规则过滤 + 局部搜索 + 多目标评分。
- 增强版：引入 CP-SAT 或整数规划求解器。
- 大门店/多资源版：混合启发式 + 求解器，支持超时返回近似最优。

### 8.2 决策变量

基础变量：

- `x[beautician, date, slot]`：美容师是否在某时段上班。
- `a[reservation, beautician]`：预约是否分配给某美容师。
- `r[resource, date, slot]`：资源是否被占用。

扩展变量：

- `overtime[beautician]`：是否产生超时工作。
- `understaffed[date, slot]`：是否低于需求人手。
- `preferencePenalty[beautician, slot]`：员工偏好违背惩罚。
- `changePenalty[slot]`：对已发布/已确认班表的改动惩罚。

### 8.3 硬约束

硬约束不满足时，方案不可发布。

| 编号 | 约束 | 说明 |
| --- | --- | --- |
| H1 | 营业时间 | 不允许排到门店营业时间外 |
| H2 | 在职状态 | 离职、休假、停用美容师不可排班 |
| H3 | 请假 | 已审批请假时段不可排班 |
| H4 | 不可用时段 | 美容师明确不可用时段不可排班 |
| H5 | 预约覆盖 | 已确认预约必须有可服务美容师覆盖 |
| H6 | 时间重叠 | 同一美容师同一时间不可服务多个预约 |
| H7 | 最大日工时 | 不得超过门店设置的日最大工时 |
| H8 | 最大周工时 | 不得超过门店设置的周最大工时 |
| H9 | 最小休息间隔 | 跨班次必须满足最小休息间隔 |
| H10 | 技能硬要求 | 项目要求认证时，未认证美容师不可服务 |
| H11 | 资源容量 | 房间、床位、设备不可超容量 |
| H12 | 已锁定预约 | 已锁定美容师的预约不可随意改派 |

### 8.4 软约束

软约束用于评分和排序。

| 编号 | 目标 | 评分方向 |
| --- | --- | --- |
| S1 | 高峰覆盖 | 高峰缺口越少越好 |
| S2 | 工时公平 | 美容师周工时差异越小越好 |
| S3 | 员工偏好 | 越符合 preferred 时段越好 |
| S4 | 技能匹配 | 技能等级越高、认证越完整越好 |
| S5 | 成本控制 | 低峰冗余越少、人力成本越低越好 |
| S6 | 班表稳定 | 对已发布/已确认班表改动越少越好 |
| S7 | 连续服务体验 | 同一客户连续项目尽量安排同一美容师 |
| S8 | 收益优先 | 高价值项目优先匹配高技能美容师 |

### 8.5 策略模式

#### 平衡模式

默认策略，适合日常排班。

权重建议：

- 预约覆盖：30
- 高峰覆盖：20
- 工时公平：15
- 技能匹配：15
- 成本控制：10
- 班表稳定：10

#### 预约优先

适合预约多、改约多的门店。

- 提高预约覆盖和技能匹配权重。
- 允许适度增加人手成本。
- 对未指定美容师预约主动分配最合适美容师。

#### 高峰优先

适合周末、活动期、节假日前后。

- 提高高峰覆盖权重。
- 高峰时段允许更多冗余人手。
- 低峰时段正常压缩。

#### 成本优先

适合客流低或需要控成本时。

- 覆盖硬约束后尽量减少排班人数。
- 低需求时段不主动补人。
- 需要明确提示可能降低现场接待弹性。

#### 公平优先

适合员工排班争议较多的门店。

- 提高工时公平和偏好满足权重。
- 控制连续晚班、连续周末班。
- 保留必要预约覆盖。

## 9. 需求预测

### 9.1 数据来源

- 当前周已预约项目。
- 历史 4-8 周同星期同时间段预约。
- 项目类型和服务时长。
- 营销活动、优惠券、活动推广带来的预期流量。
- 节假日、周末、门店特殊日。
- 线下 walk-in 经验值。

### 9.2 预测输出

每个 `date + slot` 输出：

- `expectedReservations`：预计预约数。
- `walkInBuffer`：到店咨询/临时客缓冲。
- `requiredStaff`：建议在岗人数。
- `confidence`：预测置信度。
- `demandLevel`：低、中、高、爆满。

### 9.3 MVP 简化规则

MVP 不先做复杂机器学习，先做可解释规则：

`requiredStaff = max(已预约占用人数, 历史平均需求, 高峰最低人手, walkInBuffer)`

后续再引入预测模型。

## 10. 数据模型需求

### 10.1 复用现有对象

- `Schedule`：最终发布排班。
- `Reservation`：预约来源。
- `SchedulingRuleConfig`：门店规则。
- `BeauticianAvailability`：可用/不可用/偏好时段。
- `BeauticianTimeOff`：请假。
- `SmartSchedulingRun`：算法运行记录。
- `BeauticianProjectSkill`：项目技能匹配。
- `StoreResource`：资源档案。
- `ResourceBooking`：资源占用。

### 10.2 建议新增或扩展字段

#### SchedulingRuleConfig 扩展

- `algorithmMode`：默认策略模式。
- `objectiveWeights`：目标权重 JSON。
- `allowReassignUnconfirmedReservation`：是否允许调整未确认预约美容师。
- `allowReassignConfirmedReservation`：是否允许调整已确认预约美容师。
- `walkInBufferRules`：临时到店缓冲规则。
- `lockedAfterPublished`：发布后是否默认锁定。

#### Schedule 扩展

- `source`：manual、smart、reservation、import。
- `locked`：是否锁定。
- `versionId`：所属发布版本。
- `optimizationRunId`：来源算法运行。

#### SmartSchedulingRun 扩展

- `algorithmVersion`：算法版本。
- `mode`：策略模式。
- `objectiveWeights`：本次权重。
- `inputSummary`：输入摘要。
- `solutionSummary`：方案摘要。
- `alternatives`：备选方案。
- `runtimeMs`：计算耗时。
- `solverStatus`：optimal、feasible、timeout、failed。
- `publishedScheduleVersionId`：发布版本。

#### 新增 ScheduleVersion

用途：支持发布、对比、回滚。

字段建议：

- `id`
- `storeId`
- `weekStart`
- `status`：draft、published、rolled_back。
- `sourceRunId`
- `publishedById`
- `publishedAt`
- `rollbackFromVersionId`
- `createdAt`

## 11. API 需求

### 11.1 一键生成方案

`POST /api/scheduling/smart/one-click`

入参：

```json
{
  "storeId": 1,
  "weekStart": "2026-07-06",
  "mode": "balanced",
  "generateAlternatives": true,
  "optimizeScope": "week",
  "respectPublishedLocks": true
}
```

出参：

```json
{
  "runId": "smart_1_20260706_001",
  "algorithmVersion": "v2",
  "solverStatus": "feasible",
  "score": 92,
  "summary": {
    "reservationCoverageRate": 1,
    "peakCoverageRate": 0.96,
    "skillMatchRate": 0.93,
    "fairnessScore": 0.88,
    "estimatedLaborCost": 3200,
    "hardConflictCount": 0,
    "softWarningCount": 2
  },
  "recommended": {},
  "alternatives": [],
  "explanations": []
}
```

### 11.2 方案评估

`POST /api/scheduling/smart/evaluate`

用于手动调整后重新评分。

### 11.3 发布方案

`POST /api/scheduling/smart/publish`

要求：

- 发布前重新执行硬约束校验。
- 硬冲突为 0 才能发布。
- 创建 `ScheduleVersion`。
- 写入 `Schedule`。
- 记录 `SmartSchedulingRun.confirmedAt`。

### 11.4 回滚方案

`POST /api/scheduling/smart/rollback`

入参：

```json
{
  "storeId": 1,
  "weekStart": "2026-07-06",
  "targetVersionId": 12
}
```

要求：

- 只能回滚同门店、同周期版本。
- 回滚前检查是否存在新预约冲突。
- 回滚动作写审计日志。

### 11.5 方案对比

`GET /api/scheduling/smart/runs?weekStart=2026-07-06`

返回历史运行记录、评分、发布时间、发布人、是否当前版本。

## 12. 页面需求

### 12.1 一键入口

当前“智能排班”按钮升级为主入口：

- 文案：一键智能排班。
- 默认直接进入生成确认弹窗，不先展开复杂配置。
- 旁边保留“高级设置”入口。

### 12.2 生成中状态

展示：

- 正在读取预约。
- 正在计算高峰需求。
- 正在匹配美容师技能。
- 正在检查请假和资源。
- 正在生成最优方案。

如果超过 10 秒，提示“方案较复杂，正在寻找更优解”；超过 30 秒可返回当前可行解。

### 12.3 预览结果

展示四个层级：

1. 总览卡片：
   - 综合评分。
   - 预约覆盖率。
   - 高峰覆盖率。
   - 技能匹配率。
   - 工时公平度。
   - 预计人力成本。

2. 风险卡片：
   - 硬冲突。
   - 软提醒。
   - 需要人工确认事项。

3. 差异卡片：
   - 新增排班。
   - 删除排班。
   - 调整美容师。
   - 调整资源。

4. 解释卡片：
   - 本方案为什么这样排。
   - 哪些预约被优先保障。
   - 哪些时段是高峰。
   - 哪些员工工时被平衡。

### 12.4 备选方案

如果开启备选方案，展示：

- 推荐方案。
- 成本更低方案。
- 覆盖更高方案。
- 更公平方案。

每个方案可查看差异并切换预览。

### 12.5 发布确认

发布弹窗展示：

- 发布周期。
- 覆盖预约数。
- 涉及美容师数。
- 是否存在软提醒。
- 发布后是否通知终端。
- 是否允许回滚。

硬冲突存在时，发布按钮禁用。

## 13. 终端与其他模块联动

### 13.1 项目预约

- 预约新增、取消、改约后，应标记当前排班需要重新优化。
- 排班页中“已预约”只来源于 `Reservation`。

### 13.2 终端

- 终端员工看板读取已发布排班。
- 今日预约和排班冲突需要提示。
- 终端不能绕过排班状态自行写入预约状态。

### 13.3 收银/服务

- 服务开单时可以基于排班推荐可服务美容师。
- 服务开始后占用资源和美容师时段。

### 13.4 财务/运营

- 记录预计人力成本和实际产出。
- 后续用于评估算法效果。

## 14. 权限与审计

权限建议：

- `core:store:scheduling`：查看排班。
- `core:store:scheduling:smart`：生成智能排班。
- `core:store:scheduling:publish`：发布排班。
- `core:store:scheduling:rollback`：回滚排班。
- `core:store:scheduling:config`：配置算法规则。

审计记录：

- 谁生成了方案。
- 输入参数。
- 算法版本。
- 发布版本。
- 回滚版本。
- 发布前冲突摘要。

## 15. 算法验收指标

### 15.1 必达指标

- 已确认预约覆盖率：100%。
- 请假硬冲突：0。
- 非在职员工排班：0。
- 营业时间外排班：0。
- 同美容师重叠排班：0。
- 最大日/周工时违规：0。
- 最小休息间隔违规：0。

### 15.2 优化指标

- 高峰覆盖率：>= 95%。
- 技能匹配率：>= 90%。
- 工时公平度：>= 85%。
- 低峰冗余率：逐周下降。
- 店长人工调整次数：逐周下降。
- 算法运行时长：中小门店 10 秒内返回推荐方案。

## 16. 异常处理

| 场景 | 系统处理 |
| --- | --- |
| 没有预约 | 生成基础可用班表，低峰不强制补人 |
| 美容师全部请假 | 返回不可行方案，提示需人工处理 |
| 预约超过可服务容量 | 返回硬冲突，列出无法覆盖预约 |
| 技能无人满足 | 返回硬冲突或软提醒，取决于项目配置 |
| 资源不足 | 返回资源容量冲突 |
| 求解超时 | 返回当前最佳可行方案，并标记 timeout |
| 发布前数据变化 | 重新校验，提示预约或请假已变化 |

## 17. 分阶段实施建议

### 阶段 1：一键智能排班 MVP

目标：把现有智能排班升级成默认一键入口，并补齐关键硬约束。

范围：

- 一键入口。
- 默认平衡模式。
- 预约覆盖。
- 请假避让。
- 营业时间约束。
- 最大日/周工时。
- 最小休息间隔。
- 未指定美容师预约自动分配。
- 发布前硬冲突拦截。

验收：

- 店长不配置也能生成方案。
- 已确认预约 100% 覆盖。
- 请假、非在职、营业时间外全部拦截。
- 生成方案有评分和解释。

### 阶段 2：成熟算法增强

目标：让不同策略模式真正影响结果。

范围：

- 多目标权重。
- 成本优先、预约优先、高峰优先、公平优先。
- 技能优先级。
- 资源容量。
- 备选方案。
- 方案对比。

验收：

- 不同模式生成结果存在可解释差异。
- 备选方案可切换预览。
- 资源冲突能被识别。

### 阶段 3：运营闭环

目标：让排班算法持续变聪明。

范围：

- 预约变化滚动重排。
- 发布版本回滚。
- 实际服务数据回流。
- 人力成本和产出分析。
- 算法效果看板。

验收：

- 改约/请假后可局部重排。
- 可查看历史方案效果。
- 可量化算法节省时间和减少冲突。
