# 版本变更说明

版本建议：`v0.9.0-rc.6`

更新时间：2026-07-02

发布类型：GitHub Pre-release，Gitee 同步发布

## 1. 发布摘要

本版本是 `v0.9.0-rc.5` 之后的候选预发布版本，重点补齐门店真实经营链路中的会员卡划扣、组合支付、采购入库和财务口径一致性。

对产品交付的影响是：门店在终端或管理端处理会员消费时，可以更准确地区分会员卡本金和赠送金；当会员余额不足时，可以用组合支付完成订单；采购入库能更稳定地匹配商品；财务、提成和经营分析可以沿用同一套订单与支付口径。

## 2. 核心变化

### 2.1 会员卡划扣

- 会员卡划扣支持按项目和商品明细录入，不再只能填写单一金额。
- 每条划扣明细支持数量、单价、服务人员和备注。
- 会员卡本金余额和赠送余额按比例分摊扣减。
- 扣减金额按分处理，降低财务四舍五入误差。
- 订单退款时恢复原本扣减的本金和赠送余额。

### 2.2 组合支付

- 终端收银支持会员余额不足时自动进入组合支付。
- 后端支持接收多支付方式拆分。
- 后端校验组合支付总额必须等于订单实收金额。
- 订单类型补充会员余额划扣明细，便于管理端展示和财务追踪。

### 2.3 库存采购

- 采购入库支持通过商品 ID 匹配库存商品。
- 保留 SKU 匹配能力。
- 商品名称匹配只在门店内唯一时生效，避免同名商品误入库。
- 库存批次补充成本价、总金额和供应商字段。
- 管理端库存和采购页面同步适配新增字段。

### 2.4 财务和经营分析

- 日结统计适配组合支付和会员余额划扣。
- 提成服务适配新的订单支付结构。
- 经营利润和预收负债分析同步补充会员余额相关口径。
- 商品订单和项目订单页面展示会员余额划扣信息，便于门店核对。

### 2.5 Ami Aura 终端

- 终端收银卡片增强会员余额可用性判断。
- 会员余额不足时，终端自动生成会员余额加其他方式的支付拆分。
- 终端提交支付时传递完整 `payments` 结构。
- 终端展示支付摘要，减少店员理解成本。

## 3. 交付价值

- 对店长：能看清会员消费中本金、赠送金、退款恢复和财务报表之间的关系。
- 对前台：会员余额不足时不用中断收银，可以直接补差支付。
- 对财务：日结、提成、经营利润和预收负债的数据来源更统一。
- 对仓管：采购入库匹配商品更稳定，能记录成本和供应商。
- 对终端：Ami Aura Lite 更贴近真实门店收银流程。

## 4. 兼容性说明

- 本版本不计划修改 `package.json` 的版本号，版本号仅作为 Git tag 和 GitHub Release 使用。
- 会员卡划扣接口会继续兼容金额型划扣，但推荐使用明细型划扣。
- 采购入库仍兼容 SKU 匹配；新增 `productId` 是更稳定的优先匹配方式。
- 组合支付要求支付拆分总额必须等于订单实收金额，不再容忍前后端金额不一致。

## 5. 验证建议

发布前必须验证：

1. 会员卡余额充足，全额会员卡支付成功。
2. 会员卡余额不足，组合支付成功。
3. 会员卡本金和赠送金按比例扣减。
4. 订单退款后，本金和赠送金恢复正确。
5. 采购入库通过商品 ID 命中正确商品。
6. 采购入库通过 SKU 命中正确商品。
7. 商品名称重复时，不自动按名称入库。
8. 日结、提成、经营利润、预收负债页面口径一致。
9. Ami Aura Lite 终端完成一笔组合支付订单。

## 6. 已知风险

- 会员卡本金/赠送金分摊属于财务敏感逻辑，必须以单测和真实订单样例共同验证。
- 组合支付会影响订单、支付记录、日结和提成，发布前需要跨页面核对同一笔订单。
- 商品名称匹配存在歧义风险，因此只允许唯一名称自动命中。
- Gitee `master` 如需覆盖，必须使用 `--force-with-lease`，并记录覆盖前 SHA。
- 生产部署不能只看 GitHub CI 绿色，还要确认实际部署日志和线上页面版本。

## 7. 回滚说明

- 通过 PR 合并回 `main`。
- 同步 Gitee `master` 到回滚后的 GitHub `main`。

发布回滚：

- GitHub Release 保留但标记 superseded，或删除 prerelease。
- Gitee tag 如已推送，优先保留并在发布说明中标记废弃；如必须删除，需要单独授权。

数据回滚：

- 本版本发布流程不自动执行生产写库脚本。
- 如需要数据修复或回填，必须单独输出脚本、备份方式和执行授权。
