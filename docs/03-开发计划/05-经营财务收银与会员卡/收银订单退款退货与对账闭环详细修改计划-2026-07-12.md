## 1. 已确认的产品决策

### 1.1 退款模式

退款入口必须明确选择以下一种模式：

| 模式 | API 值 | 资金处理 | 商品库存 | 项目耗材 |
| --- | --- | --- | --- | --- |
| 退款退货 | `return_and_refund` | 写退款流水并原路恢复会员余额、冲回提成和归因 | 按退款商品数量入库 | 按退款项目数量冲销原耗材扣减 |
| 仅退款不退货 | `refund_only` | 写退款流水并原路恢复会员余额、冲回提成和归因 | 不变 | 不变 |

默认规则：

1. 页面不得默认替用户选择“退款退货”，必须显式选择。
2. 商品、项目订单都支持两种模式。
3. 卡项、充值继续走现有专用退款链路，不复用商品退货入库逻辑。
4. “退款退货”必须按订单明细选择退款数量，不允许只填订单总金额后自动猜测退货数量。
5. “仅退款不退货”允许按明细退款，也允许整单按剩余可退金额退款，但仍要生成 `RefundItem`，保证利润、提成和经营分析能定位到业务明细。

### 1.2 状态机

订单状态统一为：

```text
pending -> paid/completed
paid/completed -> partially_refunded
partially_refunded -> partially_refunded
partially_refunded -> refunded
paid/completed -> refunded
```

状态判定：

```ts
const remainingRefundable = round(order.netAmount - successfulRefundAmount);
const nextStatus = remainingRefundable <= 0.01 ? 'refunded' : 'partially_refunded';
```

要求：

- `partially_refunded` 订单继续显示退款按钮。
- 只有累计成功退款达到订单 `netAmount` 才进入 `refunded`。
- 退款失败不得修改订单状态、库存、耗材、会员余额、提成或日结。
- 取消订单不等于退款；没有成功支付流水的订单不得直接生成成功退款。

### 1.3 退款金额口径

1. 退款上限使用明细剩余可退净额，不使用原价。
2. 单个明细的基础单价为 `OrderItem.netAmount / quantity`。
3. 已发生退款的明细必须扣除历史 `RefundItem.refundAmount` 和 `RefundItem.quantity`。
4. 手工调整退款金额时，不得超过本次选择明细的剩余可退净额。
5. 订单级退款金额必须等于本次 `RefundItem.refundAmount` 合计。
6. 组合支付退款本期沿用原订单支付方式展示；实际三方支付渠道原路退回属于支付网关后续能力，不在本计划内伪造成功回执。

### 1.4 库存和耗材冲销口径

商品退款退货：

- 基于原始 `sale_out` 流水反向生成 `sale_return_in`。
- 优先回到原出库批次，保留原 `batchId`、`unitCost`、`costAmount` 和成本来源。
- 入库数量不得超过原订单明细已出库数量减历史退货数量。

项目退款退货：

- 项目服务本身不可“退回”，这里的退货含义固定为冲销项目产生的耗材扣减。
- 基于原始 `service_consume` 流水反向生成 `service_consume_reverse`。
- 必须使用下单时实际扣减流水，不使用退款时最新 BOM 重新计算，避免 BOM 修改后冲销错误。
- 冲销数量按退款项目数量占原项目数量的比例计算，并使用原批次、原单位成本回补。

仅退款不退货：

- 不创建任何 `StockMovement`。
- 利润报表仍扣减退款收入，但保留原商品成本或项目耗材成本，真实反映“钱退了、货或服务成本没有回来”的损失。

### 1.5 历史订单处理

历史 `StockMovement` 缺少 `orderItemId`，不能对所有多项目订单安全推断耗材归属。处理规则：

1. 商品订单按 `ProductOrder + productId` 回填可唯一确定的 `orderItemId`。
2. 只有一个项目明细的订单，可把对应项目耗材流水回填到该 `orderItemId`。
3. 多项目且耗材归属无法唯一确定的订单标记为 `inventory_trace_ambiguous`。
4. `inventory_trace_ambiguous` 订单禁止自动执行“退款退货”，页面提示改用“仅退款不退货”并创建人工库存调整任务。
5. 不允许根据当前 BOM 猜测历史耗材归属。

## 2. 目标业务链路

```text
选择收银单/订单
  -> 加载订单明细、已退数量、已退金额、库存追溯状态
  -> 选择退款模式
  -> 选择退款明细和数量
  -> 后端校验剩余可退金额与剩余可退数量
  -> 单一数据库事务
       -> RefundRecord
       -> RefundItem[]
       -> 会员余额恢复
       -> 商品入库 / 项目耗材冲销（仅退款退货）
       -> 提成冲回
       -> 营销归因冲回
       -> 订单状态更新
  -> 事务成功后刷新退款发生日的日结
  -> 对账页展示资金、退货和耗材冲销结果
```

## 3. 数据模型设计

### 3.1 `RefundRecord` 扩展

修改 `packages/server-v2/prisma/schema.prisma`：

```prisma
model RefundRecord {
  id             Int       @id @default(autoincrement())
  orderId        Int
  refundNo       String    @unique
  requestId      String    @unique
  refundMode     String    @default("refund_only")
  amount         Decimal
  reason         String?
  status         String
  operatorId     Int?
  operatorType   String?
  inventoryStatus String   @default("not_required")
  refundedAt     DateTime?
  createdAt      DateTime  @default(now())

  order          ProductOrder @relation(fields: [orderId], references: [id], onDelete: Cascade)
  operator       User?        @relation(fields: [operatorId], references: [id], onDelete: SetNull)
  items          RefundItem[]

  @@index([orderId])
  @@index([status])
  @@index([refundMode])
  @@index([refundedAt])
}
```

字段约束：

- `requestId`：前端每次提交生成 UUID，用于幂等；重复请求返回原退款结果。
- `refundMode`：只允许 `refund_only`、`return_and_refund`。
- `inventoryStatus`：`not_required`、`pending`、`completed`、`failed`。
- `operatorId/operatorType`：用于审计管理端、终端和系统恢复任务来源。

### 3.2 新增 `RefundItem`

```prisma
model RefundItem {
  id              Int       @id @default(autoincrement())
  refundId        Int
  orderItemId     Int
  itemType        String
  itemId          Int?
  quantity        Decimal
  listAmount      Decimal   @default(0)
  discountAmount  Decimal   @default(0)
  refundAmount    Decimal
  inventoryAction String    @default("none")
  inventoryStatus String    @default("not_required")
  payload         Json?
  createdAt       DateTime  @default(now())

  refund          RefundRecord @relation(fields: [refundId], references: [id], onDelete: Cascade)
  orderItem       OrderItem    @relation(fields: [orderItemId], references: [id], onDelete: Restrict)

  @@index([refundId])
  @@index([orderItemId])
  @@index([itemType, itemId])
}
```

`inventoryAction` 取值：

- `none`
- `sale_return_in`
- `service_consume_reverse`
- `manual_adjustment_required`

### 4.3 收银组退款边界

本期继续按拆分后的实际订单退款，不新增虚拟总订单。页面通过 `checkoutGroupNo` 聚合展示，但提交时必须携带具体 `orderId + orderItemId`。

如果用户从收银组入口发起退款：

1. 前端加载组内全部订单退款预览。
2. 用户选择不同订单中的明细。
3. 后端新增 `POST /orders/checkout-groups/:checkoutGroupNo/refund`。
4. 组内多个退款记录必须在同一事务中创建。
5. 返回一个收银组退款结果和多个 `RefundRecord`，不得把整组金额挂到其中一个商品单。

## 8. 完成标准

只有同时满足以下条件，才能宣称“收银、订单、退款和退货已打通”：

1. 部分退款后仍可继续退款，累计全退后才进入已退款。
2. 每笔退款都有 `RefundItem`，金额能回溯到原订单明细净额。
3. 退款退货能准确恢复商品库存或冲销项目原耗材流水。
4. 仅退款不退货不会改库存，但利润能体现真实损失。
5. 重复请求不会重复退款或重复入库。
6. 混合收银任一拆分失败时整组回滚。
7. 支付现金流、预收资金和营业收入分开对账。
8. 对账页面使用服务端全量汇总，不受 200 条分页限制。
9. 历史歧义订单被明确标记，不自动猜测库存归属。
