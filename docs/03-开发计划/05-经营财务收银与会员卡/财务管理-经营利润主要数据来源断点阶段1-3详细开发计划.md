## 1. 目标与边界

本计划只覆盖产品方案中的阶段 1~3：

1. 阶段 1：次卡事实源补强。
2. 阶段 2：经营利润聚合改造。
3. 阶段 3：预收履约页补齐储值余额。

暂不纳入阶段 4 历史数据批量回填。阶段 1~3 的目标是保证新产生的数据完整、经营利润口径正确、页面可解释。历史数据只做兼容读取和数据缺口提示，不在本计划内强制回填。

核心业务边界：

- 次卡销售/办卡只产生现金流和预收权益，不直接计入经营收入。
- 次卡核销产生项目履约收入，应进入项目毛利和员工人效。
- 次卡核销不能伪装成普通项目订单，避免重复计项目订单收款。
- 储值充值只产生现金流和余额负债，余额消费时按对应项目/商品订单明细确认经营收入。
- 缺来源订单、缺核销单价、缺项目、缺耗材、缺提成时必须提示数据质量缺口。

## 3. 总体技术路线

### 3.1 新增事实源字段

阶段 1 通过增强现有模型完成，不优先新建大表。

| 模型 | 改造目标 |
| --- | --- |
| `CustomerCard` | 记录次卡来源订单、来源订单明细、实际成交金额、赠送次数、核销单价快照 |
| `CardUsageRecord` | 记录核销所属客户卡、卡项、项目、门店、确认收入、来源订单 |
| `CommissionRecord` | 支持 `card_usage` 来源提成，或新增 `cardUsageRecordId` |
| `StockMovement` | 继续以 `sourceType=card_usage`、`sourceId=CardUsageRecord.id` 记录核销耗材 |

### 3.2 统一项目履约收入

阶段 2 在 `OperationProfitService` 内形成统一收入事件：

```text
项目履约收入 = 项目订单收入 + 次卡核销履约收入
```

其中：

```text
项目订单收入 = OrderItem.netAmount where itemType=project
次卡核销履约收入 = CardUsageRecord.recognizedAmount
```

### 3.3 预收履约

阶段 3 把当前“会员卡履约”升级为“预收履约”视角：

```text
预收负债 = 次卡剩余履约价值 + 储值现金余额 + 储值赠送余额
```

## 4. 阶段 1：次卡事实源补强

### 4.1 阶段目标

让新产生的次卡销售、次卡核销数据天然可追溯，能支撑后续项目毛利、员工人效、预收履约和数据质量判断。

完成后应达到：

1. 新办卡后，`CustomerCard` 能追溯来源订单和成交金额。
2. 新核销后，`CardUsageRecord` 能追溯客户卡、卡项、项目、门店、确认收入。
3. 新核销后，BOM 项目能写入 `StockMovement.sourceType=card_usage`。
4. 新核销后，命中提成规则时能生成或关联核销来源提成。
5. 不新增普通项目订单收款，不重复计现金收入。

## 5. 阶段 2：经营利润聚合改造

### 5.1 阶段目标

让利润看板、项目毛利、员工人效按新的次卡事实源计算，解决“次卡核销项目服务发生了，但项目毛利和员工贡献毛利算不准”的问题。

完成后应达到：

1. 利润看板能拆分项目订单收入和次卡核销履约收入。
2. 项目毛利能展示同一项目的订单收入、核销收入、耗材、提成、贡献毛利。
3. 项目毛利支持展开项目订单明细和次卡核销明细。
4. 员工人效计入次卡核销服务收入、核销服务次数和核销提成。

## 9. 最小验收样例

建议准备 1 家门店、2 位客户、2 个项目、1 张次卡、1 次储值充值：

| 样例 | 用途 |
| --- | --- |
| 客户 A 购买 10 次卡，实收 3000 | 验证 `CustomerCard` 来源订单和核销单价 |
| 客户 A 核销 1 次项目 | 验证 `CardUsageRecord`、耗材、提成、项目毛利 |
| 客户 B 储值 1000 赠送 100 | 验证储值余额和预收负债 |
| 客户 B 用余额支付项目/商品 | 验证余额支付不重复计现金收入 |
| 项目配置 BOM 和提成规则 | 验证耗材、提成进入利润 |

验收公式：

```text
现金收入 = 办卡实收 + 储值充值 + 其它现金订单 - 退款
经营收入 = 项目订单收入 + 次卡核销收入 + 商品销售收入
项目毛利 = 项目订单收入 + 次卡核销收入 - 项目耗材 - 项目提成
预收负债 = 次卡剩余履约价值 + 储值现金余额 + 储值赠送余额
```
