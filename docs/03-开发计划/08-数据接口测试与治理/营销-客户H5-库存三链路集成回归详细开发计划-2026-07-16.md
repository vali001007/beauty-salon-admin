## 一、任务结论

本轮不把“智能推荐、公开活动页、Ami Glow H5、库存工作台分别能打开”定义为完成。

真正的业务闭环必须保留并验证同一条业务事实链：

```text
推荐实例
  → 推荐采纳
  → 活动 / 自动触达
  → 公开营销页 / Ami Glow H5
  → 浏览、留资、通知打开、领取、预约
  → 门店接待与收银订单
  → 支付完成
  → 商品销售出库 / 项目 BOM 消耗
  → 营销归因与效果事实
  → 管理端推荐、效果复盘、库存流水可查
```

闭环中的关键身份必须能够从前向后追踪：

```text
storeId
customerId
recommendationInstanceId
adoptionId
activityId / strategyId
pageId / promotionId
touchId / leadId
reservationId
orderId / orderItemId
stockMovementId
effectFactId / attributionId
```

缺失任意关键身份时，不得用时间相近、名称相同或页面文案推断链路已经打通。

- 页面事件和留资双写 `MarketingEffectFact`。

`CustomerAppService` 已支持：

#### 订单和库存

- 已支付商品订单生成 `sale_out` 库存流水。

- 同一订单库存扣减具备幂等检查。
- 支付订单触发营销归因。
- 退款链路可反向处理库存和营销效果。

### 2.3 当前确定断点

#### 断点一：公开营销页“预约”不是正式预约

Marketing H5 的 booking 请求当前写入 `MarketingPageLead` 和 `MarketingPageEvent(book)`，不会创建 `Reservation`。

产品口径必须区分：

- `book`：客户提交预约意向。
- `Reservation`：已经占用门店日期、时段和美容师产能的正式预约。

回归报告不得把 booking lead 统计为正式预约成功。

#### 断点二：H5 预约缺少结构化营销来源

当前 `Reservation` 没有以下结构化字段：

- `source`、`channel`。
- `promotionId`。
- `campaignId`。
- `recommendationInstanceId`。
- `adoptionId`。
- `sourceMarketingPageId`。

相关信息主要保存在 `remark` 和 `CustomerBehaviorEvent.metadataJson`，无法稳定支持查询、归因和数据修复。

#### 断点三：预约与订单没有直接关联

`ProductOrder` 当前没有 `reservationId` 或 `sourceReservationId`。门店完成接待和收银后，只能按客户、项目和时间推断订单来自哪个预约。

这不满足生产级归因要求。

#### 断点四：H5 预约不是订单归因候选

`MarketingAttributionService.attributeOrder` 当前只读取：

- `MarketingAutomationTouch`。
- `MarketingPageLead`。

它不会直接读取 Ami Glow H5 的 `promotion_reserved`、`h5_reservation_success` 或正式 `Reservation`。

自动触达通知链路存在 Touch 时能够完成归因；客户从首页权益或其他 H5 入口预约时，订单无法稳定归因到该 H5 行为。

#### 断点五：销售出库工作台只展示商品销售扣减

`SalesOutboundTab` 固定查询 `movementType=sale_out`，不展示项目服务产生的 `service_consume`。

因此验收必须分别查看：

- 商品订单：销售出库工作台。
- 项目订单：服务消耗与 BOM 页面。

不能只看一个库存页得出“所有库存扣减已完成”。

#### 断点六：缺少一键集成验收入口

现有测试分散在 marketing、marketing-pages、customer-app、orders、Ami Glow H5 和管理端组件中，没有统一测试数据、跨端 E2E、数据库审计脚本和验收报告。

## 四、目标业务场景

### 场景 A：推荐活动 → 公开营销页 → 留资 → 商品订单 → 销售出库

1. 门店存在有效预测批次和推荐实例。
2. 运营人员采纳推荐并发布活动页。
3. `MarketingPage` 保存推荐、采纳和活动来源。
4. 客户打开 Marketing H5，产生 view 和 click_cta。
5. 客户提交手机号和购买咨询，产生 lead。
6. 手机号匹配同门店客户。
7. 门店为该客户创建已支付商品订单，并带入活动权益。
8. 订单创建 `OrderItem`、`PaymentRecord`。
9. 订单创建 `sale_out`，库存前后数量正确。
10. 订单创建 `MarketingPageAttribution`。
11. `MarketingEffectFact` 产生 conversion 和 revenue。
12. 数据复盘显示活动收入，销售出库工作台显示订单出库。

### 场景 B：推荐自动触达 → H5 通知 → 预约 → 项目订单 → BOM 消耗

1. 运营人员采纳推荐并启用自动触达。
2. Worker 创建 execution、delivery job、touch 和站内通知。
3. Ami Glow H5 客户看到通知。
4. 客户打开通知，通知和 touch 状态更新。
5. 客户领取权益或进入预约页。
6. 客户提交正式预约，创建 `Reservation`。
7. 门店接待后使用预约创建已支付项目订单。
8. 订单保留 `sourceReservationId` 和营销来源。
9. 项目订单按 BOM 创建 `service_consume`。
10. 自动触达 Touch 被标记为 converted。
11. 创建 `MarketingAttribution` 和效果事实。
12. 数据复盘显示转化收入，服务消耗页显示真实耗材扣减。

### 场景 C：退款反向闭环

1. 对场景 A 的商品订单发起退货退款。
2. 创建退款单和库存回补流水。
3. 原 `sale_out` 保留，不直接删除或篡改。
4. 创建 refund 负向效果事实。
5. 营销净收入按退款金额减少。
6. 重复退款请求不能重复回补库存或重复冲减效果。

### 场景 D：失败和隔离

- 库存不足时订单事务整体失败，不创建半成品订单和归因。
- BOM 缺失时按现有业务规则明确失败或标记数据质量问题。
- 跨门店页面、客户、预约、订单和库存不得串读。
- 同一推荐、同一请求、同一订单重复提交不得重复创建活动、预约、扣库存或归因。
- 效果事实双写失败不能破坏客户交互，但必须进入可修复审计清单。

## 五、数据模型补齐计划

### 5.1 Reservation 结构化来源

调整 `Reservation`：

```prisma
source                   String?
channel                  String?
promotionId              Int?
campaignId               String?
recommendationInstanceId String?
adoptionId               Int?
sourceMarketingPageId    Int?
sourceContextJson        Json?
```

索引：

```prisma
@@index([storeId, source, createdAt])
@@index([promotionId])
@@index([recommendationInstanceId])
@@index([sourceMarketingPageId])
```

写入要求：

- H5 传入的营销身份必须先校验门店归属。
- `remark` 只保留用户可读信息，不再作为系统归因字段。
- 兼容旧记录，新增字段全部可空。

### 5.2 ProductOrder 关联预约

调整 `ProductOrder`：

```prisma
sourceReservationId Int?
sourceContextJson   Json?
```

并建立可选关系和索引：

```prisma
@@index([sourceReservationId])
```

规则：

- 同一预约允许存在补单或追加商品，因此不加唯一约束。
- 订单创建时校验预约、客户、门店一致。
- 订单序列化和退款详情返回 `sourceReservationId`。

### 5.3 归因候选扩展

扩展 `MarketingAttributionService.attributeOrder`，归因候选按以下顺序统一排序：

1. 订单显式关联的预约来源。
2. 最近的自动触达 Touch。
3. 最近的公开页 Lead。
4. 最近的 H5 权益领取或预约行为。

主归因继续采用 last-touch，但显式 `sourceReservationId` 优先于时间猜测。

效果事实维度至少包含：

- `recommendationInstanceId`。
- `adoptionId`。
- `activityId` 或 `strategyId`。
- `pageId`、`promotionId`。
- `customerId`、`reservationId`、`orderId`。
- `channel`。

## 十五、上线判定

### 可上线

同时满足：

- 场景 A、B、C、D 全部通过。
- 真实门店至少完成一笔商品订单和一笔项目订单闭环验收。
- 库存前后数量、订单金额、归因收入完全一致。
- 无跨门店和重复写入。
- migration、构建、自动化测试和浏览器回归全部通过。

### 不可上线

出现任一情况：

- H5 预约仍只靠 remark 关联营销来源。
- 订单无法定位预约。
- 商品订单未生成 sale_out。
- 项目订单未生成 service_consume。
- 订单有收入但无归因，或有归因但无真实订单。
- 退款重复回补库存或重复冲减效果。
- 页面使用 mock 或人工改库制造验收结果。

## 十六、风险与回滚

| 风险 | 控制方式 | 回滚方式 |
| --- | --- | --- |
| Reservation schema 变更影响预约主链 | 全部新增可空字段，先双写后切读 | 前端停止传新字段，后端继续兼容旧请求 |
| 归因候选扩大造成抢归因 | 显式预约优先、唯一主归因、辅助事实分离 | 关闭 H5 reservation attribution flag |
| 库存重复扣减 | 订单和 movementType 幂等检查、事务内执行 | 使用正式退款/冲销流水，不删除原流水 |
| H5 身份串店 | 所有来源对象按 storeId 校验 | 拒绝请求并记录安全审计 |
| 集成测试污染真实数据 | 独立测试门店、唯一前缀、写入授权门禁 | 只生成清理预览，授权后执行清理 |
