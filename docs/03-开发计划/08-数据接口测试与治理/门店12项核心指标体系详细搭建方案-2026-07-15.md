# 门店 12 项核心指标体系详细搭建方案

> 日期：2026-07-15
> 实施分支：`codex/store-metrics-core`
> 基线：`codex/stable-business-2026-07-15`

## 1. 建设目标

建立门店唯一、可解释、可下钻的经营指标事实层。管理端店长工作台、指标页与 Ami Brain 使用相同指标 key、计算服务和定义版本，不再各自聚合收入或经营目标。

本期一次性交付 12 项指标：今日实收、今日经营收入、毛利率、新客首次到店率、首次到店成交率、30 天新客复购率、现场再预约率、预约爽约率、美容师工时利用率、单位服务工时产值、会员续费率、门店月度目标达成率。

## 2. 核心口径

| 指标 key | 业务口径 | 零分母 |
| --- | --- | --- |
| `store.paid_revenue.today` | 有效外部支付减成功退款 | `null` |
| `store.operating_revenue.today` | 已交付商品和已履约服务确认收入 | `null` |
| `store.gross_margin_rate.today` | 毛利额 / 经营收入 | `null` |
| `customer.first_visit_arrival_rate` | 首次有效预约新客实际到店数 / 首次有效预约新客数 | `null` |
| `customer.first_visit_conversion_rate` | 首次到店后 24 小时内成交新客数 / 首次到店新客数 | `null` |
| `customer.new_customer_30d_repurchase_rate` | 首购后 30 天内复购新客数 / 具备完整观察窗的新客数 | `null` |
| `reservation.checkout_rebooking_rate` | 服务完成当日现场再预约人数 / 完成服务人数 | `null` |
| `reservation.no_show_rate` | 爽约及临时取消预约数 / 应到预约数 | `null` |
| `staff.service_time_utilization_rate` | 合并重叠后的实际服务分钟 / 净可售排班分钟 | `null` |
| `staff.operating_revenue_per_service_hour` | 关联服务任务的经营收入 / 实际服务小时 | `null` |
| `member.renewal_rate` | 完成续卡会员资产数 / 进入续费窗口会员资产数 | `null` |
| `store.monthly_target_completion_rate` | 月累计经营收入 / 月经营收入目标 | `null` |

## 3. 数据事实设计

- 预约新增来源、来源预约、取消/爽约时间、原因和状态事件审计。
- 订单明细与次卡核销新增 `reservationId`、`serviceTaskId`，连接成交收入、预约与服务工时。
- 服务任务与预约建立一对一显式关系。
- 会员卡新增销售类型、原卡续费关系、激活与终止信息。
- `StoreMetricTarget` 是通用目标事实；旧 `BrainStoreOperatingTarget` 仅作为 migration 映射来源。
- `StoreMetricSnapshot` 保存版本化日快照、分子、分母、样本量和质量状态。
- 新业务写显式关系；历史数据只读推断并标记 `estimated`，不自动回写。

## 4. 后端与 API

`store-metrics` 模块统一提供总览、趋势、下钻、定义、质量和目标能力。财务三项指标直接调用 `FinanceMetricsService`；今日结果缓存 5 分钟，历史趋势读取快照。

接口：

- `GET /api/store-metrics/overview`
- `GET /api/store-metrics/trends`
- `GET /api/store-metrics/:metricKey/drilldown`
- `GET /api/store-metrics/definitions`
- `GET /api/store-metrics/quality`
- `GET/POST/PUT /api/store-metrics/targets`

统一响应包含 key、值、单位、分子、分母、样本数、目标、质量、定义版本、更新时间和下钻路径。权限使用 `core:store-metrics:*`，并以 JWT 门店列表限制访问范围。

## 5. 页面与经营闭环

- `/store-operations/metrics`：经营结果、客户预约漏斗、员工产能与会员健康三屏总览。
- `/store-operations/metrics/targets`：月度经营收入目标维护。
- 每项指标可查看分子/分母、样本和事实明细；质量标签区分完整、估算、部分缺失、不可计算、冻结。
- 异常指标输出预约确认、客户回访、排班调整、续费跟进等经营动作入口。
- 店长工作台收入卡片改为消费统一指标服务。

## 6. Ami Brain

Ami Brain 店长角色通过 `StoreMetricsService` 查询同一指标 key。回答必须返回门店、日期、时区、定义版本、质量、分子、分母、样本量和证据 key；历史估算必须显式说明，不得将估算结果表述为确定事实。

## 7. 质量状态

- `complete`：显式关系与事实完整。
- `estimated`：历史记录通过只读规则推断。
- `partial`：部分事实缺失，结果可用但需提示。
- `unavailable`：缺少目标、分母为零或事实不足，值为 `null`。
- `frozen`：日结后财务快照冻结。

## 8. 验收与发布边界

- Prisma 执行 format、validate、generate；migration 仅生成并做空库/静态验证。
- 后端覆盖 12 项定义、零分母、质量、门店隔离、目标和 Brain 同源回答。
- 前端覆盖总览、目标、下钻、空数据、权限和加载体验。
- 执行后端定向测试与 build、前端定向测试与 build、`git diff --check`。
- 本分支不执行远端 migration、历史回填、commit、push、PR 或发布。
