# 门店 12 项核心指标体系详细搭建方案

版本：v1.0  
日期：2026-07-15  
适用范围：Ami Core 管理端、`packages/server-v2`、Ami Brain、Ami Aura Lite 经营摘要  
文档类型：产品口径 + 数据架构 + 开发实施 + 验收方案  

## 1. 建设结论

本项目当前已经具备订单、支付、退款、预约、服务任务、会员卡、核销、排班、提成、日结和经营利润等事实数据，但门店指标仍分散在首页、财务中心、营销分析和 Ami Brain 中，尚未形成统一、可追溯、可下钻、可预警的门店经营指标体系。

本方案建设以下 12 项店长核心指标：

1. 今日实收。
2. 今日经营收入。
3. 毛利率。
4. 新客首次到店率。
5. 首次到店成交率。
6. 30 天新客复购率。
7. 现场再预约率。
8. 预约爽约率。
9. 美容师工时利用率。
10. 单位服务工时产值。
11. 会员续费率。
12. 门店月度目标达成率。

建设重点不是再做一套独立 BI 数据源，而是建立统一的指标计算层：

```text
业务事实表
  -> 指标计算服务
  -> 实时指标 / 日结冻结指标
  -> 店长看板、财务中心、Ami Brain、终端摘要
  -> 预警、下钻、任务闭环
```

最终交付标准：同一门店、同一时间范围、同一指标，在首页、财务中心、Ami Brain 和 API 中返回一致结果，并能说明公式、分子、分母、排除项、数据质量和业务行动。

## 2. 第一性原理与指标树

### 2.1 门店经营公式

```text
门店营收
= 有效客流
× 预约转化率
× 到店率
× 到店成交率
× 客单价
× 复购频次

门店利润
= 经营收入
- 耗材成本
- 商品成本
- 直接提成
- 固定经营费用

门店产能
= 可售工时
× 工时利用率
× 单位服务工时产值
```

### 2.2 12 项指标的层级

| 层级 | 经营问题 | 指标 |
| --- | --- | --- |
| 结果层 | 今天收到多少钱、真正赚了多少 | 今日实收、今日经营收入、毛利率、目标达成率 |
| 客户增长层 | 新客是否到店、成交并留下来 | 新客首次到店率、首次到店成交率、30 天新客复购率 |
| 预约过程层 | 到店后是否锁定下一次服务、预约是否浪费 | 现场再预约率、预约爽约率 |
| 产能层 | 美容师时间是否有效转化为收入 | 美容师工时利用率、单位服务工时产值 |
| 会员层 | 预收业务是否持续、客户是否续留 | 会员续费率 |

### 2.3 北极星指标建议

店长首页不再把“销售额”作为唯一北极星指标，建议使用：

```text
月度经营目标达成率
```

原因：它可以同时承接经营收入、毛利、新客、预约等目标，并通过其余 11 项指标解释目标差距。

### 2.4 成熟产品对标依据

本方案参考成熟美业数字化产品的官方经营分析能力，但不直接复制其报表结构：

| 产品 | 官方能力重点 | 本方案吸收内容 |
| --- | --- | --- |
| Zenoti | 再预约、客户留存、会员渗透、取消/爽约、按门店/员工/服务拆解、同行基准 | 建立预约漏斗、客户 cohort、员工维度和门店基线 |
| Vagaro | 客户留存、新客/回头客、员工再预约、预约来源、预约容量、取消与爽约、员工销售 | 建立现场再预约、爽约、来源、工时利用率和员工下钻 |
| Mindbody | 客户留存、出勤分析、未来产能利用率、员工表现 | 建立完整观察窗口、净可售工时和产能趋势 |

官方参考：

- [Zenoti Business Intelligence](https://www.zenoti.com/platform/business-intelligence-that-helps-you-make-better-decisions)
- [Vagaro Reports and Dashboard](https://support.vagaro.com/hc/en-us/categories/115000066113-Reports-and-Dashboard)
- [Mindbody Retention Report](https://support.mindbodyonline.com/s/article/203256863-Retention-report?language=en_US)
- [Mindbody Utilization Report](https://support.mindbodyonline.com/s/article/Utilization-Report)
- [Mindbody Staff Performance Report](https://support.mindbodyonline.com/s/article/203256503-Staff-Performance-report?language=en_US)

对标结论：成熟产品的关键不是指标数量，而是把结果指标与客户留存、预约履约、员工产能和会员续费连接起来，并允许按门店、员工、项目和来源下钻。本方案因此优先建设可解释、可行动的 12 项指标，不建设只用于展示的大屏指标集合。

## 3. 建设目标、对象与排除项

### 3.1 建设目标

- 统一 12 项指标口径和唯一 key。
- 建立实时查询、趋势查询、明细下钻和目标配置接口。
- 建立实时态、估算态、日结冻结态和不可用态。
- 将现有首页收入计算切换到统一指标服务。
- 支持按门店、员工、项目、客户来源和时间维度拆解。
- 支持 Ami Brain 按同一口径问数。
- 支持指标异常生成经营建议和待办任务。

### 3.2 工程对象

| 层次 | 对象 |
| --- | --- |
| 页面 | 店长工作台、门店经营指标页、指标下钻抽屉、目标设置页 |
| API | 指标总览、趋势、下钻、目标、数据质量、指标定义 |
| Service | `StoreMetricsService`、指标计算器、快照服务、目标服务、数据质量服务 |
| Schema | 预约状态事件、预约来源关系、交易履约关系、续卡关系、指标目标、指标快照 |
| 数据 | 订单、支付、退款、预约、服务任务、核销、排班、会员卡、日结、成本、目标 |
| 治理 | 指标定义版本、公式版本、权限、数据血缘、质量状态、发布门禁 |

### 3.3 本方案排除项

- 不建设独立数据仓库。
- 不引入第二套客户、订单、财务或预约事实源。
- 不替代财务总账、税务和会计软件。
- 不将预测分数作为财务或经营结果事实。
- 不使用历史 mock 补齐真实门店指标。
- 不在本轮方案阶段执行 Prisma migration、回填或真实写库。

## 4. 当前数据基础与缺口

### 4.1 已有事实基础

| 业务域 | 现有模型 | 可支撑能力 |
| --- | --- | --- |
| 收款 | `PaymentRecord` | 支付金额、支付方式、支付时间、支付状态 |
| 退款 | `RefundRecord` | 退款金额、退款时间、退款状态 |
| 订单 | `ProductOrder`、`OrderItem` | 实收、折扣、项目/商品结构、美容师归属 |
| 日结 | `DailySettlement`、`DailySettlementSnapshot` | 已确认收入、退款、成本、毛利、客数、订单数 |
| 会员履约 | `CustomerCard`、`CardUsageRecord` | 卡销售、剩余次数、核销确认收入 |
| 预约 | `Reservation` | 预约日期、时间、员工、状态、到店时间 |
| 服务 | `ServiceTask` | 预约时间、服务时长、开始和完成时间 |
| 排班 | `Schedule`、`BeauticianAvailability`、`BeauticianTimeOff` | 排班时段、可用时段、请假时段 |
| 客户 | `Customer` | 来源、累计消费、到店次数、最后到店时间 |
| 目标 | `BrainStoreOperatingTarget` | 收入、毛利、预约和新客目标 |
| 语义治理 | `BusinessDefinition`、`BusinessDefinitionVersion` | 指标定义、版本、发布状态和运行时查询 |

### 4.2 关键缺口

| 缺口 | 影响指标 | 处理原则 |
| --- | --- | --- |
| 订单/订单明细未稳定关联预约和服务任务 | 首次成交率、单位工时产值 | 增加显式业务关系，不长期依赖时间窗口猜测 |
| 预约缺少来源、取消时间、爽约时间和状态事件 | 现场再预约率、爽约率 | 增加状态事件表和来源字段 |
| 续卡未关联原卡 | 会员续费率 | 增加 `renewedFromCustomerCardId` 与销售类型 |
| 目标模型仅覆盖少数目标且位于 Brain 域 | 目标达成率 | 升级为通用门店指标目标模型 |
| 首页收入使用订单创建时间与卡创建价格汇总 | 今日实收、经营收入 | 切换为支付时间、退款时间和履约确认时间 |
| 历史数据缺少完整事件时间 | 预约、续卡、现场再预约历史趋势 | 明确数据生效日，推断结果只能标记 estimated |

## 5. 统一指标定义规范

每个指标必须注册以下内容：

```json
{
  "key": "store.paid_revenue.today",
  "name": "今日实收",
  "domain": "store_operation",
  "version": 1,
  "valueType": "currency",
  "aggregation": "sum",
  "numerator": "有效支付金额 - 有效退款金额",
  "denominator": null,
  "timePolicy": "Asia/Shanghai by paidAt/refundedAt",
  "dimensions": ["store", "paymentMethod", "orderKind"],
  "permissions": ["core:finance:view"],
  "sourceModels": ["PaymentRecord", "RefundRecord"],
  "exclusions": ["failed payments", "cancelled refunds"],
  "qualityPolicy": "payment and refund event time required",
  "drilldownPath": "/finance/reconciliation"
}
```

### 5.1 通用时间规则

- 业务时区固定为门店时区，当前默认 `Asia/Shanghai`。
- “今日”使用 `[00:00:00, 次日 00:00:00)` 半开区间。
- 支付按 `PaymentRecord.paidAt` 归集。
- 退款按 `RefundRecord.refundedAt` 归集。
- 履约按 `OrderItem.recognizedAt` 或 `CardUsageRecord.verifiedAt` 归集。
- 到店按 `Reservation.checkedInAt` 归集。
- 服务时长优先使用 `ServiceTask.startedAt/completedAt`，缺失时才使用计划时长并标记 estimated。
- 月度目标按门店自然月归集。

### 5.2 通用状态规则

- `complete`：数据完整，可用于经营判断。
- `estimated`：使用兜底时间、推断关系或估算成本。
- `partial`：部分来源缺失，结果只能看趋势。
- `unavailable`：分子或分母无法可信计算。
- `frozen`：来自已确认日结/月结快照，不再被普通业务修改覆盖。

### 5.3 比率指标的零分母规则

- 分母为 0 时返回 `value = null`，不得返回 0%。
- 同时返回 `status = unavailable` 和明确原因。
- 页面显示“暂无符合口径的样本”，不得显示“表现为 0”。

## 6. 12 项指标详细口径

### 6.1 今日实收

指标 key：`store.paid_revenue.today`

### 业务定义

当前门店今日实际收到的外部支付资金，扣除今日已成功退款金额。它回答“今天现金真正进来了多少”。

### 公式

```text
今日实收
= sum(有效支付记录.amount，paidAt 在今日)
- sum(成功退款记录.amount，refundedAt 在今日)
```

### 纳入范围

- 现金、微信、支付宝、银行卡和其他外部支付。
- 项目、商品、开卡、续卡、充值等真实支付。
- 当日完成的成功退款作为负数。

### 排除范围

- 会员余额抵扣和次卡核销。
- 支付失败、取消、待支付记录。
- 仅创建订单但未成功支付的数据。
- 赠送金、赠送次数。

### 数据源与质量

| 项目 | 内容 |
| --- | --- |
| 主事实 | `PaymentRecord`、`RefundRecord` |
| 时间字段 | `paidAt`、`refundedAt` |
| 已有程度 | 可直接计算 |
| 质量门禁 | 有效支付不得缺 `paidAt`；成功退款不得缺 `refundedAt` |
| 下钻 | 支付方式、订单类型、收银员、订单号、退款号 |

### 6.2 今日经营收入

指标 key：`store.operating_revenue.today`

### 业务定义

今日已经完成商品交付或服务履约、可以归属于当日经营成果的收入。它回答“今天真正做了多少生意”，不把尚未履约的办卡和充值全部当作收入。

### 公式

```text
今日经营收入
= 单次项目/商品订单明细确认收入
+ 次卡核销确认收入
+ 储值本金对应的已履约收入
- 经营收入对应退款分摊
```

### 数据源与质量

| 项目 | 内容 |
| --- | --- |
| 主事实 | `OrderItem`、`CardUsageRecord`、`RefundItem`、余额交易 |
| 时间字段 | `recognizedAt`、`verifiedAt`、`refundedAt` |
| 已有程度 | `finance-metrics` 已具备主要计算能力 |
| 质量门禁 | 订单明细收入必须完成折扣分摊；退款必须按明细分摊 |
| 下钻 | 单次服务、商品销售、次卡核销、余额履约、退款 |

### 6.3 毛利率

指标 key：`store.gross_margin_rate.today`

### 业务定义

今日经营收入扣除直接耗材、商品成本和直接服务提成后的利润比例。

### 公式

```text
毛利
= 今日经营收入
- 实际耗材成本
- 商品成本
- 直接服务提成

毛利率
= 毛利 / 今日经营收入
```

### 质量规则

- 缺项目 BOM、实际消耗、商品批次成本或提成时，不得把缺失成本按 0 处理。
- 任一重大成本缺失时标记 `partial` 或 `estimated`。
- 日结确认后切换为 `DailySettlementSnapshot` 冻结值。

### 6.4 新客首次到店率

指标 key：`customer.first_visit_arrival_rate`

### 业务定义

统计期内首次预约的新客中，实际完成首次到店的客户比例。

### 新客定义

满足以下全部条件：

- 在当前门店此前没有有效到店记录。
- 在当前门店此前没有已履约订单或次卡核销。
- 本统计期存在首次有效预约。

### 公式

```text
新客首次到店率
= 首次预约已到店的新客数
/ 首次有效预约新客数
```

### 分母排除

- 门店在预约发生前主动取消的预约。
- 测试客户、已删除客户。
- 重复预约，只保留客户在门店的第一条有效预约。

### 数据要求

- `Reservation.checkedInAt` 或明确的 `checked_in/completed` 状态。
- 客户历史订单、核销和预约可用于判断“首次”。
- 建议保存客户门店级 `firstReservationAt`、`firstCheckedInAt` 快照，减少重复全表扫描。

### 6.5 首次到店成交率

指标 key：`customer.first_visit_conversion_rate`

### 业务定义

首次到店的新客中，在首次到店服务闭环内产生有效经营收入的客户比例。

### 公式

```text
首次到店成交率
= 首次到店且产生有效经营收入的新客数
/ 首次到店新客数
```

### 成交窗口

- 起点：首次 `checkedInAt`。
- 终点：关联服务完成后 24 小时，或门店当日营业结束时间，以较晚者为准。
- 必须优先使用显式 `reservationId/serviceTaskId` 关系。

### 当前阻断

订单和订单明细尚未稳定关联预约/服务任务。开发时必须为 `OrderItem`、`CardUsageRecord` 增加可追溯的 `reservationId`、`serviceTaskId`，不得长期使用“同一客户同一天订单”作为正式口径。

### 6.6 30 天新客复购率

指标 key：`customer.new_customer_30d_repurchase_rate`

### 业务定义

首次产生有效经营收入的新客中，在首次成交后的 30 个自然日内再次产生有效经营收入的客户比例。

### 公式

```text
30 天新客复购率
= 首次成交后 30 天内再次成交的新客数
/ 已完整观察 30 天的新客首购数
```

### 队列规则

- 按首次成交日期形成 cohort。
- 今日计算时，只纳入首次成交已超过 30 天的客户，避免观察窗口不完整。
- 第二次成交可以是项目、商品、核销或会员履约，但退款后净经营收入必须大于 0。
- 同一结账组内的多条订单明细只算一次成交。

### 展示要求

- 展示当前完整 cohort 的 30 天复购率。
- 同时展示上一个 cohort、环比变化和样本数。
- 不允许把未满 30 天客户放入分母。

### 6.7 现场再预约率

指标 key：`reservation.checkout_rebooking_rate`

### 业务定义

完成本次服务的客户中，在离店前或当日营业结束前成功预约下一次服务的比例。

### 公式

```text
现场再预约率
= 完成服务且已生成下一次预约的客户数
/ 完成服务客户数
```

### 下一次预约要求

- 下一次预约日期必须晚于本次服务日期。
- 预约创建时间在本次完成服务后至当日营业结束前。
- 预约来源为 `checkout_rebook`，或显式关联 `sourceReservationId`。
- 取消的下一次预约仍记录当时再预约行为，但需在下钻中显示后续取消状态。

### 当前阻断

`Reservation` 缺少预约来源和来源预约关系。建议新增：

```text
bookingSource: online / frontdesk / checkout_rebook / marketing / kiosk / miniapp / manual
sourceReservationId: 上一次预约 ID
createdById: 创建人
```

### 6.8 预约爽约率

指标 key：`reservation.no_show_rate`

### 业务定义

应到店预约中，客户没有到店且未在有效取消截止时间前取消的比例。

### 公式

```text
预约爽约率
= no_show 预约数
/ 应到店预约数
```

### 分母定义

应到店预约包括：

- 已到店。
- 已完成。
- 爽约。
- 截止时间后取消。

不包括：

- 门店主动取消。
- 截止时间前客户正常取消。
- 重复预约和测试预约。

### Schema 要求

建议增加 `ReservationStatusEvent`：

```text
reservationId
fromStatus
toStatus
reasonCode
actorType
actorId
occurredAt
```

并增加 `cancelledAt`、`cancelledBy`、`cancelReasonCode`、`noShowAt`。只有保留状态变化时间，才能正确判断“正常取消”与“临时取消/爽约损失”。

### 6.9 美容师工时利用率

指标 key：`staff.service_time_utilization_rate`

### 业务定义

美容师实际服务客户的有效工时占净可售工时的比例。

### 公式

```text
净可售工时
= 已发布排班工时
- 已批准请假工时
- 不可售锁定时段

实际服务工时
= sum(completedAt - startedAt)

工时利用率
= 实际服务工时 / 净可售工时
```

### 规则

- 使用实际开始/完成时间，不优先使用项目标准时长。
- 缺实际时间时使用 `ServiceTask.duration`，并标记 estimated。
- 重叠服务时间必须合并区间，不能重复累计。
- 超出排班的服务工时单独显示为“超排班工时”，不直接放大利用率分子。
- 门店利用率按总实际服务分钟/总净可售分钟计算，不做员工利用率简单平均。

### 下钻维度

- 美容师。
- 日期、星期和时段。
- 项目。
- 黄金时段/普通时段。

### 6.10 单位服务工时产值

指标 key：`staff.operating_revenue_per_service_hour`

### 业务定义

美容师每个实际服务小时产生的经营收入，用于区分“长时间忙碌”和“高价值服务产出”。

### 公式

```text
单位服务工时产值
= 归属美容师的经营收入
/ 实际服务工时小时数
```

### 收入归属规则

- 单次服务订单按 `OrderItem.beauticianId` 归属。
- 次卡核销按 `CardUsageRecord.beauticianId` 归属。
- 多美容师共同服务时按明细分摊；没有分摊关系时标记 partial。
- 商品销售、纯充值和纯开卡收入默认不计入“服务工时产值”。
- 另行提供“综合人效产值”，再纳入商品和卡项销售。

### 当前阻断

需要建立收入明细与 `ServiceTask` 的显式关系，才能保证收入和服务时长属于同一次服务。

### 6.11 会员续费率

指标 key：`member.renewal_rate`

### 业务定义

统计期内进入续费窗口的会员卡/疗程卡中，在续费窗口内完成续卡的比例。

### 公式

```text
会员续费率
= 完成续卡的到期会员资产数
/ 进入续费窗口的会员资产数
```

### 续费窗口

- 默认：到期前 30 天至到期后 30 天。
- 次数型卡：剩余次数低于总次数 20% 时也进入续费窗口。
- 储值会员：余额低于配置阈值时进入续费窗口。

### 续卡判断

- 新卡显式关联 `renewedFromCustomerCardId`。
- 同卡续期保留续费交易和续费前后资产快照。
- 单纯购买不同类型体验卡不计为原卡续费。

### 当前阻断

`CustomerCard` 没有原卡续费关系。建议增加：

```text
saleType: new / renewal / upgrade / replacement
renewedFromCustomerCardId
activatedAt
terminatedAt
terminationReason
```

### 6.12 门店月度目标达成率

指标 key：`store.monthly_target_completion_rate`

### 业务定义

当前门店本月核心经营目标的实际完成程度，并结合时间进度判断是否落后。

### 主展示公式

```text
收入目标达成率
= 本月累计经营收入 / 本月经营收入目标

时间进度
= 已经过营业日 / 本月计划营业日

预测达成率
= 按最近 7 个有效营业日趋势预测的月末经营收入
/ 本月经营收入目标
```

### 综合目标分

综合目标分用于二级分析，不替代主展示的收入目标达成率：

```text
综合目标分
= 收入达成率 × 40%
+ 毛利达成率 × 25%
+ 新客达成率 × 15%
+ 预约达成率 × 10%
+ 会员续费达成率 × 10%
```

权重必须可配置并保留版本，不得写死在前端。

### 目标模型改造

现有 `BrainStoreOperatingTarget` 只覆盖收入、毛利、预约和新客。建议升级为通用模型：

```prisma
model StoreMetricTarget {
  id              Int      @id @default(autoincrement())
  storeId         Int
  metricKey       String
  periodType      String
  periodStart     DateTime
  periodEnd       DateTime
  targetValue     Decimal
  warningValue    Decimal?
  criticalValue   Decimal?
  weight          Decimal?
  definitionVersion Int
  status          String   @default("active")
  createdById     Int?
  approvedById    Int?
  approvedAt      DateTime?
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  @@unique([storeId, metricKey, periodType, periodStart])
  @@index([storeId, periodStart, periodEnd])
}
```

迁移时将 `BrainStoreOperatingTarget` 的有效数据映射到通用目标表；Ami Brain 改为读取通用目标，不再拥有平行目标事实源。

## 7. 数据模型建设方案

### 7.1 新增关系和事件字段

| 模型 | 建议新增 | 目的 |
| --- | --- | --- |
| `Reservation` | `bookingSource`、`sourceReservationId`、`createdById` | 识别现场再预约和预约渠道 |
| `Reservation` | `cancelledAt`、`cancelReasonCode`、`cancelledByType`、`noShowAt` | 计算取消和爽约 |
| `ReservationStatusEvent` | 状态前后、原因、操作者、发生时间 | 保留状态审计链 |
| `OrderItem` | `reservationId`、`serviceTaskId` | 关联成交、服务和收入 |
| `CardUsageRecord` | `reservationId`、`serviceTaskId` | 关联核销收入与服务工时 |
| `CustomerCard` | `saleType`、`renewedFromCustomerCardId` | 计算会员续费 |
| `StoreMetricTarget` | 通用指标目标 | 统一门店目标 |
| `StoreMetricSnapshot` | 指标日快照 | 趋势、冻结、快速查询和审计 |

### 7.2 指标快照模型

```prisma
model StoreMetricSnapshot {
  id                Int      @id @default(autoincrement())
  storeId           Int
  metricKey         String
  metricDate        DateTime
  granularity       String   @default("day")
  value             Decimal?
  numerator         Decimal?
  denominator       Decimal?
  sampleCount       Int?
  qualityStatus     String
  qualityReasons    Json?
  dimensions        Json?
  definitionVersion Int
  calculationMode   String   @default("live")
  sourceVersion     String?
  generatedAt       DateTime @default(now())
  frozenAt          DateTime?

  @@unique([storeId, metricKey, metricDate, granularity, definitionVersion])
  @@index([storeId, metricDate])
  @@index([metricKey, metricDate])
}
```

### 7.3 为什么需要快照

- 解决复杂 cohort、工时区间和成本计算造成的查询延迟。
- 保存当时的公式版本、分子、分母和质量状态。
- 日结后冻结财务类指标，避免历史页面结果随业务数据修改而漂移。
- 支持趋势、目标比较和同比环比。
- 支持 Ami Brain 返回可追溯证据。

快照不是新的业务事实源。订单、支付、预约等原始表仍是事实源，快照只是经过版本化公式计算的指标结果。

### 8.2 复用边界

- 今日经营收入、毛利和成本质量必须复用 `FinanceMetricsService`，不得复制公式。
- 目标达成读取 `StoreMetricTarget`。
- 客户队列统一由 customer cohort calculator 计算。
- 排班和服务区间计算封装为共享 interval utility。
- Ami Brain 通过 resolver 或统一 API 调用 store metrics，不另写 SQL 口径。
- `DashboardService` 改为消费 `StoreMetricsService`，不再自行聚合今日收入。

### 8.3 计算模式

| 模式 | 使用场景 | 更新策略 |
| --- | --- | --- |
| `live` | 今日经营、店长实时看板 | 请求时计算 + 5 分钟缓存 |
| `snapshot` | 历史趋势、同比环比 | 每日定时生成 |
| `confirmed` | 财务日结/月结确认 | 日结确认后冻结 |
| `estimated` | 历史关联缺失或使用计划时长 | 明确质量标签，不覆盖正式值 |

### 8.4 刷新机制

- 支付成功、退款成功、核销完成、服务完成、预约状态变化、续卡成功后，失效对应门店当日缓存。
- 每 5 分钟刷新当日 live 快照。
- 每日营业结束后生成日快照。
- 日结确认后将财务相关快照切换为 confirmed/frozen。
- 每月 1 日生成上月最终目标达成快照。
- 30 天复购率每天滚动重算最近 60 天 cohort。

## 11. 预警与行动闭环

### 11.1 阈值来源

阈值优先级：

1. 门店批准目标。
2. 最近 8 周同星期/同期基线。
3. 系统初始默认阈值。

系统初始阈值只用于冷启动，不代表行业标准。门店积累 4 周有效数据后，自动生成门店基线并提示店长确认。

### 11.2 冷启动默认阈值

| 指标 | 预警 | 严重 |
| --- | --- | --- |
| 毛利率 | 低于目标 5 个百分点 | 低于目标 10 个百分点 |
| 新客首次到店率 | 低于 65% | 低于 50% |
| 首次到店成交率 | 低于 55% | 低于 40% |
| 30 天新客复购率 | 低于 30% | 低于 20% |
| 现场再预约率 | 低于 45% | 低于 30% |
| 预约爽约率 | 高于 8% | 高于 12% |
| 美容师工时利用率 | 低于 60% | 低于 45% |
| 会员续费率 | 低于 55% | 低于 40% |
| 月度目标达成率 | 落后时间进度 8 个百分点 | 落后 15 个百分点 |

### 11.3 经营动作映射

| 异常 | 建议动作 |
| --- | --- |
| 新客首次到店率低 | 检查确认时效、渠道质量，生成预约确认任务 |
| 首次到店成交率低 | 下钻项目、顾问、美容师，复盘接待和方案匹配 |
| 30 天复购率低 | 生成首购后 7/21 天回访和复购任务 |
| 现场再预约率低 | 在收银和服务完成流程增加下一次预约提示 |
| 爽约率高 | 启用预约确认、到店提醒和高风险客户定金策略 |
| 工时利用率低 | 调整排班、开放候补名单、执行空档填充 |
| 单位工时产值低 | 调整项目组合、服务时长和主推项目 |
| 会员续费率低 | 生成到期会员续费清单和负责人任务 |
| 目标达成落后 | 分解为客流、成交、客单、复购和产能差距 |

## 12. Ami Brain 与指标治理

### 12.1 统一原则

- 12 项指标必须注册为已发布 `BusinessDefinition(kind=metric)`。
- 指标定义、公式、数据源和权限与 store metrics 服务保持同版本。
- Ami Brain 查询通过已发布 runtime query/resolver 执行。
- 看板 API 和 Brain 问数必须共享指标 key。
- Brain 回答必须返回时间范围、门店范围、口径、质量和证据。

### 12.2 典型问法

- “今天实收和经营收入为什么不一样？”
- “新客到店率低在哪个渠道？”
- “哪个美容师再预约率最低？”
- “本月目标还差多少，按当前趋势能完成吗？”
- “会员续费下降是哪些卡项造成的？”

### 12.3 版本治理

本方案新增门店指标核心模块。正式实施前必须更新：

`docs/03-开发计划/01-AI智能体与问数能力/Agent与核心模块版本决策记录.md`

记录内容包括模块定位、主入口、指标 key、替代对象、兼容边界、发布门禁和旧首页指标处置。

## 13. 数据回填与生效策略

### 13.1 可回填指标

| 指标 | 回填范围 | 可信度 |
| --- | --- | --- |
| 今日实收/历史实收 | 支付、退款事件完整期间 | complete |
| 经营收入、毛利率 | 收入确认和成本链完整期间 | complete/partial |
| 新客首次到店率 | 预约状态和到店时间完整期间 | complete/partial |
| 30 天新客复购率 | 客户、支付、核销关联完整期间 | complete |
| 工时利用率 | 服务开始/完成时间完整期间 | complete/estimated |
| 单位工时产值 | 服务与收入归属完整期间 | complete/partial |
| 目标达成率 | 已有目标有效期间 | complete |

### 13.2 不强行回填指标

- 历史现场再预约率：缺预约来源关系时不做正式回填。
- 历史爽约率：缺取消时间和状态事件时只输出 estimated。
- 历史会员续费率：缺原卡续卡关系时只做候选推断，不进入正式经营看板。
- 历史首次到店成交率：订单与预约无法唯一关联时不做 complete 回填。

### 13.3 回填门禁

- 回填脚本默认 dry-run。
- 输出候选数、歧义数、无法关联数和样例。
- 低置信度关联不得写入正式关系。
- 真实写库需用户明确授权。
- 回填后必须抽样核对业务记录和指标结果。

## 15. 测试与验证方案

### 15.1 单元测试

每项指标至少覆盖：

- 正常样本。
- 分母为 0。
- 跨日和月末。
- 退款、取消、爽约。
- 重复订单或重复预约。
- 缺失时间和缺失关联。
- 多门店隔离。
- 员工重叠服务区间。
- 日结冻结后业务数据变化。
- 指标定义版本升级。

### 15.2 集成测试

- 支付成功后今日实收刷新。
- 退款成功后按退款时间扣减。
- 服务完成后经营收入、工时和单位工时产值刷新。
- 新客到店并成交后漏斗分子变化。
- 完成服务并现场创建下一预约后再预约率变化。
- 标记爽约后爽约率变化。
- 续卡后会员续费率变化。
- 配置目标后目标达成率变化。
- 店长不能读取未授权门店。

### 15.3 手工复算

每项指标至少抽取：

- 1 个正常门店日。
- 1 个退款/取消异常日。
- 1 个无样本日。
- 1 个历史估算日。

人工列出分子、分母和明细 ID，与 API 结果逐项勾稽。

### 15.4 性能门禁

- 店长总览缓存命中 p95 小于 800ms。
- 总览首次计算 p95 小于 2s。
- 单项下钻首屏 p95 小于 2s。
- 单次总览数据库查询数设置上限，禁止 12 项指标各自重复扫描同一大表。
- 趋势查询优先读取快照，不实时扫描全量历史事实。

## 16. 五层验收标准

### 16.1 代码层

- 12 项指标都有唯一 key、定义版本和计算器。
- 不复制财务中心已有收入/成本公式。
- 时间、状态、门店隔离和零分母规则有共享实现。
- 单测、集成测试和架构测试通过。

### 16.2 接口层

- 总览、趋势、下钻、目标、定义和质量接口可用。
- API 返回分子、分母、样本数、质量、版本和更新时间。
- 错误响应符合统一 `{ message, code, status, details }`。

### 16.3 页面层

- 12 项指标按结果、客户漏斗、产能和会员分组展示。
- 指标可下钻到具体客户、预约、服务、订单和卡项。
- 不可用和估算数据有明确标签。
- 页面无长时间空白等待和重复请求。

### 16.4 数据层

- 财务指标与日结/经营利润口径一致。
- 预约和成交关系可追溯。
- 续卡关系可追溯。
- 历史推断数据与正式数据严格区分。
- 随机抽样复算与 API 一致。

### 16.5 交付层

- 指标字典、API 契约、数据字典、迁移说明和验收记录同步更新。
- 12 项指标发布到业务定义目录并通过治理门禁。
- 旧首页指标完成替代或明确兼容期限。
- 未获授权前不执行真实写库、发布、commit、push 或 PR。

## 17. 风险与控制措施

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| 同名收入多口径 | 店长看到不同金额 | 同时展示实收和经营收入，并提供口径解释 |
| 历史事件字段缺失 | 历史趋势失真 | 设置生效日，历史只标 estimated |
| 订单与服务关联不完整 | 成交率和工时产值失真 | 新链路强制显式关联，旧数据不强行确认 |
| 小样本比率波动 | 错误经营判断 | 展示样本数，低样本不触发强预警 |
| 指标实时扫描过慢 | 首页加载变慢 | 事件失效缓存 + 日快照 + 批量聚合 |
| 目标权重被随意修改 | 目标失去连续性 | 目标版本、审批和生效周期 |
| Brain 和页面口径分叉 | 问数与看板冲突 | 统一 metric key 和 resolver，架构测试阻断重复实现 |

## 18. 最终推荐实施顺序

按数据可信度和经营价值排序：

1. 先统一今日实收、经营收入、毛利率和目标达成率。
2. 补预约状态事件和订单/服务关联。
3. 上线新客首次到店率、首次成交率和爽约率。
4. 补现场再预约来源并上线再预约率。
5. 上线 30 天新客复购 cohort。
6. 完成员工工时利用率和单位工时产值。
7. 补续卡关系并上线会员续费率。
8. 将 12 项指标接入 Ami Brain、终端经营摘要和异常任务闭环。

不建议先做完整大屏。正确顺序是：先冻结口径，补齐事实关系，验证指标，再建设页面。否则会得到一个视觉完整但数据无法解释的看板。

## 19. 完成定义

本项目只有同时满足以下条件才算完成：

- 12 项指标在真实门店范围内均能返回结果或明确不可用原因。
- 同一指标在页面、API、财务中心和 Ami Brain 中口径一致。
- 每个指标能够下钻到业务事实并人工复算。
- 历史估算与正式事实明确区分。
- 指标异常能够跳转到可执行的经营动作。
- 目标、趋势、预警、权限、数据质量和版本治理均已落地。
