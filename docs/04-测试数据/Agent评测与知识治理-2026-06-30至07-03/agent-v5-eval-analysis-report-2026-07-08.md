## 3. 产品结论

V5 的方向是对的，但当前还不是可直接替代 V4 的正式主入口。

主要原因不是 V5 架构失败，而是 P0 只实现了少量全业务 adapter，大量题目仍回落到 V3 受控 Text-to-SQL。V3 对口语化、跨域、缺时间范围、前台/美容师/财务细分问题的安全 SQL 计划能力不足，导致 217 次“无法生成安全查询计划”。

因此当前判断：

- V5 可以作为“新架构验证版”和“全业务统一入口候选”。
- V5 不应立刻替换 V4。
- V5 应继续以 V4 生命周期能力为第一条稳定业务链路，同时补齐前台、收银、预约、财务、员工、库存、供应链等 domain adapter。
- V5 的优先目标不是继续堆 LLM，而是减少对 V3 SQL 兜底的依赖。

这说明 V5 的最大问题是“路由后没有命中专用 adapter，只能交给 V3 问数”，而不是权限阻断或版本边界混乱。

## 7. 典型失败样本

前台接待：

- `帮我查一下张雯，她上次来是什么时候`
- `这个客人有没有办过卡，还有多少次`
- `帮我看一下这个客人的消费记录`

这些问题应该走“前台客户上下文 adapter”，不应该直接让 V3 生成 SQL。

美容师服务：

- `我今天有几个客人，分别几点`
- `下午那个客人叫什么名字，有什么要注意的`
- `我今天有没有空档，几点到几点`

这些问题应该走“美容师日程 + 客户注意事项 adapter”。

财务风控：

- `今天收了多少钱`
- `今天现金、微信、支付宝各收了多少`
- `帮我核对一下今天的收款和系统记录是否一致`

这些问题应该走“收入对账 adapter”，而不是泛化 SQL。

边界多轮：

- `帮我看看`
- `查一下`
- `这个客人`
- `钱的事情`

这些问题应该返回澄清卡片，而不是进入 V3 失败。

## 8. 改进建议

### P0：马上提分，目标从 61.1% 提升到 75%+

1. 增加 V5 澄清策略。
   - 对 `帮我看看`、`查一下`、`这个客人`、`钱的事情` 这类意图不足问题，直接返回 `clarification_card`。
   - 不再回落 V3。
   - 预计修复边界/多轮 20 条以上。

2. 新增前台客户上下文 adapter。
   - 覆盖客户搜索、手机号尾号、上次到店、办卡余量、消费记录、过敏/注意事项。
   - 复用 `Customer`、`CustomerCard`、`CardUsageRecord`、`ProductOrder`、`Reservation`。
   - 预计修复前台接待 25-35 条。

3. 新增收银与核销 adapter。
   - 覆盖今日收款、支付方式、核销、会员卡消耗、充值、退款、折扣。
   - 复用订单/支付/卡使用记录，不走泛 SQL。
   - 预计修复收银与核销、收入与对账 25-35 条。

4. 新增美容师日程 adapter。
   - 覆盖“我今天几个客人”“几点来”“有没有空档”“下一个客人注意事项”。
   - 复用 `Reservation`、`Customer`、`Project`、服务记录。
   - 预计修复美容师服务 20 条以上。

5. V3 fallback 改为“失败转诊断”。
   - V3 返回无法生成 SQL 时，V5 不应直接失败。
   - 应返回：缺哪个 domain adapter、建议补什么能力、可否给概览替代答案。

### P1：业务闭环增强，目标 85%+

1. 财务收入对账 adapter。
   - 对接收款、支付方式、日结、储值、退款、折扣、异常收款。

2. 供应链协调 adapter。
   - 覆盖供应商、采购、到货、缺货替代、库存价值、采购周期。

3. 活动策划兜底模板。
   - 没有生命周期机会或营销数据时，也应能生成“需审批的活动草稿建议”，而不是 no_data。

4. 全业务经营概览增强。
   - 今日经营概览不只返回订单/预约/机会，还要补收入、客流、新老客、员工忙闲、库存风险、异常提醒。

5. 口语化指标词典。
   - 把“客人”“在店”“最大一笔”“优惠送出去”“钱的事情”等映射为业务指标。

### P2：治理和自动修复

1. 把本次失败样本导入 V5 ontology backlog。
2. 按失败聚类自动生成 adapter 需求草案。
3. V5 eval gate 加入 CI：低于 75% 不允许作为默认入口。
4. 每个 domain adapter 建立 20 条回归题。
5. V5 结果要区分：
   - 真实可答。
   - 缺数据。
   - 缺 adapter。
   - 缺语义视图。
   - 需要用户澄清。

## 9. 是否继续推进 V5

建议继续推进，但不要马上替换 V4。

推荐策略：

1. V4 继续保留为“生命周期经营稳定版”。
2. V5 作为“全业务经营新版”进入内测。
3. V5 默认入口上线门槛设为：
   - 全量题库可用率 >= 85%。
   - 前台接待 >= 75%。
   - 店长经营 >= 85%。
   - 财务风控 >= 75%。
   - 收银与核销 >= 70%。
   - 意图模糊类必须以澄清为主，不能大量失败。

当前 V5 最大价值是架构边界清晰，能承接全业务 ontology；最大短板是 adapter 覆盖不足，导致 V3 fallback 失败集中暴露。

## 10. 下一轮开发建议

下一轮建议按以下顺序开发：

1. V5 澄清策略和失败转诊断。
2. 前台客户上下文 adapter。
3. 收银与核销 adapter。
4. 美容师日程 adapter。
5. 财务收入对账 adapter。
6. 供应链协调 adapter。
7. 重新跑 650 题评测，对比 V5 可用率是否超过 75%。

这 7 项完成后，V5 才具备从“架构验证版”进入“全业务经营内测版”的条件。
