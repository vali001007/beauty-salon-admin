# Ami Brain 管理端真实浏览器 E2E 报告（2026-07-21）

## 一、结论

本轮已使用真实管理端页面、真实店长账号、真实门店 6 数据和生产 user canary Release 359 完成核心闭环验收。

结论为：**管理端主链路通过，但不满足全量生产发布条件。**

- 已通过：登录、会话创建、真实问数、结构化回答、业务定义引用、刷新后消息持久化、多轮时间追问、结构化澄清、动作预览无数据安全收口。
- 本轮修复：Release Ontology 快照加载顺序、失败回答污染会话上下文、澄清选项误显示为真实动作、管理端角色别名导致动作计划被拒绝。
- 仍阻塞全量发布：Release 359 仅含 8 个能力，支付方式排行等已理解问题仍会因能力目录缺失失败；SSE 是完成后分块，未达到常规问数 P95 首字 3 秒目标；历史评测会话仍污染店长会话列表。

## 四、本轮发现并修复的问题

| 问题 | 产品影响 | 修复 |
| --- | --- | --- |
| Release 快照在多轮上下文校验后才加载 | 有效上下文被拒绝，追问失忆 | 先按 Release 能力引用加载冻结 Ontology，再校验上下文并复用同一快照 |
| 模型失败仍写入会话上下文 | 失败追问覆盖上一轮正确时间、指标和结果集 | 仅 `completed` 模型回答允许更新模型上下文 |
| 澄清选项复用 `suggestedActions` | 普通选项显示成真实写操作确认 | 后端澄清只写 `clarification` block；前端仅接受完整、需确认的持久化动作 |
| 管理端角色键与 Brain 角色不一致 | 店长动作在计划阶段报 `ROLE_DENIED` | 服务端统一可信角色别名，`roleHint` 只影响表达视角；计划和执行使用规范化认证角色 |

## 五、未关闭问题

### P0-1：完整生产能力目录仍未发布

问题 `其中哪种支付方式最多？` 已被模型准确解析为：

- intent：`ranking`
- metric：`metric.paid_amount`
- dimension：`dimension.paymentMethod`
- orderBy：实收金额降序

但 Release 359 只有 8 个能力，未包含 `finance_payment_breakdown`，`order_revenue_analysis` 也未声明 `ranking`，最终返回 `CAPABILITY_RETRIEVAL_NONE`。这是能力发布覆盖不足，不是语义理解失败。

### P0-2：首字延迟不达标

本轮四条成功请求后端总延迟为 `10.9s-17.6s`。当前 SSE 先发送 `run_started`，等待完整回答持久化后才将答案切成小块发送，属于“完成后分块”，不是模型实时流式输出。

PRD 要求常规问数 P95 首字延迟 `<=3s`。当前实现没有首个答案事件的独立计时，且真实页面在完整处理结束前只显示加载态，因此该门禁未通过。

### P0-3：历史评测会话污染用户工作区

店长会话列表可见大量 `评测 release_*` 与 canary smoke 会话。评测链路应使用内部会话状态或独立评测用户，并从用户工作区列表排除；不能依赖标题前缀长期过滤。

## 七、发布建议

当前 Release 359 继续保持指定店长 user canary，不扩大到全量用户。

下一步固定顺序：

1. 生成包含完整生产能力目录的新唯一候选 Release，并补支付方式排行等相邻意图合同。
2. 隔离评测会话与真实用户会话。
3. 建立 SSE 首答案事件门禁并实现真实流式/快速确定性路径，常规问数 P95 首字达到 3 秒。
4. 对上述失败点和相邻同义问法做定向回归。
5. 只在最终候选冻结后运行一次完整 650，形成正式 Go/No-Go。
6. 通过后再扩大 canary、执行回滚演练和终极成功标准验收。
