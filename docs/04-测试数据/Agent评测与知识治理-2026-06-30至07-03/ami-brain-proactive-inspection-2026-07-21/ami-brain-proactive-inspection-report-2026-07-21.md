# Ami Brain 主动巡检产品闭环验证报告

日期：2026-07-21

## 一、验证结论

本轮把原本停留在治理中心的巡检结果接入 Ami Brain 用户工作区，形成“主动发现风险 -> 展示业务证据与建议 -> 用户审批/修改/拒绝”的产品闭环。巡检仍不绕过现有动作治理，所有真实写操作继续先生成预览，再由用户确认。

本轮没有新增管理端或后端不存在的业务事实，没有执行真实业务写入，也没有扩大生产 Release 371 的单用户 Canary 范围。

## 二、已完成能力

### 2.1 后端巡检收件箱

- 新增 `GET /brain/inspections/inbox`，按当前门店、用户、角色、权限和已启用规则返回巡检结果。
- 收件箱返回风险摘要、证据、业务建议、可进入的管理端入口和是否允许审批。
- 只展示已启用规则，开发候选或停用规则不能进入用户工作区。
- 规则声明的权限必须由当前用户真实拥有；显式拒绝权限优先于通配权限。
- 美容师角色只能看到分配给本人的服务任务和预约风险，不能读取其他员工或全店明细。

### 2.2 规则调度

- 巡检调度改为每分钟检查一次到期规则。
- 每条规则独立使用自身 `scheduleCron` 判断是否到期，不再由固定每日 08:00 统一触发。
- 当前时点没有到期规则时不扫描门店，避免无效数据库访问。

### 2.3 Ami Brain 用户工作区

- 会话区顶部新增主动风险区域，展示风险总数、严重度、业务证据和处理建议。
- 默认展示前三条，可展开当前拉取的十条结果，并支持手动刷新和页面可见时每 60 秒刷新。
- 用户点击“查看并处理”后复用既有巡检修复审批弹窗，支持通过、修改和拒绝三种处理结果。
- 风险行固定为纵向信息结构，避免中间会话面板较窄时证据和操作按钮被压缩。

## 三、安全对抗性核对

首次真实数据核对发现两个产品级风险：停用候选规则进入用户收件箱，以及美容师能够看到全店服务任务异常。本轮均已修复并加入自动化测试。

| 风险 | 修复结果 |
| --- | --- |
| 停用候选规则被普通用户看到 | 收件箱只读取 enabled 规则，结果为 0 |
| 美容师读取其他员工或全店异常 | 按当前登录美容师的任务和预约归属过滤，越权结果为 0 |
| 用户缺少规则权限仍读取结果 | 规则版本权限逐条校验，缺权限直接排除 |
| 用户没有执行权限仍出现审批入口 | `canReview=false`，前端不提供审批命令 |
| 模型或巡检直接执行写操作 | 继续强制 preview 和人工确认 |

## 四、真实数据验证

验证门店：`storeId=6`。

| 验证身份 | 真实结果 | 结论 |
| --- | --- | --- |
| 店长用户 28 | 38 条已启用的客户流失中风险，具备审批权限 | 能看到当前职责范围内的主动经营风险 |
| 美容师用户 32 | 0 条 | 停用候选和其他美容师任务均未泄露 |

店长工作区抽样证据包含客户、最近到店、累计消费和统计截止时间，例如客户流失风险会给出“创建客户召回或跟进预览”的业务建议。验证过程未修改客户、预约、任务或营销数据。
