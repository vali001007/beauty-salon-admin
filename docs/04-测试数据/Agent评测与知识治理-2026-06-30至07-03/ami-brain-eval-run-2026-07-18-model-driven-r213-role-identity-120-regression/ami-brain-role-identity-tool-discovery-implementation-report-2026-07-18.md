# Ami Brain 角色身份与工具发现实施报告

生成日期：2026-07-18

## 一、目标与边界

本轮只修复当前统一后台已具备事实的角色查询，不新增管理端页面、业务表、migration 或真实写操作。重点解决美容师“我”的身份失真、个人数据越界风险、前台名单粒度和模型工具发现问题。

## 二、650 题发现基线

R205 基于 release 309 完成 650 题真实请求路径扫描：

- 自动真实可用 `254/638 = 39.8%`，另有 12 条 provider unavailable。
- 店长 `82/100`、前台 `16/100`、营销 `46/100`、美容师 `4/100`、库存 `62/97`、财务 `37/100`、边界 `7/43`。
- 自动假阳性 56 条，因此 R205 只用于发现问题，不作为真实产品完成度。
- Adapter 分布为 `none=638`；模型主路径仍主要使用治理能力卡，Domain Adapter 与能力卡执行器存在双轨，需要后续架构统一。

## 三、根因

1. 评测固定使用管理员 `userId=1`，而美容师个人技能按 `Beautician.userId` 解析身份，导致“我的排期/业绩”无法命中真实美容师。
2. 美容师个人服务技能在找不到 profile 时会省略 `beauticianId` 过滤，存在退化为全店预约的风险。
3. 美容师、前台和预约能力卡的描述与示例过薄，模型容易选择 `customer_facts`、通用概览或不选择工具。
4. “皮肤过敏事件”外部缺口门禁范围过宽，把“下一位客户过敏史”错误当成事故统计。
5. 自动评分只检查工具/引用时，仍会把通用概览误判为已回答细粒度问题。

## 四、实现

- 新增评测角色用户解析器：按门店、角色和活跃 profile 选择真实账号；门店 6 美容师题使用真实绑定美容师账号。
- 美容师服务技能未找到活跃 profile 时抛出 `beautician_identity_not_linked`，且不执行预约查询。
- 丰富 `beautician_service_overview`、`front_desk_operations_overview`、`reservation_list` 的名称、描述、示例、同义词、负例和业务定义引用。
- 前台待到店/已到店问题增加专用名单输出，返回客户、时间、项目和状态。
- 过敏事故门禁排除“下一个/这个客户的过敏和注意事项”查询，事故统计仍保持无事实拒答。

## 六、可信评测

### 6.1 R212 人工确认 targeted

`7/7 usable_exact`：

1. 我今天有几个客人，分别几点。
2. 下一个客人是谁，做什么项目。
3. 我今天第一个客人几点来。
4. 下一个客人有没有皮肤过敏或注意事项。
5. 今天预约了但还没来的客户名单。
6. 今天所有到店客户基本名单。
7. 当前下一个预约是谁、什么时候。

### 6.2 R211 自动结果不采用

R211 自动报告为 `39/75`。人工审计确认，预约确认时长、爽约预测、通知状态、VIP、空档、周排班、护理历史、服务时长、续卡建议等题仍只返回通用概览，属于隐藏假阳性。因此正式新增只认 R212 的 7 条。

### 6.3 R213 前 120 题回归

- 自动可用 `99/120 = 82.5%`，provider unavailable 为 0。
- 相比 R204 的 101 条少 2 条，均为主动剔除隐藏假阳性：流失人数/原因被沉睡名单替代、利润率实际上升却被算作下降归因成功。
- 权限绕过、跨门店读取、roleHint 绕过和假动作确认均为 0。
