## 十、改进建议

### P0：先把失败最多的 Text-to-SQL 回落变成可治理能力

1. 在 V5 Router 中增加“adapter 优先、V3 兜底需命中白名单”的硬规则。没有明确 adapter 或语义视图时，不直接执行 SQL，改为返回缺口诊断或追问。
2. 为失败样本自动生成治理 backlog：domain、intent、expectedAdapter、expectedView、缺字段、缺时间口径、失败原因。
3. AI 治理中心新增 V5 失败样本筛选：按 adapter、业务域、失败原因、题库 ID 聚合。

### P1：补四个高价值垂直 adapter

1. 财务收入对账 adapter：收入、收银、退款、优惠、支付方式、日结差异、异常单据。
2. 收银核销 adapter：会员卡核销、次卡扣减、订单关联、退款折扣、操作员责任。
3. 前台现场协调 adapter：预约、到店、排班、等候、房间/美容师冲突。
4. 供应链协调 adapter：采购建议、到货、缺货、消耗、供应商、补货周期。

### P1：强化模糊追问与记忆

1. 对“这个、刚才那个、不要这个、换成上周、客户那块怎么样”这类问题，必须先读取短期记忆和上一轮实体。
2. 多意图问题先拆解计划，不直接执行单一路径。
3. 否定纠正类问题需要记录被否定对象和替代约束，例如“不看会员卡，只看项目订单”。

### P2：把 V5 从问答型升级成经营动作型

1. 对营销、库存、预约、财务风险输出 action_card，而不是只给文字或表格。
2. 动作必须保持安全边界：只生成草稿、审批、跟进任务，不直接发券、扣库存、改订单、改排班。
3. 建立 V5 经营计划闭环：诊断 -> 计划 -> 草稿/审批 -> 归因 -> 下次策略。

## 十一、建议验收门槛

下一轮 V5 评测建议按以下门槛收口：

1. 总体可用率 >= 80%。
2. 营销、库存、店长经营 >= 85%。
3. 前台、美容师、财务 >= 75%。
4. 边界/多轮 >= 70%。
5. Text-to-SQL 安全计划失败从 162 条降到 80 条以内。
6. 动作卡覆盖率从 2.8% 提升到 20% 以上。

## 十二、最终判断

Agent V5 当前方向是对的：独立边界、全业务 ontology、可复用 V3/V4 底层能力但不递归调用旧 Agent。评测结果也证明它已经超过“概念验证”，进入“可按业务域补齐 adapter 的工程化阶段”。

但短板也很清楚：不能继续把大量真实业务问题丢给 V3 Text-to-SQL 兜底。V5 的下一步不是换模型，而是补垂直 adapter、补语义视图、补追问记忆，并把失败样本自动沉淀进 AI 治理中心。
