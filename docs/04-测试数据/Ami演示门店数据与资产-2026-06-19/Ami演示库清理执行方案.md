# Ami 演示库清理执行方案

生成日期：2026-06-09

## 结论

当前远端库将按“演示库”处理，目标是只保留 `Ami 全量演示门店` 作为对外演示、产品验证和联调入口。其他门店及其客户、订单、库存、预约、终端、预测、营销触达等业务数据将通过一次性脚本清理。

真实清理不得直接手写 SQL 执行，必须先跑 dry-run，确认影响清单和备份后再执行 apply。

## 已确认现状

- 当前远端库是 `remote_or_ssl` 类型连接。
- 当前共有 6 个门店。
- `Ami 全量演示门店` 当前是主要演示数据来源：客户 1243、订单 422、终端 3、服务任务 221、预约 362。
- 其他门店仍存在客户、订单、预约、终端、预测快照等数据；清理后不可再作为历史对照使用。

## 清理策略

- 保留门店：门店名规范化后必须唯一匹配 `Ami 全量演示门店`。
- 默认只执行 dry-run，不修改数据库。
- 真实执行必须显式传入：

```powershell
npm.cmd --prefix packages/server-v2 run db:retain:ami-demo
```

- 执行前先跑：

```powershell
npm.cmd --prefix packages/server-v2 run db:retain:ami-demo:dry-run
```

- apply 脚本内置安全阀：必须同时具备 `--apply`、`--yes`、`--confirm-retain-store-name="Ami 全量演示门店"`。

## 清理范围

脚本清理以下类型中归属非保留门店，或依赖非保留门店客户、订单、商品、项目、美容师、终端设备的数据：

- 门店、用户-门店授权、客户、健康档案、消费记录、会员卡账户和余额流水。
- 商品、批次、库存流水、供应商、供应商订单、调拨单。
- 项目、BOM、美容师、排班、预约、智能排班资源。
- 终端设备、服务任务、皮肤检测、会话、打印任务、收银班次、日结。
- 订单、订单明细、支付、退款、提成、Ami 绩效和账单。
- 预测快照、推荐事件、营销触达、营销归因、小程序身份和行为事件。

脚本不会默认删除这些全局主数据：角色、权限、品类、项目类型、卡项模板、美容师等级。原因是它们没有明确门店归属，且保留演示门店仍可能依赖这些字典。

## 执行前检查

1. 在云数据库控制台创建快照或导出备份。
2. 运行 dry-run，保存输出中的 `plannedDeleteCounts`。
3. 确认 `retainStore.name` 为 `Ami 全量演示门店`。
4. 确认 `deleteStores` 列表正是需要清理的其他门店。
5. 确认没有真实客户、订单、会员余额、财务结算数据需要迁出。

## 执行后验收

- 门店列表只剩 `Ami 全量演示门店`。
- 管理端可用 admin 登录并切换到该门店。
- 客户、订单、库存、预约、终端、营销推荐页面仍有数据。
- Dashboard 统计只显示演示门店口径。
- Ami Aura Lite 可绑定或使用该门店完成核心演示流程。

## 2026-06-09 执行结果

- 已按脚本完成远端演示库清理，最终 dry-run 的 `plannedDeleteCounts` 全部为 0。
- 当前门店列表只剩 1 个门店：`Ami 全量演示门店`。
- 保留门店核心数据仍在：客户 1244、订单 421、商品 20、项目 15、美容师 12、终端 3、服务任务 221、预约 362、预测运行 5。
- 旧 MVP/测试采购单为 0，旧营销策略为 0；系统用户只保留 `admin` 与 `ami_demo_full_*` 演示账号。
- 执行前已生成清理清单备份：`tmp/db-backups/ami-demo-retain-manifest-2026-06-09T12-05-59-737Z.json`。该文件用于核对清理范围，不等同于完整数据库恢复备份；完整恢复仍建议依赖云数据库快照。
- 清理过程中发现部分旧全局卡模板仍被 Ami 演示客户卡引用，因此脚本已调整为只清理未被引用的旧卡模板，避免破坏保留门店的会员卡记录。
