## 1. 当前结论

当前 9 条服务人归属候选涉及 5 位美容师，其中 3 位候选美容师尚未绑定系统账号，影响 4 条候选服务人归属后续生成员工维度提成：

```text
inputAssignments=9
assignmentBeauticians=5
missingUserBindingBeauticians=3
impactedAssignments=4
bindingDraftItems=1
missingUserBindingNoCandidateItems=2
```

处理原则：

- 账号绑定会影响 `CommissionRecord.staffUserId` 和员工人效归集，不能自动写回。
- `operation-profit:beautician-user-audit` 只读输出 `bindingDraft`，不会更新 `Beautician.userId`。
- `bindingDraft.missingUserBindingNoCandidates` 已结构化列出无账号候选缺口，当前为韩雨、许诺 2 位；这类缺口不能靠确认绑定候选解决，必须先创建系统用户或把美容师绑定到现有 active 门店用户。
- `operation-profit:beautician-user-backfill` 默认 dry-run；真实写库必须显式传 `--apply --yes --storeId=6`。
- 绑定 JSON 中的 `confirmedBy=pending_business_confirmation` 表示未完成业务确认，即使误传 `--apply --yes` 也会被跳过。

## 4. 确认后执行方式

先 dry-run：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:beautician-user-backfill -- --storeId=6 --file=docs/04-测试数据/operation-profit-beautician-user-bindings.pending.json
```

确认 dry-run 无异常且 `confirmedBy` 已改为业务确认人后，才允许写库：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:beautician-user-backfill -- --storeId=6 --file=<确认后的JSON文件> --apply --yes
```

写回账号绑定后，重新执行：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<确认后的服务人归属JSON文件>
npm.cmd --prefix packages/server-v2 run operation-profit:backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30
```
