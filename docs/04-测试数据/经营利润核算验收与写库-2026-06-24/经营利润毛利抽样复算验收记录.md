## 1. 当前结论

当前抽样复算脚本已能基于真实订单输出商品/项目毛利公式，但本次样本不能作为“业务验收通过”证据。

原因：

- 最新复验读取 9 条商品样本和 20 条项目样本。
- 样本均能输出 `净收入 - 成本 - 提成 = 毛利`。
- 当前样本均存在数据缺口，主要为 `missing_commission`、`missing_assignee` 和项目 `missing_actual_consumption`。
- 最新抽样脚本已能在缺提成记录时额外输出 `missing_commission_rule`，用于判断是“提成规则未覆盖”还是“规则覆盖但历史提成记录未回填”。
- 2026-06-20 复验确认：抽样脚本已能输出 `missing_project_master` 和 `projectMasterGapItems`，用于区分“项目档案不存在”和“项目存在但 BOM 未配置”。
- 目标库未执行 `CommissionRule.userId` migration 时，脚本会降级为 `commissionRuleCoverageMode=legacy_without_user_scope`，此时只能判断全局、分类、指定商品/项目和等级规则，不能完整判断指定员工规则覆盖。
- 项目样本中存在 `missing_actual_consumption`，说明有项目只能按 BOM 标准成本估算，不能证明真实耗材流水完整。
- readiness 当前仍为 `blocked`，但经营利润必需迁移、成本录入、项目 BOM 和提成规则覆盖已通过；剩余主要阻断是服务人归属、提成记录和 3 条项目档案缺口。

## 5. 后续复验模板

迁移和数据补齐后，重新执行：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30
npm.cmd --prefix packages/server-v2 run operation-profit:sample-reconcile -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --sampleSize=3
```

复验通过标准：

| 检查项 | 通过标准 |
| --- | --- |
| readiness | 至少达到 `conditional`，无 migration/schema/cost/source order 级 fail |
| 规则覆盖模式 | `commissionRuleCoverageMode` 为 `full`；如果为 `legacy_without_user_scope`，只能作为迁移前缺口证据 |
| 商品样本 | 至少 2 条商品样本无 `missing_cost`、`missing_commission`、`missing_commission_rule`、`missing_assignee` |
| 项目样本 | 至少 2 条项目样本无 `missing_project_master`、`missing_bom`、`missing_commission`、`missing_commission_rule`、`missing_assignee` |
| 项目档案缺口 | `projectMasterGapItems=0`，或业务已确认这些明细为历史异常并在验收记录中排除 |
| 公式复算 | 页面展示与脚本公式一致，允许四舍五入差异 |
| 业务确认 | 产品/运营确认成本、BOM、提成口径可接受 |
