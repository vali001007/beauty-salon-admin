# 经营利润业务确认后真实写库手册

记录日期：2026-06-20
适用门店：`storeId=6`，Ami 全量演示门店
数据窗口：`2026-06-01` 至 `2026-06-30`

---

## 1. 执行前置

本手册只适用于业务已经逐条确认正式 JSON 的场景。当前已具备真实写库授权，但授权不等于业务确认；以下条件缺一不可：

| 条件 | 通过标准 |
| --- | --- |
| 服务人候选正式 JSON | 9 条候选均已确认或修正 `beauticianId`，`confirmedBy` 不是占位值 |
| 无候选服务人正式 JSON | 24 条均已补 `resolution`；`resolution=assign` 时必须有真实 `beauticianId` |
| 美容师账号绑定正式 JSON | 已确认 `beauticianId -> userId`，且员工账号有效 |
| 员工账号创建正式 JSON | 仅在韩雨、许诺这类无账号候选时使用；已确认用户名、姓名、手机号、角色、门店和 `confirmedBy` |
| 项目档案正式 JSON | 每条为 `repair_project` 或 `historical_exception`，不能保留 TODO |
| 统一确认包预检 | `confirmationReady=true`、`writeGate.applyAllowed=true` |
| 确认后 dry-run | `status=dry_run_complete`，且业务已确认每一步计划 |

禁止把 `docs/04-测试数据/operation-profit-confirmation-drafts/*.draft.json` 原样用于写库。草稿里的 `TODO_REAL_BUSINESS_CONFIRMER`、`TODO_assign_or_historical_exception_or_ignore_non_margin`、`TODO_REQUIRED_IF_ASSIGN` 必须先被业务替换。

脚本保护：服务人写回、账号绑定写回、项目档案修复这 3 个文件型写库脚本在 `--apply` 模式下会直接拒绝 `.pending.json`、`.draft.json` 和 `operation-profit-confirmation-drafts/` 目录下的文件路径。

---

## 2. 写库前硬门禁

先执行统一确认包预检：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选正式确认JSON> --assigneeManualReviewFile=<无候选服务人正式确认JSON> --beauticianUserFile=<美容师账号绑定正式确认JSON> --projectMasterFile=<项目档案正式确认JSON> --requireReady --summaryOnly
```

如果存在员工账号创建正式 JSON，同一条硬门禁命令必须带上 `--staffUserFile`：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选正式确认JSON> --assigneeManualReviewFile=<无候选服务人正式确认JSON> --beauticianUserFile=<美容师账号绑定正式确认JSON> --staffUserFile=<员工账号创建正式确认JSON> --projectMasterFile=<项目档案正式确认JSON> --requireReady --summaryOnly
```

必须同时满足：

```text
confirmationReady=true
writeGate.applyAllowed=true
staffUser.issueCount=0
coverage.issueCount=0
inputIntegrity.issueCount=0
writeGate.blockers.unconfirmedItems=0
writeGate.blockers.missingResolutionItems=0
writeGate.blockers.missingStaffUserItems=0
writeGate.blockers.missingCoverageItems=0
writeGate.blockers.inputIntegrityItems=0
```

只要返回非 0，立即停止，不执行任何 `--apply --yes`。

脚本保护：`--requireReady` 模式必须显式传入 4 个核心正式确认 JSON；如果漏传任一核心文件，脚本会直接报错，不再回退读取 pending 默认文件。存在员工账号创建正式确认 JSON 时，也必须同步传 `--staffUserFile=<员工账号创建正式确认JSON>`。

说明：统一确认包预检会把同一个美容师账号绑定正式 JSON 中已业务确认、用户有效且门店匹配的 `beauticianId -> userId` 计划计入员工账号主体解决项；也会把 `--staffUserFile` 中已业务确认、用户名未冲突、角色和门店有效的创建账号计划计入员工账号主体解决项。也就是说，周宁这类已有候选账号、韩雨和许诺这类需新建账号的缺口，都可以通过同一个确认包预检进入 dry-run；但真实写库顺序仍必须先执行员工账号创建或账号绑定，再执行服务人归属和提成回填。

## 3. 确认后 dry-run

统一确认包通过后，先执行完整只读演练：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmed-dry-run -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选正式确认JSON> --assigneeManualReviewFile=<无候选服务人正式确认JSON> --beauticianUserFile=<美容师账号绑定正式确认JSON> --projectMasterFile=<项目档案正式确认JSON> --summaryOnly
```

如果存在员工账号创建正式 JSON，则带上可选 `--staffUserFile`，该步骤会在账号绑定 dry-run 前执行：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmed-dry-run -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选正式确认JSON> --assigneeManualReviewFile=<无候选服务人正式确认JSON> --beauticianUserFile=<美容师账号绑定正式确认JSON> --staffUserFile=<员工账号创建正式确认JSON> --projectMasterFile=<项目档案正式确认JSON> --summaryOnly
```

必须看到：

```text
status=dry_run_complete
confirmation hard gate exitCode=0
staff user create dry-run exitCode=0（传入 staffUserFile 时）
beautician user binding dry-run exitCode=0
assignee candidate dry-run exitCode=0
manual review assignee dry-run exitCode=0
project master dry-run exitCode=0
commission backfill dry-run exitCode=0
```

如果输出 `blocked_by_confirmation_gate`，说明仍有业务确认缺口，不能写库。
如果输出 `dry_run_failed`，先排查失败步骤，不能跳过失败项继续写库。

脚本保护：`operation-profit:confirmed-dry-run` 必须显式传入 4 个核心正式确认 JSON；如果漏传任一核心文件，会在执行任何 backfill dry-run 之前停止。存在员工账号创建正式确认 JSON 时，也必须同步传 `--staffUserFile=<员工账号创建正式确认JSON>`。

## 4. 推荐写库顺序

所有写库命令都必须显式指定 `--storeId=6`，并且只在 dry-run 结果经业务确认后执行。

### 4.1 无账号候选员工账号创建

仅当存在韩雨、许诺这类无系统账号候选时执行。先用正式确认 JSON dry-run：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:staff-user-backfill -- --storeId=6 --file=<员工账号创建正式确认JSON>
```

确认计划后，设置默认密码环境变量，再写库：

```powershell
$env:OPERATION_PROFIT_NEW_STAFF_DEFAULT_PASSWORD="<至少8位临时密码>"
npm.cmd --prefix packages/server-v2 run operation-profit:staff-user-backfill -- --storeId=6 --file=<员工账号创建正式确认JSON> --apply --yes
```

脚本会创建 `User`、写入 `beautician` 角色、写入 `UserStore` 门店范围，并绑定 `Beautician.userId`。它同样会拒绝 `.pending.json`、`.draft.json`、占位确认人、缺 `--storeId`、缺 `--apply --yes` 或缺默认密码环境变量。

### 4.2 美容师账号绑定

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:beautician-user-backfill -- --storeId=6 --file=<美容师账号绑定正式确认JSON> --apply --yes
```

完成后复跑同命令去掉 `--apply --yes`，期望没有新的可写计划。

### 4.3 服务人候选写回

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<服务人候选正式确认JSON> --apply --yes
```

完成后复跑 dry-run，期望已写回项不再重复计划。

### 4.4 无候选服务人处理

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<无候选服务人正式确认JSON> --apply --yes
```

脚本只会写入 `resolution=assign` 且有真实 `beauticianId` 的记录。`historical_exception` 和 `ignore_non_margin` 不会写 `OrderItem.beauticianId`，后续由 readiness 确认包模式留痕。

### 4.5 项目档案修复

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:project-master-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<项目档案正式确认JSON> --apply --yes
```

脚本只处理 `resolution=repair_project`。`historical_exception` 不会修改历史订单明细，页面仍显示项目档案缺口。

### 4.6 提成回填

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --apply --yes
```

提成回填必须放在服务人和账号绑定之后。若仍输出 `skippedMissingBeautician` 或 `skippedMissingStaffUser`，说明归属或账号仍未补齐。

## 5. 写库后复验

写库后按顺序复验：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30 --assigneeManualReviewFile=<无候选服务人正式确认JSON>
npm.cmd --prefix packages/server-v2 run operation-profit:sample-reconcile -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --sampleSize=3
npm.cmd run check:operation-profit
```

目标：

```text
readinessStatus=conditional 或 ready
assignee_data 不再 fail，或 confirmedExceptionGaps 能解释历史异常
commission_data 缺口显著下降，或剩余缺口有业务确认解释
sample-reconcile 能输出可复算商品/项目样本
check:operation-profit 通过
```

达到上述目标后，再进入 6 个经营利润页面登录态验收，并补 `docs/04-测试数据/经营利润页面验收记录模板.md`。
