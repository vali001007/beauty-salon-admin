## 3. 统一预检

任何写库前先跑统一确认预检：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选确认JSON> --assigneeManualReviewFile=<服务人无候选确认JSON> --beauticianUserFile=<账号绑定确认JSON> --projectMasterFile=<项目档案确认JSON>
```

如果存在员工账号创建正式确认 JSON，同一条预检命令必须带上可选 `--staffUserFile`：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选确认JSON> --assigneeManualReviewFile=<服务人无候选确认JSON> --beauticianUserFile=<账号绑定确认JSON> --staffUserFile=<员工账号创建正式确认JSON> --projectMasterFile=<项目档案确认JSON>
```

写库前硬门禁建议加 `--requireReady`：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选确认JSON> --assigneeManualReviewFile=<服务人无候选确认JSON> --beauticianUserFile=<账号绑定确认JSON> --projectMasterFile=<项目档案确认JSON> --requireReady
```

存在员工账号创建正式确认 JSON 时，硬门禁也必须带同一份 `--staffUserFile=<员工账号创建正式确认JSON>`。

普通预检会输出完整 JSON 并返回 0，便于排查问题；`--requireReady` 用于写库流水线，确认包没过会返回非 0，避免把“已授权写库”误当成“业务确认已完成”。
当前已具备真实写库授权，但授权只解决执行权限；业务确认 JSON 未逐条替换 TODO、未通过 `confirmationReady=true` 和 `writeGate.applyAllowed=true` 前，仍不得执行任何 `--apply --yes`。
如果只想快速判断确认包是否过期、是否能进入 dry-run，可加 `--summaryOnly`，只输出 `summary`、`writeGate` 和下一步，不展开所有明细：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmation-audit -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选确认JSON> --assigneeManualReviewFile=<服务人无候选确认JSON> --beauticianUserFile=<账号绑定确认JSON> --projectMasterFile=<项目档案确认JSON> --requireReady --summaryOnly
```

存在员工账号创建正式确认 JSON 时，短输出命令同样带 `--staffUserFile=<员工账号创建正式确认JSON>`，并重点看 `summary.staffUser.issueCount=0` 与 `writeGate.blockers.missingStaffUserItems=0`。

脚本保护：`--requireReady` 模式必须显式传入上述 4 个核心确认 JSON；漏传任一核心文件会直接失败，不会回退读取 pending 默认文件。存在员工账号创建正式确认 JSON 时，也必须同步传 `--staffUserFile=<员工账号创建正式确认JSON>`，否则韩雨、许诺这类账号主体缺口不会计入解决项。

## 4. Dry-run 顺序

确认包通过统一预检后，推荐先用编排脚本做一次完整只读演练：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmed-dry-run -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选确认JSON> --assigneeManualReviewFile=<服务人无候选确认JSON> --beauticianUserFile=<账号绑定确认JSON> --projectMasterFile=<项目档案确认JSON> --summaryOnly
```

如果存在韩雨、许诺这类无账号候选，需要把员工账号创建正式确认 JSON 作为可选参数带入：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:confirmed-dry-run -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --assigneeFile=<服务人候选确认JSON> --assigneeManualReviewFile=<服务人无候选确认JSON> --beauticianUserFile=<账号绑定确认JSON> --staffUserFile=<员工账号创建确认JSON> --projectMasterFile=<项目档案确认JSON> --summaryOnly
```

该脚本只跑 `operation-profit:confirmation-audit --requireReady` 和后续 dry-run，不接受 `--apply` / `--yes`。如果确认包未通过，会输出 `blocked_by_confirmation_gate` 并跳过员工账号创建、账号绑定、服务人写回、项目档案修复和提成回填 dry-run，避免在业务确认不完整时给出误导性写库计划；加 `--summaryOnly` 时，确认包硬门禁只输出摘要。

脚本保护：`operation-profit:confirmed-dry-run` 同样必须显式传入 4 个核心确认 JSON；漏传任一核心文件会在任何 backfill dry-run 前停止。存在员工账号创建正式确认 JSON 时，也必须同步传 `--staffUserFile=<员工账号创建正式确认JSON>`。

需要单项排查时，再按下面顺序 dry-run：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:beautician-user-backfill -- --storeId=6 --file=<账号绑定确认JSON>
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<服务人确认JSON>
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<服务人无候选确认JSON>
npm.cmd --prefix packages/server-v2 run operation-profit:project-master-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<项目档案确认JSON>
npm.cmd --prefix packages/server-v2 run operation-profit:backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30
```

`<服务人无候选确认JSON>` 可以直接来自 `operation-profit-assignee-manual-review.pending.json` 的正式确认副本；写回脚本只会处理 `resolution=assign` 且带 `beauticianId` 的记录，`resolution=historical_exception` / `ignore_non_margin` 不会进入写回计划。
`<项目档案确认JSON>` 中只有 `resolution=repair_project` 会进入 `project-master-backfill`；`resolution=historical_exception` 只用于确认包预检和验收说明，不会修历史订单。

检查点：

| 脚本 | 通过标准 |
| --- | --- |
| `beautician-user-backfill` | `plannedBindings` 等于业务确认绑定数，`updatedBeauticians=0` |
| `assignee-backfill` | `plannedAssignments` 等于业务确认可写回服务人数量，`updatedOrderItems=0` |
| `project-master-backfill` | `plannedFixes` 等于业务确认项目修复数量，`updatedOrderItems=0` |
| `backfill` | 只生成已有服务人且有账号主体的提成计划 |

## 5. Apply 顺序

用户确认 dry-run 结果后，才允许真实写库：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:beautician-user-backfill -- --storeId=6 --file=<账号绑定确认JSON> --apply --yes
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<服务人确认JSON> --apply --yes
npm.cmd --prefix packages/server-v2 run operation-profit:assignee-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<服务人无候选确认JSON> --apply --yes
npm.cmd --prefix packages/server-v2 run operation-profit:project-master-backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --file=<项目档案确认JSON> --apply --yes
npm.cmd --prefix packages/server-v2 run operation-profit:backfill -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --apply --yes
```

写库后必须重跑：

```powershell
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30
npm.cmd --prefix packages/server-v2 run operation-profit:readiness -- --storeId=6 --periodMonth=2026-06 --from=2026-06-01 --to=2026-06-30 --assigneeManualReviewFile=<服务人无候选确认JSON>
npm.cmd --prefix packages/server-v2 run operation-profit:sample-reconcile -- --storeId=6 --from=2026-06-01 --to=2026-06-30 --sampleSize=20
```

默认 readiness 保留真实库原始缺口；带 `--assigneeManualReviewFile` 的确认包模式会把已业务确认的 `historical_exception` / `ignore_non_margin` 从服务人 fail 和提成 warn 中剔除，并在 evidence 输出 `confirmedExceptionGaps`。目标是确认包模式至少从 `blocked` 进入 `conditional`。

## 6. 禁止事项

- 不允许把 `pending_business_confirmation` 原样 apply。
- 不允许把 `业务确认人`、`待确认`、`TODO` 这类占位确认人当作真实业务确认。
- 不允许跳过统一预检直接写库。
- 不允许不带 `--storeId=6` 执行 apply。
- 不允许在服务人归属未确认时生成提成记录。
- 不允许把 24 条无候选缺口停留在 Markdown 表格里；必须进入 `operation-profit-assignee-manual-review.pending.json` 并由业务给出处理结论。
- 不允许把项目 `101` 直接当作真实项目补 BOM；必须先确认项目身份或标记历史异常。
