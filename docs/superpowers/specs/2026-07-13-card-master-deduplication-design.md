# 次卡主数据归并与历史引用治理设计

日期：2026-07-13  
范围：次卡主数据、已售客户次卡、核销流水、订单明细及后续防重门禁  
状态：待用户复核

## 1. 背景与目标

当前 `Card` 主数据存在历史重复创建。重复卡同时被已售客户次卡、核销流水和订单明细引用，直接删除重复记录会破坏历史追溯并触发外键约束。

本次目标是：

- 将同一商业次卡归并到一个规范 `Card` 主记录；
- 保留客户实际购买金额、剩余次数、有效期、价格快照和核销历史；
- 更新所有结构化外键和必要的订单 JSON 快照引用；
- 重复卡不再出现在管理端和销售选项中；
- 增加同门店范围内的次卡名称防重门禁，阻止问题再次发生；
- 所有数据操作支持 dry-run、单事务执行、审计输出和执行后复核。

## 3. 数据影响范围

归并事务必须处理以下对象：

1. `CustomerCard.cardId`
   - 迁移到规范卡 ID；
   - 保留 `cardName`、`totalTimes`、`remainingTimes`、`paidAmount`、`discountAmount`、`recognizedUnitValue`、`pricingSnapshot` 和有效期；
   - 不把同一客户持有的多张同名卡合成一张，客户每次购买仍是独立资产。

2. `CardUsageRecord.cardId`
   - 迁移到规范卡 ID；
   - 保留 `customerCardId`、核销项目、次数、确认金额、来源订单和操作人；
   - 不重算历史确认收入。

3. `OrderItem.itemId`
   - 仅处理次卡类型：`card/customer_card/card_sale/member_card`；
   - 将重复卡 ID 改为规范卡 ID；
   - 保留订单成交金额、折扣和商品名称。

4. `OrderItem.payload`
   - payload 中存在 `itemId/cardId` 且等于重复卡 ID 时同步替换；
   - 仅修改标识字段，不覆盖历史成交价格、折扣或赠送项目。

5. 其他 JSON 快照
   - `CustomerCard.pricingSnapshot` 和 `CardUsageRecord.pricingSnapshot` 属于历史成交证据；
   - 保留原 `cardId` 和当时配置，不改写历史快照，以避免伪造历史事实；
   - 当前关联关系以结构化 `cardId` 为准，快照继续用于解释原始成交。

6. 重复 `Card`
   - 所有结构化引用迁移完成并校验为零后删除归并卡；
   - 删除前将完整记录写入归并审计结果；
   - 任一引用无法迁移时，整个事务回滚。

## 4. 执行流程

### 4.1 Dry-run

默认命令只读，输出每组：

- 规范卡与归并卡的字段差异；
- `CustomerCard`、`CardUsageRecord`、`OrderItem` 引用数量；
- payload 可确定替换数量；
- 无法识别或存在冲突的记录；
- 合并前后应保持不变的金额、次数和记录总数。

结果分为：

- `ready`：全部引用可确定迁移；
- `manual_review`：出现未知 itemType、异常 JSON 或规范卡商业字段冲突；
- `blocked`：规范卡不存在、归并卡被重复映射、关键统计无法守恒。

### 4.2 Apply

只有同时传入 `--apply --yes` 才允许写库。执行顺序固定为：

1. 校验映射和数据库当前值仍与 dry-run 一致；
2. 锁定涉及的 `Card` 主记录；
3. 更新 `CustomerCard.cardId`；
4. 更新 `CardUsageRecord.cardId`；
5. 更新次卡类 `OrderItem.itemId` 及 payload 标识字段；
6. 校验归并卡结构化引用为零；
7. 删除归并卡；
8. 写入归并审计记录或输出可持久化审计报告；
9. 提交事务。

任一步失败均回滚，不允许部分成功。

## 5. 防重门禁

服务层新增规范化名称：去除首尾空格、压缩连续空格、统一大小写。

创建或修改次卡时，按以下业务键检查冲突：

- 规范化名称；
- 门店范围，`storeId = null` 代表全部门店；
- 排除当前编辑记录。

冲突时返回 HTTP 409，并提示已存在的次卡 ID 和名称。数据库层增加表达式唯一索引，以 `COALESCE(storeId, 0)` 与规范化名称形成最终门禁。迁移必须在历史重复归并后再创建唯一索引。

管理端收到 409 后保留表单内容并展示明确错误，不再生成重复卡。

## 6. 守恒与验收口径

每组归并必须满足：

- `CustomerCard` 记录总数不变；
- 客户次卡 `totalTimes`、`remainingTimes`、`paidAmount` 汇总不变；
- `CardUsageRecord` 记录数、核销次数和 `recognizedAmount` 汇总不变；
- 次卡订单明细数、成交净额汇总不变；
- 归并卡的结构化引用归零；
- `Card` 主数据由 10 条减少为 6 条；
- 管理端只展示 ID 2、3、5、6、14、17；
- 销售选项和终端开卡不再返回 12、13、15、16；
- 已售客户次卡仍可正常查看和核销。

金额差异超过 0.01 元、次数差异不为 0 或记录数变化时，执行判定失败并回滚。

## 7. 测试范围

- dry-run 正确识别四组映射和所有引用数量；
- 未传 `--apply --yes` 时绝不写库；
- 单事务迁移结构化引用并删除重复卡；
- 任一守恒检查失败时回滚；
- 历史价格快照不被改写；
- 重复名称创建和改名返回 409；
- 不同门店允许存在同名门店专属卡；
- 全部门店卡与同名门店专属卡按现有业务边界分别管理；
- 管理端列表、销售选项、客户卡详情和核销主链路回归通过。

## 8. 数据安全与发布

- 执行前备份 `Card`、`CustomerCard`、`CardUsageRecord`、相关 `OrderItem`；
- 先在 dry-run 输出上复核，再执行真实归并；
- 真实归并仅执行一次，脚本重复运行应返回“已完成/无待迁移记录”；
- 不自动提交、推送或发布；
- 当前工作区存在大量其他模块改动，实施时只修改次卡治理相关文件。
