# 管理平台数据加载性能优化建议方案

生成日期：2026-06-07

## 1. 结论

当前“各模块加载数据很慢”的核心原因不是前端单个 loading 组件问题，而是以下因素叠加：

1. 首屏加载了非当前 Tab 的全量数据。
2. 多个页面仍使用全量接口，再在前端本地分页、搜索或截取前 20 条。
3. 后端部分接口查询跨表关联、无分页或返回体过大。
4. Axios 对 5xx、网络错误、超时会自动重试 3 次，慢接口会被放大成更长等待。
5. 客户画像相关 chunk 包体较大，进入客户画像页时下载和解析成本高。

建议先做“首屏只加载当前视图 + 全量接口改分页/搜索 + 慢查询索引和接口缓存”，这是对用户感知提升最大的路径。

- 前端：`http://localhost:5173`

## 3. 首屏慢的主要代码位置

### 3.1 客户数据页

文件：`src/app/pages/CustomerData.tsx`

当前页面进入后，不管用户停留在“基础信息 / 消费记录 / 肌肤档案”哪个 Tab，都会同时触发：

- `getCustomerConsumptionRecords()`
- `getCustomerHealthProfiles()`
- `getCustomers({ storeName })`
- `usePagination(getCustomersPaginated)`

影响：

- 基础信息页本来只需要 `/customers/paginated`。
- 但页面同时请求全量消费记录、全量肌肤档案、全量客户。
- 其中全量消费记录和全量客户已实测 30 秒超时。

### 3.2 智能推荐页

文件：`src/app/pages/MarketingRecommendation.tsx`

当前模式：

- 首屏调用 `/marketing/recommendations`。
- 后台继续加载客户画像上下文：
  - `getCustomersPaginated({ page: 1, pageSize: 2000 })`
  - `getCustomerHealthProfiles()`
  - `getCustomerConsumptionRecords()`
- 页面还会执行 `computeBehaviorProfiles(customers, consumptionRecords, healthProfiles)`。

影响：

- 推荐接口本身已实测 45 秒超时。
- 后台大数据加载完成后还会触发前端计算，容易造成页面二次卡顿。

### 3.3 订单、预约、商品相关弹窗

典型代码：

- `src/app/pages/ProductOrderManagement.tsx`
- `src/app/pages/ProjectOrderManagement.tsx`
- `src/app/pages/ProjectReservation.tsx`
- `src/app/pages/GoodsProductManagement.tsx`

当前模式：

- 创建订单或预约时，通过 `getCustomers({ keyword, storeName })` 拉全量客户，然后前端 `slice(0, 20)`。

影响：

- 用户只是搜索客户，但接口仍可能扫描和返回大量客户。
- 在客户量变大后，开单、预约、收银都会受影响。

## 4. 后端查询问题

### 4.1 全量接口仍在返回大数组

需要重点处理：

- `GET /api/customers`
- `GET /api/customers/consumption-records`
- `GET /api/customers/health-profiles`
- `GET /api/projects`
- `GET /api/products`

建议：

- 保留兼容，但前端主流程不再依赖。
- 新增或改用分页接口、搜索接口、摘要接口。
- 对全量接口增加默认 `take` 上限，例如 200 或 500，避免误用拖垮页面。

### 4.2 客户分页接口仍偏慢

`/customers/paginated?page=1&pageSize=10` 实测 2509ms。

可能原因：

- 查询 include 了 `store`、`balanceAccounts`、`customerCards`。
- `count` 与 `findMany` 并行，但都要处理相同 where。
- “全部门店”下数据范围更大。

建议：

- 首屏列表只 select 必要字段，避免 include 余额和卡数量。
- 余额、会员卡数量改成客户详情或独立懒加载。
- 增加组合索引：
  - `Customer(storeId, deletedAt, createdAt)`
  - `Customer(storeId, deletedAt, name)`
  - `Customer(storeId, deletedAt, phone)`
  - `ConsumptionRecord(customerId, consumeTime)`
  - 如消费记录按门店查，建议增加 `ConsumptionRecord(storeId, consumeTime)`，或在消费记录表冗余 `storeId`。

### 4.3 智能推荐接口需要缓存

`/marketing/recommendations` 实测 45 秒超时。

代码中已存在“每日推荐运行”的方向，但接口仍会做：

- 获取最新 run。
- 获取 snapshot。
- 获取近期行为信号。
- 获取实时信号。
- 组合推荐卡。

建议：

- 每天首次计算后落库缓存推荐卡。
- 页面进入只读当天缓存，不重新组合重计算。
- “刷新推荐”才触发异步任务，前端显示任务状态。
- 推荐接口目标响应时间应小于 800ms。

## 5. 前端优化建议

### 5.1 首屏只加载当前 Tab

客户数据页改为：

- 默认 Tab “基础信息”：只请求 `/customers/paginated`。
- 切到“消费记录”：再请求 `/customers/consumption-records/paginated`。
- 切到“肌肤档案”：再请求 `/customers/health-profiles/paginated`。
- AI 肤质检测弹窗打开时，再按关键词搜索客户。

### 5.2 所有搜索下拉改为远程搜索

以下场景不要再 `getCustomers()` 全量拉取：

- 商品订单新增客户选择。
- 项目订单新增客户选择。
- 项目预约客户选择。
- 商品售卖客户选择。
- AI 肤质检测客户选择。

建议新增统一接口：

```text
GET /api/customers/search?keyword=&storeId=&limit=20
```

返回字段：

```json
[
  { "id": 1, "name": "客户名", "phone": "138****0000", "storeName": "门店名", "memberLevel": "会员等级" }
]
```

### 5.3 Loading 状态拆分

当前多个页面用一个 loading 覆盖整个内容。

建议拆分为：

- 首屏主列表 loading。
- 辅助数据 loading。
- 弹窗下拉 loading。
- 后台画像增强 loading。

这样基础列表先显示，画像、消费记录、推荐证据可以后加载。

### 5.4 前端缓存

建议引入 React Query 或 SWR 风格缓存，至少对以下数据加内存缓存：

- 门店列表。
- 商品类型/产品类型。
- 项目类型。
- 美容师等级。
- 自动营销触发规则。
- 商品/项目下拉候选。

缓存策略：

- 基础字典 5-10 分钟。
- 当前页分页数据 30-60 秒。
- 智能推荐当天缓存。

### 5.5 包体拆分

当前较大 chunk：

- `UserProfile`：约 1172KB。
- `customerSegmentation`：约 921KB。
- `vendor-utils`：约 452KB。

建议：

- 客户画像算法和大图表按 Tab 懒加载。
- `xlsx`、复杂图表、客户分群算法按需 import。
- 客户画像页初始只加载概览，预测视角/消费画像点击后再加载。

## 6. 后端接口改造清单

建议新增：

```text
GET /api/customers/search
GET /api/customers/consumption-records/paginated
GET /api/customers/health-profiles/paginated
GET /api/customers/health-profiles/summary
GET /api/customers/consumption-records/summary
GET /api/marketing/recommendations/latest
POST /api/marketing/recommendations/refresh
GET /api/marketing/recommendations/refresh-status/:jobId
```

建议调整：

- `/api/customers`：增加默认 limit 上限，或只保留给导出/兼容场景。
- `/api/projects`：项目选择下拉使用轻量字段。
- `/api/products`：订单/营销页使用轻量字段，不返回完整商品详情。
- `/api/marketing/automation/effects`：改为缓存聚合结果，避免每次实时计算。

## 7. 推荐实施顺序

### 第一批：直接改善用户感知

1. 客户数据页按 Tab 懒加载。
2. 新增客户远程搜索接口，替换订单/预约/AI 检测中的全量客户加载。
3. 新增消费记录、肌肤档案分页接口。
4. 智能推荐改读当天缓存，刷新改为异步。

预期效果：

- 客户数据页首屏从 30 秒级降到 1-2 秒内。
- 订单/预约弹窗打开不再卡顿。
- 智能推荐页不再超时。

### 第二批：后端查询和索引

1. 客户列表 select 精简。
2. 增加组合索引。
3. 消费记录冗余 `storeId` 或优化 join 查询。
4. 自动营销 effects 聚合缓存。

预期效果：

- 分页接口稳定进入 300-800ms。
- 大店/全部门店场景不会明显拖慢。

### 第三批：前端工程优化

1. 引入请求缓存。
2. 大 chunk 拆分。
3. 导出功能改为后端异步导出，避免前端一次性拿全量数据。

## 8. 验收指标

建议设定以下性能门槛：

| 场景 | 目标 |
| --- | ---: |
| 客户数据基础信息首屏 | P95 < 1500ms |
| 客户分页接口 | P95 < 800ms |
| 消费记录分页接口 | P95 < 1000ms |
| 肌肤档案分页接口 | P95 < 1000ms |
| 客户远程搜索 | P95 < 500ms |
| 智能推荐读取缓存 | P95 < 800ms |
| 商品/项目/库存分页接口 | P95 < 1000ms |
| 前端单页面首包解析 | P95 < 1500ms |

## 9. 风险与注意事项

1. 不建议继续用全量接口支撑页面搜索，否则数据规模上来后必然卡顿。
2. 不建议在页面 mounted 时预加载所有 Tab 数据，应按用户操作懒加载。
3. 导出可以例外使用全量数据，但应改成后端异步导出或单独导出接口。
4. Axios 当前 15 秒超时 + 3 次重试会放大慢接口等待，应区分 GET 全量接口和关键轻量接口的重试策略。
5. “全部门店”比单门店更重，应在后端按用户权限和当前门店做明确范围限制。

## 10. 建议立即改动点

建议下一步直接实施：

1. `CustomerData.tsx`：基础信息页只加载分页客户；消费记录和肌肤档案切 Tab 后加载。
2. `customers.service.ts`：新增消费记录/肌肤档案分页接口。
3. `customers.service.ts`：新增客户搜索轻量接口。
4. 替换 `ProductOrderManagement`、`ProjectOrderManagement`、`ProjectReservation`、`GoodsProductManagement` 中的 `getCustomers()` 全量搜索。
5. `marketing.service.ts`：推荐卡每日缓存读取，刷新走异步任务。
