# 智能推荐算法升级需求文档

版本：v1.0  
日期：2026-06-02  
适用范围：管理端智能推荐、自动营销、活动创建、小程序活动页、客户画像预测视角  
相关文档：`docs/marketing-trigger-rules-requirements.md`、`docs/算法优化.md`、`docs/api-contract.md`

## 1. 背景

当前系统已具备第一版“准模型化预测”能力，后端通过 `rules-v1` 生成客户级预测快照，覆盖流失风险、30 天复购概率、营销响应分、LTV 分层，并在智能推荐页生成推荐卡。

但从美业门店真实经营看，仅依赖流失、复购、响应、LTV 四类分数仍偏薄。门店日常营销还高度依赖节假日、季节换肤、优惠券到期、次卡到期、护理周期、小程序浏览行为、预约放弃、领券未核销等即时信号。若这些信号没有进入推荐算法，智能推荐容易变成“泛泛建议”，无法稳定支撑下一步创建营销活动或自动营销规则。

本次升级目标是把智能推荐从“预测分驱动”升级为“事件 + 场景 + 资产 + 生命周期驱动”的推荐引擎。

## 2. 当前现状

### 2.1 已具备能力

- 后端可生成 `PredictionRun` 与 `CustomerPredictionSnapshot`。
- 智能推荐页可展示推荐卡、匹配度、推荐原因、目标客户、预计营收、数据依据。
- 推荐卡可进入两条路径：创建普通营销活动、创建自动营销规则。
- 自动营销执行可生成 `MarketingAutomationExecution` 与客户级 `MarketingAutomationTouch`。
- 已存在部分规则类型：沉睡客户、会员等级、肤质、护理周期、卡项到期等。

### 2.2 主要不足

- 推荐卡目前偏“文案型”，推荐策略、优惠方式、项目/商品多为文本，不够结构化。
- `preferAutoRule: boolean` 只能表达是否优先自动规则，无法表达“仅活动、仅自动规则、两者均可”。
- 节假日、季节变化、优惠券到期、浏览行为、预约放弃等关键经营信号未系统化纳入推荐。
- 普通活动创建时没有固化推荐命中的客户名单与预测批次，后续归因弱。
- 活动优惠、项目/商品、触达渠道、规则参数缺少统一结构，导致下一步创建活动/规则时仍需要人工补全。
- 推荐算法暂未充分考虑触达疲劳、门店排班、库存、项目容量和渠道偏好。

## 3. 升级目标

### 3.1 产品目标

- 让推荐卡从“建议展示”升级为“下一步动作卡”。
- 让每张推荐卡明确推荐执行方式：单次营销活动、自动营销规则、两者均可。
- 让推荐卡携带结构化策略、优惠、项目/商品、渠道、触发规则、目标客户快照。
- 支持美业成熟场景：节日营销、季节护肤、优惠券到期、次卡到期、护理周期、浏览未预约、领券未核销、预约放弃。
- 让活动创建和自动规则创建都能从推荐卡直接带入关键配置，减少运营人员二次填写。

### 3.2 业务目标

- 提高推荐可执行率：推荐卡可直接创建活动或规则。
- 提高转化率：按客户所处时机和行为意图推荐合适动作。
- 降低打扰：加入触达频控、疲劳惩罚和排除规则。
- 支持复盘：推荐、触达、预约、核销、下单、收入能串成闭环。

## 5. 推荐执行方式分层

推荐卡需要支持三种执行方式：

| 执行方式 | 说明 | 适合场景 |
| --- | --- | --- |
| `activity` 单次营销活动 | 有明确主题、时间、海报、小程序落地页 | 节日活动、季节活动、老带新、限时体验 |
| `automation` 自动营销规则 | 按客户事件或状态持续触发 | 券到期、次卡到期、护理周期、浏览未预约、沉睡唤醒 |
| `both` 两者均可 | 可做一次活动，也可沉淀成自动规则 | 高 LTV 维护、交叉销售、流失唤醒、季节护肤 |

原字段 `preferAutoRule` 建议升级为：

```ts
executionModes: Array<'activity' | 'automation'>;
preferredMode: 'activity' | 'automation';
modeReason: string;
```

示例：

```json
{
  "executionModes": ["automation", "activity"],
  "preferredMode": "automation",
  "modeReason": "客户命中条件会持续变化，适合配置为长期自动触达规则，同时可作为本月唤醒活动使用"
}
```

## 6. 推荐算法升级方向

### 6.1 推荐生成公式

推荐卡综合评分建议从单一预测分升级为多维评分：

```text
recommendationScore =
  customerValueScore * 0.18
  + lifecycleScore * 0.14
  + behaviorIntentScore * 0.18
  + timingUrgencyScore * 0.16
  + offerFitScore * 0.12
  + channelFitScore * 0.08
  + operationFitScore * 0.08
  - fatiguePenalty * 0.10
```

各维度含义：

| 维度 | 含义 | 示例数据 |
| --- | --- | --- |
| `customerValueScore` | 客户价值 | 累计消费、LTV、会员等级 |
| `lifecycleScore` | 生命周期状态 | 新客、活跃、复购期、沉睡、流失 |
| `behaviorIntentScore` | 行为意图 | 浏览项目、领券、预约放弃、加购 |
| `timingUrgencyScore` | 时间紧迫度 | 券到期、卡到期、节日前、护理周期到期 |
| `offerFitScore` | 优惠适配度 | 客户促销敏感度、历史优惠使用 |
| `channelFitScore` | 渠道适配度 | 短信、小程序、微信、门店顾问 |
| `operationFitScore` | 门店承接能力 | 库存、排班、项目容量、过期库存 |
| `fatiguePenalty` | 触达疲劳惩罚 | 近 7/14/30 天触达次数、未响应次数 |

### 6.2 推荐优先级

| 优先级 | 推荐类型 | 原因 |
| --- | --- | --- |
| P0 | 优惠券即将到期、次卡即将到期、预约放弃、极高流失风险 | 时间敏感，错过即损失 |
| P1 | 护理周期到期、浏览未预约、领券未核销、高营销响应客户 | 转化窗口明显 |
| P2 | 节假日活动、季节护肤、高 LTV 维护、产品补货 | 适合批量运营 |
| P3 | 老带新、拼团、泛主题活动 | 依赖门店活动策略 |

## 9. 新增推荐类型需求

### 9.1 优惠券到期推荐

规则类型：`coupon_expiry`

触发条件：

- 客户有未使用优惠券。
- 优惠券将在 7 天、3 天、1 天内到期。
- 客户最近未预约或未下单。

推荐动作：

- D-7：小程序提醒 + 项目推荐。
- D-3：短信/微信提醒 + 预约入口。
- D-1：强提醒，但需受频控限制。

执行方式：自动规则。

### 9.2 次卡/套餐到期推荐

规则类型：`card_expiry`

触发条件：

- 次卡或套餐将在 30 天内到期。
- 剩余次数小于等于 1。
- 客户最近 14 天未预约。

推荐动作：

- 提醒使用剩余权益。
- 推荐续卡优惠。
- 推荐同类升级套餐。

执行方式：自动规则。

### 9.3 护理周期到期推荐

规则类型：`care_cycle`

触发条件：

- 上次完成面部护理后满 21/28/45 天。
- 按项目类型配置不同周期。
- 排除最近已预约客户。

推荐动作：

- 推送复购提醒。
- 推荐同类护理或升级护理。
- 搭配小额优惠券。

执行方式：自动规则。

### 9.4 小程序浏览未预约推荐

规则类型：`browse_abandonment`

触发条件：

- 客户浏览项目详情、活动页或商品详情。
- 24 小时内未预约、未领券、未下单。
- 浏览次数达到阈值，或停留时长达到阈值。

推荐动作：

- 推荐浏览项目对应的体验券。
- 推送案例、功效、适用肤质说明。
- 给顾问生成跟进话术。

执行方式：自动规则。

### 9.5 预约放弃推荐

规则类型：`booking_abandonment`

触发条件：

- 客户进入预约流程但未提交。
- 选择过项目、美容师或时间段。
- 2 小时内未再次预约。

推荐动作：

- 小程序提醒继续预约。
- 若时间段冲突，推荐相邻时间段。
- 若顾客多次放弃，推送顾问协助。

执行方式：自动规则。

### 9.6 季节换肤推荐

规则类型：`seasonal_skin_care`

触发条件：

- 当前处于季节换肤窗口。
- 客户肤质或历史护理偏好匹配当季护理主题。

推荐动作：

- 春季：敏感修护、屏障修复。
- 夏季：防晒、控油、清洁、美白。
- 秋冬：补水、抗干燥、深层滋养。

执行方式：单次活动、自动规则。

### 9.7 节假日营销推荐

规则类型：`holiday_campaign`

触发条件：

- 节日前 15-30 天进入预热期。
- 节日主题与客户群体匹配。
- 门店项目、库存、排班可承接。

推荐动作：

- 女神节：女性会员护理礼遇。
- 母亲节：亲情套餐、礼品卡。
- 520/七夕：双人护理、闺蜜同行。
- 双 11/双 12：储值、套餐、商品组合。

执行方式：单次活动。

### 9.8 高价值客户权益维护

规则类型：`vip_privilege_care`

触发条件：

- LTV 分层为铂金/黄金。
- 最近消费趋势下降，或即将进入生日/周年/季度维护窗口。

推荐动作：

- 顾问 1v1 跟进。
- VIP 专属护理权益。
- 预约优先权、生日礼遇、季度礼包。

执行方式：两者均可。

## 10. 创建活动承接需求

从推荐卡创建普通活动时，应自动带入：

- 活动名称：来自推荐卡 `title`。
- 活动说明：来自推荐卡 `reason`，但需要转化为客户可见文案。
- 活动类型：根据 `category` 和 `preferredMode` 自动映射。
- 目标客户：使用 `audienceSnapshot.customerIds` 固化。
- 推荐项目/商品：使用 `recommendedItems`。
- 优惠配置：使用 `offer`。
- 渠道：使用 `recommendedChannels`。
- 活动周期：根据规则类型自动生成。
- 来源归因：写入 `sourceRecommendationId`、`predictionRunId`、`audienceSnapshotId`。

活动创建后，应保留推荐来源，便于效果分析回答：

- 这次活动来自哪张推荐卡？
- 哪批预测生成？
- 当时命中了哪些客户？
- 后续多少客户预约、到店、核销、下单？

## 11. 创建自动营销规则承接需求

从推荐卡创建自动规则时，应自动带入：

- 策略名称。
- 策略说明。
- 触发规则类型 `triggerRule.type`。
- 触发参数 `triggerRule.params`。
- 规则关系，默认 `AND`。
- 动作类型，如优惠券、小程序推送、短信、顾问任务。
- 渠道 `recommendedChannels`。
- 优惠 `offer`。
- 推荐项目/商品 `recommendedItems`。
- 预计命中客户。

创建前必须支持预览：

- 预计命中人数。
- 预计触达人数。
- 预计转化人数。
- 预计收入。
- 抽样客户及命中原因。

## 14. 前端需求

### 14.1 智能推荐页

- 推荐卡展示执行方式标签：单次活动、自动规则、两者均可。
- 推荐卡展示推荐来源：预测、行为、权益到期、节令、库存、排班。
- 推荐卡展示结构化优惠和推荐项目/商品。
- 推荐卡提供两个 CTA：创建活动、创建自动规则。
- 若某推荐只适合一种方式，另一个 CTA 置灰并说明原因。
- 点击目标客户可查看命中客户、命中原因、预测分、行为事件、权益状态。

### 14.2 创建活动弹窗

- 从推荐卡带入目标客户快照。
- 从推荐卡带入优惠、项目/商品、渠道、周期。
- 明确展示“来源于智能推荐”。
- 支持用户编辑，但保留原始推荐参数用于归因。

### 14.3 自动营销页

- 从推荐卡带入规则参数。
- 展示“已使用系统推荐值 / 已自定义”。
- 创建前可预览受众和预计收益。
- 执行后展示客户级触达和预测 vs 实际效果。

## 15. 频控与风控

必须增加触达疲劳控制：

- 同客户同策略 7 天内最多触达 1 次。
- 同客户同渠道 1 天内最多触达 1 次。
- 同客户 30 天内营销触达超过阈值后降低推荐分。
- 已预约客户不再重复推同类预约提醒。
- 已购买或已核销客户不再推同一优惠。
- 高价值客户避免过度低价促销，优先权益和顾问服务。

隐私与合规要求：

- 小程序浏览记录需遵守用户授权与隐私政策。
- 敏感字段脱敏展示。
- 推荐原因避免暴露“流失风险”“高价值”等内部标签给客户。

## 16. 验收标准

### 16.1 推荐生成

- 系统可生成至少 8 类推荐：券到期、卡到期、护理周期、浏览未预约、预约放弃、节日活动、季节护肤、VIP 维护。
- 每张推荐卡必须包含 `executionModes`、`preferredMode`、`offer`、`recommendedItems`、`recommendedChannels`。
- 每张推荐卡必须包含可解释数据依据。

### 16.2 活动承接

- 从推荐卡创建活动时，活动名称、目标客户、优惠、项目/商品、渠道、周期可自动带入。
- 活动保存后能记录 `sourceRecommendationId` 与 `predictionRunId`。
- 活动效果分析可回溯推荐来源。

### 16.3 自动规则承接

- 从推荐卡创建自动规则时，规则类型和默认参数可自动带入。
- 创建前可预览命中客户、预计转化和预计收入。
- 执行后能生成客户级触达记录。

### 16.4 用户体验

- 运营人员能理解为什么推荐。
- 运营人员能判断应该创建活动还是自动规则。
- 运营人员可以修改系统推荐值。
- 推荐卡不会把内部敏感标签直接展示给客户。

## 17. 分阶段实施计划

### Phase 1：推荐卡结构升级

- 增加 `executionModes`、`preferredMode`、`offer`、`recommendedItems`、`recommendedChannels`。
- 保持旧字段兼容。
- 智能推荐页展示执行方式和结构化信息。

### Phase 2：P0/P1 事件规则接入

- 接入优惠券到期。
- 接入次卡/套餐到期。
- 接入护理周期到期。
- 接入小程序浏览未预约。
- 接入预约放弃。

### Phase 3：活动/规则草稿生成

- 新增推荐生成活动草稿接口。
- 新增推荐生成自动规则草稿接口。
- 创建活动和自动规则页面使用结构化推荐参数。

### Phase 4：推荐归因闭环

- 落推荐事件。
- 落推荐受众快照。
- 活动、自动规则、触达、预约、核销、订单收入串联。
- 效果分析支持推荐来源维度。

### Phase 5：经营场景增强

- 加入节假日活动日历。
- 加入季节护肤规则。
- 加入门店排班和库存承接能力。
- 加入触达疲劳惩罚和顾问任务推荐。

## 18. 风险与待确认问题

| 问题 | 影响 | 建议 |
| --- | --- | --- |
| 小程序行为事件目前是否完整采集 | 浏览未预约、预约放弃依赖事件数据 | 先定义事件埋点协议 |
| 优惠券/次卡数据是否已完整落库 | 券到期、卡到期推荐依赖权益资产 | 梳理券、卡、核销表 |
| 推荐项目/商品如何选择 | 影响活动创建质量 | 先用规则匹配，后续引入协同过滤 |
| 触达频控是否已有统一表 | 影响客户打扰控制 | 优先复用 `MarketingAutomationTouch` |
| 活动是否需要固化客户名单 | 影响归因准确性 | 建议新增受众快照 |
| 高价值客户是否适合优惠券 | 影响品牌体验 | 高价值客户优先权益和顾问服务 |

## 19. 结论

智能推荐下一步的核心不是增加更多文案，而是把推荐卡升级为可执行、可解释、可归因的结构化决策包。

优先落地顺序建议：

1. 升级推荐卡结构，明确活动/自动规则/两者均可。
2. 补齐优惠券到期、次卡到期、护理周期、小程序浏览未预约、预约放弃五类高价值规则。
3. 让推荐卡自动生成活动草稿和自动规则草稿。
4. 固化推荐受众快照和推荐来源，打通后续归因。

完成后，智能推荐将从“算法建议页”升级为“门店营销作战台”。
